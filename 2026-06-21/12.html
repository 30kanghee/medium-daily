Value Objects in .NET 10: Stop Passing Strings Around Your Domain

Michael Maurice · 8 min read · 원문 · freedium

.NET 10에서 값 객체(Value Object)를 사용해 도메인 모델의 원시 타입 집착을 제거하고, EF Core의 값 변환·복합 타입·소유 엔티티를 통해 올바르게 영속화하는 방법을 실전 코드로 설명한다.

요약

원시 타입 집착(Primitive Obsession)은 코드베이스를 조용히 망가뜨리는 주요 원인이다. 이메일·화폐·수량 같은 도메인 개념을 단순 문자열·정수로 다루면 유효성 검사가 애플리케이션 전체에 흩어진다. 값 객체는 불변·값 동등성을 가진 작은 타입으로 이 문제를 해결하며, private 생성자와 팩토리 메서드로 잘못된 상태가 애당초 생성되지 못하게 한다. EF Core에서는 단일 컬럼 객체에 값 변환, 다중 컬럼 구조에 복합 타입, 집계 컬렉션에 소유 엔티티를 활용한다. 값 객체 테스트는 저렴하면서도 효과적이고, 과도한 복잡화(DB 접근·서비스 호출 포함)는 피해야 한다.

원시 타입 집착(Primitive Obsession)은 코드베이스가 조용히 취약해지는 방식 중 하나다. 고객 ID는 문자열이다. 이메일은 문자열이다. 통화는 문자열이다. 금액은 decimal이다. 수량은 int다. 모든 것이 컴파일되지만, 모델은 스스로를 보호하지 않는다.

값 객체(Value Object)는 중요한 도메인 개념을 명시적 타입으로 바꿔 이 문제를 해결한다. 값 객체는 데이터베이스 키가 아닌 자신의 값으로 동등성이 결정되는, 작고 불변한 객체다. Milan Jovanovic이 도메인 모델을 더 표현력 있게 만들기 위해 자주 사용하는 패턴이며, EF Core는 이제 소유 엔티티 타입, 값 변환, 복합 타입을 통해 .NET 개발자에게 더 나은 영속화 옵션을 제공한다.

이 글은 .NET 10에서 프로덕션 수준의 값 객체 설계를 구축하고, EF Core로 영속화하는 방법을 보여주며, 값 변환·소유 엔티티·복합 타입 중 언제 무엇을 사용해야 하는지 설명한다.

문제

이 코드는 정상처럼 보이지만 경계가 약하다:

public sealed record CreateOrderRequest(
    string CustomerId,
    string Email,
    decimal Amount,
    string Currency);

유효하지 않은 이메일, 음수 금액, 소문자 통화 코드가 애플리케이션을 흘러 들어오는 것을 막는 것이 아무것도 없다. 컨트롤러, 핸들러, 엔티티, 데이터베이스 모두 동일한 규칙을 기억해야 한다.

더 나은 모델은 유효하지 않은 상태를 표현하기 어렵게 만든다.

설계 규칙

개념에 유효성 검사, 서식 지정, 정규화, 비교 규칙, 또는 도메인 언어가 붙어 있을 때 값 객체를 사용하라.

  • 이메일은 단순한 문자열이 아니다.
  • 금액은 단순한 decimal이 아니다.
  • 통화는 단순한 세 글자가 아니다.
  • 수량은 단순한 int가 아니다.
  • 주문 번호는 단순한 텍스트가 아니다.

모든 원시 타입에 자동으로 값 객체를 만들지 마라. 타입은 모호성을 제거하거나 규칙을 강제함으로써 자신의 자리를 얻어야 한다.

Email 값 객체

이메일은 단일 값 객체로 적합하다. 입력을 정규화하고 생성 시점에 유효하지 않은 값을 거부한다.

public readonly record struct Email
{
    public string Value { get; }
    private Email(string value)
    {
        Value = value;
    }
    public static Email Create(string value)
    {
        if (string.IsNullOrWhiteSpace(value))
        {
            throw new DomainException("Email is required.");
        }
        var normalized = value.Trim().ToLowerInvariant();
        if (!normalized.Contains('@'))
        {
            throw new DomainException("Email is invalid.");
        }
        return new Email(normalized);
    }
    public override string ToString() => Value;
}

생성자는 private이다. 호출자는 팩토리 메서드를 사용해야 하며, 이는 유효성 검사를 한 곳에 집중시킨다.

Currency 값 객체

통화도 단일 값 객체지만, 다른 규칙이 있다: 길이, 대소문자, 허용 값.

public readonly record struct Currency
{
    private static readonly HashSet<string> Allowed = new(StringComparer.Ordinal)
    {
        "USD",
        "EUR",
        "AED",
        "GBP"
    };
    public string Code { get; }
    private Currency(string code)
    {
        Code = code;
    }
    public static Currency Create(string code)
    {
        var normalized = code.Trim().ToUpperInvariant();
        if (!Allowed.Contains(normalized))
        {
            throw new DomainException($"Currency '{code}' is not supported.");
        }
        return new Currency(normalized);
    }
    public override string ToString() => Code;
}

이렇게 하면 지원하지 않는 통화가 조용히 도메인에 진입하는 대신 경계에서 실패한다.

Money 값 객체

Money는 복합 값 객체다. 금액과 통화를 결합하며, 서로 다른 통화 간의 산술을 방지한다.

public readonly record struct Money
{
    public decimal Amount { get; }
    public Currency Currency { get; }
    private Money(decimal amount, Currency currency)
    {
        Amount = amount;
        Currency = currency;
    }
    public static Money Create(decimal amount, Currency currency)
    {
        if (amount < 0)
        {
            throw new DomainException("Money amount cannot be negative.");
        }
        return new Money(decimal.Round(amount, 2), currency);
    }
    public Money Add(Money other)
    {
        if (Currency != other.Currency)
        {
            throw new DomainException("Cannot add money with different currencies.");
        }
        return Create(Amount + other.Amount, Currency);
    }
}

핵심 장점은 도메인 규칙이 자신이 보호하는 개념과 함께 위치한다는 것이다.

Quantity 값 객체

Quantity는 0이나 음수 주문 라인을 방지한다.

public readonly record struct Quantity
{
    public int Value { get; }
    private Quantity(int value)
    {
        Value = value;
    }
    public static Quantity Create(int value)
    {
        if (value <= 0)
        {
            throw new DomainException("Quantity must be greater than zero.");
        }
        return new Quantity(value);
    }
    public override string ToString() => Value.ToString();
}

작아 보이지만, 애플리케이션 전체에서 반복되는 가드 절을 제거한다.

도메인 예외

유효하지 않은 도메인 입력에는 특정 예외 타입을 사용하라. API 레이어가 이를 400 응답으로 변환할 수 있다.

public sealed class DomainException : Exception
{
    public DomainException(string message)
        : base(message)
    {
    }
}

이것을 인프라 장애를 숨기는 데 사용하지 마라. 데이터베이스 오류, 네트워크 타임아웃, 손상된 의존성은 다른 범주다.

Order 엔티티

엔티티는 이제 원시 타입 대신 도메인 타입을 사용한다.

public sealed class Order
{
    private readonly List<OrderLine> _lines = [];
    private Order(Guid id, Email customerEmail, Money total)
    {
        Id = id;
        CustomerEmail = customerEmail;
        Total = total;
        CreatedAt = DateTimeOffset.UtcNow;
    }
    public Guid Id { get; private set; }
    public Email CustomerEmail { get; private set; }
    public Money Total { get; private set; }
    public DateTimeOffset CreatedAt { get; private set; }
    public IReadOnlyCollection<OrderLine> Lines => _lines;
    public static Order Create(Email customerEmail, IReadOnlyCollection<OrderLine> lines)
    {
        if (lines.Count == 0)
        {
            throw new DomainException("Order must contain at least one line.");
        }
        var total = lines
            .Select(line => line.Total)
            .Aggregate((left, right) => left.Add(right));
        var order = new Order(Guid.NewGuid(), customerEmail, total);
        order._lines.AddRange(lines);
        return order;
    }
}

값 객체가 이미 자체 규칙을 강제하기 때문에 엔티티는 더 적은 유효하지 않은 경로를 노출한다.

OrderLine 엔티티

OrderLine은 집계의 일부다. 값 객체도 사용한다.

public sealed class OrderLine
{
    private OrderLine(Guid productId, Quantity quantity, Money unitPrice)
    {
        ProductId = productId;
        Quantity = quantity;
        UnitPrice = unitPrice;
        Total = Money.Create(unitPrice.Amount * quantity.Value, unitPrice.Currency);
    }
    public Guid ProductId { get; private set; }
    public Quantity Quantity { get; private set; }
    public Money UnitPrice { get; private set; }
    public Money Total { get; private set; }
    public static OrderLine Create(Guid productId, Quantity quantity, Money unitPrice)
    {
        return new OrderLine(productId, quantity, unitPrice);
    }
}

Total은 클라이언트에서 신뢰하는 것이 아니라 모델이 계산한다.

요청 모델

API 요청은 여전히 원시 타입을 사용한다. 괜찮다. HTTP는 당신의 도메인 모델이 아니다.

public sealed record CreateOrderRequest(
    string CustomerEmail,
    IReadOnlyCollection<CreateOrderLineRequest> Lines);

각 주문 라인 요청은 클라이언트의 원시 입력을 담는다.

public sealed record CreateOrderLineRequest(
    Guid ProductId,
    int Quantity,
    decimal UnitPrice,
    string Currency);

애플리케이션 경계에서 원시 타입을 값 객체로 변환한다.

커맨드 핸들러

핸들러는 요청 데이터를 도메인 모델로 변환하고 집계를 영속화한다.

public sealed class CreateOrderHandler(AppDbContext db)
{
    public async Task<Guid> Handle(
        CreateOrderRequest request,
        CancellationToken cancellationToken)
    {
        var email = Email.Create(request.CustomerEmail);
        var lines = request.Lines
            .Select(line => OrderLine.Create(
                line.ProductId,
                Quantity.Create(line.Quantity),
                Money.Create(line.UnitPrice, Currency.Create(line.Currency))))
            .ToArray();
        var order = Order.Create(email, lines);
        db.Orders.Add(order);
        await db.SaveChangesAsync(cancellationToken);
        return order.Id;
    }
}

여기서 유효하지 않은 클라이언트 입력이 거부된다. 도메인 모델은 부분적으로 유효한 데이터를 받지 않는다.

엔드포인트

엔드포인트는 HTTP 변환을 처리한다.

app.MapPost("/orders", async (
    CreateOrderRequest request,
    CreateOrderHandler handler,
    CancellationToken cancellationToken) =>
{
    try
    {
        var orderId = await handler.Handle(request, cancellationToken);
        return Results.Created($"/orders/{orderId}", new { orderId });
    }
    catch (DomainException exception)
    {
        return Results.BadRequest(new { error = exception.Message });
    }
});

대규모 API의 경우 이 예외 매핑을 미들웨어 또는 엔드포인트 필터로 이동하라.

EF Core 매핑: 단일 값 객체

Email은 값 변환을 통해 단일 컬럼으로 저장할 수 있다.

modelBuilder.Entity<Order>()
    .Property(order => order.CustomerEmail)
    .HasConversion(
        email => email.Value,
        value => Email.Create(value))
    .HasMaxLength(320)
    .HasColumnName("customer_email");

Quantity도 동일한 방식을 사용할 수 있다.

modelBuilder.Entity<OrderLine>()
    .Property(line => line.Quantity)
    .HasConversion(
        quantity => quantity.Value,
        value => Quantity.Create(value))
    .HasColumnName("quantity");

값 변환은 값 객체가 하나의 데이터베이스 컬럼을 래핑할 때 적합하다.

EF Core 매핑: Money를 복합 타입으로

Money는 여러 값을 포함한다. EF Core 복합 타입은 하나의 컬럼으로 직렬화하는 것보다 더 적합하다.

modelBuilder.Entity<Order>()
    .ComplexProperty(order => order.Total, money =>
    {
        money.Property(x => x.Amount)
            .HasColumnName("total_amount")
            .HasPrecision(18, 2);
        money.Property(x => x.Currency)
            .HasConversion(
                currency => currency.Code,
                code => Currency.Create(code))
            .HasColumnName("total_currency")
            .HasMaxLength(3);
    });

복합 타입은 값 시맨틱을 가지며 구조화된 값 객체에 유용하다. EF Core 10은 이전의 소유 엔티티 우회 방법과 비교해 복합 타입 지원도 개선했다.

EF Core 매핑: 주문 라인

주문 라인은 주문 집계에 소유된다.

modelBuilder.Entity<Order>()
    .OwnsMany(order => order.Lines, lines =>
    {
        lines.ToTable("order_lines");
        lines.WithOwner().HasForeignKey("order_id");
        lines.Property<Guid>("id");
        lines.HasKey("id");
        lines.Property(line => line.ProductId).HasColumnName("product_id");
    });

주문 라인은 주문과 독립적으로 존재하지 않으므로 소유 컬렉션이 적합하다.

DbContext

DbContext는 영속화 규칙을 한 곳에 유지한다.

public sealed class AppDbContext(DbContextOptions<AppDbContext> options)
    : DbContext(options)
{
    public DbSet<Order> Orders => Set<Order>();
    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.ApplyConfigurationsFromAssembly(typeof(AppDbContext).Assembly);
    }
}

실제 프로젝트에서는 긴 OnModelCreating 메서드 대신 IEntityTypeConfiguration 클래스에 엔티티 매핑을 배치하라.

올바른 영속화 패턴 선택하기

Email, Currency, OrderNumber, Quantity처럼 하나의 컬럼에 매핑되는 값 객체에는 값 변환을 사용하라.

Money, Address, DateRange, GeoPoint처럼 여러 컬럼에 매핑되는 구조화된 값 객체에는 복합 타입을 사용하라.

객체가 집계의 일부이고 컬렉션 동작, 그림자 키, 또는 소유권 시맨틱이 필요할 때는 소유 엔티티를 사용하라.

실패 패턴

값 객체는 너무 영리해지면 문제가 생긴다. 데이터베이스 접근, 서비스 호출, 또는 인프라 로직을 그 안에 넣지 마라. 작고, 결정론적이며, 테스트하기 쉬워야 한다.

또 다른 흔한 실수는 EF Core를 위해 public setter를 노출하고 애플리케이션 코드가 객체를 유효하지 않은 상태로 변형하게 허용하는 것이다. private 생성자, private setter, 팩토리 메서드를 선호하라.

마지막으로, 직렬화에 주의하라. API가 값 객체를 직접 반환하면 클라이언트가 어색한 형태를 볼 수 있다. 보통 API는 도메인 객체를 응답 DTO로 매핑해야 한다.

값 객체 테스트

값 객체 테스트는 저렴하고 가치 있다.

[Fact]
public void Create_Throws_WhenEmailIsInvalid()
{
    var exception = Assert.Throws<DomainException>(
        () => Email.Create("not-an-email"));
    Assert.Equal("Email is invalid.", exception.Message);
}

Money 테스트는 통화 규칙을 확인해야 한다.

[Fact]
public void Add_Throws_WhenCurrenciesAreDifferent()
{
    var usd = Money.Create(10, Currency.Create("USD"));
    var eur = Money.Create(10, Currency.Create("EUR"));
    Assert.Throws<DomainException>(() => usd.Add(eur));
}

값 객체를 사용하지 말아야 할 때

도메인 주도 설계처럼 보이기 위해서만 값 객체를 만들지 마라. 값에 규칙, 동작, 서식, 모호성이 없다면 원시 타입으로도 충분할 수 있다.

목표는 더 많은 타입이 아니다. 더 적은 유효하지 않은 상태와 더 명확한 도메인 언어다.

마치며

값 객체는 .NET 도메인 모델의 품질을 높이는 가장 간단한 방법 중 하나다. 유효성 검사를 해당 개념 가까이 이동시키고, 유효하지 않은 상태를 만들기 어렵게 하며, 애플리케이션 전체의 원시 타입 노이즈를 줄인다.

EF Core 값 변환, 소유 엔티티 타입, 복합 타입을 통해 영속화는 더 이상 값 객체를 피해야 할 이유가 아니다. 중요한 결정은 객체의 형태에 맞는 올바른 매핑을 선택하는 것이다.

이미 버그를 유발하는 개념들부터 시작하라: 이메일, 금액, 통화, 수량, 식별자, 범위, 주소. 그 개념들에 이름을 붙여라. 그러면 컴파일러가 그것들을 보호하도록 도와줄 것이다.
