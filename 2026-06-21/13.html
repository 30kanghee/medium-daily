Spark Structured Streaming: Output Modes Explained with Real-World Examples

Kaizen · 4 min read · 원문 · freedium

Spark Structured Streaming의 세 가지 출력 모드(Append, Update, Complete)를 언제, 왜 사용해야 하는지 실전 예시와 함께 명확히 설명한다.

요약

Spark Structured Streaming에서 출력 모드(Output Mode)는 각 마이크로 배치 처리 후 결과의 어느 부분을 출력 싱크에 기록할지를 결정하는 핵심 설정이다. Append 모드는 이전 배치 이후 새로 추가된 행만 기록하며, 집계 없는 이벤트 스트림이나 워터마크가 있는 윈도우 집계에 적합하다. Update 모드는 변경되거나 새로 추가된 행만 기록하여 실시간 대시보드처럼 업데이트가 잦은 경우에 가장 실용적이다. Complete 모드는 매 배치마다 전체 결과 테이블을 기록하므로 디버깅이나 소규모 데이터셋에 적합하지만 프로덕션에서는 비용이 크다. 잘못된 출력 모드 선택은 런타임 오류나 심각한 성능 저하로 이어질 수 있으므로, 집계 여부·결과 변경 가능성·데이터 규모를 기준으로 올바른 모드를 선택해야 한다.

이제 겉으로는 단순해 보이지만 인터뷰와 프로덕션 시스템에서 매우 중요한 개념을 다룬다:

Structured Streaming의 출력 모드(Output Modes)

많은 스트리밍 버그와 성능 이슈는 잘못된 출력 모드 선택에서 비롯된다. 이 글은 각 출력 모드가 무엇인지뿐 아니라, 언제 그리고 왜 사용해야 하는지를 이해하는 데 도움을 줄 것이다.

1. 출력 모드란 무엇인가?

스트리밍에서는 데이터가 마이크로 배치 단위로 지속적으로 처리된다. 각 배치 처리 후 Spark는 다음과 같은 결정을 내려야 한다:

결과의 어느 부분을 출력에 기록할 것인가?

이 결정을 제어하는 것이 바로 출력 모드다.

2. 세 가지 출력 모드

Spark Structured Streaming은 세 가지 출력 모드를 지원한다:

  • Append (추가)
  • Update (업데이트)
  • Complete (완전)

각각의 동작 방식은 매우 다르다.

3. Append 모드

정의:

마지막 마이크로 배치 이후 새로 추가된 행만 기록한다.

예시:

df.writeStream \
  .format("console") \
  .outputMode("append")

동작 방식:

새로운 행만 기록된다.

중요한 규칙:

Append 모드는 나중에 행이 변경되지 않을 때만 동작한다.

사용 가능한 경우:

  • 집계 없는 이벤트 스트림
  • 워터마크가 있는 윈도우 집계

사용 불가한 경우:

  • 전역 집계 (예: groupBy("category"))
  • 카운트가 계속 업데이트되기 때문

4. Update 모드

정의:

마지막 배치 이후 변경된 행만 기록한다.

예시:

df.writeStream \
  .format("console") \
  .outputMode("update")

동작 방식:

변경되거나 새로 추가된 행만 기록된다.

주요 장점:

  • Complete 모드보다 효율적
  • 전체 데이터셋을 재작성하지 않아도 됨

사용 사례:

  • 집계 처리
  • 상태 기반(Stateful) 처리
  • 업데이트가 빈번한 경우

5. Complete 모드

정의:

매 마이크로 배치 후 전체 결과 테이블을 기록한다.

예시:

df.writeStream \
  .format("console") \
  .outputMode("complete")

동작 방식:

출력:

  • 매번 전체 테이블이 출력됨

주요 특성:

  • 항상 최신 전체 상태를 보여줌
  • 대용량 데이터셋에서는 비용이 큼

사용 사례:

  • 디버깅
  • 소규모 데이터셋
  • 학습 목적

대부분의 사람들이 어려움을 겪는 부분이다.

의사 결정 테이블:

7. 출력 모드와 상태 기반 연산의 관계

무상태(Stateless) 연산:

  • Append 모드가 완벽하게 동작

상태 기반(Stateful) 연산:

  • Update 또는 Complete 모드가 필요

그 이유는?

  • 결과가 계속 변경되기 때문

8. 실전 예시

시나리오: 실시간 주문 대시보드

목표:

  • 카테고리별 주문 수 집계
  • 몇 초마다 업데이트

옵션 1: Complete 모드

  • 구현이 쉬움
  • 확장성이 낮음

옵션 2: Update 모드 (더 나은 선택)

  • 변경된 카테고리만 업데이트
  • 대시보드에 효율적

옵션 3: Append 모드

  • 다음 조건을 충족하는 경우에만 동작:
  • window + watermark를 사용할 때
  • 이후 결과가 변경되지 않을 때

9. 흔한 실수

1. 집계에 Append 모드 사용

결과:

  • 런타임 오류 발생

2. 프로덕션에서 Complete 모드 사용

결과:

  • 높은 컴퓨팅 비용
  • 성능 문제

3. 워터마크 요구사항 무시

Append 모드에는 다음이 필요:

  • 확정된 결과
  • 그렇지 않으면 Spark가 "새로운" 것을 판단할 수 없음

10. 인터뷰 관점

이 주제에 대한 질문은 거의 반드시 나온다.

자주 나오는 질문:

  • Append, Update, Complete 모드의 차이점
  • 왜 Append 모드는 집계와 함께 동작하지 않는가
  • 대규모 파이프라인에 가장 적합한 모드는 무엇인가
  • 워터마크와 Append 모드의 관계

강력한 답변 팁:

Append 모드는 새 행만 출력하고, Update 모드는 변경된 행만 출력하며, Complete 모드는 전체 결과를 출력한다. 선택 기준은 데이터가 불변인지, 집계가 사용되는지, 결과가 시간에 따라 변경될 수 있는지에 달려 있다.

11. 핵심 요약

  • 출력 모드는 각 마이크로 배치 후 기록되는 내용을 제어한다
  • Append 모드는 효율적이지만 제한적이다
  • Update 모드는 실제 시스템에서 가장 실용적이다
  • Complete 모드는 주로 디버깅용이다
  • 잘못된 모드 선택은 파이프라인을 망가뜨릴 수 있다

12. 다음 학습 주제

출력 모드를 이해했다면, 다음 중요 주제들이 기다리고 있다:

  • 트리거 유형 (처리 시간 vs 한 번만 vs 연속)
  • 정확히 한 번(Exactly-once) 보장에 대한 심화 학습
  • 엔드투엔드 스트리밍 아키텍처 설계

출력 모드는 작은 설정처럼 보일 수 있지만, 실제로는 전체 스트리밍 파이프라인의 동작 방식을 정의한다. 이를 마스터하는 것은 신뢰할 수 있고 확장 가능한 실시간 시스템을 구축하는 데 필수적이다.
