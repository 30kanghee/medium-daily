Google L5 level Interview: If You Join 10 Tables in an SQL Query, How Would You Optimize Performance?

Umesh Kumar Yadav · Javarevisited · 5 min read · 원문 · freedium

10개 테이블 조인 SQL이 30초 이상 걸릴 때, Google L5 면접 수준의 체계적 진단과 7가지 계층별 최적화 전략을 단계적으로 설명한다.

요약

구글 L5 면접에서 나온 실제 질문을 바탕으로, 10개 테이블 조인 SQL 쿼리가 30초 이상 걸릴 때 어떻게 진단하고 최적화할지를 체계적으로 설명한다. 진단 단계에서는 EXPLAIN 실행 계획 분석, SQL 프로파일링, DB 파라미터 검토 등 3단계 접근법을 제시한다. 성능 저하의 근본 원인은 중첩 루프 조인(Nested Loop Join) 알고리즘의 복잡도이며, 최적화 방법으로는 인덱스 추가, JOIN 순서 조정, 애플리케이션 레이어 분리, 임시 테이블 활용, 요약 테이블 사전 계산, OLAP 엔진 전환, 수직 파티셔닝 등 7가지 기법을 레이어 순서로 제시한다. 단일 기법이 아닌 데이터 볼륨과 비즈니스 요구에 맞는 복합적 접근이 핵심이다.

"운영 환경에서 10개 테이블을 조인하는 SQL 쿼리가 실행에 30초 이상 걸린다면, 성능을 단계적으로 어떻게 트러블슈팅하고 최적화하겠는가?"

진짜 어려운 점은 단순히 SQL을 작성하는 것이 아니라 체계적인 트러블슈팅 방법론을 적용하고 다층적인 최적화 기법을 조합하는 것이다.

이 글은 인터뷰와 실제 현장 모두에서 활용할 수 있는 구조화된 접근법을 제시한다.

1. 병목을 파악하는 3단계

10개 테이블 조인이 포함된 느린 SQL 쿼리를 만났을 때, 즉시 쿼리를 수정하지 마라. 대신 근본 원인을 정확히 찾아내기 위해 아래의 진단 단계를 따라라.

1.1 EXPLAIN으로 실행 계획 분석

EXPLAIN 명령어를 실행해 각 테이블의 접근 유형을 확인한다. 특히 주목해야 할 항목:

  • type 컬럼: ALL(전체 테이블 스캔)이나 index(전체 인덱스 스캔) 같은 값은 최적화 여지를 나타낸다. 이상적인 값은 ref, eq_ref, const다.
  • rows 컬럼: 스캔할 행 수의 추정치다. 예상보다 훨씬 높은 숫자는 추가 인덱스가 필요함을 시사한다.
  • Extra 컬럼: Using temporary(임시 테이블)나 Using filesort(파일 정렬)가 있으면 주요 성능 병목이다.

예시:

EXPLAIN SELECT o.id, o.amount, u.name, p.title
FROM orders o
JOIN users u ON o.user_id = u.id
JOIN products p ON o.product_id = p.id
WHERE o.status = 'PAID';

users 테이블의 type이 ALL이라면, user_id에 인덱스가 없어 전체 테이블 스캔이 발생하고 있음을 의미한다. 명확한 최적화 대상이다.

1.2 실제 SQL 실행 시간 분포 확인

SET profiling = 1;
-- 느린 SQL 쿼리 실행
SELECT ...;
SHOW PROFILES;
SHOW PROFILE FOR QUERY 1;

이 방법으로 각 단계(예: "sending data", "creating sort index")에서 소요된 시간을 확인해 I/O 병목인지 CPU/정렬 병목인지 구분할 수 있다.

1.3 데이터베이스 파라미터 설정 검토

  • join_buffer_size: 값이 작으면 다중 스캔이 발생할 수 있다.
  • tmp_table_size / max_heap_table_size: 값이 부족하면 임시 테이블이 디스크로 밀려난다.
  • innodb_buffer_pool_size: 자주 접근하는 "핫" 데이터를 수용할 수 있는지 확인한다.

2. 10개 테이블 조인이 느린 이유

MySQL(InnoDB)에서 다중 테이블 조인은 기본적으로 중첩 루프 조인(Nested Loop Join) 알고리즘을 사용한다. 드라이빙 테이블에서 행을 선택하고 이어지는 테이블을 반복해 매칭 행을 찾는다.

대략적인 시간 복잡도: 드라이빙 테이블 행 수 × 연관 테이블별 인덱스 스캔 비용.

드라이빙 테이블에 10만 개 행이 있고 각 연관 테이블의 인덱스 스캔 비용이 1ms라면, 10개 테이블의 경우 최악의 상황에서 100,000 × (10 × 1ms) = 1,000초가 될 수 있다.

MySQL 8.0.18 이상은 특정 조건에서 Hash Join을 사용할 수 있다. 인덱스가 있을 때 성능이 향상되지만 메모리와 해시 테이블 구성 오버헤드에 제약이 있다.

3. 7가지 최적화 기법: SQL에서 아키텍처까지

다음 전략들은 비용이 낮고 빠른 SQL 레벨 변경부터 노력이 많이 드는 아키텍처 솔루션 순으로 제시된다.

기법 1: 인덱스 최적화 (가장 빠른 효과)

ON과 WHERE 절에 사용된 모든 컬럼에 인덱스가 있는지 확인한다. LEFT JOIN의 경우 오른쪽 테이블의 조인 컬럼에 인덱스를 추가한다. 복합 인덱스는 좌측 접두사 원칙을 따라야 한다.

최적화 예시:

-- orders에 복합 인덱스 추가
ALTER TABLE orders ADD INDEX idx_create_user (create_time, user_id);

-- 관련 테이블에 인덱스 추가
ALTER TABLE users ADD INDEX idx_vip (vip_level);
ALTER TABLE categories ADD INDEX idx_status (status);

장점: 간단하고 직접적이며 비침습적이다.

단점: 과도한 인덱스는 쓰기 성능을 저하시킬 수 있다.

적합한 경우: 선택도가 높은 컬럼(중복 값이 적은 컬럼).

기법 2: JOIN 순서 조정

가장 작은 결과 집합이 조인을 드라이빙하게 한다. 필요한 경우 STRAIGHT_JOIN을 사용해 순서를 강제한다.

예시:

-- 작은 테이블(블랙리스트)이 조인을 드라이빙하도록 강제
SELECT STRAIGHT_JOIN o.*
FROM blacklist b
JOIN orders o ON b.user_id = o.user_id;

기법 3: JOIN을 분리하고 애플리케이션 레이어에서 조합

데이터 볼륨이 적당한 목록 조회 쿼리의 경우, 별도의 쿼리로 데이터를 가져와 애플리케이션 코드(예: Java)에서 결과를 병합한다.

이 접근법은 데이터베이스 부하를 줄이고 대용량 카테시안 곱을 방지한다. IN 절을 사용한 배치 쿼리(한계를 피하기 위해 500개씩 분할)를 사용하고 효율적인 메모리 내 조합을 위해 맵을 활용한다.

장점: 데이터베이스 부하 감소 및 독립적인 쿼리 최적화 가능.

적합한 경우: 메인 테이블 크기가 적당한 경우(<100,000 행)이고 관련 테이블이 여러 개인 경우.

기법 4: 임시 테이블 또는 파생 테이블

자주 재사용되는 중간 결과를 구체화(materialize)한다.

예시:

CREATE TEMPORARY TABLE tmp_user_stat (
    user_id BIGINT PRIMARY KEY,
    total DECIMAL(10,2),
    INDEX(user_id)
) ENGINE=InnoDB;

INSERT INTO tmp_user_stat (user_id, total)
SELECT user_id, SUM(amount) FROM orders GROUP BY user_id;

기법 5: 구체화된 뷰 또는 요약 테이블

고정된 BI 스타일 보고서의 경우, 예약된 ETL 작업을 통해 결과를 사전 계산해 전용 요약 테이블에 저장한다. 쿼리가 요약 테이블을 조회해 밀리초 단위 응답을 얻는다.

기법 6: OLAP 엔진으로 전환

복잡한 분석 쿼리를 ClickHouse나 Apache Doris 같은 컬럼형 데이터베이스로 마이그레이션한다. 이러한 데이터베이스는 와이드 테이블과 스타 스키마 조인에서 탁월한 성능을 발휘한다.

기법 7: 수직 파티셔닝 + 읽기/쓰기 분리

와이드 테이블을 기본(고빈도) 테이블과 확장(대용량 텍스트, 저빈도) 테이블로 분리한다. 복잡한 읽기 쿼리를 레플리카 데이터베이스로 라우팅한다.

4. 최적화 방법 비교

최적화는 단계적으로 접근해야 한다:

  • 1단계 (즉시 효과): 인덱스 추가, JOIN 순서 조정 — 구현이 쉽고 변경 영향 최소화
  • 2단계 (중간 노력): 애플리케이션 레이어 분리, 임시 테이블 — 데이터베이스 부하 감소
  • 3단계 (아키텍처 변경): 요약 테이블, OLAP 엔진, 수직 파티셔닝 — 장기적 확장성 확보

5. 인터뷰에서 권장하는 답변 방식

"인터뷰에서는 다음과 같이 답변할 수 있다: 먼저 EXPLAIN을 실행해 전체 테이블 스캔, 임시 테이블, 파일 정렬을 파악한다. 그런 다음 적절한 인덱스를 추가하거나 STRAIGHT_JOIN을 사용해 조인 순서를 조정한다. SQL 레벨 튜닝이 여전히 부족하다면 애플리케이션 레이어 변경(메모리에서 조합하는 여러 개의 타깃 쿼리)이나 임시 테이블을 사용한 중간 결과 구체화를 고려할 것이다. 최적화는 단일 기법으로 달성되는 것이 아니라 데이터 볼륨과 비즈니스 요구사항에 맞는 인덱싱, 쿼리 재작성, 아키텍처 조정의 조합으로 이루어진다. 또한 performance_schema나 프로파일링 도구를 활용해 병목이 I/O인지 CPU인지 확인할 것이다."

결론

복잡한 다중 테이블 조인 최적화에는 만능 해결책이 없다. 성공은 철저한 진단, 데이터 특성, 단기 성과와 장기 유지보수성의 균형에 달려 있다.

이러한 계층적 기법을 마스터함으로써, 개발자와 아키텍트는 가장 까다로운 10개 테이블 쿼리도 효율적이고 확장 가능한 운영으로 탈바꿈시킬 수 있다.
