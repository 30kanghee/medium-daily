A Step-by-Step Guide for Developing Your Personal Agentic System.

Erdogan T · Data Science Collective · 36 min read · 원문 · freedium

LLM을 CPU처럼 활용하는 에이전트 시스템을 단계별로 직접 구축하는 실전 가이드.

요약

에이전틱 시스템은 복잡한 작업을 자동화할 때 핵심 패러다임으로 자리 잡고 있으며, LLM을 단순 텍스트 생성기가 아닌 운영체제의 CPU처럼 다루는 관점(LLMos)이 중요하다. 단일 프롬프트 접근법은 컨텍스트 포화, 작업 혼재, 프롬프트 불안정, 구조 부재라는 네 가지 근본적 한계를 지닌다. 이를 극복하기 위해 RAG(검색 증강 생성), 청킹, 임베딩, 글로벌 추론 등을 결합한 파이프라인과 역할별 전문 에이전트를 구성해야 한다. 저자는 오픈소스 파이썬 라이브러리 LLMlight를 활용해 로컬에서 프라이빗하게 동작하는 멀티 에이전트 시스템(토론 에이전트, 모더레이터, 스코어링 에이전트)을 직접 구현하는 실습 예제 5개를 단계별로 제공한다. 핵심 교훈은 "빠르게 가지 말고, 구조적으로 가라"이다.

핸즈온 튜토리얼

에이전틱 시스템의 활용은 복잡한 작업을 자동화할 때 가장 적합한 방식으로 자리 잡고 있습니다. 데이터 과학자로서, 자연어가 컴퓨팅의 기본 인터페이스가 되는 자동화의 새로운 패러다임을 따라가야 합니다. 이 새로운 개념을 이해하고 나면, 진짜 도전은 확률적 모델 주변에 신뢰할 수 있고 구조화되고 재현 가능한 시스템을 만드는 엔지니어링 문제라는 것을 깨닫게 됩니다. 목표는 반복적인 인지적 작업 부하를 줄이고 인간의 업무를 보다 효율적이고 일관되게 지원할 수 있는 에이전틱 시스템을 설계하는 방법을 배우는 것입니다.

이 글을 통해 언어 모델이 도구, 메모리, 검색, 다수의 전문 에이전트를 조율하는 중앙 추론 엔진 역할을 하는 LLM 운영체제(LLMos)라는 새로운 개념을 배우게 됩니다. 이후 신뢰할 수 있는 에이전틱 파이프라인을 구축하는 실질적인 설계로 들어갑니다. 글의 말미에는 구조화된 프롬프트, 검색 증강 생성, 임베딩 기반 검색, 협력하여 복잡한 문제를 해결하는 전문 에이전트를 사용해 Python 라이브러리 LLMlight로 경량 멀티 에이전트 시스템을 직접 구축하게 됩니다.

에이전틱 시스템 소개

에이전틱 시스템에 대한 간략한 소개를 쓰기 시작했는데, 설명해야 할 개념이 너무 많아 상당히 긴 소개가 되었습니다. 이 소개를 건너뛰지 않기를 강력히 권합니다. 이 새로운 개념들은 에이전틱 시스템 설계의 근본이기 때문입니다.

데이터 과학자로서, 단순히 소비하는 것이 아니라 자신만의 시스템을 직접 구축하는 데 큰 즐거움이 있습니다. 그럼에도 불구하고 "not invented here 증후군"을 경계해야 합니다. 이는 처음부터 끝까지 모든 것이 어떻게 작동하는지 완전히 이해할 수 있다는 이유만으로 기술이나 소프트웨어를 재구축하려는 엔지니어의 경향을 말합니다. 때로는 불필요한 재발명으로 이어질 수 있지만, AI 에이전트와 같은 신흥 분야에서 새로운 기술을 배울 때는 중요한 단계이기도 합니다. 시스템을 직접 실험하고 구축해봄으로써 근본적인 작동 원리를 훨씬 깊이 이해할 수 있고, 에이전틱 워크플로우의 장점과 한계를 파악하기 쉬워집니다.

에이전틱 시스템을 직접 구축해보는 학습 과정을 통해 시스템이 근본적으로 어떻게 작동하는지 더 깊이 이해하게 되고, 그 역량과 한계를 파악하기 쉬워집니다.

ChatGPT, Gemini, Grok의 채팅 프롬프트를 넘어서 언어 모델 주변에 시스템을 구축하기 시작하면, 진짜 도전은 텍스트를 생성하는 것이 아님을 금방 깨닫게 됩니다. 진짜 도전은 구조화되고 재현 가능한 방식으로 실질적인 문제를 안정적으로 해결할 수 있는 워크플로우를 설계하는 것입니다. 이는 문서 요약, 보고서 생성, 정보 추출, 텍스트 분석, 대용량 데이터 추론 등의 작업에서 조직과 연구자 모두에게 엄청난 기회를 만들어냅니다.

한때 세심하게 규칙 기반으로 설계된 시스템이 필요했던 작업들을 이제 자연어 지침으로 처리할 수 있는 경우가 많습니다. 하지만 현실은 더 복잡합니다. 핵심 과제는 비슷하게 남아 있습니다. 신뢰할 수 있고 구조화되고 재현 가능한 결과를 제공하는 에이전틱 워크플로우를 설계해야 합니다.

전통적인 결정론적 시스템은 대개 매우 안정적입니다. 일단 배포되면, 기반 규칙이 변경되지 않는 한 예측 가능하게 동작합니다. 언어 모델은 다릅니다. 프롬프트, 모델 버전, 온도 같은 파라미터 설정, 컨텍스트 창, 심지어 소소한 표현 변화에 따라 출력이 달라질 수 있습니다. 이 유연성이 강력함의 원천이지만, 프로덕션 환경에서 제어하기 어렵게 만드는 요인이기도 합니다.

테이크어웨이 1: LLM은 작가가 아니라 CPU다

우리는 이제 대형 언어 모델 자체가 시스템의 컴퓨팅 핵심이 되는 새로운 컴퓨팅 패러다임에 진입하고 있습니다. 전통적인 운영체제의 하드웨어 추상화와 많은 유사점이 있습니다. 현대 AI 시스템은 LLM 운영체제(LLMos)라고 표현할 수 있는 것을 중심으로 돌아갑니다: 언어 모델이 여러 작업, 도구, 에이전트를 조율하는 중앙 추론 엔진 역할을 하는 아키텍처 설계입니다. LLMos 용어는 Andrej Karpathy가 만들었습니다.

전통적 컴퓨팅과의 유추는 놀라울 정도로 직관적입니다:

  • CPU → 대형 언어 모델(LLM)
  • 바이트 → 토큰
  • RAM → 컨텍스트 창

바이트에 대한 결정론적 명령을 실행하는 대신, 이제 이 시스템들은 토큰에 대해 확률적으로 작동합니다. 컨텍스트 창은 임시 작업 메모리로 기능하고, 검색 시스템은 외부 메모리 저장소 역할을 하며, 프롬프트는 점점 더 컴포넌트 간의 소프트웨어 인터페이스를 닮아갑니다.

우리는 대형 언어 모델 자체가 시스템의 컴퓨팅 핵심이 되는 새로운 컴퓨팅 패러다임에 진입하고 있습니다.

여러 면에서, 우리는 이제 완전히 새로운 소프트웨어 아키텍처를 설계할 때처럼 언어 모델을 사용하고 있으며, 이는 도전적입니다. 복잡한 작업에 언어 모델을 사용하려 할 때, 여러 형태의 추론을 동시에 요구하기 때문입니다. 단일 프롬프트에서 원하는 것은:

  1. 정보 검색,
  2. 문서 추론,
  3. 구조 검증,
  4. 피드백 제공,
  5. 일관성 유지,
  6. 최종적으로 일관된 응답 생성.

하나의 큰 프롬프트로 모든 것을 해결하려는 시도는 혼란스럽고 신뢰할 수 없게 됩니다. 바로 이 지점에서 AI 에이전트가 등장합니다. 하나의 거대한 모델에게 전체 문제를 한 번에 해결하도록 요청하는 대신, 에이전틱 시스템은 작업을 더 작고 전문화된 하위 작업으로 분해합니다. 여러 에이전트가 공동 목표를 향해 협력할 수 있으며, 각 에이전트는 특정 책임에 집중합니다. 한 에이전트는 정보를 검색하고, 다른 에이전트는 품질을 평가하며, 또 다른 에이전트는 참고 문헌을 구조화하고, 마지막 에이전트는 최종 출력을 요약합니다.

테이크어웨이 2: 수십억 개의 파라미터가 더 필요한 게 아니라 전문화된 "에이전트 팀"이 필요하다

에이전틱 시스템을 구축하는 아이디어는 간단해 보이지만, 완전히 새로운 엔지니어링 과제들을 도입합니다:

  • 에이전트들은 어떻게 소통하는가?
  • 어떤 작업이 결정론적이어야 하고 어떤 것이 확률적이어야 하는가?
  • 제한된 컨텍스트 창을 어떻게 관리하는가?
  • 관련 정보를 어떻게 검색하고 통계적으로 검증하는가?
  • 환각(hallucination)을 어떻게 방지하는가?
  • 일관된 채점과 평가를 어떻게 유지하는가?

이 질문들은 로컬의 소형 모델로 작업할 때 더욱 강조됩니다. 소형 모델은 프라이버시, 이식성, 비용 면에서 장점을 제공하지만, 더 작은 컨텍스트 창과 더 큰 클라우드 기반 시스템에 비해 낮은 추론 능력이라는 제약도 있습니다.

에이전틱 시스템을 구축할 때 핸즈온 실습에서 깨달을 가장 중요한 것 중 하나는, 성공이 하나의 영리한 프롬프트에서 오는 경우는 드물다는 것입니다. 실제로 좋은 시스템은 신중하게 설계된 파이프라인의 결과입니다: 청킹 전략, 검색 시스템, 임베딩 모델, 랭킹 메커니즘, 구조화된 프롬프트, 채점 워크플로우, 협력하는 전문 에이전트들.

실세계 문제: 인간 대 기계

많은 실제 문제들은 대형 언어 모델로 자동화하려고 할 때까지 간단해 보입니다. 가장 일반적인 자동화 작업은 문서 평가입니다. 언뜻 보면 워크플로우는 간단해 보입니다: 1. 문서 업로드, 2. 질문하기, 3. 기다리고 모델이 답변 생성하게 두기. 하지만 실제로는 이런 작업이 격리되거나 결정론적인 경우가 드뭅니다. 여러 질문을 해야 하고, 더 많은 맥락을 제공해야 하며, 모델이 환각하는 것을 막고 팩트 체크를 해야 합니다.

단일 거대 프롬프트는 함정입니다.

단일 문서조차도 여러 형태의 추론을 요구할 수 있습니다. 문서가 더 커지고 상호 연결될수록 복잡성은 급격히 증가합니다. 인간으로서 우리는 도메인 지식과 전문성을 사용해 문서를 탐색합니다. 전체적인 이해를 유지하면서 필요할 때 특정 세부 사항에 집중하는 방식으로 로컬 추론과 글로벌 추론 사이를 자연스럽게 전환합니다. 반면 언어 모델은 제한된 컨텍스트 창을 통해 텍스트를 처리하고 토큰별로 확률적으로 출력을 생성합니다.

단일 프롬프트가 실패하고 에이전틱 시스템이 작동하는 이유

대형 언어 모델 채팅 인터페이스에 접근할 수 있을 때, 전체 문서를 LLM에 넣고 한 번에 내용을 평가하도록 요청하는 가장 단순한 방법으로 문제를 해결하려는 유혹을 받습니다. 이 단일적 접근법은 크고 복잡한 작업을 해결할 때 네 가지 핵심 한계가 있습니다:

  • 컨텍스트 창 포화: 모델이 200K 토큰과 같은 거대한 컨텍스트 창을 지원하더라도, 밀도 높은 시스템 지침, 기준, 엄격한 출력 형식 스키마와 결합된 대용량 문서는 모델을 실질적 한계까지 쉽게 밀어붙일 수 있습니다. 결과적으로 문서의 중요 세그먼트가 암묵적으로 압축되거나 무시되어 불완전한 추론으로 이어집니다.
  • 작업 얽힘: 단일 프롬프트는 모델이 근본적으로 다른 인지 작업을 동시에 수행하도록 강제합니다: 관련 데이터 추출, 맥락 해석, 세밀한 규칙 적용, 섹션 간 일관성 유지, 구조화된 피드백 생성. 이러한 관심사를 분리하는 명시적 메커니즘 없이는 모델이 종종 피상적이거나 자기 모순적인 출력을 만들어냅니다.
  • 프롬프트 불안정성: 단일 패스 아키텍처는 사소한 구조적 변화에 매우 민감합니다. 표현의 작은 변화가 동일한 문서를 평가할 때도 출력 점수와 피드백 품질을 크게 바꿀 수 있습니다.
  • 구조 부재: 단일 프롬프트는 단일 출력을 만들어냅니다. 체크포인트도 없고 추론 단계도 없습니다. 이로 인해 시스템을 디버그하고 검증하고 감사하기가 극도로 어려워집니다. 최종 답변이 틀렸을 때 추론 체인이 어디서 끊어졌는지 추적할 명확한 방법이 없습니다.

복잡한 작업은 반복과 검증을 요구합니다. 원샷 프롬프팅은 판단을 신뢰할 수 있게 만드는 구조를 제거하여, 고영향 환경에서 LLM 출력을 취약하고 일관성 없게 만듭니다.

해결책? 전문화된 작업을 위해 작은 로컬 언어 모델을 사용해 더 간단한 부분들로 큰 모듈식 시스템을 만드는 것입니다.

에이전트 아키텍처란 무엇인가?

실제로 대부분의 에이전틱 시스템은 여러 전문화된 LLM 모델이 특정 작업을 갖고 공동 목표를 향해 협력하는 구조화된 파이프라인입니다. 간단히 말해, 하나의 대형 언어 모델을 사용할 때는 모델이라고 부르지만, 공동 목표를 위해 여러 LLM 모델을 사용할 때는 에이전틱 시스템이라고 부릅니다. LLM 에이전트에 대한 오해는 그것들이 본질적으로 지능적인 존재라는 것입니다. 실제로는 특정 지침과 사전 정의된 작업을 가진 단일 (소형 및/또는 로컬) LLM 모델에 불과합니다.

작은 로컬 모델은 팀을 이룰 때 이깁니다.

  • A1: 검색 에이전트: 문서의 관련 섹션을 식별합니다.
  • A2: 추론 에이전트: 검색된 컨텍스트를 분석합니다.
  • A3: 요약 에이전트: 검색된 컨텍스트를 요약합니다.
  • A4: 평가 에이전트: 완전성이나 품질을 검증합니다.
  • A5: 채점 에이전트: 신뢰도나 순위를 할당합니다.
  • A6: 오케스트레이션 에이전트: 최종 응답을 집계하고 작업이 완료됐는지 또는 또 다른 사고 과정이 필요한지 결정합니다.

전문 에이전트를 위한 다양한 작업의 이러한 분해는 여러 장점을 도입합니다. 첫째, 모델의 인지 과부하를 줄입니다. 둘째, 중간 체크포인트를 생성합니다. 셋째, 확장성을 향상시킵니다. 넷째, 하이브리드 추론 전략을 가능하게 합니다. 일부 에이전트는 검색이나 규칙 기반 로직을 사용해 결정론적으로 작동하고, 다른 에이전트는 LLM을 사용해 확률적 추론을 수행할 수 있습니다.

검색 증강 생성(RAG) 파이프라인

대형 언어 모델의 핵심 한계 중 하나는 특정 질문과 관련된 대용량 문서의 어떤 부분이 관련 있는지 본질적으로 "알지 못한다"는 것입니다. 전체 책, 보고서, 데이터베이스를 직접 모델에 넣는 것은 제한된 컨텍스트 창으로 인해 종종 비효율적이고 비용이 많이 들며 신뢰할 수 없습니다.

검색 증강 생성(RAG)은 생성 전에 외부 검색 레이어를 도입하여 이 문제를 해결합니다. 전체 문서를 모델에 보내는 대신, 시스템은 먼저 가장 관련 있는 텍스트 조각을 검색하고 그 조각들만 LLM에 맥락적 입력으로 제공합니다.

RAG를 직관적으로 이해하는 방법은 인간 기억의 유추를 통한 것입니다. 뇌가 수천 개의 작고 검색 가능한 메모리 조각으로 분할되었다고 상상해보세요. "내 열쇠는 어디에 있지?"라고 자문할 때, 당신은 전체 인생 이야기를 재생하지 않습니다. 대신, 관련 있어 보이는 기억들만 무의식적으로 검색합니다. RAG도 유사하게 작동합니다. 대용량 문서는 청크라고 불리는 더 작은 조각으로 나뉩니다. 질문이 있을 때, 시스템은 그 청크들을 검색하고 가장 관련 있는 것들만 검색합니다.

RAG 시스템의 과제

RAG는 또한 핵심 의존성을 도입합니다: 검색 품질. 중요한 청크가 검색되지 않으면, 모델은 빠진 공백을 올바른 것이 아닌 그럴듯한 답변으로 채울 수 있습니다. 다시 말해, 생성 품질은 근본적으로 검색 파이프라인에 의해 제한됩니다.

RAG 파이프라인에 필요한 네 가지 단계

일반적인 RAG 파이프라인은 여러 단계로 구성되며, 가장 단순한 형태는 "Simple RAG" 또는 "Naive RAG"라고 합니다. 이 기본 접근법은 네 가지 주요 단계에서 최적화될 수 있습니다: 청킹, 검색, 임베딩, 채점.

  • 청킹 전략: 소스 문서를 더 작고 일관된 세그먼트로 분할하는 방법을 정의합니다. 청킹은 단어, 문장, 단락, 토큰, 또는 의미론적 섹션으로 할 수 있습니다.
  • 임베딩 전략: 각 청크는 임베딩이라는 수치 표현으로 변환됩니다. 임베딩 모델은 텍스트를 고차원 벡터로 변환하여 의미론적으로 유사한 텍스트 조각이 수학적으로 가까워지게 합니다. 옵션에는 TF-IDF, Word2Vec, Sentence-BERT, 또는 최신 OpenAI 임베딩이 있습니다.
  • 검색/검색 전략: 사용자가 쿼리를 제출하면, 쿼리 자체도 벡터 공간에 임베딩됩니다. 시스템은 쿼리 임베딩과 모든 문서 청크 사이의 유사도 점수를 계산합니다.
  • 채점 전략: 가장 높은 유사도 점수를 가진 청크가 질문과 가장 관련 있다고 가정합니다. 코사인 유사도가 가장 일반적입니다.

청크의 랭킹 및 집계

검색 후, 가장 관련 있는 상위 k개 청크가 선택되고 새로운 컨텍스트 창에 집계됩니다. 검색된 청크는 겹치거나 중복된 정보를 포함하거나 중요한 지원 맥락을 놓칠 수 있습니다. 재순위 모델, 메타데이터 필터링, 하이브리드 검색, 또는 방법의 발견적 집계와 같은 추가 랭킹 전략이 종종 사용됩니다.

최종 단계: 생성

마지막 단계는 선택된 컨텍스트 + 지침 + 프롬프트를 기반으로 답변을 생성하는 것입니다. 이 단계에서 LLM은 더 이상 전체 문서를 추론할 필요가 없습니다. 대신, 파이프라인에서 앞서 검색된 가장 관련 있는 정보에만 집중합니다.

이 파이프라인이 매우 잘 작동하는 작업들

이 접근법은 다음과 같은 조회 질문에 매우 잘 작동합니다:

  • "문서에 모델 비교가 포함되어 있나요?"
  • "방법론 섹션이 있나요?"
  • "어떤 데이터셋이 사용됐나요?"

하지만 일부 질문은 특정 문장이나 단락을 찾는 것이 아니라 문서 전체를 이해해야 합니다:

  • "조언이 잘 생각되어 있나요?"
  • "논증이 일관성이 있나요?"
  • "방법론이 논리적으로 따라갈 수 있나요?"

이 접근법은 로컬 관련성을 최적화하지 글로벌 관련성을 최적화하지 않는다는 것이 주요 한계입니다.

글로벌 추론 워크플로우

검색 시스템은 쿼리와 가장 유사한 청크를 검색하고 그 조각들만 언어 모델에 전달합니다. "확대"와 "축소"의 한계를 해결하기 위해, 많은 실용적인 시스템들은 최종 생성 전에 추가 추론 레이어를 도입합니다. 이것이 다단계 글로벌 추론 전략입니다.

2단계 글로벌 추론 전략

글로벌 추론 워크플로우는 계층적 추론 시스템과 밀접하게 관련됩니다: 가장 낮은 수준에서 청크는 로컬 정보를 포함합니다. 중간 수준에서 요약은 섹션 수준 의미를 포착합니다. 더 높은 수준에서 집계된 요약은 문서를 전역적으로 나타냅니다.

단순화된 워크플로우:

  1. 문서를 청크로 분할합니다.
  2. 각 청크를 독립적으로 분석합니다.
  3. 로컬 요약이나 관찰을 생성합니다.
  4. 이 중간 요약들을 집계합니다.
  5. 문서의 글로벌 표현을 구성합니다.
  6. 집계된 컨텍스트를 사용해 원래 질문에 답변합니다.

이것은 워크플로우를 다음에서: 검색 → 생성을 다음으로 변환합니다: 분석 → 요약 → 집계 → 추론 → 생성

쿼리 재작성

글로벌 추론의 또 다른 중요한 구성 요소는 쿼리 재작성입니다. 예를 들어:

  • 원래 쿼리: "제안이 잘 생각되어 있나요?"
  • 재작성된 추론 목표: "문서의 일관성, 구조, 실현 가능성, 일관성을 분석하세요."

로컬 추론과 글로벌 추론을 함께

실제로는 로컬 전략과 글로벌 전략을 결합하는 것이 가장 좋습니다:

  • 결정론적 질문을 위한 로컬 검색 파이프라인,
  • 추상적이거나 전체적 평가를 위한 글로벌 추론 워크플로우.

이 하이브리드 접근법은 현대 에이전틱 시스템에서 부상하는 핵심 아키텍처 패턴 중 하나입니다.

작은 언어 모델이 이기고 있습니다

LLM 분야에서 올해 가장 중요한 변화 중 하나는 단일 모놀리식 밀집 언어 모델에서 여러 전문 에이전트로 구성된 에이전틱 AI 시스템으로의 이동입니다. NVIDIA 연구는 "소형 언어 모델이 에이전틱 AI의 미래다"라고 주장했으며, Microsoft는 메모리 요구 사항을 대폭 줄이고 로컬 배포를 점점 더 실현 가능하게 만드는 1비트 LLM을 도입했습니다. 예를 들어 Gemma 4-26B-a4B 모델은 약 260억 개의 총 파라미터를 가지지만, 추론 시 약 40억 개의 파라미터만 활성화되는 MoE(전문가 혼합) 아키텍처입니다.

LM Studio로 로컬 모델 설정하기

LLM 및 에이전틱 시스템을 실험하기 전에, 자신만의 로컬 모델을 설정해야 합니다. 가장 많이 사용되는 소프트웨어 중 하나는 LMstudio입니다. 무료이고, 사용자 인터페이스가 있으며, 기계의 하드웨어와 잘 연결됩니다. 이 밖에 좋은 대안으로는 Ollama나 Claude가 있습니다.

LLMlight 라이브러리

LLMlight 라이브러리는 최소 의존성으로 언어 모델을 로컬에서 실행하기 위한 사용하기 쉽고 경량화된 Python 패키지입니다. 입력 프롬프트 처리부터 모델 출력까지 다양한 단계가 있는 에이전틱 파이프라인입니다. 일반적으로 파이프라인에는 여섯 가지 핵심 단계가 있습니다: 청킹, 검색 전략, 임베딩 전략, 채점 전략, 컨텍스트 전략, 마지막으로 프롬프팅 전략.

청킹 전략

LLMlight 라이브러리는 대용량 문서를 더 작고 일관된 세그먼트로 분할하는 모듈식이고 구성 가능한 청킹 전략을 활용합니다. 청킹 프로세스는 chunks 파라미터를 통해 높은 수준으로 사용자 정의할 수 있습니다.

  • 방법: 청크는 문자 또는 단어 기준으로 생성할 수 있습니다.
  • 기본 구성: 라이브러리의 기본 설정은 {'method': 'chars', 'size': 1000, 'overlap': 200}입니다.
  • 크기: 이 기본값은 1000문자마다 세그먼트를 생성합니다.
  • 겹침: 경계에서 지식 단편화를 방지하기 위해 인접한 청크 사이에 200문자 겹침이 유지됩니다.

검색 전략 — 로컬 데이터베이스

검색 전략은 (벡터) 데이터베이스에서 후보 청크를 검색하는 방법을 지정합니다. LLMlight는 SQLite + HNSW를 통합합니다. 기본 백엔드는 고속 근사 최근접 이웃 검색을 위한 옵션 HNSW 인덱스가 있는 로컬 SQLite 데이터베이스를 사용합니다.

임베딩 및 채점 전략

LLMlight에서 사용 가능한 임베딩 방법들:

  • TF-IDF: 쿼리 용어가 일치하는 구조화된 문서에 최적
  • 단어 빈도(Bag of Words): 간단한 단어 빈도 접근법
  • BERT: 자유 형식 텍스트를 위한 고급 문맥 임베딩
  • BGE-small: 일반 용도를 위한 효율적인 임베딩 모델

채점 전략은 후보 청크에 관련성 점수를 할당합니다. 코사인 유사도가 기본입니다.

컨텍스트 전략

LLMlight의 컨텍스트 전략에는 모든 입력 청크를 기반으로 단일 포괄적 텍스트를 생성하기 위해 협력하는 두 에이전트가 있습니다. chunk-analyzer-agent는 각 청크와 다른 청크와의 관계를 분석하는 특정 지침을 가집니다. 그런 다음 smart-context-agent가 처리된 청크를 사용해 특정 지침으로 하나의 일관된 텍스트로 요약합니다.

  • None: 전처리 없음. 원시 전체 컨텍스트 텍스트 사용
  • chunk-wise: 컨텍스트를 관리 가능한 청크로 분할
  • global-reasoning: 컨텍스트의 글로벌 요약 생성

프롬프트 최적화

최종 프롬프트는 사용자가 제공하거나 에이전트가 생성한 다양한 부분의 조합입니다:

  • 시스템 메시지: AI의 역할과 동작을 정의
  • 컨텍스트: 처리되고 검색된 관련 정보
  • 사용자 쿼리: 특정 질문 또는 요청
  • 지침: 응답 생성을 위한 추가 안내

실습 1: 단일 모델 로드 및 간단한 채팅

첫 번째 실습에서는 LM studio의 엔드포인트를 사용해 사용 가능한 모델에 연결하고, 모델에 기본 질문을 합니다. 제 경우, LM의 기본 엔드포인트는 http://localhost:1234입니다.

# 라이브러리 설치
pip install llmlight
from LLMlight import LLMlight

# LLMlight 클라이언트 초기화
client = LLMlight(model='google/gemma-4-26b-a4b-qat')

# 질문하기
response = client.prompt('What is the capital of France?')
print(response)

실습 2: 로컬 지식 베이스 생성

로컬 지식 베이스는 로컬 검증된 사실을 사용해 추론하기 위한 데이터베이스 역할을 하므로 프라이빗 에이전틱 시스템 구축의 중요한 첫 번째 단계입니다. LLMlight 라이브러리로 memory_add 함수를 사용해 PDF, 텍스트 파일, 전체 디렉토리를 포함한 원시 데이터를 쉽게 수집할 수 있습니다. MemVid는 수백만 개의 텍스트 임베딩을 단일 MP4 비디오 파일로 압축하는 특히 혁신적인 백엔드입니다.

# 라이브러리 임포트
from LLMlight import LLMlight

# 모델과 메모리 초기화
client = LLMlight(model='google/gemma-4-26b-a4b-qat')
# 데이터베이스 생성 (또는 로드)
client.memory_init(store_path='knowledge_base.db')

# PDF 파일을 데이터베이스에 추가 (자동으로 텍스트 추출 및 청킹)
url = 'https://proceedings.neurips.cc/paper_files/paper/2017/file/3f5ee243547dee91fbd053c1c4a845aa-Paper.pdf'
pdf_text = client.read_pdf(url)

# DB에 작성
client.memory_add(text=pdf_text)

# 청크 표시
client.memory_chunks(1)

# 디스크에 저장
client.memory_save()

# 새 지식으로 쿼리
response = client.prompt(query='What are attention networks?', response_format='Summarize in 3 sentences.')
print(response)

실습 3: 컨텍스트 전략 방법별 출력 차이

LLMlight 라이브러리에는 검색된 정보가 언어 모델에 제시되는 방식을 최적화하는 세 가지 컨텍스트 전략이 있습니다: None, chunk-wise, global-reasoning.

None(또는 "처리 없음") 전략은 원시 및 전체 검색된 컨텍스트를 그대로 사용하는 가장 단순한 접근법입니다:

from LLMlight import LLMlight

# 컨텍스트 전략 없이 초기화
client = LLMlight(model='google/gemma-4-26b-a4b-qat', retrieval_method='naive_rag', context_strategy=None, top_chunks=6)

client.memory_init(store_path='knowledge_base.db')

response = client.prompt(query='What are attention networks?', response_format='Summarize in 2 sentence')
print(response)
# Based on the provided text, attention networks utilize mechanisms like self-attention
# to perform tasks such as reading comprehension, summarization, and machine translation
# by capturing the syntactic and semantic structures of sentences.

chunk-wise 전략은 컨텍스트를 관리 가능한 세그먼트로 분할하고 각 상위 청크에 특정 변환을 적용합니다:

from LLMlight import LLMlight

# CHUNK-WISE 컨텍스트 전략으로 초기화
client = LLMlight(model='google/gemma-4-26b-a4b-qat', retrieval_method='naive_rag', context_strategy='chunk-wise', top_chunks=6)

client.memory_init(store_path='knowledge_base.db')

response = client.prompt(query='What are attention networks?', response_format='Summarize in 2 sentence')
print(response)
# The provided context does not contain a formal definition of "attention networks."
# It only describes the mathematical mechanism of attention, which uses queries, keys, and values to
# compute output weights through methods like scaled dot-product or additive attention.

global-reasoning 전략은 청크를 분석, 요약, 집계하여 문서의 글로벌 표현을 생성하는 정교한 계층적 접근법을 사용합니다:

from LLMlight import LLMlight

# GLOBAL-REASONING 컨텍스트 전략으로 초기화
client = LLMlight(model='google/gemma-4-26b-a4b-qat', retrieval_method='naive_rag', context_strategy='global-reasoning', top_chunks=6)

client.memory_init(store_path='knowledge_base.db')

response = client.prompt(query='What are attention networks?', response_format='Summarize in 2 sentence')
print(response)
# Based on the provided text, attention networks are computational mechanisms that use queries, keys,
# and values to determine weights via a scaled dot-product and a softmax function.

실습 4: 관심 주제에 대한 두 에이전트 간 토론 만들기

가장 흥미로운 응용 중 하나는 여러 AI 에이전트 사이의 자율적 토론을 만드는 것입니다. 이 예에서는 각자 고유한 지식 베이스, 검색 설정, 성격을 가진 두 가지 다른 에이전트를 설정하는 방법을 배웁니다. 토론 중 각 에이전트는 RAG를 사용해 응답 생성 전에 메모리에서 관련 정보를 검색합니다.

# 초기화
from LLMlight import LLMlight

# ====================================================
# 에이전트 A: 데이터 과학자
# ====================================================
agent_a = LLMlight(
    model="google/gemma-4-26b-a4b-qat",
    retrieval_method="naive_rag",
    embedding="bert",
    context_strategy=None,
    top_chunks=5,
    temperature=0.7,
)

agent_a.memory_init(store_path="agent_a.db")
agent_a.memory_add("""
Large Language Models are one of the most important step we did in the field of AI
It helps the workload and the work easier and faster.
""")
agent_a.memory_add("""
Large Language Models use transformer architectures and are trained on
massive text corpora using self-supervised learning.
""")

# ====================================================
# 에이전트 B: 농부
# ====================================================
agent_b = LLMlight(
    model="google/gemma-4-26b-a4b-qat",
    retrieval_method="naive_rag",
    embedding="bert",
    context_strategy=None,
    top_chunks=5,
    temperature=0.7,
)

agent_b.memory_init(store_path="agent_b.db")
agent_b.memory_add("""
The use of AI and machine learning consumes to much power and there is no need for this
new technology. The human work was good enough and there is no need to change that.
""")

# ====================================================
# 토론 루프
# ====================================================
topic = "Discuss the importance of the use of Large Language Models and AI."
message = topic

for turn in range(5):
    print(f"\n{'='*80}")
    print(f"ROUND {turn+1}")
    print(f"{'='*80}")

    response_a = agent_a.prompt(
        system='You are a Data Scientist.',
        query=f"""Topic: {message}""",
        response_format='Give your opinion in 1-2 paragraphs and ask a question to the other agent.',
    )
    print("\nAgent A:")
    print(response_a)

    response_b = agent_b.prompt(
        system='You are a farmer.',
        query=f"""The Data Scientist said: {response_a}""",
        response_format='Respond to the discussion in 1-2 paragraphs and ask a follow-up question.',
    )
    print("\nAgent B:")
    print(response_b)

    message = response_b

세 번째 에이전트 도입: 모더레이터

두 에이전트는 풍부하고 통찰력 있는 토론을 할 수 있지만, 반복적인 논쟁에 갇히거나 원래 목표에서 벗어날 수 있습니다. 모더레이터 또는 중재자 역할을 하는 세 번째 에이전트를 도입하면 대화의 품질을 크게 향상시킬 수 있습니다. 모더레이터는 새로운 도메인 지식을 기여하는 대신, 대화를 관찰하고 동의와 불일치 영역을 식별하며 주기적으로 양측이 제시한 핵심 포인트를 요약합니다.

from LLMlight import LLMlight

# 에이전트 A, B 설정 (이전과 동일)...

# ====================================================
# 에이전트 C: 모더레이터
# ====================================================
moderator = LLMlight(
    model="openai/gpt-oss-20b",
    retrieval_method="naive_rag",
    context_strategy=None,
    top_chunks=5,
    temperature=0.3,  # 객관적 요약을 위한 낮은 온도
)
moderator.memory_init(store_path="moderator.db")

# 공유 토론 메모리
shared_memory = LLMlight(model="openai/gpt-oss-20b")
shared_memory.memory_init(store_path="discussion.db")

# ====================================================
# 토론 루프
# ====================================================
for turn in range(5):
    # 에이전트 A 응답...
    # 에이전트 B 응답...

    # 모더레이터 요약
    moderator_summary = moderator.prompt(
        system='You are a neutral moderator.',
        query=f"""
        Data Scientist: {response_a}
        Farmer: {response_b}

        수행 과제:
        1. 핵심 논거 요약.
        2. 동의 사항 식별.
        3. 불일치 사항 식별.
        4. 합의를 향해 나아가도록 돕는 질문 제안.
        """,
        response_format='Keep the output concise.'
    )

    shared_memory.memory_add(response_a)
    shared_memory.memory_add(response_b)
    shared_memory.memory_add(moderator_summary)
    message = moderator_summary

채점 에이전트로 대화 제어하기

멀티 에이전트 토론을 더 제어 가능하고 목표 지향적으로 만들기 위해, 대화를 고정된 반복 횟수(예: 5라운드)로 제한하거나, 더 동적으로는 채점 에이전트를 도입할 수 있습니다. 이 네 번째 에이전트의 임무는 각 토론 라운드를 평가하고 에이전트들 사이에 얼마나 많은 동의가 존재하는지를 기반으로 수렴 점수를 할당하는 것입니다. 점수가 사전 정의된 임계값을 초과하면, 시스템은 일찍 종료하여 만족스러운 합의에 도달했음을 나타낼 수 있습니다.

from LLMlight import LLMlight

# ====================================================
# 채점 에이전트 (수렴/중지 결정)
# ====================================================
scoring_agent = LLMlight(
    model="liquid/lfm2-24b-a2b",
    retrieval_method="naive_rag",
    context_strategy=None,
    top_chunks=5,
    temperature=0.0,  # 결정론적 채점
)
scoring_agent.memory_init(store_path="scoring.db", overwrite=True)

MAX_ROUNDS = 5
AGREEMENT_THRESHOLD = 0.85  # 수렴이 충분히 높으면 중지

for turn in range(MAX_ROUNDS):
    # 에이전트 A, B 응답...
    # 모더레이터 요약...

    # 채점 에이전트가 수렴 점수 평가
    convergence_score = scoring_agent.prompt(
        system='You are an objective scoring agent.',
        query=f"""
        Moderator summary: {moderator_summary}
        Rate the convergence between 0.0 and 1.0.
        """,
        response_format='Return only a float between 0.0 and 1.0.'
    )

    try:
        score = float(convergence_score.strip())
        if score >= AGREEMENT_THRESHOLD:
            print(f"합의 달성! 점수: {score}")
            break
    except:
        pass

좋은 지침이 핵심입니다

실습을 통해 에이전트들이 모호한 또는 불명확한 지침을 받을 때 더 나쁜 성능을 보인다는 것을 알 수 있습니다. 좋은 시스템 지침을 작성하는 것은 숙련된 인간 관리자에게 업무 브리핑을 작성하는 것과 매우 유사합니다. 역할(너는 누구야), 목표(무엇을 달성해야 하나), 제약(허용되는 것과 허용되지 않는 것), 출력 형식(구조화된 JSON, 키워드 목록, 자유 텍스트)을 지정하는 것이 중요합니다.

언어 모델 생성 전 테이크어웨이

  1. 모든 솔루션이 언어 모델을 필요로 하지는 않습니다. 간단한 솔루션이 원하는 결과를 주지 않을 때만 복잡한 모델을 고려하세요. 조회 목록, 정규 표현식, TF-IDF로 시작하세요.
  2. 언어 모델은 특정 사용 사례에 최적화되어야 합니다. 컨텍스트 전략, 검색 방법, 임베딩, 온도, 모델을 신중하게 선택하세요.
  3. 항상 모든 입력 텍스트가 UTF-8 또는 Latin-1로 인코딩되었는지 확인하세요. 인코딩을 전환하지 마세요. 일부 모델은 특정 인코딩을 건너뛰어 전체 문서를 놓칠 수 있습니다.
  4. 너무 작게 청킹하지 마세요 (200자 미만). 청크에 완전한 정보가 없어 환각이 발생합니다.
  5. 헤더, 푸터, 보일러플레이트 등을 제거하세요. 이것들은 종종 시스템에 너무 많은 노이즈를 생성합니다.
  6. 항상 임베딩에 대해 비판적으로 생각하세요. QA 시스템을 만들 때, TF-IDF가 특정 도메인에서 BERT 같은 의미론적 임베딩보다 뛰어날 수 있습니다.
  7. 모호한 쿼리는 모호한 답변을 줄 것입니다. 예를 들어 "스포츠 팁을 알려줘"는 관심 없는 청크를 검색할 수 있습니다.
  8. 각 언어 모델에는 가드레일이 있어야 합니다. 사용 사례에 따라, 모델이 인용을 제공하거나 정보가 발견된 청크를 표시하도록 강제하세요.

마무리: 빠르게 가지 말고, 구조적으로 가라

이것이 "컨텍스트 엔지니어링"의 해입니다. 성공적인 LLM 통합은 배포 속도가 아니라 기반이 되는 인지 아키텍처에 관한 것입니다. 결정론적 바이트에서 확률적 토큰으로 전환함에 따라, 전통적인 운영체제에 적용하는 것과 같은 엄격함으로 시스템을 관리해야 합니다. 우리는 때때로 자체 논리를 환각하는 "운영체제"를 관리할 준비가 되어 있나요? 모듈식, 로컬, 통계적으로 검증된 인프라를 구축하는 데 답이 있다고 생각합니다. 작게 시작하고, 자신만의 멀티 에이전트 파이프라인을 구축하고, 모델을 작가로 취급하는 것을 멈추세요. 새로운 운영체제의 핵심으로 취급하세요.
