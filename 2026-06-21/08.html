10 Data Models Every Data Engineer Must Know (Before They Break Production)

Rohan Dutt · 12 min read · 원문 · freedium

프로덕션을 망가뜨리기 전에 데이터 엔지니어가 반드시 알아야 할 10가지 데이터 모델과 그 함정 및 해법.

요약

스타 스키마, 스노우플레이크 스키마, 데이터 볼트 등 전통적 모델부터 스트리밍 이벤트 소싱, 컬럼형 스토리지, 그래프 모델, 유니파이드 서빙 레이어까지 10가지 핵심 데이터 모델을 실제 장단점과 코드 예제로 설명한다. 각 모델은 특정 시나리오에 최적화되어 있으며, 잘못 선택하면 쿼리 타임아웃, 성능 저하, 복잡성 폭발로 이어진다. 저자는 각 모델의 핵심 문제점, 실제 재앙 사례, 그리고 현대적 대안을 제시하며, 궁극적으로 아이스버그(Iceberg)를 중심으로 한 유니파이드 서빙 레이어가 미래 아키텍처라고 주장한다.

프로덕션 시스템에서 발생하는 장애를 예방하는 필수 데이터 모델들을 배웁니다. 올바른 모델 선택은 확장성, 일관성, 신뢰할 수 있는 분석을 보장합니다.

10. 스타 스키마: 레거시 강자 (그러나 규모에서 실패한다)

왜 중요한가

스타 스키마는 직관적이고 분석가 친화적이지만, 대규모에서는 성능 병목이 됩니다. 특히 거대한 팩트 테이블, 고카디널리티 차원, 준실시간 워크로드에서 그렇습니다.

핵심 문제

  • 팩트 테이블이 커질수록(수십억 행) 쿼리 성능이 급격히 저하됩니다.
  • 복잡한 조인 + 원시 팩트에 대한 GROUP BY = 느린 스캔, 타임아웃.
-- 원시 팩트 테이블에 대한 클래식 스타 스키마 쿼리
SELECT
    d.region,
    d.store_type,
    SUM(f.sales_amount) AS total_sales
FROM fact_sales f
JOIN dim_store d
  ON f.store_id = d.store_id
JOIN dim_date dt
  ON f.date_id = dt.date_id
WHERE dt.date BETWEEN '2025-11-28' AND '2025-11-29'
GROUP BY d.region, d.store_type;

무엇이 잘못되는가

  • 수억 개의 팩트 행을 스캔
  • 모든 쿼리에 값비싼 조인
  • 고카디널리티 차원에서 GROUP BY

실제 재앙 사례: 한 대형 유통업체의 블랙프라이데이 대시보드가 타임아웃됐습니다. 시간당 1천만 건 이상의 트랜잭션이 유입되자 스타 스키마가 따라가지 못했던 것입니다.

해결책: 집계 인식 모델링

사전 집계 요약 테이블

CREATE TABLE sales_daily_summary AS
SELECT
    date_id,
    store_id,
    SUM(sales_amount) AS total_sales,
    COUNT(*) AS txn_count
FROM fact_sales
GROUP BY date_id, store_id;

최적화된 쿼리

SELECT
    d.region,
    d.store_type,
    SUM(s.total_sales) AS total_sales
FROM sales_daily_summary s
JOIN dim_store d
  ON s.store_id = d.store_id
JOIN dim_date dt
  ON s.date_id = dt.date_id
WHERE dt.date BETWEEN '2025-11-28' AND '2025-11-29'
GROUP BY d.region, d.store_type;

결과: 10~100배 빠른 쿼리, 피크 부하에서도 예측 가능한 성능.

장점

  • 단순한 멘탈 모델 (분석가가 이해하기 쉬움)
  • 중소 규모 데이터에 잘 동작
  • 대부분의 BI 도구와 호환

단점

  • 대용량 팩트 테이블에서 확장성 부족
  • 쿼리 시점에 비용이 큰 조인
  • 실시간 또는 고인입 시스템에 부적합

결론: 스타 스키마는 정적, 저용량 분석에는 괜찮습니다. 고처리량 OLAP에서는 사전 집계, 와이드 테이블, 또는 현대적 레이크하우스 모델이 필수입니다.

9. 스노우플레이크 스키마: 과도하게 설계되고 느리다

왜 중요한가: 스노우플레이크 스키마는 스토리지를 최적화하지 쿼리 성능을 최적화하지 않습니다. 현대 분석(클라우드 OLAP, 대시보드, 애드혹 쿼리)에서 병목은 디스크가 아닌 컴퓨팅입니다. 과도한 정규화는 조인 깊이를 폭발적으로 늘리고 지연 시간을 죽입니다.

핵심 문제

  • 차원 테이블이 여러 하위 차원으로 분할
  • 단순 지표에도 5~10개 조인 필요
  • 최적화기가 고전 → 느린 계획, 높은 CPU
SELECT
    r.region_name,
    c.category_name,
    SUM(f.sales_amount) AS total_sales
FROM fact_sales f
JOIN dim_product p        ON f.product_id = p.product_id
JOIN dim_category c       ON p.category_id = c.category_id
JOIN dim_subcategory sc   ON c.subcategory_id = sc.subcategory_id
JOIN dim_store s          ON f.store_id = s.store_id
JOIN dim_region r         ON s.region_id = r.region_id
GROUP BY r.region_name, c.category_name;

관찰된 영향: 벤치마크에서 정규화 해제된 차원 대비 300% 이상 높은 쿼리 지연이 일관되게 나타납니다.

현대적 대안: 차원 정규화 해제

CREATE TABLE dim_product_flat AS
SELECT
    p.product_id,
    p.product_name,
    c.category_name,
    sc.subcategory_name
FROM dim_product p
JOIN dim_category c     ON p.category_id = c.category_id
JOIN dim_subcategory sc ON c.subcategory_id = sc.subcategory_id;

장점

  • 차원 스토리지 크기 감소
  • 정규화를 통한 강력한 데이터 무결성
  • OLTP나 제한된 환경에서 유용

단점

  • 과도한 조인이 OLAP 성능을 망침
  • 쿼리 최적화 및 디버그가 어려움
  • 현대 BI와 셀프 서비스 분석에 부적합

결론: 스노우플레이크 스키마는 어제의 스토리지 문제를 해결하면서 오늘의 성능 악몽을 만들어냅니다. 분석 규모에서는 차원을 평탄화하고, 조인을 구체화하고, 컬럼형 엔진이 무거운 작업을 하도록 두세요.

8. 데이터 볼트: 엔터프라이즈 몬스터 (감사 가능성이 필요할 때)

왜 중요한가: 데이터 볼트는 감사 가능성, 계보, 완전한 이력화에 탁월합니다. 규제 산업(은행, 의료)에 필수적입니다. 하지만 다층 아키텍처는 저지연 분석에 근본적으로 맞지 않습니다.

핵심 구조 (왜 느린가)

  • 허브 → 비즈니스 키
  • 링크 → 관계
  • 새틀라이트 → 설명적 + 이력 속성. 모든 쿼리가 시간 변형 테이블에 걸쳐 조인 폭발이 됩니다.

원시 볼트의 고통

SELECT
    h_cust.customer_id,
    s_cust.customer_name,
    SUM(s_tx.amount) AS total_spend
FROM hub_customer h_cust
JOIN link_customer_tx l_ct
  ON h_cust.hub_customer_key = l_ct.hub_customer_key
JOIN hub_transaction h_tx
  ON l_ct.hub_transaction_key = h_tx.hub_transaction_key
JOIN sat_customer s_cust
  ON h_cust.hub_customer_key = s_cust.hub_customer_key
 AND s_cust.is_current = true
JOIN sat_transaction s_tx
  ON h_tx.hub_transaction_key = s_tx.hub_transaction_key
 AND s_tx.is_current = true
GROUP BY h_cust.customer_id, s_cust.customer_name;

탈출구: 데이터 볼트 → 차원 마트

CREATE TABLE customer_spend_mart AS
SELECT
    h_cust.customer_id,
    s_cust.customer_name,
    s_tx.amount,
    s_tx.tx_timestamp
FROM hub_customer h_cust
JOIN link_customer_tx l_ct ON h_cust.hub_customer_key = l_ct.hub_customer_key
JOIN hub_transaction h_tx ON l_ct.hub_transaction_key = h_tx.hub_transaction_key
JOIN sat_customer s_cust ON h_cust.hub_customer_key = s_cust.hub_customer_key AND s_cust.is_current = true
JOIN sat_transaction s_tx ON h_tx.hub_transaction_key = s_tx.hub_transaction_key AND s_tx.is_current = true;
SELECT customer_name, SUM(amount) AS total_spend
FROM customer_spend_mart
GROUP BY customer_name;

결과: 컴플라이언스를 희생하지 않고 서브초 분석 가능.

장점

  • 완전한 이력 추적 (데이터 손실 없음)
  • 탁월한 감사 가능성 및 계보
  • 수집 및 스키마 변경에 잘 확장

단점

  • 분석에서 극도로 느림
  • 높은 모델링 및 운영 복잡성
  • 어차피 BI를 위한 마트가 필요

결론: 데이터 볼트는 통합 및 컴플라이언스 레이어이지 분석 레이어가 아닙니다. 규제 기관이 이력을 요구할 때 사용하세요. 속도를 위해서는 스트리밍 CDC + 차원 또는 와이드 마트를 위에 레이어링하세요.

7. 와이드 컬럼 스토어 (Cassandra, Bigtable): 시계열 혼돈용

왜 중요한가: 와이드 컬럼 데이터베이스는 쓰기가 멈추지 않는 고속 수집(IoT, 지표, 로그)을 지배합니다. 하지만 쿼리 유연성을 희생합니다 — 조인 없음, 제한된 필터링, 엄격한 접근 패턴. 쓰기에서 이기고, 탐색에서 집니다.

핵심 현실

  • 추가 중심의 시계열 워크로드용으로 구축됨
  • 쿼리는 기본 키 설계와 일치해야 함
  • 애드혹 분석 = 전체 테이블 스캔 = 고통

흔한 실수: Cassandra를 관계형 DB처럼 사용 → 보조 인덱스, 랜덤 필터, 조인 → 즉각적 후회.

CQL (올바른 패턴)

CREATE TABLE device_metrics (
    device_id TEXT,
    metric_date DATE,
    event_ts TIMESTAMP,
    metric_type TEXT,
    metric_value DOUBLE,
    PRIMARY KEY (
        (device_id, metric_date),  -- 파티션 키
        event_ts                   -- 클러스터링 키
    )
) WITH CLUSTERING ORDER BY (event_ts DESC);

확장되는 쿼리

SELECT event_ts, metric_value
FROM device_metrics
WHERE device_id = 'dev_42'
  AND metric_date = '2025-12-26'
  AND event_ts >= '2025-12-26 10:00:00'
  AND event_ts <  '2025-12-26 11:00:00';

결과: 초당 수백만 건의 쓰기에서 예측 가능한 지연.

하지 말아야 할 것

-- 안티패턴: 파티션 키 없음
SELECT * FROM device_metrics WHERE metric_type = 'cpu';
  • 전체 클러스터 스캔
  • 무한한 지연

장점

  • 대규모 쓰기 처리량
  • 선형적 수평 확장성
  • 설계 상 고가용성

단점

  • 조인 없음, 진정한 SQL 유연성 없음
  • 쿼리 패턴에 묶인 엄격한 스키마
  • BI와 애드혹 분석에 부적합

결론: 와이드 컬럼 스토어는 쓰기 최적화된 시계열 엔진이지 분석 데이터베이스가 아닙니다. 미리 쿼리에 맞게 설계하거나, 실제 분석을 위해 OLAP 엔진(Druid, ClickHouse, BigQuery)과 페어링하세요.

6. 그래프 모델 (Neo4j, TigerGraph): 숨겨진 관계용

왜 중요한가: 관계(사기 링, 소셜 영향, 네트워크 홉)에 인사이트가 있을 때, 관계형 조인은 재귀적 깊이에서 무너집니다. 그래프 데이터베이스는 관계를 일급 시민으로 취급하여 다중 홉 순회를 빠르고 자연스럽게 만듭니다.

SQL이 실패하는 곳

-- SQL에서 N차 연결 (안티패턴)
-- 친구의 친구 찾기 (2홉)
SELECT DISTINCT f2.user_id
FROM friendships f1
JOIN friendships f2
  ON f1.friend_id = f2.user_id
WHERE f1.user_id = 123
  AND f2.user_id <> 123;

홉마다 조인이 기하급수적으로 늘어납니다. 3+ 레벨에서 쿼리를 읽을 수 없게 되고, 성능이 초 → 분/시간으로 저하됩니다.

그래프 모델 (승리)

MATCH (u:User {id: 123})-[:FRIEND*2..4]->(conn)
RETURN DISTINCT conn.id
LIMIT 100;

N차 순회가 밀리초 만에. 조인 폭발 없음. LinkedIn의 당신이 알 수도 있는 사람이 대규모 멀티홉 연결을 계산하기 위해 그래프 순회에 의존합니다.

사기 탐지

MATCH (a:Account)-[:TRANSFER]->(b:Account)-[:TRANSFER]->(c:Account)
WHERE a.country <> c.country
RETURN a, b, c;

SQL에서는 고통스럽거나 불가능한 의심스러운 자금 흐름을 즉시 표면화합니다.

장점

  • 번개처럼 빠른 N홉 순회
  • 네트워크의 자연스러운 모델링
  • 사기, 소셜, 추천 시스템에 이상적

단점

  • 대규모 집계나 OLAP에 부적합
  • SQL 대비 작은 생태계
  • 그래프 네이티브 사고방식 필요

결론: 질문이 "누가 누구와 연결되어 있나"로 시작한다면, SQL은 잘못된 도구입니다. 관계 중심 인사이트에서 그래프 데이터베이스는 조인보다 수십 배 뛰어납니다.

5. 스트리밍 이벤트 소싱 (Kafka + CDC)

왜 중요한가: 배치 ETL은 실시간 시스템과 근본적으로 호환되지 않습니다. CDC는 데이터베이스 변경을 불변 이벤트로 바꾸어 거의 제로 지연 파이프라인, 재생 가능한 상태, 마이크로서비스 전반의 시스템 일관성을 가능하게 합니다.

핵심 아이디어

  • 데이터베이스 → 이벤트 프로듀서
  • Kafka → 내구성 있는 이벤트 로그
  • 소비자 → 언제든지 상태 재구축
CREATE TABLE orders (
    order_id BIGINT PRIMARY KEY,
    user_id  BIGINT,
    status   TEXT,
    amount   DECIMAL(10,2),
    updated_at TIMESTAMP
);

CDC 이벤트 (Debezium 스타일)

{
  "op": "u",
  "before": { "status": "CREATED" },
  "after":  { "status": "PAID" },
  "ts_ms": 1735209123123,
  "source": { "table": "orders" }
}

모든 행 변경이 재생 가능한 사실이 됩니다.

멱등 싱크 (중요)

-- Kafka 이벤트를 소비하는 싱크 테이블
INSERT INTO orders_state (order_id, status, amount)
VALUES (:order_id, :status, :amount)
ON CONFLICT (order_id)
DO UPDATE SET
  status = EXCLUDED.status,
  amount = EXCLUDED.amount;

순서 및 정확히 한 번 (어려운 부분)

  • 비즈니스 키(order_id)로 Kafka 토픽 파티셔닝
  • 원자적 produce + commit을 위해 Kafka 트랜잭션 사용
  • DB 쓰기와 같은 트랜잭션에서 오프셋 추적

이를 잘못하면 → 자동 데이터 손상.

실제 증명: Uber는 이벤트 소싱을 사용해 100개 이상의 마이크로서비스를 동기화하며, 서비스가 데이터베이스 복원 대신 이벤트 재생으로 상태를 복구할 수 있습니다.

장점

  • 실시간 데이터 전파
  • 재생을 통한 내장 재해 복구
  • 프로듀서와 소비자 분리

단점

  • 높은 운영 복잡성
  • 순서 및 멱등성이 쉽지 않음
  • 디버깅에 이벤트 수준의 가시성 필요

결론: 이벤트 소싱은 최적화가 아닌 아키텍처적 전환입니다. 실시간 일관성, 감사 가능성, 복구가 필요하다면, Kafka + CDC가 백본이지만 올바름을 위해 먼저 엔지니어링하세요.

4. 컬럼형 스토리지 (Parquet, Delta Lake): 저비용 고속 분석용

왜 중요한가: 행 기반 데이터베이스는 포인트 조회를 위해 최적화되어 있지, 스캔을 위한 것이 아닙니다. 분석 워크로드는 수십억 행에서 몇 개의 컬럼을 읽습니다 — 컬럼형 스토리지가 구축된 목적이 바로 이것입니다. 결과: 비용의 일부로 수십 배 빠른 쿼리.

핵심 장점

  • 필요한 컬럼만 읽음
  • 높은 압축률 (RLE, 딕셔너리 인코딩)
  • 벡터화된 실행
CREATE TABLE sales_parquet (
    order_id BIGINT,
    region   STRING,
    amount   DECIMAL(10,2),
    order_ts TIMESTAMP
)
USING PARQUET
PARTITIONED BY (region, order_date);
SELECT
    region,
    SUM(amount) AS total_sales
FROM sales_parquet
WHERE order_date = '2025-12-25'
  AND region = 'US'
GROUP BY region;

벤치마크 현실: 50TB Parquet 테이블은 CSV로 저장된 동일 데이터보다 약 10배 빠르게 스캔할 수 있습니다.

Delta Lake 보너스 (실제 안전성)

MERGE INTO sales_parquet t
USING updates u
ON t.order_id = u.order_id
WHEN MATCHED THEN UPDATE SET amount = u.amount
WHEN NOT MATCHED THEN INSERT *;

데이터 레이크에서 ACID 보장 — 더 이상 깨진 배치 작업 없음.

장점

  • 대규모 스캔 성능
  • 탁월한 압축률
  • 클라우드 규모 분석에 완벽

단점

  • 고빈도 포인트 업데이트에 부적합
  • 신중한 파티션 설계 필요
  • 소형 파일 관리 오버헤드

결론: 컬럼형 스토리지는 원시 데이터를 분석 수퍼파워로 바꿉니다. 행 스토어에서 빅데이터를 스캔하고 있다면, 돈과 시간을 낭비하는 것입니다.

3. 멀티 모델 하이브리드 (SQL + NoSQL가 충돌할 때)

왜 중요한가: 실제 데이터는 하나의 형태인 경우가 드뭅니다. 현대 앱은 관계형 사실, 반구조화된 JSON, 관계를 혼합합니다. 멀티 모델 데이터베이스는 하나의 장소에서 모든 것을 쿼리할 수 있게 합니다.

전형적인 사용 사례

  • 구조화: 예약, 가격, 재고
  • 반구조화: 리스팅 메타데이터, 편의시설, 사용자 선호도
  • 그래프 같은: 호스트-게스트-리스팅 관계

(에어비앤비는 구조화된 테이블과 JSON 기반 리스팅 속성을 혼합합니다.)

CREATE TABLE listings (
    listing_id BIGINT PRIMARY KEY,
    city TEXT,
    price NUMERIC,
    attributes JSONB
);
SELECT
    city,
    COUNT(*) AS luxury_listings
FROM listings
WHERE price > 300
  AND attributes @> '{"amenities": ["pool", "wifi"]}'
  AND (attributes->>'pet_friendly')::boolean = true
GROUP BY city;

성능을 위한 인덱싱 (중요)

CREATE INDEX idx_listings_attrs
ON listings
USING GIN (attributes);

JSON 쿼리가 초 단위 → 밀리초로 단축됩니다.

장점

  • 여러 데이터 형태를 위한 하나의 시스템
  • 강력한 SQL + 유연한 스키마
  • ETL 및 데이터 중복 감소

단점

  • 스키마 거버넌스가 더 어려움
  • 쿼리 계획이 복잡해질 수 있음
  • 특화된 최고의 엔진만큼 빠르지 않음

결론: 멀티 모델 하이브리드는 현실적인 절충안입니다. 데이터가 테이블, 문서, 관계에 걸쳐 있을 때, 뷰와 인덱스 뒤에 복잡성을 숨기는 한 민첩성을 제공합니다.

2. 리버스 ETL (운영 분석): 데이터를 앱에 다시 넣기

왜 중요한가: 전통적 분석은 대시보드에서 멈춥니다. 리버스 ETL은 루프를 닫아 웨어하우스 모델링된 데이터를 운영 도구로 다시 밀어 넣어, 인사이트가 보고서가 아닌 행동을 유도하게 합니다.

핵심 사용 사례

  • 준실시간 개인화
  • 실시간 고객 건강 및 이탈 점수
  • 영업 및 마케팅 자동화 (CRM, ESP)

웨어하우스가 인텔리전스 시스템이 되고, 앱이 실행 엔진이 됩니다.

-- customer_360.sql
SELECT
    customer_id,
    MAX(last_login_ts)              AS last_active,
    SUM(lifetime_spend)             AS ltv,
    COUNT(DISTINCT order_id)        AS order_count,
    CASE
        WHEN SUM(lifetime_spend) > 5000 THEN 'VIP'
        WHEN SUM(lifetime_spend) > 1000 THEN 'High'
        ELSE 'Standard'
    END AS customer_tier
FROM orders
GROUP BY customer_id;
warehouse.customer_360
   → Hightouch / Census
       → Salesforce (Account Tier)
       → HubSpot (Lifecycle Stage)
       → Zendesk (Support Priority)

증분 업데이트 (중요)

SELECT *
FROM customer_360
WHERE updated_at > :last_sync_ts;

장점

  • 분석을 행동으로 전환
  • 비즈니스 로직의 단일 소스
  • 엔지니어링 변경 없이 빠른 실험

단점

  • SaaS API 제한에 의존
  • 앱 소유 필드 덮어쓰기 위험
  • 강력한 데이터 거버넌스 필요

결론: 리버스 ETL은 웨어하우스를 보고 묘지에서 운영 엔진으로 변환합니다. 인사이트가 고객 대면 도구에 도달하지 않는다면, 가치를 창출하지 못하는 것입니다.

1. 유니파이드 서빙 레이어: 프로덕션 데이터의 미래

왜 중요한가: 현대 데이터 스택은 OLTP, OLAP, 검색, 스트리밍 시스템에 걸쳐 데이터를 분열시켜 동기화 지연과 중복 로직을 만듭니다. 유니파이드 서빙 레이어는 하나의 논리적 데이터 레이어(Iceberg/Hudi/Delta)를 사용해 SQL 분석, 준실시간 읽기, ML, 그래프/검색 워크로드 등 다중 액세스 모드를 지원합니다.

하나의 데이터셋. 여러 엔진. 리라이트 없음.

무엇을 대체하는가

  • ETL 분산 (웨어하우스 → 레이크 → 피처 스토어)
  • 시스템 간 데이터 드리프트
  • SQL, Spark, 앱에서 동일한 로직 재구현

빅테크 증명: LinkedIn Data Hub, Uber Marmaray, Netflix Iceberg가 페타바이트 규모 추천을 구동합니다.

Iceberg 통합 테이블

CREATE TABLE user_events (
    user_id BIGINT,
    event_type STRING,
    event_ts TIMESTAMP,
    attributes MAP<STRING, STRING>
)
USING ICEBERG
PARTITIONED BY (days(event_ts));

OLAP 쿼리 (Trino / Spark SQL)

SELECT
    event_type,
    COUNT(*) AS cnt
FROM user_events
WHERE event_ts >= current_date - INTERVAL '7' DAY
GROUP BY event_type;

증분 / 준실시간 읽기

SELECT *
FROM user_events
FOR SYSTEM_TIME AS OF TIMESTAMP '2025-12-26 10:00:00';

타임 트래블, 백필, 재현 가능한 ML 피처를 가능하게 합니다.

같은 테이블에서 스트리밍 + 배치

  • Kafka / Flink → Iceberg에 쓰기
  • Trino / Spark → 즉시 읽기
  • ML → 동일한 스냅샷에서 훈련

오픈 테이블 포맷은 스토리지와 컴퓨팅을 분리합니다 — 데이터 마이그레이션 없이 엔진을 교체할 수 있습니다.

장점

  • 단일 진실 소스
  • 멀티 엔진 액세스 (SQL, ML, 스트리밍)
  • ACID, 타임 트래블, 스키마 진화
  • 벤더 잠금 없음

단점

  • 더 높은 플랫폼 복잡성
  • 강력한 거버넌스 및 메타데이터 관리 필요
  • OLTP를 대체하는 드롭인이 아님

결론: 유니파이드 서빙 레이어는 최종 아키텍처입니다: 하나의 데이터셋, 여러 워크로드, 제로 사일로. 다음 10년을 위해 설계한다면, 오픈 테이블 포맷은 협상 불가능한 요소입니다.
