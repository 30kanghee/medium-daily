100,000 GitHub Stars in Weeks: Why Developers Can't Stop Talking About GStack

The Latency Gambler · 4 min read · 원문 · freedium

YC CEO Garry Tan이 공개한 AI 코딩 워크플로우 툴킷 GStack이 몇 주 만에 GitHub 스타 10만 개를 획득하며 개발자 커뮤니티의 주목을 받고 있다.

요약

2026년 3월, Y Combinator CEO Garry Tan이 자신의 AI 코딩 워크플로우 전체를 담은 오픈소스 저장소 GStack을 공개했고, 48시간 만에 스타 1만 개, 몇 주 만에 10만 개를 돌파했다. GStack은 프레임워크나 라이브러리가 아닌, Claude Code가 자동으로 불러오는 23개의 역할 기반 슬래시 커맨드 스킬 모음으로, 기획·설계·개발·리뷰·테스트·배포·회고의 전 주기를 가상의 팀원들이 담당하는 구조다. 특히 /qa 스킬은 Bun과 Playwright 기반의 영구 브라우저 데몬을 활용해 스테이징 환경에서 60초 만에 앱 헬스 점수를 산출하는 기술적 차별점을 제공한다. Tan은 GStack을 사용해 60일간 60만 줄의 프로덕션 코드를 작성했다고 밝혔으며, 이 숫자에 대한 논란이 있었으나 프로세스의 일관성이 진짜 가치라는 반론도 제기됐다. 이 프로젝트는 AI 코딩 에이전트에서 앞서나가는 개발자들이 더 나은 모델이 아닌 더 나은 프로세스를 채택하고 있음을 보여준다.

YC CEO가 자신의 AI 코딩 셋업 전체를 오픈소스로 공개하다. 개발자 커뮤니티가 즉각 반응했다.

2026년 3월 12일, Y Combinator의 사장 겸 CEO인 Garry Tan이 gstack이라는 저장소를 GitHub에 올렸다. 48시간 만에 스타 1만 개를 기록했고, 몇 주 만에 10만 개에 근접했다.

이런 속도는 우연이 아니다. 사람들이 조용히 씨름해왔지만 깔끔하게 해결된 것을 본 적 없던 문제를 어떤 도구가 해결할 때 일어나는 일이다.

그 문제: Claude Code(그리고 AI 코딩 에이전트 일반)는 단독으로는 강력하지만 프로세스를 제공하지 않는다. 기획, 리뷰, 테스트, 배포를 둘러싼 구조가 없다. 모든 개발자가 매 프로젝트, 매번 그 비계(scaffolding)를 처음부터 다시 쌓는다.

GStack은 그 비계를, 23개의 의견이 담긴 슬래시 커맨드 스킬로 코드화한 것이다.

저장소: github.com/garrytan/gstack

GStack이 실제로 무엇인가

GStack은 프레임워크도, 라이브러리도, 새로운 AI 모델도 아니다. SKILL.md 파일들의 모음이다. Claude Code가 .claude/skills/gstack/에서 자동으로 불러오고 역할 정의로 취급하는 구조화된 지시 집합이다.

각 스킬은 소프트웨어 팀의 한 구성원을 시뮬레이션한다:

GStack 가상 팀 구조:
─────────────────────────────────────────────────
/office-hours       ──►  제품 사고, 한 줄도 쓰기 전에
                         약한 아이디어를 밀어낸다

/plan-ceo-review    ──►  까다로운 CEO처럼 행동한다
                         (Brian Chesky를 모델로)
                         지루한 기능을 만들지 못하게 막는다

/plan-eng-review    ──►  코딩 전에 시퀀스 다이어그램과
                         상태 머신을 그린다. 아키텍처를 확정한다

/review             ──►  N+1 쿼리, 오래된 읽기,
                         보안 이슈(XSS 등)를 스캔하는 코드 리뷰어

/qa                 ──►  Bun + Playwright로 실제 브라우저를 연다.
                         스테이징에 로그인하고, 클릭하고,
                         스크린샷을 찍고, 60초 만에 앱 헬스 점수를 준다

/ship               ──►  릴리즈 엔지니어. 커밋하고,
                         PR을 열고, 체인지로그를 처리한다

/retro              ──►  엔지니어링 회고.
                         LOC, 커밋, 출시된 것, 망가진 것

역할들은 전체 개발 라이프사이클을 커버한다: 사고 → 계획 → 구축 → 리뷰 → 테스트 → 배포 → 회고.

30초 만에 설치

git clone --single-branch --depth 1 \
  https://github.com/garrytan/gstack.git \
  ~/.claude/skills/gstack \
  && cd ~/.claude/skills/gstack && ./setup

그 후 Claude Code에서 아무 프로젝트나 열면 스킬이 슬래시 커맨드로 사용 가능해진다. 시스템 PATH에 아무것도 건드리지 않는다. 백그라운드에서 지속적으로 실행되는 것도 없다. 모든 것이 .claude/ 안에 있다.

QA 도구가 기술적 핵심이다

GStack의 대부분은 좋은 프롬프팅으로 근사할 수 있다. /qa 스킬은 다르다.

Bun과 Playwright로 구축된 영구 브라우저 데몬으로, 스테이징 환경에 로그인하고, 앱을 탐색하고, 스크린샷을 찍고, 헬스 리포트를 만든다. 명령 사이에 페이지 리로드 없음. 브라우저가 무엇을 보고 있는지 다시 설명하느라 컨텍스트가 부풀지 않는다.

아키텍처가 주목할 만하다:

표준 Claude Chrome MCP:
──────────────────────────────
Claude ──► MCP 호출 ──► Chrome (매번 새 컨텍스트)
                              │
                    호출 사이에 상태가 사라진다.
                    컨텍스트 창이 빠르게 찬다.

GStack /qa (Bun + Playwright 데몬):
──────────────────────────────────────
Claude ──► /qa 스킬 ──► 영구 브라우저 데몬
                              │
                    같은 세션, 같은 탭.
                    스크린샷이 컨텍스트로 전달된다.
                    컨텍스트 오버헤드 20배 감소.
                    ~60초 만에 헬스 점수 산출.

호출마다 새 브라우저 컨텍스트를 생성하는 대신 영구 데몬을 실행하는 것은 작은 아키텍처 결정이지만 의미 있는 성능 차이를 만든다. 이것이 /qa 스킬이 1분 이내에 전체 앱 헬스 점수를 산출하는 이유이고, 반면 기본 Chrome MCP 세션은 부풀어 오르는 경향이 있는 이유다.

프로덕션 수치

Tan은 자신의 처리량 지표를 README에 포함시켰고, 이것이 논쟁을 촉발했다.

그는 GStack을 사용해 60일간 60만 줄의 프로덕션 코드를 작성했고, Y Combinator를 풀타임으로 운영하면서 하루 평균 1만~2만 줄의 사용 가능한 코드를 산출했다고 보고했다. 그의 2026년 GitHub 기여 횟수는 이미 2013년 전체 기여량의 240배를 넘어섰다.

예상대로 "하지만 그건 AI가 쓴 코드일 뿐"이라는 비판이 따랐다. 그의 반응은 직접적이었다: 중요한 것은 누가 타이핑했느냐가 아니라 무엇이 출시됐느냐다. 인플레이션을 감안해 정규화하면, 이전보다 더 높은 품질과 더 빠른 속도였다.

수치를 받아들이든 아니든, 저장소 구조가 같은 이야기를 한다. GStack은 대부분 Claude Opus와 공동 저술되었으며 커밋 히스토리가 이를 보여준다. 그것이 코드베이스에 내장된 개념 증명이다: 자신의 워크플로우를 사용해 구축된 도구.

빠른 채택이 실제로 신호하는 것

10만 스타는 워크플로우 도구치고 의미 있는 수치다, 라이브러리가 아닌. 아무도 GStack을 의존성으로 추가하지 않는다. 그들은 일하는 방식을 바꾸고 있다.

그것이 나타내는 패턴이 명확해지고 있다: AI 코딩 에이전트에서 가장 앞서나가는 개발자들은 더 나은 모델이나 더 많은 컴퓨팅을 사용하고 있지 않다. 더 나은 프로세스를 사용하고 있다. 그들은 개발 철학, 즉 계획 기준, 리뷰 기준, 테스트 기대치를 매번 일관되게 실행되는 지시 집합으로 인코딩했다.

GStack은 한 개발자의 완전한 철학을 23개의 스킬로 인코딩한 것이다. 스타 수는 많은 다른 개발자들이 그 철학을 공유하거나, 그것의 어떤 버전을 빌려올 것을 찾고 있었음을 시사한다.

그것이 가리키는 전환:

기존 모델:
  개발자가 코드를 작성한다
  개발자가 (선택적으로) 코드를 리뷰한다
  개발자가 배포한다

GStack 모델:
  /office-hours  아이디어를 검증한다
  /plan-ceo-review  범위를 압박 테스트한다
  /plan-eng-review  아키텍처를 확정한다
  개발자가 코드를 작성한다 (또는 에이전트가 한다)
  /review  프로덕션 이슈를 잡는다
  /qa  실제 브라우저에서 검증한다
  /ship  PR을 연다
  /retro  무슨 일이 있었는지 측정한다

8개의 체크포인트. 개발자 한 명. 얼마나 늦었든, 얼마나 피곤하든 일관된 프로세스.

하지 않는 것

GStack은 의견이 강하고 워크플로우가 무겁다. 한 사람이 여러 역할을 맡는 1인 개발자나 소규모 팀에서 가장 잘 작동한다. 전담 QA, 디자인, 엔지니어링 매니지먼트가 있는 큰 팀에서는 대부분의 스킬이 불필요해진다. 실제 사람들이 그 역할을 채우고 있으니까.

또한 초기 투자가 필요하다. CLAUDE.md 설정, 프로젝트 컨텍스트, 스프린트 구조에 익숙해지기. GStack과의 첫 프로젝트는 없을 때보다 느리다. 복리 효과는 그 이후에 온다.
