If You Understand These 5 Data Engineering Terms, You're Ahead of 90% of the Industry

B V Sarath Chandra · Towards Data Engineering · 5 min read · 원문 · freedium

요약

데이터 엔지니어링의 핵심 개념 다섯 가지만 제대로 이해해도 업계 상위 10%에 진입할 수 있다.

데이터 엔지니어링 업계는 SaaS 도구 목록 암기와 실제 분산 시스템 이해 사이에서 크게 나뉜다. 저자는 멱등성(Idempotency), 네트워크 셔플, 1:N 팬아웃 함정, 메달리온 아키텍처, 시맨틱 레이어라는 다섯 가지 개념을 진정으로 이해하는 것만으로 거의 모든 동료보다 앞설 수 있다고 주장한다. 이 개념들은 파이프라인이 왜 느린지, 왜 클라우드 비용이 폭등하는지, 왜 AI 챗봇이 엉뚱한 답을 내놓는지를 설명하는 물리 법칙과 같다. 새로운 도구를 쫓는 대신, 시스템의 근본 원리를 이해하는 것이 진짜 데이터 엔지니어를 구별 짓는다.

솔직하게 말하겠다. 요즘 데이터 엔지니어링에 대해 이야기하는 사람들은 대부분 두 부류다. Airflow, Snowflake, dbt, Kafka 같은 SaaS 도구 목록을 줄줄 읊거나, PySpark 작업이 왜 클러스터를 다운시켰는지 실제로 물어보면 눈이 멍해지거나.

그 두 부류 중 어느 쪽도 될 필요가 없다.

다섯 가지 핵심 개념이 있다. 이 개념들을 단순히 암기하는 것이 아니라 실제로 이해하면, 방 안의 거의 모든 사람보다 훨씬 앞서게 된다. 주니어 개발자든, 파이프라인 장애에 지친 데이터 사이언티스트든, 클라우드 비용이 왜 이렇게 나오는지 이해하려는 리더든 상관없이.

1. 멱등성 (Idempotency)

가장 먼저 머릿속에 새겨야 할 것은, 데이터 파이프라인은 실패한다는 사실이다. 끊임없이 실패한다. API가 다운되고, 클러스터가 메모리 부족에 빠지고, 소스 데이터가 늦게 도착한다. 그렇다면 파이프라인이 절반쯤 실행되다 실패했을 때 다시 실행하면 어떻게 될까?

바로 이 지점에서 멱등성이 등장한다. 멱등성 파이프라인이란, 한 번이든 두 번이든 쉰 번이든 연속으로 실행해도 데이터베이스의 최종 상태가 딱 한 번만 실행한 것과 동일한 파이프라인이다.

파이프라인이 단순히 INSERT 명령을 사용한다면, 두 번 실행하면 데이터가 그대로 복제된다. 대시보드가 갑자기 매출을 두 배로 보여주기 시작하고, CEO가 패닉 상태에 빠진다.

멱등성 파이프라인은 MERGE(Upsert) 명령을 쓰거나 특정 파티션을 안전하게 삭제하고 교체한다. 데이터베이스를 보며 이렇게 말한다. "이 행이 이미 있으면 업데이트하고, 없으면 삽입해."

주니어 엔지니어는 모든 게 잘 될 것이라 가정하며 파이프라인을 만든다. 시니어 엔지니어는 모든 게 잘못될 것이라 알기 때문에 멱등성 파이프라인을 만든다.

멱등성을 이해하면, Airflow DAG에서 '재시도' 버튼을 눌렀을 때 프로덕션 데이터베이스를 망가뜨린 건 아닌지 새벽 3시에 식은땀을 흘리며 깨는 일이 없어진다.

2. 네트워크 셔플 (The Network Shuffle)

상업용 주방을 상상해보자. 셰프 10명(Executor)이 각자 자리에서 채소를 썰고 있다. 엄청나게 빠르다. 그런데 갑자기 다진 양파는 한 테이블에, 당근은 다른 테이블에, 셀러리는 또 다른 테이블에 모아달라고 한다. 혼란이 펼쳐진다. 셰프들이 주방을 가로질러 뛰고, 서로 부딪히고, 그릇을 날라 줄을 선다. 더 이상 아무도 채소를 썰지 않는다. 그냥 음식을 옮기고 있을 뿐이다.

이것이 셔플이다.

Spark나 Databricks 같은 분산 컴퓨팅에서 데이터는 여러 워커 노드에 분산된다. .filter() 같은 명령은 노드가 데이터를 제자리에서 처리한다. 번개처럼 빠르다. 하지만 .groupBy()나 .join()을 실행하면, 엔진이 일치하는 키를 한곳에 모으기 위해 네트워크를 통해 방대한 데이터를 물리적으로 이동시켜야 한다.

셔플은 데이터 엔지니어링에서 가장 비싸고 시간이 많이 걸리는 단일 작업이다. 파이프라인이 4분이 아닌 4시간 걸리는 이유가 바로 이것이다.

셔플을 이해하면, 느린 코드를 고치겠다고 무작정 클러스터에 메모리를 더 추가하는 짓을 멈추게 된다. 대신 로직을 바꾼다. 조인하기 전에 데이터를 필터링하고, 작은 테이블은 브로드캐스트한다. 하드웨어가 아니라 데이터의 물리 법칙을 최적화하는 것이다.

3. 1:N 팬아웃 함정 (The 1:N Fan-Out Trap)

이것은 내가 가장 즐겨 설명하는 개념이다. 클라우드 컴퓨팅 비용을 조용히 갉아먹는 무성의 살인자이기 때문이다.

SQL로 두 테이블을 조인할 때, 키를 기반으로 행을 매칭한다. 테이블 A에 1,000개 행이 있고, 테이블 B에 LEFT JOIN하면 결과도 1,000개 행이어야 한다고 생각하지 않는가? 테이블 B에 중복이 없다면 그렇다.

하지만 사용자 테이블을 구매 테이블에 조인하는데, 한 사용자가 5번 구매했다면? 그 사용자 행 하나가 결과에서 5개로 복제된다. 이것이 일대다(1:N) 관계다.

이제 1억 행짜리 테이블을 수십억 클릭이 담긴 로그 테이블에 실수로 조인했다고 상상해보자. 카르테시안 폭발이 일어난다. 1억 행짜리 테이블이 순식간에 500억 행짜리 괴물로 팬아웃된다. Spark 클러스터는 RAM이 부족해서 디스크로 스필되고, 결국 크래시한다. 클라우드 컴퓨팅 비용으로 수백 달러가 사라진다.

팬아웃 함정을 이해하면, 연결하는 테이블의 그래뉼래리티(고유성)를 꼼꼼히 검증하지 않고는 절대 JOIN을 작성하지 않게 된다.

4. 메달리온 아키텍처 (Medallion Architecture)

모두가 쓰는 용어지만, 제대로 실행하는 사람은 드물다.

앱에서 원시 JSON 파일을 비즈니스가 바로 읽는 대시보드에 직접 던지면 골치 아파진다. 데이터는 지저분하고, 중첩되어 있고, 오류투성이다.

메달리온 아키텍처는 데이터 레이크하우스에 질서를 가져오는 방법이다. 데이터를 세 가지 품질 수준으로 구성하는 간단한 개념이다.

  • Bronze(브론즈): 원본 그대로의 필터링되지 않은 데이터. 소스 시스템이 망가져도 항상 원본 브론즈 데이터로 재처리할 수 있다.
  • Silver(실버): 정제되고, 필터링되고, 타입이 지정된 데이터. 중복 제거와 표준화가 완료된 상태.
  • Gold(골드): 비즈니스 수준의 집계 데이터. 임원이 바로 쿼리할 수 있도록 사전 조인된 고도로 정제된 Star Schema.

비즈니스 관계자가 어떤 지표가 틀렸다고 불평하면, 메달리온 아키텍처는 정확히 어디서 디버깅해야 하는지 알려준다. 500줄짜리 SQL 쿼리를 풀 필요가 없다. 골드 테이블을 확인하고, 거기서 로직이 잘못됐다면 고치고, 골드 테이블이 괜찮다면 실버 테이블에서 누락된 데이터를 확인한다. 혼돈스러운 데이터 늪지를 관리 가능하고 감사 가능한 공급망으로 바꿔준다.

5. 시맨틱 레이어 (The Semantic Layer)

이것은 다섯 가지 중 가장 오해받는 개념이다. 특히 AI가 등장한 지금은 더욱 그렇다.

요즘 모든 CEO는 "지난달 매출이 얼마였어?"라고 타이핑하면 즉각 답이 나오는 "텍스트-to-SQL" 챗봇을 원한다. LLM을 날것의 데이터베이스에 직접 연결하면 환각이 일어난다. "매출"이 gross_sales 컬럼인지 net_arr 컬럼인지 알지 못하고, 환불된 거래를 필터링해야 한다는 것도 모른다.

이것이 시맨틱 레이어가 필요한 이유다.

시맨틱 레이어(dbt 같은 도구 사용)는 비즈니스 지표를 코드로 정의하는 곳이다. "매출 = gross_sales — refunds, status = 'active'인 것만"이라는 엄격한 정의를 작성한다.

시맨틱 레이어가 있으면, AI가 SQL 조인을 추측하도록 내버려 두지 않아도 된다. AI가 그냥 시맨틱 레이어에 '매출' 지표를 요청하면, 시맨틱 레이어가 사람이 검증한 완벽한 SQL 코드로 변환한다.

엔터프라이즈에서 실제로 작동하는 AI 에이전트를 만들고 싶다면 더 좋은 프롬프트 엔지니어링이 필요한 게 아니다. 시맨틱 레이어가 필요하다.

상위 10%에 오신 것을 환영합니다

Python 스크립트를 작성할 줄 아는 사람과 분산 시스템이 실제로 어떻게 작동하는지 이해하는 사람 사이의 간극은 어마어마하다.

ProductHunt에서 새로 나오는 모든 도구를 암기할 필요가 없다. 하지만 멱등성을 이해하면 파이프라인이 밤새 살아남는다. 셔플과 팬아웃 함정을 이해하면 회사의 클라우드 비용을 수천 달러 아낀다. 메달리온 아키텍처를 이해하면 데이터가 실제로 거버넌스된다. 시맨틱 레이어를 이해하면 내일의 AI 인프라를 구축할 준비가 된 것이다.

다섯 개의 개념. 진짜 엔지니어링. 상위 10%에 오신 것을 환영한다.
