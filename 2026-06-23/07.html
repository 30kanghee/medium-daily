Learning Data Engineering from Scratch? Start with these Skills to Get Hired

Satyam Sahu · Stackademic · 9 min read · 원문 · freedium

요약

데이터 엔지니어링을 처음 배우는 사람이 취업까지 닿으려면, 20가지 도구를 동시에 익히려는 욕심을 버리고 SQL·Python·클라우드 웨어하우스·dbt·오케스트레이션이라는 핵심 다섯 축을 순서대로 탄탄히 쌓아야 한다.

데이터 엔지니어의 역할은 "디지털 배관공"에 가깝다. 원천 시스템에서 쏟아지는 날 것의 데이터를 비즈니스가 쓸 수 있는 깔끔한 형태로 만들어 전달하는 파이프라인을 구축하는 것이 핵심 업무다. 이 글은 그 여정을 5단계 Phase로 나눠 설명한다. Phase 1에서는 SQL과 Python을, Phase 2에서는 BigQuery나 Snowflake 같은 클라우드 데이터 웨어하우스를, Phase 3에서는 변환 도구인 dbt를, Phase 4에서는 Airbyte 같은 수집 도구와 Airflow 같은 오케스트레이션을, Phase 5에서는 파이프라인 모니터링과 장애 진단(Observability)을 다룬다. 저자는 "튜토리얼 지옥"을 경계하며, 실제로 손을 움직여 간단한 프로젝트를 완성하는 것이 이력서에 도구 이름을 나열하는 것보다 훨씬 효과적임을 강조한다.

데이터 엔지니어링을 처음 시작하면 막막함부터 밀려온다. 도구 이름, 유행어, 튜토리얼이 넘쳐나서 어디서부터 손을 대야 할지조차 모르겠다는 느낌. 그 감각은 누구나 겪는다.

20가지 도구를 나열한 "로드맵"을 보고 나면 오히려 더 혼란스러워진다. 사실 한꺼번에 모든 것을 익힐 필요가 없다. 현장에서 실제로 쓰이는 핵심 기술에 집중하는 것이 가장 빠른 성장 경로다.

대부분의 입문자가 이 벽에 부딪히는 이유는, 필드가 어려운 게 아니라 "모든 것을 동시에 배워라"라는 잘못된 조언 때문이다. SQL 쿼리 하나 제대로 못 짜는 상태에서 복잡한 파이프라인과 고급 인프라를 건드리다 보면 몇 달이 지나도 자신감만 바닥으로 떨어진다.

진실은 간단하다. 데이터 엔지니어가 되려면 스무 가지 도구가 필요하지 않다. 소수의 기술을 제대로 이해하고, 그것들이 서로 어떻게 맞물리는지를 파악하면 된다. 이 글은 그 명확한 길을 제시한다.

데이터 엔지니어가 하는 일: 한 문장으로

Python이나 SQL을 이야기하기 전에, 데이터 엔지니어가 실제로 무엇을 하는지부터 살펴보자. 전문 용어를 걷어내면 이 직업은 놀랍도록 단순하다.

쉽게 말하면 디지털 배관공이다.

실제 배관공은 수원지에서 나온 깨끗한 물이 주방 수도꼭지까지 안전하게 도달하도록 파이프를 관리한다. 압력도 맞추고, 새는 곳도 없애고, 물도 안전하게 유지한다. 데이터 엔지니어가 하는 일이 정확히 그것이다.

  • 수원지: TikTok, Stripe, 웹사이트 같은 앱에서 쏟아지는 날 것의 지저분한 데이터.
  • 파이프라인: 그 데이터를 이동시키는 도구들.
  • 수도꼭지: 경영진이 의사결정에 활용하는 최종 대시보드나 리포트.

당신의 역할은 물(데이터)을 만들거나 마시는(분석하는) 게 아니다. A 지점에서 B 지점까지 데이터가 오염 없이 흐르도록 신뢰할 수 있는 파이프라인을 구축하는 것이다.

모던 데이터 스택

예전에는 엔지니어가 이 파이프를 처음부터 직접 만들어야 했다. 지금은 레고 블록처럼 조립 가능한 고품질 오픈소스·SaaS 도구들이 즐비하다. 이것이 바로 모던 데이터 스택(Modern Data Stack)이다. 바퀴를 새로 발명할 필요 없이, 어떤 블록들이 어떻게 맞물리는지를 익히는 것이 목표다.

결국 "데이터를 옮기고 다듬는다"는 본질을 이해하면, "뭘 먼저 배워야 하지?"라는 공황 상태는 자연스럽게 사라진다.

Phase 1: 반드시 갖춰야 할 기반 — SQL과 Python

클라우드 로고를 들여다보기 전에 토대부터 닦아야 한다. 이 분야에서 절대 타협할 수 없는 두 가지 기술이 있다. 이것을 건너뛰고 고급 도구로 직행하면 반드시 고생한다.

SQL: 데이터의 언어

SQL(Structured Query Language)은 데이터베이스가 이해하는 유일한 언어다. 이걸 모르면 데이터를 꺼낼 수 없다. 우리가 하는 모든 일의 밑바탕이다.

  • 하는 일: 데이터를 조회하고, 필터링하고, 집계한다. "지난 11월 인도에서 아이폰 15가 몇 대 팔렸나?"처럼 질문을 던지는 언어다.
  • 집중할 것: 복잡한 데이터베이스 관리는 나중 문제다. SELECT 문, JOIN(테이블 결합), GROUP BY(집계), 그리고 윈도우 함수에 집중하라.
  • 현실 점검: Python 실력이 평범해도 훌륭한 데이터 엔지니어가 될 수 있지만, SQL이 흔들리면 살아남기 어렵다.

Python: 자동화의 언어

SQL이 웨어하우스 내부의 데이터를 다룬다면, Python은 그 외부의 모든 것을 담당한다. 파이프라인 전체를 하나로 이어주는 접착제다.

  • 하는 일: 파이프라인을 만들고 반복 작업을 자동화한다. 매일 아침 9시에 이메일 첨부 파일을 받아 데이터베이스에 업로드해야 한다면, Python 스크립트 하나로 해결된다.
  • 집중할 것: 소프트웨어 개발자처럼 대형 앱을 만들 필요 없다. 파일을 이동시키거나, 지저분한 텍스트를 정리하거나, API와 통신하는 작은 스크립트를 짜는 "스크립팅" 능력에 집중하라.

Git: 코드의 저장 버튼

로드맵에서 자주 빠지지만 꼭 필요한 것이 Git이다. Git은 버전 관리 시스템으로, 500줄짜리 코드를 작성하다가 핵심 부분을 실수로 지웠을 때 살려주는 "되돌리기" 버튼이다. 모든 변경 사항을 추적하고, 다른 엔지니어의 작업을 덮어쓰지 않고 협업할 수 있게 해준다.

황금 원칙: 이 세 가지를 서두르지 말고 제대로 익혀라. 이것들이 뒤에 나오는 모든 도구를 받쳐주는 기둥이다.

Phase 2: 실습 무대 — 클라우드 데이터 웨어하우스

SQL과 Python을 갖췄다면, 이제 실제로 작업할 공간이 필요하다. 예전에는 회사가 거대한 물리 서버를 직접 구매했지만, 지금은 클라우드 제공업체에서 강력한 저장 공간을 빌려 쓴다. 이것이 클라우드 데이터 웨어하우스다.

하나를 고르고 나머지는 무시하라

입문자들이 여기서 자주 패닉에 빠진다. 채용 공고에 AWS, Azure, Google Cloud, Snowflake, Databricks가 모두 나오면 전부 배워야 할 것 같은 느낌이 든다. 하지만 그럴 필요 없다.

이 플랫폼들은 근본 원리가 거의 같다. 하나를 제대로 익히면 나머지는 주말 안에 따라잡을 수 있다.

  • BigQuery (Google): 절대 초보에게 강력 추천. Gmail 계정만 있으면 설정 없이 바로 쿼리를 시작할 수 있다.
  • Snowflake: 저장과 연산을 분리해 비용을 절감하는 구조로 기업 환경에서 매우 인기 있고, 사용자 친화적이다.
  • Databricks: 나중에 데이터 사이언스 쪽으로 넘어가고 싶다면 좋은 선택지지만, 학습 곡선이 다소 가파르다.

실습 체크리스트

튜토리얼만 보지 말고 직접 손을 더럽혀라. 플랫폼 하나를 골라 데이터의 전체 생애 주기를 경험해보자.

  1. 생성: 새 데이터베이스와 테이블을 만드는 법을 익혀라.
  2. 적재: 로컬 CSV 파일을 테이블에 불러오는 법을 익혀라.
  3. 조회: SQL로 데이터에 질문을 던져라.
  4. 최적화: 비용이 폭증하지 않도록 파티셔닝 같은 기본 저장 전략을 파악해라.

Phase 3: 변환 사고방식 — dbt

이 단계가 전환점이다. "SQL을 아는 사람"에서 "데이터 엔지니어"로 졸업하는 구간이 바로 여기다.

과거 방식 vs. 현재 방식(ELT)

과거에는 ETL(Extract-Transform-Load) 방식을 썼다. 데이터를 웨어하우스에 넣기 전에 미리 정제했다. 지금은 ELT(Extract-Load-Transform)가 표준이다. 날 것의 지저분한 데이터를 일단 웨어하우스에 던져 넣고, 웨어하우스 자체의 연산 능력으로 나중에 정제하는 방식이다.

dbt란 무엇인가

이 "변환(Transform)" 단계의 업계 표준 도구가 dbt(data build tool)다. 클라우드 웨어하우스가 식재료를 보관하는 냉장고라면, dbt는 재료를 썰고 간을 맞추고 요리하는 작업대다.

dbt는 소프트웨어 엔지니어링의 모범 사례를 데이터 세계로 가져온다.

  • 모듈형 SQL: 1000줄짜리 거대한 스크립트 대신, 재사용 가능한 작은 논리 단위("모델")로 쪼개서 작성한다.
  • 깔끔한 데이터 모델링: table_A와 table_B가 의미 있는 방식으로 연결되도록 데이터 구조를 잡아준다.
  • 테스트: "order_id가 null이면 즉시 경고" 같은 자동화 테스트를 작성할 수 있다.
  • 문서화: 팀원 누구나 각 테이블의 의미를 알 수 있도록 가이드북을 자동 생성한다.
  • 버전 관리: Git 사용을 강제해 작업 내역을 절대 잃지 않게 한다.

핵심 메시지: dbt를 배우는 것은 단순히 도구 하나를 배우는 게 아니다. 신뢰할 수 있고, 테스트 가능하고, 깔끔한 파이프라인을 만드는 방법을 배우는 것이다. 이력서에 "dbt"를 올리면 채용 담당자는 당신이 현대적이고 품질 높은 데이터 워크플로우를 이해한다고 읽는다.

Phase 4: 파이프 연결하기 — 수집과 오케스트레이션

웨어하우스(저장)와 dbt(처리)가 갖춰졌으니, 이제 흐름을 자동화해야 한다. 매일 아침 6시에 직접 "실행" 버튼을 누르는 사람이 될 수는 없다. 로봇이 그 일을 대신하게 해야 한다.

수집(Ingestion): 배달 트럭

수집이란 Stripe, Salesforce, Facebook Ads 같은 원천(Source A)에서 웨어하우스(목적지 B)로 데이터를 옮기는 과정이다.

  • 초보자의 실수: 모든 데이터 소스에 맞춰 Python 스크립트를 직접 짜는 것. "Facebook용 스크립트 하나, Google Sheets용 하나, Shopify용 하나…"
  • 문제점: API는 끊임없이 바뀐다. 잠자는 사이 Facebook이 API를 업데이트하면 스크립트가 죽고, 아침에 눈을 뜨면 비상사태가 기다린다.
  • 전문가의 해법: Airbyte나 Fivetran 같은 전문 수집 도구를 사용하라. 험한 길에서도 흔들리지 않는 장갑차 배달 트럭 같은 도구들이다.
  • 실습 과제: Airbyte를 로컬에 내려받아 직접 실행해보라. 무료 오픈소스이며, 비싼 엔터프라이즈 소프트웨어 없이도 데이터 이동을 직접 경험할 수 있다.

오케스트레이션(Orchestration): 지휘자

오케스트레이터는 파이프라인 전체의 두뇌다. 서로 다른 도구들에게 언제 시작하고 언제 멈출지를 지시한다.

  • 비유: 공사 현장을 생각해보자. 배관공, 전기 기사, 도장공이 있는데, 벽이 완성되기 전에 도장공이 붓을 들면 재앙이다. 오케스트레이터는 "벽 완성(작업 A). 이제 도장 시작(작업 B). 도장공 결근 시(오류) 대체 인력 투입(재시도)"을 관리하는 현장 소장이다.
  • 도구: 업계 표준은 Apache Airflow지만 무겁고 복잡하다. 입문자에게는 Prefect가 더 친숙하다.
  • 배울 것: 전문가가 될 필요는 없다. DAG(방향 비순환 그래프)가 무엇인지, 즉 어떤 작업이 다른 작업에 의존하는지를 나타내는 지도를 이해하는 것이 먼저다.

Phase 5: 프로의 마무리 — 관찰 가능성(Observability)

이 마지막 단계가 "지도 없이는 못 움직이는 주니어"와 "혼자서도 굴러가는 엔지니어"를 가른다.

관찰 가능성: 장애의 '이유'를 읽는 능력

누구나 모든 게 완벽할 때 돌아가는 파이프라인은 만들 수 있다. 진짜 엔지니어는 뭔가 잘못됐을 때 정확히 어디서 무엇이 깨졌는지 즉시 알 수 있는 파이프라인을 만든다.

  • 현실: 파이프라인은 반드시 실패한다. 서버가 다운되고, 데이터가 깨진 채 도착한다.
  • 핵심 역량: 시스템을 들여다보고 문제를 즉각 진단하는 능력이 관찰 가능성이다. 대시보드가 비어 있을 때, 원인이 웨어하우스 장애인지, 수집 도구 실패인지, 아니면 소스 데이터 자체가 비어 있는 것인지를 빠르게 파악해야 한다.
  • 로그를 읽고 알림을 설정하는 법을 익히면, 상사가 알아채기 전에 문제를 먼저 해결할 수 있다.

실시간 스트리밍(Kafka/Flink)에 대해

Kafka, Flink 같은 "실시간 데이터" 도구 이야기가 많이 들린다. 하지만 솔직히 말하면, 스트리밍은 복잡하고 비용도 많이 들고 디버깅도 어렵다. 실제로 초 단위 데이터 업데이트가 꼭 필요한 회사는 생각보다 많지 않다. 대부분은 일별 배치 처리로 충분하다.

지금은 여기에 신경 쓰지 말고 견고한 배치 파이프라인을 먼저 완성하라. 그 이후에 스트리밍 세계로 넘어가도 늦지 않다.

마치며: 읽기만 하지 말고, 직접 만들어라

이런 글을 백 편 읽어도 되고, YouTube 튜토리얼을 50시간 볼 수도 있다. 하지만 직접 무언가를 망가뜨리고 고쳐보기 전까지는 진짜로 배우는 게 아니다.

데이터 엔지니어링에서 가장 큰 함정은 "튜토리얼 지옥"이다. 코드 한 줄 짜지 않고 영상만 수동적으로 소비하는 안락한 상태. 비행기 영화를 보는 것만으로 조종사가 될 수 없는 것처럼, 직접 조종석에 앉아야 배운다.

이번 주말 도전 과제

"완벽한" 포트폴리오 프로젝트를 기획하는 일을 멈춰라. 그런 건 존재하지 않는다. 대신 이렇게 해보자.

  1. 데이터 찾기: Kaggle에서 "우버 운행 데이터"나 "IMDb 영화 평점" 같은 단순한 데이터를 내려받는다.
  2. 적재하기: 그 CSV를 BigQuery나 Snowflake에 올린다.
  3. 모델링하기: dbt로 데이터를 정제한다(예: 이동 거리가 음수인 운행 기록 제거).
  4. 시각화하기: Google Looker Studio 같은 도구를 연결해 차트 하나를 만든다.

이 과정을 완수한다면, 이미 이력서에 도구 이름만 나열한 80%의 지원자보다 앞서 있다.

글을 닫고, 코드 에디터를 열어라. 그리고 첫 번째 쿼리를 작성하라. 로드맵은 이것으로 충분하다. 나머지는 당신의 몫이다.
