The Architecture Decision Every Senior .NET Developer Regrets After 3 Years

Kerim Kara · Stackademic · 16 min read · 원문 · freedium

요약

시니어 .NET 개발자들이 3년이 지나서야 후회하는 아키텍처 결정들 — 처음엔 옳아 보였지만 시간이 지나면서 발목을 잡는 선택들에 대한 솔직한 회고록이다.

아키텍처 실수는 대부분 첫날에 드러나지 않는다. 처음에는 잘 돌아가는 것처럼 보이다가, 3년이 지날 무렵 시스템 전체가 무거워진 것을 깨닫게 된다. 이 글은 과도한 추상화, 성급한 마이크로서비스 도입, ORM이 도메인 설계를 지배하도록 방치한 것, Clean Architecture의 맹목적 적용, 비동기 코드의 무분별한 사용 등 경험 많은 .NET 개발자들이 공통으로 꼽는 20가지 후회를 다룬다. 핵심 교훈은 하나다. 아키텍처는 완성하는 것이 아니라 끊임없이 가꿔가는 것이다.

아키텍처 결정이 실패하는 방식은 대부분 요란하지 않다. 첫날 프로덕션을 다운시키거나 긴급 롤백을 유발하지 않는다. 오히려 충분히 잘 동작해서 신뢰를 얻은 뒤, 조용히 이자를 뽑아가기 시작한다. 3년 후에는 여전히 기능을 출시하고 있지만 모든 것이 무겁게 느껴진다. 간단한 변경에도 신중한 조율이 필요하고, 새로 합류한 개발자들은 코드를 어디에 추가해야 할지 갈피를 잡지 못한다. 리팩터링은 도메인이 복잡해서가 아니라 시스템이 취약해졌기 때문에 두렵다.

대부분의 아키텍처 관련 블로그가 다루지 않는 국면이 바로 이 지점이다.

1년 차에 아키텍처는 이상적이다. 패턴은 깔끔하고, 추상화는 정당해 보이며, 다이어그램이 여전히 현실을 반영한다. 2년 차에는 압박이 온다. 마감은 촉박해지고, 팀은 커지며, 타협이 '실용주의'라는 명목으로 슬그머니 끼어든다. 3년 차에 이르러서야 결과가 수면 위로 떠오른다. 단 하나의 실수가 아니라, 모든 것을 느리게 만드는 수천 개의 작은 마찰로.

이 글은 잘못된 아키텍처를 비판하는 것이 아니다. 선한 의도로 내린 결정들, 하지만 실제 사용을 거치면서만 진짜 비용이 드러나는 결정들을 돌아보는 것이다.

1. 너무 이른 과도한 추상화 — 오지 않을 미래를 위한 설계

시니어 .NET 개발자들이 가장 많이 고백하는 후회 중 하나는, 너무 이른 단계에 과도한 추상화를 도입한 것이다. 의도는 거의 항상 좋다. 유연성, 테스트 가능성, 미래 확장성을 바란다. 모든 곳에 인터페이스를 추가하고, 서비스 위에 팩토리를 쌓고, 이미 리포지토리를 구현한 ORM 위에 또 리포지토리를 감싼다. 코드베이스가 '엔터프라이즈스럽게' 느껴지기 시작한다.

처음에는 깔끔하고 인상적으로 보인다. 6개월 후에는 짐이 된다.

문제는 추상화 자체가 아니라 투기적 추상화다. 구체적인 현재 요구사항이 아닌 가상의 미래 요구사항을 위해 설계하면, 이득 없는 간접 참조만 생겨난다. 새 개발자는 요청 하나가 시스템을 통과할 때 어떤 일이 벌어지는지 파악하려면 여러 레이어를 머릿속으로 따라가야 한다. 디버깅은 느려지고, 온보딩은 길어지며, 작은 변경 하나에도 여러 파일을 건드려야 한다. 결국 속도가 떨어진다. 시스템이 복잡해서가 아니라 인위적으로 복잡하게 만들었기 때문에.

// 실제 대안 구현체도 없고 확장 계획도 없는데 인터페이스를 만드는 전형적인 예
public interface IOrderService {
    Task PlaceOrderAsync(OrderDto order);
}

public class OrderService : IOrderService {
    private readonly IOrderRepository _repository;
    public OrderService(IOrderRepository repository) {
        _repository = repository;
    }
    public async Task PlaceOrderAsync(OrderDto order) {
        await _repository.SaveAsync(order);
    }
}

겉보기에는 문제없어 보인다. 하지만 구현체가 하나뿐이고, 대안 동작도 없고, 여러 구현체가 생길 현실적인 가능성도 없다면, 인터페이스는 거의 가치를 제공하지 않는다. 3년 후에 시니어 개발자들은 추상화의 절반이 한 번도 교체되거나 다르게 모킹되거나 확장된 적이 없다는 것을 깨닫는다. 그냥 유지만 됐을 뿐이다.

경험 많은 개발자들은 결국 이것을 배운다. 추상화는 가정하는 것이 아니라 획득하는 것이다. 구체적으로 시작하고, 중복이나 변동성이 자연스럽게 추상화를 요구하도록 두어라.

2. 너무 이른 마이크로서비스 선택 — 아무도 예산에 넣지 않은 분산 시스템 세금

또 다른 빈번한 후회는 마이크로서비스를 섣불리 채택한 것이다. 독립 배포, 확장성, 팀 자율성이라는 약속이 매혹적이다. 하지만 간과하는 것이 있다. 운영 비용과 인지 부하다.

마이크로서비스는 단순한 배포 모델이 아니다. 분산 시스템에 대한 전면적인 헌신이다. .NET 생태계에서는 서비스 디스커버리, 서비스 간 인증, 재시도 로직, 서킷 브레이커, 결과적 일관성, 로그 상관관계, 버전 관리를 일상적으로 다뤄야 한다는 뜻이다.

// 한때 메서드 호출이었던 것이 이제는 네트워크 홉이 된다
var response = await _httpClient.GetAsync("https://inventory/api/stock/123");

이제 실패는 예외가 아니라 타임아웃, 부분 장애, 연쇄 실패가 된다. 3년이 지나면 많은 팀이 깨닫는다. 자신들의 시스템은 사실 독립 확장이나 별도 배포 주기가 필요하지 않았다는 것을. 필요했던 건 모듈 경계였지, 네트워크 경계가 아니었다.

돌이켜 보면 시니어 개발자들은 인정한다. 마이크로서비스를 선택한 것이 잘못이 아니라, 기본값으로 선택한 것이 잘못이었다고. 3년을 되돌릴 수 있다면 모듈형 모놀리스로 시작해서 고통이 강제할 때만 서비스를 분리했을 것이라고.

3. ORM이 아키텍처를 지배하도록 방치하기

Entity Framework Core는 강력한 도구지만, 시니어 .NET 개발자들이 가장 미묘하게 후회하는 것 중 하나는 ORM이 도메인 대신 아키텍처를 형성하도록 내버려 둔 것이다.

초기에는 EF Core 엔티티를 시스템의 핵심으로 삼는 것이 자연스러워 보인다. 쿼리하기 쉽고, 저장하기 쉽고, 노출하기 쉽다. 시간이 지나면 이 편의성이 제약이 된다.

// 처음엔 단순해 보이지만...
public class Order {
    public int Id { get; set; }
    public decimal Total { get; set; }
    public bool IsPaid { get; set; }
}

여기에 비즈니스 로직이 추가되고, 유효성 검사가 붙고, 불변식이 일관성 없이 적용된다. 3년 후에는 지속성 관심사가 동작에 단단히 결합되어 리팩터링이 위험해진다.

EF Core는 조용히 성공하기 때문에 실패가 눈에 띄지 않는다. 너무 쉽게 성공한다. 수년간 쌓인 지름길들이 쌓인 후에야 아키텍처가 변화에 저항적이 된다.

4. 팀 현실을 무시한 Clean Architecture 과잉 투자

Clean Architecture는 종이 위에서는 매력적이다. 실제로는 많은 시니어 .NET 개발자들이 너무 엄격하게 적용한 것을 후회한다. 문제는 Clean Architecture 자체가 아니라 팀 규모, 경험 수준, 제품 성숙도를 무시한 것이다.

엄격한 레이어 분리, 과도한 DTO 매핑, 장황한 use case 클래스는 오버헤드를 추가한다. 소규모나 중간 규모 팀에서는 이 오버헤드가 빠르게 누적된다. 모든 기능을 구현하려면 여러 레이어를 건드려야 하고, 보일러플레이트 코드를 작성해야 하며, 비즈니스 가치가 거의 없는 매핑을 유지해야 한다.

3년 후 팀들은 종종 깨닫는다. 이론적 순수성을 위해 최적화했지 실제 납품을 위해 최적화하지 않았다는 것을. 아키텍처는 팀을 섬겨야 한다. 팀이 아키텍처를 섬기는 것이 아니라.

5. 초기 설계에서 운영 관심사 무시하기

시니어 .NET 개발자들이 가장 뼈아프게 공유하는 후회는 아마 이것이다. 운영을 과소평가한 것. 로깅, 메트릭, 트레이싱, 배포 관심사는 '기능을 먼저 내보내자'는 명목으로 미뤄진다.

3년 후에는 프로덕션 문제 진단이 악몽이 된다. 구조화된 로그도, 상관관계 ID도, 의미 있는 메트릭도 없으면 팀은 맹목적으로 비행하게 된다.

// 처음에는 사소해 보이지만 일관된 관찰가능성 관행이 엄청난 가치를 낸다
_logger.LogInformation("Processing payment for OrderId {OrderId}", orderId);

새벽 3시에 텔레메트리도 없이 프로덕션 장애를 겪어본 사람은 절대 잊지 못한다. 아키텍처는 코드에 관한 것만이 아니다. 시스템이 프로덕션에서 어떻게 동작하는지에 관한 것이기도 하다.

6. 필요하지도 않은데 모든 곳에 비동기 코드 적용하기

거의 모든 시니어 .NET 개발자가 결국 후회하는 결정 중 하나는, 시스템이 실제로 필요하기 훨씬 전에 모든 것을 기본적으로 비동기로 만든 것이다.

// 이름만 비동기인 메서드 — 실제로는 동기 연산
public async Task<bool> IsUserActiveAsync(User user) {
    return await Task.FromResult(user.LastLogin > DateTime.UtcNow.AddDays(-30));
}

이 메서드는 이름만 비동기다. 상태 머신을 추가하고, 스택 트레이스를 읽기 어렵게 만들고, 디버깅을 복잡하게 한다. 더 나쁜 것은 비동기 코드가 미묘한 함정을 만든다. 교착 상태, 컨텍스트 스위칭 비용, 주니어 개발자들이 자주 틀리는 예외 처리 패턴들.

비동기 코드는 실제 동시성 요구사항이 있을 때 써야 한다. 확장성 문제는 존재할 때 해결해야 하지, 미리 상상해서는 안 된다.

7. 설정을 정적이고 부차적인 것으로 취급하기

초기에는 무해해 보이지만 시간이 지나면 고통스러워지는 또 다른 아키텍처 결정은, 설정을 정적이고 부차적인 것으로 취급하는 것이다. 연결 문자열, 기능 플래그, 타임아웃, 임계값들이 하드코딩되거나 느슨하게 관리되어 여러 appsettings.json 파일에 흩어진다.

// 설정이 로직이 되기 시작하면 문제가 생긴다
if (_config.EnableNewPricingLogic) {
    // 새 동작
} else {
    // 레거시 동작
}

3년 후에는 어떤 플래그를 제거해도 안전한지, 어떤 조합이 유효한지 아무도 기억하지 못한다. 설정은 유연성으로 위장한 기술 부채가 된다.

8. 내부 모델 기반으로 API 설계하기

시니어 .NET 개발자들이 후회하는 미묘하지만 파괴적인 실수 중 하나는, 소비자 요구가 아닌 내부 도메인이나 지속성 모델을 기반으로 API를 설계하는 것이다.

[HttpGet("/orders/{id}")]
public Order GetOrder(int id) {
    return _dbContext.Orders.Find(id);
}

이 방식은 외부 소비자를 내부 표현에 단단히 결합시킨다. 도메인이 진화하면 파괴적 변경이 불가피해진다. 3년 후에는 팀이 아무것도 바꾸기 두려워한다. API는 오래된 가정의 냉동 스냅샷이 된다. API는 제품이다. 내부 모델의 거울이 아니다.

9. 테스트 가능성을 위해 비즈니스 변화 적응력을 희생하기

테스트 가능성은 중요하지만, 시니어 .NET 개발자들이 솔직하게 고백하는 후회 중 하나는 명확성과 적응력을 희생하면서까지 테스트 가능성에 과잉 최적화한 것이다.

목킹 프레임워크를 중심으로 아키텍처 전체를 설계하면, 비즈니스 결과보다 구현 세부사항을 검증하는 취약한 테스트가 만들어진다. 3년 후 리팩터링이 위험해진다. 로직이 복잡해서가 아니라 테스트가 깨지기 쉽기 때문에. 테스트는 아키텍처를 따라야 한다. 아키텍처를 지배해서는 안 된다.

10. 지속적인 관리 없이 초기 아키텍처가 잘 늙을 것이라 가정하기

마지막으로 가장 보편적인 후회다. '좋은' 초기 아키텍처가 지속적인 관리 없이도 좋은 상태를 유지할 것이라 가정하는 것.

아키텍처는 한 번의 결정이 아니다. 끊임없는 관리가 필요한 살아있는 시스템이다. 경계가 강제되지 않고, 지름길이 해결되지 않고, 트레이드오프가 재검토되지 않으면 엔트로피가 이긴다.

아키텍처는 완성하는 것이 아니다. 실천하는 것이다.

11. 프레임워크 지식을 아키텍처 역량으로 혼동하기

여러 해가 지나면서 시니어 .NET 개발자들이 깨닫는 것 중 하나는, 깊은 프레임워크 지식이 좋은 아키텍처로 자동 변환되지 않는다는 것이다. ASP.NET Core, EF Core, 의존성 주입, 미들웨어 파이프라인을 마스터한 자신감이 있을 수 있다. 이 지식은 가치 있지만 오해를 부를 수 있다.

프레임워크 중심 아키텍처는 도메인이 요구하는 것이 아닌, 프레임워크가 쉽게 만들어주는 것을 중심으로 시스템을 형성하는 경향이 있다. 프레임워크는 아키텍처의 종이어야 한다. 아키텍트가 되어서는 안 된다.

12. 수직 복잡성 해결 전에 수평 확장성을 위해 최적화하기

많은 시니어 .NET 개발자들이 돌아보면 깨닫는다. 이해하기 전에 규모를 위해 최적화했다는 것을. 수평 확장성은 매혹적이다. 하지만 대부분의 시스템은 수평으로 확장할 수 없어서 실패하지 않는다. 안전하게 변경하기에 너무 복잡해서 실패한다.

수직 명확성이 수평 규모에 선행한다. 추론하기 쉬운 시스템은 확장할 수 있다. 이미 혼란스러운 시스템을 분산하면 지수적으로 더 나빠진다.

13. 데이터베이스 설계를 구현 세부사항으로 취급하기

오래된 후회 중 하나는 데이터베이스의 아키텍처적 역할을 과소평가한 것이다. 스키마, 관계, 트랜잭션 경계에 대한 초기 결정들은 되돌릴 수 있는 것처럼 느껴진다. 실제로는 그렇지 않다.

-- 3년 후 아무도 왜 있는지 모르지만 지우기가 두렵다
ALTER TABLE Orders ADD LegacyFlag BIT DEFAULT 0;

데이터베이스 설계는 아키텍처다. 단지 코드보다 리팩터링하기 느릴 뿐이다.

14. 시스템 장수보다 개발자 경험을 위해 최적화하기

현대 .NET 도구는 개발을 편안하게 만든다. 하지만 개발자 편의성만을 위해 최적화된 아키텍처는 시간이 지나면 노후화된다. 편의성을 추구하되, 한계를 두어라.

15. 비즈니스 생존보다 아키텍처 순수성이 더 중요하다고 믿기

많은 시니어 .NET 개발자들이 결국 깨닫는 철학적 교훈이 있다. 아키텍처 순수성을 너무 높이 평가했다는 것. 현실은 더 혹독한 교훈을 가르친다. 아키텍처는 비즈니스를 섬기기 위해 존재한다. 개발자의 자아를 위해서가 아니라.

3년을 살아남은 시스템은 가장 깔끔한 것들이 아니다. 적응한 것들이다. 지저분한 요구사항을 흡수하고, 급박한 릴리스를 버텨내고, 제품과 함께 진화한 것들이다.

16. 레이어드 아키텍처를 관심사 분리로 혼동하기

시니어 .NET 개발자들이 몇 년 후 도달하는 깨달음 중 하나는, 시스템을 레이어로 나누는 것이 자동으로 관심사 분리를 만들지 않는다는 것이다. 레이어는 도구다. 보장이 아니다. 의도적 경계 없이 레이어드 시스템은 조용히 추가 단계를 가진 절차적 코드로 퇴화한다.

17. '임시' 아키텍처 결정이 영구화되도록 방치하기

모든 경험 있는 .NET 개발자는 '임시'라고 쓴 주석을 달아본 적이 있다. 초기에는 시스템이 작고 관리 가능하기 때문에 임시 결정이 무해하게 느껴진다. 하지만 시간은 지름길을 굳혀버리는 습성이 있다.

// TEMP: 지금은 도메인 유효성 검사를 우회함
await _dbContext.Orders.AddAsync(order);

3년 후 그 우회로는 여전히 존재하고, 여러 기능이 그것에 의존한다. '임시'는 아키텍처에서 가장 위험한 단어 중 하나다.

18. 팀의 인지 부하 비용 무시하기

시니어 .NET 개발자들이 배우는 가장 힘든 교훈 중 하나는, 아키텍처적 복잡성이 기계보다 인간의 주의력에 먼저 세금을 부과한다는 것이다. 더 이해하기 쉬운 시스템이 더 빠르게 움직인다. 덜 정교해 보여도.

19. 조직 변화의 비용 간과하기

아키텍처는 종종 안정적인 팀을 가정하고 설계된다. 현실에서는 사람들이 떠나고, 새 개발자가 합류하고, 팀 구조가 바뀐다. 위대한 아키텍처는 사람의 변화를 이겨낸다. 취약한 아키텍처는 그렇지 않다.

20. 아키텍처가 통제보다 가능화에 관한 것임을 이해하지 못하기

많은 시니어 .NET 개발자들이 인정하는 마지막 철학적 후회는, 아키텍처를 가능화의 메커니즘이 아닌 통제의 메커니즘으로 취급한 것이다.

아키텍처는 변화를 가능하게 하기 위해 존재한다. 제약하기 위해서가 아니라. 살아남는 시스템은 올바른 것을 쉽게 만들고, 잘못된 것을 명확하게 만드는 것들이다. 개발자들을 경직된 경로에 가두는 것들이 아니라.

좋은 아키텍처는 자유를 만든다. 변화할 자유, 적응할 자유, 현실에 반응할 자유를.
