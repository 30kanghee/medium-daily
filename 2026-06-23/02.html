A Step-by-Step Guide for Developing Your Personal Agentic System.

Erdogan T · Data Science Collective · 36 min read · 원문 · freedium

요약

대형 언어 모델을 중심 추론 엔진으로 삼는 LLM 운영체제(LLMos) 개념을 소개하고, 로컬에서 동작하는 Python 라이브러리 LLMlight로 RAG·임베딩·멀티 에이전트 파이프라인을 직접 구축하는 방법을 단계별로 안내한다.

에이전트 시스템이 복잡한 작업 자동화의 표준으로 자리잡고 있다. 이 글은 그 핵심 개념인 LLMos를 소개하는 것으로 시작한다. LLM이 CPU, 토큰이 바이트, 컨텍스트 창이 RAM에 해당하는 새로운 컴퓨팅 패러다임이다. 단일 프롬프트로 복잡한 작업을 처리하려는 '모놀리식 접근'의 한계를 짚고, 작업을 분해해 여러 전문 에이전트가 협력하는 아키텍처가 왜 효과적인지 설명한다. 이어서 RAG 파이프라인의 청킹·임베딩·검색·스코링 전략, 로컬 추론과 전역 추론을 결합하는 방법, LLMlight 라이브러리를 활용한 실습 예제 5개를 상세히 다룬다. 마지막으로 "빠르게 가려 하지 말고, 구조적으로 가라"는 조언으로 마무리한다.

들어가며

에이전트 시스템이 복잡한 작업 자동화의 기본 도구로 자리잡고 있다. 데이터 과학자라면 이 새로운 패러다임을 따라잡아야 한다. 자연어가 연산의 기본 인터페이스가 되는 세상이다. 그런데 막상 들어가보면 핵심 과제가 생성 자체가 아님을 금방 깨닫게 된다. 진짜 문제는 확률적 모델 위에서 신뢰할 수 있고, 구조화되어 있으며, 재현 가능한 시스템을 설계하는 엔지니어링 문제다.

이 글에서는 LLM 운영체제(LLMos)라는 새로운 개념, 신뢰할 수 있는 에이전트 파이프라인을 만드는 실제 설계 방법, 그리고 Python 라이브러리 LLMlight를 활용해 로컬에서 직접 멀티 에이전트 시스템을 구축하는 실습 예제를 다룬다.

에이전트 시스템 입문

ChatGPT, Gemini, Grok 같은 채팅 인터페이스를 넘어서서 언어 모델을 중심으로 시스템을 직접 만들어보면 금방 알게 된다. 어려운 것은 텍스트를 생성하는 일이 아니다. 실제 문제를 구조적이고 재현 가능하게 해결하는 워크플로를 설계하는 일이다.

전통적인 결정론적 시스템은 배포하고 나면 규칙이 바뀌지 않는 한 예측 가능하게 동작한다. 언어 모델은 다르다. 프롬프트, 모델 버전, 온도(temperature) 설정, 컨텍스트 창, 심지어 단어 표현의 미묘한 차이에 따라 출력이 달라진다. 이 유연성이 언어 모델을 강력하게 만드는 동시에, 프로덕션 환경에서 제어하기 어렵게 만드는 요인이기도 하다.

스스로 만들어보는 것이 중요하다. 직접 구축하는 과정에서 에이전트 시스템이 어떻게 근본적으로 작동하는지 훨씬 깊이 이해할 수 있고, 그래야 이런 워크플로의 가능성과 한계를 제대로 볼 수 있다.

핵심 개념 1: LLM은 작가가 아니라 CPU다

우리는 지금 새로운 컴퓨팅 패러다임에 들어서고 있다. 대형 언어 모델이 시스템의 연산 핵심이 되는 세상이다. Andrej Karpathy가 처음 명명한 LLM 운영체제(LLMos)는 언어 모델이 중앙 추론 엔진으로서 여러 작업, 도구, 에이전트를 조율하는 아키텍처 설계 방식이다.

전통적인 컴퓨팅과의 유사성은 놀라울 정도로 직관적이다.

  • CPU → 대형 언어 모델(LLM)
  • 바이트 → 토큰
  • RAM → 컨텍스트 창

결정론적 명령을 바이트 단위로 실행하는 대신, 이 시스템은 이제 토큰을 확률적으로 처리한다. 컨텍스트 창은 임시 작업 메모리 역할을 하고, 검색 시스템은 외부 메모리 저장소 역할을 맡는다. 프롬프트는 점점 컴포넌트 간 소프트웨어 인터페이스처럼 기능한다.

복잡한 작업을 단일 프롬프트로 처리하려고 하면 곧 한계에 부딪힌다. 정보 검색, 문서 추론, 구조 검증, 피드백 제공, 일관성 유지, 최종 응답 생성이 동시에 요구되는데, 이 모든 것을 하나의 거대한 프롬프트에 우겨넣으면 결과는 지저분하고 신뢰하기 어렵다. 에이전트 시스템이 여기서 등장한다.

핵심 개념 2: 더 많은 파라미터가 아니라 전문화된 에이전트 팀이 필요하다

에이전트 시스템을 구축한다는 것은 단순해 보이지만, 완전히 새로운 엔지니어링 과제를 안겨준다.

  • 에이전트들이 어떻게 소통하는가?
  • 어떤 작업은 결정론적으로, 어떤 작업은 확률적으로 처리해야 하는가?
  • 제한된 컨텍스트 창을 어떻게 관리하는가?
  • 관련 정보를 어떻게 검색하고 통계적으로 검증하는가?
  • 환각(hallucination)을 어떻게 방지하는가?
  • 일관된 채점과 평가를 어떻게 유지하는가?

실습 경험을 통해 깨닫게 되는 것이 있다. 에이전트 시스템에서 성공은 단 하나의 영리한 프롬프트에서 오지 않는다. 좋은 시스템은 신중하게 설계된 파이프라인의 결과다. 청킹 전략, 검색 시스템, 임베딩 모델, 랭킹 메커니즘, 구조화된 프롬프트, 스코링 워크플로, 그리고 함께 작동하는 전문화된 에이전트들이 모두 맞물려야 한다.

단일 프롬프트가 실패하고 에이전트 시스템이 작동하는 이유

대형 언어 모델 채팅 인터페이스를 쓰다 보면 유혹이 생긴다. 문서 전체를 LLM에 넣고 한 번에 평가하게 하면 되지 않을까. 이 '모놀리식' 접근법에는 복잡하고 큰 작업을 처리할 때 네 가지 근본적인 한계가 있다.

컨텍스트 창 포화: 모델이 20만 토큰의 컨텍스트 창을 지원하더라도, 긴 문서에 시스템 지시, 평가 기준, 엄격한 출력 형식 스키마를 함께 넣으면 실제 한계에 금방 부딪힌다. 결과적으로 문서의 중요한 부분이 암묵적으로 압축되거나 무시돼 불완전한 추론으로 이어진다. LLMos와 PC의 유추에서 컨텍스트 창은 RAM에 해당한다. 한계를 넘으면 오류가 발생한다.

작업 뒤얽힘: 단일 프롬프트는 모델이 근본적으로 다른 인지 작업을 동시에 수행하게 강제한다. 관련 데이터 추출, 맥락 해석, 미묘한 규칙 적용, 단면 간 일관성 유지, 구조화된 피드백 생성. 이런 작업들을 명시적으로 분리하는 메커니즘 없이는 모델이 종종 피상적이거나 자기 모순적인 출력을 만들어낸다.

프롬프트 불안정성: 단일 패스 아키텍처는 구조적 변화에 매우 민감하다. 표현의 작은 변화가 같은 문서를 평가할 때도 출력 품질을 크게 바꿀 수 있다.

구조 부재: 단일 프롬프트는 단일 출력을 낸다. 체크포인트도 없고, 추론 단계도 없다. 최종 답변이 틀렸을 때 추론 사슬 어디에서 틀렸는지 추적할 방법이 없다. 투명하고 신뢰할 수 있는 시스템을 만들기가 극히 어렵다.

복잡한 작업은 본질적으로 반복적이다. 읽고, 교차 확인하고, 검증하고, 판단을 다듬는 과정이 필요하다. 이 모든 단계를 하나의 변환으로 압축하면, 우리 인간 평가자가 자연스럽게 의존하는 구조가 사라진다.

에이전트 아키텍처란 무엇인가

실제로 대부분의 에이전트 시스템은 여러 전문화된 LLM 모델이 각자 특정 역할을 맡아 공동 목표를 향해 협력하는 구조화된 파이프라인이다. 하나의 LLM을 쓰면 그것은 그냥 모델이다. 여러 LLM이 공동 목표를 향해 작동하면 에이전트 시스템이라고 말할 수 있다.

LLM 에이전트에 대한 흔한 오해가 있다. 에이전트 자체가 근본적으로 지능적인 존재라는 것. 사실 에이전트는 특정 지시와 미리 정의된 작업을 가진 단일한 (작고 로컬에서 실행되는) LLM 모델일 뿐이다.

에이전트 시스템의 아이디어는 단일 프롬프트로 모든 것을 해결하려는 대신, 문제를 더 작은 하위 작업으로 쪼개고 각 에이전트에게 미리 정해진 책임을 부여해 제한된 컨텍스트 안에서만 작동하게 하는 것이다. 가장 흔한 에이전트 유형은 다음과 같다.

  • A1 검색 에이전트: 문서에서 관련 섹션을 찾는다
  • A2 추론 에이전트: 검색된 맥락을 분석한다
  • A3 요약 에이전트: 검색된 맥락을 요약한다
  • A4 평가 에이전트: 완전성이나 품질을 검증한다
  • A5 스코링 에이전트: 신뢰도나 순위를 부여한다
  • A6 조율 에이전트: 최종 응답을 통합하고, 작업이 완료됐는지 또는 다음 라운드가 필요한지 결정한다

이런 분해 방식은 몇 가지 이점을 만들어낸다. 모델의 인지 부담이 줄어든다. 중간 체크포인트가 생기므로 실패를 디버깅하고 검증하기 쉬워진다. 개별 에이전트를 전체 시스템을 다시 설계하지 않고도 독립적으로 교체하거나 업그레이드할 수 있다. 일부 에이전트는 결정론적 검색이나 규칙 기반 로직으로 작동하고, 다른 에이전트는 LLM으로 확률적 추론을 수행하는 하이브리드 전략도 가능해진다.

RAG 파이프라인

대형 언어 모델의 핵심 한계 중 하나는 긴 문서에서 어떤 부분이 특정 질문과 관련 있는지 본질적으로 '알지 못한다'는 것이다. 책이나 보고서, 데이터베이스 전체를 모델에 직접 넣는 방식은 제한된 컨텍스트 창 때문에 비효율적이고 비용이 많이 들며 신뢰하기 어렵다.

검색 증강 생성(RAG)은 생성 전에 외부 검색 레이어를 도입해 이 문제를 해결한다. 전체 문서를 모델에 보내는 대신, 시스템이 먼저 가장 관련성 높은 텍스트 조각을 찾아서 그 조각들만 LLM의 컨텍스트 입력으로 제공한다.

RAG를 이해하는 직관적인 방법은 인간 기억과의 유사성이다. 뇌가 수천 개의 작고 검색 가능한 기억 조각으로 나뉘어 있다고 상상해보자. "내 열쇠가 어디 있지?"라고 자문할 때, 전체 인생사를 처음부터 재생하지 않는다. 집에 도착했을 때, 테이블 위에 뭔가를 놓았을 때, 재킷을 집었을 때처럼 관련성 있어 보이는 기억만 무의식적으로 떠올린다. RAG가 이렇게 작동한다.

대형 문서를 '청크'라는 작은 조각들로 나눈다. 질문이 들어오면 시스템이 청크들을 검색해 가장 관련성 높은 것들만 가져온다. 이 상위 조각들이 모델의 제한된 컨텍스트 창, 즉 임시 작업 메모리에 주입된다. 그러면 언어 모델은 전체 정보를 보지 않고도 그 조각들을 엮어 일관된 답변을 만들어내는 데 매우 능숙하다.

RAG 파이프라인의 네 단계는 다음과 같다.

청킹 전략: 원본 문서를 더 작고 일관된 세그먼트로 쪼개는 방법을 정의한다. 청킹의 단위는 단어, 문장, 단락, 토큰, 의미 섹션 등 다양하다.

임베딩 전략: 각 청크를 수치 표현, 즉 임베딩으로 변환한다. 임베딩 모델은 텍스트를 고차원 벡터로 변환해서 의미적으로 유사한 텍스트 조각들이 수학적으로 서로 가까이 위치하게 만든다. 이를 통해 시스템이 정확한 키워드 매칭이 아닌 의미에 기반해 문서를 검색할 수 있다. 옵션으로는 TF-IDF, Word2Vec, Sentence-BERT, 최신 OpenAI 임베딩 등이 있다.

검색 전략: 사용자가 쿼리를 제출하면 쿼리 자체도 벡터 공간에 임베딩된다. 시스템이 쿼리 임베딩과 모든 문서 청크 사이의 유사도를 계산한다.

스코링 전략: 유사도 점수가 가장 높은 청크가 질문에 가장 관련성 있다고 간주된다. 코사인 유사도가 가장 흔히 쓰인다.

RAG가 잘 작동하는 작업과 한계

이 접근법은 룩업 방식의 질문에서 매우 잘 작동한다.

  • "이 문서에 모델 비교가 있는가?"
  • "방법론 섹션이 있는가?"
  • "어떤 데이터셋이 사용됐는가?"

그러나 어떤 질문들은 특정 문장이나 단락을 찾는 것이 아니라 문서 전체를 이해하는 것을 필요로 한다.

  • "이 제안서는 충분히 고민된 내용인가?"
  • "논거가 일관성 있는가?"
  • "방법론이 논리적으로 따라올 수 있는가?"

이런 질문들에서 RAG의 한계가 드러난다. 지역적 관련성에 최적화되어 있고 전역적 관련성에는 최적화되어 있지 않다.

전역 추론 워크플로

이 한계를 해결하기 위해 많은 실제 에이전트 시스템이 최종 생성 전에 추가 추론 레이어를 도입한다. 2단계 전역 추론 전략이다.

검색된 청크에서 바로 질문에 답하는 대신, 시스템이 먼저 문서의 중간 전역 표현을 만든다. 단순화된 워크플로는 다음과 같다.

  1. 문서를 청크로 나눈다
  2. 각 청크를 독립적으로 분석한다
  3. 로컬 요약이나 관찰 사항을 생성한다
  4. 중간 요약들을 집계한다
  5. 문서의 전역 표현을 구성한다
  6. 집계된 맥락으로 원래 질문에 답한다

이렇게 하면 워크플로가 검색→생성에서 분석→요약→집계→추론→생성으로 바뀐다. 이 차이는 미묘해 보이지만 아키텍처적으로는 매우 중요하다. 모델이 더 이상 모든 추론을 하나의 생성 단계에 압축할 필요가 없다. 대신 시스템이 문서에 대해 점진적으로 더 높은 수준의 추상화를 구축해나간다.

실제로 로컬 검색 파이프라인과 전역 추론 워크플로를 함께 사용하는 것이 가장 좋다. 결정론적 질문에는 로컬 검색을, 추상적이거나 전체론적 평가에는 전역 추론을 적용한다.

소형 언어 모델이 팀으로서 이기고 있다

올해 LLM 분야에서 가장 중요한 흐름 중 하나는 단일 모놀리식 거대 언어 모델에서 여러 전문화된 에이전트로 구성된 에이전트 AI 시스템으로의 이동이다. NVIDIA 연구팀은 "소형 언어 모델이 에이전트 AI의 미래"라고 주장했다. Microsoft는 메모리 요구량을 대폭 줄여 로컬 배포를 점점 더 실현 가능하게 만드는 1비트 LLM을 선보였다.

Gemma 4-26B-a4B 같은 모델이 이 트렌드를 잘 보여준다. 26억 개의 파라미터를 가지지만 추론 시에는 약 4억 개만 활성화된다. 일반적인 밀집 모델이 아니라 혼합 전문가(Mixture-of-Experts, MoE) 아키텍처다. 에이전트 AI와 결합하면 단일 모델에 의존할 필요 없이 각 작업에 특화된 여러 모델을 만들어 함께 작업을 해결할 수 있다.

LM Studio로 로컬 모델 설정하기

LLM과 에이전트 시스템을 실험하려면 먼저 로컬 모델을 설정해야 한다. LM Studio는 가장 널리 쓰이는 소프트웨어다. 무료이고 사용자 인터페이스가 있으며, 컴퓨터 하드웨어와 놀라울 정도로 잘 연동된다. 인기 있는 모델을 쉽게 다운로드해 바로 사용할 수 있다. Ollama나 Claude 같은 대안도 있다. 툴에 엔드포인트가 있으면 로컬 Python 환경에서 언어 모델을 사용할 수 있다.

LLMlight 라이브러리

LLMlight는 최소한의 의존성으로 로컬에서 언어 모델을 실행하기 위한 사용하기 쉽고 가벼운 Python 패키지다. 입력 프롬프트 처리부터 모델 출력까지 다양한 단계를 갖춘 에이전트 파이프라인이다. 파이프라인의 핵심 6단계는 텍스트 청킹, 검색 전략, 임베딩 전략, 스코링 전략, 컨텍스트 전략, 프롬프팅 전략이다.

청킹 전략

LLMlight는 큰 문서를 작고 일관된 세그먼트로 나누는 모듈식이고 설정 가능한 청킹 전략을 사용한다. 기본 설정은 {'method': 'chars', 'size': 1000, 'overlap': 200}이다. 청크가 너무 작으면(200자 미만) 완전한 정보를 담지 못해 환각을 유발할 수 있어 강력히 경고한다.

검색 전략 — 로컬 데이터베이스

LLMlight는 SQLite와 HNSW를 결합해 가볍고 사용하기 쉬운 검색을 지원한다. 벡터 데이터베이스가 빠르고 강력하지만 복잡한 설치 절차가 필요한 데 반해, LLMlight는 로컬 SQLite 데이터베이스에 선택적 HNSW 인덱스를 결합한 기본 백엔드를 제공한다.

임베딩 및 스코링 전략

임베딩 전략은 청크와 쿼리가 벡터 공간에서 어떻게 표현될지 결정한다. LLMlight에서 사용 가능한 임베딩 방법은 TF-IDF(구조화된 문서에 적합), Bag of Words(단순 단어 빈도), BERT(자유 형식 텍스트를 위한 고급 문맥 임베딩), BGE-small(일반 사용을 위한 효율적인 임베딩 모델)이다. 스코링 전략은 상위 청크를 반환하기 위해 후보 청크에 관련성 점수를 부여한다. 기본값은 코사인 유사도다.

컨텍스트 전략

LLMlight의 컨텍스트 전략에는 두 에이전트가 함께 작동해 모든 입력 청크를 바탕으로 단일 포괄적인 텍스트를 만든다. 청크 분석기 에이전트는 각 청크와 다른 청크와의 관계를 분석하고, 스마트 컨텍스트 에이전트는 처리된 청크를 특정 지시에 따라 하나의 일관된 텍스트로 요약한다. 세 가지 컨텍스트 전략이 있다. None(처리 없음), chunk-wise(청크 단위 처리), global-reasoning(전역 추론)이다.

실습 예제 1: 단일 모델 로드 및 간단한 대화

# 라이브러리 설치
pip install llmlight
from LLMlight import LLMlight

# LLMlight 클라이언트 초기화
client = LLMlight(model='google/gemma-4-26b-a4b-qat')

# 질문하기
response = client.prompt('What is the capital of France?')
print(response)

# [LLMlight.LLM] [INFO ] Model : google/gemma-4-26b-a4b-qat
# [LLMlight.LLM] [INFO ] Context strategy : disabled
# [LLMlight.LLM] [INFO ] Retrieval method : naive_rag
# ...
# The capital of France is Paris.

실습 예제 2: 로컬 지식 베이스 만들기

로컬 지식 베이스는 로컬 검증된 사실로 추론하기 위한 데이터베이스로서 프라이빗 에이전트 시스템 구축의 결정적인 첫 단계다. LLMlight로 PDF, 텍스트 파일, 심지어 디렉토리 전체를 memory_add 함수로 쉽게 수집할 수 있다. MemVid는 특히 혁신적인데, 수백만 개의 텍스트 임베딩을 단일 MP4 비디오 파일로 압축해 전체 지식 베이스를 오프라인에서 사용하기 쉽고 엣지 장치 간 이동이 용이하게 만든다.

from LLMlight import LLMlight

# 모델과 메모리 초기화
client = LLMlight(model='google/gemma-4-26b-a4b-qat')
client.memory_init(store_path='knowledge_base.db')

# PDF 파일을 데이터베이스에 추가
url = 'https://proceedings.neurips.cc/paper_files/paper/2017/file/3f5ee243547dee91fbd053c1c4a845aa-Paper.pdf'
pdf_text = client.read_pdf(url)
client.memory_add(text=pdf_text)
client.memory_save()

# 새로운 지식에 쿼리
response = client.prompt(
    query='What are attention networks?',
    response_format='Summarize in 3 sentences.'
)
print(response)

실습 예제 3: 컨텍스트 전략별 출력 차이

LLMlight 라이브러리의 세 가지 컨텍스트 전략, 즉 None, chunk-wise, global-reasoning이 결과에 어떤 차이를 만드는지 직접 실험해볼 수 있다.

  • None: 검색된 원본 청크를 그대로 사용. 가장 단순하지만 중복이 많거나 맥락이 조각날 수 있다.
  • chunk-wise: 청크 분석기 에이전트가 각 청크를 처리한 후 스마트 컨텍스트 에이전트가 일관된 텍스트로 통합한다.
  • global-reasoning: 전역 문서 요약을 먼저 만들고 그 위에서 최종 답변을 생성한다. 전체적 이해가 필요한 질문에 가장 효과적이다.

실전 조언: 언어 모델 도입 전에 생각할 것들

실습 예제들을 통해 에이전트 시스템을 어떻게 설계하는지 경험했다면, 도입에 앞서 몇 가지 중요한 원칙을 기억할 필요가 있다.

  • 모든 솔루션에 언어 모델이 필요한 것은 아니다. 단순한 솔루션이 원하는 결과를 주지 않을 때만 복잡한 모델을 고려하라. 조회 목록, 정규 표현식, TF-IDF부터 시작하라. 시간, 복잡성, 유지보수 비용, 좌절감을 아낄 수 있다.
  • 언어 모델은 특정 사용 사례에 맞게 최적화해야 한다. 잘 작동하는 모델을 만들려면 파라미터 최적화가 필요하다. 컨텍스트 전략, 검색 방법, 임베딩, 온도, 모델을 신중하게 선택하라.
  • 청킹은 너무 작게 하지 마라(200자 미만). 작은 청크는 완전한 정보를 담지 못해 환각을 유발한다.
  • 헤더, 바닥글, 반복 텍스트를 제거하라. 이런 것들은 시스템에 불필요한 노이즈를 만들어 모델 성능을 저하시킨다.
  • 임베딩에 대해 항상 비판적으로 생각하라. 일부 임베딩은 의미적 유사도를 위해 훈련됐지만, QA 시스템에서는 TF-IDF가 특정 도메인에서 BERT 같은 의미 임베딩을 능가할 수 있다.
  • 각 언어 모델에는 가드레일이 있어야 한다. 사용 사례에 따라 모델이 인용을 제공하거나 정보를 찾은 청크를 표시하도록 강제하라.

마무리: 빠르게 가지 말고, 구조적으로 가라

올해는 '컨텍스트 엔지니어링'의 해다. 성공적인 LLM 통합은 빠른 배포가 아니라 기반이 되는 인지 아키텍처에 달려 있다. 결정론적 바이트에서 확률론적 토큰으로 전환하는 지금, 전통적인 운영체제에 적용하던 것과 같은 엄격함으로 시스템을 관리해야 한다.

간혹 자신의 논리를 환각으로 만들어낼 수 있는 '운영체제'를 관리할 준비가 됐는가? 모듈식이고, 로컬에서 실행되며, 통계적으로 검증된 인프라를 구축하는 것이 그 답이라고 생각한다. 작게 시작하고, 나만의 멀티 에이전트 파이프라인을 직접 만들고, 모델을 작가로 취급하는 것을 멈춰라. 모델을 새로운 운영체제의 핵심으로 대하라.
