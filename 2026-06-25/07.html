Everyone Said AI Would Replace Developers. GitHub's Numbers Show the Opposite.

The Latency Gambler · 4분 · 원문 · freedium

요약

AI가 개발자를 대체할 거라는 예측과 달리, GitHub의 실제 수치는 커밋·PR·저장소 수가 역사상 가장 빠른 속도로 늘고 있음을 보여준다.

NVIDIA CEO 젠슨 황이 공개한 GitHub 통계에 따르면 2026년 월간 PR은 9천만 건, 커밋은 14억 건, 신규 저장소는 2천만 개에 달한다. 2023년 대비 3년도 안 되어 커밋 수가 3배 가까이 증가한 것이다. 이 추세는 AI가 개발자를 밀어내는 것이 아니라, AI 덕분에 더 많은 사람이 더 많은 것을 만들 수 있게 됐음을 의미한다. 스프레드시트가 회계사를 없애지 않고 오히려 회계사가 다룰 수 있는 문제의 범위를 넓혔듯, AI 도구는 개발자 한 명이 이전에 소팀이 해야 했던 일을 혼자 처리할 수 있게 만들었다. 결국 커밋 수 급증은 AI가 코드를 쓰는 이야기가 아니라, 인간이 역사상 가장 많은 것을 만들어내고 있다는 이야기다.

대체론을 뒤집는 숫자들

AI가 개발자를 대체할 거라는 이야기는 2년째 끊이지 않았다. Copilot이 코드를 쓰고, 에이전트가 테스트를 돌리고, Cursor가 리팩터링을 처리한다. 1~2년만 지나면 개발자가 하는 일 대부분을 개발자 없이 할 수 있게 될 거라는 목소리였다.

그런데 NVIDIA의 젠슨 황이 최근 키노트에서 차트 하나를 띄웠고, 데이터는 전혀 다른 이야기를 했다. GitHub는 현재 월 9천만 건의 PR 병합, 14억 건의 커밋, 그리고 매달 2천만 개의 신규 저장소 생성을 처리하고 있다. 플랫폼 활동이 쪼그라드는 곳의 숫자가 아니다. 역사상 가장 빠른 성장을 기록 중인 플랫폼의 숫자다.

GitHub 연간 커밋 추이를 보면 흐름이 더 분명해진다. 2023년 약 3억 건이던 커밋은 2024년 4억 건, 2025년 5억 건으로 꾸준히 올랐다. 그런데 2026년 초 몇 달치를 연간으로 환산하면 약 14억 건에 달한다. 2023년에서 3년도 채 안 되는 사이에 거의 3배가 됐고, 속도는 줄기는커녕 오히려 붙고 있다.

AI가 개발자의 필요성을 줄이고 있었다면, 활동이 정체되거나 감소했을 것이다. 저장소가 줄고, 커밋이 줄고, PR이 줄었을 것이다. 모든 지표에서 정반대가 벌어지고 있다.

이게 실제로 의미하는 것

가장 가까운 비유는 스프레드시트가 회계에 끼친 영향이다. Excel 이전에 재무 분석은 느리고, 인력 의존적이었으며, 비쌌다. 스프레드시트는 회계사를 없애지 않았다. 회계사가 다룰 수 있는 문제의 천장을 없앴다. 3명짜리 팀이 예전에는 15명이 해야 했던 일을 처리할 수 있게 됐고, 이전에는 엄두도 못 낼 문제에 도전할 수 있게 됐다.

소프트웨어 개발에서도 같은 역학이 펼쳐지고 있다. AI 도구 이전에는 아이디어에서 프로토타입까지 몇 주가 걸렸고, MVP를 만들려면 최소 4~6명의 팀이 필요했다. 지금은 개발자 한 명이 Claude Code나 Cursor, Copilot을 가지고 하루 이틀 만에 프로토타입을 완성하고, 혼자 또는 2인 팀으로 수주 안에 MVP를 낼 수 있다. 그 결과 더 많은 아이디어가 코드가 되고, 저장소가 늘고, 커밋이 늘어난다.

노트북에만 머물던 아이디어가 프로토타입이 되고 있다. 주말 사이드 프로젝트가 실제 제품으로 출시되고 있다. 개념에서 코드로 가는 진입 장벽이 드라마틱하게 낮아졌고, GitHub 수치는 그 직접적인 증거다.

개발자의 하루가 바뀐 방식

AI가 코드를 쓰고 개발자는 구경하는 것이 아니다. 변화의 핵심은 개발자가 더 높은 추상화 수준에서 일하게 됐다는 데 있다. AI가 구현 세부사항을 더 많이 맡으면서, 개발자는 어떤 것이 "좋은 코드"인지 판단하는 일에 집중할 수 있게 됐다.

예를 들어 S3에 파일을 올리는 코드를 보자. 예전에는 리서치하고, 직접 작성하고, 테스트하는 데 45분이 걸렸다. 기본적인 에러 처리 정도만 넣은 채로.

# AI 도구 이전: 직접 작성
import boto3

def upload_to_s3(file_path: str, bucket: str, key: str) -> bool:
    s3 = boto3.client("s3")
    try:
        s3.upload_file(file_path, bucket, key)
        return True
    except Exception as e:
        print(f"Upload failed: {e}")
        return False

AI 도구를 쓰면 "재시도 로직, 지수 백오프, 구조화된 로깅을 갖춘 S3 업로드 함수"라고 의도를 기술하고, 결과를 검토해서 통합하는 데 3분이면 된다.

# AI 도구 활용: 의도를 기술하고 결과를 검토
import boto3
import time
import logging
from botocore.exceptions import ClientError

logger = logging.getLogger(__name__)

def upload_to_s3(
    file_path: str,
    bucket: str,
    key: str,
    max_retries: int = 3
) -> bool:
    s3 = boto3.client("s3")
    for attempt in range(max_retries):
        try:
            s3.upload_file(file_path, bucket, key)
            logger.info("Uploaded %s to s3://%s/%s", file_path, bucket, key)
            return True
        except ClientError as e:
            wait = 2 ** attempt
            logger.warning("Attempt %d failed, retrying in %ds: %s", attempt + 1, wait, e)
            time.sleep(wait)
    logger.error("All %d upload attempts failed", max_retries)
    return False

두 번째 코드가 더 낫다. 그리고 만드는 데 시간이 덜 걸렸다. 개발자의 기여는 "좋은 코드"가 어떤 모습이어야 하는지 판단하는 것이었다. 그 판단은 여전히 온전히 인간의 몫이다. 그리고 시간이 줄었기 때문에, 같은 개발자가 예전에 두 명이 필요했던 일을 혼자 처리할 수 있게 됐다.

생산성 스택에 새 층이 추가됐다

모든 세대의 도구는 개발자를 더 유능하게 만드는 추상화 층을 하나씩 쌓아왔다. 개발자를 쓸모없게 만든 게 아니라.

2026년 개발자 생산성 스택을 보면, 하드웨어·프로세스·네트워킹을 다루는 OS/인프라 층 위에 메모리 관리·타입 안전성·컴파일을 맡는 언어 층, 패턴·관례·스캐폴딩을 처리하는 프레임워크 층이 있다. 그리고 이제 맨 위에 AI 층(Claude Code, Cursor, Copilot, Devin)이 올라앉아 구현 세부사항·보일러플레이트·리팩터링을 담당한다. 개발자는 아키텍처, 제품 판단, 시스템 설계, 코드 리뷰, 그리고 특정 문제에서 "좋다"는 게 무엇인지 이해하는 일에서 활동한다.

새 층이 추가될 때마다 비관론이 따라왔다. 고수준 언어가 어셈블리 프로그래머를 없앨 거라고 했다. IDE가 코드를 이해할 필요성을 없앨 거라고 했다. 프레임워크가 백엔드 엔지니어를 없앨 거라고 했다. 어느 것도 현실이 되지 않았다. 각 층이 가져온 생산성 향상이 만들어질 수 있는 것의 영역을 넓혔고, 그것을 이끌 수 있는 인간에 대한 수요를 더 키웠기 때문이다.

차트가 실제로 말하는 것

GitHub의 수치는 AI에도 불구하고 개발자가 안전하다는 증거가 아니다. AI 덕분에 더 많은 개발자, 더 많은 프로젝트, 역사상 그 어느 때보다 많은 코드가 만들어지고 있다는 증거다.

개발자 한 명이 이제 예전에 소팀이 필요했던 것을 만들 수 있다. 더 많은 사람이 실현 가능하게 만들기 시작할 수 있다는 뜻이다. 더 작은 그룹이 더 많은 문제에 달려들 수 있다. 더 많은 소프트웨어가 작성되고, 출시되고, 유지된다.

몇 달 만에 커밋 수가 거의 세 배로 뛴 것은 AI가 코드를 쓰는 이야기가 아니다. 도구가 드디어 그것을 가능하게 만들어줬기 때문에, 인간이 그 어느 때보다 많은 것을 만들고 있다는 이야기다. 그것은 직업에 대한 위협이 아니다. 직업이 확장되고 있는 것이다.

GitHub 통계 출처: 젠슨 황의 키노트 발표, 2026년 4월. 모든 수치는 공개적으로 보고된 플랫폼 지표다.
