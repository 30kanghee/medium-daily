Apache Iceberg for Data Engineers

MHAMDI Chiheb · 9 min read · 원문 · freedium

요약

Apache Iceberg는 데이터 레이크 위에 카탈로그 계층을 추가해 ACID 트랜잭션, 타임 트래블, 스키마 진화를 가능하게 하는 오픈 테이블 포맷이다.

S3에 Parquet 파일을 그냥 쌓는 방식은 부분 쓰기 실패, 동시 접근 충돌, 과거 데이터 재현 불가 등 고질적인 문제를 안고 있다. Iceberg는 쿼리 엔진이 아니라 테이블 포맷 명세로, 스냅샷·매니페스트·카탈로그로 구성된 메타데이터 계층을 통해 이 문제를 해결한다. MERGE INTO로 CDC 파이프라인을 구현하고, TIMESTAMP AS OF로 과거 시점 데이터를 조회하며, 데이터를 다시 쓰지 않고도 파티션 방식을 변경할 수 있다. Spark, Trino, Flink 등 여러 엔진이 동일 테이블을 공유할 수 있고, AWS Glue·Nessie·REST 카탈로그와 연동된다. 운영 관점에서는 소형 파일 컴팩션과 스냅샷 만료 정책이 필수이며, 이를 소홀히 하면 메타데이터 비대화로 성능이 급격히 저하된다.

1. Iceberg가 해결하는 문제

창고에 수천 권의 책을 분류 없이 쌓아둔 도서관을 상상해보자. "2023년 출간된 추리 소설"을 찾으려면 상자를 일일이 열어 레이블을 확인해야 한다. 그 사이 누군가 책을 추가라도 하면 집계 결과가 틀려진다.

데이터 레이크도 마찬가지다. S3에 Parquet 파일만 쌓아두면 안전한 업데이트, 효율적인 쿼리, 일관된 상태 유지 중 어느 것도 보장되지 않는다. Iceberg는 이 파일들 위에 신뢰할 수 있는 카탈로그 계층을 얹어 이 세 가지 문제를 한꺼번에 해결한다.

Iceberg 없이 데이터를 관리하면 어떤 일이 벌어지는지 한 예를 보자.

s3://lake/orders/
  year=2023/month=06/part-00001.parquet
  year=2023/month=06/part-00002.parquet
  year=2024/month=01/part-00001.parquet   ← job failed mid-write, corrupt state

이 상태에서는 다음 질문에 깔끔하게 답할 수 없다.

  • 현재 테이블을 구성하는 파일이 정확히 어떤 것들인가?
  • 어제 이 테이블은 어떤 모습이었는가?
  • 두 작업자가 동시에 테이블을 수정해도 데이터가 깨지지 않는가?

Iceberg는 이 세 질문 모두에 답한다.

"그냥 Parquet on S3"의 한계

Iceberg의 접근 방식은 다르다. 테이블을 "폴더 안에 있는 파일들의 집합"이 아니라, 메타데이터가 관리하는 버전화된 파일 집합으로 취급한다.

실무에서 Iceberg를 도입하는 시점

다음 상황에 해당한다면 Iceberg 도입을 진지하게 검토해야 한다.

  • Spark, Trino, Flink 등 여러 엔진이 동일 테이블을 공유해야 할 때
  • 데이터 레이크에서 MERGE INTO, DELETE, UPDATE가 필요할 때
  • 재무·컴플라이언스 팀에서 "3월 1일 기준 보고서를 그대로 재현해달라"고 요청할 때
  • Hive 스타일의 파티션 디렉터리 구조가 페타바이트 규모에서 감당이 안 될 때

반면 아직 Iceberg가 필요 없을 수 있는 경우도 있다.

  • 데이터 규모가 작고, 단일 작성자에 append-only 패턴, 단일 엔진만 사용할 때
  • BigQuery나 Snowflake 네이티브 테이블만 쓰고 오픈 레이크 스토리지는 전혀 사용하지 않을 때

2. Iceberg의 본질

MP3는 파일 포맷이고, Spotify는 그 포맷을 재생하는 앱이다. Iceberg도 마찬가지다. Iceberg는 포맷이지, Spark도 S3도 쿼리 엔진도 아니다.

Iceberg는 오픈 테이블 포맷 명세다. 데이터 파일과 메타데이터를 구성하는 표준 방식을 정의하여, 호환되는 어떤 엔진이든 동일 테이블을 읽고 쓸 수 있게 한다.

Iceberg가 아닌 것들을 명확히 해두자.

전체 스택은 다음과 같이 구성된다.

Query Engine (Spark / Trino / Flink)
        ↓
   Iceberg API
        ↓
   Catalog (Glue / Nessie / REST)
        ↓
   Metadata files (snapshots, manifests)
        ↓
   Data files (Parquet / ORC / Avro on S3)

세 가지 계층

테이블 식별자는 다음 형식을 따른다.

catalog_name.database_name.table_name
-- e.g. glue.prod.orders

카탈로그는 최신 메타데이터 파일의 위치를 가리키는 포인터를 저장한다. 엔진은 카탈로그를 통해 테이블을 찾고, Iceberg 메타데이터를 읽어 실제로 스캔할 데이터 파일을 파악한다.

실무 현황

2024~2026년 기준으로 AWS Glue, Databricks(UniForm), Snowflake(외부 테이블), Apache Flink, Trino, Presto, ClickHouse 등이 Iceberg를 폭넓게 지원한다. 원래 Netflix와 Apple이 대규모 환경을 위해 이 설계를 주도했다.

파일은 여전히 S3 위의 Parquet이다. Iceberg 메타데이터가 이식 가능한 계약 역할을 하므로, 독점 레이크 포맷 대비 벤더 종속 위험이 낮다.

3. Iceberg 내부 동작 방식

Iceberg는 데이터 테이블의 Git과 같다.

  • 커밋 = 스냅샷 (특정 시점의 테이블 상태)
  • 커밋의 파일 목록 = 매니페스트
  • 커밋 히스토리 = 스냅샷 로그

데이터를 복사하지 않고도 과거 커밋을 체크아웃할 수 있다(타임 트래블).

메타데이터 계층 구조

매니페스트가 성능에 미치는 영향

매니페스트 파일은 각 데이터 파일의 컬럼별 통계(최솟값/최댓값/null 개수)를 저장한다. 쿼리 엔진은 Parquet 파일을 직접 열기 전에 매니페스트를 먼저 읽어, 필터 조건을 만족할 수 없는 파일 전체를 건너뛴다.

예를 들어 WHERE order_date = '2024-06-01'이라는 조건이 있을 때, 특정 파일의 order_date 최댓값이 2024-05-30이라면 Iceberg는 해당 파일을 Parquet 접근 전에 제거한다.

Copy-on-Write vs Merge-on-Read

대부분의 프로덕션 배치 워크로드는 COW를 사용한다. 스트리밍 업서트에는 MOR을 쓰고 주기적으로 컴팩션을 수행하는 패턴이 일반적이다.

실무에서 자주 만나는 문제

  • 메타데이터 비대화 — 수백만 건의 소형 커밋이 쌓이면 쿼리 계획이 느려진다. expire_snapshots와 rewrite_manifests로 해결한다.
  • 소형 파일 문제 — 수천 개의 1 MB짜리 파일이 쌓이면 매니페스트 오버헤드가 지배적이 된다. rewrite_data_files(컴팩션)로 해결한다.
  • 스냅샷 폭증 — 스트리밍 마이크로배치가 분마다 스냅샷을 생성한다. 스냅샷 만료 정책과 쓰기 배치 처리로 대응한다.

낙관적 동시성 제어

두 작성자가 테이블을 잠그지 않고 동시에 커밋을 시도한다. 첫 번째가 성공하면 두 번째는 기반 스냅샷이 바뀐 것을 감지하고 재시도한다. 동시 쓰기가 많은 환경에서는 커밋 충돌이 빈번해지므로, 재시도 로직을 튜닝하거나 테이블별로 작성자를 직렬화해야 한다.

면접 대비 포인트

자주 나오는 질문들이다.

  • "Iceberg는 오브젝트 스토리지에서 ACID를 어떻게 달성하나요?"
  • "스냅샷이 무엇인가요?"
  • "Iceberg에서 파티션 프루닝은 어떻게 작동하나요?"

핵심 답변: "Iceberg의 커밋은 원자적인 메타데이터 스왑이다. 스냅샷은 불변의 데이터 파일 집합이다. 리더는 항상 단일 스냅샷을 사용하므로 부분 쓰기를 절대 보지 않는다. 매니페스트는 파일 수준 통계를 저장하므로, 디렉터리 전체를 나열하지 않고도 파일을 프루닝할 수 있다."

4. 데이터 엔지니어가 매일 쓰는 핵심 기능

버전 히스토리와 실행 취소 기능이 있고, 기존 시트를 깨뜨리지 않고 컬럼을 추가할 수 있는 스프레드시트를 떠올리면 된다.

ACID 트랜잭션

쓰기 작업은 새 스냅샷을 생성하고, 카탈로그 포인터가 원자적으로 전환된다. 리더는 절대 절반만 쓰인 데이터를 보지 않는다.

Writer A: snapshot 101 → 102  ✓
Writer B: based on 101 → conflict, retry from 102
Reader:   always reads exactly one snapshot (e.g. 102)

타임 트래블

스냅샷 ID나 타임스탬프를 기준으로 과거 시점의 테이블을 조회할 수 있다.

-- Spark SQL
SELECT * FROM prod.orders TIMESTAMP AS OF '2024-06-01 00:00:00';
SELECT * FROM prod.orders VERSION AS OF 48291347;

감사, 버그 재현, 모델 백테스트, 컴플라이언스 등 다양한 용도로 활용된다.

스키마 진화

Iceberg는 컬럼 인덱스가 아닌 컬럼 ID로 스키마를 추적한다. 따라서 컬럼 순서를 바꿔도 기존 파일이 깨지지 않는다.

파티션 진화

Hive에서의 문제를 생각해보자. days(order_date)로 파티셔닝했다가 월별로 바꾸고 싶어졌다. 기존 데이터는 일별 폴더에 그대로 갇혀있다.

Iceberg에서는 새로운 쓰기에만 월별 파티션 스펙을 적용하면 된다. 기존 데이터는 기존 스펙을 유지한 채 하나의 테이블 안에 공존하며, 쿼리도 정상적으로 작동한다.

히든 파티셔닝

사용자는 단순하게 쿼리를 작성한다.

WHERE order_date = DATE '2024-06-01'

Iceberg가 자동으로 파티션 변환(day(order_date))을 적용한다. 물리적 경로 order_date_day=2024-06-01을 알 필요가 없다.

MERGE INTO — CDC 파이프라인의 핵심

MERGE INTO prod.orders AS target
USING staging.orders_updates AS source
ON target.order_id = source.order_id
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *

Debezium/Kafka CDC 스트림이 멱등성 업서트로 레이크에 안착하는 방식이 바로 이것이다.

주의해야 할 부분도 있다.

  • 소스 스키마 드리프트 — CDC가 컬럼을 추가하면 MERGE가 실패한다. 스키마 진화 정책과 계약 테스트로 대응한다.
  • 만료된 스냅샷에서의 타임 트래블 — 삭제된 데이터는 조회할 수 없다. 컴플라이언스 요건에 맞는 보존 정책을 수립해야 한다.
  • 스키마 진화 남용 — 500개 컬럼 중 대부분이 미사용 상태. 거버넌스 정책이 필요하다.

면접 대비 포인트

CDC 파이프라인 구현 방법을 물어볼 때의 핵심 답변 구조는 이렇다: Kafka → Flink/Spark 스트리밍 → 기본키 기반 Iceberg MERGE → 스케줄 컴팩션 → 보존 정책에 따른 오래된 스냅샷 만료.

정확히 한 번(exactly-once) 처리에 대한 후속 질문도 자주 나온다. Iceberg + Flink 2단계 커밋, 또는 Spark 구조적 스트리밍 체크포인트 + 멱등성 MERGE가 답이다. 완전한 exactly-once는 어렵고, at-least-once + 멱등성 MERGE가 실무의 기본값이다.

실습 예제

-- Create table (Spark SQL)
CREATE TABLE glue.prod.orders (
    order_id    BIGINT,
    customer_id BIGINT,
    order_date  DATE,
    amount      DECIMAL(12,2)
)
USING iceberg
PARTITIONED BY (months(order_date))
TBLPROPERTIES ('format-version' = '2');

-- Time travel
SELECT COUNT(*) FROM glue.prod.orders VERSION AS OF 42;
SELECT * FROM glue.prod.orders TIMESTAMP AS OF '2024-06-01 08:00:00' LIMIT 10;

-- Schema evolution
ALTER TABLE glue.prod.orders ADD COLUMN discount DECIMAL(5,2);
ALTER TABLE glue.prod.orders RENAME COLUMN amount TO total_amount;

-- Incremental read (streaming)
-- Spark Structured Streaming: read stream from Iceberg table
-- Flink: Iceberg source with start snapshot / timestamp

# PySpark — read Iceberg table
df = spark.read.format("iceberg").load("glue.prod.orders")

# Write append
df.writeTo("glue.prod.orders").append()

# Overwrite one partition
(
    df.filter("order_date = '2024-06-01'")
    .writeTo("glue.prod.orders")
    .overwritePartitions()
)

-- Maintenance (Spark procedures or engine-specific syntax)
CALL glue.system.rewrite_data_files('prod.orders');      -- compact small files
CALL glue.system.expire_snapshots('prod.orders',          -- drop old history
  TIMESTAMP '2024-01-01 00:00:00');
CALL glue.system.remove_orphan_files('prod.orders');     -- clean failed writes

5. Iceberg vs Delta Lake vs Hudi vs Hive

같은 창고의 상자들을 관리하는 네 가지 카탈로그 시스템이라고 생각하면 된다. 모두 "저렴한 스토리지 위의 파일 관리"라는 문제를 해결하지만, 강점과 생태계가 다르다.

어떤 하나가 절대적으로 우월하지 않다. 다음 기준으로 선택한다.

실무 현황

상호운용성 트렌드를 보면 Databricks UniForm, Snowflake 외부 Iceberg 테이블, AWS Glue 카탈로그 등 오픈 스토리지의 표준으로 Iceberg가 수렴하고 있다.

Hive에서 마이그레이션하는 경로는 간단하다.

-- Spark: migrate Hive table to Iceberg in place (simplified)
CALL catalog.system.migrate('db.hive_orders');

마이그레이션 후에는 스냅샷 확인, 행 수 검증, 컴팩션 재실행, 다운스트림 파이프라인 업데이트 순서로 진행한다.

포맷을 계속 비교만 하다 시간을 낭비하지 말자. 동일 데이터에 두 포맷을 병행하는 비용이 하나를 잘 운영하는 것보다 훨씬 크다.

면접 대비 포인트

"Iceberg vs Delta Lake?" 질문에 대한 핵심 답변: "둘 다 오브젝트 스토리지 위에 ACID와 타임 트래블을 제공한다. Iceberg는 엔진 중립성이 더 강하고 파티션 진화와 히든 파티셔닝이 뛰어나다. Delta는 Databricks 생태계에서 성숙도가 높다. 멀티 엔진 오픈 레이크하우스라면 Iceberg를, Databricks 중심 조직이라면 Delta를 선택하되 UniForm이 간극을 메워줄 수 있는지 확인하겠다."

6. 카탈로그, 엔진, 레이크하우스 스택

모든 독자(엔진)에게 최신 챕터(메타데이터 파일)의 위치를 알려주는 목차라고 생각하면 된다.

카탈로그가 없으면 엔진들이 각자 경로를 추측해야 한다. 카탈로그가 있으면 모두가 glue.prod.orders라는 이름으로 현재 메타데이터 위치를 합의한다.

주요 카탈로그

전형적인 레이크하우스 스택

Nessie 브랜칭 (고급)

테이블을 위한 git 브랜치처럼 동작한다. dev.orders에서 실험하고 main.orders로 머지한다. 페타바이트를 복사하지 않고 격리된 환경에서 테스트할 수 있어 유용하다.

실무 주의사항

카탈로그는 거버넌스의 병목 지점이다. IAM 정책, 테이블 접근 제어 목록, 감사 로그가 모두 여기에 있다. 카탈로그 접근 권한을 잃으면 S3 파일이 멀쩡해도 테이블을 쓸 수 없다.

크로스 계정/크로스 리전 구성에서는 데이터 파일 복제와 카탈로그 메타데이터 동기화를 함께 처리해야 하며, S3 복제 시 일관성 순서에도 주의해야 한다.

7. 운영 관리 — 컴팩션, 스냅샷 만료, 유지 보수

계절마다 정리가 필요한 옷장과 같다. 마이크로배치 파일(새 옷)이 계속 쌓이고, 오래된 스냅샷(안 입는 옷)은 자리를 차지한다. 한 달에 한 번은 정리하고 비워야 한다.

유지 보수 작업의 종류

파일 크기 목표값은 쿼리 패턴에 따라 다르지만, S3 위의 Parquet 기준으로 보통 256 MB ~ 1 GB를 권장한다.

보존 기간 트레이드오프

실무 운영 패턴

유지 보수는 반드시 자동화해야 한다. 컴팩션을 방치한 테이블은 느려지고 비용이 증가한다. Iceberg 자체의 문제가 아니라 운영 일정을 잡지 않은 결과다.

전형적인 Airflow/dbt 유지 보수 스케줄은 다음과 같다.

Sunday 02:00  → rewrite_data_files (all tier-1 tables)
Sunday 03:00  → expire_snapshots (retain 60 days)
Sunday 04:00  → remove_orphan_files (with dry-run first)

흔히 발생하는 문제들도 있다.

  • 과도한 orphan 파일 정리 — 아직 진행 중인 쓰기의 파일을 삭제할 수 있다. 반드시 최소 연령 기준을 설정해야 한다.
  • 피크 타임 컴팩션 — I/O 경합이 발생한다. 비피크 시간대에 스케줄링한다.
  • 스냅샷 만료 후 타임 트래블 — 보존 기간 밖의 스냅샷을 조회하는 쿼리가 실패한다. BI 툴링의 사용 패턴을 보존 정책과 맞춰야 한다.

실습 예제

# PySpark maintenance job (conceptual)
spark.sql("""
    CALL glue.system.rewrite_data_files(
        table => 'prod.orders',
        options => map('min-file-size-bytes', '134217728',
                       'target-file-size-bytes', '536870912')
    )
""")

-- Check table health (Spark)
SELECT * FROM glue.prod.orders.files;       -- data files + sizes
SELECT * FROM glue.prod.orders.snapshots;   -- snapshot history
SELECT * FROM glue.prod.orders.partitions;  -- partition stats

면접 치트 시트

마치며

Apache Iceberg는 마법 같은 스토리지가 아니다. 저렴한 Parquet 파일 위에 올려진 규율 있는 메타데이터 관리 체계다. 테이블 포맷 없는 데이터 레이크가 결국 무너지는 이유는 명확하다. 부분 쓰기, 재현 불가능한 보고서, 파티션 폴더 난립, 조인을 조용히 망가뜨리는 스키마 변경이 그 이유다.

데이터 엔지니어에게 중요한 것들을 정리하면 이렇다.

  • 왜 — S3의 raw 파일만으로는 ACID, 타임 트래블, 안전한 진화가 불가능하다.
  • 무엇 — 오픈 테이블 포맷: 카탈로그 + 메타데이터 + 데이터 파일.
  • 어떻게 — 스냅샷, 매니페스트, 낙관적 동시성, 히든 파티셔닝.
  • 일상 활용 — CDC를 위한 MERGE, 스키마/파티션 진화, 감사를 위한 타임 트래블.
  • 운영 — 컴팩션과 스냅샷 만료는 선택이 아니라 제품의 일부다.

Iceberg는 이미 알고 있어야 할 기반 기술들 위에 올려진다. 오브젝트 스토리지(S3), 컬럼형 파일(Parquet), 데이터 모델링에서의 그레인과 스키마 계약, 파이프라인 설계에서의 at-least-once와 멱등성 쓰기. 이 기반 없이 Iceberg부터 파고들면, 아무도 유지 보수할 수 없는 테이블을 만드는 Spark 잡을 양산하게 된다. 수천 개의 소형 파일, 감사를 망가뜨리는 만료된 스냅샷, 커밋을 두고 서로 싸우는 MERGE 잡들이 그 결과다.

Iceberg 전에 Parquet과 파티셔닝을 익혀라. MERGE INTO 전에 데이터 모델링을 배워라. CDC 전에 파이프라인 멱등성을 이해하라.

기초를 탄탄히 하면, Iceberg는 Stack Overflow에서 복붙하는 블랙박스가 아니라 자신 있게 운영하는 도구가 된다.
