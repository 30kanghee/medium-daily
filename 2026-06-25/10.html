Automating the Entire Data Engineering Lifecycle with AI: An AI-First Approach to TDLC, SDLC, and…

Nayan Paul · 17분 · 원문 · freedium

요약

슬래시 커맨드와 자율 서브에이전트로 PySpark + Delta Lake 파이프라인 전 과정을 자동화하는 AI 퍼스트 데이터 엔지니어링 접근법

데이터 엔지니어링 프로젝트를 처음부터 구축할 때마다 반복되는 의례적 작업들 — 폴더 구조 생성, 보일러플레이트 코드 작성, 스키마 추론 씨름 — 은 창의적이지도, 차별화되지도 않으면서 며칠을 잡아먹는다. 저자가 만든 SDLC Data Pipeline 패키지는 Claude Code의 슬래시 커맨드 7개와 자율 서브에이전트들로 이 전 과정을 자동화한다. /project-init부터 /checkin까지 각 단계는 인간이 질문에 답하는 대화형 프런트엔드와, 실제 코드를 생성하는 자율 에이전트로 역할이 명확히 분리된다. 특히 큐레이션, 데이터 프로덕트 설계, git 푸시의 세 지점에서는 인간의 명시적 승인 없이 어떤 코드도 생성되지 않는다는 점이 핵심 설계 원칙이다. AI는 가능한 것을 알고, 인간은 적절한 것을 결정한다.

슬래시 커맨드와 자율 서브에이전트 묶음이 어떻게 운영 수준의 PySpark + Delta Lake 파이프라인을 처음부터 끝까지 자동으로 구축하는지, 그리고 모든 중요한 결정 지점에서 어떻게 인간이 개입하는지 살펴본다.

우리가 풀려 했던 문제

데이터 엔지니어링 프로젝트를 밑바닥부터 구축해본 사람이라면 그 의례를 잘 안다. 폴더 구조를 만들고, SparkSession 팩토리 보일러플레이트를 작성하고, 수집 스크립트 뼈대를 세운다. 스키마 추론과 증분 워터마크, 파티션 전략을 붙들고 씨름한다. SCD Type 2 병합 로직을 직접 코딩하고, README를 쓰고, .env를 .gitignore에 넣는 것을 빠뜨리지 않았는지 확인한다.

이 과정은 대부분 기계적인 작업이다. 창의적이지도 않고, 차별화되지도 않지만, 프로젝트마다 며칠을 잡아먹는다. 더 심각한 문제는 일관성이 없다는 것이다. 같은 팀의 두 엔지니어가 증분 워터마크를 다르게 구현하고, 한 명은 불량 행을 격리하지만 다른 한 명은 조용히 삭제한다. 한 명은 bronze 레이어를 ingest_date로 파티션하지만 다른 한 명은 비즈니스 컬럼으로 파티션해 하위 단계를 망가뜨린다.

우리가 답하려 한 질문은 단순했다. mkdir부터 git push까지, 선택 사항을 기억하고 맥락을 이어받아 일관된 아키텍처의 운영 품질 코드를 생성하는 대화형 AI가 전체 파이프라인을 이끌 수 있을까?

그 답이 바로 우리가 SDLC Data Pipeline이라 부르는 Claude Code 패키지다. 슬래시 커맨드와 자율 서브에이전트의 묶음으로, PySpark + Delta Lake 프로젝트의 기술 납품 생명주기(TDLC), 소프트웨어 개발 생명주기(SDLC), DevOps 체크인 프로세스를 자동화한다.

우리가 만든 것: 7단계 AI 파이프라인

이 패키지는 프로젝트를 7개 단계로 이끈다. 각 단계는 Claude Code에서 입력하는 슬래시 커맨드 하나다.

/project-init → /ingest → /curate → /cdc → /data-product → /test-harness → /checkin

각 커맨드는 질문을 던지는 상태 유지형 대화 프런트엔드다. 커맨드 뒤에서는 하나 이상의 자율 서브에이전트가 무거운 작업을 처리한다. 스키마를 읽고, 코드를 생성하고, 파일을 쓰는 것이 그 역할이다. 두 계층 사이에서 영속 상태 파일 .sdlc/state.json이 모든 정보를 다음 단계로 전달하므로 같은 질문을 두 번 묻는 일이 없다.

1단계: /project-init — 기반 다지기

새 데이터 프로젝트는 구조에서 시작한다. /project-init 커맨드는 다섯 가지를 묻는다. 프로젝트 이름, 설명, 기술 스택 확인(기본값은 pyspark-delta), 기본 출력 경로, 팀 이름이다. 딱 다섯 가지다. 이 답변을 받은 뒤 project-init-agent 서브에이전트에게 넘겨 전체 프로젝트 트리를 뼈대로 세운다.

my-pipeline/
├── README.md
├── .env.example
├── .gitignore
├── requirements.txt
├── conf/
│   └── project.yaml
├── src/
│   ├── __init__.py
│   └── common/
│       ├── spark.py       ← 모든 단계가 공유하는 SparkSession 팩토리
│       └── config.py
├── tests/
│   └── __init__.py
└── .sdlc/
    └── state.json         ← 전체 파이프라인의 영속 컨텍스트 객체

모든 하위 단계는 src/common/spark.py를 가져다 쓴다. Delta Lake 확장이 활성화된 SparkSession을 만드는 재사용 헬퍼다. 단일 공유 팩토리를 두면 프로젝트 내 모든 생성 스크립트가 일관되게 유지된다. 카탈로그 설정도, Delta 확장도, 환경 변수를 통한 확장 패턴도 똑같다.

state.json은 이곳에서 초기화되며, 프로젝트 메타데이터와 다섯 파이프라인 단계가 모두 pending 상태로 기록된다. 이 파일이 파이프라인의 기억이다. 단계가 완료될 때마다 산출물, 선택 사항, 결과물이 여기에 기록되고, 이후 단계는 사용자에게 다시 묻지 않고 이 파일에서 맥락을 복원한다.

에이전트가 협상 없이 강제하는 설계 원칙이 하나 있다. 기존 .sdlc/state.json을 절대 덮어쓰지 않는다는 것이다. 이미 상태 파일이 있는 디렉터리에서 /project-init를 실행하면 에이전트는 즉시 중단하고 프로젝트가 이미 초기화되었다고 보고한다. 이는 흔한 실수 — 다시 초기화를 실행해 반쯤 완성된 파이프라인의 상태를 날리는 것 — 를 방지한다.

AI 퍼스트 관점에서 본 핵심 통찰: 프로젝트 초기화는 창의적 작업이 아니다. 모든 단계가 동의하는 약속의 집합이다. 그 약속을 인간의 기억이 아니라 서브에이전트의 지시에 인코딩함으로써, 모든 프로젝트가 같은 토대에서 시작한다.

2단계: /ingest — 소스에서 Bronze Delta Lake로

수집 단계에서 진짜 다양성이 시작된다. 소스는 ADLS/S3/로컬 파일, JDBC 데이터베이스, Kafka 토픽, REST API, Snowflake 테이블이 될 수 있다. 로드는 배치 또는 스트리밍이고, 전체 또는 증분 방식이다. 증분 전략도 소스마다 다르다. JDBC는 타임스탬프 워터마크, 파일 소스는 Auto Loader, Kafka는 오프셋 체크포인트, REST API는 커서 파라미터를 사용한다.

/ingest 커맨드는 선택에 따라 갈라지는 일련의 질문을 던진다. "배치 JDBC 증분"이라고 답하면 워터마크 컬럼, 타입(타임스탬프 대 숫자), 초기 워터마크 값을 묻는다. "스트리밍 Kafka"라고 하면 startingOffsets와 체크포인트 경로를 확인한다.

커맨드가 명시적으로 묻지 않는 것이 하나 있다. 어떤 비즈니스 파티션 키를 쓸지다. 답은 항상 ingest_date이며, 이것이 이 패키지가 인코딩하는 가장 중요한 아키텍처 결정 중 하나다.

Bronze는 항상 ingest_date(로드 날짜)로 파티션된다. 매 실행은 자체 ingest_date=YYYY-MM-DD/ 폴더에 저장된다. 파티션이 곧 로드 경계다.

이 말은 "새 로드를 처리한다"는 것이 항상 "새 ingest_date 파티션을 읽는다"와 동일하다는 의미다. 워터마크 메타데이터도, "마지막 실행 날짜가 뭐였지?"라는 질문도 필요 없다. 큐레이션 단계는 이름으로 정확한 로드를 지정할 수 있다. 로드를 다시 실행해도 다른 파티션이 오염되지 않는다. 로드 경계가 파일 시스템에 자체 문서화된다.

ingest-agent 서브에이전트는 컨텍스트 JSON — 소스 타입, 모드, 연결 환경변수 이름, 경로, 로드 타입, 증분 전략 — 을 받아 다음을 생성한다.

  • src/ingest.py: main() 가드, 타입 힌트, docstring, 단계별 로깅이 포함된 완전히 실행 가능한 PySpark + Delta 스크립트. 증분 JDBC 로드의 경우 conf/_watermarks/ 아래 JSON 파일에 고수위 표시를 영속화하는 read_watermark() / write_watermark() 헬퍼를 생성한다.
  • README-ingestion.md: 소스, 모드, 로드 타입, 증분 전략, 필요한 환경변수, 스크립트 실행 방법을 문서화한다.
  • requirements.txt와 .env.example 업데이트: 생성된 스크립트에서 참조하는 모든 환경변수가 .env.example에 플레이스홀더로 등록된다. 특히 에이전트는 Spark 데이터소스 커넥터(Kafka, JDBC 드라이버, Snowflake)를 pip 패키지가 아닌 spark-submit --packages 좌표로 문서화하도록 강제한다. 미묘하지만 운영 환경에서 결정적으로 중요한 구분이다.

AI 퍼스트 관점에서 본 핵심 통찰: 수집의 분기 결정 트리(소스 타입 × 로드 타입 × 모드 × 증분 전략)는 어떤 인간도 일관되게 머릿속에 담아두기엔 너무 깊다. 모든 조합을 파악하고 각 경로에 필요한 질문을 아는 AI는 설정 실수라는 아예 통째의 오류 유형을 제거한다.

3단계: /curate — 추천-승인-생성 게이트

큐레이션은 가장 아키텍처적으로 흥미로운 단계이며, 인간이 진정으로 루프 안에 있어야 할 때 "AI 퍼스트"가 무엇을 의미하는지 가장 명확하게 보여주는 단계다.

이 단계는 세 하위 단계로 구성된다.

1단계 — 추천(코드 미작성): curate-advisor 서브에이전트가 bronze 스키마나 샘플 데이터 파일을 받아 JSON 형태의 권장 규칙 세트를 만든다. 컬럼 이름 변경, 데이터 타입 캐스팅, 표준화 규칙, 보강 컬럼, null 처리, 중복 제거, 데이터 품질 검사, 불량 행 처리 방식의 여덟 범주를 다룬다. 모든 추천에는 한 줄짜리 근거가 딸린다.

결정적으로, 어드바이저는 파일을 쓰지 않는다. 출력은 오직 JSON 추천뿐이며, 메인 대화 스레드로 반환된다.

2단계 — 인간 검토 및 승인(게이트): /curate 커맨드가 추천 사항을 범주별로, 각각의 근거와 함께 제시한다. 승인하거나, 수정하거나, 개별 규칙을 거부하거나, 새 규칙을 추가할 수 있다. 만족할 때까지 반복한 후, 어드바이저가 제안할 수는 있지만 결정할 수는 없는 한 가지를 선택한다. 불량 행을 어떻게 처리할 것인가?

  • 격리(Quarantine): 정상 행은 silver 경로로, error 심각도의 데이터 품질 검사에 실패한 행은 dq_failure_reason 컬럼이 태그된 채 별도의 격리 경로로 간다. 둘 다 파티션을 인식하는 Delta 쓰기다.
  • 건너뛰기/인라인 수정: 실패한 행은 제자리에서 삭제되거나 복구되며, 별도의 격리 저장소는 없다.

warn 심각도 검사만 실패한 행은 절대 격리되거나 삭제되지 않는다. dq_warnings 컬럼이 태그된 채 silver에 유지되어 관측 가능성을 확보한다. 이 구분 — error 대 warn, 격리 대 유지 — 은 단순한 데이터 품질 정책이 아니다. 데이터셋에서 "올바름"이 무엇인지에 대한 선언이며, 인간의 판단이 필요하다.

승인이 있기 전까지 코드는 생성되지 않는다.

3단계 — 코드 생성: curate-agent 서브에이전트가 승인된 규칙 세트를 받아 src/curate.py에 구현한다. 스크립트는 정의된 순서로 변환을 적용한다. 읽기, 이름 변경, 캐스팅, 표준화, 보강, null 처리, 중복 제거, DQ 검사, 쓰기 순이며, 각각 docstring이 있는 명확한 이름의 함수로 분리된다.

AI 퍼스트 관점에서 본 핵심 통찰: 어드바이저 패턴은 가능한 것(AI가 큐레이션 모범 사례에 대해 아는 모든 것)과 적절한 것(특정 데이터와 도메인에 맞는 규칙)을 분리한다. AI는 dt라는 컬럼이 아마 날짜를 의미하며 timestamp로 캐스팅해야 한다는 것을 알지만, null dt가 행을 삭제해야 하는지 로드 날짜로 채워야 하는지는 당신이 안다. 게이트는 그 경계를 강제한다.

4단계: /cdc — 변경 데이터 캡처와 이력 관리

모든 데이터셋이 CDC를 필요로 하지는 않는다. /cdc 커맨드가 던지는 첫 번째 질문은 간단하다. 이 데이터셋에 CDC나 이력 관리가 필요한가? 필요 없다고 하면 CDC는 상태에 skipped로 기록되고 파이프라인이 진행된다. 이것이 중요한 이유는, 데이터 프로덕트 단계가 이력이 필요 없을 때 큐레이션된 silver 레이어에서 직접 작업할 수 있다는 의미이기 때문이다.

필요하다고 하면, 커맨드가 데이터를 기반으로 SCD 전략을 추천하고 트레이드오프를 설명한 뒤 명시적 확인을 요구한다.

  • SCD Type 1(덮어쓰기/upsert): 현재 값만 유지한다. Delta MERGE가 매칭된 키를 업데이트하고 새 키를 삽입한다. 단순하고 빠르며 이력이 없다. 하위 단계가 최신 상태만 필요할 때 권장한다.
  • SCD Type 2(유효 날짜가 있는 이력): 모든 속성 변경의 전체 이력을 유지하며, effective_from, effective_to, is_current, 대리 키가 포함된다. 구현에는 손으로 작성하기 까다롭기로 유명한 두 단계 staged-updates 병합 패턴을 사용한다.

에이전트는 삭제 감지 전략 사이의 중요한 구분도 안내한다. 소스에 삭제 마커 컬럼이 있는 경우와, 전체 스냅샷에서 키가 없어진 것을 삭제로 처리하는 경우다. 스냅샷 부재 방식은 오직 완전한 현재 스냅샷 로드에서만 작동하며, 증분 델타 로드에 절대 사용해서는 안 된다고 명시적으로 경고한다.

cdc-agent는 전체 병합 로직이 담긴 src/cdc.py를 생성한다. gold/CDC 레이어는 유일하게 커스텀 비즈니스 파티션을 쓰는 레이어다. bronze와 silver는 ingest_date로 파티션되고, gold는 비즈니스 키로 파티션된다. 이 3계층 파티션 전략은 우연이 아니다. 각 레이어에서 로드 격리와 쿼리 성능 사이의 의도적인 트레이드오프를 인코딩한다.

5단계: /data-product — Gold 테이블에서 소비 가능한 프로덕트로

데이터 프로덕트 단계는 파이프라인이 엔지니어링 관심사(로드 경계, 파티션, CDC 로직)에서 비즈니스 관심사(BI 소비자가 실제로 보는 것이 무엇인가?)로 전환되는 지점이다.

큐레이션과 마찬가지로 어드바이저→게이트→생성기 패턴을 사용한다. data-product-advisor 서브에이전트가 gold 스키마, 원하는 BI 리포팅 단위, 알려진 KPI를 받아 다음을 추천한다.

  • 비즈니스 데이터 계약: 소비자 대상 컬럼 이름, 타입, 설명, 기본 키, PII 플래그, 데이터 소유자, 수집 주기에서 추론된 신선도 SLA.
  • BI 뷰: 소스 위에 비즈니스 친화적 뷰. Type 2 소스의 경우 기본적으로 현재/삭제되지 않은 행을 대상으로 하며, 선택적 이력 뷰도 포함.
  • 집계 및 KPI 정의: 이름, Spark SQL 표현식, 롤업 단위, kind 분류(measure는 단순 집계, kpi는 평균 주문 금액이나 기간 대비 성장률 같은 파생 비즈니스 지표).
  • 최적화 키: 제안된 ZORDER 컬럼과 프로덕트 테이블 파티션 여부.

스펙을 검토하고, KPI 목록을 편집하고, BI 뷰 필터를 확인한 후 승인한다. 그러면 data-product-agent가 생성한다. src/data_product.py, conf/data_contract.yaml, DATA_DICTIONARY.md, 그리고 전체 프로젝트를 포괄하는 새 README.md가 만들어진다.

AI 퍼스트 관점에서 본 핵심 통찰: 데이터 프로덕트는 테이블이 아니라 소비자와의 계약이다. 인간이 검토하고 승인한 계약을 일관되게 생성함으로써, 대부분의 데이터 팀이 갖추지 못한 것을 얻는다. 빌드 프로세스의 자연스러운 산출물로, 사후에 덧붙이는 것이 아닌 문서화되고 버전 관리된 기계 판독 가능한 약속이다.

6단계: /test-harness — 템플릿이 아닌 실제 코드에 맞는 테스트

데이터 파이프라인 테스트가 어려운 이유는, 테스트가 실제 생성된 코드의 실제 함수와 일치해야 하기 때문이다. 함수 이름을 가정하는 일반 테스트 템플릿은 AI가 rename_columns 대신 apply_column_renames라는 이름의 함수를 생성하는 순간 망가진다.

/test-harness 단계는 test-agent가 생성된 소스 파일을 먼저 읽게 함으로써 이 문제를 해결한다. 채워 넣을 템플릿이 없다. src/ingest.py, src/curate.py, src/cdc.py, src/data_product.py를 직접 읽고, 실제 함수 이름과 시그니처를 파악한 뒤, 그 함수들을 대상으로 테스트를 작성한다.

생성된 테스트 묶음에는 다음이 포함된다.

  • tests/conftest.py: 세션 범위의 spark픽스처와, DQ 규칙을 의도적으로 위반하도록 설계된 행이 포함된 합성 픽스처.
  • tests/test_ingest.py: 순수 헬퍼 테스트. 파티션 컬럼 도출, 워터마크 읽기/쓰기 왕복, 외부 리더는 목 처리.
  • tests/test_curate.py: 존재하는 각 변환에 대한 테스트. 특히 DQ 심각도 분리 — error 검사 실패 행은 격리되고, warn 검사만 실패한 행은 dq_warnings 태그가 붙어 silver에 유지된다.
  • tests/test_cdc.py: 작은 타겟 + 입력 DataFrame으로 Type 1 upsert, Type 2 close-and-insert, 소프트/하드 삭제 처리를 검증. 임시 Delta 테이블에는 tmp_path를 사용.
  • tests/test_data_product.py: 작은 DataFrame에 대한 각 KPI/집계 표현식 검사, BI 뷰 필터 정확도.

AI 퍼스트 관점에서 본 핵심 통찰: 테스트 생성은 코드 읽기 AI의 가장 높은 레버리지 활용 중 하나다. 인간은 코드가 한다고 생각하는 것을 기반으로 테스트를 쓴다. AI는 코드를 직접 읽는다. 해석 간극이 없고, "요구사항이 안정되면 테스트를 쓸게"도 없다. 코드가 존재하는 순간, 코드가 노출하는 정확한 API에 맞게 테스트가 생성된다.

7단계: /checkin — 안전하고 신중하며 게이트가 있는 DevOps

체크인 단계는 파이프라인이 돌이킬 수 없는 것, 즉 원격 저장소를 건드리는 지점이다. 설계에 그 무게가 반영되어 있다.

/checkin 커맨드는 git 제공자(GitHub/Git 또는 Azure DevOps), 저장소 좌표, 인증 방식(CLI 인증 또는 PAT 환경변수), 타겟 브랜치, PR 개설 여부를 수집한다. PAT를 담은 환경변수의 이름을 묻는다 — PAT 자체가 아니라. 리터럴 토큰을 붙여 넣으면 커맨드가 거부하고 환경변수로 설정하도록 안내한다.

커밋 전에 필수 사전 점검이 실행된다.

  1. .gitignore 검증 또는 생성 — .env, *.log, __pycache__/, .venv/, spark-warehouse/, metastore_db/, /data/, conf/_watermarks/가 반드시 포함되어야 한다.
  2. 모든 파일을 스테이징하고 전체 git status --short를 표시한다.
  3. 비밀 스캔: .env와 자격증명 파일이 스테이징되지 않았는지 확인하고, 스테이징된 파일에서 명백한 비밀 패턴(토큰, PASSWORD=, BEGIN ... PRIVATE KEY, 내장된 연결 문자열)을 검색한다. 의심스러운 것이 있으면 중단하고, 스테이징을 취소하고, .gitignore를 수정한 뒤 보고한다.
  4. 커밋될 정확한 파일, 원격지, 브랜치 흐름, 커밋 메시지를 표시하고 checkin-agent가 호출되기 전 명시적 확인을 요구한다.

checkin-agent는 환경변수 이름만 받는다. 런타임에 환경에서 토큰을 읽어 .git/config나 원격 URL에 저장하지 않고 푸시에 사용하며, 커밋 SHA, 브랜치, 원격 URL, PR 링크를 보고한다 — 토큰은 절대 노출하지 않는다.

AI 퍼스트 관점에서 본 핵심 통찰: DevOps 자동화에서 보안은 AI에게 비밀을 신뢰하는 문제가 아니다. 비밀이 AI의 컨텍스트에 처음부터 들어오지 않도록 시스템을 설계하는 문제다. 에이전트는 받지 않은 토큰을 유출할 수 없다. 아키텍처가 정책이 아니라 구조적으로 이를 강제한다.

아키텍처: 슬래시 커맨드 + 서브에이전트가 필요한 이유

이 패키지에서 가장 중요한 설계 결정은 하이브리드 모델이다. 대화형 질문을 위한 슬래시 커맨드, 자율 코드 생성을 위한 서브에이전트, 그리고 영속 컨텍스트 객체로서의 .sdlc/state.json이다.

이 분리는 Claude Code의 하드 제약에서 비롯된다. 서브에이전트는 자율적이며 사용자에게 질문할 수 없다. 메인 대화에 접근할 수 없는 상태로 시작한다. 컨텍스트 JSON을 프롬프트에 받아 작업을 수행하고 결과를 반환한다. "잠깐, 비즈니스 키가 어떤 컬럼이었죠?"라고 물을 수 없다.

이 제약이 사실은 기능이다. 깔끔한 아키텍처를 강제한다.

결과적으로 각 단계는 독립적으로 실행 가능하다. DQ 규칙을 바꾼 뒤 큐레이션 스크립트를 다시 생성해야 한다면, /curate를 직접 실행하면 된다. .sdlc/state.json을 읽고 이전 단계에서 필요한 모든 것을 복원한 뒤, 큐레이션 경로와 샘플 스키마만 다시 묻는다. 수집을 다시 실행하거나 모든 질문을 처음부터 다시 답할 필요가 없다.

세 가지 승인 게이트: 인간이 루프 안에 있는 곳

이 패키지에는 인간이 명시적으로 승인하기 전까지 코드가 생성되지 않는 세 단계가 있다.

  1. /curate: curate-advisor가 추천한 큐레이션 규칙을 검토·편집하고, 불량 행 처리 방식을 선택한 뒤 curate-agent가 코드를 생성한다.
  2. /data-product: data-product-advisor가 추천한 BI 계약, 뷰, KPI를 검토·편집한 뒤 data-product-agent가 코드를 생성한다.
  3. /checkin: 사전 점검, 스테이징된 파일 검토, checkin-agent가 실행되기 전 명시적 푸시 확인.

이 게이트들은 관료적 마찰이 아니다. AI의 가능한 것에 대한 지식과 인간의 적절한 것에 대한 지식이 만나는 지점이다. 큐레이션 규칙 세트는 데이터 의미론에 맞을 때만 가치 있다. KPI 정의는 비즈니스 정의와 일치할 때만 유용하다. git 푸시는 무엇을 배포하는지 알 때만 안전하다.

레이어드 파티션 모델: 관례로 인코딩된 아키텍처

이 패키지의 가장 가치 있는 역할 중 하나는 세 개의 Delta Lake 레이어 전체에 걸쳐 일관되고 설명 가능한 파티션 전략을 강제하는 것이다.

이 아키텍처는 임의적이지 않다. Bronze를 ingest_date로 파티션하면 모든 로드가 격리된다. 다른 파티션에 영향을 주지 않고 어느 로드든 재처리할 수 있고, 특정 날짜에 들어온 것을 감사할 수 있으며, 큐레이션 단계는 각 실행의 자연스러운 "범위"를 워터마크 없이 갖는다. Silver는 큐레이션이 한 번에 하나의 로드를 처리하므로 같은 파티션을 이어받는다. Gold는 BI 도구가 실제로 쿼리하는 유일한 레이어이므로, 비즈니스 파티션이 효과를 발휘하는 곳이 바로 여기다.

AI 퍼스트 데이터 엔지니어링의 의미

이 패키지가 처음부터 끝까지 하는 일을 돌아보면, 어느 단일 단계보다 더 큰 패턴이 드러난다. AI 퍼스트 SDLC/TDLC가 실제로 어떤 모습인지가 보인다.

AI는 조직의 암묵지를 인코딩한다. JDBC 증분 로드의 워터마크 패턴, 두 단계 SCD Type 2 병합, 데이터 품질의 warn 대 error 심각도 분리 — 이것들은 경험 많은 데이터 엔지니어가 알고, 주니어 엔지니어가 어렵게 배우는 것들이다. 이 패키지는 그것들을 에이전트 지시의 협상 불가능한 동작으로 인코딩한다. 누가 파이프라인을 실행하든, 모든 프로젝트가 그 지식의 혜택을 받는다.

AI는 설정 드리프트를 제거한다. 이 패키지 없이는 두 엔지니어가 파티션 키, 큐레이션 순서, DQ 규칙 명명 관례, 테스트 구조에 대해 서로 다른 선택을 한다. 이 패키지가 있으면 선택들이 아키텍처에 내재된다. 사람들이 때로는 따르고 때로는 무시하는 스타일 가이드 때문이 아니라, 생성기가 그 선택을 하기 때문에 프로젝트들이 일관성을 갖는다.

인간은 인간만이 결정할 수 있는 것에 집중한다. 승인 게이트는 비즈니스 지식이 필요한 결정에 정확히 배치된다. 어떤 큐레이션 규칙이 데이터 의미론에 맞는가, 어떤 KPI가 비즈니스에 중요한가, 데이터 계약이 소비자에게 무엇을 약속하는가. AI가 나머지를 처리한다.

생명주기는 연속적이다. TDLC는 파이프라인 구축에서 끝나지 않는다. 테스트, 문서화, 안전한 배포까지 이어진다. 파이프라인은 data_product.py에서 멈추지 않는다. /test-harness(실제 생성된 코드에 맞는 테스트 생성)와 /checkin(보안 가드레일로 커밋 및 푸시)을 거쳐 계속된다. "데이터 파이프라인이 필요하다"부터 "코드가 main에 있고 PR이 열려있다"까지 전 과정이 단일 자동화 워크플로우다.

시작하기

이 패키지는 .claude/ 아래 두 하위 디렉터리에 들어있다.

패키지 디렉터리에서 Claude Code 세션을 열고 새 파이프라인을 시작하려면 이렇게 입력한다.

/project-init my-sales-pipeline

파이프라인은 단계별로 질문을 던지며, 큐레이션, 데이터 프로덕트 설계, 체크인에서 인간 승인 게이트를 거친다. 나머지는 자동화된다. 끝에는 mkdir부터 git push까지, 완성되고 문서화되고 테스트되고 버전 관리된 PySpark + Delta Lake 파이프라인이 완성된다.

결론

기존의 SDLC 접근법은 AI를 보조자로 취급한다. 보일러플레이트를 더 빠르게 쓰는 수단, 더 나은 자동완성, 코드도 생성할 수 있는 고무 오리. 여기서 설명하는 접근법은 다르다. AI가 프로세스다. 단계를 이끌고, 맥락을 이어받고, 아키텍처를 강제하고, 코드를 생성한다. 인간이 루프 밖에 있는 것이 아니다. 인간은 정확히 올바른 결정 지점에 있다. AI가 나머지를 모두 처리한다.

이것이 AI 퍼스트의 실제 의미다. 사용하는 도구로서의 AI가 아니라, 그 위에 구축하는 인프라로서의 AI. 가치는 단일 생성 파일에 있지 않다. 일관성, 완전성, 규모에 인코딩된 조직의 암묵지, 그리고 엔지니어들이 실제로 인간의 판단이 필요한 결정에 집중할 수 있는 자유에 있다.

파이프라인은 완성됐다. 코드는 일관적이다. 테스트는 코드와 일치한다. README가 모든 것을 문서화한다. 그리고 이 중 어느 것도 워터마크 헬퍼, DQ 심각도 분리, SCD Type 2 staged-updates 병합, 사전 점검 비밀 스캔을 손으로 직접 작성할 필요가 없었다.

그것이 핵심이다.
