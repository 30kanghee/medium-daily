Stop Designing REST APIs Like a Mid-Level Dev: 4 Advanced Patterns Senior Engineers Use Instead

HabibWahid · Stackademic · 8 min · 원문 보기 (Medium)

요약

미들레벨 개발자의 REST API와 시니어의 API를 가르는 건 결제 재시도, 동시 수정, 폐기 전략 같은 디테일이다.

이 글은 한 프린서플 엔지니어가 던진 질문, 즉 결제 요청이 실패해 재시도되면 어떻게 되는지, 두 클라이언트가 같은 레코드를 동시에 수정하면 어떻게 되는지, 엔드포인트를 폐기할 때 클라이언트에게 무엇을 알려줄지에서 출발한다. 저자는 이런 질문에 답하지 못하는 설계가 바로 미들레벨 개발자 수준에 머무는 이유라고 지적한다. 첫 번째 패턴은 멱등성으로, 결제처럼 부수효과가 있는 POST 요청에는 Idempotency-Key를 부여하고 서버에서 중복 요청을 걸러내야 한다고 설명한다. 두 번째로 스트라이프의 이중 버전 관리 방식을 예로 들어, URL에는 메이저 버전만 노출하되 API 키마다 날짜 기반 버전을 고정시켜 기존 클라이언트가 안전하게 옛 버전을 계속 쓸 수 있게 하는 전략을 소개한다. 마지막으로 오프셋 기반 페이지네이션은 실시간으로 바뀌는 데이터에 취약하므로, 커서 기반 페이지네이션으로 이를 해결해야 한다고 강조한다.
