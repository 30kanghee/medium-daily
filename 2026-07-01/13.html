I Wish I Knew This Before Building an AI Second Brain

JD Wilkins · 15분 읽기 · 원문 · freedium

AI를 얼마나 많이 쏟아붓든 기초 구조가 제대로 갖춰지지 않으면 AI 세컨드 브레인은 결국 무너진다.

요약

저자는 Obsidian 볼트를 AI 에이전트(Claude Code)와 연결한 세컨드 브레인을 구축하면서 여러 번 실패한 경험을 바탕으로, 성공의 핵심이 AI 도구 선택이 아닌 볼트 구조의 철저한 설계에 있음을 강조한다. 폴더 계층, 링크 방식, 프론트매터 필드, 토픽 페이지, Obsidian CLI 연동까지 구체적인 시스템 설계 원칙을 공유하며, AI를 투입하기 전에 사람이 직접 시스템을 완전히 익혀야 한다고 역설한다. 시스템이 잘 갖춰지면 Claude는 볼트 전체를 인식하고 검색하며, 파일을 직접 넘기거나 이름을 정확히 알 필요 없이 자연스러운 대화로 지식을 활용할 수 있게 된다.

왜 기초가 먼저인가

AI 모델에 제공하는 컨텍스트를 늘리는 것은 응답 품질을 높이는 가장 확실한 방법이다. 저자의 세컨드 브레인에는 본인이 쓴 글, 웹 클리핑, 노트, 일일 로그, 프로젝트, AI와의 대화 중 남길 가치가 있는 것들이 모두 담겨 있다. Claude는 그 모든 것에 접근할 수 있고, 무엇이 있는지 알고, 필요한 것을 스스로 찾아 꺼낸다. 파일을 수동으로 올릴 필요도, 정확한 파일 이름을 알 필요도 없다.

그런데 이것이 첫 번째 시도에서 완성된 게 아니다. 수개월에 걸쳐 다양한 아키텍처와 도구 조합을 실험했고, 최근까지 같은 문제가 반복됐다. 바로 AI의 예측 불가능성이다. 돌파구는 볼트에 대한 명확한 규칙 집합과 관리 프로토콜을 정의하고 나서야 열렸다. 자신이 완전히 숙달한 후에야 AI 에이전트를 가르쳐 자동화할 수 있었다.

시스템은 담고 싶은 데이터를 충분히 담을 만큼 복잡하되, 실제로 꾸준히 유지할 수 있을 만큼 단순해야 한다.

AI를 투입하기 전에 볼트를 직접 관리하는 시간을 거치길 강력히 권한다. 토픽을 수동으로 만들고, 모든 노트에 태그를 달고, 링크를 연결하는 과정에서 작고 중요한 규칙들이 드러나기 때문이다. AI 에이전트에게 시스템을 가르치려면 먼저 자신이 그것을 속속들이 알아야 한다.

1. 에이전트 선택

커맨드라인 접근과 루틴 프로그래밍이 가능한 AI 에이전트라면 무엇이든 작동하지만, 대부분의 경우 Claude Code가 최선이다. Claude.ai 채팅이나 Claude Cowork는 샌드박스 외부 쉘 접근이 없어 적합하지 않다. 기술적으로 Cowork에서 볼트 폴더를 같은 위치에 두면 가능하지만, Claude Code가 어디서든 사용하기 편하고 기술 파일(skill)을 채팅에서 직접 수정할 수 있다는 장점도 있다.

2. 볼트 구조

Obsidian에서 폴더는 노트 간 연결에 영향을 주지 않는다. 그래프 모델은 폴더 계층을 고려하지 않기 때문이다. 그러나 에이전트가 Obsidian CLI를 통해 작업할 때는 폴더가 세 가지 측면에서 중요하다.

  • 범위 파악 — folders 명령으로 전체 볼트 구조를 빠르게 이해할 수 있다.
  • 성능 효율 — search 같은 명령을 특정 폴더로 제한해 속도를 크게 높일 수 있다.
  • 안전 격리 — 새 루틴을 실험할 때 에이전트의 작업 범위를 특정 폴더로 제한해 실수가 퍼지는 것을 방지할 수 있다.

저자가 채택한 최상위 폴더 구조는 다음과 같다.

  • 00 Daily Notes
  • 10 Raw — 웹 클리핑, 팟캐스트 스크립트, 원본 자료. 여기서는 직접 편집하지 않는다.
  • 20 Notes — 정제되고 간결한 노트. 원본이 아닌, 바로 활용 가능한 깔끔한 노트.
  • 30 Projects — 프로젝트별 페이지. 관련 원본과 노트를 sources 필드로 연결.
  • 97 Topics — 토픽 페이지 폴더.
  • 98 Assets — 미디어 파일 모음.
  • 99 Templates — 각 폴더용 템플릿. 매번 올바른 프론트매터 필드가 자동으로 준비된다.

하위 폴더 조직

하위 폴더는 의미론적이 아닌 체계적으로 구성하는 것을 권장한다. "미완성", "진행중", "완료" 같은 동적 폴더는 노트를 자주 이동시켜야 하므로 피하는 것이 좋다.

3. 링크 방식

프론트매터를 일관되게 유지하는 것이 세컨드 브레인의 생명을 유지하는 핵심이다. Templater 플러그인으로 폴더 템플릿을 설정하면 새 노트를 만들 때 올바른 필드가 자동으로 채워진다. 모든 노트에 적용하는 4가지 핵심 필드는 다음과 같다.

Types (유형)

최상위 폴더와 일치하는 정적 분류다. '/'로 하위 유형을 구분한다. 예: Raw/Article, Raw/Podcast, Raw/Research-Paper, Notes, Topic.

Topics (토픽)

토픽은 대체로 정적이지만 시간이 지나면서 확장된다. 하나의 토픽에 수백 개의 노트가 쌓이면 세분화할 때가 된 것이다. 이를 위해 별도의 토픽 폴더를 만들고, 각 토픽 페이지에 Up 필드로 상위 토픽을 연결한다. 예를 들어 'Security' 아래에 'data breaches', 'AI security', 'security tips' 같은 하위 토픽이 위를 가리키도록 한다.

토픽이 너무 포화 상태가 되면 에이전트에게 재조직을 맡긴다. 관련 노트를 찾고, 새 하위 토픽을 정의하고, 기존 템플릿을 바탕으로 토픽 페이지를 만들고, 각 노트를 가장 적합한 토픽으로 재배정하는 작업을 Claude가 수행한다. 저자는 최근 AI-news 토픽이 너무 많아지자 AI-companies, product-releases 하위 토픽을 만들고 재정리했다. 이 루틴은 직접 먼저 해보고 나서 에이전트 기술 파일로 자동화했다.

Tags (태그)

태그는 노트의 일시적 상태를 표시한다. 초안 중인 글에는 status/draft, 발행 후에는 status/published를 붙인다. 저자는 뉴스레터에 사용할 노트에 임시 태그 the-clique를 달았다가 뉴스레터 작성 후 제거한다. 이처럼 태그는 언제든 달고 뗄 수 있는 유동적 상태 표시자다.

Sources (출처)

현재 노트의 원본이 되는 노트를 링크로 연결한다. 예를 들어 작성한 글은 몇 가지 'Notes'를 출처로 가리키고, 각 노트는 원본 'Raw' 파일을 가리킨다. SNS 게시글은 해당 글 노트를 출처로 삼는다. 이렇게 링크에 방향성(하향)을 부여하면 무작위 연결 없이 어느 노트가 어느 노트를 링크해야 하는지 명확해지고, 새 노트를 만들 때만 링크를 추가하면 되므로 기존 노트로 돌아가 링크를 추가하는 수고도 없다.

본문 내 위키링크

프론트매터 외에도 본문 작성 중 다른 노트나 아이디어를 참조할 때는 Wikilink([[노트명]])를 활용한다. [[를 입력하고 키워드를 타이핑하면 자동완성으로 링크가 연결된다. 이렇게 구체적인 아이디어 간의 연결이 만들어진다.

4. 에이전트 접근 프로토콜 — Obsidian-CLI 기술 파일

저자는 자주 사용하는 기술 파일들과 CLAUDE.md를 하나의 폴더에 모아두고 Claude Code 작업의 대부분을 그 안에서 한다. Obsidian과 연동하기 위해 Obsidian-CLI 기술 파일을 만들었는데, 이를 통해 표준 파일 작업보다 훨씬 효과적으로 Obsidian의 내부 파일 인덱스와 검색 기능을 활용할 수 있다.

사용법은 간단하다. "볼트에서 원본 자료를 찾을 수 있어" 한마디면 충분하다. Claude가 그 세션에서는 볼트 위치를 인식하고 읽기/쓰기를 알아서 처리한다. 파일을 직접 올리거나 정확한 파일 이름을 알 필요가 없다. 이 기술 파일은 볼트 아키텍처나 복잡한 워크플로우 없이, 검색·파일 읽기·수정 같은 기본 작업만 처리하는 드라이버 역할을 한다. Obsidian CLI가 자주 업데이트되므로 주기적으로 Claude에게 문서를 리뷰하고 기술 파일을 최신 상태로 유지하도록 하는 것이 좋다.

5. 에이전트 볼트 인덱싱

볼트와 접근 프로토콜이 준비되면 CLAUDE.md 파일을 수정할 차례다. 먼저 에이전트가 볼트를 탐색하고 폴더와 주요 파일을 파악하게 한다. 저자의 경우 컨텍스트 파일, 브랜드 스타일 가이드, 현재 업무 목표와 집중 사항이 이에 해당한다. 에이전트가 이것들을 확인하고 메모리 파일에 기록하도록 한다. 새 파일이 추가될 때마다 이 인덱싱 과정을 주기적으로 재실행하면 된다.

6. 실제 활용 사례

콘텐츠 수집 및 일괄 작업

가끔 노트에 토픽을 빠뜨리는 경우가 있다. Claude에게 토픽 페이지 폴더에서 토픽 구조를 파악하고, 각 노트를 읽어 적절한 토픽을 결정하고, 해당 필드를 추가하도록 시키면 된다. 태그 리팩토링, 오래된 파일 보관, 노트 정리, 중복 제거 같은 일괄 작업도 마찬가지다.

일일 브리핑

Claude 예약 작업으로 날씨 예보, 최신 뉴스, 오늘 할 일을 정리해 매일 아침 일일 노트에 직접 작성하게 할 수 있다. 저자가 주기적으로 실험하는 기능으로, 생산적인 하루를 시작하는 데 도움이 된다.

최근 탐구 중인 것

Obsidian CLI의 개발자 기능 모음이 흥미롭다. Obsidian 런타임 내에서 JavaScript를 실행하고, 라이브 UI를 쿼리하고, Obsidian 창을 스크린샷하고, 명령 팔레트의 모든 명령을 프로그래밍 방식으로 트리거할 수 있다. 예를 들어 Claude에게 특정 노트를 찾아 Obsidian에서 새 탭으로 바로 열어달라고 하면 검색 없이 눈앞에 나타난다. 스크린샷 기능으로 Excalidraw 스케치를 열어 다이어그램으로 변환하거나, 설명된 워크플로우를 실행하는 것도 가능하다.

세컨드 브레인으로 학습하기

이 시스템의 가장 강력한 활용은 축적된 지식으로부터 직접 배우는 것이다. 팟캐스트, 읽은 글, 진행 중인 작업, 프로젝트 업데이트, 일일 로그와 성찰이 모두 담겨 있다면 AI 에이전트가 아이디어를 연결하고 패턴을 발견하는 데 큰 도움을 준다.

다음과 같이 물어보면 놀라운 결과를 얻을 수 있다.

"내 세컨드 브레인에 접근할 수 있어. 업무와 관심사에 관한 노트가 모두 링크로 연결되어 있어. 전부 읽어봐. 연결, 토픽, 링크를 살펴봐. 내가 여기서 무엇을 배울 수 있을까? 어떤 패턴이 보여? 다음에 무슨 프로젝트를 해야 할까?"

일상적인 작업도 훨씬 빨라진다. 자주 하는 작업 중 하나가 리서치 보고서 작성인데, 주제에 대한 간단한 노트가 이미 있으면 Claude를 그쪽으로 안내하고 리서치와 보고서 작성을 볼트에 직접 시킨다. 파일이 어디에 저장될지 걱정할 필요도, 형식을 지정할 필요도 없다. 이미 토픽이 태그되고, 원본 노트에 링크되고, 상태가 'to-read'로 설정된 채 대기 중이다.

파일을 채팅에 드래그할 필요도, 스타일 가이드를 매번 전달할 필요도, 예시 파일을 찾아 헤맬 필요도 없다. Claude가 이미 필요한 모든 것에 접근할 수 있기 때문이다. 이 설정이 일하는 방식 자체를 바꿔놓았다. 새 기술 파일을 추가할수록 기능이 늘어나고, 데이터가 쌓일수록 점점 더 강력한 자산이 된다. 그리고 그 모든 것은 기초를 제대로 갖추는 것에서 시작된다.
