Claude Code agents: what they actually are

Jose Parreño · Data Science Collective · 15분 읽기 · 원문 · freedium

Claude Code의 서브에이전트(subagent)가 단순한 프롬프트 변형이 아니라 독립적인 컨텍스트 창에서 실행되는 별도 실행 단위임을 설명한 실용 가이드.

요약

많은 개발자들이 Claude Code 에이전트를 단순히 "더 똑똑한 스킬" 정도로 오해하지만, 서브에이전트는 메인 대화와 완전히 분리된 자체 컨텍스트 창, 시스템 프롬프트, 도구 권한을 갖는 독립 실행 단위다. 서브에이전트가 존재하는 핵심 이유는 컨텍스트 관리에 있다. 무거운 탐색 작업을 메인 세션이 아닌 서브에이전트에서 처리하면 "컨텍스트 오염(context rot)"을 방지하고 주요 대화의 품질을 유지할 수 있다. .claude/agents/ 디렉터리에 마크다운 파일로 정의하며, YAML 프론트매터의 description 필드가 언제 이 에이전트를 호출할지 결정하는 라우팅 기준이 된다는 점이 핵심이다. 스킬은 메인 세션에서 Claude가 알아야 할 것을 정의하는 반면, 서브에이전트는 누가, 어디서, 어떤 도구로 작업하는지를 결정한다는 구분을 명확히 이해해야 한다.

들어가며

제 뉴스레터 파이프라인에서 처음으로 Claude가 서브에이전트를 생성하는 장면을 목격했을 때, 저는 그저 범위를 좁힌 프롬프트가 실행되는 거라고 생각했습니다. 수십 개 파일을 처리한 뒤 깔끔한 요약을 돌려주는 걸 지켜봤는데, 꽤 인상적이었습니다. 하지만 내부에서 무슨 일이 일어났는지는 전혀 이해하지 못했고, 그 찜찜함이 오래 남았습니다.

대부분의 사람들은 Claude Code 에이전트를 처음 접할 때 "스텝이 추가된 더 나은 스킬" 정도로 받아들입니다. 지침을 다르게 작성하기 때문에 더 자율적으로 동작한다고 생각하는 거죠.

그 이해는 틀렸습니다. 에이전트는 스킬과 비슷한 프롬프트 구조를 갖고 있지만, 결정적인 차이가 있습니다. 메인 대화 안에서 실행되지 않는다는 점입니다. 에이전트는 자체 컨텍스트 창에서, 자체 시스템 프롬프트와 도구 접근 권한, 독립적인 퍼미션을 가지고 동작하다가 결과만 반환합니다.

이 글은 에이전트를 밑바닥부터 설명합니다. 에이전트가 무엇인지, 어떻게 만드는지, 언제 투자할 가치가 있는지를 다룹니다. Claude Code를 이미 사용 중이고 이 기능을 제대로 이해한 뒤 활용하고 싶은 분들을 대상으로 합니다.

명칭 문제: "Claude Code 에이전트"란 무엇인가?

무엇보다 먼저 짚고 넘어가야 할 점이 있습니다. "에이전트"라는 단어는 Claude Code에서 최소 네 가지 서로 다른 의미로 쓰이며, 이것들을 뒤섞으면 기능 전체를 이해하기 어려워집니다. 다소 꼼꼼한 정리처럼 보일 수 있지만, 이 작업이 개념을 실제 사용 가능한 자산으로 매핑하는 데 도움이 되었습니다.

Claude Code 자체가 이미 에이전틱 환경입니다. Anthropic은 Claude Code를 에이전틱 코딩 어시스턴트로 설명합니다. 도구를 갖추고, 실행 루프를 돌며, 매번 프롬프트를 기다리지 않고도 여러 단계에 걸쳐 계획하고 행동할 수 있습니다. 이것이 기본 상태입니다. 아무것도 설정하지 않아도 이미 에이전틱입니다.

서브에이전트는 제품 내 커스텀 에이전트 기본 요소입니다. 실무자가 "Claude Code에서 에이전트를 만들고 있어"라고 말할 때 거의 항상 서브에이전트를 의미합니다. 공식 문서는 서브에이전트를 "특정 유형의 작업을 처리하는 특수 AI 어시스턴트"로 정의합니다. 각 서브에이전트는 자체 컨텍스트 창에서, 커스텀 시스템 프롬프트와 특정 도구 접근 권한, 독립적인 퍼미션을 가지고 실행됩니다.

에이전트 팀은 별도의 멀티 세션 기능입니다. 에이전트 팀 문서는 이 경계를 명확히 합니다. "서브에이전트는 단일 세션 안에서 동작하고, 에이전트 팀은 별도 세션 간에 조율합니다." 에이전트 팀은 공유 작업 목록과 직접 메시지를 통해 소통하는 독립적인 Claude Code 인스턴스들입니다. 실험적 기능이며 기본적으로 비활성화되어 있고, 환경 변수 설정이 필요합니다. "더 많은 서브에이전트"가 아니라 아키텍처의 다른 계층입니다.

Claude Agent SDK는 네 번째 계층입니다. Claude Code를 구동하는 동일한 도구와 에이전트 루프를 Python 및 TypeScript 라이브러리로 공개한 것입니다. CLI 바깥에 존재하며, 커스텀 에이전트 애플리케이션을 구축하는 팀을 위한 것입니다. 이 글의 주제는 아닙니다.

이 정의를 바탕으로, 이 글은 서브에이전트(위 2번)를 다룹니다. 에이전트 팀에 대한 글도 따로 쓰고 싶지만, 먼저 서브에이전트 하나를 제대로 파악하는 것이 우선입니다.

서브에이전트가 존재하는 이유

서브에이전트가 존재하는 주된 이유는 LLM 모델의 컨텍스트가 유한하기 때문입니다. 결국 어느 시점에서는 컨텍스트 한계에 부딪힙니다.

대응하지 않으면 이 문제는 반복해서 나타납니다. Anthropic의 컨텍스트 엔지니어링 관련 글은 이 문제를 정확하게 표현합니다.

"컨텍스트 엔지니어링은 제한된 컨텍스트 창에 무엇을 넣을지 선별하는 기술이자 과학이다."

도구 호출 하나, 파일 읽기 하나, 중간 분석 결과 하나 — 이 모든 것이 컨텍스트에 쌓이고 거기 머뭅니다. 세션이 길어질수록 창은 결과를 내기 위해 필요했지만 이제는 더 이상 참조할 필요 없는 중간 작업물로 가득 찹니다. 이 노이즈가 성능을 떨어뜨립니다(이를 "컨텍스트 오염(context rot)"이라고 하는데, 이 주제만으로 별도 글을 쓸 계획입니다). Claude가 작업할 공간이 줄어들고, 주의를 빼앗는 불필요한 내용이 늘어납니다.

서브에이전트는 바로 이 문제를 해결하기 위해 존재합니다.

공식 문서는 직접적으로 설명합니다. "서브에이전트는 수십 개의 파일을 읽거나 광범위한 검색을 수행할 수 있지만, 메인 대화에는 요약만 전달됩니다. 서브에이전트의 작업이 메인 컨텍스트를 소비하지 않기 때문에, 중간 과정을 메인 창에 남겨둘 필요가 없을 때도 유용합니다."

실제로 어떻게 작동하는지 살펴봅시다. 테스트 스위트가 실패하는 이유를 조사해 달라고 요청한다고 가정해 보세요. 메인 세션에서 이 조사를 진행하면 파일 20개 읽기, 에러 패턴 검색, 테스트 설정 확인, 최근 커밋 검토가 모두 컨텍스트에 쌓입니다. 조사가 끝날 때쯤이면 수천 토큰에 달하는 중간 작업물이 쌓이고, 대부분은 다음 단계와 무관합니다.

서브에이전트는 이 문제의 구조를 바꿉니다. 자체 컨텍스트 창에서 조사를 수행하고, 자체 토큰 예산을 사용합니다. 메인 대화에는 이런 결과만 돌아옵니다. "CI 설정에 환경 변수가 누락되어 테스트가 실패합니다. 영향받는 테스트 세 개와 수정 방법은 다음과 같습니다." 깔끔합니다. 메인 스레드는 검색 과정의 컨텍스트 오염을 전혀 흡수하지 않습니다.

컨텍스트 격리가 핵심 이유입니다. 여기에 더해 세 가지 부가적인 이유도 언급할 만합니다.

전문화. 서브에이전트는 자체 시스템 프롬프트를 가집니다. 팀의 코딩 기준에 특화된 코드 리뷰어는, 그 기준을 다른 모든 작업과 함께 유지하려는 범용 어시스턴트보다 훨씬 신뢰할 수 있습니다.

제약 강제. 서브에이전트가 사용할 수 있는 도구를 정확히 지정할 수 있습니다. 파일을 쓰거나 셸 명령을 실행하지 못하는 읽기 전용 리서치 에이전트는 메인 어시스턴트와 실질적으로 다른 위험 프로파일을 갖습니다. 제약이 선언되고 강제됩니다.

비용 제어. 서브에이전트는 다른 모델에서 실행할 수 있습니다. 메인 세션은 Sonnet으로 유지하면서 가벼운 작업을 Haiku로 처리하는 것은, 청구서를 보고 나서야 발견하게 되는 실용적인 비용 조절 방법입니다.

첫 번째 서브에이전트 만들기

서브에이전트는 YAML 프론트매터가 포함된 마크다운 파일입니다.

두 곳 중 한 곳에 저장합니다.

프로젝트 디렉터리의 .claude/agents/ — 특정 코드베이스에 한정된 에이전트. 팀이 공유하고 개선할 수 있도록 버전 관리에 포함시키세요.

홈 디렉터리의 ~/.claude/agents/ — 모든 프로젝트에서 사용 가능한 개인 에이전트.

YAML 프론트매터 살펴보기

서브에이전트에서 사람들이 주의를 덜 기울이는 부분이 바로 YAML 프론트매터입니다. 이 프론트매터는 Claude Code가 메인 세션의 메모리에 보관하는 메타데이터로, 에이전트가 어떤 역할을 하는지 파악하는 데 사용됩니다. 즉, 에이전트의 요약본 역할을 합니다.

필수 필드는 2개이고, 선택 필드도 몇 가지 있습니다. 간단한 예제로 살펴봅시다.

---
name: code-reviewer
description: Reviews code changes for quality, security, and best practices.
Use proactively after any significant code change.
tools: Read, Glob, Grep
model: sonnet
---
You are a code reviewer. When invoked, read the changed files and provide
specific, actionable feedback on code quality, security, and best practices.
Focus on what matters. Don't restate what the code does — say what's wrong
or good about it. Flag security issues first, then logic problems, then style.

각 부분이 무엇을 하는지 살펴봅시다.

name은 고유 식별자입니다. 소문자와 하이픈만 사용합니다. 로그에서 에이전트를 구분하고 Claude가 내부적으로 참조하는 이름입니다.

description은 가장 중요한 필드입니다. Claude는 이 설명을 읽고 작업을 이 에이전트에 위임할지 결정합니다. 행동 중심으로 작성하세요. 에이전트가 무엇을 하는지, 언제 사용해야 하는지를 담으세요. "use proactively"라는 표현은 공식 문서에서도 언급된 신호로, 명시적 요청 없이도 Claude가 이 에이전트를 활성화하도록 유도합니다. 직접 요청할 때만 실행되길 원한다면 이 표현을 제외하세요.

tools는 이 에이전트가 접근할 수 있는 도구를 제한합니다. 이 리뷰어는 파일을 읽고 검색할 수 있지만, 편집하거나 셸 명령을 실행하거나 네트워크 호출을 할 수 없습니다. 의도적인 제약입니다. 읽기 전용 리뷰어는 자신이 리뷰하는 대상을 실수로 수정할 수 없습니다. 공식 문서는 에이전트가 실제로 필요한 도구만 부여하도록 명시적으로 권고합니다. 보안뿐 아니라 동작 예측 가능성을 위해서도 중요합니다.

model은 특정 모델로 라우팅합니다. 선택지는 sonnet, opus, haiku, 또는 claude-opus-4-6과 같은 전체 모델 ID입니다. 생략하면 메인 세션과 동일한 모델을 상속합니다.

프론트매터 아래의 본문은 에이전트의 시스템 프롬프트가 됩니다. 에이전트는 이 내용과 작업 디렉터리만 메인 세션으로부터 받습니다.

선택 가능한 프론트매터 필드는 더 많습니다. permissionMode, maxTurns, skills, mcpServers, effort, memory, hooks, disallowedTools, color 등이 있습니다.

디렉터리 구조

위 예시를 프로젝트 안에 .claude/agents/code-reviewer.md로 저장합니다. 구조는 다음과 같습니다.

your-project/
├── .claude/
│   ├── agents/
│   │   └── code-reviewer.md   ← 새 에이전트 파일
│   ├── skills/
│   └── rules/
├── src/
├── tests/
├── README.md
└── package.json

파일을 저장하면 Claude가 에이전트를 감지합니다. 코드 변경 사항을 리뷰해 달라고 요청하면, 설명 내용과 일치한다고 판단할 때 자동으로 code-reviewer에 위임합니다.

모든 프로젝트에서 사용할 개인 에이전트라면 ~/.claude/agents/code-reviewer.md로 옮기면 됩니다. 형식과 내용은 동일하고 위치만 다릅니다.

에이전트 vs 스킬: 모두가 혼동하는 지점

왜 헷갈리는가

서브에이전트와 스킬을 혼동하는 이유는 구조가 매우 비슷하기 때문입니다.

둘 다 간단한 디렉터리 구조로 설정합니다. .claude/skills 또는 .claude/agents 안에 파일을 두면 됩니다. 둘 다 프론트매터 구조를 사용하며, description이 가장 중요한 설정 값입니다. 둘 다 프롬프트 형태의 지침 집합입니다. 같은 내용의 프롬프트를 스킬과 에이전트 각각에 넣으면 동일한 작업을 수행할 수 있습니다.

작업이 일어나는 곳

하지만 핵심적인 차이가 하나 있습니다. 바로 작업이 이루어지는 장소, 다시 말해 토큰이 소비되는 위치입니다.

스킬은 현재 대화에 로드됩니다. 지침, 템플릿, 참조 파일이 활성 컨텍스트의 일부가 되어 같은 스레드에서 진행되며, 모든 단계가 메인 창에 누적됩니다.

서브에이전트는 별도의 컨텍스트 창에서 실행됩니다. 작업을 완료한 뒤 메인 대화에 결과만 전달합니다.

또 다른 차이: 튜닝 매개변수

앞서 에이전트의 프론트매터에 설정할 수 있는 매개변수 목록을 살펴봤는데, 꽤 깁니다. 반면 스킬은 훨씬 제한적인 목록을 가지고 있습니다. 이것만 봐도 에이전트가 스킬보다 얼마나 더 세밀하게 제어 가능한지 알 수 있습니다.

구체적인 예시

팀의 코드 리뷰 기준을 강제 적용하고 싶다고 해봅시다.

세션 전반에 걸쳐 Claude가 그 기준을 알고 있기를 원한다면 — 코드를 작성할 때 적용하고, 제안할 때 참조하고, 위반 사항을 인라인으로 표시해 주기를 원한다면 — 이것은 스킬의 영역입니다. 기준을 한 번 작성해 두면 Claude가 관련 상황에서 컨텍스트에 로드합니다.

반면, 메인 컨텍스트를 채우지 않고 파일 40개를 한 번에 리뷰하고 싶다면 이것은 서브에이전트의 영역입니다. code-reviewer 에이전트를 설정하면, 자체 컨텍스트 창에서 리뷰를 수행하고 구조화된 요약만 반환합니다. 40개 파일 각각에 서브에이전트를 병렬로 생성할 수도 있습니다. 메인 대화에는 결과만 남고, 그 결과를 만들어낸 과정은 남지 않습니다.

한 문장으로 정리하면 이렇습니다.

스킬은 Claude가 무엇을 알거나 어떻게 접근해야 하는지를 정의합니다. 서브에이전트는 누가, 어디서, 어떤 도구로 작업하는지를 정의합니다.

서브에이전트 실전 팁

description은 라우팅 인터페이스다

이 부분은 보통 강조되는 것보다 훨씬 중요합니다.

Claude는 서브에이전트의 description을 읽고 위임 여부를 결정합니다. 본문은 결정 전에 읽지 않습니다. 즉, 본문에 세부 내용을 작성해도 서브에이전트가 실제로 생성되기 전까지는 Claude Code가 그것을 볼 수 없습니다. 그래서 description이 언제 이 서브에이전트가 필요한지 Claude에게 명확히 알려주어야 합니다.

구체적이고 행동 중심으로 작성하세요. 에이전트가 무엇을 하는지, 언제 사용해야 하는지 설명하세요. 잘 처리하는 작업 유형과 처리하지 않아야 할 유형도 포함하면 좋습니다. "코드 변경 후 자동으로 실행하세요"라는 표현은 검증된 패턴입니다. 문서에서도 명시적 요청 없이 위임을 유도하는 방법으로 언급됩니다. 직접 호출할 때만 실행되길 원한다면 이 표현을 빼세요.

피해야 할 것은 에이전트의 정체성을 설명하는 description입니다("당신은 20년 경력의 코드 리뷰 전문가입니다…"). 그런 내용은 시스템 프롬프트 본문에 들어가야 합니다. description은 에이전트 자신을 위한 것이 아니라 라우팅 모델을 위한 것입니다.

구체적인 대비 예시를 보면 이렇습니다.

너무 모호한 description: "코드 품질 및 리뷰 작업을 지원합니다."

라우팅에 적합한 description: "보안 문제, 로직 버그, 스타일 위반에 대해 스테이징된 코드 변경 사항을 리뷰합니다. 중요한 편집 후에는 자동으로 실행하세요. 코드를 생성하거나 파일을 편집하지 않습니다."

도구는 의도적으로 제한하라

메인 세션의 모든 도구를 상속받은 에이전트는, 문제가 발생했을 때 원인을 추론하기 어려운 에이전트입니다. 작업에 실제로 필요한 도구만 부여하세요.

코드 리뷰어에게는 읽기 접근만 필요합니다. 쓰기 접근, 셸 실행, 네트워크 호출은 필요 없습니다. 리서치 에이전트에게는 검색과 가져오기가 필요합니다. 파일 편집은 불필요합니다. 테스트 실행 에이전트에게는 Bash 실행이 필요합니다. 웹 접근은 필요 없습니다.

도구 제한은 보안 관행이자 명확성 관행입니다. 좁은 도구 집합은 에이전트의 동작을 예측 가능하게 만듭니다. 에이전트가 실패하거나 예상치 못한 출력을 내보낼 때, 가능한 도구 목록이 짧을수록 진단이 빠릅니다. 그리고 실패 가능한 경우의 수가 유한하고 파악 가능해집니다.

disallowedTools 필드를 사용하면 메인 세션의 도구 집합을 상속받은 뒤 특정 도구만 제거할 수 있습니다. "Bash만 빼고 전부" 또는 "Write만 빼고 전부" 같은 설정이 필요할 때 유용합니다.

에이전트 하나에 작업 하나

서브에이전트 문서는 직접적으로 이야기합니다. 각 서브에이전트는 한 가지 특정 작업에 탁월해야 합니다. "코드를 리뷰하고, 테스트를 작성하고, 문서를 업데이트하고, 배포 검사를 처리합니다"라는 description을 가진 에이전트는 네 가지 설명이 하나의 필드에서 경쟁하는 꼴입니다. Claude가 일관성 없이 라우팅할 수 있고, 시스템 프롬프트는 네 가지 작동 모드를 동시에 처리하려 할 것입니다.

범위를 좁히세요. 하나의 명확한 설명을 가진 세 개의 별도 파일로 만드세요.

데이터 사이언스 맥락에서는 모델 검증 에이전트, 데이터 품질 검사 에이전트, 노트북 리뷰어를 따로 만드세요. 소프트웨어 개발 맥락에서는 코드 리뷰어, 테스트 생성기, 문서 업데이터를 별도로 두세요. 도메인에 관계없이 원칙은 같습니다. 한 가지 역할, 하나의 파일, 명확하게 라우팅되는 하나의 description.

model 필드를 비용 조절 수단으로 활용하라

이것은 청구서를 보고 나서야 발견하는 실무자들이 가장 많이 간과하는 필드입니다.

여러 에이전트가 세션에서 실행되는 경우, 해당 에이전트에 model: haiku를 설정하면 비용이 상당히 줄어듭니다. 긴 컨텍스트에 걸쳐 지속적인 추론이 필요한 에이전트에는 model: opus를 사용하면 비용을 정당화할 수 있습니다. 기본값인 inherit는 모든 에이전트가 메인 세션과 동일한 모델(대체로 가장 강력한 모델)에서 실행되도록 합니다. 그 수준의 성능이 필요하지 않은 작업에도 동일하게 적용되는 셈입니다.

커뮤니티의 합의

VoltAgent의 서브에이전트 컬렉션에는 언어 전문가, 보안 리뷰어, 인프라 에이전트, 디버거, 오케스트레이터 등 100개가 넘는 전문 서브에이전트가 정리되어 있습니다. Everything Claude Code 저장소에도 훌륭한 에이전트 컬렉션이 있습니다.

100개가 넘는 에이전트를 모두 읽은 건 아니지만 이름들을 훑어보면서 느낀 것은, 그 중 20%도 필요하지 않을 것 같다는 점입니다. 따라서 에이전트의 수가 아니라 구조에 주목해야 합니다.

컬렉션의 모든 에이전트는 좁은 범위를 가집니다. TypeScript 전문가. SQL 전문가. 보안 감사자. Kubernetes 운영자. 하나의 도메인을 넘으려는 에이전트는 없습니다.

1년 이상의 실제 사용 끝에 실무자들이 내린 결론은 하나입니다. 범용 에이전트 하나보다 전문화된 에이전트 여럿이 더 낫다는 것입니다.

에이전트 팀 간략 소개

에이전트 팀은 알아둘 가치가 있지만, 서브에이전트와는 다른 기능입니다.

서브에이전트는 단일 세션 안에서 동작합니다. 메인 Claude Code 인스턴스가 서브에이전트를 생성하고, 서브에이전트는 격리된 환경에서 작업을 수행한 뒤 결과를 반환합니다. 오케스트레이션은 여러분이 합니다.

에이전트 팀은 공유 작업 목록과 직접 메시지를 통해 서로 소통하는 독립적인 Claude Code 세션들입니다. 각 팀원은 자체 컨텍스트 창과 세션 상태를 가지며, 다른 팀원에게 작업을 할당하거나 받을 수 있습니다.

에이전트 팀 문서는 효과적인 상황을 이렇게 설명합니다. 여러 팀원이 서로 다른 각도에서 동시에 조사하는 리서치 및 리뷰, 서로의 가설에 이의를 제기하는 경쟁적 디버깅, 각 팀원이 충돌 없이 독립된 컴포넌트를 담당하는 새 모듈 개발 등입니다.

이 글에서 에이전트 팀을 깊이 다루지 않겠습니다. 설정이 간단하지 않고, FOMO 때문에 에이전트 팀으로 뛰어들 필요는 없습니다. 에이전트 팀은 조율 오버헤드를 추가하고 토큰을 훨씬 많이 소비합니다. 각 팀원이 독립적인 컨텍스트 창을 가지며 별도로 청구됩니다. 실험적 기능이며 기본적으로 비활성화되어 있어 환경 변수 설정이 필요합니다.

실용적인 기준은 간단합니다. 서브에이전트들이 여러분에게 결과를 보고하고 서로 직접 소통할 필요가 없다면, 에이전트 팀은 필요 없습니다. 한 서브에이전트가 다른 서브에이전트에게 결과를 전달하거나, 상대방의 결론에 이의를 제기해야 하는 순간이 오면 그때 업그레이드를 고려하세요. 그 전까지는 서브에이전트로 충분합니다.

에이전트가 적합하지 않은 경우

Anthropic의 자체 가이드는 진지하게 받아들일 만합니다. "가능한 가장 단순한 해결책을 찾고, 필요할 때만 복잡성을 높이세요. 이 말은 에이전틱 시스템을 전혀 구축하지 않는 것을 의미할 수도 있습니다."

이것은 서브에이전트에도 직접 적용됩니다. 서브에이전트가 오히려 번거로움만 더하는 상황이 있습니다.

일회성 작업. 서브에이전트는 명칭 설계, description 작성, 도구 선택, 테스트, 유지보수 등의 초기 투자를 요구합니다. 한 번만 할 작업에는 그런 투자가 필요 없습니다. 좋은 프롬프트 하나로 충분합니다. 서브에이전트의 투자는 해당 워크플로우가 정의를 유지보수할 만큼 자주 반복될 때만 회수됩니다.

컨텍스트가 문제가 아닐 때. 서브에이전트를 사용하는 주된 이유는 무거운 작업으로부터 메인 컨텍스트를 보호하는 것입니다. 작업이 가볍고, 컨텍스트 창이 차지 않으며, 특별한 도구 제한도 필요 없다면 서브에이전트는 불필요한 간접 계층만 추가합니다. 메인 세션에서 잘 작성된 프롬프트가 더 빠르고, 더 단순하고, 디버깅하기 더 쉽습니다.

모든 것을 한 에이전트에 담으려는 함정. 리서치, 구현, 리뷰, 문서화를 모두 처리하는 에이전트는 포괄적이지 않습니다. 테스트할 수 없고, 일관성 없이 선택됩니다. description이 짧고 구체적일수록 라우팅이 더 안정적입니다.

일반 원칙은 이렇습니다. 실제 문제를 해결하는 가장 단순한 도구를 선택하세요. 컨텍스트 격리, 특수한 제약, 별도 실행이 구체적으로 필요할 때 에이전트를 사용하세요. 에이전트가 더 강력해 보이기 때문이 아니라요.

마치며

서브에이전트는 컨텍스트 관리 도구이자 실행 경계입니다. 그러나 대부분의 사람들은 이를 기능 업그레이드로 접근합니다.

더 나은 관점은 이것입니다. Claude가 올바른 작업을 올바른 장소에서 수행하게 만드는 방법. 스킬은 Claude가 무엇을 알아야 하는지를 정의합니다. 서브에이전트는 누가, 어디서, 어떤 제약 속에서 작업하는지를 정의합니다.

신중하게 설정된 에이전트 하나는 오후 시간 정도의 투자로 만들 수 있습니다. 오케스트레이션 시스템이 필요하지 않아도 이 기능에서 가치를 얻을 수 있습니다. 명확한 description, 제한된 도구 집합, 단 하나의 명확한 역할. 그로부터 복잡성을 키울 수 있지만, 처음부터 그렇게 시작할 필요는 없습니다.
