9 RAG Architectures Every AI Developer Must Know: A Complete Guide

Naren · 9 min read · 원문 · freedium

바닐라 RAG만으론 부족하다 — 문제 유형에 맞는 9가지 RAG 아키텍처를 알아야 진짜 AI 엔지니어다.

요약

RAG(검색 증강 생성)는 단일 기법이 아니라 각기 다른 문제에 최적화된 아키텍처 패밀리다. 대부분의 튜토리얼이 '바닐라 RAG'만 가르치고 끝내지만, 이는 소금만으로 요리를 배우는 것과 같다. 이 가이드는 입문·중급·고급 단계로 나뉜 9가지 RAG 아키텍처(Standard RAG, DeepRAG, MA-RAG, Speculative RAG, Corrective RAG, Fusion RAG, RAG-Gym, Modular RAG, SAM-RAG)를 각각 언제 사용해야 하는지 명확한 판단 기준과 함께 설명한다. 핵심 원칙은 간단하다: 단순하게 시작하고, 어디서 망가지는지 측정하고, 그때 의도적으로 업그레이드하라.

RAG 현실 점검

RAG(검색 증강 생성)가 AI 애플리케이션의 환각을 막는 비법이라는 말을 들어봤을 것이다. 맞는 말이다 — 어느 정도는.

문제는? 대부분의 튜토리얼은 '바닐라 RAG'만 가르치고 끝낸다. 하지만 바닐라 RAG는 소금만으로 요리를 배우는 것과 같다. 단순한 요리에는 통하지만, 좀 더 정교한 것이 필요한 순간, AI가 계속 반쯤 익은 답변을 내놓는 이유를 알 수 없어 막막해진다.

실제로는 최소 9가지의 뚜렷한 RAG 아키텍처가 있으며, 각각 서로 다른 문제를 위해 설계되었다. 잘못된 것을 고르는 것은 액자를 걸기 위해 대형 망치를 쓰는 것과 같다 — 기술적으로 가능하지만 고통스럽게 비효율적이다.

각 아키텍처, 언제 사용할지, 올바르게 구현하는 방법을 하나씩 살펴보자.

입문 단계

1. Standard RAG — 기반

흐름: 사용자 쿼리 → 쿼리 처리 → 검색 + 문서 선택 → 컨텍스트 통합 → LLM 응답

이것이 기본 RAG다. 사용자가 질문하면 벡터 데이터베이스에서 관련 문서를 검색하고, 컨텍스트에 넣은 뒤, LLM이 답변을 생성한다.

마치 사서가 질문을 받으면 가장 관련성 높은 책을 찾아와 딱 맞는 페이지를 열어 답을 읽어주는 것과 같다. 단순하고, 효과적이고, 신뢰할 수 있다.

사용 시기:

  • 정확도는 중요하지만 복잡도가 낮은 단순 사실 검색
  • 최신 회사 정책이나 지식 베이스 항목을 표시하는 FAQ 봇
  • MVP 또는 개념 증명 구축 시
  • 지식 베이스가 비교적 균일할 때

팁: 더 작은 청크 크기(256~512 토큰)로 시작하고 오버랩을 실험하여 컨텍스트 창을 압도하지 않으면서 커버리지를 개선하라. 많은 사람들이 청크를 너무 크게 만드는 실수를 한다 — 결국 신호와 함께 관련 없는 노이즈를 검색하게 된다.

한계: Standard RAG는 쿼리가 검색 가능한 문서에 명확하게 매핑된다고 가정한다. 다단계 추론이나 여러 출처의 정보가 필요한 순간 금이 가기 시작한다.

중급 단계

2. DeepRAG — 계층적 분해 + 프로세스 감독

흐름: 사용자 쿼리 → 질문 분해 → 하위 질문별 검색 → 추론 + 검증 → 최종 응답

DeepRAG는 근본적인 진실을 인식한다: 복잡한 질문은 단일체가 아니라 — 실제로는 트렌치코트를 입은 여러 질문이다.

탐정이 사건을 조사하는 것을 상상해보라. "누가 한 거야?"라고만 묻지 않는다. 분해한다: 동기는 누구에게 있었나? 기회는? 수단은? 그 당시 각자는 어디에 있었나? DeepRAG도 마찬가지다 — 쿼리를 하위 질문으로 분해하고, 각각의 정보를 검색한 뒤, 결과를 종합한다.

사용 시기:

  • 구조적 분해가 필요한 다중 홉 또는 도메인별 질문
  • UMLS 같은 연결된 개념 데이터베이스에 대한 생의학 쿼리
  • 여러 지식 영역에 걸쳐 점들을 연결해야 하는 시나리오

실제 예시: "신장 질환 환자에서 메트포르민과 상호작용하는 약물은?"이라는 쿼리를 생각해보라. 이것은 하나의 질문이 아니다 — 약물 상호작용, 신장 기능, 금기 사항을 모두 이해해야 한다. DeepRAG는 각 측면을 별도의 검색 작업으로 처리한다.

3. MA-RAG — 다중 에이전트 RAG

흐름: 사용자 쿼리 → 여러 전문 에이전트(계획자, 추출자, 검색자, 합성자) → 결합된 검색 & 추론 → LLM 출력

하나의 모놀리식 검색 시스템 대신, MA-RAG는 전문화된 에이전트 팀을 배치한다. 각 에이전트는 하위 작업을 처리하고, 쿼리당 필요한 에이전트만 호출된다.

병원을 생각해보라. 모든 것을 하는 의사 한 명이 아니라 전문의들이 있다. 트리아지 간호사가 상태를 평가하고, 적절한 전문의에게 보내고, 다른 전문의와 협의하여 치료 계획을 결정한다. MA-RAG가 이와 같다.

사용 시기:

  • 분해와 에이전트 협업이 도움이 되는 모호하거나 복잡한 쿼리
  • 이질적인 출처에 걸친 개방형 QA
  • 회사 위키, Slack 대화, 외부 문서 모두에서 가져오는 질문

팁: 에이전트가 해석 가능성을 위해 체인 오브 쏘트 프롬프팅(추론 흔적)을 사용하게 하라. 문제가 생겼을 때 어느 에이전트가 실수했는지 알아야 한다. 블랙박스를 디버깅하는 것은 전문가들 간의 대화를 추적하는 것보다 훨씬 어렵다.

4. Speculative RAG — 속도와 정확도

흐름: 사용자 쿼리 → 소형 모델 초안 → 병렬 검색 → 대형 모델 검증 → 최종 응답

전문가들이 실제로 일하는 방식을 반영하는 영리한 트릭이다: 빠른 소형 모델이 초안 답변을 빠르게 생성하는 동안 대형 모델이 검색된 문서와 대조하여 초안을 검증하는 병렬 검색을 실행한다.

주니어 애널리스트가 즉시 예비 답변 초안을 작성하는 동안 시니어 팀이 지원 증거를 모으는 것과 같다. 증거가 도착할 때쯤, 대조할 구체적인 것이 있다.

사용 시기:

  • 응답 속도가 중요하지만 정확도를 희생할 수 없을 때
  • 제품 세부 사항을 확인하면서 빠른 초안 답변이 필요한 이커머스 라이브 채팅 에이전트
  • 사용자가 기다리기 싫어하지만 틀린 답변은 더 싫어하는 시나리오

고급 단계

5. Corrective RAG — 신뢰하되 검증하라

흐름: 사용자 쿼리 → 초기 응답 → 오류 감지 → 검색 → 응답 수정 → 최종 응답

Corrective RAG는 초기 응답 후에 검증 루프를 추가한다. 본질적으로 시스템이 제출하기 전에 자신의 숙제를 확인하는 것이다.

모든 기사를 출판 전에 읽고, 주요 출처에 대해 주장을 팩트 체크하며, 검증을 통과하지 못한 것은 모두 표시하는 꼼꼼한 편집자가 있는 것과 같다.

사용 시기:

  • 틀리면 결과가 있는 고위험 시나리오
  • 신뢰할 수 있는 의학 문헌과 교차 검증하는 임상 의사결정 지원 어시스턴트
  • 정확도가 협상 불가능한 법률 또는 규정 준수 애플리케이션

팁: 가벼운 검증 루프를 사용하라 — 별도의 분류기 또는 보조 LLM — 불확실한 세그먼트를 표시하고 수정 검색을 트리거하도록. 모든 것을 검증할 필요는 없다. 위험한 부분에만 집중하라.

6. Fusion RAG — 합성기

흐름: 사용자 쿼리 → 다중 쿼리 생성 → 다중 검색 → 검색된 문서 융합 → LLM 응답

Fusion RAG는 원래 쿼리가 찾고 있는 것을 찾는 최선의 방법이 아닐 수 있다는 것을 인정한다. 쿼리의 여러 변형을 생성하고, 각 변형에 대한 문서를 검색한 다음, 답변 생성 전에 모든 결과를 융합한다.

"기후 변화 영향"만 검색하는 것이 아니라 "지구 온난화 영향", "온도 상승 결과", "환경 악화 결과"도 검색하는 연구원과 같다. 각 표현이 다른 용어를 사용하는 다른 논문을 표면화할 수 있다.

사용 시기:

  • 다양한 유형의 출처나 형식에서 증거가 필요한 쿼리
  • 저널 기사, 특허, 기술 블로그를 통해 합성하는 학술 연구 어시스턴트
  • 용어가 현저히 다른 도메인

팁: 상호 순위 융합(Reciprocal Rank Fusion)과 같은 융합 방법을 사용하고, 신뢰성과 도메인 관련성에 따라 출처에 가중치를 부여하라. 랜덤 블로그 게시물이 동료 검토 논문과 같은 가중치를 가져서는 안 된다.

7. RAG-Gym / 프로세스 감독 에이전틱 RAG

흐름: 사용자 쿼리 → 에이전틱 계획/추론 → 단계별 검색/검색 에이전트 → LLM 생성 + 프로세스 감독

이 아키텍처는 최종 답변 감독만이 아닌 프로세스 수준 보상을 사용하여 에이전트를 훈련시킨다. 학생들을 정답 여부만이 아닌 그들의 작업 과정으로 채점하는 것과 같다.

전통적인 훈련은 이렇게 말한다: "프랑스의 수도로 파리를 얻었나? 좋아." 프로세스 감독은 이렇게 말한다: "먼저 이것이 지리 질문임을 파악했나? '프랑스 수도'를 검색했나? 여러 출처로 검증했나? 그런 다음 답변을 작성했나? 좋아."

사용 시기:

  • 중간 단계 감독이 도움이 되는 다단계 추론 또는 작업
  • 원샷 검색보다 검색 경로를 계획해야 하는 QA 시스템
  • 인용 체인 추적이 필요한 복잡한 연구 쿼리

팁: 프로세스 수준 보상으로 에이전트를 훈련시켜라. 최종 답변이 맞는지가 아니라 어떤 검색 또는 재구성 단계가 더 나은지에 대해 보상하라. 이는 시스템이 무엇을 찾을지가 아니라 어떻게 검색할지를 배우도록 돕는다.

8. Modular RAG — 레고 접근법

흐름: 사용자 쿼리 → 모듈 라우팅/스케줄링 → 모듈을 통한 검색/재작성/융합 → LLM 생성

Modular RAG는 RAG 파이프라인을 교환 가능한 연산자로 분해한다: 라우팅, 쿼리 재작성, 융합, 재랭킹 등. 전체 시스템을 재구축하지 않고 구성 요소를 독립적으로 교체할 수 있다.

레고 블록으로 건물 짓는 것을 생각해보라. 이 검색기가 마음에 안 든다고? 교체하라. 새 재랭커를 실험하고 싶다고? 꽂아라. 인터페이스는 동일하게 유지된다; 구현만 바뀐다.

사용 시기:

  • RAG 파이프라인의 부분을 커스터마이징하거나 교체하는 유연성이 필요할 때
  • 새로운 검색/융합 전략을 탐구하는 연구 시스템
  • 검색 로직을 건드리지 않고 다른 재랭커를 A/B 테스트할 때

팁: Modular RAG 논문은 선형, 조건부, 분기, 루프 흐름과 같은 패턴을 식별한다. 선형으로 시작하고, 필요할 때만 복잡도를 추가하라. 섣부른 최적화가 모든 악의 근원이라면, 섣부른 복잡도는 그 사촌이다.

9. Self-Adaptive Multimodal RAG (SAM-RAG)

흐름: 사용자 쿼리(텍스트/멀티모달) → 모달리티 전반의 적응형 검색 → 필터링 & 품질 확인 → LLM 응답

SAM-RAG는 쿼리를 기반으로 어떤 모달리티(텍스트, 이미지, 오디오)에서 검색할지를 동적으로 선택한다. 텍스트만 가정하지 않고 적응한다.

책만 가져오는 것이 아니라 관련 사진도 가져오고, 오디오 녹음을 재생하고, 질문에 답하는 데 도움이 될 때 동영상을 보여주는 사서를 상상해보라. SAM-RAG가 정확히 그렇게 한다.

사용 시기:

  • 이미지, 오디오, 혼합 모달리티를 포함하는 쿼리나 데이터
  • 텍스트 + 이미지를 결합하는 QA, 예를 들어 사진이 있는 제품 설명
  • 다이어그램이 있는 기술 문서

팁: 어떤 모달리티에서 검색할지를 동적으로 선택하고, 검색된 멀티모달 데이터의 관련성을 확인하고, 컨텍스트에서 통합하라. 모델은 쿼리당 검색 수와 모달리티 선택을 적응시켜야 한다 — 모든 질문에 이미지가 필요한 것은 아니다.

아키텍처 선택: 의사결정 프레임워크

아직도 어떤 것을 선택할지 모르겠다고? 실용적인 결정 트리가 있다:

다음과 같다면 Standard RAG로 시작하라:

  • 쿼리가 단순하고 직접적일 때
  • MVP나 개념 증명을 구축할 때
  • 지식 베이스가 비교적 균일할 때

다음과 같다면 DeepRAG 또는 MA-RAG로 이동하라:

  • 질문이 여러 추론 단계를 필요로 할 때
  • 도메인별 복잡도를 다룰 때
  • 단일 검색이 충분하지 않을 때

다음과 같다면 Corrective 또는 Speculative RAG를 고려하라:

  • 정확도가 협상 불가능할 때 (의료, 법률, 금융)
  • 속도가 중요하지만 정확도를 희생할 수 없을 때
  • 사람과 같은 "두 번 확인" 동작이 필요할 때

다음과 같다면 Fusion 또는 Modular로 가라:

  • 다양한 출처 유형에서 가져올 때
  • 실험하고 반복하는 유연성이 필요할 때
  • 팀이 재구축 없이 구성 요소를 교체하길 원할 때

다음과 같다면 SAM-RAG를 수용하라:

  • 데이터에 이미지, 오디오, 비디오가 포함될 때
  • 사용자가 모달리티에 걸친 질문을 할 때
  • 텍스트 전용 검색이 가치를 놓치고 있을 때

결론

RAG는 단일 기법이 아니다 — 속도, 정확도, 복잡도, 유연성 간의 서로 다른 트레이드오프에 최적화된 아키텍처 패밀리다.

팀들이 저지르는 가장 큰 실수는 Standard RAG를 이미 넘어섰음에도 그것을 고수하는 것이다. 실제로 근본적으로 다른 아키텍처가 필요한데 프롬프트와 청킹 전략만 계속 조정한다. 레이싱 스트라이프를 추가하여 세단을 빠르게 만들려는 것과 같다 — 스포츠카가 필요한데.

단순하게 시작하라. 어디서 망가지는지 측정하라. 그런 다음 의도적으로 업그레이드하라.

RAG 시스템은 문제의 복잡도와 맞아야 한다 — 그 이상도 이하도 아니다.
