The Complete Claude Architect Study Guide (With Code and Tutor Prompts)

Paolo Perrone · Data Science Collective · 19 min read · 원문 · freedium

Claude Certified Architect 시험이 다루는 다섯 개 영역을, 실제 돌아가는 코드와 흔한 실수, 그리고 Claude를 과외 선생처럼 쓰는 프롬프트까지 곁들여 통째로 파헤친 실전 가이드.

요약

저자 Paolo Perrone은 MCP 서버와 Claude Code, 멀티 에이전트 파이프라인을 직접 만들며 숱하게 깨져 본 경험을 토대로 Claude Certified Architect 시험 가이드를 처음부터 끝까지 분해했다. 시험 자체는 Anthropic 파트너사만 응시할 수 있지만 가이드는 공개돼 있고, 프로덕션에서 Claude를 다룰 때 실제로 필요한 다섯 개 영역(에이전트 아키텍처 27%, 도구 설계 18%, Claude Code 설정 20%, 프롬프트 엔지니어링 20%, 컨텍스트 관리 15%)을 다룬다. 핵심은 에이전트 루프에서 stop_reason만이 믿을 수 있는 종료 신호라는 점, 모든 통신이 코디네이터를 거치는 허브앤스포크 구조, CLAUDE.md의 세 단계 계층, 구조화된 출력을 위한 JSON 스키마, 그리고 컨텍스트를 잃지 않게 지키는 패턴들이다. 각 영역마다 실제 작동하는 코드와, Claude에게 그대로 붙여넣어 과외를 받을 수 있는 프롬프트가 함께 들어 있다.

나는 한동안 Claude를 기반으로 이것저것 만들어 왔다. Airtable과 Notion에 연결한 MCP 서버, 커스텀 스킬과 CLAUDE.md 계층까지 갖춰 팀 프로젝트 전반에 적용한 Claude Code 설정, 콘텐츠 추출과 리서치를 돌리는 멀티 에이전트 파이프라인 같은 것들이다. 그러면서 오케스트레이션이 어긋나는 걸 디버깅했고, 컨텍스트가 격리되는 벽에 부딪혀 봤으며, 도구 설명을 성의 없이 써놓은 탓에 엉뚱한 도구로 일이 넘어가는 것도 직접 겪었다.

Anthropic이 Claude Certified Architect 시험 가이드를 공개했을 때, 거기 적힌 개념들이 하나같이 낯익었다. 대부분은 내가 이미 한 번씩 잘못 구현해 보면서 몸으로 배운 것들이었기 때문이다.

시험은 Claude 파트너사만 볼 수 있어서, 파트너가 아니면 응시 자체가 안 된다. 하지만 가이드는 누구나 볼 수 있고, 프로덕션에서 Claude를 개발할 때 필요한 것들과 그대로 맞아떨어지는 다섯 개 영역으로 짜여 있다.

나는 가이드 전체를 훑으면서 알아야 할 개념을 모두 뽑아냈고, 그중 가장 중요한 패턴은 실제로 돌아가는 코드로 옮겨 봤으며, 영역마다 Claude를 과외 선생처럼 활용할 수 있는 복붙용 프롬프트를 만들었다.

한마디로, 내가 Claude로 에이전트를 처음 만들기 시작할 때 곁에 있었으면 했던 그런 가이드다.

나는 매주 The AI Engineer에서 이런 개념들을 뜯어본다. 소프트웨어 엔지니어가 위험할 만큼 유능한 AI 엔지니어로 거듭나는 곳이다. 구독은 무료.

영역 1 — 에이전트 아키텍처와 오케스트레이션 (시험의 27%)

비중이 가장 큰 영역이다. 에이전트 루프, 여러 에이전트의 협업, 워크플로 강제, 작업 분해, 세션 관리를 다룬다.

핵심 개념: 에이전트 루프

에이전트가 실제로 돌아가는 방식

믿을 수 있는 종료 신호는 오직 stop_reason 하나뿐이다.

모든 에이전트는 같은 핵심 루프를 따른다. 이 루프를 이해하면 에이전트를 이해한 것이나 다름없다.

  • 1단계: Messages API로 Claude에 요청을 보낸다.
  • 2단계: 응답에 담긴 stop_reason 필드를 확인한다.
  • 3단계: stop_reason === "tool_use"이면 도구를 실행하고, 그 결과를 대화에 덧붙인 뒤 다시 요청을 보낸다.
  • 4단계: stop_reason === "end_turn"이면 에이전트가 일을 끝낸 것이므로 루프를 빠져나온다.
# 표준 에이전트 루프
while True:
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        messages=conversation,
        tools=tools
    )

    if response.stop_reason == "end_turn":
        break  # 에이전트가 끝냄

    if response.stop_reason == "tool_use":
        tool_results = execute_tools(response)
        conversation.append(response)
        conversation.append(tool_results)

시험에 나오는 세 가지 잘못된 패턴

시험은 이 세 가지를 그럴듯한 보기로 깔아 둔다. 셋 다 정답이 아니다.

  1. 루프를 끝낼지 말지를 자연어로 판단하는 것. Claude가 "다 했어요"나 "작업 완료"라고 말하는지 들여다보는 방식인데, 이건 믿을 수 없다.
  2. 반복 횟수 상한을 주된 종료 장치로 쓰는 것. max_iterations=10 정도로 막아 두고 그 안에 끝나기를 바라는 식이다. 횟수 상한은 어디까지나 안전장치일 뿐, 흐름을 제어하는 수단이 아니다.
  3. 모델이 내놓은 텍스트가 있는지로 완료를 판단하는 것. 답변 텍스트가 달렸다는 사실 자체를 신호로 삼는 방식인데, 유일한 신호는 stop_reason이다.

여러 에이전트의 오케스트레이션

허브앤스포크 구조 — 모든 통신은 코디네이터를 거친다

멀티 에이전트 시스템은 허브앤스포크(중앙 집중형) 구조를 쓴다. 가운데에 코디네이터 하나가 있고, 그 둘레에 여러 서브에이전트가 붙는다.

  • 규칙 1: 모든 통신은 코디네이터를 거친다. 서브에이전트끼리 직접 주고받는 일은 없다.
  • 규칙 2: 서브에이전트는 코디네이터와 메모리를 공유하지 않는다. 각자 격리된 컨텍스트에서 돌아간다.
  • 규칙 3: 서브에이전트에 필요한 정보는 전부 프롬프트에 담아 명시적으로 넘겨야 한다.
"흔히들 서브에이전트가 코디네이터와 메모리를 공유한다고 착각한다. 그렇지 않다. 서브에이전트는 격리된 컨텍스트에서 동작하고, 필요한 정보는 모두 프롬프트로 직접 전달해야 한다." — 시험에서 가장 많이 틀리는 부분

워크플로 강제

프롬프트로 안내하기 (확률적): 시스템 프롬프트에서 에이전트에게 무엇을 하라고 일러두는 방식이다. 대략 95%는 지시를 따르지만 나머지는 빗나간다. 위험이 낮은 결정에 적당하다.

코드로 강제하기 (결정론적): 훅으로 중간에 가로채서 규칙을 강제한다. 에이전트가 이 검사를 우회할 방법이 없다. 결제 한도, 보안 정책, 규정 준수처럼 어겨선 안 되는 곳에 쓴다.

판단 기준: 일이 틀어졌을 때 결제·보안·규정 준수가 걸린다면, 무조건 코드로 강제하라.

Agent SDK 훅

# PostToolUse 훅: 파일 변경을 모두 기록
async def log_file_change(input_data, tool_use_id, context):
    file_path = input_data.get("tool_input", {}).get("file_path", "unknown")
    with open("./audit.log", "a") as f:
        f.write(f"{datetime.now()}: modified {file_path}\n")
    return {}
연습 시나리오: 실제 프로덕션 데이터를 보니, 고객 지원 에이전트가 계정 소유자 확인을 건너뛴 채 환불을 처리하는 경우가 8%나 됐다. 올바른 해결책은?
정답: 계정 확인이 끝나기 전까지 환불 도구를 막아 두는, 코드로 강제하는 사전 조건 게이트(훅)다. 프롬프트를 다듬거나 예시를 더 주거나 라우팅 분류기를 붙이는 게 아니다. 돈이 오가는 작업은 반드시 결정론적으로 강제해야 한다.
영역 1 과외 프롬프트 (Claude에 붙여넣기):

나는 Claude Certified Architect 시험을 준비하고 있어. 영역 1: 에이전트 아키텍처와 오케스트레이션(시험의 27%)으로 퀴즈를 내줘. 에이전트 루프(종료 신호는 stop_reason뿐), 여러 에이전트의 협업(허브앤스포크, 격리된 컨텍스트), 워크플로 강제(프롬프트 vs 훅), Agent SDK 훅(PostToolUse vs 도구 호출 가로채기), 작업 분해(고정 파이프라인 vs 동적 적응), 세션 관리를 중심으로. 상황을 하나 던져 주고, 거기서 올바른 아키텍처 결정을 고르게 하는 식으로 진행해 줘.

영역 2 — 도구 설계와 MCP 통합 (시험의 18%)

도구 설명, 오류 응답, 도구 배포, MCP 서버 구성, 기본 내장 도구를 다룬다.

도구 설명 — 도구를 고르는 1순위 기준

설명이 애매하면 엉뚱한 도구로 넘어간다. 그러니 설명부터 손봐라.

Claude는 도구 설명을 보고 어떤 도구를 쓸지 정한다. 그래서 Claude가 잘못된 도구를 고른다면, 해결책은 거의 언제나 설명을 더 잘 쓰는 것이다.

구조화된 오류 응답 — 오류의 네 가지 갈래

  • 일시적 오류 (재시도 가능): 타임아웃이나 서비스 일시 중단 같은 경우다. 잠깐 기다렸다가 다시 시도(백오프)하면 된다.
  • 입력 검증 오류 (재시도 가능): 값이 틀렸거나 형식이 어긋났거나 필수 항목이 빠진 경우다. 입력을 바로잡아 다시 보내면 된다.
  • 업무 규칙 위반 (재시도 불가): 정책에 어긋나거나 잔액이 부족한 경우처럼, 같은 요청을 다시 보내도 소용없다. 다른 경로를 찾아야 한다.
  • 권한 문제 (사람에게 이관): 접근이 거부되거나 권한이 없는 경우다. 사람에게 넘겨야 한다.

꼭 구분해야 할 것: 접근 실패 vs 정상적인 '결과 없음'

도구가 아무것도 돌려주지 않을 때, 에이전트는 "데이터에 접근하지 못한 것"(오류)인지 "데이터가 정말로 없는 것"(정상 결과)인지를 가려야 한다. 시험은 이 구분을 콕 집어서 묻는다.

도구 배포와 MCP

도구가 너무 많으면 선택 정확도가 떨어진다: 에이전트 하나에 도구 18개는 과하다. 도구가 18개쯤 되면 Claude가 알맞은 도구를 고르는 정확도가 눈에 띄게 떨어진다. 에이전트당 4~5개가 가장 좋다.

프로젝트 단위 설정 (.mcp.json): 프로젝트 루트에 두고 버전 관리하며 팀과 공유한다. 비밀값은 ${GITHUB_TOKEN}처럼 변수를 펼치는 문법으로 넣는다.

사용자 단위 설정 (~/.claude.json): 홈 디렉터리에 있고, 버전 관리도 공유도 되지 않는다. 시험에 자주 나오는 함정: 새로 합류한 팀원에게만 도구가 안 보인다면, 그 설정이 사용자 단위에 들어가 있는지 확인하라.

기본 내장 도구

  • Grep: 파일 내용에서 패턴을 찾는다.
  • Glob: 이름 패턴으로 파일 경로를 찾는다.
  • Edit: 파일에서 특정 부분만 골라 고친다.
  • 탐색은 좁혀 가며: 먼저 Grep으로 진입점을 찾고, 그다음 Read로 흐름을 따라간다. 처음부터 모든 파일을 다 읽지 마라.
# 나쁜 방법: 처음부터 모든 파일을 읽는다
Read src/api/*.ts  # 관련 없는 파일까지 읽어 컨텍스트만 낭비

# 좋은 방법: 좁혀 가며 탐색한다
Grep "processPayment"  # 진입점부터 찾고
Read src/payments/handler.ts  # 거기서 흐름을 따라간다
연습 시나리오: "주문 #12345 상태 확인해 줘"라는 요청을 에이전트가 lookup_order가 아니라 get_customer로 보내 버린다. 두 도구 모두 설명이 빈약하다. 가장 먼저 할 일은?
정답: 도구 설명을 더 낫게 다듬는 것. 예시를 추가하거나 라우팅 분류기를 붙이는 게 아니다. 도구 설명이 곧 도구를 고르는 핵심 기준이다.

영역 3 — Claude Code 설정과 워크플로 (시험의 20%)

CLAUDE.md 계층, 커스텀 명령, 스킬, 경로별 규칙, 계획 모드, CI/CD 연동을 다룬다.

CLAUDE.md 계층 — 세 단계

  1. 사용자 단위: ~/.claude/CLAUDE.md — 나에게만 적용된다. 공유되지 않고 버전 관리도 되지 않는다.
  2. 프로젝트 단위: .claude/CLAUDE.md — 팀 전체에 적용된다. 버전 관리되고 git으로 공유된다.
  3. 디렉터리 단위: subdirectory/CLAUDE.md — 해당 디렉터리에서 작업할 때만 적용되며, 프로젝트 단위 설정 위에 얹힌다.

시험 함정 — '지침이 안 먹히는' 문제: 새 팀원이 들어왔는데 Claude가 팀 컨벤션을 따르지 않는다. 그 지침이 사용자 단위 설정에만 들어 있으면 새 팀원에게는 없는 셈이다. 해결책은 프로젝트 단위(.claude/CLAUDE.md)로 옮기는 것이다.

계획 모드 vs 곧장 실행

계획 모드가 맞는 경우: 모놀리스 구조 개편, 여러 파일에 걸친 마이그레이션, 아키텍처 결정, 요구사항이 모호할 때.

곧장 실행이 맞는 경우: 파일 하나짜리 버그 수정, 범위가 분명한 일, 반복 작업, 규모가 작고 명확한 변경.

CI/CD 연동

# CI 파이프라인에서는 비대화형 모드로
claude -p "이 PR을 분석해서 결과를 JSON 형식으로 내보내 줘" \
  --output-format json

-p 플래그: Claude Code를 비대화형(출력 전용) 모드로 돌린다. 이걸 빼면 CI가 입력을 기다리며 멈춰 버린다.

연습 시나리오: 개발자 A의 Claude Code는 API 명명 규칙을 완벽히 지킨다. 지난주에 합류한 개발자 B는 명명이 들쭉날쭉하다. 둘은 같은 저장소에서 일한다. 근본 원인은?
정답: 명명 규칙이 프로젝트 단위(.claude/CLAUDE.md)가 아니라 사용자 단위(~/.claude/CLAUDE.md)에 들어 있다. 사용자 단위는 버전 관리도 공유도 되지 않기 때문이다.
영역 3 과외 프롬프트:

나는 Claude Certified Architect 시험을 준비하고 있어. 영역 3: Claude Code 설정과 워크플로(시험의 20%)로 퀴즈를 내줘. CLAUDE.md 계층(사용자/프로젝트/디렉터리 단위), 경로별 규칙(.claude/rules/), 커스텀 슬래시 명령, 스킬(SKILL.md 프론트매터), 계획 모드 vs 곧장 실행, CI/CD 연동(-p 플래그, JSON 출력), 그리고 '지침이 안 먹히는' 문제를 중심으로.

영역 4 — 프롬프트 엔지니어링과 구조화된 출력 (시험의 20%)

명확한 기준, 퓨샷 프롬프팅, 구조화된 출력을 위한 tool_use, 검증·재시도 루프, 배치 처리를 다룬다.

기준을 명확히 — 두루뭉술한 지시는 실패한다

"보수적으로 해"라는 지시는 통하지 않는다. 모호한 지시는 결과를 들쭉날쭉하게 만든다. 해결책은, 무엇을 보고하고 무엇을 넘길지를 콕 집어 정의하는 것이다.

일관성을 끌어올리는 가장 효과적인 방법: 헷갈리는 상황에 대해 딱 맞는 예시 2~4개를 주는 것이다. 예시마다 왜 그 행동을 골랐는지 근거를 함께 보여 주고, 경계에 걸치는 사례도 꼭 넣어라.

구조화된 출력과 Batch API

JSON 스키마와 tool_use: 문법 오류는 없애 주지만 의미상의 오류까지 막아 주지는 않는다.

스키마를 지킨다고 해서 내용을 지어내지 않는 것은 아니다. Claude는 형식상 완벽한 JSON 안에 꾸며 낸 데이터를 담아 돌려줄 수도 있다. 그러니 값이 비어도 되는 필드와 "불명확" 같은 선택지를 스키마에 미리 넣어 두자.

Batch API: 동기 호출보다 비용이 50% 싸고 최대 24시간까지 돌릴 수 있다. 다만 여러 차례 주고받는 도구 호출은 안 된다. 응답 속도가 급하지 않은 대량 추출에 적합하다.

동기 API: 결과가 실시간으로 나오고, 여러 차례 주고받는 도구 호출도 온전히 된다. 대신 비용이 더 든다. 머지 전 점검이나 대화형 에이전트에 적합하다.

"개발자가 결과를 기다려야 하면 동기, 밤새 돌려도 되면 배치." — 판단 기준
연습 시나리오: 코드 리뷰 도구의 전체 정확도가 97%인데도 개발자들은 여전히 믿지 못한다. 왜일까?
정답: 전체 97%라는 숫자가, 특정 문서 유형에서의 40% 오류율을 가려 버릴 수 있기 때문이다. 한 종류에서 오탐이 자주 나면 모든 종류에 대한 신뢰가 무너진다. 해결책은 유형별·카테고리별로 정확도를 따로 검증하는 것이다.

영역 5 — 컨텍스트 관리와 신뢰성 (시험의 15%)

컨텍스트 보존, 사람에게 넘기는 기준, 오류 전파, 코드베이스 탐색, 정보 출처를 다룬다.

컨텍스트 보존 — 요약이 망쳐 놓는 것

거래 데이터는 절대 요약하지 마라

중간중간 내용을 요약하다 보면 정작 중요한 값들이 날아간다. 금액, 케이스 ID, 시각, 주문 번호 같은 것들이다.

해결책: 절대 요약되지 않는 '사건 핵심 정보' 블록을 따로 유지하라.

'중간이 묻히는' 현상: 모델은 길게 주어진 내용 중 가운데 부분을 잘 놓친다. 그러니 핵심 요약은 끝이 아니라 맨 앞에 두고, 도구가 돌려준 결과에서 관련 없는 90%는 잘라 내라.

언제 사람에게 넘겨야 하나

사람에게 넘기는 게 맞는 신호:

  • 고객이 직접 상담원을 찾을 때 — 두말 말고 바로 연결한다. 더 설득하려 들지 않는다.
  • 정책에 빈틈이 있을 때 — 주어진 지침만으로는 판단할 수 없는 상황을 마주한 경우다.
  • 더는 진척이 없을 때 — 할 만큼 해 봤는데도 문제가 풀리지 않는 경우다.

넘김의 근거로 믿어선 안 되는 신호:

  • 감정 분석 — 빈정거림이나 문화 차이, 말투 오해 탓에 엉뚱한 순간에 넘기게 된다.
  • 모델이 스스로 매긴 신뢰도 — Claude가 '얼마나 확신하는지' 스스로 보고한 값은 실제 정확도와 잘 맞아떨어지지 않는다.

오류 전파

오류는 맥락을 갖춰 위로 전달하라. 무엇이 실패했는지, 무엇을 시도했는지, 건진 부분 결과는 무엇인지를 함께 넘기는 것이다.

나쁜 패턴 1: 오류를 조용히 묻기. 오류를 붙잡아 놓고는 아무 일 없었던 듯 그냥 진행해 버린다.

나쁜 패턴 2: 한 번 실패했다고 전체를 중단하기. 일부라도 건진 결과는 그 자체로 쓸모 있는 경우가 많다.

연습 시나리오: 대화가 여러 차례 오간 끝에, 지원 에이전트가 $247.83이라는 금액은 기억하면서도 주문 번호를 놓치는 바람에 엉뚱한 주문에 환불해 버렸다. 이걸 막을 아키텍처 차원의 해결책은?
정답: 추출한 거래 정보(금액, 날짜, 주문 번호)를 담아 모든 프롬프트에 함께 넣는 '사건 핵심 정보' 블록을 두는 것이다. 거래 데이터를 중간 요약에 맡겨선 절대 안 된다.
연습 시나리오: 고객이 "사람이랑 통화하고 싶어요"라고 말한다. 그런데 에이전트는 일단 도와주겠다며 문제부터 풀려고 한다. 이게 맞을까?
정답: 아니다. 고객이 사람을 명시적으로 찾으면 즉시 연결해야 한다. 먼저 해결하려 들지 마라. 이건 예외 없는 규칙이다.

시험 전날 치트시트

항상 참인 핵심 사실 10가지

  1. 서브에이전트는 메모리를 공유하지 않는다 — 필요한 정보는 모두 프롬프트로 직접 넘겨야 한다.
  2. 믿을 수 있는 종료 신호는 stop_reason뿐이다 — "다 했어요"도, 반복 횟수 상한도, 텍스트가 있는지도 아니다. 오직 stop_reason === "end_turn"이다.
  3. 결제·보안은 코드로 강제한다 — 결제 한도나 보안 정책에는 절대 프롬프트 안내에 기대지 마라.
  4. 도구 설명이 도구 선택의 핵심 기준이다 — Claude가 엉뚱한 도구를 고르면 설명부터 손봐라.
  5. CI/CD에는 -p 플래그 — 이게 없으면 CI 안의 Claude Code가 대화형 입력을 기다리며 멈춘다.
  6. 코드 리뷰는 별도 인스턴스로 — 자기 출력을 같은 세션에서 검토하면, 만들 때의 사고에 갇혀 봐주게 된다.
  7. Batch API는 여러 차례 도구 호출이 안 된다 — 50% 싸고 최대 24시간이지만, 한 번 묻고 한 번 답하는 방식만 된다.
  8. 중간 요약은 거래 데이터를 망친다 — 절대 요약되지 않는 '사건 핵심 정보' 블록을 유지하라.
  9. 접근 실패와 '결과 없음'은 다르다 — 타임아웃은 오류이고, 0건 반환은 정상 결과다.
  10. 예시가 산문 지침보다 낫다 — 근거를 곁들인 예시 2~4개가 더 일관된 결과를 만든다.

항상 거부해야 할 잘못된 패턴 7가지

  1. 루프 종료를 "다 했어요" 파싱으로 판단하기
  2. 반복 횟수 상한을 주된 종료 장치로 쓰기
  3. 에이전트 하나에 도구 18개 몰아넣기
  4. 감정 분석으로 사람에게 넘길지 정하기
  5. 모델이 스스로 매긴 신뢰도로 넘김 판단하기
  6. 오류를 조용히 묻기
  7. 처음부터 모든 파일 읽기

시험 전에 직접 만들어 볼 다섯 가지

아래 다섯 가지를 모두 만들 수 있다면, 시험을 통과할 만큼은 안다고 봐도 된다.

  1. stop_reason 처리와 훅을 갖춘 멀티 도구 에이전트 — MCP 도구 3~4개, 제대로 된 에이전트 루프, 데이터 정규화를 위한 PostToolUse 훅, 정책 강제를 위한 도구 호출 가로채기. 영역 1을 아우른다.
  2. 일부러 비슷하게 설명을 단 MCP 도구들 — 기능이 겹치는 도구 2~3개를 만들어 두고, 엉뚱한 라우팅이 사라지도록 설명을 다듬는 연습을 한다. 영역 2를 아우른다.
  3. CLAUDE.md 계층과 규칙, 스킬을 모두 갖춘 프로젝트 — 사용자·프로젝트·디렉터리 단위 CLAUDE.md, 경로별 규칙, 그리고 SKILL.md 프론트매터를 갖춘 스킬 하나 이상. 영역 3을 아우른다.
  4. tool_use와 검증·재시도를 갖춘 추출 파이프라인 — 구조화된 출력을 위한 JSON 스키마, tool_choice 강제, 검증 단계, 재시도 루프. 영역 4를 아우른다.
  5. 오류 전파와 출처 표기를 갖춘 코디네이터-서브에이전트 구조 — 코디네이터가 격리된 컨텍스트로 서브에이전트 둘 이상에 일을 맡기고, 오류는 맥락을 갖춰 위로 전달하며, 리서치 결과에는 출처·발췌·날짜를 함께 단다. 영역 5를 아우른다.

다시 말하지만, 이건 내가 Claude로 에이전트를 막 만들기 시작했을 때 곁에 있었으면 했던 가이드다. 시험에 나오는 개념은 하나같이 프로덕션에서 마주칠 것들이다 — 인증이 목표가 아니더라도 말이다.
