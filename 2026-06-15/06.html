Junior Devs Use try-catch Everywhere. Senior Devs Use These 4 Exception Handling Patterns

HabibWahid · Stackademic · 6 min read · 원문 · freedium

try-catch를 모든 곳에 쓰는 건 '안전한 코딩'이 아니라 '두려움 기반 프로그래밍'이다 — 시니어 개발자가 실제로 쓰는 4가지 예외 처리 패턴을 알아보자.

요약

많은 주니어 개발자들이 모든 메서드에 try-catch를 감싸는 '안전한 코딩'을 실천하지만, 이는 오히려 오류를 조용히 삼켜버려 프로덕션 버그를 숨기는 위험한 습관이다. 시니어 개발자들은 예외를 방어적으로 처리하는 대신, 먼저 입력값을 검증해 예외 자체를 예방하고, 범용 Exception 대신 도메인에 특화된 커스텀 예외를 사용한다. 또한 전역 미들웨어 하나로 예외를 중앙에서 처리해 코드 중복을 없애고, 예상 가능한 실패에는 예외 대신 Result 객체를 반환한다. 마지막으로 예외를 잡았지만 완전히 처리할 수 없을 때는 반드시 로그를 남긴 후 다시 throw해야 오류가 사라지지 않는다.

들어가며: 실제 프로젝트에서 흔히 보이는 실수

몇 년 전 한 스타트업과 함께 일했을 때, 코드베이스에서 뭔가 이상한 점을 발견했다.

거의 모든 메서드에 try-catch 블록이 있었다.

  • 컨트롤러
  • 서비스
  • 리포지토리
  • 유틸리티 클래스

팀은 그것을 "안전한 코딩"이라고 불렀다.

하지만 전혀 안전하지 않았다.

몇 달 후, 프로덕션 문제들이 나타나기 시작했다:

  • 오류가 조용히 삼켜졌다
  • 로그가 없거나 쓸모가 없었다
  • 알림이 전혀 트리거되지 않았다
  • 버그는 사용자 불만 제기 후에야 발견됐다

그 try-catch 블록들은 시스템을 보호하지 않았다. 오히려 실제 장애를 숨겨버렸다.

그 경험에서 중요한 교훈을 얻었다:

try-catch를 모든 곳에 사용하는 것은 방어적 프로그래밍이 아니다.
그것은 두려움 기반 프로그래밍이다.

시니어 개발자들은 기본적으로 try-catch를 추가하지 않는다. 진정으로 가치가 있는 곳에만 추가한다.

시니어 개발자들이 실제로 예외를 처리하는 방법을 간단하고 실용적인 예제와 함께 살펴보자.

주니어 개발자가 try-catch를 기본으로 쓰는 이유

대부분의 주니어들은 좋은 의도로 시작한다:

  • "이게 실패하면 어떡하지?"
  • "컴파일러가 경고했잖아"
  • "튜토리얼에서 이렇게 했는데"
  • "앱이 크래시 나면 안 되잖아"

그래서 이런 코드를 작성한다:

try
{
    DoSomething();
}
catch (Exception ex)
{
    // handle error
}

문제는?

  • try-catch는 실패 이후에 반응한다
  • 버그를 예방하지 않는다
  • 실제 원인을 종종 숨긴다
  • 디버깅을 더 어렵게 만든다

try-catch를 모든 곳에 쓰는 것이 위험한 이유

어떻게 되는가

Request
   ↓
Controller (try-catch)
   ↓
Service (try-catch)
   ↓
Repository (try-catch)
   ↓
Exception occurs
   ↓
Caught too early
   ↓
Logged inconsistently or ignored
   ↓
Fake success / unclear failure

문제점

  • 예외가 삼켜진다
  • 로그가 중복되거나 누락된다
  • 실제 근본 원인이 사라진다
  • 프로덕션 버그가 숨겨진 채 방치된다

이것이 바로 "안전한 코딩"이 위험한 코딩으로 변하는 이유다.

시니어 개발자들은 먼저 더 나은 질문을 한다:

이 실패를 잡는 것 대신 예방할 수 있지 않을까?

깔끔한 흐름

Request
   ↓
Controller (no try-catch)
   ↓
Service
   ↓
Repository
   ↓
Exception occurs
   ↓
Bubble up
   ↓
Global Exception Middleware
   ↓
✔ Log once
✔ Correct HTTP status
✔ Clear error response

이 방식이 효과적인 이유

  • 오류를 한 곳에서만 처리한다
  • 중복이 없다
  • 깨끗한 스택 트레이스를 유지한다

시니어 개발자들이 사용하는 예외 처리 패턴

패턴 1: 먼저 검증하고, 나중에 잡지 마라

불필요한 try-catch 블록의 대부분은 잘못된 입력값 때문에 존재한다.

주니어 방식 (잘못된 방법)

[HttpPost("users")]
public IActionResult CreateUser(UserRequest request)
{
    try
    {
        var user = _userService.Create(request);
        return Ok(user);
    }
    catch (NullReferenceException)
    {
        return BadRequest();
    }
    catch (ArgumentException)
    {
        return BadRequest();
    }
}

이 코드는 취약하고 유지보수하기 어렵다.

시니어 방식 (올바른 방법)

메서드가 실행되기 전에 검증한다.

[HttpPost("users")]
public IActionResult CreateUser([FromBody] UserRequest request)
{
    var user = _userService.Create(request);
    return CreatedAtAction(nameof(CreateUser), user);
}

검증이 포함된 요청 모델

public class UserRequest
{
    [Required]
    public string Name { get; set; }

    [Required, EmailAddress]
    public string Email { get; set; }

    [Range(18, 100)]
    public int Age { get; set; }
}

ASP.NET Core가 자동으로:

  • 입력값을 검증하고
  • 400 Bad Request를 반환하며
  • 잘못된 데이터를 초기에 차단한다
규칙: 잘못된 입력으로 인한 예외를 잡고 있다면, 예외 문제가 아니라 검증 문제다.

패턴 2: 커스텀 예외를 사용하라 (Exception은 절대 쓰지 마라)

주니어 방식

try
{
    _orderService.Process(orderId);
}
catch (Exception ex)
{
    _logger.LogError(ex.Message);
    return StatusCode(500);
}

이것은 아무런 유용한 정보를 제공하지 않는다.

시니어 방식

기본 애플리케이션 예외

public abstract class AppException : Exception
{
    public int StatusCode { get; }

    protected AppException(string message, int statusCode)
        : base(message)
    {
        StatusCode = statusCode;
    }
}

구체적인 예외 클래스들

public class NotFoundException : AppException
{
    public NotFoundException(string resource)
        : base($"{resource} not found", StatusCodes.Status404NotFound) { }
}

public class BusinessRuleException : AppException
{
    public BusinessRuleException(string message)
        : base(message, StatusCodes.Status409Conflict) { }
}

사용법 (try-catch 없음)

public Order ProcessOrder(int orderId)
{
    var order = _repository.Get(orderId)
        ?? throw new NotFoundException("Order");

    if (order.IsProcessed)
        throw new BusinessRuleException("Order already processed");

    return order;
}
규칙: Exception을 잡은 후 무슨 문제인지 파악하려 한다면 — 예외가 너무 범용적인 것이다.

패턴 3: 예외를 한 곳에서만 처리하라 (전역 미들웨어)

주니어 개발자들은 모든 곳에 try-catch를 추가한다. 시니어 개발자들은 한 곳에서만 예외를 처리한다.

주니어 방식

[HttpGet("{id}")]
public IActionResult GetOrder(int id)
{
    try
    {
        return Ok(_service.Get(id));
    }
    catch (NotFoundException)
    {
        return NotFound();
    }
    catch (Exception)
    {
        return StatusCode(500);
    }
}

시니어 방식 — 전역 예외 미들웨어

public class ExceptionMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<ExceptionMiddleware> _logger;

    public ExceptionMiddleware(RequestDelegate next, ILogger<ExceptionMiddleware> logger)
    {
        _next = next;
        _logger = logger;
    }

    public async Task Invoke(HttpContext context)
    {
        try
        {
            await _next(context);
        }
        catch (AppException ex)
        {
            context.Response.StatusCode = ex.StatusCode;
            await context.Response.WriteAsJsonAsync(new { error = ex.Message });
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Unhandled error");
            context.Response.StatusCode = 500;
            await context.Response.WriteAsJsonAsync(new { error = "Something went wrong" });
        }
    }
}

한 번만 등록하면 된다:

app.UseMiddleware<ExceptionMiddleware>();

컨트롤러가 깔끔해진다:

[HttpGet("{id}")]
public Order GetOrder(int id)
{
    return _service.Get(id);
}
규칙: 같은 catch 로직이 여러 곳에 나타난다면 — 중앙화하라.

패턴 4: 예상된 실패에는 Result 객체를 사용하라

모든 실패가 예외적인 것은 아니다.

예상 가능한 실패:

  • 사용자를 찾을 수 없음
  • 검증 실패
  • 결제 거절

주니어 방식

public User GetUser(int id)
{
    return _repo.Get(id) ?? throw new NotFoundException("User");
}

이것은 모든 호출자가 예외를 처리하도록 강제한다.

시니어 방식 — Result 패턴

public class Result<T>
{
    public bool IsSuccess { get; }
    public T Value { get; }
    public string Error { get; }

    private Result(bool success, T value, string error)
    {
        IsSuccess = success;
        Value = value;
        Error = error;
    }

    public static Result<T> Success(T value) => new(true, value, null);
    public static Result<T> Failure(string error) => new(false, default, error);
}

사용법:

public Result<User> GetUser(int id)
{
    var user = _repo.Get(id);
    return user == null
        ? Result<User>.Failure("User not found")
        : Result<User>.Success(user);
}

컨트롤러:

var result = _service.GetUser(id);

if (!result.IsSuccess)
    return NotFound(result.Error);

return Ok(result.Value);
규칙: 실패가 예상 가능하다면 throw하지 말고 — result를 반환하라.

(중요) 패턴 5: 로그 남기고 다시 throw하라, 삼키지 마라

많은 블로그에서 놓치는 핵심 규칙 하나.

잘못된 방법

catch (Exception ex)
{
    _logger.LogError(ex.Message);
}

이것은 실패를 숨긴다.

올바른 방법

catch (Exception ex)
{
    _logger.LogError(ex, "Error while processing order");
    throw;
}
규칙: 예외를 잡았지만 완전히 처리할 수 없다면 — 로그를 남기고 다시 throw하라.

시니어 개발자들이 사용하는 멘탈 모델

예외적인 상황 (try-catch 사용)

  • 데이터베이스 다운
  • 네트워크 장애
  • 외부 API 타임아웃
  • 인프라 문제

예상 가능한 상황 (try-catch 불필요)

  • 사용자를 찾을 수 없음
  • 잘못된 입력값
  • 비즈니스 규칙 위반
  • 중복 요청

이렇게 생각하기 시작하면 코드가:

  • 더 깔끔해지고
  • 읽기 쉬워지며
  • 디버깅이 편해지고
  • 프로덕션에서 더 안전해진다

간단한 실천 계획

내일 업무에서 딱 한 가지만 해보자:

  • 검증용 try-catch 하나를 모델 검증으로 교체하기
  • catch (Exception) 하나를 커스텀 예외로 교체하기
  • 반복되는 catch 로직을 미들웨어로 옮기기
  • 예상 가능한 예외 하나를 Result 객체로 교체하기

작은 한 걸음 → 시니어 수준의 코드.
