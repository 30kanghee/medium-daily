CAP 정리부터 카오스 엔지니어링에 이르기까지, 현대 데이터 플랫폼에서 실제로 맞닥뜨리는 문제들을 중심으로 구성됐다. 각 주제마다 면접관이 기대하는 수준의 설명, 장단점, 동작 원리를 보여주는 Python 코드 예제, 그리고 주니어와 시니어를 가르는 심화 개념까지 다룬다. Kafka, Saga 패턴, CRDTs, Parquet, Raft 합의 알고리즘, 데이터 계약, 일관된 해시 샤딩, 재해 복구 등 2026년 현재 현업에서 표준으로 자리 잡은 기술들을 망라한다.

데이터 엔지니어가 2026년 면접에서 반드시 숙지해야 할 시스템 설계 주제들이다. 이 개념들은 실제 프로덕션 환경의 문제와 현대적인 데이터 아키텍처를 반영한다.

10. 아키텍처 트레이드오프: CAP 정리

2026년에도 CAP 정리는 분산 시스템의 근본적인 틀로 여전히 유효하다. 이 정리는 네트워크 파티션(P)이 발생했을 때 시스템이 일관성(C)과 가용성(A) 중 하나를 선택해야 하며, 둘 다 동시에 보장할 수 없다는 것을 증명한다.

왜 중요한가

Lakehouse(Iceberg)를 선택하는 것부터 Kafka의 acks 설정을 조정하는 것까지, 모든 아키텍처 결정은 CAP 트레이드오프의 연속이다. 면접에서는 "완벽한" 해답을 찾는 능력보다, 왜 그 희생을 선택했는지 (예: 일관성을 포기하고 낮은 지연 시간을 선택한 이유) 논리적으로 설명할 수 있는지를 더 중요하게 본다.

장점

  • 운영 예측 가능성: "회색 장애(gray failure)"나 네트워크 혼잡 상황에서 시스템이 어떻게 동작할지 미리 예측할 수 있다.
  • 글로벌 확장성: AP 시스템은 값비싼 전역 동기화 잠금을 피하기 때문에 더 빠르고 저렴하게 확장할 수 있다.
  • SLA 복원력: AP 모델은 특정 클라우드 리전에 장애가 발생해도 쓰기 가용성을 높게 유지한다.

단점

  • 애플리케이션 복잡도 증가: AP 시스템에서는 "오래된 읽기(stale reads)"와 충돌 해소 로직을 개발자가 코드로 직접 처리해야 한다.
  • 마이그레이션 부채: AP 모델에서 CP 모델로 운영 중 전환하는 것은 위험도가 높고 수개월이 걸리는 엔지니어링 작업이다.

CAP 파티션 시뮬레이션

이 스크립트는 네트워크 장애 상황에서 CP 노드(정확성 우선)와 AP 노드(가동 시간 우선)가 어떻게 다르게 동작하는지를 보여준다.

class DistributedNode:
    def __init__(self, node_id, mode="AP"):
        self.node_id = node_id
        self.mode = mode  # "AP" or "CP"
        self.data = "Initial Value"
        self.is_partitioned = False

    def get_value(self):
        """
        Simulates a Read Request.
        """
        if self.is_partitioned:
            if self.mode == "CP":
                # In CP mode, we fail rather than return potentially stale data
                raise ConnectionError(f"Node {self.node_id}: System unavailable (Consistency check failed)")
            else:
                # In AP mode, we return what we have, even if it might be old
                return f"{self.data} (STALE/AVAILABLE)"
        return self.data

    def set_partition(self, status: bool):
        self.is_partitioned = status

# Scenario Simulation
if __name__ == "__main__":
    node_cp = DistributedNode(1, mode="CP")
    node_ap = DistributedNode(2, mode="AP")

    # Simulate a network cut (Partition)
    node_cp.set_partition(True)
    node_ap.set_partition(True)

    print(f"AP Node Response: {node_ap.get_value()}")
    try:
        print(f"CP Node Response: {node_cp.get_value()}")
    except ConnectionError as e:
        print(f"CP Node Error: {e}")

2026년의 변화: PACELC가 새로운 CAP이다

한 발 더 나아가 PACELC를 언급하면 인상을 남길 수 있다. PACELC는 파티션이 없는 정상 상태에서의 동작까지 포함해 CAP을 확장한 모델이다.

  • PAC: 파티션이 발생하면 가용성과 일관성 중 하나를 선택한다.
  • ELC: 그 외의 평상시에는 지연 시간과 일관성 중 하나를 선택한다.

9. 분산 트랜잭션: Saga vs. 2PC

모놀리식 애플리케이션은 단일 데이터베이스 트랜잭션을 사용한다. 분산 시스템에서는 "주문"이 결제, 재고, 배송 서비스로 나뉘어 처리된다. 한 단계가 실패하면 나머지도 롤백되어야 한다. 2PC(Two-Phase Commit)는 이를 동기적이고 엄격하게 처리한다. Saga는 최종 일관성(eventual consistency)을 통해 비동기적으로 관리한다.

왜 중요한가

전략이 없으면 결제는 됐는데 배송은 안 되거나, 재고는 줄었는데 판매 내역은 없는 "유령 주문" 상황이 생긴다. 2026년 현재 Saga는 서비스 간 독립성과 부분 네트워크 장애 상황에서의 높은 가용성을 유지하기 때문에 업계 표준으로 자리 잡았다.

2PC (Two-Phase Commit)

  • 장점: 원자성 보장. 클러스터 전체가 하나의 일관된 데이터베이스처럼 동작한다.
  • 단점: 블로킹 방식. 코디네이터가 장애를 일으키면 참여 서비스들이 모두 잠긴다. "Prepare" 단계와 "Commit" 단계의 동기적 특성 때문에 지연 시간이 높다.

Saga 패턴

  • 장점: 높은 가용성. 서비스가 즉시 로컬 커밋을 진행하며 잠금이 없다. 여행 예약처럼 오래 걸리는 대규모 프로세스에 적합하다.
  • 단점: 복잡성. 모든 단계마다 취소 로직인 "보상 트랜잭션(Compensating Transaction)"을 작성해야 한다. 중간 상태가 외부에 노출되는 "더러운 읽기(Dirty Reads)" 위험이 있다.

오케스트레이션 기반 Saga

이 스크립트는 "여행 예약" Saga를 구현한다. 결제가 실패하면 오케스트레이터가 "호텔 취소" 보상 작업을 실행해 시스템 무결성을 복원한다.

import time

class BookingOrchestrator:
    def __init__(self):
        self.steps = []
        self.compensations = []

    def execute_saga(self, booking_id):
        try:
            # Step 1: Reserve Hotel
            self._hotel_reserve(booking_id)
            self.compensations.append(lambda: self._hotel_cancel(booking_id))

            # Step 2: Process Payment (Simulate Failure)
            self._process_payment(booking_id)
            self.compensations.append(lambda: self._refund_payment(booking_id))

            print("Saga Completed Successfully!")
        except Exception as e:
            print(f"Saga Failed: {e}. Starting Compensations...")
            self._rollback()

    def _rollback(self):
        # Reverse the list to undo steps in correct order (LIFO)
        for undo_action in reversed(self.compensations):
            undo_action()
        print("Rollback Finished. System is Consistent.")

    def _hotel_reserve(self, bid):
        print(f"[HOTEL] Room reserved for {bid}")

    def _hotel_cancel(self, bid):
        print(f"[HOTEL] COMPENSATING: Room reservation released for {bid}")

    def _process_payment(self, bid):
        # Simulated Failure logic
        time.sleep(1)
        raise ConnectionError("Payment Gateway Timeout")

    def _refund_payment(self, bid):
        print(f"[PAYMENT] COMPENSATING: Refunding payment for {bid}")

# Execution
if __name__ == "__main__":
    orchestrator = BookingOrchestrator()
    orchestrator.execute_saga("TX-9901")

2026년의 변화: "Transactional Outbox" 패턴

현대 면접에서는 Saga를 아는 것만으로는 부족하다. DB에 쓰기와 Kafka 메시지 전송을 동시에 하는 "이중 쓰기(Dual Write)" 문제를 해결하는 Outbox 패턴도 이해해야 한다.

  • 로컬 DB를 통한 원자성: 비즈니스 데이터와 이벤트("Outbox 메시지")를 동일한 로컬 데이터베이스 트랜잭션에 함께 기록한다.
  • 릴레이 서비스: 별도 프로세스가 Outbox 테이블을 읽어 Kafka나 RabbitMQ 같은 메시지 브로커에 이벤트를 발행한다.
  • 혼합 전략: 단일 클러스터 내 고가치·저지연 금융 원장 업데이트에는 2PC를(예: CockroachDB 활용), 서비스 경계를 넘나드는 모든 처리에는 Saga를 사용한다.

8. 고가용성 상태 관리: 최종 일관성과 CRDTs

강한 일관성은 비용이 크다. 쓰기가 성공하려면 모든 노드가 합의해야 하기 때문이다. 최종 일관성(Eventual Consistency)은 가용성을 우선시한다. 노드가 즉시 쓰기를 수락하고, 나중에 조정한다. CRDT(Conflict-Free Replicated Data Type)는 이 조정을 자동으로, 수학적으로 확정적인 방식으로 처리하는 특수 자료구조다.

왜 중요한가

글로벌 시스템에서는 빛의 속도 때문에 처리량이 높은 앱에서 중앙 잠금을 기다리는 것이 불가능하다. CRDT는 Figma나 Apple Notes 같은 플랫폼이 중앙 중재자 없이 동시 편집을 병합할 수 있게 해준다. 데이터 엔지니어에게 이것은 네트워크 파티션이 발생해도 사용자 데이터를 잃지 않는 시스템을 만드는 핵심이다.

장점

  • 지연 없는 쓰기: 리더 노드로의 네트워크 왕복을 기다리지 않고 복제본이 즉시 로컬 상태를 업데이트한다.
  • 파티션 내성: 네트워크가 끊겨도 시스템은 100% 가용 상태를 유지하며, 연결이 복원되면 노드들이 자연스럽게 동기화된다.
  • 수학적 수렴: "Last-Write-Wins(LWW)"와 달리, CRDTs는 업데이트 순서에 관계없이 모든 복제본이 동일한 상태에 도달하도록 보장한다.

단점

  • 메타데이터 오버헤드: CRDTs는 인과 관계를 추적하기 위해 버전 벡터 같은 추가 데이터가 필요하며, 시간이 지나면 저장 공간이 늘어날 수 있다.
  • 제한된 연산 종류: 연산이 교환 가능(x + y = y + x)하고, 결합 가능하며, 멱등성을 가져야 한다.
  • 사고 방식의 전환: 현재 값이 몇 밀리초 동안 노드마다 다를 수 있다는 점을 개발자가 항상 염두에 두어야 한다.

G-Counter (증가 전용 카운터)

이 스크립트는 상태 기반 CRDT 카운터를 구현한다. 별도 노드에서 증가가 일어나고 병합 순서가 달라져도, 최종 결과는 어디서나 동일하다.

class GCounter:
    """
    A Grow-only Counter CRDT.
    State is a dictionary mapping node_ids to their local counts.
    """
    def __init__(self, node_id):
        self.node_id = node_id
        self.state = {node_id: 0}

    def increment(self):
        """Update only the local node's counter."""
        self.state[self.node_id] += 1

    def merge(self, other_gcounter):
        """
        The 'Join' operation: takes the maximum value for each node_id.
        This ensures commutativity and associativity.
        """
        all_nodes = set(self.state.keys()).union(other_gcounter.state.keys())
        for nid in all_nodes:
            self.state[nid] = max(
                self.state.get(nid, 0),
                other_gcounter.state.get(nid, 0)
            )

    @property
    def value(self):
        """The total value is the sum of all individual node counts."""
        return sum(self.state.values())

# Simulation of three distributed nodes
node_a = GCounter("node_a")
node_b = GCounter("node_b")

# Node A and B work independently
node_a.increment()
node_a.increment()
node_b.increment()

print(f"Node A value: {node_a.value}")  # 2
print(f"Node B value: {node_b.value}")  # 1

# Sync/Merge: Node A receives Node B's state
node_a.merge(node_b)
print(f"Node A value after merge: {node_a.value}")  # 3

2026년의 변화: 로컬 퍼스트 데이터 엔지니어링

시니어 면접에서는 최종 일관성이 어떻게 진화하고 있는지 언급하는 것이 좋다.

  • 상태 기반 vs. 연산 기반: 상태 기반(전체 객체 전송)은 구현이 단순하지만 대역폭을 더 사용한다. 연산 기반(변경 사항만 전송)은 효율적이지만 신뢰성 있는 전달이 필요하다.
  • 인과적 일관성(Causal Consistency): 사용자 A가 사용자 B의 메시지에 답장하면, 모든 이가 B의 메시지를 먼저 보도록 보장하는 "중간 지점"이다.
  • 내구성 있는 실행의 부상: Temporal 같은 프레임워크를 사용해 최종 일관성 시스템이 촉발하는 장기 실행 워크플로를 처리하는 방식이다.

7. 저장소의 진화: 컬럼형 포맷 (Parquet & Arrow)

2026년에는 분석 데이터를 더 이상 행(row) 단위로 저장하지 않는다. "콜드" 디스크 저장에는 Parquet를, "핫" 인메모리 처리에는 Apache Arrow를 사용한다. 이 분리 덕분에 쿼리에 필요한 특정 컬럼만 스캔할 수 있어 I/O 비용이 대폭 줄어든다.

왜 중요한가

대부분의 "빅데이터" 병목은 사실 I/O 문제다. CSV나 JSON 같은 행 기반 포맷은 하나의 컬럼을 평균 내기 위해서도 100개 컬럼 전체를 읽어야 한다. 컬럼형 포맷은 이 O(n) 행 스캔을 선택적인 O(k) 컬럼 가져오기로 바꾼다. 2026년에 데이터 레이크하우스(Iceberg/Delta Lake)가 업계의 중심이 된 만큼, 이 물리적 구조를 이해하는 것은 필수다.

장점

  • 뛰어난 압축률: 같은 타입의 데이터를 모아 저장하기 때문에(예: 정수끼리) 딕셔너리 인코딩이나 런-렝스 인코딩 같은 효율적인 압축이 가능하다.
  • 프레디케이트 푸시다운(Predicate Pushdown): 실제 데이터 블록을 열지 않고도 메타데이터(최솟값/최댓값)만 확인해 전체 "Row Group"을 건너뛸 수 있다.
  • 제로 카피 읽기: Apache Arrow 덕분에 Python, Spark, C++ 같은 서로 다른 시스템이 데이터를 직렬화하는 느린 과정 없이 동일한 메모리 버퍼를 공유할 수 있다.

단점

  • 쓰기 지연: 컬럼형 파일 생성은 CPU를 많이 사용한다. 단일 행 추가는 비효율적이며, 반드시 큰 배치 단위로 버퍼링 후 쓰기를 해야 한다.
  • 스키마 경직성: JSON 블롭을 단순히 수정하는 것보다 필드를 추가하는 작업이 더 복잡하다.
  • 업데이트 패널티: 단일 셀을 쉽게 업데이트할 수 없다. 파일 전체를 다시 쓰거나 변경 사항을 표시하는 "Delete Vectors"를 사용해야 한다.

고성능 컬럼 필터링

이 예제는 PyArrow를 사용해 현대적인 워크플로를 시뮬레이션한다. 관련 컬럼만 로드하고 "프레디케이트 푸시다운"으로 메모리를 절약하는 방식이다.

import pyarrow as pa
import pyarrow.parquet as pq
import pyarrow.compute as pc
import os

# 1. Setup: Create a Table with 1 Million Rows
data = {
    "timestamp": pa.array(range(1000000), type=pa.int64()),
    "user_id": pa.array([f"user_{i%1000}" for i in range(1000000)]),
    "revenue": pa.array([float(i % 100) for i in range(1000000)]),

2026년의 변화: 파일 포맷 너머로

시니어 면접에서는 "컬럼형"과 관련한 세 가지 고급 개념을 언급하는 것이 좋다.

  • 벡터화 실행(Vectorized Execution): 현대 CPU는 단일 명령어(SIMD)로 전체 "벡터"(예: 1024개의 정수)를 처리할 수 있다. 이를 가능하게 하는 유일한 포맷이 바로 컬럼형 데이터다.
  • 매니페스트 파일(Iceberg): Iceberg는 단순히 Parquet 파일을 저장하는 것이 아니라 파일의 "스냅샷"을 저장한다. 이것이 지난주 화요일 데이터를 조회하는 타임 트래블(Time Travel)을 가능하게 한다.
  • 지연 구체화(Late Materialization): 쿼리 엔진이 압축된 컬럼형 데이터를 먼저 필터링하고, 맨 마지막 순간에야 완전한 레코드를 가져오는 전략으로 메모리를 절약한다.

6. 비동기 통신: 메시지 큐 (Kafka vs. RabbitMQ vs. SQS)

비동기 통신은 서비스가 응답을 기다리지 않고 서로 메시지를 주고받게 한다. 덕분에 시스템이 대규모 트래픽 급증을 처리하고 부분적인 장애에서도 살아남을 수 있다. 2026년에는 보통 스마트 파이프(RabbitMQ 또는 SQS)와 덤 파이프/스마트 컨슈머(Kafka) 중 하나를 선택하는 구도다.

왜 중요한가

일반적인 API 호출은 취약하다. 수신 측이 느려지면 발신 측이 다운된다. 메시지 큐는 이런 연쇄 장애를 막는 완충 역할을 한다. 데이터 엔지니어에게는 로그와 클릭스트림을 수집할 때 다운스트림 데이터베이스에 부하를 주지 않는 가장 좋은 방법이다.

장점

  • 수평적 확장성: 프로듀서를 건드리지 않고 컨슈머를 늘려 밀린 메시지를 처리할 수 있다.
  • 장애 격리: 컨슈머가 다운되어도 메시지는 큐에 안전하게 보관되며, 재시작 후 처리된다.
  • 트래픽 급증 완충: 블랙 프라이데이 같은 고트래픽 이벤트 때 큐가 충격 흡수 장치 역할을 한다.

단점

  • 인프라 복잡도 추가: 모니터링하고 관리해야 할 인프라가 하나 더 생긴다.
  • 디버깅 어려움: 여러 서비스에 걸친 단일 트랜잭션을 추적하려면 복잡한 트레이싱 도구가 필요하다.
  • 일관성 지연: 업데이트가 즉각적이지 않다. 시스템은 설계상 최종 일관성을 가진다.

고성능 Kafka 프로듀서

confluent-kafka 라이브러리는 속도 면에서 2026년 표준이다. 이 예제는 파티션 키잉을 사용해 이벤트 순서를 올바르게 유지하는 방법을 보여준다.

from confluent_kafka import Producer
import json
import socket

# Configuration for a 2026-ready production environment
conf = {
    'bootstrap.servers': 'localhost:9092',
    'client.id': socket.gethostname(),
    'acks': 'all',               # Guarantee no data loss (Wait for all replicas)
    'retries': 5,                # Resiliency against transient network jitters
    'compression.type': 'snappy' # High-speed compression for data-heavy streams
}

producer = Producer(conf)

def delivery_report(err, msg):
    """Callback triggered on success or failure of message delivery."""
    if err is not None:
        print(f"Message delivery failed: {err}")
    else:
        print(f"Message delivered to {msg.topic()} [{msg.partition()}] at offset {msg.offset()}")

def stream_event(topic, user_id, action):
    payload = {
        'user_id': user_id,
        'action': action,
        'timestamp': '2026-01-07T00:00:00Z'
    }
    # Keying by user_id ensures all events for ONE user stay in the SAME partition (Ordering)
    producer.produce(
        topic,
        key=str(user_id),
        value=json.dumps(payload),
        callback=delivery_report
    )
    # Serve background callbacks
    producer.poll(0)

# Simulate streaming
events = [
    (101, 'login'),
    (102, 'view_item'),
    (101, 'add_to_cart')  # Will go to same partition as first event due to key 101
]

for uid, act in events:
    stream_event('user_behavior_logs', uid, act)

producer.flush()  # Ensure all messages are sent before exiting

2026년의 변화: 도구 선택의 기준

시니어 면접에서는 순서 보장과 메시지 보존 필요성을 기준으로 도구 선택을 정당화할 수 있어야 한다.

  • 재생 가능성(Replayability): Kafka에서는 컨슈머를 지난주 화요일로 "되감아" 지표를 재계산할 수 있다. RabbitMQ나 SQS에서는 이것이 불가능하다.
  • 스마트 라우팅: "헤더 X가 있는 메시지를 큐 Y로 보내라" 같은 복잡한 라우팅 로직이 필요하면 RabbitMQ가 탁월하다. Kafka는 컨슈머가 직접 필터링을 처리하는 구조다.
  • 서버리스 단순성: AWS 환경이고 초당 수백만 건의 이벤트가 필요 없다면, 서버 관리가 전혀 필요 없는 SQS가 최선이다.

5. 분산 조정: 합의 알고리즘 (Raft vs. Paxos)

합의 알고리즘은 신뢰할 수 없는 노드 집합이 단일 상태에 동의할 수 있게 한다. 네트워크가 느려지거나 노드가 다운돼도 시스템은 일관된 단위로 동작한다. Paxos가 원래의 수학적 증명이었다면, Raft는 더 이해하기 쉽고 구현하기 쉬워서 2026년 업계 표준이 됐다.

왜 중요한가

대규모 시스템에서 가장 큰 위험은 두 노드가 모두 자신이 리더라고 생각하며 충돌하는 변경을 만드는 "스플릿 브레인(Split-Brain)" 시나리오다. 합의 알고리즘은 모든 결정에 쿼럼(과반수)을 요구함으로써 이를 방지한다. Kubernetes가 클러스터 상태를 유지하고 Amazon Aurora가 빠른 페일오버를 처리하는 방식이 바로 이것이다.

장점

  • 높은 가용성: 노드의 과반수(50% 초과)가 정상이면 시스템이 온라인 상태를 유지한다.
  • 강한 일관성: 데이터가 커밋되면 모든 읽기 요청이 정확히 같은 값을 보도록 보장한다.
  • 자동 복구: 리더 노드가 다운되면 남은 노드들이 인간의 개입 없이 자동으로 새 리더를 선출한다.

단점

  • 지연 시간 패널티: 모든 쓰기가 과반수 노드를 왕복해야 하므로 단순 복제보다 느리다.
  • 극도의 복잡성: 합의 알고리즘을 처음부터 직접 작성하는 것은 매우 어렵고 버그가 발생하기 쉽다.
  • 네트워크 민감성: 네트워크 지터가 심하면 불필요한 리더 선출이 반복적으로 발생해 시스템이 느려진다.

Raft 리더 선출 시뮬레이션

이 스크립트는 노드들이 리더와의 연결을 잃었을 때 팔로워에서 후보로 전환하고, 결국 선출에서 승리하는 과정을 시뮬레이션한다.

import random
import time
import threading

class RaftNode:
    def __init__(self, node_id, total_nodes):
        self.node_id = node_id
        self.total_nodes = total_nodes
        self.state = "Follower"
        self.current_term = 0
        self.votes_received = 0
        self.heartbeat_timeout = random.uniform(1.5, 3.0)  # Randomized for Raft
        self.last_heartbeat = time.time()

    def start(self):
        """Monitor heartbeats in a background thread."""
        threading.Thread(target=self._run_election_timer, daemon=True).start()

    def _run_election_timer(self):
        while True:
            time.sleep(0.1)
            # If no heartbeat received within timeout, become a candidate
            if self.state != "Leader" and (time.time() - self.last_heartbeat) > self.heartbeat_timeout:
                self._become_candidate()

    def _become_candidate(self):
        self.state = "Candidate"
        self.current_term += 1
        self.votes_received = 1  # Vote for self
        print(f"Node {self.node_id}: Heartbeat lost. Starting Election (Term {self.current_term})")
        # Simulate requesting votes from other nodes
        self._request_votes()

    def _request_votes(self):
        # In a real system, this would be an RPC call to other nodes
        votes_from_peers = random.randint(0, self.total_nodes - 1)
        self.votes_received += votes_from_peers
        quorum = (self.total_nodes // 2) + 1
        if self.votes_received >= quorum:
            self._become_leader()

    def _become_leader(self):
        self.state = "Leader"
        print(f"Node {self.node_id}: Elected LEADER for Term {self.current_term} with {self.votes_received} votes")
        # Leaders send heartbeats to suppress further elections
        self._send_heartbeats()

    def _send_heartbeats(self):
        # Leaders send periodic signals to reset followers' timers
        print(f"Node {self.node_id}: Sending heartbeats to maintain authority...")

# Execution
if __name__ == "__main__":
    # Create a 5-node cluster
    cluster_size = 5
    nodes = [RaftNode(i, cluster_size) for i in range(cluster_size)]
    for node in nodes:
        node.start()
    time.sleep(4)  # Let the simulation run

2026년의 변화: Zookeeper의 퇴장

업계는 Zookeeper 같은 외부 합의 도구에서 벗어나 조정 로직을 자체 아키텍처에 직접 내장하는 방향으로 이동했다.

  • KRaft (Kafka Raft): Kafka 3.x 이후 Zookeeper가 제거됐다. 합의 로직이 Kafka 브로커 내부로 이동해 메타데이터 업데이트가 빨라지고 클러스터가 단순해졌다.
  • 내장 합의: CockroachDB와 TiDB 같은 데이터베이스는 Raft를 스토리지 레이어에 내장해 샤딩과 복제를 하나의 시스템에서 관리한다.
  • 비잔틴 장애 허용(BFT): 표준 Raft는 노드가 정직하지만 장애가 날 수 있다고 가정한다. 2026년 고보안 또는 Web3 플랫폼 면접에서는 노드가 악의적으로 거짓 정보를 보내는 경우까지 처리하는 더 강력한 합의 방식인 BFT에 대해 묻는 경우가 많다.

4. 내결함성: 멱등성과 정확히-한번 처리

분산 시스템에서 "최소-한번(At-Least-Once)" 전달 보장은 흔하지만 위험하다. 레코드가 처리된 후 확인 응답이 전송되기 전에 네트워크 순단이 발생하면, 시스템이 재시도하면서 중복 데이터가 생긴다. 멱등성(Idempotency)은 동일한 작업을 여러 번 수행해도 한 번 수행한 것과 같은 결과를 보장한다.

왜 중요한가

금융 데이터 엔지니어링이나 재고 관리에서 "대충 맞는" 것은 실패다. 중복 이벤트는 지표 부풀리기, 이중 청구, 상태 오염으로 이어진다. 2026년의 이벤트 기반 아키텍처로 나아가면서, 정확히-한번 시맨틱스(Exactly-Once Semantics, EoS)로 재시도를 우아하게 처리하는 능력이 주니어 스크립트 작성자와 시니어 시스템 아키텍트를 가르는 기준이 된다.

장점

  • 데이터 무결성: 분석 지표와 금융 원장의 "이중 계산"을 제거한다.
  • 안전한 재시도: 부작용 없이 실패한 작업을 적극적으로 재시도할 수 있다.
  • 로직 단순화: 업스트림이 EoS를 보장하면, 다운스트림 컨슈머가 복잡한 중복 제거 로직을 별도로 만들 필요가 없다.

단점

  • 성능 오버헤드: "처리된" 키를 추적하거나 분산 트랜잭션(2PC 등)을 사용하면 지연이 생긴다.
  • 저장 비용: 처리된 Request ID를 추적하기 위한 "중복 제거 저장소"(Redis나 블룸 필터 등)를 유지해야 한다.

멱등적 데이터 수집

이 예제는 해시 기반 중복 제거 저장소(여기서는 집합으로 시뮬레이션)를 사용하는 멱등적 "싱크"를 보여준다. process_payment 함수가 동일한 이벤트로 여러 번 호출되어도 데이터베이스는 단 한 번만 업데이트된다.

import hashlib
import uuid

class ResilientProcessor:
    def __init__(self):
        # In production, this would be a distributed store like Redis with a TTL
        self.dedup_store = set()
        self.db = {}

    def _generate_idempotency_key(self, payload: dict) -> str:
        """
        Creates a unique deterministic hash of the business logic
        to prevent duplicate processing of the same data.
        """
        # Sorting keys to ensure the hash is deterministic
        payload_string = str(sorted(payload.items()))
        return hashlib.sha256(payload_string.encode()).hexdigest()

    def process_transaction(self, request_id: str, payload: dict):
        """
        Processes a payment idempotently using a Request ID.
        """
        # 1. Check if the unique Request ID has already been processed
        if request_id in self.dedup_store:
            print(f"Duplicate detected for Request {request_id}. Skipping...")
            return self.db.get(request_id)

        # 2. Business Logic (e.g., updating a balance)
        user_id = payload['user_id']
        amount = payload['amount']

        # Simulate DB Write
        self.db[request_id] = {"status": "confirmed", "amount": amount}

        # 3. Commit to Deduplication Store
        self.dedup_store.add(request_id)
        print(f"Successfully processed transaction {request_id} for ${amount}")
        return self.db[request_id]

# Simulation
processor = ResilientProcessor()
payment_event = {"user_id": "user_99", "amount": 250.00}
unique_token = str(uuid.uuid4())

# First attempt
processor.process_transaction(unique_token, payment_event)

# Second attempt (Retry due to simulated network timeout)
processor.process_transaction(unique_token, payment_event)

2026년의 변화: 현대 프레임워크의 EoS

2026년 면접에서는 특정 도구들이 이것을 어떻게 규모에 맞게 처리하는지 논의할 수 있어야 한다.

  • Kafka 트랜잭션: enable.idempotence=true와 트랜잭션 프로듀서를 사용해 메시지가 여러 파티션에 원자적으로 기록되도록 보장한다.
  • Upsert 로직: INSERT ... ON CONFLICT(SQL)나 merge()(Delta Lake/Hudi)를 사용해 늦게 도착한 중복 메시지가 새 레코드를 만드는 대신 기존 레코드를 업데이트하도록 한다.
  • 상태 저장 처리: Apache Flink가 "체크포인트"를 사용해 장애 발생 시 전체 시스템 상태를 일관된 시점으로 롤백하는 방식.
  • 부작용: 데이터베이스 쓰기는 멱등적으로 만들 수 있지만, 이메일 발송 같은 외부 부작용은 롤백하기가 훨씬 어려우며 "Saga 패턴"이 필요한 경우가 많다.

3. 데이터 계약과 스키마 진화

2026년에는 "데이터 계약"이 수동 QA와 "파이프라인이 깨졌다"는 Slack 알림을 대체했다. 데이터 계약은 데이터 제공자와 소비자 사이에서 교환되는 데이터의 구조, 품질, 의미를 정의하는 공식 합의다.

왜 중요한가

현대 데이터 플랫폼의 가장 큰 병목은 "업스트림 드리프트"다. 백엔드 엔지니어가 데이터베이스 스키마를 변경해 의도치 않게 다운스트림 대시보드를 망가뜨리는 상황이다. 데이터 계약은 이 "조용한 실패"를 "빌드 실패"로 바꿔, 나쁜 데이터가 데이터 웨어하우스에 들어오기 전에 막는다.

장점

  • 분리된 아키텍처: 계약이 지켜지는 한 프로듀서가 내부 로직을 변경해도 다운스트림 컨슈머가 영향받지 않는다.
  • 자가 치유 파이프라인: 계약을 위반하는 데이터를 자동으로 격리하는 도구를 구축할 수 있다.
  • 감사 추적성: GDPR/CCPA 같은 규정 준수에 필수적인 스키마 변경 이력을 명확하게 제공한다.

단점

  • 초기 마찰: 여러 팀(제품, 엔지니어링, 데이터)의 참여가 필요해 초기 개발이 느려질 수 있다.
  • 도구 성숙도: 많은 조직이 커스텀 스크립트 없이 기존 CI/CD 파이프라인에 계약 검증을 통합하는 데 어려움을 겪는다.

Pydantic 기반 계약 검증

이 예제는 Pydantic V2를 사용해 수집 레이어에서 계약을 검증하는 방법을 보여준다. 데이터가 저장소에 도달하기 전에 타입을 검증하고, 필수 필드를 강제하며, 범위 검사 같은 커스텀 로직을 적용한다.

from pydantic import BaseModel, Field, ValidationError, field_validator
from datetime import datetime
from typing import List, Optional

# The 'Contract' Definition
class UserActivityContract(BaseModel):
    user_id: int = Field(gt=0)
    event_type: str = Field(pattern="^(click|view|purchase)$")
    timestamp: datetime
    metadata: Optional[dict] = None
    tags: List[str] = Field(default_factory=list)

    @field_validator('timestamp')
    @classmethod
    def ensure_not_future(cls, v):
        if v > datetime.now():
            raise ValueError('Timestamp cannot be in the future')
        return v

def ingest_data(raw_payload: dict):
    """
    Simulates an ingestion entry point that enforces the contract.
    """
    try:
        # Validating the payload against the contract
        contract_validated_data = UserActivityContract(**raw_payload)
        # If valid, proceed to sink (S3/Snowflake)
        print(f"Contract Passed: Processing user {contract_validated_data.user_id}")
        return True
    except ValidationError as e:
        # In 2026, we 'Dead Letter' this instead of letting the pipeline crash
        print(f"Contract Violated: Sending to Dead Letter Queue (DLQ). Details: {e.json()}")
        return False

# Example: Data drift where a producer sends a future timestamp
bad_payload = {
    "user_id": 42,
    "event_type": "click",
    "timestamp": "2099-01-01T12:00:00"  # Invalid: Future date
}
ingest_data(bad_payload)

2026년의 변화: 왼쪽으로 이동하는 데이터 품질

면접관들은 품질 검사를 웨어하우스(dbt 테스트)에서 소스(Kafka/API)로 앞당기는 "Shift-Left" 능력을 찾는다. 언급해야 할 핵심 개념들은 다음과 같다.

  • 스키마 레지스트리: Confluent Schema Registry나 AWS Glue를 사용해 메시지 버스 레벨에서 Avro/Protobuf 스키마를 강제한다.
  • 코드로서의 계약(Contract-as-Code): 계약을 YAML이나 JSON으로 Git 저장소에 저장하고 프로듀서와 컨슈머가 모두 참조한다.
  • CI/CD를 통한 거버넌스: Pull Request 생성 시 "계약 호환성" 검사를 실행해 파괴적인 변경이 병합되지 않도록 막는다.

2. 수평 확장: 파티셔닝과 샤딩

"무한 데이터" 시대에 단일 데이터베이스 인스턴스는 결국 하드웨어의 한계에 부딪힌다. 파티셔닝과 샤딩은 대규모 확장을 위해 데이터를 여러 논리적 또는 물리적 경계로 나누는 아키텍처 패턴이다.

왜 중요한가

파티셔닝 없이는 백업, 청소(vacuum), 스키마 변경 같은 유지보수 작업이 프로덕션을 며칠씩 막는 작업이 된다. 샤딩은 선형 확장을 가능하게 한다. 데이터가 두 배로 늘면 노드를 두 배로 늘리면 되고, 성능은 일정하게 유지된다. 2026년에 하루 수십억 건의 이벤트를 처리하는 시스템에서 이것은 설계의 성패를 가르는 결정이다.

장점

  • 병렬 처리: 여러 CPU 코어나 노드가 서로 다른 파티션을 동시에 처리할 수 있다.
  • 가용성 향상: 한 샤드의 장애(예: "미국 동부 사용자" 샤드)가 다른 사용자의 시스템 전체를 반드시 망가뜨리지는 않는다.
  • 효율적인 데이터 보존: "2025년 1월" 파티션 전체를 비싼 DELETE 쿼리 없이 즉각 삭제할 수 있다.

단점

  • 쿼리 복잡성: "크로스-샤드 조인"은 악명 높을 정도로 느리며, 복잡한 애플리케이션 수준의 로직이 필요한 경우가 많다.
  • 핫스팟: 샤드 키를 잘못 선택하면(예: 뉴스 사이트를 날짜로 샤딩) 한 샤드에 트래픽이 몰리고 나머지는 한가해진다.
  • 리샤딩의 고통: 시스템이 운영 중인 상태에서 샤드 간 데이터를 이동하는 것은 위험도 높고 지연 시간도 큰 작업이다.

일관된 해시 샤더 구현

일관된 해싱(Consistent Hashing)은 2026년 샤드 간 데이터 분산의 표준 방법이다. 새 데이터베이스 노드를 추가할 때 전체 데이터를 재분배하는 대신 아주 일부만 이동시키면 된다.

import hashlib

class ShardManager:
    def __init__(self, nodes=None):
        """
        Initializes a shard manager with a list of physical DB nodes.
        """
        self.nodes = nodes or []
        self.node_map = {}
        self._generate_hashes()

    def _generate_hashes(self):
        # In a real system, we would use virtual nodes to ensure even distribution
        for node in self.nodes:
            h_key = self._hash_func(node)
            self.node_map[h_key] = node

    def _hash_func(self, key):
        """Standard MD5 hash to determine placement."""
        return int(hashlib.md5(key.encode()).hexdigest(), 16)

    def get_node(self, user_id):
        """
        Determines which database shard a user's data lives on.
        """
        if not self.node_map:
            return None
        user_hash = self._hash_func(str(user_id))
        # Find the first node hash greater than the user hash (clockwise on the ring)
        sorted_keys = sorted(self.node_map.keys())
        for node_hash in sorted_keys:
            if user_hash <= node_hash:
                return self.node_map[node_hash]
        # Wrap around to the first node
        return self.node_map[sorted_keys[0]]

# Example Usage
shards = ["db-node-01", "db-node-02", "db-node-03"]
manager = ShardManager(shards)

users = [1001, 2045, 3099, 4500]
for u in users:
    target_node = manager.get_node(u)
    print(f"User {u} is routed to shard: {target_node}")

2026년의 변화: 선언적 파티셔닝

현대 면접은 선언적 파티셔닝과 애플리케이션 수준 샤딩에 대한 지식을 테스트한다.

  • 목록 파티셔닝(List Partitioning): Country_Code 같은 이산 값으로 데이터를 분류한다.
  • 범위 파티셔닝(Range Partitioning): 월별 created_at처럼 시간 간격으로 데이터를 분할한다. BigQuery와 Snowflake 같은 데이터 웨어하우스의 업계 표준이다.
  • 해시 샤딩(Hash Sharding): 기본 키의 해시를 사용해 균등한 분산을 보장하고 "핫 샤드"를 방지한다.
  • 멀티 테넌트 샤딩(Multi-tenant Sharding): 대형 엔터프라이즈 고객은 전용 하드웨어로 격리하고, 소규모 고객은 공유 노드에 그룹화한다.

1. 복원력 엔지니어링: 재해 복구와 카오스 엔지니어링

2026년에 데이터 플랫폼은 처리량만으로 평가받지 않는다. "평균 복구 시간(MTTR)"이 중요한 기준이 됐다. 단순한 백업을 넘어, 현대 데이터 엔지니어링은 파이프라인이 인프라 장애 시 우아하게 성능을 낮추거나 자가 치유할 수 있는 복원력 엔지니어링에 집중한다.

왜 중요한가

분산 시스템은 본질적으로 불안정하다. 데이터 아키텍처가 초대규모로 이동할수록, 구성 요소 장애(스팟 인스턴스 회수, 네트워크 파티션, 리전 전체 클라우드 장애)의 확률은 100%에 가까워진다. 사전에 테스트하지 않으면, 재해 복구의 첫 교훈은 긴급 상황의 프로덕션 인시던트에서 얻게 된다.

장점

  • MTTR 단축: 카오스 엔지니어링을 실천하는 팀은 인시던트에 훨씬 빠르게 대응한다.
  • 인프라 신뢰성: 오토 스케일링 그룹, 서킷 브레이커, 페일오버 레플리카가 실제로 설정대로 동작하는지 검증한다.
  • 비용 관리: 다운타임과 SLA 위반으로 인한 막대한 재정적 손실을 예방한다.

단점

  • 운영 부담: 실험을 설계하고 안전하게 실행하는 데 상당한 엔지니어링 시간이 필요하다.
  • 의도치 않은 장애 위험: 범위(폭발 반경)를 엄격히 제한하지 않으면 카오스 실험이 프로덕션으로 누출돼 예방하려던 바로 그 다운타임을 유발할 수 있다.

데이터 파이프라인의 카오스 시뮬레이션

이 예제는 데코레이터 패턴을 이용한 "카오스 래퍼"를 보여준다. 데이터 수집 태스크에 제어된 장애를 주입해 간헐적인 API 실패나 데이터베이스 타임아웃을 시스템이 어떻게 처리하는지 테스트한다.

import random
import time
import logging
from functools import wraps

# Configure logging for chaos events
logging.basicConfig(level=logging.INFO)
logger = logging.getLogger("ChaosEngineer")

def inject_chaos(failure_rate=0.3, max_delay=2):
    """
    Decorator to simulate network latency and intermittent failures.
    """
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            # Simulate Latency
            delay = random.uniform(0, max_delay)
            time.sleep(delay)
            # Simulate Random Failure (e.g., Connection Reset)
            if random.random() < failure_rate:
                logger.error(f"Chaos injected: Simulating failure in {func.__name__}")
                raise ConnectionError("Simulated Chaos: Database/API Unreachable")
            return func(*args, **kwargs)
        return wrapper
    return decorator

class DataPipeline:
    def __init__(self):
        self.retry_limit = 3

    @inject_chaos(failure_rate=0.4)
    def fetch_records(self):
        return {"status": "success", "data": [101, 102, 103]}

    def run_safe_ingestion(self):
        """
        Implementation of a resilient pattern (Retry Logic) to counter chaos.
        """
        attempt = 0
        while attempt < self.retry_limit:
            try:
                result = self.fetch_records()
                logger.info(f"Ingestion successful: {result}")
                return result
            except ConnectionError:
                attempt += 1
                wait_time = 2 ** attempt  # Exponential backoff
                logger.warning(f"Attempt {attempt} failed. Retrying in {wait_time}s...")
                time.sleep(wait_time)
        logger.critical("Circuit Breaker: All retries failed. Triggering Disaster Recovery.")

# Execution
if __name__ == "__main__":
    pipeline = DataPipeline()
    pipeline.run_safe_ingestion()

2026년의 변화: "백업"에서 "관찰 기반 복구"로

현대 면접은 재해 복구가 단순히 pg_dump나 S3 버전 관리 이상이라는 것을 알고 있는지 확인한다.

  • 폭발 반경 격리(Blast Radius Isolation): 셀 기반 아키텍처를 사용해 한 마이크로서비스의 장애가 연쇄적으로 확산되지 않도록 한다.
  • 관찰 기반 페일오버: Prometheus나 OpenTelemetry 신호를 사용해 서비스 메시(Istio 등)를 통한 트래픽 재지정을 자동으로 트리거한다.
  • 상태 재구성: 오래된 스냅샷에 의존하는 대신, 메시지 버스(Kafka 등)에서 이벤트를 재생해 손상된 상태를 재구축하는 능력.
