DuckDB vs PostgreSQL: The Analytics Revolution Nobody Saw Coming

Yamishift · 6분 읽기 · 원문 · freedium

요약

아무도 예상하지 못한 사이, DuckDB가 분석 워크로드의 판도를 바꾸며 복잡한 데이터 인프라를 단 몇 줄의 코드로 대체하고 있다.

PostgreSQL은 트랜잭션 처리에 최적화된 데이터베이스로, 현대 백엔드 아키텍처의 근간을 이루고 있다. 그러나 대규모 분석 쿼리에는 근본적으로 적합하지 않아, 팀들은 Kafka, Spark, 데이터 웨어하우스 등 수많은 레이어를 쌓는 방식으로 대응해 왔다. DuckDB는 컬럼형 저장 방식과 고도로 최적화된 실행 엔진으로 이 문제에 정면 도전한다. 노트북 한 대에서 수억 건의 Parquet 파일을 별도 인프라 없이 직접 쿼리할 수 있다. 두 데이터베이스는 경쟁 상대가 아니라 역할이 다른 도구다. PostgreSQL은 트랜잭션을, DuckDB는 분석을 각각 담당하는 구조가 현명한 선택이다.

이 10년을 통틀어 가장 흥미로운 데이터베이스 이야기는 AI가 아니다

수년간 데이터베이스 대화는 늘 같은 흐름이었다.

트랜잭션이 필요하면 PostgreSQL을 쓴다. 분석이 필요하면 웨어하우스를 구축한다. 규모를 키워야 하면 인프라를 더 늘린다.

누구나 이 공식을 알고 있었다.

그런데 어느 순간 이상한 일이 벌어졌다. 엔지니어들이 노트북 한 대로 돌린 분석 쿼리가, 클러스터와 컨테이너와 오케스트레이션 레이어에 적지 않은 클라우드 비용까지 쏟아부은 시스템을 성능에서 앞지르기 시작한 것이다.

그 주인공은 수십억 달러짜리 플랫폼이 아니었다. DuckDB였다.

그러자 데이터 엔지니어, 백엔드 개발자, 분석가, 머신러닝 팀 할 것 없이 모두 같은 질문을 던지기 시작했다.

쿼리 엔진 자체가 이렇게 빨라졌는데, 왜 우리는 아직도 데이터를 이리저리 옮기고 있는 걸까?

이 질문의 답이 현대 분석 아키텍처에서 가장 큰 변화 중 하나를 만들어내고 있다. 아직 대부분의 사람들이 눈치채지 못했을 뿐이다.

PostgreSQL이 지난 시대를 지배한 이유

DuckDB의 부상을 이해하려면, 먼저 PostgreSQL이 왜 거의 모든 문제의 기본 답이 되었는지를 살펴볼 필요가 있다.

PostgreSQL은 역대 가장 성공적인 소프트웨어 프로젝트 중 하나다. OLTP 워크로드, ACID 트랜잭션, 복잡한 관계형 모델, 동시 사용자 처리, 복제, 고가용성, 프로덕션 API, 인증 시스템, 금융 트랜잭션까지 폭넓게 처리한다.

현대 백엔드 아키텍처의 상당 부분이 조용히 PostgreSQL 위에서 돌아가고 있다.

전형적인 프로덕션 아키텍처를 보면 이렇다.

└──────┬───────┘
       │

│  PostgreSQL  │
└──────┬───────┘
       │
┌─────────▼─────────┐
│   Redis / Kafka   │
└───────────────────┘

문제는 PostgreSQL이 대규모 분석과는 전혀 다른 목적에 맞게 설계되었다는 점이다.

PostgreSQL이 잘 답하도록 설계된 질문은 이런 것이다.

이 고객 한 명에게 무슨 일이 있었는가?

반면 분석이 던지는 질문은 다르다.

지난 18개월 동안 5억 명의 고객에게 무슨 일이 있었는가?

이 둘은 근본적으로 다른 문제다.

PostgreSQL에서 분석을 돌릴 때 치르는 숨은 비용

많은 팀이 이 사실을 뒤늦게, 그것도 고통스럽게 깨닫는다.

처음에는 모든 게 잘 돌아간다. 그러다 제품 관리자가 대시보드를 요청하기 시작한다. 마케팅팀은 퍼널 분석을 원하고, 재무팀은 코호트 분석을 요구하고, 경영진은 주간 트렌드를 보고 싶어하고, 데이터 과학팀은 과거 집계 데이터를 달라고 한다.

어느새 프로덕션 트래픽을 처리하던 데이터베이스가 분석 워크로드까지 떠안게 된다. 이런 쿼리가 곳곳에 등장한다.

SELECT
  DATE(created_at),
  COUNT(*),
  SUM(total_amount)
FROM orders
WHERE created_at >= NOW() - INTERVAL '365 days'
GROUP BY DATE(created_at)
ORDER BY DATE(created_at);

얼핏 보면 무해해 보인다. 그렇지 않다.

데이터가 많으면 이 쿼리 하나가 수백만 건의 행을 스캔한다. 그게 하루에 수백 번, 수십 명이 동시에 실행한다. 트랜잭션 데이터베이스가 분석 워크로드와 싸움을 벌이는 상황이 된다. 결국 양쪽 모두가 피해를 입는다.

전통적인 해결책은 지나치게 복잡해졌다

업계의 반응은 예측 가능했다. 인프라를 더 쌓는 것이다. 훨씬 더 많이.

기술적으로는 맞는 접근이다. 운영 측면에서는 소모적이다.

레이어가 하나씩 추가될 때마다 장애 지점도, 지연도, 비용도, 조율 부담도 늘어난다.

그리고 불편한 진실이 하나 있다. 많은 회사들이 실제로 필요해지기 훨씬 전에 이런 아키텍처를 도입했다. 그 결과는 실질적인 생산성보다 이론적인 규모에 최적화된 시스템 세대였다.

DuckDB의 등장

DuckDB는 분석 문제에 완전히 다른 각도로 접근했다.

기존의 질문은 이것이었다. "어떻게 하면 클러스터를 더 크게 만들 수 있을까?"

DuckDB의 질문은 달랐다. "분석 실행 자체를 극도로 효율적으로 만들면 어떨까?"

DuckDB는 OLAP 워크로드를 위해 설계된 컬럼형 분석 데이터베이스다. 설명만 들으면 평범하게 들린다. 하지만 그 의미는 전혀 평범하지 않다.

수억 건의 행이 담긴 Parquet 데이터셋을 생각해보자. DuckDB는 이걸 직접 쿼리할 수 있다. 데이터를 불러올 필요도, ETL도, 웨어하우스 마이그레이션도, Spark 클러스터도, Kubernetes 배포도 필요 없다.

import duckdb
result = duckdb.sql("""
  SELECT
    country,
    SUM(revenue) as revenue
  FROM 'sales.parquet'
  GROUP BY country
  ORDER BY revenue DESC
""").df()

이게 전부다. 쿼리 하나. 인프라 의식 없음.

왜 이렇게 빠른가

이 속도는 마법이 아니다. 아키텍처의 차이다.

PostgreSQL은 행 단위로 데이터를 저장한다. DuckDB는 열 단위로 저장한다.

아래 테이블을 예로 들자.

id | country | revenue | created_at

revenue와 country만 필요한 쿼리가 있다면, PostgreSQL은 전체 행을 읽는 경우가 많다. DuckDB는 필요한 열만 읽는다. 분석 워크로드에서는 이게 엄청난 차이를 만든다. 데이터가 수십억 건에 달할수록 더욱 그렇다. I/O가 줄고, 메모리 부담이 줄고, 불필요한 작업이 사라진다. 현대 CPU 구조와도 잘 맞는다.

아무도 말하지 않는 개발자 생산성 혁명

가장 큰 변화는 성능이 아니다. 생산성이다.

수년간 분석 작업은 인프라 전문가의 영역이었다. 오늘날 개발자 혼자서도 직접 질문에 답을 찾을 수 있다.

  • 백엔드 개발자가 프로덕션 내보내기 데이터를 로컬에서 바로 분석할 수 있다.
  • 데이터 과학자가 웨어하우스 접근 권한을 요청하지 않고도 Parquet 파일을 쿼리할 수 있다.
  • 제품 엔지니어가 몇 분 만에 사용자 행동 패턴을 파악할 수 있다.

궁금증에서 통찰까지 이어지는 장벽이 무너지고 있다. 이것이 진짜 혁명이다.

실제 백엔드 사례

이커머스 플랫폼을 운영한다고 가정하자. 프로덕션 시스템은 PostgreSQL을 쓰고, 주문은 FastAPI 서비스로 들어온다.

프로덕션 쓰기 경로

@router.post("/orders")
async def create_order(
  payload: OrderCreate,
  db: AsyncSession
):
  order = Order(
    customer_id=payload.customer_id,
    total=payload.total
  )
  db.add(order)
  await db.commit()
  return {"order_id": order.id}

이런 경우는 PostgreSQL이 딱 맞는 도구다. 트랜잭션, 일관성, 동시성 — 모두 중요하다. DuckDB가 이걸 대체해서는 안 된다. 많은 사람들이 범하는 실수가 바로 이것이다.

잘못된 아키텍처

DuckDB 벤치마크를 보고 이렇게 생각하는 팀이 있다.

DuckDB
   │
모든 것

이 접근은 대부분 잘못된 결과로 끝난다. DuckDB는 트랜잭션 데이터베이스를 대체하도록 설계되지 않았다. 주된 쓰기 저장소로 사용하려 하면 문제가 생긴다. 성숙한 트랜잭션 보장, 다중 사용자 쓰기 패턴, 운영 도구, 프로덕션 생태계 지원 — 이 모두를 잃게 된다. 잘못된 도구, 잘못된 문제다.

실용적인 아키텍처

영리한 팀들은 트랜잭션과 분석 역할을 점점 분리하고 있다.

┌──────────────┐
│  FastAPI API │
└──────┬───────┘
       │
┌──────▼───────┐
│  PostgreSQL  │
└──────┬───────┘
       │
  Outbox Pattern
       │
┌──────▼───────┐
│    Kafka     │
└──────┬───────┘
       │
┌──────▼───────┐
│   Parquet    │
└──────┬───────┘
       │
┌──────▼───────┐
│    DuckDB    │
└──────────────┘

이 아키텍처는 각 시스템이 가장 잘하는 일을 하도록 역할을 나눈다.

Outbox 패턴이 판도를 바꾼다

확장 가능한 백엔드 시스템에서 흔히 저지르는 실수 중 하나는 쓰기 작업과 하위 분석 파이프라인을 직접 연결하는 것이다.

나쁜 구현:

async def create_order(order):
  await save_order(order)
  await kafka.publish(
    "orders",
    order.dict()
  )

Kafka가 실패하면 막힌다. 일관성 문제가 복잡해진다.

더 나은 구현:

async def create_order(order, db):
  async with db.begin():
    db.add(Order(**order))
    db.add(
      OutboxEvent(
        event_type="order_created",
        payload=order
      )
    )

백그라운드 워커:

while True:
  events = fetch_pending_events()
  for event in events:
    kafka.publish(
      event.event_type,
      event.payload
    )
    mark_processed(event.id)

이 패턴은 안정성을 크게 높인다. 같은 이벤트가 나중에 DuckDB가 소비하는 분석 데이터셋으로 흘러들어갈 수도 있다.

멱등성이 데이터베이스보다 중요하다

많은 확장 실패는 데이터베이스 문제가 아니다. 재시도 처리 문제다.

나쁜 구현:

@app.post("/payments")
async def process_payment(request):
  charge_card()
  save_payment()
  return {"success": True}

요청을 재시도하면 결제가 두 번 발생한다. 금융 사고가 생긴 것이다.

더 나은 구현:

@app.post("/payments")
async def process_payment(
  request,
  idempotency_key: str
):
  existing = await redis.get(
    idempotency_key
  )
  if existing:
    return existing
  result = charge_card()
  await redis.set(
    idempotency_key,
    result
  )
  return result

데이터베이스 선택이 나쁜 시스템 설계를 구해줄 수는 없다.

인프라를 늘리는 것보다 가시성이 낫다

성능 문제가 생기면 팀들은 흔히 서비스를 추가하는 방식으로 반응한다. 사실 대부분 필요한 것은 가시성이다.

구조화된 로깅:

logger.info(
  "query_executed",
  query_type="analytics",
  duration_ms=duration,
  rows_scanned=rows
)

메트릭:

analytics_query_duration.observe(
  execution_time
)

트레이싱:

with tracer.start_as_current_span(
  "analytics_query"
):
  run_query()

대부분의 확장성 문제는 CPU 문제처럼 보이지만, 사실은 조율 문제다.

우리가 이토록 복잡성을 쌓아온 이유

이것은 무능함의 결과가 아니다. 역사의 결과다.

10년 전에는 스토리지가 비쌌고, CPU는 지금보다 훨씬 약했고, 로컬 분석은 제한적이었으며, 클라우드 웨어하우스는 실제 문제를 해결해 주었다. 엔지니어들은 당시의 제약에 맞게 최적화했다. 그 제약이 바뀌었다. 도구도 바뀌었다. 하지만 아키텍처 습관은 그대로 남았다.

원래 문제가 사라진 뒤에도 조직은 복잡성 부채를 계속 갚고 있는 경우가 많다.

마이크로서비스 대 모놀리스에서도 같은 교훈을 얻었다

DuckDB 이야기는 또 다른 흐름과 닮아 있다. 수년간 팀들은 시스템을 수십 개의 서비스로 나눴다. 사용자 서비스, 주문 서비스, 재고 서비스, 청구 서비스, 알림 서비스, 분석 서비스로 말이다.

그러다 결국 깨달았다. 대부분의 통신이 내부에서 일어나고 있었고, 복잡성의 대부분은 스스로 만들어낸 것이었다. 이제 많은 조직이 모듈형 모놀리스로 돌아오고 있다.

┌─────────────────────────┐
│      Modular Monolith   │
├─────────────────────────┤
│      Orders Module      │
│      Billing Module     │
│    Inventory Module     │
│     Analytics Module    │
└─────────────────────────┘

목표는 서비스를 줄이는 것이 아니다. 불필요한 결정을 줄이는 것이다. 개발자 생산성이 인프라보다 빠르게 성장하는 경우가 많다.

백그라운드 작업은 여전히 중요하다

분석 파이프라인은 사용자 요청을 막아서는 안 된다.

나쁜 예:

@app.post("/generate-report")
async def generate_report():
  run_heavy_report()
  return {"done": True}

좋은 예:

@app.post("/generate-report")
async def generate_report():
  task_id = queue.enqueue(
    "generate_report"
  )
  return {"task_id": task_id}

워커:

@consumer.task
async def generate_report():
  data = duckdb.sql("""
    SELECT *
    FROM reports.parquet
  """).df()
  export_report(data)

비동기 워크플로는 가장 쉽게 얻을 수 있는 확장성 향상 중 하나다.

속도 제한이 최적화보다 더 많은 시스템을 살린다

데이터베이스 부하의 상당 부분은 규모의 문제가 아니라 남용에서 비롯된다.

limiter = Limiter(
  key_func=get_remote_address
)
@app.get("/analytics")

  ...

때로는 나쁜 트래픽을 막는 것이 최고의 성능 최적화다.

이 조언이 통하지 않는 경우

DuckDB는 만능 해결책이 아니다. 다음 상황에서 PostgreSQL을 대체해서는 안 된다.

  • 높은 동시 쓰기가 필요할 때
  • 금융 트랜잭션을 처리할 때
  • 운영 워크로드를 다룰 때
  • 사용자 대면 애플리케이션을 구축할 때
  • 복잡한 트랜잭션 보장이 필요할 때

반면 PostgreSQL이 힘들어지는 상황도 있다.

  • 수십억 건의 레코드를 스캔할 때
  • 분석 집계 쿼리를 실행할 때
  • 대용량 Parquet 데이터셋을 조회할 때
  • 탐색적 분석을 수행할 때

워크로드가 다르면 도구도 달라야 한다. 엔지니어링 성숙도란 그 차이를 아는 것이다.

오늘날 영리한 팀들이 실제로 하는 것

점점 더 많은 팀의 아키텍처가 이런 모습을 갖춰가고 있다.

운영 레이어: FastAPI, PostgreSQL, Redis, RabbitMQ 또는 Kafka

분석 레이어: Parquet, DuckDB, Python, 데이터 노트북

관찰 가능성: OpenTelemetry, Prometheus, Grafana

배포: Docker, Kubernetes (정말 필요할 때만)

유행을 따르는 것이 아니다. 마찰을 최소화하기 때문이다. 그리고 마찰은 쌓인다.

진짜 교훈

DuckDB 대 PostgreSQL 논쟁의 프레임이 잘못되어 있다. 이것은 경쟁이 아니다. 역할 분담이다.

PostgreSQL은 역대 가장 뛰어난 트랜잭션 데이터베이스 중 하나로 여전히 건재하다. DuckDB는 분석 분야에서 무엇이 가능한지를 다시 정의하고 있다.

놀라운 점은 DuckDB가 빠르다는 사실이 아니다. 분석 실행이 느렸던 시절에 그 느림을 해결하기 위해 얼마나 많은 인프라가 존재했는지가 놀라운 것이다.

그 전제는 이제 사실이 아니다. 전제가 바뀌면 아키텍처가 따라온다.

다음 세대의 데이터 플랫폼은 더 큰 클러스터로 정의되지 않을지도 모른다. 훨씬 적은 클러스터가 필요한 것으로 정의될지도 모른다.

그것이 아무도 예상하지 못한 분석 혁명이다.
