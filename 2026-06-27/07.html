AI Agents Can't Read Your Database — Build This Layer

Unknown Author · 7분 읽기 · 원문 · freedium

요약

AI 에이전트에게 직접 데이터베이스를 열어주면 환각이 발생한다. 시맨틱 레이어가 그 사이의 필수 중간층이다.

LLM을 데이터 웨어하우스에 연결할 때 가장 먼저 떠오르는 방법은 스키마와 프롬프트를 넘겨주는 것이다. 데모에서는 완벽하게 작동하지만, 프로덕션에 투입하는 순간 세 가지 구조적 결함이 드러난다. 스키마에는 비즈니스 의미가 없고, JOIN 그래프는 지뢰밭이며, SQL SELECT 문에는 접근 제어 개념이 존재하지 않는다.

시맨틱 레이어(semantic layer)는 이 문제를 해결하는 거버넌스 중간층이다. 메트릭과 차원, 올바른 조인 경로, 접근 정책을 사전에 정의해두면 에이전트는 raw 테이블이 아닌 이름 붙은 지표를 조회한다. 이것이 단순한 접근 권한 부여와 진정한 이해 사이의 차이다.

dbt Semantic Layer, Cube Core, LookML 같은 도구를 사용해 구축하고, 2026년 기준으로는 MCP(Model Context Protocol)를 통해 에이전트와 통합하는 것이 표준 패턴이 되었다. 이 구조를 갖춘 조직은 복잡한 분석 쿼리에서 80~90%의 정확도를 달성한다.

1. Text-to-SQL이 프로덕션에서 실패하는 이유

LLM을 데이터 웨어하우스에 연결하는 첫 번째 본능은 단순하다. 테이블 스키마를 넘겨주고, 예시 쿼리 몇 개를 붙이고, 프롬프트에 "이 질문에 답하라"고 지시하는 것이다. 데모 환경에서는 인상적으로 작동한다. 모델은 그럴듯한 SQL을 척척 생성하고, 숫자가 화면에 나타나며, 청중은 감탄한다.

그러나 프로덕션에 배포하는 순간 현실이 달라진다. 실패의 양상은 우리가 예상하는 것과 다르다. 현대 LLM은 문법적으로 완벽한 SQL을 종일 생성할 수 있다. 구문 오류가 문제가 아니다. 진짜 문제는 SQL 자체가 에이전트에게 넘겨줄 올바른 추상화 수준이 아니라는 점이다.

한 핀테크 기업의 사례가 이를 잘 보여준다. 그 팀은 text-to-SQL 에이전트를 500개 테스트 케이스로 검증했고, 모두 통과했다. 그러나 701번째 케이스, 그것도 VP 앞에서 진행된 라이브 데모 도중에 에이전트는 조용히 환각을 일으켰다. 틀린 숫자를 자신 있게 제시했고, 그 숫자가 왜 틀렸는지 아무도 즉시 알아채지 못했다. 환각을 모델의 문제가 아닌 인프라의 문제로 접근하는 조직이 복잡한 분석 쿼리에서 80~90%의 정확도를 달성한다는 사실은 이 맥락에서 매우 시사적이다.

같은 질문 — "지난 분기 매출이 얼마였나?" — 을 세 번 물었을 때 세 가지 다른 숫자가 반환된다면, 그것은 모델이 멍청해서가 아니다. 모델에게 해결할 수 없는 문제를 던져놓은 구조가 원인이다. 그리고 그 숫자 중 어느 것이 맞는지 알 수 없다는 사실이 더 무섭다.

2. 에이전트가 데이터에서 환각하는 세 가지 구조적 원인

text-to-SQL 에이전트가 raw 테이블을 직접 조회할 때 약 80%의 경우에 틀린 결과를 낸다는 관찰은 여러 독립적인 연구에서 수렴한다. 이 실패는 무작위적이지 않다. "에이전트가 경로를 이탈하는 지점은 정확히 시맨틱 레이어가 약한 곳"이라는 관찰, 그리고 "표준화된 스키마 없이는 AI가 탈선한다"는 결론은 같은 구조적 원인을 가리킨다.

첫째, 의미론적 정보가 스키마에 없다. orders라는 테이블 이름은 모델에게 "매출"이 총매출인지 순매출인지, 세금이 포함되는지, 환불은 제외되는지, 주문 시점 기준인지 출고 시점 기준인지를 전혀 알려주지 않는다. 모델은 그럴듯한 것을 추측한다. 그리고 그 추측은 매번 자신 있고, 매번 다를 수 있다. 세 부서가 "활성 사용자"를 각각 다르게 정의하고 있다면, AI는 인사이트를 제공하는 것이 아니라 구조화된 환각을 제공하는 것이다.

둘째, JOIN 그래프는 지뢰밭이다. 실제 데이터 웨어하우스에는 팬아웃 조인(fan-out join), 천천히 변하는 차원(slowly-changing dimension), 그리고 모두 "고객 테이블"처럼 보이는 세 개의 테이블이 공존한다. 데이터 집계 단위(grain)를 잘못 선택하면 숫자가 조용히 두 배로 집계된다. 모델에게는 자신이 틀렸다는 것을 알 방법이 없다. 결과는 항상 그럴듯해 보인다.

셋째, SELECT 문에는 접근 제어 개념이 없다. 올바른 쿼리와 다른 테넌트의 데이터를 노출하는 쿼리는 모델의 관점에서 동일하게 보인다. 권한 로직이 다운스트림에 있거나 아예 없다면, 에이전트는 교묘한 프롬프트 하나만으로 데이터 유출의 경로가 된다. 이것은 단순한 정확도 문제가 아니라 보안 문제다.

3. 시맨틱 레이어란 무엇인가

시맨틱 레이어(semantic layer)는 raw 테이블과 에이전트 사이에 존재하는 이해의 중간층이다. 단순한 뷰(view)나 API 래퍼가 아니다. 비즈니스 로직을 코드로 표현하고, 한 곳에서 관리하며, 모든 쿼리 경로에 일관되게 적용하는 거버넌스 계층이다.

시맨틱 레이어가 정의하는 것은 네 가지다. 첫째, 메트릭(metrics) — "매출"이 무엇을 의미하는지, 어느 테이블에서 오는지, 어떤 집계 함수를 쓰는지, 어떤 필터를 적용하는지를 명시한다. 둘째, 차원(dimensions) — 지역별, 코호트별, 제품 라인별로 어떻게 데이터를 슬라이스하는지를 정의한다. 셋째, 조인 경로(joins) — 올바른 JOIN 그래프를 사전에 정의하여 에이전트가 임의로 조인을 만들어낼 수 없게 한다. 넷째, 접근 정책(access policies) — 행 수준 보안(row-level security)과 역할 기반 접근 제어(role-based access control)를 쿼리 컴파일 시점에 적용한다.

에이전트가 시맨틱 레이어를 통해 조회할 때, 더 이상 raw 테이블에 SQL을 직접 작성하지 않는다. 대신, 거버넌스된 명명된 메트릭과 차원을 선택한다. 에이전트의 요청은 이렇게 바뀐다: "지난 분기의 제품 라인별 revenue를 보여줘." 시맨틱 레이어는 이 요청을 받아 올바른 조인, 올바른 필터, 올바른 접근 제어가 적용된 SQL을 생성한다.

이것이 접근 권한(access)과 이해(understanding)의 차이다. text-to-SQL은 에이전트에게 데이터에 대한 접근 권한을 준다. 시맨틱 레이어는 에이전트에게 데이터에 대한 이해를 준다. 전자는 에이전트가 무엇이든 시도할 수 있게 하고, 후자는 에이전트가 무엇을 요청할 수 있는지를 정의한다. 이 차이가 프로덕션의 안정성을 결정한다.

4. 시맨틱 레이어 구축 방법: dbt, Cube, LookML

시맨틱 레이어를 구축하는 도구는 여러 가지가 있으며, 2026년 기준으로 세 가지가 주요 선택지로 자리 잡았다. 어떤 도구를 선택하든, 구축 패턴은 동일한 단계를 따른다.

dbt Semantic Layer / MetricFlow는 이미 dbt를 사용하는 팀에게 자연스러운 선택이다. 메트릭을 dbt 모델 옆에 YAML 파일로 정의하고, Semantic Layer API를 통해 에이전트가 이를 조회한다. dbt의 데이터 변환 파이프라인과 메트릭 정의가 동일한 리포지토리에서 관리된다는 점이 강점이다. 코드 리뷰와 버전 관리가 메트릭 정의에도 자동으로 적용된다.

Cube Core(오픈소스)는 JS 또는 YAML로 데이터 스키마를 정의하고, REST, GraphQL, SQL 인터페이스를 통해 외부에 노출한다. 2026년에는 MCP 서버를 통한 에이전트 연결이 표준 통합 방식으로 자리 잡았다. 특정 데이터 플랫폼에 종속되지 않고 독립적으로 운영할 수 있다는 유연성이 장점이다.

LookML(Looker)은 Google Cloud 데이터 스택을 사용하는 조직에 적합한 비즈니스 로직 레이어다. 에이전트는 Looker API를 통해 사전 정의된 Explore와 메트릭을 조회한다. 수년간 엔터프라이즈 환경에서 검증된 거버넌스 모델을 갖추고 있다.

도구와 무관하게 구축 단계는 다섯 가지다. 첫째, 데이터 모델을 감사하여 비즈니스가 실제로 신경 쓰는 20~30개의 메트릭을 식별한다. 둘째, 메트릭 이름, 소스 테이블, 집계 방식, 필터, 그리고 평이한 한국어로 쓴 설명을 YAML로 정의한다. 셋째, 올바른 조인 경로를 명시하여 에이전트가 임의로 조인을 발명하지 못하게 한다. 넷째, 쿼리 실행 시점에 강제되는 행 수준 및 역할 기반 접근 규칙을 추가한다. 다섯째, MCP를 통해 에이전트가 메트릭을 발견하고 이름으로 조회할 수 있게 노출한다.

이 과정은 처음에는 느리게 느껴질 수 있다. 20개 메트릭을 정의하는 작업은 단순히 에이전트에게 데이터베이스 자격증명을 넘겨주는 것보다 훨씬 더 많은 노력을 요구한다. 그러나 그 노력은 한 번만 하면 된다. 그리고 그 이후의 모든 쿼리는 정의된 의미론적 경계 안에서 안전하게 실행된다.

5. MCP를 통한 에이전트 통합: 2026년 패턴

MCP(Model Context Protocol)는 2026년 기준으로 에이전트가 거버넌스된 메트릭을 발견하고 조회하는 표준 방식이 되었다. 이 프로토콜의 핵심 아이디어는 단순하다: 에이전트가 raw 테이블 이름, 컬럼 이름, 조인 로직을 볼 필요가 없다는 것이다. 에이전트는 시맨틱 레이어가 노출하는 것만 본다.

에이전트와 시맨틱 레이어 사이의 대화는 이렇게 흘러간다. 먼저 에이전트가 사용 가능한 메트릭 목록을 요청한다. MCP 서버는 이름 붙은 메트릭과 차원의 목록을 반환한다. 에이전트는 이 목록에서 필요한 것을 선택해 조회를 요청하고, 시맨틱 레이어는 올바른 조인과 필터, 접근 제어가 적용된 SQL을 생성해 실행한다.

# MCP를 통한 에이전트 쿼리 흐름

Agent → MCP server: "list available metrics"
MCP   → Agent:      [revenue, active_users, churn_rate, ...]

Agent → MCP: "get revenue by region for last_quarter"
MCP   → SQL: SELECT region, SUM(net_revenue)
             FROM orders
             JOIN regions ON orders.region_id = regions.id
             WHERE order_date >= '2025-10-01'
               AND order_date < '2026-01-01'
               AND tenant_id = :current_tenant  -- 접근 제어 자동 적용
             GROUP BY region

이 흐름에서 에이전트는 orders 테이블의 존재를 모른다. net_revenue와 gross_revenue의 차이를 모른다. region_id로 조인해야 한다는 사실을 모른다. 이 모든 지식이 시맨틱 레이어 안에 캡슐화되어 있기 때문이다. 에이전트가 알아야 할 것은 오직 두 가지다: 어떤 메트릭이 존재하는지, 그리고 어떻게 요청하는지.

이것이 text-to-SQL과의 근본적인 차이다. text-to-SQL에서 에이전트는 SQL 작성자 역할을 맡는다. MCP + 시맨틱 레이어 패턴에서 에이전트는 메트릭 소비자 역할을 맡는다. 전자는 데이터베이스 전문가의 지식을 요구하고, 후자는 비즈니스 언어로 대화한다. 멀티 테넌트 환경에서 접근 제어가 쿼리 컴파일 시점에 자동으로 주입된다는 점은 특히 중요하다. 프롬프트에 "다른 고객의 데이터는 보지 마"라고 쓰는 것과는 차원이 다른 보안이다.

6. 하지 말아야 할 것들

시맨틱 레이어를 구축하기 전에, 혹은 구축하는 대신 시도하고 싶어지는 지름길들이 있다. 각각이 왜 위험한지 이해하는 것이 구축을 결정하는 데 도움이 된다.

에이전트에게 직접 데이터베이스 자격증명을 주지 말라. 편리해 보이지만, 에이전트는 그 자격증명으로 접근할 수 있는 모든 것을 조회할 수 있다. 의도치 않은 데이터 노출의 표면적이 무한히 넓어진다.

프롬프트 엔지니어링만으로 스키마를 가르치려 하지 말라. "이 테이블의 revenue는 세금을 제외한 순매출입니다"라고 프롬프트에 써넣는 것은 단기적으로 작동할 수 있다. 하지만 프롬프트는 버전 관리되지 않고, 테스트되지 않으며, 팀 간에 일관되게 공유되지 않는다. 스키마가 바뀌면 프롬프트를 업데이트했는지 아무도 모른다.

접근 제어 로직을 프롬프트에 넣지 말라. "사용자 A의 데이터만 보여줘"를 프롬프트로 강제하는 것은 보안이 아니다. 프롬프트 인젝션 하나로 우회될 수 있다. 접근 제어는 쿼리 실행 경로 안에 있어야 한다.

검증 레이어 없이 text-to-SQL을 사용하지 말라. text-to-SQL이 반드시 나쁜 것은 아니다. 그러나 생성된 SQL이 의미론적으로 올바른지, 접근 제어가 올바르게 적용되었는지 검증하는 레이어 없이 사용하는 것은 위험하다.

에이전트가 프로덕션에 나간 뒤에 시맨틱 레이어를 사후 추가하지 말라. 에이전트가 먼저 배포되고 나면, 시맨틱 레이어를 끼워 넣는 작업은 이미 사용자들이 기대하게 된 동작을 변경할 수 있다. 시맨틱 레이어는 인프라다. 인프라는 처음부터 계획에 포함되어야 한다.

7. 결론: 접근 권한이 아니라 이해가 필요하다

Brex의 사례는 이 논의를 잘 마무리한다. 그들은 시맨틱 레이어 접근 방식을 채택한 뒤 이렇게 말했다: "시맨틱 레이어가 AI를 유용하게 만드는 것이다." 여기서 "유용하다"는 말은 단지 동작한다는 의미가 아니다. 신뢰할 수 있다는 의미다. 매번 동일한 질문에 동일한 답을 내놓고, 그 답이 왜 그런지 추적할 수 있다는 의미다.

raw 테이블에 직접 연결된 에이전트는 접근 권한을 가진 것이다. 시맨틱 레이어를 통해 조회하는 에이전트는 이해를 가진 것이다. 이 둘의 차이는 데모에서는 보이지 않는다. 501번째 쿼리에서, VP 앞에서, 혹은 보안 감사 때 드러난다.

시맨틱 레이어가 해결하는 문제는 모델의 능력 문제가 아니다. 구조의 문제다. 아무리 뛰어난 LLM이라도 orders 테이블의 revenue 컬럼이 세금을 포함하는지 알 수 없다. 그 정보가 어디에도 없기 때문이다. 시맨틱 레이어는 그 정보를 코드로 표현하고, 검증 가능하게 만들고, 모든 에이전트에게 일관되게 적용한다.

구축 순서는 명확하다. 비즈니스가 실제로 사용하는 메트릭 20~30개를 정의하고, YAML로 명문화하고, 올바른 조인 경로를 지정하고, 접근 정책을 추가하고, MCP를 통해 에이전트에게 노출한다. 이것이 선택 사항이 아닌 이유는 간단하다. 에이전트가 데이터에 대한 이해 없이 접근 권한만 가지는 순간, 그것은 단지 비싸고 자신감 넘치는 추측 기계가 된다.

"시맨틱 레이어가 AI를 유용하게 만드는 것이다." — Brex 팀

인프라는 선택 사항이 아니다. 시맨틱 레이어는 AI 에이전트 스택의 데이터 인프라다. 처음부터 계획하라.
