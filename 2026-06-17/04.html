The Spark Revolution You Didn't See Coming: How Apache Spark 4.0 in Databricks Just Changed Everything

MatMaq · 10 min read
원문 · freedium

요약

Apache Spark 4.0은 VARIANT 데이터 타입, ANSI SQL 모드 기본 활성화, Spark Connect, 선언적 파이프라인 등 혁신적 기능으로 빅데이터 처리 패러다임을 근본적으로 바꾸며 JSON 워크로드에서 최대 8배 성능 향상을 제공한다.

Apache Spark 4.0은 5년 만의 가장 큰 업그레이드로, Databricks Runtime 18.0에서 구동되며 데이터 엔지니어링 분야에 혁명적인 변화를 가져옵니다. 핵심 기능인 VARIANT 데이터 타입은 JSON 처리 시 최대 8배의 성능 향상과 80% 이상의 비용 절감을 가능하게 하며, ANSI SQL 모드 기본 활성화로 조용한 데이터 품질 문제를 사전에 탐지할 수 있습니다. Spark Connect는 클라이언트-서버 아키텍처를 통해 중앙화된 컴퓨팅과 탈중앙화된 접근을 동시에 실현하며, 선언적 파이프라인(SDP)은 Airflow 같은 별도의 오케스트레이션 도구를 불필요하게 만들 수 있습니다. Python 네이티브 기능 강화(데이터 소스 API, Arrow 네이티브 UDF)로 PySpark 개발자들이 이제 진정한 일급 시민이 되었으며, 구조화 스트리밍 실시간 모드를 통해 서브초 지연 시간도 지원됩니다. 저자는 3단계 마이그레이션 전략을 제시하며, 이 변화가 데이터 엔지니어들에게 레거시 최적화 스킬 대신 더 고수준의 아키텍처 역량을 요구할 것이라고 강조합니다.

아무도 예상 못한 스파크 혁명: Apache Spark 4.0이 Databricks에서 모든 것을 바꾼 방법

5년 만의 최대 업그레이드가 여러분의 데이터 엔지니어링 스택 절반을 구식으로 만들 이유

여러분의 프로덕션 파이프라인은 방금 40% 빨라졌고, 코드 한 줄도 바꾸지 않았습니다.

Spark로 작업하는 데이터 엔지니어나 데이터 과학자라면, 작업 최적화, 반정형 데이터와의 씨름, 그리고 이해관계자들에게 "그냥 Spark가 원래 그래요"라고 설명하는 데 수많은 시간을 보냈을 것입니다. 여기서 대부분의 사람들이 잘못 이해하는 게 있습니다: 그들은 Spark의 발전이 단순한 점진적 성능 향상에 불과하다고 생각합니다.

진실은? Apache Spark 4.0은 빅데이터 처리의 규칙을 근본적으로 다시 씁니다.

이 글에서는 Apache Spark 4.0과 그 후속 버전인 4.1로 구동되는 Databricks Runtime 18.0이, 프로덕션 배포 경험, 벤치마크 데이터, 그리고 이미 대규모 데이터 처리 방식을 뒤흔들고 있는 패러다임 전환을 근거로, 클라우드 마이그레이션 이후 데이터 엔지니어링 분야에서 가장 중요한 변화를 어떻게 대표하는지 보여 드리겠습니다.

아무도 말하지 않는 위기: 레거시 Spark 배포가 실패하는 이유

제가 데이터 인프라에 대한 생각을 바꾸게 된 어떤 대화를 들려드리겠습니다.

세 달 전, 저는 수십억 건의 환자 기록을 처리하는 헬스케어 분석 회사와 컨설팅을 하고 있었습니다. 그 회사는 Spark 인프라에 수백만 달러를 투자했고—완벽하게 최적화된 클러스터, 몇 년에 걸쳐 미세 조정된 파이프라인, 그리고 이를 유지 관리하는 시니어 엔지니어 팀을 갖추고 있었습니다.

그러다 CTO가 모두를 멈추게 만드는 질문을 했습니다: "왜 우리 JSON 데이터를 처리하는 데 드는 컴퓨팅 비용이 모든 구조화된 ETL을 합친 것보다 더 많이 드나요?"

정답은 기술 부채나 잘못된 최적화가 아니었습니다. 바로 아키텍처적 한계였습니다.

전통적인 Spark는 반정형 데이터를 문자열이나 복잡한 맵으로 처리합니다. 모든 쿼리는 단 하나의 필드만 필요로 해도 전체 JSON 블롭을 파싱해야 합니다. 그 회사의 경우, 이는 수 기가바이트의 데이터를 처리해서 수 킬로바이트의 인사이트를 추출한다는 것을 의미했습니다.

최근의 성능 분석에 따르면, 조직들은 반정형 데이터 처리에 막대한 컴퓨팅 자원을 낭비하고 있으며, 이는 "사업을 하는 비용"으로 받아들여질 만큼 광범위하게 퍼진 문제입니다.

하지만 여기에 논란의 여지가 있는 진실이 있습니다: 이것은 Spark의 문제가 아니라 데이터 타입의 문제였습니다. 그리고 Spark 4.0이 이를 해결했습니다.

게임 체인저: VARIANT 데이터 타입과 8배 성능 향상

Spark 4.0의 VARIANT 데이터 타입은 대부분의 엔지니어들이 아직 완전히 파악하지 못한 것을 나타냅니다: 이것은 단순히 JSON을 저장하는 새로운 방법이 아닙니다—이것은 분산 시스템에서 반정형 데이터가 작동하는 방식에 대한 근본적인 재구상입니다.

오늘날 데이터베이스가 JSON을 처리하는 방법을 생각해 보세요. 다음 중 하나입니다:

  • 매번 파싱 (비용이 많이 듦)
  • 모든 것을 인덱싱 (낭비적임)
  • 미리 스키마를 정의하도록 강제 (유연성 없음)

VARIANT는 근본적으로 다른 것을 합니다. 자주 쿼리되는 필드를 자동으로 식별하고 이를 별도의 타입이 지정된 Parquet 컬럼으로 추출하는 "슈레딩(shredding)"이라는 기법을 사용합니다. 중첩된 JSON은 유연하게 유지되지만, 자주 접근하는 데이터는 구조화된 컬럼의 성능을 얻게 됩니다.

실제 세계의 영향

마이그레이션 테스트에서 얻은 구체적인 수치를 알려드리겠습니다:

Spark 4.0 이전 (전통적인 JSON 처리):

  • 중첩 필드 추출을 위한 쿼리 시간: 45초
  • 스캔된 데이터: 실제 결과 15GB에 대해 2.3TB
  • 컴퓨팅 비용: 쿼리당 $12.50

Spark 4.0 이후 (슈레딩을 적용한 VARIANT):

  • 쿼리 시간: 5.8초
  • 스캔된 데이터: 280GB (슈레딩된 필드만)
  • 컴퓨팅 비용: 쿼리당 $1.60

이것은 점진적인 개선이 아닙니다. 이것은 경제적 변혁입니다.

초기 도입자들은 JSON 중심 워크로드에서 최대 8배의 성능 향상을 보고하고 있지만, 진짜 가치는 속도만이 아닙니다—전통적인 성능 패널티 없이 스키마-온-리드(schema-on-read) 데이터로 작업할 수 있는 유연성입니다.

ANSI SQL 모드: 실제로 모든 것을 고치는 Breaking Change

여기서 논쟁의 여지가 생깁니다: Spark 4.0은 기본적으로 ANSI SQL 모드를 활성화하여, 엔진이 오류 및 타입 변환을 처리하는 방식을 근본적으로 바꿉니다.

제가 이야기한 대부분의 엔지니어들은 이것을 마이그레이션 골치거리로 봅니다. 그들이 틀렸습니다.

이것은 Spark 역사에서 가장 중요한 단일 신뢰성 개선입니다. 이유는 이렇습니다: 조용한 실패가 요란한 실패보다 더 비쌉니다.

실제로 무엇이 달라지나

레거시 Spark에서는 유효하지 않은 연산이 null을 반환합니다. 파이프라인은 성공하지만, 데이터는 조용히 손상됩니다. 수천 개의 다운스트림 프로세스가 잘못된 데이터를 소비한 후, 분기별 감사 중에 문제를 발견하게 됩니다.

ANSI 모드에서는 0으로 나누기 및 숫자 오버플로와 같은 연산이 null을 반환하는 대신 명시적인 오류를 발생시킵니다. 삽입 중 유효하지 않은 타입 변환은 자동으로 강제 변환되는 대신 거부됩니다.

이것은 의도적으로 무언가를 깨뜨립니다.

하지만 프로덕션이 아닌 개발 중에 깨뜨립니다. 데이터 품질 문제가 데이터 레이크 전반으로 전파되기 전에 드러냅니다. Spark가 조직이 실제로 필요로 하는 엔터프라이즈 데이터베이스처럼 동작하게 만듭니다.

아무도 사용하지 않지만 사용해야 할 마이그레이션 전략

ANSI 모드를 비활성화하지 마세요. 대신, 이를 데이터 품질 발견 도구로 사용하세요.

병렬 환경에서 Spark 4.0으로 기존 파이프라인을 실행하세요. 발생하는 모든 오류는 프로덕션 데이터에 숨겨져 있던 조용한 데이터 품질 문제입니다. 증상이 아닌 근본 원인을 고치세요.

제가 함께 일했던 한 금융 서비스 회사는 이 방법을 통해 이전에는 알려지지 않았던 147개의 데이터 품질 문제를 발견했습니다. 그 중 각각은 그들이 몰랐던 잠재적인 컴플라이언스 위반이었습니다.

Spark Connect: 왜 여러분의 아키텍처가 변할 예정인가

Spark Connect는 애플리케이션을 Spark 클러스터로부터 분리하는 클라이언트-서버 아키텍처를 도입하여, DataFrame API와 미해석 논리 플랜을 통한 원격 연결을 가능하게 합니다.

하지만 진정한 혁명은 기술적인 것이 아닙니다—조직적인 것입니다.

중앙화 전환

전통적인 Spark는 선택을 강요합니다:

  • 옵션 A: 모든 팀에게 자체 클러스터를 주기 (비용이 많이 들고, 거버넌스가 어려움)
  • 옵션 B: 클러스터를 공유하고 의존성 지옥을 감당하기 (불안정하고, 좌절스러움)

Spark Connect는 옵션 C를 도입합니다: 탈중앙화된 접근과 함께 중앙화된 컴퓨팅.

데이터 팀은 경량 클라이언트로 작업할 수 있습니다—더 이상 클러스터 구성을 관리하거나 호환되지 않는 라이브러리 버전을 처리할 필요가 없습니다. Python으로 작업하는 데이터 과학 팀, R로 작업하는 분석 팀, Scala로 작업하는 ML 엔지니어—모두 충돌 없이 동일하게 관리되는 Spark 인스턴스에 연결할 수 있습니다.

Databricks Runtime 18.0은 이제 Spark 4.1과 함께 출시되어, Spark Connect와 클래식 Spark 간의 완전한 기능 동등성을 달성하여 대부분의 워크로드에서 마이그레이션이 원활하게 이루어집니다.

아무도 논의하지 않는 보안 함의

이 아키텍처 전환에는 주목 받지 못하고 있는 엄청난 보안 이점이 있습니다: 코드가 클라이언트 머신에서 절대 실행되지 않습니다.

분석가들은 로컬에서 DataFrame 변환을 작성합니다. 해당 변환은 실행 가능한 코드가 아닌 논리 플랜으로 서버에 전송됩니다. 서버는 이를 검증하고, 최적화하고, 통제된 환경에서 실행합니다.

이것이 의미하는 바:

  • 사용자 노트북의 악성 코드가 클러스터에 도달하지 않음
  • 실제로 실행된 것의 완전한 감사 가능성
  • 규제 산업에서의 간소화된 컴플라이언스

Python 네이티브 기능: 마침내 일급 시민이 되다

수년간 PySpark 사용자들은 Spark 생태계에서 이류 시민이었습니다. 대부분의 것을 할 수 있었지만, 항상 Scala 모양의 구멍이 기능 세트에 있었습니다.

Spark 4.0이 이것을 완전히 바꿉니다.

Python 데이터 소스 API

새로운 Python 데이터 소스 API를 통해 개발자는 Scala나 Java 지식 없이도 완전히 Python으로 커스텀 데이터 소스와 싱크를 생성할 수 있습니다.

이는 Spark 확장 개발을 민주화합니다. 독점 API에서 읽고 싶으신가요? Python 커넥터를 작성하면 됩니다. 커스텀 파일 포맷이 필요하신가요? Python으로 구현하면 됩니다.

이전에는 다음이 필요했습니다:

  1. Scala 학습
  2. Spark 내부 이해
  3. JAR 컴파일 및 배포
  4. 모든 사람이 커스텀 빌드를 채택하도록 설득

이제는 그냥 Python을 작성하고 패키지로 공유하면 됩니다.

Arrow 네이티브 UDF: 성능 배수기

Spark 4.1은 Pandas 변환 오버헤드를 우회하고 PyArrow 배열과 직접 작동하는 arrow_udf와 arrow_udtf 데코레이터를 도입합니다.

기술적인 세부 사항보다 실질적인 영향이 더 중요합니다: Python UDF는 이제 프로덕션 워크로드에 충분히 빠릅니다.

저는 PySpark 성능이 마침내 따라잡으면서 전체 Scala 코드베이스를 버리는 팀을 봤습니다. 한 전자상거래 플랫폼은 23개의 Scala UDF를 Python으로 마이그레이션하고 유지 관리할 코드는 70% 줄이면서 유사한 성능을 달성했습니다.

선언적 파이프라인: DAG 도구를 없앨 기능

Spark 선언적 파이프라인(SDP)은 데이터셋과 쿼리를 정의하면 Spark가 실행 그래프, 의존성 순서, 병렬성, 체크포인트, 재시도를 처리하는 새로운 프레임워크입니다.

익숙하게 들리시나요? 그럴 것입니다—이것은 Airflow와 Dagster가 하는 모든 것이지만, Spark에 네이티브입니다.

이것이 중요한 이유

대부분의 조직은 외부 DAG 도구에 의해 오케스트레이션되는 Spark 작업을 실행합니다. 이는 몇 가지 문제를 만들어 냅니다:

  • 의존성 관리가 두 곳에서 발생함 (DAG 도구 + Spark)
  • 재시도가 일관성 없이 처리됨
  • 계보 추적에 커스텀 통합이 필요함
  • 테스트에 두 시스템 모두 가동이 필요함

SDP는 이 복잡성을 줄입니다. 어떻게 오케스트레이션할지가 아니라 무엇을 원하는지를 정의합니다. Spark가 실행 플랜을 파악하고, 실패를 처리하고, 상태를 관리하고, 계보를 추적합니다—모두 네이티브로.

오케스트레이션 엔지니어들을 불안하게 만드는 부분이 여기 있습니다: 데이터 변환 파이프라인의 경우, 이것은 독립적인 오케스트레이터를 불필요하게 만듭니다.

모든 것을 위한 것은 아닙니다—Airflow는 여전히 이종 워크플로에서 뛰어납니다. 하지만 순수한 Spark 파이프라인의 경우? SDP는 태스크 의존성이 아닌 데이터 의존성을 이해하기 때문에 더 잘합니다.

실시간 스트리밍: 서브초 지연 시간 도래

구조화 스트리밍 실시간 모드(RTM)는 서브초 지연 시간의 지속 처리를 공식적으로 지원하며—무상태 연산의 경우 지연 시간이 한 자릿수 밀리초까지 낮아질 수 있습니다.

이는 Spark와 Flink 같은 전용 스트리밍 엔진 사이의 마지막 남은 격차를 닫습니다.

이제 열리는 사용 사례

10ms 미만의 지연 시간으로, 이제 Spark를 다음에 사용할 수 있습니다:

  • 실시간 사기 탐지 (이전에는 전문 시스템이 필요했음)
  • 라이브 추천 엔진 (배치 지연을 허용할 수 없었음)
  • IoT 센서 분석 (더 빠른 피드백 루프가 필요했음)
  • 트레이딩 시스템 (밀리초가 중요함)

아름다운 점은 새로운 시스템을 배울 필요가 없다는 것입니다. 기존 Spark 지식이 실시간 처리에 직접 적용됩니다.

마이그레이션 전략: 혼란 없이 승리하는 방법

제가 성공적으로 안내한 마이그레이션을 기반으로, 여기에 전술적 플레이북이 있습니다:

1단계: 병렬 배포 (1~2주)

이중 환경 설정:

  • 현재 런타임에서 기존 프로덕션 유지
  • Databricks Runtime 18.0 Beta에서 새 클러스터 구축
  • 비핵심 워크로드의 5%를 새 런타임으로 라우팅

집중 영역:

  • JSON 중심 파이프라인에서 VARIANT 성능 테스트
  • ANSI 모드 위반 식별 (데이터 품질 이점으로 처리)
  • Python UDF 성능 향상 벤치마킹

2단계: 점진적 마이그레이션 (3~8주)

우선순위 순서:

  1. 반정형 데이터 워크로드 먼저 (VARIANT에서 가장 큰 ROI)
  2. Python 중심 파이프라인 두 번째 (Arrow 최적화 활용)
  3. 스트리밍 작업 세 번째 (RTM 활용)
  4. 레거시 Scala 작업 마지막 (가장 낮은 위험, 가장 낮은 이익)

피해야 할 안티패턴: 한꺼번에 모든 것을 마이그레이션하지 마세요. 새로운 프로젝트로 시작한 다음 고가치 기존 워크로드를 이동하세요.

3단계: 아키텍처 발전 (9~16주)

새 프로젝트에 Spark Connect 구현:

  • 컴퓨팅 거버넌스 중앙화
  • 경량 클라이언트 접근 활성화
  • 보안 경계 수립

적절한 곳에 선언적 파이프라인 도입:

  • 새로운 변환 파이프라인으로 시작
  • 단순한 Airflow DAG을 점진적으로 교체
  • 복잡한 다중 시스템 오케스트레이션은 기존 도구에 유지

프로덕션 마이그레이션의 실전 팁

1. VARIANT를 전략적으로 사용하되, 보편적으로 사용하지 마세요 모든 JSON이 슈레딩으로 이익을 얻는 것은 아닙니다. 드물게 쿼리되고 고도로 변수가 많은 스키마는 전통적인 문자열/맵 처리로 더 나은 성능을 낼 수 있습니다.

2. ANSI 모드 오류를 환영하세요 모든 오류는 몰랐던 데이터 품질 문제입니다. 추적 시트를 만들고, 수정 사항의 우선순위를 정하고, 이것이 CEO가 발견하기 전에 찾아준 Spark 4.0에 감사하세요.

3. 모든 것을 벤치마킹하세요 여러분의 워크로드는 독특합니다. 마케팅 수치(제 수치 포함)를 믿지 마세요. 병렬 작업을 실행하고 실제 성능 차이를 측정하세요.

4. Spark Connect 패턴 교육에 투자하세요 "코드가 내가 작성하는 곳에서 실행된다"에서 "논리 플랜이 원격으로 실행된다"로의 정신적 모델 전환은 시니어 엔지니어조차 실수하게 만듭니다. 교육에 투자하세요.

눈에 보이는 곳에 숨겨진 경쟁 우위

업계 분석가들이 아직 말하지 않는 것: Databricks Runtime 18.0의 Spark 4.0은 경쟁적 해자를 만드는 도구입니다.

일찍 마이그레이션하는 조직은 다음을 얻습니다:

비용 우위: 8배 빠른 JSON 처리는 반정형 워크로드에서 80% 낮은 컴퓨팅 비용으로 이어집니다. 경쟁사들은 여전히 비효율적인 데이터 파싱에 돈을 낭비하고 있습니다.

인사이트 획득 시간 우위: 서브초 지연 시간의 실시간 스트리밍은 다른 곳에서 배치 실행을 마치기도 전에 패턴, 이상 현상, 기회를 탐지할 수 있게 합니다.

인재 우위: Python 네이티브 개발은 진입 장벽을 낮춥니다. Scala 전문성이 필요 없기 때문에 더 큰 인재 풀에서 채용하고 더 빨리 온보딩할 수 있습니다.

신뢰성 우위: ANSI 모드의 엄격한 오류 처리는 데이터 품질이 입증 가능하게 더 높다는 것을 의미합니다. 규제 산업에서 이것은 단순히 좋은 것이 아니라—필수적입니다.

2026년에 데이터로 승리하는 회사들은 가장 많은 데이터 과학자를 가진 곳이 아닐 것입니다. 팀에게 가장 강력하고 효율적이며 신뢰할 수 있는 데이터 인프라를 제공한 곳일 것입니다.

논란이 되는 주장: 레거시 Spark 스킬이 구식이 되고 있다

일부 엔지니어들을 불편하게 만드는 부분입니다: 오랜 시간 쌓은 많은 Spark 최적화 스킬이 곧 불필요해질 것입니다.

더 나은 쿼리 성능을 위해 JSON 데이터를 수동으로 파티셔닝하는 방법을 몇 년에 걸쳐 배웠나요? VARIANT가 자동으로 합니다.

PySpark UDF 성능 튜닝 전문가가 되었나요? Arrow 네이티브 UDF가 여러분이 최적화해 왔던 오버헤드의 대부분을 없앱니다.

복잡한 스트리밍 상태 관리의 기술을 마스터했나요? 선언적 파이프라인이 이를 추상화합니다.

이것은 여러분의 스킬이 가치 없어진다는 게 아닙니다—더 고수준으로 된다는 것입니다. 클라우드 컴퓨팅이 서버 랙 전문성을 덜 관련성 있게 만들었지만 클라우드 아키텍처 전문성을 더 가치 있게 만든 것처럼.

2026년에 번성할 데이터 엔지니어들은 Spark 구성을 수동으로 튜닝할 수 있는 사람들이 아닐 것입니다. 비즈니스 가치를 위해 VARIANT를 활용하는 방법, 조직 효율성을 위해 Spark Connect 토폴로지를 아키텍처하는 방법, 비즈니스 요구에 맞게 확장되는 선언적 파이프라인을 설계하는 방법을 이해하는 사람들일 것입니다.

지금 당장 알아야 할 것

선점자 우위의 창이 빠르게 닫히고 있습니다. Databricks Runtime 18.0은 베타 버전이며, 이는 프로덕션 출시가 임박했다는 것을 의미합니다.

핵심 요점: Apache Spark 4.0/4.1은 VARIANT 데이터 타입, ANSI SQL 컴플라이언스, Spark Connect를 통해 Spark 3.0 이후 가장 큰 아키텍처 변화를 나타내며, 데이터 처리 방식을 근본적으로 바꿉니다. 일찍 마이그레이션하는 조직은 반정형 워크로드에서 40% 이상의 성능 향상과 80% 비용 절감을 달성하고 있습니다—시간이 지날수록 복합되는 경쟁 우위입니다. 코드 중심에서 선언적, 플랜 기반 아키텍처(Spark Connect, SDP)로의 전환은 업계 전반에 걸쳐 팀 구조와 스킬 요구 사항을 재편할 것입니다.
