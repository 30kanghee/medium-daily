Stop Designing REST APIs Like a Mid-Level Dev: 4 Advanced Patterns Senior Engineers Use Instead

Habib Wahid · in Stackademic · 8 min read
원문 · freedium

요약

시니어 엔지니어가 REST API 설계 시 반드시 적용하는 4가지 고급 패턴 — 멱등성 키, PATCH/PUT 구분, ETag 동시성 제어, 스파스 필드셋 — 을 Java 코드 예제와 함께 설명한다.

단순히 동작하는 API가 아니라 분산 환경의 현실을 견디는 API를 설계하기 위해 시니어 엔지니어들이 사용하는 4가지 핵심 패턴을 다룬다. 첫째, 쓰기 작업에 클라이언트 생성 멱등성 키를 요구해 네트워크 재시도로 인한 중복 처리를 원천 차단한다. 둘째, 부분 업데이트에 PUT 대신 PATCH와 JsonMergePatch를 사용해 의도치 않은 필드 손실을 방지한다. 셋째, ETag와 If-Match 헤더로 낙관적 동시성 제어를 구현해 두 클라이언트가 동시에 같은 리소스를 수정할 때 발생하는 조용한 데이터 손실을 막는다. 넷째, fields 쿼리 파라미터로 스파스 필드셋을 지원해 모바일 클라이언트의 페이로드를 80% 이상 줄인다.

중급 개발자처럼 REST API를 설계하지 마라: 시니어 엔지니어가 사용하는 4가지 고급 패턴

백엔드 커리어를 시작한 지 8개월이 됐을 때, 나는 REST를 다 알고 있다고 생각했다. DTO, 적절한 상태 코드, 페이지네이션, 버저닝 — 관련 글들을 읽고 패턴을 적용했으며, 내 API는 모든 코드 리뷰를 통과했다.

그러던 어느 날 수석 엔지니어가 내 결제 서비스 API를 리뷰했다. 그녀는 무엇이 잘못됐는지 지적하지 않았다. 대신 질문을 했다: "클라이언트가 실패한 결제 요청을 재시도하면 어떻게 되나요? 두 클라이언트가 동시에 같은 레코드를 업데이트하면 어떻게 되죠? 엔드포인트가 없어질 때 API가 클라이언트에게 뭐라고 알려주나요?"

나는 답이 없었다. API는 독립적으로는 완벽하게 동작했다. 하지만 분산된 클라이언트, 동시 요청, 변화하는 계약이라는 지저분한 현실을 다루도록 설계된 적이 없었다.

그 대화가 나에게 가르쳐 준 것은 "만들어진 API"와 "설계된 API"의 차이였다. 오늘은 기본기를 다진 후 수석 엔지니어들이 적용하는 4가지 고급 패턴을 공유한다.

"충분히 괜찮은" API의 문제점

기본기를 넘어선 대부분의 개발자들은 이런 API를 작성한다:

@RestController
@RequestMapping("/api/v1/orders")
public class OrderController {

  @PostMapping
  public ResponseEntity<OrderResponse> createOrder(
          @Valid @RequestBody CreateOrderRequest request) {
      Order order = orderService.create(request);
      return ResponseEntity.status(HttpStatus.CREATED).body(OrderResponse.from(order));
  }

  @PutMapping("/{id}")
  public ResponseEntity<OrderResponse> updateOrder(
          @PathVariable Long id,
          @Valid @RequestBody UpdateOrderRequest request) {
      Order order = orderService.update(id, request);
      return ResponseEntity.ok(OrderResponse.from(order));
  }
}

이것은 주니어의 API보다 낫다. 하지만 여전히 심각한 문제들이 있다:

  • 멱등성(idempotency) 보호 없음 — 재시도된 POST 요청이 중복 주문을 생성한다
  • PUT이 전체를 교체함 — 클라이언트가 의도치 않게 변경하지 않으려 했던 필드를 지워버린다
  • 동시성 제어 없음 — 두 개의 동시 업데이트가 조용히 서로를 덮어쓴다
  • 필드 선택 없음 — 클라이언트가 두 개 필드만 필요해도 모든 필드를 다운로드한다

이제 수석 엔지니어들이 처음부터 적용하는 패턴으로 각각을 고쳐보자.

실수 1: 쓰기 작업에 멱등성이 없다

문제: 모바일 클라이언트가 결제를 제출한다. 응답이 도착하기 전에 네트워크가 끊긴다. 클라이언트가 재시도한다. 서버는 결제를 두 번 처리한다. 사용자는 두 번 청구된다. 서버 측에서는 이것을 감지할 방법이 없다.

// ❌ 멱등성 없음 — 재시도할 때마다 새 주문이 생성됨
@PostMapping("/orders")
public ResponseEntity<OrderResponse> createOrder(
        @Valid @RequestBody CreateOrderRequest request) {
    Order order = orderService.create(request);
    return ResponseEntity.status(HttpStatus.CREATED).body(OrderResponse.from(order));
    // 네트워크 타임아웃으로 클라이언트가 재시도하면?
    // 주문이 두 개 생성됨. 결제도 두 번 처리됨. 알 방법이 없음.
}

시니어 엔지니어가 하는 것: 모든 쓰기 작업에 클라이언트가 생성하는 멱등성 키(idempotency key)를 요구한다. 첫 번째 실행 결과를 저장하고, 중복 요청에 대해서는 로직을 다시 실행하지 않고 저장된 결과를 반환한다.

// ✅ 멱등성 키로 중복 처리 방지
@PostMapping("/orders")
public ResponseEntity<OrderResponse> createOrder(
        @RequestHeader("Idempotency-Key") String idempotencyKey,
        @Valid @RequestBody CreateOrderRequest request) {

    // 이미 처리됨? 캐시된 결과를 즉시 반환
    return idempotencyService.findResult(idempotencyKey)
        .map(cached -> ResponseEntity.ok(cached))
        .orElseGet(() -> {
            Order order = orderService.create(request);
            idempotencyService.save(idempotencyKey, OrderResponse.from(order));
            return ResponseEntity.status(HttpStatus.CREATED).body(OrderResponse.from(order));
        });
}

왜 중요한가:

  • 모바일 클라이언트와 로드 밸런서는 실패한 요청을 재시도한다 — 멱등성 없이는 재시도가 위험하다
  • 클라이언트가 키를 생성하므로, 서버는 재시도와 새 요청을 구별할 필요가 없다
  • 동일한 키 + 동일한 요청 = 동일한 응답, 재처리 없음
  • Stripe, PayPal, 모든 주요 결제 API는 멱등성 키를 요구한다 — 쓰기 작업의 업계 표준이다

실수 2: PATCH를 써야 할 때 PUT을 사용한다

문제: PUT은 리소스 전체를 교체한다. 클라이언트가 변경하려는 필드만 포함해서 PUT 요청을 보내면, 포함되지 않은 모든 필드는 조용히 null로 설정된다. 부분 업데이트에는 부분 교체가 필요하다 — 그것이 PATCH가 존재하는 이유다.

// ❌ PUT은 모든 것을 교체 — 누락된 필드는 null이 됨
@PutMapping("/users/{id}")
public ResponseEntity<UserResponse> updateUser(
        @PathVariable Long id,
        @RequestBody UpdateUserRequest request) {
    // 클라이언트가 {"email": "new@email.com"}만 보냄
    // username, phone, address → 모두 조용히 null로 설정됨
    return ResponseEntity.ok(UserResponse.from(userService.update(id, request)));
}

시니어 엔지니어가 하는 것: 부분 업데이트에는 JsonMergePatch와 함께 PATCH를 사용한다. 요청에 있는 필드만 수정된다. 누락된 필드는 그대로 유지된다.

// ✅ PATCH는 클라이언트가 보낸 것만 업데이트
@PatchMapping(value = "/users/{id}",
              consumes = "application/merge-patch+json")
public ResponseEntity<UserResponse> patchUser(
        @PathVariable Long id,
        @RequestBody JsonMergePatch patch) {

    // 현재 상태 로드
    User existing = userService.findById(id);
    UserRequest currentRequest = UserRequest.from(existing);

    // 패치에 있는 필드만 적용
    UserRequest patched = applyPatch(patch, currentRequest);
    return ResponseEntity.ok(UserResponse.from(userService.update(id, patched)));
}

클라이언트는 변경된 것만 보낸다:

PATCH /users/42
Content-Type: application/merge-patch+json

{"email": "new@email.com"}
// username, phone, address - 변경 없음. 이메일만 업데이트됨.

왜 중요한가:

  • PUT과 PATCH는 의미가 다르다 — 부분 업데이트에 PUT을 사용하는 것은 잘못된 REST다
  • application/merge-patch+json은 부분 업데이트를 위한 RFC 7396 표준 콘텐츠 타입이다
  • 클라이언트는 최소한의 페이로드를 보낸다 — 하나의 필드를 업데이트하기 전에 전체 리소스를 가져올 필요가 없다
  • 누락된 필드로 인한 우발적 데이터 손실이 구조적으로 불가능해진다

실수 3: 업데이트에 동시성 제어가 없다

문제: 두 클라이언트가 동시에 같은 리소스를 로드한다. 둘 다 수정한다. 첫 번째가 성공적으로 저장한다. 두 번째가 그 위에 저장한다 — 조용히 첫 번째 클라이언트의 변경 사항을 덮어쓴다. 아무 오류도 발생하지 않는다. 데이터는 영구적으로 손실된다.

// ❌ 마지막 쓰기가 이김 — 첫 번째 클라이언트의 변경이 조용히 덮어써짐
@PutMapping("/products/{id}")
public ResponseEntity<ProductResponse> updateProduct(
        @PathVariable Long id,
        @RequestBody UpdateProductRequest request) {
    // 클라이언트 A가 버전 1을 로드, 가격을 $20으로 변경
    // 클라이언트 B가 버전 1을 로드, 재고를 50으로 변경
    // 클라이언트 A 저장 → 버전 2, 가격 $20, 재고 원래 값
    // 클라이언트 B 저장 → 버전 2, 가격 원래 값, 재고 50
    // 클라이언트 A의 가격 변경이 사라짐
}

시니어 엔지니어가 하는 것: 낙관적 동시성을 위해 ETag를 사용한다. 서버는 모든 응답과 함께 ETag를 보낸다. 클라이언트는 업데이트 시 반드시 ETag를 포함해야 한다. 클라이언트가 로드한 이후 리소스가 변경됐다면, 서버는 412로 업데이트를 거부한다.

// ✅ ETag 기반 동시성 — 오래된 업데이트를 명확하게 거부
@GetMapping("/products/{id}")
public ResponseEntity<ProductResponse> getProduct(@PathVariable Long id) {
    Product product = productService.findById(id);
    String eTag = """ + product.getVersion() + """;

    return ResponseEntity.ok()
        .eTag(eTag)  // 응답 헤더에 ETag: "5" 전송
        .body(ProductResponse.from(product));
}

@PutMapping("/products/{id}")
public ResponseEntity<ProductResponse> updateProduct(
        @PathVariable Long id,
        @RequestHeader("If-Match") String ifMatch,
        @RequestBody UpdateProductRequest request) {
    Product product = productService.findById(id);
    String currentETag = """ + product.getVersion() + """;

    if (!currentETag.equals(ifMatch)) {
        return ResponseEntity.status(412).build(); // 리소스가 변경됨 — 재로드 필요
    }
    return ResponseEntity.ok(ProductResponse.from(productService.update(id, request)));
}

왜 중요한가:

  • 동시성 제어가 없는 모든 API에서 업데이트 손실은 조용히 발생한다 — ETag는 이것을 가시화한다
  • 412 Precondition Failed는 클라이언트에게 재로드 후 충돌을 해결하라는 명확한 신호를 준다
  • product.getVersion()은 JPA의 @Version 필드와 직접 매핑된다 — 추가 인프라가 필요 없다
  • GitHub, Google Drive, 모든 협업 API가 ETag를 사용한다 — 클라이언트는 이미 처리 방법을 알고 있다

실수 4: 필드 선택 없이 항상 모든 것을 보낸다

문제: 사용자 목록을 로드하는 모바일 클라이언트는 id와 name만 필요하다. API는 주소, 설정, 메타데이터를 포함한 15개 필드를 반환한다. 모든 필드가 직렬화되고, 전송되고, 파싱된다 — 즉시 버려지는 것들까지 포함해서.

// ❌ 클라이언트가 필요한 것과 상관없이 모든 필드 반환
@GetMapping("/users")
public ResponseEntity<Page<UserResponse>> getUsers(Pageable pageable) {
    return ResponseEntity.ok(
        userService.findAll(pageable).map(UserResponse::from));
    // 모바일 클라이언트는 2개 필드가 필요. 15개를 받음. 매 요청마다 13개 필드가 낭비됨.
}

시니어 엔지니어가 하는 것: fields 쿼리 파라미터를 받는다. 요청된 필드만 반환한다. 클라이언트는 가벼운 페이로드를 얻고, 대역폭과 파싱 시간이 크게 줄어든다.

// ✅ 스파스 필드셋 — 클라이언트가 필요한 것만 요청
@GetMapping("/users")
public ResponseEntity<Page<Map<String, Object>>> getUsers(
        Pageable pageable,
        @RequestParam(required = false) Set<String> fields) {

    Page<User> users = userService.findAll(pageable);

    Page<Map<String, Object>> response = users.map(user -> {
        Map<String, Object> full = Map.of(
            "id",    user.getId(),
            "name",  user.getName()
            // ... 나머지 필드들
        );
        return (fields != null) ? filterFields(full, fields) : full;
    });
    return ResponseEntity.ok(response);
}

클라이언트는 필요한 것만 요청한다:

GET /users?fields=id,name
[{"id": 1, "name": "Alice"}, {"id": 2, "name": "Bob"}]
// 2개 필드. 15개가 아니라.

왜 중요한가:

  • 느린 연결의 모바일 클라이언트는 렌더링하는 것만 다운로드한다 — 응답 크기가 80% 이상 줄어들 수 있다
  • 하나의 엔드포인트가 서로 다른 필드 요구를 가진 여러 클라이언트를 서빙한다 — 엔드포인트 증가 없이
  • 이 패턴은 GraphQL의 기반이다 — REST는 하나의 쿼리 파라미터로 구현할 수 있다
  • fields가 선택 사항이므로, 보내지 않는 기존 클라이언트는 여전히 전체 응답을 받는다

좋은 API에서 계약 수준의 API로

좋은 API와 계약 수준의 API의 차이는 동작하느냐의 문제가 아니다 — 실제 분산 시스템이 만들어내는 조건들, 즉 재시도, 동시 편집, 부분 업데이트, 대역폭 제약, 변화하는 계약을 처리하느냐의 문제다.

이전 (문제 있는 코드):

@PostMapping("/orders")
public ResponseEntity<OrderResponse> createOrder(
        @Valid @RequestBody CreateOrderRequest request) {
    return ResponseEntity.status(201).body(OrderResponse.from(orderService.create(request)));
    // 재시도 = 중복 주문. 동시 편집 = 업데이트 손실.
    // 모바일 페이로드 = 15개 필드. V1 제거 = 예상치 못한 장애.
}

이후 (개선된 코드):

@PostMapping("/orders")
public ResponseEntity<OrderResponse> createOrder(
        @RequestHeader("Idempotency-Key") String key,  // 재시도 안전
        @Valid @RequestBody CreateOrderRequest request) {

    return idempotencyService.findResult(key)
        .map(ResponseEntity::ok)
        .orElseGet(() -> {
            OrderResponse response = OrderResponse.from(orderService.create(request));
            idempotencyService.save(key, response);    // 재시도를 위해 캐싱
            return ResponseEntity.status(201).body(response);
        });
}

실행 계획

한꺼번에 다 적용하려 하지 마라. 가장 빠른 효과를 내는 순서는 다음과 같다:

1주차: 주문을 생성하거나 결제를 처리하는 모든 POST 엔드포인트에 Idempotency-Key를 추가하라 — 헤더 하나가 중복 처리 버그 전체를 방지한다.

2주차: 모든 PUT 엔드포인트를 감사하라 — 부분 업데이트를 PATCH와 JsonMergePatch로 교체하라.

3주차: 가장 많이 업데이트되는 엔티티에 @Version을 추가하고, GET과 PUT 엔드포인트에 ETag 헤더를 연결하라.

4주차: 트래픽이 가장 많은 목록 엔드포인트에 ?fields= 지원을 추가하고 — 페이로드 크기 감소를 측정하라.

다음에 @PostMapping을 작성할 때, 이렇게 물어라: "이 요청이 두 번 도착하면 어떻게 되나?" 그 질문 하나가 모든 레벨에서 API를 설계하는 방식을 바꿀 것이다.

이 패턴들은 괜히 고급인 것이 아니다. 개발 환경에서 동작하는 API와, 분산된 클라이언트·불안정한 네트워크·이에 의존하는 팀들에게 영향을 주지 않고 발전해야 하는 시스템의 현실을 견뎌내는 API의 차이다.

지금 당신의 가장 중요한 엔드포인트에 이 패턴 중 어떤 것이 빠져 있는가?
