8 Data Pipeline Patterns Exist. Most Engineers Only Know 3

Pravash · Towards Data Engineering · 10분 읽기 · 원문 · freedium

요약

데이터 파이프라인에는 8가지 패턴이 있고, 문제에 맞는 패턴을 고르는 것이 성패를 좌우한다.

대부분의 파이프라인이 실패하는 이유는 코드 품질이 아니라 잘못된 패턴 선택 때문이다. ETL·ELT·스트리밍·람다·카파·마이크로배치·팬아웃·이벤트 드리븐, 이 8가지 패턴은 실무에서 마주치는 사실상 모든 데이터 엔지니어링 시나리오를 커버한다. 각 패턴이 어떤 문제를 풀기 위해 존재하는지, 언제 쓰고 언제 피해야 하는지 실제 코드 예시와 함께 정리한다. 패턴을 제대로 이해하면 처음부터 다시 짜는 일을 없앨 수 있다.

매주 어딘가에서 데이터 팀 하나가 파이프라인을 통째로 다시 짜고 있다. 코드가 버그투성이여서가 아니고, 인프라가 잘못돼서도 아니다. 비즈니스가 실제로 필요로 하는 것과 맞지 않는 패턴을 처음부터 골랐기 때문이다.

한 스타트업이 매일 매출 리포트를 처리하는 깔끔한 ETL 파이프라인을 구축했다. 그런데 프로덕트팀에서 실시간 사기 탐지 기능을 요청하는 순간, 그 파이프라인은 무용지물이 된다. 핀테크 기업은 모든 것에 스트리밍 파이프라인을 적용했지만, 그러고 보니 월 단위로 돌아가는 청구 작업은 단순한 배치 잡으로도 충분했다. 엔지니어 여섯 달치 노력이 한순간에 날아간다.

핵심은 이것이다. 데이터 파이프라인 설계는 가장 "좋은" 기술을 고르는 문제가 아니라, 문제에 맞는 패턴을 찾는 문제다. 실무에서 만나는 데이터 엔지니어링 시나리오를 사실상 전부 커버하는 패턴은 정확히 8가지다. 이 8가지를 제대로 익히면 파이프라인을 처음부터 다시 짜는 일은 없어진다.

1. ETL — Extract, Transform, Load (검증된 고전)

이 패턴이 해결하는 문제: 여러 소스에서 오는 지저분하고 일관성 없는 데이터를 데이터베이스에 적재하기 전에 반드시 정제해야 할 때.

이커머스 주문 데이터가 세 개 시스템에서 세 가지 형식으로 들어온다고 하자. 통화 단위도 제각각이고 데이터 품질도 들쭉날쭉한데, 재무팀은 내일 아침까지 한곳에 모아달라고 한다. ETL이 이 문제를 해결한다. 데이터를 꺼내 중간에서 정제한 다음, 유효한 것만 적재한다. 불량 레코드는 Dead Letter Queue(DLQ)에 보관했다가 근본 원인을 고친 뒤 재처리한다. 원칙은 단순하다. 오염된 데이터는 웨어하우스에 절대 들어가지 않는다.

실무 적용처: 레거시 엔터프라이즈 데이터 웨어하우스, 규제 준수 리포팅 파이프라인, 저장소에 불량 데이터가 들어가면 안 되는 금융·의료 시스템.

# etl_pipeline.py
# 핵심: 적재 전 변환 — 유효하지 않은 레코드는 DLQ로

def transform(record):
    if not record.get("order_id") or record["amount"] <= 0:
        dlq.append(record)          # 불량 레코드 → DLQ
        return None
    record["amount_usd"] = record["amount"] * fx_rate(record["currency"])
    record["loaded_at"] = datetime.utcnow().isoformat()
    return record

cleaned  = [r for r in (transform(row) for row in extract()) if r]
warehouse.load(cleaned)            # 정제된 레코드만 저장소에 적재
dlq_table.load(dlq)                # 불량 레코드는 재처리 대기

ETL은 웨어하우스를 깨끗하게 유지해 주지만, 요구사항이 바뀌었을 때 원본 데이터를 잃을 수 없다면 어떻게 해야 할까? 그게 바로 ELT가 등장하는 지점이다.

2. ELT — Extract, Load, Transform (현대적 전환)

이 패턴이 해결하는 문제: 방대한 원본 데이터를 보유하고 있지만, 나중에 어떤 질문을 던질지 지금은 알 수 없을 때.

한 미디어 기업이 하루 5,000만 건의 클릭 데이터를 수집한다. 어떤 이벤트가 ML에 필요하고 어떤 것이 BI에 필요한지 아직 모르는 상황에서는 사전 변환을 하다가 중요한 데이터를 버리게 될 수 있다. ELT는 순서를 뒤집어 이 문제를 해결한다. 모든 데이터를 raw 레이어에 그대로 올린 뒤, 웨어하우스 내부에서 나중에 변환한다. BigQuery는 100GB를 수 초 만에 처리하므로, 한 번 저장해두고 필요에 따라 원하는 형태로 가공하면 된다. raw 레이어 하나에서 목적별 산출물을 여럿 뽑아낼 수 있다.

실무 적용처: 현대적 클라우드 데이터 스택, dbt 기반 파이프라인, 애널리틱스 엔지니어링 워크플로, 유연하고 다목적인 데이터가 필요한 모든 상황.

# elt_pipeline.py
# 핵심: raw를 그대로 적재한 뒤 BigQuery 내부에서 dbt/SQL로 변환

# 1단계 — 원본 데이터를 손대지 않고 그대로 적재
df_raw = extract_from_api()
bq_client.load_table_from_dataframe(df_raw, "project.raw.events")

# 2단계 — 웨어하우스 내부에서 변환 (dbt가 이 SQL을 실행)
curated_sql = """
    SELECT
        user_id,
        COUNT(*)        AS total_events,
        SUM(revenue)    AS total_revenue
    FROM raw.events
    WHERE _loaded_at >= CURRENT_DATE - 1
    GROUP BY user_id
"""
# 소비자(BI, ML, API)마다 각자의 curated 뷰를 가짐
bq_client.query(curated_sql).to_dataframe()

3. 스트리밍 파이프라인 (실시간 데이터, 대기 없음)

이 패턴이 해결하는 문제: 데이터가 도착하는 순간 바로 처리해야 하고, 밀리초 단위의 지연도 허용할 수 없을 때.

파리에서 카드가 사용됐다. 3초 뒤 같은 카드가 서울에서 사용된다. 1시간 주기로 돌아가는 배치 파이프라인은 이 이상 거래를 6시간 뒤에나 잡아낸다. 스트리밍 파이프라인은 두 번째 거래가 승인되기 전에 포착한다.

실무 적용처: 사기 탐지, 실시간 추천 엔진, 애드테크 실시간 입찰, IoT 센서 모니터링, 주식 거래 시스템, 차량 공유 서비스의 요금 급등 제어.

# streaming_pipeline.py
# 핵심: Kafka 소비 → 각 이벤트 처리 → 싱크로 라우팅

consumer = KafkaConsumer("transactions", bootstrap_servers=KAFKA_BROKER)

for msg in consumer:
    event = json.loads(msg.value)
    try:
        if is_fraudulent(event):          # 처리 중에 바로 판별
            alert_sink.send(event)        # → PagerDuty / Slack
        else:
            db_sink.insert(event)         # → 정상 저장
    except Exception as e:
        dlq_sink.send(event, error=str(e))  # → 재처리 큐

4. 람다 아키텍처 (두 세계의 장점, 그에 따른 비용)

이 패턴이 해결하는 문제: 정확성과 속도 모두 필요하고, 어느 하나도 양보할 수 없을 때.

한 물류 기업이 동시에 두 가지가 필요하다. 몇 초마다 업데이트되는 실시간 지도와, GPS 지연 데이터를 보정해서 내놓는 정확한 배송 리포트. 하나는 속도가, 다른 하나는 정확성이 생명이다. 람다 아키텍처는 이 둘을 각각의 레이어로 분리해서 해결한다. 배치 레이어가 정확성을, 스피드 레이어가 신선함을 담당하고, 서빙 레이어가 쿼리 시점에 둘을 합친다.

실무 적용처: 실시간 대시보드를 갖춘 금융 리포팅, 물류·공급망 관리, 소셜 미디어 분석, "현재 최선 추정치"와 "정확한 역사적 기록" 모두 필요한 경우.

# lambda_architecture.py
# 핵심: 배치 레이어(정확성) + 스피드 레이어(신선함)
# 서빙 레이어가 쿼리 시 두 레이어를 병합

# 배치 레이어 — Spark가 매일 밤 전체 이력 재계산
batch_view = (spark.read.parquet("s3://events/history/")
    .groupBy("user_id")
    .agg(F.sum("revenue").alias("total_revenue"))  # 정확한 값
)

# 스피드 레이어 — Kafka 컨슈머가 최근 N분만 업데이트
speed_view = redis_client.hgetall("recent_revenue")  # 빠른 근사치

# 서빙 레이어 — 최신 데이터를 배치 뷰에 덮어써서 응답
def query_total_revenue(user_id):
    base = batch_view.filter(f"user_id='{user_id}'")
    delta = speed_view.get(user_id, 0)
    return base["total_revenue"] + delta

5. 카파 아키텍처 (스트림 하나로 모든 것을)

이 패턴이 해결하는 문제: 람다 아키텍처의 강력함은 원하지만, 두 시스템을 동시에 유지하는 복잡성은 피하고 싶을 때.

Kafka의 공동 창시자 Jay Kreps는 단순한 질문을 던졌다. 배치 레이어가 아예 필요 없다면 어떨까? 카파 아키텍처에서는 과거 이력 재처리를 포함한 모든 것이 스트림이다. 처리 로직이 바뀌면 새 버전의 스트림 잡을 띄워서 Kafka 오프셋 처음부터 재생하고, 준비가 되면 트래픽을 넘긴다. 배치 레이어가 없으니 유지보수할 코드베이스도 하나다.

실무 적용처: 통합된 코드베이스를 원하는 팀, 처리 로직이 자주 바뀌는 시스템, Kafka 생태계를 깊이 활용 중인 조직.

# kappa_architecture.py
# 핵심: t=0부터 재생해서 새 로직으로 이력 재처리
# 별도 배치 레이어 없음 — 스트림 잡 하나가 모든 것 처리

def run_stream_job(version, start_offset="latest"):
    consumer = KafkaConsumer(
        "events",
        auto_offset_reset=start_offset,   # "earliest" = 전체 이력 재생
        group_id=f"job-v{version}"        # 새 그룹 = 독립적인 재생
    )
    for msg in consumer:
        event = json.loads(msg.value)
        result = process_v2(event)        # 새 버전 로직 적용
        output_topic.send(result)

# 로직 변경 시: 새 버전을 처음부터 재생한 뒤 트래픽 전환
run_stream_job(version=2, start_offset="earliest")

6. 마이크로배치 파이프라인 (Spark의 숨겨진 기본값)

이 패턴이 해결하는 문제: 거의 실시간에 가까운 수십 초 주기의 신선함이 필요하지만, 배치 처리의 단순함도 유지하고 싶을 때.

한 리테일 체인이 재고 대시보드를 30초마다 갱신하고 싶다. 본격적인 스트리밍은 과도하고, 1시간 배치는 너무 느리다. 마이크로배치가 딱 맞는 중간 지점이다. 실제로 Spark Structured Streaming의 기본 동작 방식이 바로 이것이다. 데이터를 작은 덩어리로 잘라 짧은 간격으로 처리한다.

실무 적용처: "초" 단위 신선함이 필요한 대시보드·리포팅, 로그 집계, 전체 스트리밍 복잡성 없이 exactly-once 의미론이 필요한 ETL 파이프라인.

# microbatch_pipeline.py
# 핵심: Spark Structured Streaming — N초마다 트리거
# 체크포인트로 재시작 시에도 exactly-once 보장

query = (
    spark.readStream
        .format("kafka")
        .option("subscribe", "orders")
        .load()
    .withColumn("value", F.from_json("value", schema))
    .groupBy(F.window("event_time", "1 minute"))
    .agg(F.sum("amount").alias("total"))
    .writeStream
        .trigger(processingTime="30 seconds")   # 30초마다 마이크로배치
        .option("checkpointLocation", CHECKPOINT)
        .outputMode("update")
        .start()
)
query.awaitTermination()

7. 팬아웃 파이프라인 (소스 하나, 목적지 여럿)

이 패턴이 해결하는 문제: 동일한 데이터를 여러 시스템이 동시에, 서로 독립적으로 소비해야 할 때.

이커머스 플랫폼에 주문 하나가 들어왔다. 재고·풀필먼트·분석·추천·검색·이메일, 6개 시스템이 즉시 알아야 한다. 각 시스템마다 별도 파이프라인을 이어 붙이면 소스에 과부하가 걸리고 하나가 실패하면 연쇄 장애가 난다. 팬아웃은 이벤트를 SNS 토픽으로 발행하고, 각 소비자가 독립적으로 구독하게 함으로써 이 결합을 끊는다.

실무 적용처: 이커머스 주문 처리, 사용자 활동 이벤트, 알림 시스템, 두 개 이상의 다운스트림 시스템이 동일 이벤트를 소비해야 하는 모든 상황.

8. 이벤트 드리븐 파이프라인 (스케줄러 없이, 기다림 없이)

이 패턴이 해결하는 문제: 각 단계가 다음 단계를 자동으로 트리거하는 파이프라인이 필요한데, 중앙 오케스트레이터도 cron 잡도 두고 싶지 않을 때.

대부분의 파이프라인은 시계에 따라 돌아간다. 이벤트 드리븐 파이프라인은 결과에 따라 돌아간다. 계약서 PDF가 S3에 올라오면 핸들러 A가 발동해 텍스트를 추출하고, 그 결과가 핸들러 B를 트리거해 개체명을 인식하고, 완료되면 핸들러 C가 CRM을 업데이트한다. 중앙 스케줄러는 없다. 각 단계는 이전 단계의 성공에 반응해서 실행된다.

실무 적용처: 서버리스 아키텍처(AWS Lambda, Google Cloud Functions), 문서·미디어 처리 파이프라인, 마이크로서비스 코레오그래피, 단계가 시간이 아닌 이전 결과에 의존하는 모든 워크플로.

# event_driven_pipeline.py
# 핵심: 각 핸들러가 다음을 트리거 — 스케줄러 불필요
# 이벤트 내용에 따른 조건부 라우팅

def handler_a(event):               # S3 업로드로 트리거
    text = extract_text(event["file"])
    db.save(event["id"], text)
    event_bus.emit("text_extracted", {**event, "text": text})  # B 발동

def handler_b(event):               # "text_extracted" 이벤트로 트리거
    entities = ner_model.extract(event["text"])
    db.save(event["id"], entities)
    if entities.get("contract_value"):
        event_bus.emit("high_value_contract", event)           # C 발동
    else:
        event_bus.emit("standard_contract", event)

def handler_c(event):               # 이벤트 유형에 따라 라우팅
    crm.update(event["id"], priority="high")

올바른 패턴 선택: 빠른 의사결정 가이드

현실에서 성숙한 데이터 플랫폼은 이 패턴들을 하나만 쓰지 않는다. 과거 분석에는 ELT, 사기 탐지에는 스트리밍, 대시보드에는 마이크로배치, 다운스트림 팀에 이벤트를 전파할 때는 팬아웃을 조합해서 쓰는 식이다.

처음 파이프라인을 설계할 때 하나를 골라야 한다면, 다음 질문을 순서대로 해보자.

데이터 신선도가 얼마나 필요한가? 시간 단위면 ETL 또는 ELT. 초 단위면 마이크로배치. 밀리초 단위면 스트리밍.

처리 로직이 자주 바뀌는가? 그렇다면 카파. 아니라면 람다 또는 스트리밍.

동일한 이벤트를 여러 시스템이 소비해야 하는가? 그렇다면 팬아웃.

파이프라인 단계가 시간이 아닌 결과에 반응해야 하는가? 그렇다면 이벤트 드리븐.

이력 정확성과 실시간 신선함이 모두 필요한가? 그렇다면 람다.

오래 살아남는 시스템을 만드는 엔지니어는 가장 많은 도구를 아는 사람이 아니다. 어떤 패턴이 어떤 문제를 해결하는지 깊이 이해하고, 더 화려한 대안이 있어도 맞는 것을 고를 수 있는 판단력을 가진 사람이다.
