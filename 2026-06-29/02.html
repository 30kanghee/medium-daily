LLM vs RAG vs MCP: I Finally Know When to Use Each One

Pravash · Level Up Coding · 7분 읽기 · 원문 · freedium

요약

LLM·RAG·MCP는 각각 다른 문제를 풀기 위한 도구이며, 셋을 언제 쓰는지 알면 AI 시스템 설계 실수를 줄일 수 있다.

저자는 내부 파이프라인 문서를 AI로 조회하는 도구를 만들다가 LLM만으로는 부족해 RAG를 도입했고, 실시간 DB 쿼리가 필요해지자 MCP를 발견했다. 이 직접 경험을 바탕으로 세 기술의 역할을 명확히 정리한다. LLM은 학습된 지식으로 추론하는 두뇌, RAG는 고정 문서 코퍼스에서 근거를 찾아주는 사서, MCP는 살아있는 시스템에 실제로 접근해 행동할 수 있는 도구 세트다. 어느 것이 더 낫다는 문제가 아니라 AI가 실제로 무엇을 해야 하는지에 따라 고르는 문제다.

모든 것이 맞아떨어진 순간

몇 달 전, 나는 팀이 데이터 파이프라인 문서를 자연어로 조회할 수 있는 내부 도구를 만들고 있었다. 스키마, 계보 그래프, 200개 마크다운 파일에 흩어진 변환 로직이 대상이었다. 처음 든 생각은 전부 시스템 프롬프트에 집어넣자는 것이었다. 당연히 금방 컨텍스트 한도에 부딪혔다.

그래서 RAG로 방향을 틀었다. 문서를 임베딩하고, 리트리버를 만들고, 벡터 스토어를 연결했다. 꽤 잘 됐다. 대부분은. 그런데 요구사항이 바뀌었다. 문서 검색을 넘어, AI가 실제로 운영 중인 Postgres 데이터베이스에 쿼리를 실행하고 Delta Lake 테이블 통계를 실시간으로 확인해야 했다. 갑자기 RAG는 완전히 잘못된 도구처럼 느껴졌다.

그때 MCP를 발견했고, AI 시스템 설계에 대한 사고방식이 바뀌었다.

핵심 질문은 "어떤 AI가 더 똑똑한가"가 아니다. "이 문제를 풀기 위해 AI가 실제로 무엇을 해야 하는가"다.

LLM: 두뇌

대형 언어 모델(LLM): 학습 시점에 고정된 내부 지식.

기본 LLM을 세상에 존재하는 모든 책을 읽은 뒤 인터넷도, 파일도, 이전 대화 기억도 없는 방에 갇힌 탁월한 동료라고 생각하면 된다.

LLM의 지식은 전적으로 학습에 내재되어 있다. 정적인 범용 지식으로 충분한 작업, 즉 논리 문제를 추론하거나, 잘 알려진 언어로 코드를 짜거나, 개념을 요약하는 일에 강하다.

잘 맞는 경우: "Spark에서 broadcast join이 뭔지 설명해줘." LLM은 이걸 이미 안다. 검색도, 외부 호출도 필요 없다. 순수한 추론이다.

한계:

  • 시간에 민감한 정보 — 학습 컷오프가 있어 지난주 무슨 일이 있었는지 모른다
  • 회사 내부나 비공개 데이터 — 내부 위키를 본 적이 없다
  • 대규모 코퍼스에서의 정밀 검색 — 검색이 아닌 요약에 치우친다
from anthropic import Anthropic

client = Anthropic()

# 단순 LLM 호출 — 검색도 도구도 없음
# 적합: 추론, 작문, 잘 알려진 개념
response = client.messages.create(
    model="claude-opus-4-5",
    max_tokens=1024,
    messages=[
        {
            "role": "user",
            "content": "Explain what a broadcast join is in Spark"
        }
    ]
)

print(response.content[0].text)

# 사용 시기:
#   + 범용 지식 질문
#   + 추론 및 합성
#   + 알려진 라이브러리 코드 생성
# 사용하지 말아야 할 때:
#   - 내부/비공개 데이터
#   - 실시간 쿼리

RAG: 사서

검색 증강 생성(RAG): 관련 컨텍스트를 먼저 가져온 뒤 생성.

RAG는 LLM에게 도서관 이용권을 준다. 사용자가 질문하면 리트리버가 먼저 벡터 데이터베이스나 문서 저장소에서 관련 청크를 찾아, 그 내용을 질문과 함께 LLM의 컨텍스트 창에 넣는다.

흐름은 단순하다. 사용자 쿼리 → 리트리버가 지식 베이스(PDF, 코드, 벡터 DB)에서 검색 → 문서를 쿼리와 함께 LLM에 전달 → 응답 생성. 정적으로 구워 넣는 게 아니라 런타임에 지식을 검색하는 방식이다.

나는 답이 고정된 코퍼스에 있을 때 RAG를 많이 쓴다. 문서, 정책서, 과거 기록, 내부 위키처럼 몇 분마다 바뀌지 않는 데이터가 대상이다.

잘 맞는 경우: "3년치 장애 사후분석 리포트를 검색해서 파이프라인 장애의 가장 흔한 근본 원인이 뭔지 알려줘." 고정 코퍼스에 시맨틱 검색이 필요한 전형적인 RAG 영역이다.

한계:

  • 실시간 데이터 — RAG는 스냅샷이지 라이브 연결이 아니다
  • 행동 수행 — 검색하고 요약할 뿐, 무언가를 실행하지는 못한다
  • 빠르게 바뀌는 데이터 — 데이터가 시간 단위로 갱신되면 임베딩이 금방 낡아진다
from anthropic import Anthropic
from sentence_transformers import SentenceTransformer
import chromadb

client   = Anthropic()
embedder = SentenceTransformer("all-MiniLM-L6-v2")
db       = chromadb.Client()
coll     = db.get_or_create_collection("pipeline_docs")

# --- 1단계: 코퍼스 인덱싱 (최초 1회 실행) ---
docs = [
    "Delta Lake supports ACID transactions via transaction logs.",
    "Spark broadcast joins send small tables to every executor.",
    "Kafka retention is configured with retention.ms per topic.",
]
coll.add(
    documents=docs,
    embeddings=embedder.encode(docs).tolist(),
    ids=[f"doc-{i}" for i in range(len(docs))]
)

# --- 2단계: 쿼리 시점에 검색 + 생성 ---
query   = "How does Spark handle joins with small tables?"
results = coll.query(
    query_embeddings=embedder.encode([query]).tolist(),
    n_results=2
)
context = "\n".join(results["documents"][0])

response = client.messages.create(
    model="claude-opus-4-5",
    max_tokens=512,
    system=f"Answer using only this context:\n{context}",
    messages=[{"role": "user", "content": query}]
)

print(response.content[0].text)

# 사용 시기:
#   + 내부 문서, 위키, 운영 가이드
#   + 출처 근거로 환각 감소
#   + 컨텍스트 창에 들어가지 않는 대규모 정적 코퍼스
# 사용하지 말아야 할 때:
#   - 실시간/라이브 데이터 (임베딩이 낡아짐)
#   - 외부 시스템에서 행동 수행

MCP: 도구 세트

Model Context Protocol(MCP): LLM을 위한 표준화된 라이브 도구 접근.

MCP는 LLM 호스트(Claude Desktop이나 IDE 등)가 MCP 서버에 연결할 수 있게 해주는 표준화된 프로토콜이다. 각 서버는 모델이 실시간으로 호출할 수 있는 도구 모음을 제공한다. 라이브 데이터베이스 쿼리, GitHub API 호출, 디스크 파일 읽기·쓰기 같은 것들이다.

MCP 호스트(Claude Desktop, IDE, AI 도구)는 MCP 프로토콜을 통해 서버들과 통신한다. 서버 A는 GitHub·Slack 같은 웹 API를 호출하고, 서버 B는 데이터베이스 쿼리를 실행하고, 서버 C는 파일 시스템을 읽고 쓴다.

사전 인덱싱된 저장소에서 검색하는 게 아니다. 실제 실행이다. 모델이 어떤 도구를 쓸지 결정하고, 호출하고, 실시간 결과를 받아서 응답에 반영한다.

잘 맞는 경우: "orders_delta 테이블의 현재 행 수를 확인해서 어제 스냅샷과 비교하고 이상 징후를 찾아줘." MCP는 쿼리를 직접 실행한다. RAG는 이걸 할 수 없다.

한계:

  • 설정 복잡도 — MCP 서버를 구축하고 운영해야 한다
  • 지연 — 도구 호출마다 왕복 시간이 추가된다
  • 신뢰·보안 표면 — 모델에게 실제 시스템 접근 권한을 주는 것이라 가드레일이 필요하다
import anthropic, psycopg2, json

client = anthropic.Anthropic()

# --- LLM이 호출할 수 있는 도구 정의 ---
tools = [
    {
        "name": "query_database",
        "description": "Run a read-only SQL query on the data warehouse",
        "input_schema": {
            "type": "object",
            "properties": {
                "sql": {"type": "string", "description": "SQL to execute"}
            },
            "required": ["sql"]
        }
    }
]

# --- 도구 실행 (모델이 사용하기로 결정하면 호출) ---
def run_tool(name, inputs):
    if name == "query_database":
        conn = psycopg2.connect("postgresql://localhost/warehouse")
        cur  = conn.cursor()
        cur.execute(inputs["sql"])
        rows = cur.fetchall()
        conn.close()
        return json.dumps({"rows": rows})

# --- 에이전틱 루프: 모델이 답을 얻을 때까지 도구 반복 호출 ---
messages = [{
    "role": "user",
    "content": "How many rows landed in orders_delta today vs yesterday?"
}]

while True:
    resp = client.messages.create(
        model="claude-opus-4-5",
        max_tokens=1024,
        tools=tools,
        messages=messages
    )

    if resp.stop_reason == "end_turn":
        print(resp.content[0].text)
        break

    # 모델이 도구 요청 → 실행 후 결과 반환
    tool_use = next(b for b in resp.content if b.type == "tool_use")
    result   = run_tool(tool_use.name, tool_use.input)

    messages += [
        {"role": "assistant", "content": resp.content},
        {"role": "user", "content": [{
            "type": "tool_result",
            "tool_use_id": tool_use.id,
            "content": result
        }]}
    ]

# 사용 시기:
#   + 라이브 DB 쿼리, 실시간 데이터
#   + 행동 수행 (쓰기, 트리거, 커밋)
#   + 로컬 파일/현재 코드베이스 분석
# 사용하지 말아야 할 때:
#   - 외부 상태가 필요 없는 단순 Q&A
#   - 고빈도 저지연 호출 (도구 왕복이 쌓임)

실전 경험: 코드베이스 분석 도구를 만들며

세 기술이 구체적으로 와닿았던 프로젝트가 있다. 엔지니어들이 Spark 코드베이스에 자연어로 질문할 수 있는 도구를 만들었다. "customer 테이블에 쓰는 잡이 뭐야?" 또는 "shuffle hint 없이 repartition을 쓴 곳을 다 찾아줘" 같은 질문이었다.

첫 번째 시도는 순수 RAG였다. Python 파일을 전부 임베딩하고, 관련 청크를 검색해서 LLM에 넘겼다. 어느 정도 됐다. 하지만 청킹이 문제였다. Python 파일은 깔끔하게 잘리지 않는다. 함수 컨텍스트가 쪼개지고, 검색 결과는 엉뚱한 파일 섹션을 가져오기 일쑤였다.

"RAG는 모델에게 코드베이스를 갈가리 찢은 버전을 건네는 느낌이었다. MCP는 터미널을 직접 쥐어주는 느낌이었다."

디스크에서 파일을 직접 읽고, 디렉터리를 나열하고, 패턴을 grep하고, 경량 AST 분석 스크립트를 실행할 수 있는 MCP 서버로 전환하자 경험이 완전히 달라졌다. 모델이 도구에게 "/src/jobs 아래 .py 파일을 모두 나열해줘"라고 물어보고, 특정 파일을 필요할 때 직접 읽고, repartition 호출을 전부 grep할 수 있었다. 낡은 임베딩도, 청킹 부작용도 없었다. 그냥 라이브 파일 접근이었다.

반면 온보딩 가이드, 아키텍처 ADR, 운영 가이드 같은 문서 쪽은 RAG를 유지했다. 천천히 바뀌고, 시맨틱 검색이 잘 맞고, 검색 지연이 크게 문제 되지 않는다. 하지만 라이브 코드베이스나 운영 중인 시스템을 건드리는 모든 것에는 MCP가 맞는 답이었다.

치트 시트: 필요한 것에 따라 고르는 법

  • 범용 지식 질문, 추론, 코드 생성 → LLM 단독
  • 내부 문서·위키 검색, 환각 억제, 대규모 정적 코퍼스 → RAG
  • 실시간 DB 쿼리, 외부 시스템 행동 수행, 라이브 코드베이스 분석 → MCP
  • 문서 기반 컨텍스트 + 라이브 시스템 행동 모두 필요 → RAG + MCP 조합

마치며

처음에 나는 LLM을 마법 상자처럼 다루며 컨텍스트를 계속 채워 넣으면 나아질 거라 생각했다. RAG는 검색과 생성을 분리하는 법을 가르쳐줬다. MCP는 모델이 모든 것을 알 필요가 없다는 것을, 그저 올바른 도구와 그것을 쓸 판단력만 있으면 된다는 것을 알려줬다.

지금 AI로 의미 있는 것을 만들고 있다면, 세 기술이 각각 어디에 맞는지 이해하는 것이 수많은 아키텍처 후회를 줄여줄 것이다. 가장 단순하게 작동하는 것부터 시작하라. 보통은 LLM 단독 호출이다. 특정 코퍼스에 근거를 둔 답변이 필요하면 RAG를 추가하라. AI가 세상을 묘사하는 게 아니라 세상과 상호작용해야 할 때 MCP를 사용하라.

문제에 맞는 도구를 고르자. 최고의 AI 아키텍처는 가장 복잡한 것이 아니라 실제 문제를 해결하는 것이다.
