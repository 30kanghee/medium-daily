Claude Code Has Four Agent Modes. Most Developers Only Use One.

Kristopher Dunham · 10분 읽기 · 원문 · freedium

요약

Claude Code는 단순한 대화형 세션을 넘어 네 가지 위임 패턴을 제공하며, 대부분의 개발자는 그 중 첫 번째만 사용하고 있다.

Claude Code는 인터랙티브 세션 하나에만 머물기에는 너무 강력한 도구다. 이 글은 대부분의 개발자가 놓치고 있는 세 가지 추가 모드—서브에이전트(subagent), 헤드리스(headless) 자동화, 다중 에이전트 조율—를 소개한다. 핵심 통찰은 하나의 창 안에서 모든 작업을 처리하려는 습관이 Claude Code의 가장 큰 걸림돌이라는 것이다. 컨텍스트 창이 포화되면 품질이 저하되고, 작업이 직렬화되어 속도가 느려진다. 서브에이전트를 활용하면 작업을 병렬로 분산시켜 메인 세션을 날카롭게 유지하면서 훨씬 빠른 성과를 낼 수 있다.

개발자들의 터미널에서 이상한 일이 벌어지고 있다

현직 개발자들의 터미널에서 이상한 일이 벌어지고 있다. 모두가 사랑하게 된 그 대화형 AI 채팅 박스가 뒷줄로 밀려나고 있다. 여전히 잘 작동한다. 다만 더 이상 생산성이 집중되는 곳이 아닐 뿐이다.

진짜 작업은 이제 당신이 잠든 사이에 실행되는 에이전트들, 공유 메일함을 통해 서로 소통하는 에이전트들, 그리고 대화 대신 JSON 설정 파일을 받아 처리하는 에이전트들에서 이루어진다. Anthropic의 터미널 기반 코딩 도구인 Claude Code는 이제 네 가지 위임 패턴을 제공한다. 대부분의 개발자는 그 중 첫 번째인 인터랙티브 세션에만 머물고 있다.

이것은 포뮬러 1 경주차를 사서 슈퍼마켓만 다니는 것과 같다.

핵심 명제: Claude Code에서 겪는 거의 모든 문제의 해법은, 모든 작업을 하나의 창 안에서 처리하려는 습관을 버리는 것이다.

네 가지 패턴 한눈에 보기

깊이 들어가기 전에 전체 지도를 살펴보자.

  • 패턴 1 - 인터랙티브 세션: 터미널을 열고, 작업을 주고, 결과를 기다리는 익숙한 방식. 모든 것이 한 창 안에서 순서대로 진행된다.
  • 패턴 2 - 서브에이전트(Subagents): 메인 세션이 조율자 역할을 맡아 작업을 분할하고 각 조각을 독립된 서브에이전트에 위임한다. 병렬 실행이 가능하다.
  • 패턴 3 - 헤드리스(Headless) 자동화: 대화 없이 스크립트나 JSON 설정으로 에이전트를 실행한다. CI/CD 파이프라인이나 야간 자동화 작업에 적합하다.
  • 패턴 4 - 다중 에이전트 조율: 여러 에이전트가 공유 상태나 메시지를 통해 서로 소통하며 협력한다. 팀처럼 작동한다.

왜 하나의 세션만으로는 한계가 있는가

대부분의 사람들이 Claude Code를 사용하는 방식은 이렇다. 터미널을 열고, 작업을 주고, 파일을 읽고 수정하는 것을 지켜본다. 완료되면 다음 작업을 준다. 에이전트 하나, 작업 하나, 처음부터 끝까지 순서대로.

이것은 대부분의 상황에서 충분히 잘 작동한다. 하지만 한계가 있고, 그 한계는 진짜 복잡한 작업을 시작하는 순간 드러난다. 새로운 기능을 구현하면서 동시에 테스트를 작성해야 할 때. 모듈을 리팩토링하면서 문서도 업데이트해야 할 때. 코드베이스를 세 가지 각도에서 동시에 탐색해야 할 때. 단일 세션 모델에서는 이 모든 것이 대기열이다. 모든 것이 기다린다.

두 번째 문제는 컨텍스트다. 세션이 길어질수록 컨텍스트 창은 모든 것으로 가득 찬다. 파일 내용, 중간 추론 과정, 검색 결과, 두 시간 전에 내린 결정들. 스레드가 길어질수록 모델의 집중도가 낮아진다. 처음에는 미묘하게, 나중에는 눈에 띄게 품질이 떨어진다.

서브에이전트는 이 두 문제를 동시에 해결한다.

서브에이전트 설정 방법

생각보다 훨씬 간단하다. 특별한 플래그나 복잡한 설정 파일이 필요하지 않다. Claude Code의 서브에이전트는 단순히 폴더에 있는 마크다운 파일이다.

두 가지 위치에 둘 수 있다. ~/.claude/agents/는 전역 범위로, 해당 기기의 모든 Claude Code 세션에서 사용할 수 있다. 프로젝트 내부의 .claude/agents/는 해당 프로젝트에만 적용되며, 팀과 공유하고 싶을 때 저장소에 커밋할 수 있어 유용하다.

각 에이전트는 상단에 YAML 프론트매터 블록과 본문의 시스템 프롬프트로 구성된 단일 마크다운 파일이다. 기본적인 예시를 보면 이렇다.

---
name: swift-engineer
description: SwiftUI 뷰나 Swift 네트워킹 코드를 작성해야 할 때 사용하라. iOS 화면, ViewModel, async/await 네트워킹 작업에 호출하라.
model: claude-sonnet-4-6
tools: Read, Write, Glob
---

당신은 Swift만 작성하는 시니어 iOS 엔지니어다. MVVM을 엄격히 따른다. 뷰는 더미(dumb)로 유지하고, 로직은 ViewModel에, 모든 네트워크 호출은 async/await로 처리한다.

무언가를 작성하기 전에 Glob으로 기존 프로젝트 구조를 읽고 이미 정착된 컨벤션을 따르라. 생성하거나 수정한 모든 파일 목록을 반환하라.

name은 당신과 Claude가 에이전트를 참조하는 방식이다. description은 메인 에이전트가 언제 이 에이전트를 사용할지 결정하는 핵심 부분이다. 라우팅 규칙처럼 작성하라. 언제 호출할지와 무엇을 돌려주는지를 구체적으로 명시해야 한다. 모호한 설명은 모호한 라우팅을 낳는다. "이럴 때 사용하라..." 다음에 구체적인 조건을 적는 패턴이 잘 작동한다.

tools 필드는 허용 목록이다. 테스트 작성 에이전트는 파일을 삭제할 필요가 없다. 코드 리뷰어는 쓰기 권한이 없어도 된다. 에이전트별로 도구를 제한하는 것은 보안과 예측 가능성 면에서 가장 유용한 설정 중 하나다.

병렬 실행: 실제로 어떻게 작동하는가

병렬 작업을 위한 메인 에이전트 프롬프트는 간단하다. 예를 들면 이렇다.

Swift 프론트엔드와 Swift Vapor 백엔드로 구성된 iOS 앱을 만들고 있다.

세 개의 서브에이전트를 병렬로 실행하라:
1. swift-engineer 서브에이전트를 사용하라. SwiftUI 로그인 화면과 대시보드를 구축하라.
2. backend-engineer 서브에이전트를 사용하라. /auth/login 엔드포인트를 위한 Vapor API를 구축하라.
3. test-writer 서브에이전트를 사용하라. API 엔드포인트와 Swift ViewModel 테스트를 작성하라.

각 서브에이전트는 자체 컨텍스트 안에서 독립적으로 동시에 실행되며 결과를 따로 보고한다. 세 가지 순차 검색이 점점 부풀어 오르는 컨텍스트 스레드 안에서 이루어지는 것 대신, 세 개의 깔끔하게 격리된 작업자가 동시에 실행된다.

서브에이전트가 빛나는 상황들

모든 작업이 서브에이전트의 혜택을 받는 것은 아니다. 변수 이름을 바꾸는 데 에이전트 세 개를 쓰는 것은 시간도 토큰도 낭비다. 토큰 비용은 현실이다. 서브에이전트 중심의 워크플로는 동일한 작업을 단일 세션으로 처리할 때보다 훨씬 많은 토큰을 소모할 수 있다.

하지만 이 투자가 명확히 가치 있는 상황들이 있다.

코드베이스 탐색이 가장 명확한 이득을 주는 경우다. 대규모 프로젝트에서 어떤 것이 어떻게 작동하는지 이해하려 할 때, 사실상 여러 방향에서 동시에 검색을 실행하는 것이다. 각 방향에 하나의 서브에이전트를 두면 모두 동시에 실행되고, 각자 자신의 컨텍스트에서 필요한 것을 읽고 집중된 요약을 돌려준다.

테스트 커버리지도 반복해서 돌아오는 사례다. 테스트 작성은 본질적으로 병렬화할 수 있다. 모듈당 하나의 서브에이전트를 두면 각자 독립적으로 커버리지를 작성하며 서로를 기다리지 않는다.

독립성이 핵심이다

이 모든 것을 묶는 패턴은 독립성이다. 두 작업이 상태를 공유해야 하거나 한 작업이 다른 것의 결과에 의존한다면, 그것들은 메인 스레드에 속한다. 서로를 알 필요 없이 실행될 수 있다면, 서브에이전트 후보다. 이 경계선은 눈을 뜨고 보면 대개 꽤 명확하다.

인터랙티브 세션은 여전히 유용하다. 다만 이제는 Claude Code가 할 수 있는 것 중 가장 기초적인 레벨이라는 사실을 알게 되었을 뿐이다. 나머지 세 가지 패턴은 이미 거기 있었다. 대부분의 개발자들이 찾아보지 않았을 뿐이다.
