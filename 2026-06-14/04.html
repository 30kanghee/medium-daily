5 Data Ingestion Methods Every Data Engineer Should Know

Henry · Data Engineer Things · 11 min read · 원문 · freedium

데이터 엔지니어라면 반드시 알아야 할 5가지 데이터 수집 방식을 파이프라인 실무 관점에서 명쾌하게 정리한 글이다.

요약

파이프라인 장애의 원인은 변환 로직보다 잘못된 수집 방식 선택에 있는 경우가 더 많다. 이 글은 전체 로드, 증분 로드, CDC(변경 데이터 캡처), 스트리밍, API 기반 수집이라는 다섯 가지 방식을 언제 어떻게 쓰는지 파이썬 예제와 함께 설명한다. 전체 로드는 소규모 정적 테이블에, 증분 로드는 대형 이력 테이블에, CDC는 삭제 추적이 필요한 실시간 환경에 각각 적합하다. 스트리밍은 지연이 허용되지 않는 이벤트 데이터에, API 수집은 외부 SaaS 연동에 쓴다. 복잡성이 낮은 방식부터 시작해 실제 한계에 부딪힐 때 이동하라는 것이 핵심 조언이다.

수집 방식이 중요한 이유

파이프라인이 실패하는 원인은 대부분 변환 로직이 아니다. 실시간에 가까운 처리가 필요한 데이터에 배치 잡을 돌리거나, 증분 로드가 필요한 곳에 전체 로드를 써버리는 선택 실수 때문이다. 올바른 수집 방식은 데이터 엔지니어링에서 가장 먼저 내려야 할 결정이다.

방식 1: 전체 로드 (Full Load)

전체 로드는 매번 소스 테이블의 모든 데이터를 읽어 대상 시스템에 덮어쓴다. 구현이 단순하고, 소규모 참조 테이블이나 스냅샷 데이터처럼 변경이 드물고 크기가 작은 테이블에 적합하다.

언제 쓰는가: 행 수가 수만 건 이하이고, 변경 추적이 필요 없으며, 소스에서 삭제가 발생하는 경우.

import pandas as pd
from sqlalchemy import create_engine

engine = create_engine("postgresql://user:pass@host/db")
df = pd.read_sql("SELECT * FROM products", engine)
df.to_sql("products_snapshot", engine, if_exists="replace", index=False)
print(f"Full load complete: {len(df)} rows")

단점은 테이블이 커질수록 비용과 시간이 선형으로 증가한다는 것이다.

방식 2: 증분 로드 (Incremental Load)

증분 로드는 마지막 실행 이후 변경된 레코드만 읽어온다. 타임스탬프 컬럼이나 자동 증가 ID를 워터마크로 활용하며, 수백만 행 규모의 트랜잭션 테이블에 적합하다.

언제 쓰는가: 소스에 updated_at 또는 단조 증가 ID가 있고, 볼륨이 커서 전체 로드가 비현실적인 경우.

import pandas as pd
from datetime import datetime

def incremental_load(engine, last_run):
    query = ("SELECT * FROM orders "
             "WHERE updated_at > '" + last_run.isoformat() + "' "
             "ORDER BY updated_at")
    df = pd.read_sql(query, engine)
    if not df.empty:
        df.to_sql("orders_dw", engine, if_exists="append", index=False)
    return df["updated_at"].max() if not df.empty else last_run

증분 로드의 함정은 소스에서 레코드가 삭제될 때 대상 시스템이 이를 알 수 없다는 점이다. 삭제 추적이 필요하다면 CDC가 필요하다.

방식 3: 변경 데이터 캡처 (CDC)

CDC는 데이터베이스 트랜잭션 로그(WAL, binlog 등)를 읽어 INSERT·UPDATE·DELETE를 모두 캡처한다. Debezium, AWS DMS 같은 도구가 이 역할을 한다.

언제 쓰는가: 소스 DB를 수정할 수 없고, 삭제 이벤트를 캡처해야 하며, 지연이 수 초 이내여야 하는 경우.

connector_config = {
    "name": "postgres-orders-cdc",
    "config": {
        "connector.class": "io.debezium.connector.postgresql.PostgresConnector",
        "database.hostname": "db-host",
        "database.port": "5432",
        "database.user": "debezium",
        "database.dbname": "production",
        "table.include.list": "public.orders",
        "plugin.name": "pgoutput"
    }
}

CDC는 강력하지만 복잡성이 높다. 트랜잭션 로그 보존 설정, 컨슈머의 오프셋 관리, 스키마 변경 처리를 신경 써야 한다.

방식 4: 스트리밍 수집

스트리밍은 이벤트 발생 즉시 처리한다. Kafka, Kinesis, Pub/Sub 같은 메시지 브로커를 통해 데이터가 흐른다. 클릭스트림, IoT 센서, 결제 이벤트처럼 지연이 허용되지 않는 데이터에 사용한다.

언제 쓰는가: 지연 허용치가 수 초 이내이고, 데이터가 지속적으로 생성되며, 실시간 집계나 이상 감지가 필요한 경우.

from kafka import KafkaConsumer
import json

consumer = KafkaConsumer(
    "clickstream-events",
    bootstrap_servers=["kafka:9092"],
    value_deserializer=lambda m: json.loads(m.decode("utf-8")),
    auto_offset_reset="latest",
    group_id="analytics-pipeline"
)

for message in consumer:
    event = message.value
    process_event(event)

스트리밍은 배치 대비 운영 복잡도가 훨씬 높다. 상태 관리, 윈도우 집계, 정확히-한-번(exactly-once) 처리, 장애 복구를 모두 고려해야 한다.

방식 5: API 기반 수집

외부 SaaS 서비스에서 데이터를 가져올 때는 REST 또는 GraphQL API를 통한 수집이 유일한 선택지인 경우가 많다. Rate limit, 페이지네이션, 토큰 갱신을 모두 처리해야 한다.

언제 쓰는가: 소스가 API만 제공하거나, DB 직접 접속이 불가능한 외부 서비스인 경우.

import requests, time

def fetch_stripe_charges(api_key, starting_after=None):
    params = {"limit": 100}
    if starting_after:
        params["starting_after"] = starting_after
    resp = requests.get("https://api.stripe.com/v1/charges",
                        auth=(api_key, ""), params=params)
    resp.raise_for_status()
    data = resp.json()
    cursor = data["data"][-1]["id"] if data["data"] else None
    return data["data"], data.get("has_more", False), cursor

all_charges, cursor = [], None
while True:
    charges, has_more, cursor = fetch_stripe_charges("sk_live_...", cursor)
    all_charges.extend(charges)
    if not has_more:
        break
    time.sleep(0.1)

올바른 방식 선택의 기준

세 가지 질문으로 결정할 수 있다.

  • 데이터 볼륨과 변경 빈도: 소규모·저빈도면 전체 로드, 대규모·고빈도면 증분 로드나 CDC.
  • 지연 허용치: 수 시간이면 배치, 수 분이면 마이크로배치, 수 초면 스트리밍.
  • 소스 시스템 인터페이스: DB 직접 접속 가능하면 CDC·증분, API만 있으면 API 수집.

복잡성이 낮은 방식부터 시작하고, 실제 한계에 부딪혔을 때 더 정교한 방식으로 이동하라. 수집 방식 선택은 파이프라인의 안정성과 운영 비용을 결정짓는 가장 중요한 설계 결정이다.
