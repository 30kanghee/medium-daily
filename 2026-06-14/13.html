Agentic Architectures — Article 6: Multi-Agent Orchestration Patterns

Ali Süleyman TOPUZ · 21 min read · 원문 · freedium

단일 에이전트 하니스로 모든 것을 처리할 수 있다는 믿음에서 벗어나, 멀티 에이전트 시스템의 다양한 오케스트레이션 패턴을 프로덕션 관점에서 깊이 있게 분석합니다.

요약

저자는 처음에 잘 설계된 단일 에이전트 하니스로 무엇이든 처리할 수 있다고 믿었지만, 약 3주 만에 그 믿음이 무너졌습니다. 이 글은 멀티 에이전트 시스템의 핵심 오케스트레이션 패턴들—오케스트레이터-워커, 파이프라인, 병렬 팬아웃, 슈퍼바이저-서브에이전트, 브로드캐스트-집계, 계층형 위임—을 실제 프로덕션 환경에서의 트레이드오프와 함께 설명합니다. 각 패턴이 언제 적합하고 언제 적합하지 않은지를 구체적인 인프라 관점에서 다루며, 멀티 에이전트 시스템을 분산 시스템 문제로 접근해야 한다는 핵심 교훈을 전달합니다.

단일 에이전트의 한계를 발견하기까지

저는 약 3주 동안 잘 설계된 단일 에이전트 하니스로 무엇이든 처리할 수 있다고 확신했습니다. 충분한 도구, 충분한 컨텍스트 관리, 충분한 재시도 로직이 있다면 단일 ReAct 루프가 웬만한 것은 다 할 수 있다고 생각했습니다.

그 믿음이 틀렸다는 걸 깨달은 순간은, 에이전트가 코드를 작성하면서 동시에 그 코드를 리뷰하도록 요청했을 때였습니다. 모델이 자신이 방금 만든 결함에 대해 검토하면서 그것을 좋다고 평가하는 모습을 보았습니다. 평가자와 피평가자가 같은 존재일 때 발생하는 확증 편향이었습니다.

그것이 멀티 에이전트 오케스트레이션 패턴을 진지하게 연구하기 시작한 계기였습니다.

왜 패턴이 중요한가

멀티 에이전트 시스템을 구축할 때 가장 많이 범하는 실수는 패턴을 선택하는 것이 아니라 그냥 에이전트를 연결하는 것입니다. "오케스트레이터가 있고, 워커가 있고, 나머지는 알아서 되겠지"라고 생각하는 것입니다. 이 접근 방식은 두 번째 또는 세 번째 복잡한 작업에서 무너집니다.

패턴은 단순히 코드 구성이 아닙니다. 패턴은 세 가지를 강제합니다:

  • 상태가 어디에 살아야 하는가
  • 실패가 어디서 발생할 수 있으며 어떻게 격리되는가
  • 비용이 어떻게 증가하는가

패턴 없이 멀티 에이전트 시스템을 구축하는 것은 서비스 경계를 정의하지 않고 마이크로서비스를 구축하는 것과 같습니다. 기술적으로는 가능하지만, 3개월 후에 디버깅하는 것은 악몽입니다.

패턴 1: 오케스트레이터-워커(Orchestrator-Worker)

가장 기본적이고 가장 많이 오용되는 패턴입니다.

오케스트레이터
    ├── 워커 A (코드 작성)
    ├── 워커 B (테스트)
    └── 워커 C (문서화)

오케스트레이터는 작업을 받아 분해하고, 워커에게 하위 작업을 할당하고, 결과를 조합합니다. 구현하기 쉽고 추론하기 쉽고 디버깅하기 쉽습니다.

언제 적합한가

  • 하위 작업이 잘 정의되어 있고 상호 의존성이 낮을 때
  • 오케스트레이터가 전체 계획을 사전에 알 수 있을 때
  • 실패가 격리될 수 있을 때 (워커 A가 실패해도 B, C는 계속 진행 가능)

언제 부적합한가

오케스트레이터 자체가 복잡한 추론을 수행해야 할 때입니다. 오케스트레이터가 30단계짜리 계획을 세우고 있다면, 이미 그것은 오케스트레이터가 아니라 에이전트입니다. 역할이 뒤섞입니다.

인프라 시사점

오케스트레이터는 모든 워커의 상태를 유지해야 합니다. 오케스트레이터가 실패하면 전체가 실패합니다. AWS Step Functions나 LangGraph의 StateGraph 같은 내구성 있는 실행 환경이 필요합니다.

패턴 2: 파이프라인(Pipeline)

각 에이전트가 이전 에이전트의 출력을 입력으로 받는 선형 체인입니다.

에이전트 A → 에이전트 B → 에이전트 C → 최종 출력
(리서치)    (초안 작성)  (편집)

파이프라인이 잘 작동하는 경우

각 단계가 이전 단계의 출력을 의미 있게 변환할 때입니다. 리서치 → 초안 → 편집은 실제로 파이프라인이 적합합니다. 각 단계에서 출력이 실질적으로 달라집니다.

파이프라인이 실패하는 경우

초기 단계에서 오류가 발생하면 이후 모든 단계로 전파됩니다. 에이전트 A가 잘못된 사실을 리서치하면, 에이전트 B는 그 잘못된 사실로 초안을 작성하고, 에이전트 C는 잘못된 사실이 담긴 초안을 편집합니다. 파이프라인은 정확도를 증폭시키는 게 아니라 오류를 증폭시킵니다.

해결책

각 단계에 검증 레이어를 추가하세요. 파이프라인 에이전트는 이전 단계의 출력을 수동적으로 수락하는 것이 아니라 능동적으로 검사해야 합니다.

패턴 3: 병렬 팬아웃(Parallel Fan-out)

동일한 입력을 여러 에이전트에게 동시에 보내고 결과를 집계합니다.

입력
 ├──→ 에이전트 A ──┐
 ├──→ 에이전트 B ──┼──→ 집계기 → 출력
 └──→ 에이전트 C ──┘

언제 사용하는가

  • 동일한 문제에 대해 다양한 관점이 필요할 때 (예: 코드를 여러 리뷰어에게 동시에 보내기)
  • 독립적인 하위 작업을 병렬로 처리할 때 (예: 여러 시장을 동시에 분석하기)
  • 앙상블 출력으로 신뢰도를 높이고 싶을 때

비용 경고

팬아웃은 비용을 즉시 N배로 증가시킵니다. N개의 에이전트가 병렬로 실행되면 N번의 LLM 호출이 발생합니다. 집계기도 또 하나의 LLM 호출입니다. 이것이 정당화되려면 병렬성의 이점이 비용을 초과해야 합니다.

집계 문제

집계기가 N개의 출력을 어떻게 결합하는지가 핵심입니다. 단순 다수결은 동의한 에이전트 수를 최대화하지만 최선의 답을 최대화하지 않습니다. 품질 기반 선택은 또 다른 판단 레이어가 필요합니다.

패턴 4: 슈퍼바이저-서브에이전트(Supervisor-Subagent)

오케스트레이터-워커와 비슷하지만 핵심 차이가 있습니다. 슈퍼바이저는 단순히 작업을 위임하는 것이 아니라 서브에이전트의 출력을 평가하고 거부하거나 수정할 권한을 갖습니다.

슈퍼바이저
    ↕ (평가 + 거부 + 재지시)
서브에이전트

이것이 중요한 이유

이것이 순수한 오케스트레이터-워커와의 차이입니다. 오케스트레이터는 결과를 받아 다음 단계로 전달합니다. 슈퍼바이저는 결과를 받아 충분히 좋은지 판단하고, 충분하지 않으면 거부합니다.

이 패턴은 제가 처음에 언급한 "에이전트가 자신의 출력을 리뷰하는" 문제를 해결합니다. 슈퍼바이저가 서브에이전트와 다른 모델에서 실행되면, 확증 편향이 구조적으로 불가능해집니다.

구현 시 주의사항

슈퍼바이저는 언제 "충분히 좋은가"를 판단하는 명확한 기준이 필요합니다. 그 기준이 모호하면 슈퍼바이저가 루프를 멈추지 않을 수 있습니다. 항상 최대 반복 횟수를 설정하세요.

패턴 5: 브로드캐스트-집계(Broadcast-Aggregate)

팬아웃의 변형이지만, 에이전트들이 같은 작업이 아니라 다른 도구나 데이터 소스에 질의합니다.

질의
 ├──→ 에이전트 (웹 검색)    ──┐
 ├──→ 에이전트 (데이터베이스) ──┼──→ 합성기 → 응답
 └──→ 에이전트 (내부 문서)   ──┘

핵심 사용 사례

멀티소스 리서치. 사용자가 질문을 던지면 여러 에이전트가 각자의 전문 소스에서 동시에 검색하고, 합성기가 그 결과를 통합된 답변으로 만듭니다. 단일 에이전트가 소스를 순차적으로 검색하는 것보다 훨씬 빠릅니다.

합성의 어려움

소스들이 서로 모순될 때가 문제입니다. 합성기는 단순히 결합하는 것이 아니라 충돌을 해결해야 합니다. 이는 생각보다 훨씬 어려운 추론 작업입니다.

패턴 6: 계층형 위임(Hierarchical Delegation)

오케스트레이터-워커를 여러 레벨로 중첩합니다. 최상위 오케스트레이터가 중간 레벨 오케스트레이터에게 위임하고, 그 중간 레벨이 리프(leaf) 에이전트에게 위임합니다.

최상위 오케스트레이터
    ├── 중간 오케스트레이터 (개발팀)
    │       ├── 리프 에이전트 (백엔드)
    │       └── 리프 에이전트 (프론트엔드)
    └── 중간 오케스트레이터 (QA팀)
            ├── 리프 에이전트 (단위 테스트)
            └── 리프 에이전트 (통합 테스트)

이 패턴이 빛나는 순간

작업이 진정으로 계층적인 분해를 요구할 때입니다. 소프트웨어 개발 프로젝트가 좋은 예입니다. 자연스럽게 팀(개발, QA, 배포)으로 분해되고, 각 팀은 다시 역할(백엔드, 프론트엔드, DB)로 분해됩니다.

운영 비용 경고

이 패턴은 분산 시스템 문제를 완전히 가져옵니다. 중간 레벨 오케스트레이터가 실패하면 그 아래의 모든 에이전트가 고아가 됩니다. 각 레벨에 내결함성이 필요합니다. LangGraph나 AWS Bedrock AgentCore 같은 내구성 있는 실행 런타임 없이는 이 패턴을 프로덕션에서 실행하지 마세요.

패턴 선택: 의사결정 프레임워크

이 모든 패턴 중에서 어떤 것을 선택해야 할까요? 저는 다음 질문들을 사용합니다:

1. 하위 작업들이 독립적인가?

  • 예: 병렬 팬아웃 또는 브로드캐스트-집계를 고려하세요.
  • 아니오: 파이프라인 또는 오케스트레이터-워커를 사용하세요.

2. 출력 품질 검증이 필요한가?

  • 예: 슈퍼바이저-서브에이전트를 추가하세요. 비용이 증가하지만 품질은 향상됩니다.
  • 아니오: 더 단순한 패턴으로 시작하고 필요하면 추가하세요.

3. 작업이 자연스럽게 계층적으로 분해되는가?

  • 예: 계층형 위임이 적합하지만 운영 복잡성에 준비하세요.
  • 아니오: 단일 레벨 패턴을 사용하세요. 불필요한 계층은 디버깅만 어렵게 합니다.

4. 예산 제약이 있는가?

  • 엄격한 비용 한도: 오케스트레이터-워커 또는 파이프라인을 사용하세요. 예측 가능한 비용.
  • 비용보다 품질이 우선: 슈퍼바이저 레이어나 팬아웃 확인을 추가하는 것을 고려하세요.

모든 패턴에 적용되는 프로덕션 원칙들

특정 패턴에 관계없이, 멀티 에이전트 시스템을 프로덕션에서 운영할 때 항상 적용하는 원칙들이 있습니다:

에이전트 경계를 좁게 유지하세요

에이전트가 수행하는 일이 좁을수록 실패 원인을 추적하기 쉽습니다. 모든 것을 할 수 있는 에이전트는 모든 방식으로 실패할 수 있습니다.

루프에는 항상 최대 반복 횟수를 설정하세요

슈퍼바이저 패턴, 반성(reflection) 루프, 재시도 메커니즘. 모두 상한이 없으면 무한 루프가 됩니다. 무한 루프는 무한한 비용을 의미합니다.

에이전트 경계를 넘는 관찰 가능성을 구축하세요

단일 에이전트의 추론 트레이스는 LangSmith나 Arize Phoenix로 추적할 수 있습니다. 멀티 에이전트 시스템에서는 에이전트 간의 상호작용을 추적해야 합니다. 에이전트 B가 에이전트 A에서 받은 입력과 그것으로 한 일을 볼 수 있어야 합니다.

멱등성(idempotency)을 설계하세요

에이전트가 실패하면 어디서나 재시작할 수 있어야 합니다. 이는 에이전트가 수행하는 모든 작업이 멱등해야 한다는 것을 의미합니다. 두 번 실행해도 다른 결과가 나오지 않아야 합니다.

멀티 에이전트 오케스트레이션 패턴 비교

패턴              비용 모델        주요 실패 모드      최적 사용 사례
──────────────────────────────────────────────────────────────────────
오케스트레이터-워커  예측 가능       오케스트레이터 실패  잘 정의된 하위 작업
파이프라인          선형            오류 전파            순차적 변환
병렬 팬아웃         N× 즉시 증가   집계 복잡성          독립적 병렬 작업
슈퍼바이저-서브에이전트 반복 시 높음  반복 루프          품질 검증 필요 시
브로드캐스트-집계   N× 즉시 증가   소스 충돌            멀티소스 리서치

마치며

멀티 에이전트 오케스트레이션 패턴은 소프트웨어 패턴이 아닙니다. 이것들은 분산 시스템 패턴입니다. 그 렌즈로 접근하면 훨씬 더 명확해집니다.

마이크로서비스에서 서비스 경계를 설계할 때와 같은 질문을 하세요:

  • 이 컴포넌트가 실패하면 다른 것들이 계속 동작할 수 있나요?
  • 이 상태는 어디에 살아야 하고 누가 소유하나요?
  • 이 인터페이스는 테스트 가능하고 관찰 가능한가요?

이 질문들에 답할 수 있다면, 어떤 패턴을 선택하든 프로덕션 준비가 된 시스템을 구축할 수 있습니다.

다음 글에서는 이 패턴들을 실제 엔터프라이즈 사용 사례에 매핑하고, 각각의 비용-품질 곡선을 측정하는 방법을 살펴보겠습니다.
