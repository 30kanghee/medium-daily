AI Agent Best Practices: Production-Ready Harness Engineering (2026 Guide)

Tort Mario · 8분 · 원문 · freedium
AI 에이전트가 프로덕션에서 실패하는 이유는 LLM이 약해서가 아니라 하네스(harness)가 취약하기 때문이다 — 2026년 에이전트 하네스 엔지니어링 완전 가이드.

AI 에이전트 생태계가 폭발적으로 성장하지만, 대부분의 에이전트는 프로덕션에서 실패한다. LLM 자체가 약해서가 아니라, 에이전트를 감싸는 런타임 래퍼인 "하네스"가 취약하기 때문이다. 에이전트 하네스는 모델이 제안하는 모든 액션을 검증·인가·실행·로깅하는 결정론적 레이어로, "모델이 제안하고 하네스가 실행"하는 원칙이 핵심이다. 이 가이드는 Map-Identify-Blueprint-Implement-Launch 5단계 방법론과 예산(step/time/token/cost), 권한 매트릭스, 컨텍스트 레이어링, 보안 평가 체크리스트를 포함하는 오픈소스 저장소를 바탕으로 제작 가능한 AI 에이전트 하네스 설계의 실전 기준을 제시한다.

AI 에이전트 환경이 폭발적으로 성장하고 있다. 고객 지원 봇부터 금융 분석 코파일럿까지, 에이전트는 복잡한 워크플로우 자동화를 약속한다. 그러나 대부분의 에이전트는 프로덕션에서 실패한다. LLM이 약해서가 아니라, 에이전트를 관장하는 런타임 래퍼인 하네스가 취약하거나, 안전하지 않거나, 예측 불가능하기 때문이다.

수백 건의 실패한 에이전트 배포를 분석한 결과, 하나의 명확한 패턴이 나온다. 부재한 것은 훈련된 하네스 엔지니어링이다.

여기 agents-best-practices를 소개한다. Claude Code와 Codex의 내부 구조에서 영감을 받아 Denis Sergeevitch가 구축한 제공자 중립적이고 프로덕션 준비된 오픈소스 저장소로, 구체적인 결과물을 제공한다. 컴포넌트 모델부터 체크리스트까지.

에이전트 하네스란 무엇이고 왜 필요한가?

에이전트 하네스는 LLM을 감싸는 결정론적 런타임 레이어다. 모델이 제안하는 모든 액션을 검증하고, 인가하고, 실행하고, 로깅한다. 핵심 아이디어는 명확한 책임 분리다:

  • 모델은 액션과 도구 호출을 제안한다.
  • 하네스는 이를 실행한다 — 스키마, 권한, 예산, 안전 규칙을 확인하면서.

적절한 하네스 없이는 에이전트가 토큰을 누수하고, 통제 불가능한 루프를 돌고, 위험한 명령을 실행하는 블랙박스가 된다. 하네스가 있으면 예측 가능성, 보안, 관찰 가능성을 얻는다.

타협 불가능한 원칙들

이 원칙들은 제공자 중립적이다 — OpenAI, Anthropic, 오픈소스 모델, 또는 미래의 어떤 LLM과도 작동한다.

저장소 심층 분석

저장소는 에이전트 스킬 명세를 따른다. SKILL.md가 진입점이고, 상세 참조 문서는 references/ 폴더에 있다.

왜 "또 다른 팁 목록"이 아닌가

대부분의 블로그 포스트는 "도구 입력을 검증하라"처럼 추상적인 조언을 준다. 이 스킬은 실행 가능한 결과물을 제공한다:

  • 15개 모듈로 구성된 컴포넌트 모델 (명령 관리자, 컨텍스트 빌더, 모델 어댑터, 도구 레지스트리, 권한 리졸버, 예산 추적기 등)
  • 예산, 압축 트리거, 중단 조건을 포함한 의사 코드로 작성된 표준 에이전틱 루프
  • 리스크 분류 체계 (read_only, financial, destructive 등)와 권한 매트릭스
  • 프롬프트 캐싱 비용을 줄이기 위한 캐시 인식 순서 전략
  • 모델뿐 아니라 하네스 자체를 테스트하는 보안 평가 (주입, 타임아웃, 과다 도구화)

프로덕션 준비 에이전트를 위한 핵심 원칙

1. 모델이 제안하고 — 하네스가 실행한다

절대로 LLM이 도구를 직접 호출하게 하지 마라. 대신 모델이 구조화된 도구 호출을 반환하면, 하네스가 스키마를 검증하고, 권한을 확인하고, 실행하고, 결과를 다시 주입한다. 이것이 프롬프트 주입이 임의 코드 실행으로 확산되는 것을 막는다.

2. 모든 도구 호출은 결과를 반환한다 — 실패 시에도

성공적인 API 응답이든, 권한 거부든, 타임아웃이든, 에이전트는 항상 구조화된 관찰값을 받는다. 미처리 약속(dangling promises)은 없다.

3. 리스크가 프로세스를 바꾼다

최소 세 가지 리스크 수준을 사용하라:

  • 읽기 전용 (자율)
  • 초안 (내부 시뮬레이션, 외부 부작용 없음)
  • 외부 쓰기 (승인 필요)

이것이 초안-커밋 패턴이다 — 위험한 액션은 먼저 초안으로 만들어지고, 그 다음 명시적으로 커밋된다.

4. 컨텍스트는 조립되는 것이지 덤프되는 것이 아니다

전체 대화 기록을 매 턴에 밀어넣지 마라. 레이어를 사용하라:

  • 정책 (시스템 레벨, 거의 변경되지 않음)
  • 범위 지정 명령 (작업별 또는 도메인별)
  • 런타임 힌트 (메모리나 도구에서 JIT 검색)

신뢰할 수 없는 데이터 (예: 사용자 입력)에 신뢰 레이블을 표시하여 하네스가 다르게 처리할 수 있게 하라.

5. 장기 작업에는 예산이 있다

모든 에이전트 루프는 다음을 갖춰야 한다:

  • 단계 예산 (최대 반복 횟수)
  • 시간 예산 (실제 시간)
  • 토큰 예산 (턴당 및 누적)
  • 비용 예산 (USD 한도)

예산이 소진되면, 하네스는 우아하게 종료하고 구조화된 실패를 반환한다.

6. 반복되는 실패는 하네스 기능이 된다

도구가 잘못된 응답을 반환해서 에이전트가 반복적으로 실패한다면, 프롬프트에서 수정하지 말고 — 하네스 안에 유효성 검사 함수를 작성하라. 에이전트가 항상 같은 누락 정보를 요청한다면, 자동으로 검색하는 도구를 만들어라.

아이디어에서 프로덕션까지: 단계별 가이드

저장소는 구체적인 방법론을 제공한다: Map → Identify → Blueprint → Implement → Launch.

1단계: Map (올바른 질문하기)

코드를 쓰기 전에 답하라:

  • 어떤 도메인인가? (고객 지원, 금융, DevOps 등)
  • 어떤 수준의 자율성인가? (레벨 0 = 사람이 모든 것을 하는 것, 레벨 4 = 완전 자율)
  • 어떤 리스크 수준인가? (읽기 전용, 금융, 파괴적)
  • 어떤 외부 시스템인가? (Slack, Linear, Drive, 데이터베이스, API)

2단계: Identify (MVP 수준 선택)

답변에 따라 mvp-agent-blueprint.md 표에서 MVP 수준을 선택한다. 대부분의 첫 번째 에이전트에는 레벨 1 (사람이 모든 외부 쓰기를 승인) 또는 레벨 2 (계획은 사람이 승인하고, 저위험 단계는 자율 실행 가능)가 적합하다.

3단계: Blueprint (하네스 설계 생성)

스킬에 청사진을 생성하도록 요청하라. 출력물에는 다음이 포함된다:

  • 목표와 도메인 경계
  • 에이전틱 루프 (중단 조건, 예산)
  • 도구 레지스트리 (타입 스키마와 리스크 클래스가 있는 각 도구)
  • 권한 매트릭스 (누가 무엇을 언제 호출할 수 있는지, 언제 승인이 필요한지)
  • 컨텍스트 및 메모리 레이어링 (캐시에 남는 것, JIT 검색되는 것)
  • 스킬 및 커넥터 (사용할 에이전트 스킬 또는 MCP 서버)

4단계: Implement (청사진을 엄격히 따르기)

설명된 경계 내에서 정확히 MVP를 구축한다. 골격과 검증 경로부터 시작한 다음 측정된 확장을 추가하라. checklists.md 파일이 줄별 구현 체크리스트를 제공한다.

5단계: Launch (라이브 전 감사)

프로덕션 전에 checklists.md의 감사 체크리스트를 실행하라. 확인할 것:

  • 예산이 적용되었는가
  • 권한이 올바른가 (execute_anything 도구가 없는가)
  • 주입 및 타임아웃 평가가 통과되는가
  • 관찰 가능성 (트레이스, 로그)이 갖춰져 있는가

실전 사례: 계약 리스크 분석 에이전트

스킬을 사용하여 한 팀이 다음과 같은 에이전트를 구축했다:

  • 계약 초안 읽기 (읽기 전용, 자율)
  • 리스크 브리프 및 초안 액션 생성 (초안 모드)
  • 명시적 승인 후에만 이메일 발송 (외부 쓰기, 승인 필요)

하네스 청사진 생성에 15분, 구현에 이틀이 걸렸으며, 에이전트는 6개월간 무단 액션 없이 운영됐다.

구현 팁: 표준 에이전틱 루프

budgets = Budgets(step=25, time=120, tokens=8000, cost=0.50)
context = build_initial_context()
permissions = load_permission_matrix()

while not budgets.exhausted():
    response = model.generate(context, tools=typed_tool_schemas)
    if response.finish_reason == "stop":
        break
    if response.tool_calls:
        for tool_call in response.tool_calls:
            if not permissions.is_allowed(tool_call):
                observation = "Permission denied: " + tool_call.name
            else:
                # 리스크에 맞는 검사와 함께 실행
                if permissions.risk(tool_call) == "external_write":
                    approval = request_human_approval(tool_call.draft)
                    if not approval:
                        observation = "Human rejected: " + tool_call.name
                    else:
                        observation = execute_tool(tool_call)
                else:
                    observation = execute_tool(tool_call)
            context.append(observation)
        # 선택적 압축 트리거
        if context.token_count() > budgets.token_per_turn:
            context = compact_context(context, preserve_approvals=True)
    else:
        # 도구 호출 없음, 최종 답변
        break

캐시 효율을 위한 컨텍스트 레이어링

Layer 0: System policies (stable prefix)   → Cached
Layer 1: Agent skill definitions (rarely change) → Cached
Layer 2: User session instructions (per conversation) → Not cached
Layer 3: JIT-retrieved tool outputs (fresh) → Not cached

가장 안정적인 것에서 가장 불안정한 것 순서로 레이어를 배열하면 프롬프트 캐싱을 극대화하고 비용을 줄인다.

보안, 관찰 가능성, 그리고 평가 (건너뛰지 마라)

security-evals-observability.md는 다음을 제공한다:

  • 에이전트 하네스에 대한 위협 모델 (주입, 서비스 거부, 도구 남용, 승인 스푸핑)
  • 다단계 가드레일 (입력 위생화, 권한 검사, 출력 검증)
  • 추적 형식 (모든 단계: 프롬프트, 도구 호출, 관찰값, 지연, 비용)
  • 하네스 자체에 대한 평가 — 단순 모델 정확도만이 아니라:
    • 주입 저항성: 사용자 프롬프트가 시스템 명령을 덮어쓸 수 있는가?
    • 타임아웃 복원력: 도구가 멈추면 하네스가 중단하는가?
    • 과다 도구화: 에이전트가 불필요한 도구를 요청하는가?
