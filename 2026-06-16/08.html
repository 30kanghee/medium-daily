11 Kafka Design Patterns for Every Backend Engineer

Vineet Sharma · 15분 · 원문 · freedium
Kafka를 제대로 쓰려면 패턴이 필요하다 — 이벤트 소싱부터 사가 패턴까지, 백엔드 엔지니어가 반드시 알아야 할 11가지 Kafka 설계 패턴을 AWS 실전 예제와 코드로 완전 정복.

Apache Kafka가 AWS 환경의 이벤트 스트리밍 표준으로 자리잡았지만, 효과적으로 쓰려면 단순한 메시지 생산·소비를 넘어 검증된 설계 패턴이 필수다. 이 글은 4부 시리즈의 파트 1로, 11가지 핵심 패턴 전체를 개요 수준으로 다루며 각 패턴마다 실전 AWS 예제, 다이어그램, Java·Python 코드 스니펫을 제공한다. 이벤트 소싱, CQRS, 클레임 체크, DLQ, 멱등성 컨슈머, 트랜잭셔널 아웃박스, 컴팩션 토픽, 파티션 키 순서 보장, 스트림-테이블 이중성, 사가(코레오그래피)까지 11가지 패턴을 한 번에 조망할 수 있다. 각 패턴의 "언제 쓰는가"와 "AWS에서 어떻게 구현하는가"를 명확히 이해하면 장애에 강하고 확장 가능한 이벤트 드리븐 시스템을 설계할 수 있다.

소개

AWS 위의 Apache Kafka는 스타트업부터 대기업까지 현대 아키텍처의 사실상 표준 이벤트 스트리밍 백본이 됐습니다. 하지만 Kafka를 효과적으로 쓰려면 단순히 메시지를 생산하고 소비하는 것 이상이 필요합니다. 순서 보장, 정확히 한 번 처리 시맨틱, 대용량 페이로드, 장애 복구, MSK·S3·DynamoDB·Lambda 같은 AWS 서비스와의 통합을 다루는 검증된 설계 패턴이 있어야 합니다.

이 4부 시리즈는 11가지 핵심 Kafka 패턴을 다룹니다 — 각각 실전 AWS 예제, 다이어그램, 코드 스니펫(Java & Python)과 함께 설명합니다. 각 패턴의 "언제 쓰는지", "AWS에서 어떻게 구현하는지", 흔한 함정까지 배울 수 있습니다.

11가지 패턴 개요

1. 이벤트 소싱 (Event Sourcing)

무엇인가: 엔티티의 현재 상태만 저장하는 대신, 모든 상태 변경 이벤트를 Kafka에 불변·추가 전용 시퀀스로 저장합니다. 현재 상태는 처음부터 모든 이벤트를 재생해서 도출합니다. Kafka의 로그 기반 저장 방식이 이를 자연스럽게 지원합니다.

언제 쓰는가:

  • 모든 변경의 완전한 감사 이력이 필요할 때
  • 과거 상태를 재구성하거나 디버깅을 위해 이력을 재생해야 할 때
  • 시간적 쿼리("어제 오후 3시 주문 상태는 어땠나?")가 필요할 때

AWS에서: 무제한 보존 기간 MSK 토픽을 생성하거나 MSK Connect S3 Sink Connector로 S3에 백업합니다. 컨슈머는 가장 이른 오프셋부터 읽어 상태를 재구성합니다. ksqlDB로 이벤트 로그에서 뷰를 구체화할 수 있습니다.

# 프로듀서 - 추가 전용
event = {"event_type": "OrderCreated", "order_id": "123", "amount": 100, "status": "pending"}
producer.send("order_events", key="123", value=json.dumps(event))

# 컨슈머 - 재생으로 상태 재구성
state = {}
for msg in consumer:
    event = json.loads(msg.value)
    order_id = event["order_id"]
    if order_id not in state:
        state[order_id] = {}
    if event["event_type"] == "OrderCreated":
        state[order_id] = {"amount": event["amount"], "status": "pending"}
    elif event["event_type"] == "OrderPaid":
        state[order_id]["status"] = "paid"

AWS 팁: Glue Schema Registry로 이벤트 스키마를 안전하게 진화시키세요. 컴플라이언스 요건에 따라 retention.bytes 또는 retention.ms를 설정합니다.

2. CQRS (명령-조회 책임 분리)

무엇인가: 상태를 변경하는 쓰기 경로(커맨드)와 데이터를 조회하는 읽기 경로(쿼리)를 분리합니다. 커맨드는 Kafka로 가고, Kafka가 비동기적으로 하나 이상의 읽기 최적화 모델을 업데이트합니다. 읽기와 쓰기가 독립적으로 확장되고 서로 다른 데이터 구조를 사용할 수 있습니다.

언제 쓰는가:

  • 읽기·쓰기 워크로드의 확장 요건이 다를 때(쓰기 1,000/초, 읽기 100,000/초)
  • 모바일 앱·대시보드·분석 등 다양한 뷰에 맞는 여러 읽기 모델이 필요할 때
  • 읽기 경로에서 복잡한 JOIN을 피하고 싶을 때

AWS에서: MSK 토픽에 커맨드를 씁니다. 컨슈머(Lambda, Kafka Streams 또는 ksqlDB)가 DynamoDB, Aurora 또는 OpenSearch Service의 읽기 모델을 업데이트합니다. 읽기는 이 최적화 저장소에 직접 접근하며 소스는 절대 건드리지 않습니다.

// Java 컨슈머 - DynamoDB 읽기 모델 업데이트
@KafkaListener(topics = "order_events")
public void updateReadModels(OrderEvent event) {
    DashboardView view = new DashboardView();
    view.setOrderId(event.getOrderId());
    view.setAmount(event.getAmount());
    view.setStatus(event.getStatus());
    dynamoDbMapper.save(view);

    // OpenSearch에도 분석용으로 인덱싱
    IndexRequest request = new IndexRequest("orders")
        .id(event.getOrderId())
        .source(Map.of("amount", event.getAmount(), "status", event.getStatus()));
    openSearchClient.index(request);
}

AWS 팁: 다양한 쿼리 패턴을 위해 DynamoDB 글로벌 보조 인덱스를 사용하세요. Lambda 컨슈머의 배치 크기와 동시성 제한을 적절히 설정해 스로틀링을 방지합니다.

3. 이벤트 운반 상태 전이 (Event Carried State Transfer)

무엇인가: 이벤트에 식별자만 담는 것이 아니라 다운스트림 컨슈머가 필요한 완전한 상태 정보를 포함시킵니다. 컨슈머가 소스 서비스에서 추가 데이터를 가져올 필요가 없어 결합도, 네트워크 호출, 지연 시간이 줄어듭니다.

언제 쓰는가:

  • 컨슈머가 이벤트와 함께 동일한 컨텍스트 데이터를 자주 필요로 할 때
  • 서비스를 분리해 프로듀서가 이벤트 계약만 지키면 변경 가능하도록 할 때
  • 추가 네트워크 홉이 중요한 저지연 시스템을 구축할 때
# user_id만 보내는 대신 완전한 상태를 포함
event = {
    "event_type": "UserUpdated",
    "user_id": "123",
    "name": "Alice Chen",
    "email": "alice@example.com",
    "timezone": "America/New_York",
    "preferences": {"notifications_enabled": True, "email_digest": "daily"},
    "last_login_ts": 1700000000
}
producer.send("user_events", key="123", value=json.dumps(event))

# 컨슈머 - 추가 데이터 조회 불필요
def handle_user_update(event):
    send_email(event["email"], "프로필이 업데이트됐습니다")
    if event["preferences"]["notifications_enabled"]:
        send_push_notification(event["user_id"])

AWS 팁: 호환성을 BACKWARD 또는 FORWARD로 설정한 Glue Schema Registry를 사용해 기존 컨슈머를 깨지 않고 이벤트 스키마를 진화시키세요. CloudWatch로 메시지 크기를 모니터링합니다.

4. 클레임 체크 (Claim Check)

무엇인가: 대용량 메시지 페이로드(이미지, 동영상, 대형 JSON)를 S3 같은 외부 저장소에 저장합니다. Kafka 메시지에는 외부 데이터에 대한 참조(클레임 체크)만 담습니다. 컨슈머는 참조를 받아 필요할 때 외부 저장소에서 실제 데이터를 가져옵니다.

언제 쓰는가:

  • 메시지가 Kafka 기본 1MB 제한을 초과할 때
  • 브로커 디스크 I/O와 네트워크 트래픽을 줄이고 싶을 때
  • 페이로드가 크지만 컨슈머의 접근 빈도가 낮을 때
import boto3
from uuid import uuid4

s3 = boto3.client('s3')
s3_key = f"order-attachments/{uuid4()}/invoice.pdf"
s3.upload_file("/tmp/invoice.pdf", "my-kafka-bucket", s3_key)

# 클레임 체크 - S3 참조만 포함하는 소형 메시지
claim_check = {
    "order_id": "123",
    "s3_bucket": "my-kafka-bucket",
    "s3_key": s3_key,
    "size_bytes": 10485760
}
producer.send("order_attachments", key="123", value=json.dumps(claim_check))

AWS 팁: S3 수명 주기 규칙으로 7일 후 객체를 삭제하세요. 컨슈머가 IAM 권한 없이 직접 접근해야 할 때만 사전 서명 URL을 생성합니다.

5. 데드 레터 큐 (Dead Letter Queue, DLQ)

무엇인가: 컨슈머가 모든 재시도 후에도 메시지 처리에 실패하면, 해당 메시지를 잃거나 메인 토픽을 막는 대신 별도의 "데드 레터" 토픽으로 보냅니다. DLQ는 수동 검사나 지연 재생이 필요한 문제 메시지의 격리 공간 역할을 합니다.

언제 쓰는가:

  • 항상 실패를 유발하는 독성 메시지(poison pill)가 있을 때
  • 재시도로 해결할 수 없는 방식으로 다운스트림 의존성이 실패할 때
  • 파이프라인을 멈추지 않고 실패 메시지에 대한 운영 가시성이 필요할 때
def consume_with_dlq():
    dlq_producer = KafkaProducer(bootstrap_servers='msk-broker:9092')
    for msg in consumer:
        try:
            process_message(msg)
            consumer.commit()
        except MalformedDataError as e:
            # 독성 메시지 - DLQ로 보내고 커밋
            dlq_producer.send(f"{msg.topic}.dlq", msg.value, key=msg.key)
            consumer.commit()
            logger.error(f"기형 메시지를 DLQ로 전송: {e}")
        except TransientError as e:
            # 최대 재시도 후 DLQ로
            if retry_count[msg.offset] > MAX_RETRIES:
                dlq_producer.send(f"{msg.topic}.dlq", msg.value)
                consumer.commit()

AWS 팁: Lambda를 컨슈머로 사용하는 경우, Lambda 이벤트 소스 매핑에서 DLQ 설정을 활성화하세요. 복잡한 MSK 컨슈머는 애플리케이션 수준에서 DLQ를 구현합니다.

6. 멱등성 컨슈머 (Idempotent Consumer)

무엇인가: 동일한 메시지를 여러 번 처리해도 중복 부작용(이중 청구, 환영 이메일 두 번 발송, 카운터 이중 증가)이 발생하지 않는 컨슈머입니다. Kafka는 기본적으로 최소 한 번 전달을 보장하기 때문에 이 패턴이 필수적입니다.

언제 쓰는가:

  • 컨슈머가 비멱등 작업(DB 쓰기, 이메일 발송, 결제 API 호출)을 수행할 때
  • Kafka의 정확히 한 번 시맨틱을 사용할 수 없을 때
  • 복잡한 분산 트랜잭션 없이 단순한 멱등 비즈니스 로직을 원할 때

AWS에서: DynamoDB(TTL 포함) 또는 Redis ElastiCache를 멱등성 저장소로 유지합니다. 메시지 키 + 파티션 + 오프셋마다 처리 완료 표시를 저장한 뒤 오프셋을 커밋합니다.

AWS 팁: 가변 처리량을 위해 DynamoDB 온디맨드 모드를 사용하세요. TTL로 재생 윈도우(예: 7일) 이후 오래된 멱등성 레코드를 자동 정리합니다.

7. 트랜잭셔널 아웃박스 (Transactional Outbox)

무엇인가: 이중 쓰기 문제를 해결하는 패턴입니다. DB 업데이트와 Kafka 메시지 발행을 원자적으로 보장합니다. 두 시스템에 동시에 쓰는 대신, DB와 같은 로컬 트랜잭션 안에서 "아웃박스" 테이블에도 함께 씁니다. 별도 프로세스가 아웃박스를 폴링해 Kafka에 발행하고 전송 완료를 표시합니다.

언제 쓰는가:

  • DB와 Kafka 사이에 정확히 한 번 시맨틱이 필요할 때
  • Kafka 트랜잭션을 사용할 수 없을 때
  • 이중 쓰기의 경쟁 조건과 부분 실패를 피하고 싶을 때
# 애플리케이션 - 같은 DB 트랜잭션 안에서
with db.transaction():
    # 비즈니스 데이터
    db.execute("INSERT INTO orders (id, amount, status) VALUES (%s, %s, %s)",
               (order_id, 100, 'created'))
    # 아웃박스 레코드
    outbox_payload = json.dumps({"event_type": "OrderCreated", "order_id": order_id, "amount": 100})
    db.execute("INSERT INTO outbox (id, payload, status, created_at) VALUES (%s, %s, %s, %s)",
               (uuid4(), outbox_payload, 'pending', datetime.utcnow()))

# Lambda 폴러
def lambda_handler(event, context):
    conn = get_db_connection()
    cursor = conn.cursor()
    cursor.execute("SELECT id, payload FROM outbox WHERE status='pending' ORDER BY created_at LIMIT 100")
    rows = cursor.fetchall()
    for row in rows:
        kafka_producer.send('order_events', value=row['payload'])
        cursor.execute("UPDATE outbox SET status='sent', sent_at=NOW() WHERE id=%s", (row['id'],))
    conn.commit()

AWS 팁: 폴링을 피하려면 Debezium(MSK Connect에서 제공)을 사용해 아웃박스 테이블의 변경을 CDC로 직접 Kafka에 스트리밍하세요.

8. 컴팩션 토픽 (Compacted Topic)

무엇인가: cleanup.policy=compact로 설정된 Kafka 토픽입니다. 시간이나 크기로 오래된 메시지를 삭제하는 대신, Kafka가 각 키의 최신 메시지만 보존합니다. 이전 값들은 로그 컴팩션 과정에서 결국 제거됩니다. 이 토픽은 분산되고 복제된 내결함성 키-값 저장소가 됩니다.

언제 쓰는가:

  • 각 엔티티의 최신 상태를 재구성해야 할 때(현재 사용자 프로필, 상품 가격, 설정)
  • 새 컨슈머가 전체 이력을 재생하지 않고 최신 상태로 시작하길 원할 때
  • DB 테이블의 변경 로그를 구축할 때(Debezium CDC가 컴팩션 토픽 사용)
# 이 토픽에서는 키당 최신 값만 중요
producer.send("user_profiles", key="user123", value='{"name":"Alice","theme":"dark"}')
producer.send("user_profiles", key="user123", value='{"name":"Alice","theme":"light"}')
producer.send("user_profiles", key="user123", value='{"name":"Alice","theme":"light","email":"alice@x.com"}')

# 끝에서 읽는 컨슈머는 최신 값만 받음
consumer.assign([TopicPartition("user_profiles", 0)])
consumer.seek_to_end()
msg = consumer.poll()  # {"name":"Alice","theme":"light","email":"alice@x.com"}

AWS 팁: Kafka의 로그-랙 메트릭으로 컴팩션 지연을 모니터링하세요. delete.retention.ms로 툼스톤 메시지 보존 기간을 제어합니다.

9. 파티션 키 / 순서 보장 패턴

무엇인가: Kafka는 단일 파티션 내에서만 순서를 보장합니다. 메시지 키를 설정하면 동일한 키의 모든 메시지가 같은 파티션으로 가서 순서가 유지됩니다. 이는 엔티티별 순서가 중요한 이벤트 소싱 시스템의 기본입니다.

언제 쓰는가:

  • 엔티티(주문, 사용자, 디바이스, 세션)별 엄격한 순서 보장이 필요할 때
  • 엔티티별 순서를 보존하면서 다른 엔티티 간에는 병렬 처리를 원할 때
  • 동일한 키의 이벤트를 순차적으로 처리해야 하는 상태 저장 애플리케이션을 구축할 때
# 같은 order_id의 모든 이벤트는 같은 파티션 → 순서 보장
producer.send("order_events", key="order_123", value=json.dumps({"event": "created", "ts": 100}))
producer.send("order_events", key="order_123", value=json.dumps({"event": "paid", "ts": 200}))
producer.send("order_events", key="order_123", value=json.dumps({"event": "shipped", "ts": 300}))

# 다른 order_id는 다른 파티션 → 병렬 처리
producer.send("order_events", key="order_456", value=json.dumps({"event": "created", "ts": 150}))

AWS 팁: CloudWatch 메트릭(BytesInPerPartition)으로 파티션 불균형을 모니터링하세요. 특정 파티션이 과부하되면 키 솔팅이나 파티션 수 증가를 고려합니다.

10. 스트림-테이블 이중성 (Stream-Table Duality)

무엇인가: Kafka는 스트림(개별 이벤트의 시퀀스)과 테이블(키별 현재 상태) 모두로 볼 수 있습니다. 이 이중성이 Kafka Streams와 ksqlDB의 핵심입니다. 스트림을 집계해 테이블로 만들거나(사용자별 누계), 테이블의 변경 로그를 스트림으로 내보낼 수 있습니다.

언제 쓰는가:

  • 실시간 이벤트 스트림을 느리게 변하는 참조 데이터와 조인해야 할 때(클릭 스트림 + 사용자 프로필 테이블)
  • 새 이벤트가 도착할 때 업데이트되는 구체화 뷰를 유지하고 싶을 때
  • 실시간 대시보드나 모니터링 시스템을 구축할 때
-- 컴팩션 토픽에서 테이블 생성 (사용자별 최신)
CREATE TABLE user_profiles (
    user_id VARCHAR PRIMARY KEY,
    name VARCHAR,
    email VARCHAR
) WITH (KAFKA_TOPIC = 'user_changes', VALUE_FORMAT = 'JSON');

-- 일반 토픽에서 스트림 생성
CREATE STREAM clickstream (
    user_id VARCHAR,
    page VARCHAR,
    timestamp BIGINT
) WITH (KAFKA_TOPIC = 'clicks', VALUE_FORMAT = 'JSON');

-- 스트림-테이블 조인: 클릭에 사용자 프로필 보강
CREATE STREAM enriched_clicks AS
SELECT c.user_id, c.page, c.timestamp, u.name, u.email
FROM clickstream c
JOIN user_profiles u ON c.user_id = u.user_id
EMIT CHANGES;

AWS 팁: 관리형 서비스를 원하면 Confluent Cloud의 ksqlDB를 사용하세요. 자체 관리를 원하면 ECS Fargate에서 Kafka Streams를 실행하고 EBS로 RocksDB 상태 저장소를 백업합니다.

11. 사가 - 코레오그래피 (Saga Choreography)

무엇인가: 각 로컬 트랜잭션이 다른 서비스의 다음 트랜잭션을 트리거하는 이벤트를 발행하는 분산 트랜잭션 패턴입니다. 중앙 코디네이터가 없습니다. 단계가 실패하면 이전 서비스는 실패 이벤트를 수신해 보상 트랜잭션(취소 작업)을 실행합니다. Kafka가 통신 백본 역할을 합니다.

언제 쓰는가:

  • 2단계 커밋(2PC) 없이 마이크로서비스 간 분산 트랜잭션이 필요할 때
  • 서비스가 느슨하게 결합되고 다른 팀이 소유할 때
  • 2PC의 락 보유로는 너무 긴 장기 실행 트랜잭션(분·시간 단위)일 때
# 주문 서비스 - 사가 시작
def create_order(order):
    db.insert(order, status="PENDING")
    producer.send("order_events", value={"event": "OrderCreated", "order_id": order.id})

# 결제 서비스 - 수신 및 보상
def handle_order_created(event):
    try:
        charge_credit_card(event.order_id)
        producer.send("payment_events", value={"event": "PaymentProcessed", "order_id": event.order_id})
    except InsufficientFunds:
        # 보상: 실패 이벤트 발행
        producer.send("payment_events", value={"event": "PaymentFailed", "order_id": event.order_id})
        producer.send("order_events", value={"event": "OrderCancelled", "order_id": event.order_id})

# 주문 서비스 - 보상 핸들러
def handle_order_cancelled(event):
    db.update(event.order_id, status="CANCELLED")
    refund_any_pre_authorizations(event.order_id)

AWS 팁: 코레오그래피에서는 서비스별 MSK 토픽(order_events, payment_events, inventory_events)을 사용하세요. 실패한 사가 단계에 대해 데드 레터 토픽을 모니터링합니다. 분기/병합이 복잡한 사가는 MSK 통합이 포함된 AWS Step Functions 오케스트레이션 패턴을 고려합니다.
