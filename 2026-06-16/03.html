An Interviewer Asked Me Difference Between @Bean vs @Component — Here's the Real Answer

kaurharjeet122 · 3분 · 원문 · freedium
면접관이 "@Bean과 @Component 차이" 질문으로 진짜 원하는 건 암기한 답이 아니라 Spring이 왜 두 방법을 모두 제공하는지 이해하는지 확인하는 것이다.

면접에서 @Bean과 @Component의 차이를 묻자 교과서적 답변을 했더니 면접관이 "그건 구글 답이고, 숙련된 개발자처럼 진짜 차이를 설명해 보라"고 했다. @Component는 Spring이 자동으로 스캔해 등록하는 반면, @Bean은 @Configuration 클래스 안에서 개발자가 직접 객체 생성 방식을 지정한다. 핵심 차이는 소유권과 제어권이다. 내 클래스면 @Component, 서드파티 라이브러리처럼 소스를 수정할 수 없는 클래스라면 @Bean을 써야 한다. 면접관이 정말 평가하는 것은 "Spring이 왜 같은 문제를 해결하는 방법을 두 가지 제공하는가"를 이해하는지 여부다.

"@Bean과 @Component의 차이가 뭐죠?"

나는 표준 답변을 했습니다:

  • @Component는 Spring이 자동으로 감지합니다
  • @Bean은 @Configuration 클래스 안에서 수동으로 선언합니다

면접관은 나를 바라보며 말했습니다:

"그건 구글 답입니다. 이제 숙련된 개발자처럼 진짜 차이를 설명해 보세요."

솔직히 말하면… 그 질문이 Spring을 진짜로 이해하는지, 아니면 애너테이션을 암기만 했는지를 드러내는 질문이었습니다.

진짜 답은 이렇습니다 👇

@Bean vs @Component — 진짜 차이

둘 다 Spring이 관리하는 객체(Bean)를 만드는 데 사용됩니다.

하지만 Spring이 객체를 생성하고 제어하는 방식이 완전히 다릅니다.

@Component

@Component를 사용하면 Spring이 자동으로 스캔해서 객체를 생성합니다.

예시:

@Component
public class PaymentService {
}

컴포넌트 스캔이 활성화되어 있으면, Spring이 이 클래스를 자동으로 찾아 Bean으로 등록합니다.

그래서 이것을 자동 감지(Auto-detection) 방식이라고 부릅니다.

@Bean

@Bean은 설정 클래스(Configuration class) 안에서 사용하며, 개발자가 직접 Spring에게 어떤 객체를 만들지 알려줍니다.

예시:

@Configuration
public class AppConfig {

    @Bean
    public PaymentService paymentService() {
        return new PaymentService();
    }
}

이것은 수동 Bean 생성(Manual bean creation) 방식입니다.

대부분의 개발자가 놓치는 진짜 면접 포인트

면접관이 이어서 물었습니다:

"둘 다 Bean을 만드는데, Spring은 왜 두 가지가 필요한가요?"

그게 진짜 논의의 시작이었습니다.

@Component가 BEST인 경우

@Component를 사용하세요:

  • 내가 소유한 클래스일 때
  • 클래스가 내 애플리케이션에 속할 때
  • 복잡한 객체 생성 로직이 필요 없을 때

다음에 적합합니다:

  • 서비스(Services)
  • 리포지토리(Repositories)
  • 컨트롤러(Controllers)
  • 유틸리티 클래스(Utility classes)

예시:

@Service
@Repository
@Controller

이것들은 모두 @Component의 특수화된 형태입니다.

@Bean이 BEST인 경우

@Bean을 사용하세요:

  • 내가 클래스를 소유하지 않을 때
  • 서드파티 라이브러리 객체를 Spring이 관리하게 해야 할 때
  • 커스텀 초기화가 필요할 때

실제 사례 👇

다음과 같은 서드파티 클래스는 수정할 수 없습니다:

ObjectMapper
RestTemplate
ModelMapper

그래서 직접 생성합니다:

@Bean
public ObjectMapper objectMapper() {
    return new ObjectMapper();
}

이것이 @Bean의 가장 실용적인 사용 사례 중 하나입니다.

프로덕션 레벨의 차이

시니어 개발자들이 보통 알고 있는 것들:

@Component — Spring이 제어:

  • 객체 생성
  • 의존성 주입
  • 생명주기 자동 관리

코드가 적습니다. 개발이 더 깔끔합니다.

@Bean — 개발자가 제어:

  • 객체가 어떻게 생성되는지
  • 생성자 값
  • 설정 로직
  • 서드파티 통합

유연성이 더 높습니다. 제어력이 더 큽니다.

간단한 비유

면접관에게 이렇게 말했습니다:

@Component는 메뉴에서 음식을 바로 주문하는 것과 같습니다.

Spring이 모든 걸 자동으로 준비합니다.

하지만…

@Bean은 주방에 들어가서 원하는 방식으로 직접 요리하는 것과 같습니다.

면접관은 웃으며 말했습니다: "드디어 실용적으로 설명하는 사람이 나왔네요."
