Everything you need to know about Spark Structured Streaming

Vu Trinh · Data Engineer Things · 17분 · 원문 · freedium
Spark Structured Streaming의 아키텍처부터 워터마크, 상태 저장, 장애 허용, 정확히-한-번 보장까지 한 편으로 완벽 정리.

Spark Structured Streaming은 무한히 들어오는 스트림을 유한한 데이터의 연속 조각(마이크로 배치)으로 처리하는 스트림 처리 엔진이다. 트리거 유형과 배치 크기 조정으로 처리 빈도와 지연 시간을 제어하며, 이벤트 타임 처리를 위해 워터마크를 사용해 늦게 도착하는 데이터를 다룬다. 스트리밍 집계·조인·중복 제거 같은 상태형 연산은 HDFS 기반 또는 RocksDB 상태 저장소에 상태를 유지한다. 체크포인팅 메커니즘으로 장애 복구를 지원하며, 소스·처리·싱크 세 계층이 모두 보장할 때만 정확히-한-번 의미론이 성립한다.

저자 Vu Trinh은 vutr.substack.com에서 심층 데이터 엔지니어링 아티클을 발행합니다.

소개

저자는 배치 처리를 위한 Spark 전반을 다룬 글을 이미 썼습니다. 하지만 Spark는 신뢰할 수 있는 스트림 처리 엔진이기도 합니다. 이번 주 글에서는 Spark Structured Streaming과 Apache Flink를 비교하려 했으나, 두 주제 모두 논의해야 할 내용이 많아 Spark의 스트림 처리 엔진만 다루는 전용 글을 먼저 쓰기로 했습니다.

이 글에서는 먼저 아키텍처를 살펴보고, 이벤트 타임 처리, 상태형 처리, 장애 허용이 어떻게 지원되는지 탐구합니다.

참고: 이 글은 윈도잉 같은 스트림 처리 개념을 자세히 다루지 않습니다. 저자의 이전 글 Batch and Stream Processing을 먼저 읽어보길 권장합니다.

아키텍처

Structured Streaming은 Spark SQL 엔진 위에 구축된 스트림 처리 엔진입니다. 핵심 설계 원칙은 연속 스트림을 유한한 데이터의 부분 집합으로 취급하는 것입니다. 이 접근 방식 덕분에 Spark 개발자들은 강력한 배치 엔진을 스트림 처리에 활용할 수 있었습니다.

Spark 클러스터

본론에 앞서 Spark에서 배치 처리 애플리케이션이 어떻게 처리되는지 다시 살펴보면 도움이 됩니다.

Spark 애플리케이션은 다음으로 구성됩니다:

  • 드라이버: 사용자 입력 처리부터 익스큐터에게 태스크를 배분하는 것까지 전체 Spark 애플리케이션을 관리하는 JVM 프로세스입니다.
  • 익스큐터: 드라이버가 할당한 태스크를 실행하고 상태와 결과를 보고하는 프로세스입니다. 각 Spark 애플리케이션은 자체 익스큐터 집합을 가집니다.

구조

Spark 애플리케이션은 드라이버와 익스큐터의 클러스터입니다. 각 애플리케이션은 여러 개의 잡(Job)을 가질 수 있습니다. 잡은 데이터에 적용되는 일련의 변환을 나타내며, 처음부터 끝까지 전체 워크플로우를 포괄합니다.

잡은 액션(Action)에 의해 정의되며, 변환이 파티션 간 데이터 셔플을 요구할 때 서로 다른 스테이지로 분할됩니다. 각 스테이지는 다양한 파티션에서 병렬로 처리를 실행하는 여러 태스크로 나뉩니다.

애플리케이션의 전형적인 생명주기

사용자는 애플리케이션의 처리 로직을 정의합니다. Spark의 모든 기능과 상호작용하는 중앙 게이트웨이 역할을 하는 SparkSession 객체가 반드시 포함되어야 합니다.

사용자가 Spark 애플리케이션을 클러스터 매니저에 제출하고 드라이버 리소스를 요청하면, 클러스터 매니저는 제출을 수락하고 드라이버 프로세스를 시작합니다. 드라이버는 클러스터에 익스큐터를 실행하도록 요청합니다. 사용자가 정의한 로직에 따라 드라이버는 실행 계획을 수립하고 익스큐터에 태스크 스케줄링을 시작합니다. 그런 다음 익스큐터가 물리적으로 태스크를 실행하고 드라이버에 상태를 보냅니다. 모든 태스크가 처리될 때까지 이 과정이 계속됩니다.

Spark Structured Streaming

Spark Structured Streaming에서 스트리밍 애플리케이션을 시작하면 장기 실행 Spark 애플리케이션이 생성됩니다. 드라이버 프로세스는 전체 스트리밍 쿼리 생명주기를 관리하며 지속적으로 활성 상태를 유지합니다. 각 스트림에는 트리거가 있으며, 이 트리거는 Spark가 새 데이터를 확인해야 하는 시점을 정의합니다. 트리거가 발생하면 Spark 엔진은 다음을 수행합니다:

  • 소스에 쿼리합니다(예: Kafka에 "최신 오프셋은 무엇인가?" 질문).
  • 마지막 배치 이후 도착한 새 데이터를 식별합니다(예: Kafka 오프셋 1001~5000).
  • 이 새 데이터 덩어리를 마이크로 배치로 개념화합니다. 내부적으로 Spark는 이 마이크로 배치를 작고 정적인 DataFrame으로 취급합니다.

Spark가 마이크로 배치를 DataFrame으로 정의한 뒤에는 코드에 작성한 모든 변환(select, withColumn, groupBy, filter 등)을 적용합니다. Spark는 각 배치에 대한 잡을 구성합니다. Spark의 변환은 지연(lazy) 실행된다는 점을 기억하세요. 액션이 호출되기 전까지는 실제로 아무것도 실행되지 않습니다. Structured Streaming에서 액션은 싱크에 쓰는 것입니다(예: .writeStream.format("parquet").start("/path/to/sink")).

각 마이크로 배치에서:

  • 전체 쿼리 계획(변환 + 싱크 액션)이 마이크로 배치 DataFrame에 적용됩니다.
  • 논리 및 물리 계획이 생성됩니다.
  • 드라이버가 물리 계획에 기반해 익스큐터에 태스크를 스케줄링합니다.

이 과정은 각 마이크로 배치가 증분적으로 검색됨에 따라 이론상 영원히 계속됩니다. 실제 애플리케이션에서 각 마이크로 배치는 Kafka나 Kinesis 같은 소스에서 여러 파일 또는 오프셋 범위를 포함할 수 있습니다.

트리거

트리거 빈도를 정의하기 위해 사용자는 다음 유형 중에서 선택할 수 있습니다:

  • 기본 트리거(Default Trigger): 명시적 트리거가 지정되지 않습니다. Spark는 최대한 빠르게 쿼리를 실행합니다. 이전 배치가 완료되고 새 데이터가 있으면 새 마이크로 배치가 즉시 시작됩니다. 예약된 간격을 기다리는 시간을 제거합니다.
  • 고정 간격(Fixed-Interval): 쿼리는 데이터가 언제 도착하든 관계없이 고정된 간격으로 마이크로 배치를 실행, 검색, 처리하려고 시도합니다.
    • 배치가 간격보다 빨리 완료되면 Spark는 남은 시간을 기다립니다.
    • 배치가 간격보다 오래 걸리면 현재 배치 완료 후 즉시 다음 배치가 시작됩니다.
  • 일회성(One-Time): 트리거가 유한한 양의 데이터를 처리한 후 스트리밍 쿼리를 중지합니다. 스트리밍 로직을 단일 배치 잡으로 실행하는 데 이상적입니다.
  • Available-Now 마이크로 배치: 일회성 트리거와 유사하지만, 소스에 따라 여러 배치로 데이터를 처리할 수 있습니다.

배치 크기

대부분의 경우 Apache Spark Structured Streaming을 사용할 때 가장 중요한 요소는 배치 크기입니다. 사용자는 단일 마이크로 배치에서 소비되는 데이터의 최대량을 제한하는 스로틀 역할의 옵션을 지정할 수 있습니다.

  • maxOffsetsPerTrigger: Apache Kafka, Kinesis 같은 소스에 대해 하나의 마이크로 배치에서 소비되는 레코드(오프셋)의 최대 수를 제한합니다.
  • maxFilesPerTrigger: Delta Lake, JSON, Parquet 파일 같은 소스에 대해 하나의 마이크로 배치에서 처리되는 파일의 최대 수를 제한합니다.
  • maxBytesPerTrigger: Delta Lake, JSON, Parquet 파일 같은 소스에 대해 하나의 마이크로 배치에서 처리되는 데이터의 최대 총 바이트를 (소프트하게) 제한합니다. 마지막 파일이 남은 한도보다 큰 경우 총 바이트가 한도를 초과할 수 있습니다.

배치 크기를 선택할 때는 트레이드오프가 있습니다:

  • 너무 작은 배치: Spark가 마이크로 배치를 처리할 때 계획 수립부터 익스큐터에 태스크 스케줄링까지 오버헤드가 발생합니다. 배치가 작으면 처리할 배치 수가 늘어나 배치마다 오버헤드 시간이 증가합니다. 또한 소규모 배치는 Spark 클러스터의 가용 리소스를 충분히 활용하지 못해 리소스가 유휴 상태로 남을 수 있습니다.
  • 너무 큰 배치: 지연 시간(latency)이 증가합니다. 모든 레코드는 전체 배치가 처리될 때까지 기다려야 합니다. 또한 대형 배치, 특히 상태형 연산(복잡한 집계나 조인)의 경우 Spark 익스큐터의 메모리에 압력을 줄 수 있습니다.

워터마크, 이벤트 타임 처리, 지연 데이터

이벤트 타임, 처리 타임, 스큐

스트림 처리에서 시간은 중요한 요소입니다. 일부 애플리케이션은 실생활에서 발생하는 순서대로 데이터 레코드를 처리해야 합니다. 예를 들어 멀티플레이어 게임의 실시간 리더보드에서 플레이어 A가 08:45:14.100에 챌린지를 완료하고 플레이어 B가 08:45:15.250에 완료했다면, 처리 엔진은 최종 결과에서 플레이어 A가 플레이어 B보다 높은 순위를 갖도록 해야 합니다.

이로 인해 스트림 처리 시스템에서 일반적으로 두 가지 시간 개념이 등장합니다:

  • 이벤트 타임(Event Time): 데이터 이벤트 자체가 발생한 시간.
  • 처리 타임(Processing Time): 처리 중 임의의 지점에서 이벤트가 관찰되는 시간.

이상적인 세계에서는 이벤트 타임이 처리 타임과 같아 모든 이벤트를 발생 즉시 처리할 수 있습니다. 하지만 네트워크 지연, 기계 고장, 인적 오류로 인해 처리 타임은 항상 이벤트 타임보다 큽니다.

이벤트 타임과 처리 타임의 차이를 시간 스큐(time skew)라고 합니다. 워터마크(Watermark)는 스큐를 측정하고 데이터 완전성을 검증하는 스트리밍 시스템의 도구입니다:

  • 이벤트 타임 윈도우를 언제 트리거할지 결정
  • 이벤트가 얼마나 늦게 도착할 수 있는지 지정

윈도우

윈도우는 sum, count, join 같은 상태형 집계의 검색 공간을 제한하는 메커니즘입니다. 윈도잉은 무한한 스트림을 유한한 부분으로 나눕니다. 보통 시스템은 시간 기반 개념을 사용해 데이터를 윈도우로 구성합니다.

Spark는 세 가지 유형의 윈도우를 지원합니다:

  • 고정(Tumbling): 정적 윈도우 크기로 정의됩니다(예: 시간별 윈도우).
  • 슬라이딩(Sliding): 윈도우 크기와 슬라이드 기간으로 정의됩니다(예: 5분마다 시작하는 30분 윈도우).
  • 세션(Session): 키별 데이터 서브셋에 대한 활동 기간을 캡처합니다. 보통 타임아웃 갭으로 정의됩니다.

Spark의 워터마크

Spark의 워터마크는 이벤트 타임 필드와 임계값으로 정의됩니다. 이 임계값은 데이터가 도착할 수 있는 최대 지연 시간을 지정합니다. 워터마크는 관찰된 최대 이벤트 타임에서 임계값을 빼서 계산되며 이를 W라고 합니다.

이 접근 방식의 장점은 단순성입니다. 사용자는 이벤트 타임 필드와 임계값만 지정하면 됩니다.

단, 몇 가지 단점이 있습니다. 첫째, 마이크로 배치 실행 사이클에 의존하기 때문에 워터마크는 각 배치 끝에서만 진행됩니다. 이로 인해 배치 간격과 워터마크 지연을 합산한 최소 지연이 발생해 실제 밀리초 단위 지연 요구 사항에는 적합하지 않을 수 있습니다. 둘째, 고정 임계값은 이벤트 타임과 처리 타임의 차이가 일정할 때만 효과적입니다.

상태형 처리

상태형(Stateful) 스트리밍 쿼리는 Spark가 마이크로 배치 전반에 걸쳐 올바른 결과를 계산하기 위해 중간 상태(과거 이벤트에 대한 기억)를 유지해야 합니다. 간단히 말해 상태는 최종 결과를 산출하기 위해 새 데이터로 업데이트되는 변수입니다.

상태형 연산

Spark의 상태형 연산에는 스트리밍 집계, dropDuplicates, 스트림-스트림 조인, 사용자 정의 상태형 애플리케이션이 포함됩니다:

  • 스트리밍 집계(Streaming Aggregation): 전체 스트림이나 윈도우에 걸쳐 count, sum, average 같은 집계 값을 지속적으로 계산합니다.
  • 스트림-스트림 조인(Stream-Stream Joins): 두 개의 연속적이고 무한한 데이터 스트림을 조인합니다. 어느 시점에서나 엔진이 데이터셋의 완전한 뷰를 갖지 못하기 때문에 배치-배치 조인보다 구현이 더 어렵습니다. 두 스트림 모두 상태로 버퍼링되며, 워터마크가 버퍼 크기를 제어하는 데도 사용됩니다.
  • dropDuplicates: 고유 키로 식별되는 레코드가 지정된 기간 내에 한 번만 처리되도록 보장합니다.
  • 사용자 정의 상태형 애플리케이션(Arbitrary State): Spark는 사용자가 필요에 따라 커스텀 상태형 연산을 구현할 수 있도록 허용합니다.

상태 저장소

상태는 어딘가에 저장되어야 합니다. 상태 저장소는 Spark 익스큐터가 상태를 읽고 쓸 수 있게 하는 키-값 저장소입니다. Spark Structured Streaming은 두 가지 저장소를 지원합니다:

  • HDFS 기반 상태 저장소(기본값): 처음에는 익스큐터의 메모리에 상태 데이터를 저장합니다. 상태 크기가 커지면 OOM(Out-of-Memory) 오류가 쉽게 발생할 수 있습니다. 상태는 HDFS 호환 시스템의 파일에 트랜잭션 방식으로 영속화됩니다.
  • RocksDB 상태 저장소: Spark 3.2부터 도입된 옵션으로, 유명한 임베디드 C++ 키-값 저장소인 RocksDB를 사용합니다. 상태를 익스큐터의 JVM 메모리 대신 RocksDB 인스턴스의 메모리와 디스크에 저장합니다.

장애 허용

기계는 언제든 고장날 수 있습니다. 장애 허용은 Apache Spark Structured Streaming의 가장 핵심 기능 중 하나입니다. Spark는 체크포인팅을 통해 이 기능을 제공합니다.

체크포인팅

사용자는 스트리밍 애플리케이션에 체크포인트 위치(HDFS 호환 시스템)를 설정할 수 있습니다. Spark는 주기적으로 전체 진행 상태(처리된 소스 오프셋 포함), 스트리밍 쿼리의 "아이덴티티", 상태 정보를 해당 위치에 저장합니다. 이를 통해 스트림 쿼리가 장애 직전 상태로 복구할 수 있습니다.

체크포인트 디렉토리의 핵심 하위 디렉토리:

  • offsets: WAL(Write-Ahead Log) 역할을 합니다. 배치 처리 전에 Spark는 처리하려는 데이터 범위를 여기에 씁니다(예: "배치 10에서 Kafka 오프셋 501~600을 처리할 것임").
  • commits: 커밋 파일을 포함합니다. 파일 9가 있다면 배치 #9가 완전히 처리되고 커밋되었음을 의미합니다. 이것이 완료된 작업의 신뢰 소스입니다.

정확히-한-번 보장

장애 허용은 훌륭합니다. 하지만 일부 비즈니스 운영은 그 이상을 요구합니다. 데이터 손실이나 중복이 허용되지 않는 정확한 의미론이 필요합니다. Spark Structured Streaming은 이를 제공할 수 있습니다:

정확히-한-번 의미론을 종단간으로 제공하는 것이 Structured Streaming 설계의 핵심 목표 중 하나였습니다. 이를 달성하기 위해 Structured Streaming 소스, 싱크, 실행 엔진을 설계해 처리 진행 상태를 정확하게 추적하고 재시작 및/또는 재처리를 통해 모든 장애를 처리할 수 있도록 했습니다. — Apache Spark 공식 문서

핵심은 종단간 파이프라인이 정확히-한-번을 보장하려면 소스, 데이터 변환 프로세스, 싱크 모두가 이 속성을 보장해야 한다는 것입니다:

  • 소스: Kafka의 기본 보장은 At-Least-Once입니다. 따라서 Kafka에서 오는 데이터는 중복될 수 있습니다. 정확히-한-번 전달을 보장하려면 메시지를 중복 제거하는 메커니즘이 필요합니다.
  • 처리: Structured Streaming은 체크포인트 메커니즘 덕분에 데이터 처리가 누락되지 않도록 보장할 수 있습니다. 각 마이크로 배치는 커밋 폴더에 하나의 커밋 파일만 가질 수 있어 같은 데이터가 성공적으로 두 번 처리되지 않도록 합니다.
  • 싱크: Structured Streaming의 결과는 싱크에 두 번 이상 구체화될 수 있습니다. 따라서 싱크는 정확히-한-번을 보장하기 위해 멱등성(idempotent)이어야 합니다. 전체 테이블을 덮어쓰는 것이 좋은 예시입니다.

정확히-한-번 전달은 Spark Structured Streaming 애플리케이션에만 달려 있지 않습니다. 소스와 싱크도 이 속성을 제공할 수 있는지 신중하게 검토해야 합니다.

마무리

이 글에서는 먼저 Spark의 아키텍처를 다시 살펴보고, Spark Structured Streaming의 아키텍처를 탐구했습니다. 배치 크기가 마이크로 배치 접근 방식의 성능에 핵심 요소임을 이해했습니다. 이벤트 타임 처리(워터마크 포함)와 상태형 처리를 살펴보고, 마지막으로 Spark Structured Streaming이 장애 허용과 정확히-한-번 처리를 어떻게 달성하는지 알아봤습니다.

Spark는 널리 채택되어 있고 많은 엔지니어가 이미 경험을 갖고 있기 때문에, Spark Structured Streaming은 준실시간(near-real-time) 요구 사항의 스트림 처리에 매력적인 선택입니다. 하지만 초저지연, 강력한 이벤트 타임 및 상태형 처리가 필요한 경우에는 마이크로 배치 접근 방식과 단순한 워터마크가 요구를 충족하지 못할 수 있습니다. 이러한 경우 Flink 같은 진정한 스트리밍 엔진이 더 나은 선택이며, 트레이드오프는 명확합니다: 새로운 시스템을 이해하는 데 더 많은 시간과 노력이 필요합니다.
