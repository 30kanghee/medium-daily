How Claude Code Actually Works for Traders

AI in Trading · 6분 · 원문 · freedium
Claude Code는 단순한 AI 챗봇이 아니라, 트레이더의 반복 작업을 자동으로 실행하는 에이전트 루프 엔진이다.

많은 트레이더들이 Claude Code를 그냥 채팅창처럼 쓰고 있지만, 그것은 진짜 가치를 놓치는 것이다. Claude Code의 핵심은 "프롬프트 → 컨텍스트 → 도구 → 행동 → 검증 → 반복"으로 이어지는 에이전트 루프다. 트레이더 입장에서 병목은 아이디어가 아니라 그 아이디어를 실제 파일·스크립트·차트·숫자로 만드는 기계적 작업이며, Claude Code는 바로 그 층위를 공격한다. 컨텍스트 창 관리, 권한 설정, 검증 강제 등 실전 사용 원칙을 제대로 이해해야 이 도구의 진짜 위력이 발휘된다.

들어가며

당신은 아마 Claude Code가 일반 챗봇과 다르다는 건 이미 알고 있을 것이다. 그런데도 많은 트레이더와 투자자들이 여전히 잘못 쓰고 있다. Claude를 열고, 질문을 입력하고, 텍스트 답변을 받고, 쓸 만한 내용을 복사한 뒤 탭을 닫는다. 이게 다다. 그게 아니다. 가치는 거기 있지 않다.

Claude Code가 가치 있는 이유는 멋진 설명을 써주기 때문이 아니다. 루프를 실행할 수 있기 때문이다.

프롬프트 → 컨텍스트 → 도구 → 행동 → 검증 → 반복

그 루프가 곧 제품의 전부다. 이걸 이해하고 나면 Claude Code는 "개발자용 AI 채팅"이 아니라 당신이 일반 언어로 제어하는 트레이딩 리서치 엔진처럼 보이기 시작한다.

루프의 구조

간단하게 설명하면 이렇다. Claude Code에 작업을 준다. 예를 들면:

로컬 가격 데이터를 사용해 S&P 500 종목의 일간 모멘텀 스크리너를 만들어라. 6개월 수익률로 종목을 정렬하고, 200일 이동평균 아래 종목은 제외한 뒤, 상위 25개를 CSV로 내보내라.

Claude Code는 그냥 단락으로 답변하지 않는다. 일을 시작한다.

  • 먼저 컨텍스트를 수집한다. 프로젝트의 파일을 읽는다. prices.csv, 기존 Python 스크립트, CLAUDE.md, 폴더 구조, 이전 백테스트 결과, 마지막 실행의 에러 로그.
  • 다음으로 어떤 도구가 필요한지 결정한다. 파일 읽기, 프로젝트 검색, 새 스크립트 작성, 명령어 실행, 오류 함수 수정.
  • 그리고 무슨 일이 생겼는지 확인한다. 스크립트가 실행됐는가? CSV가 생성됐는가? 결과물에 25개 행이 있는가? 랭킹 로직이 당신의 규칙과 일치하는가? 맞으면 멈추고, 아니면 다시 루프를 돈다.

그게 진짜 차이다. 챗봇은 답변을 준다. Claude Code는 디스크에 실제 결과물이 생길 때까지 계속 행동을 취한다. 스크립트, 차트, 정제된 데이터셋, 백테스트 보고서, 대시보드, 트레이딩 저널 업데이트.

트레이더 버전의 에이전트 루프

1. 작업을 정의한다

"내 전략을 분석해줘"가 아니다. 그런 프롬프트는 2022년에나 유행했다. 어떤 입력이 있고, 어떤 규칙이 중요하며, 어떤 결과물을 원하는지 명확히 써라.

data/trades.csv를 읽어라. 승률, 평균 수익, 평균 손실, 기대값, 최대 낙폭, 셋업 유형별 성과를 계산하라. 그 다음 HTML 대시보드를 만들어 /reports/trade_review.html에 저장하라.

이제 Claude는 경계가 있는 작업을 갖게 됐다. 입력: 거래 CSV. 로직: 지표 계산. 출력: HTML 대시보드. 이게 실행 가능한 프롬프트다.

2. Claude가 컨텍스트를 수집한다

Claude Code는 프로젝트 주변을 살펴본다. CSV 헤더를 점검하고, 기존 대시보드 파일을 열고, 어떤 Python 라이브러리가 이미 설치됐는지 확인하고, CLAUDE.md를 읽어 당신의 규칙을 파악한다.

주식 작업 시 수정 종가(adjusted close)를 사용하라.
모든 백테스트에 수수료와 슬리피지를 포함하라.
신호 생성에 미래 데이터를 절대 사용하지 마라.
차트는 /reports에 저장하라.
모든 성과 지표는 퍼센트로 반환하라.

대부분의 트레이더들이 Claude Code를 과소평가하는 지점이 바로 여기다. 프롬프트만이 컨텍스트가 아니다. 당신의 프로젝트가 컨텍스트다. 파일이 컨텍스트다. 명령 출력이 컨텍스트다. 에러가 컨텍스트다. 더 나은 컨텍스트가 들어가면, 더 나은 자동화가 나온다.

3. Claude가 도구를 선택한다

이 지점에서 Claude Code는 채팅 창이기를 멈춘다. 파일을 검사해야 하면 읽기 도구를 쓴다. 함수가 어디 있는지 찾아야 하면 검색한다. 새 백테스트 스크립트를 만들어야 하면 파일을 작성한다. 스크립트를 실행해야 하면 터미널을 사용한다.

"읽기 도구를 쓴 다음 bash를 쓰고 그다음 42번째 줄을 수정해"라고 직접 지시할 필요가 없다. 당신은 결과를 묘사하고, Claude가 도구를 고른다.

왜냐하면 트레이딩 리서치는 대부분 도구 작업이기 때문이다. 데이터 다운로드, 데이터 정제, 컬럼 확인, 스크립트 실행, 에러 수정, 결과 내보내기, 버전 비교, 보고서 업데이트. 2026년에 이걸 수동으로 해야 할 이유가 없다.

4. Claude가 행동을 취한다

일반 AI 채팅이 할 수 없는 부분이 여기다. Claude Code는 프로젝트를 실제로 변경할 수 있다.

  • momentum_screener.py를 만들 수 있다
  • python momentum_screener.py를 실행할 수 있다
  • 에러를 확인할 수 있다
  • import를 수정할 수 있다
  • 다시 실행할 수 있다
  • 출력 파일을 만들 수 있다
  • 출력을 열어 요청한 구조와 일치하는지 확인할 수 있다

투자자/트레이더에게 이게 중요한 이유는, 병목이 보통 아이디어가 아니기 때문이다. 병목은 아이디어와 결과 사이에 있는 모든 기계적 작업이다. 당신은 이미 무엇을 테스트할지 안다. 짜증나는 건 그 아이디어를 파일, 스크립트, 차트, 숫자로 만드는 과정이다. Claude Code는 그 층위를 공격한다. 전략이 아니라, 전략 주변의 작업을.

5. Claude가 검증한다

이게 가장 중요한 부분이다. Claude가 백테스트를 작성하고 검증하지 않는다면 당신에겐 아무것도 없다. 깔끔한 수익 곡선은 증거가 아니다. 초록색 샤프 비율도 증거가 아니다. 수익률 표도 증거가 아니다.

트레이딩에서 검증이란 Claude에게 로직을 테스트하도록 요청하는 것을 의미한다.

  • 신호가 과거 데이터만 사용했는가?
  • 거래 비용이 포함됐는가?
  • 누락된 가격은 처리됐는가?
  • 상장 폐지 종목은 제외됐는가?
  • 배당과 주식 분할이 조정됐는가?
  • 벤치마크가 동일한 방식으로 계산됐는가?
  • 포지션 사이징이 규칙과 일치하는가?
  • 거래 횟수가 타당한가?

실행되는 백테스트로는 부족하다. 검증을 통과한 백테스트가 최소 기준이다. 그러니 프롬프트가 검증을 강제하도록 설계해야 한다.

백테스트를 구축한 후 실행하라. 그 다음 처음 10개 거래를 수동으로 검사해 진입·청산 날짜가 규칙을 따르는지 확인하라. 룩어헤드 편향을 점검하라. 거래 비용을 추가하라. 무언가 실패하면 최종 보고서를 주기 전에 수정하라.

이것이 Claude가 겉만 그럴듯한 쓰레기를 생산하는 걸 막는 방법이다.

컨텍스트 창: Claude의 작업 기억

Claude Code에는 컨텍스트 창이 있다. 현재 볼 수 있고 추론할 수 있는 모든 것이다. 당신의 프롬프트, 그것의 답변, 파일 내용, 명령 출력, 에러, 계획, 이전 결정들.

그런데 창에는 한계가 있다. 트레이딩에서 이게 매우 중요하다. 트레이딩 데이터는 빠르게 커지기 때문이다. 작은 전략 설명은 괜찮다. 3년치 거래 저널도 괜찮다. 하지만 분봉 OHLCV 파일로 가득 찬 폴더를 구조 없이 집어넣으면 세션을 망칠 수 있다.

모든 걸 컨텍스트에 쑤셔넣지 마라. Claude가 필요한 것만 검사하게 하라. 데이터가 어디 있는지 알려줘라. 스키마를 줘라. 샘플 행을 줘라. 집중된 세션을 사용하라. 작업 하나당 세션 하나. "스크리너 구축"은 하나의 세션. "실적 반응 분석"은 또 다른 세션. "포트폴리오 리스크 계산 디버깅"은 또 다른 세션.

컨텍스트가 지저분해지면 품질이 떨어진다. 성격 문제가 아니다. 정보 문제다. 관련 없는 상태가 많으면 잘못된 결정이 늘어난다. 세션이 아직 유용하지만 비대해졌을 때는 /compact를 사용하라. 작업이 바뀌었을 때는 /clear를 사용하라. 깔끔한 컨텍스트가 훨씬 명확한 결과를 만든다.

도구가 핵심이다

대부분의 AI 도구는 여전히 텍스트 입력, 텍스트 출력이다. RSI가 뭔지 묻는 데는 괜찮다. 워크플로를 구축하는 데는 부족하다.

Claude Code에는 도구가 있다. 트레이더에게 중요한 것들은 간단하다.

  • 파일 읽기
  • 파일 쓰기
  • 파일 편집
  • 터미널 명령 실행
  • 프로젝트 검색
  • 필요 시 웹 검색
  • MCP를 통해 외부 시스템 연결

마지막 항목이 진지해지는 지점이다. 시장 데이터에 연결된 Claude Code는 스크린샷을 붙여넣는 Claude Code와 다르다. 거래 저널에 연결된 Claude Code는 기억으로 추측하는 Claude Code와 다르다. 브로커 데이터, 공시, 노트, 로컬 리서치 폴더에 연결된 Claude Code는 작동하는 시스템이 된다. 마법이라서가 아니라, 입력에 접근하고 그에 대해 행동을 취할 수 있기 때문이다.

권한: 어리석게 굴지 마라

Claude Code는 파일을 편집하고 명령을 실행할 수 있다. 그건 강력하다. 그렇기 때문에 권한 관리가 필요하다.

기본적으로 Claude는 위험한 행동 전에 확인을 요청한다. 짜증스럽지만 좋은 일이다. 배우는 중일 때는 승인을 켜두어라. 워크플로를 신뢰하게 되면 안전한 부분은 느슨하게 해라.

  • 샌드박스 프로젝트 내 파일 편집 자동 승인? 괜찮다.
  • 로컬 CSV 파일 읽기? 괜찮다.
  • 알려진 Python 백테스트 스크립트 실행? 보통은 괜찮다.
  • 무작위 패키지 설치, API 키 접근, 검토 없이 실제 브로커 주문 실행? 절대 안 된다.

작업이 워크플로를 손상시킬 수 있을 때는 플랜 모드를 사용하라. 플랜 모드는 Claude가 파일을 검사하고 아무것도 편집하지 않은 채 계획을 제안하게 한다. 트레이딩 시스템에 안성맞춤이다. Claude가 백테스터를 재작성하기 전에 계획을 요청하라. 데이터 파이프라인을 변경하기 전에 계획을 요청하라. 실제 돈과 연결된 무언가에 손대기 전에는 반드시 계획을 요청하라.

빠른 것은 좋다. 통제되지 않은 것은 기계 속도로 숨겨진 에러를 만드는 방법이다.
