Google L5 level Interview: If You Join 10 Tables in an SQL Query, How Would You Optimize

Umesh Kumar Yadav · Javarevisited · 5분 · 원문 · freedium
10개 테이블 조인 쿼리가 30초 넘게 걸린다면? Google L5 면접관이 원하는 체계적 진단과 7가지 최적화 전략을 단계별로 공개합니다.

구글 L5 기술 면접에서 "10개 테이블을 조인하는 SQL 쿼리가 프로덕션에서 30초 넘게 걸린다면 어떻게 최적화하겠는가?"라는 질문이 출제됐다. 단순히 SQL을 잘 쓰는 것을 넘어 체계적인 진단 방법론과 다층적 최적화 기법을 결합하는 능력이 핵심이다. EXPLAIN 실행 계획 분석, 프로파일링으로 병목 구간 특정, 인덱스 추가·조인 순서 조정 같은 SQL 단계 최적화부터 애플리케이션 레이어 분리, 집계 테이블, OLAP 엔진 전환까지 7가지 무기를 제시한다. 진단 없이 쿼리를 수정하는 것은 금물이며, 데이터 특성과 비즈니스 요건에 맞는 기법 조합이 정답이다.

"프로덕션 환경에서 10개 테이블을 조인하는 SQL 쿼리가 30초 이상 걸린다면, 단계별로 어떻게 성능을 진단하고 최적화하겠습니까?"

이 질문의 진짜 난도는 SQL 작성 능력이 아니라, 체계적인 트러블슈팅 방법론과 다층적 최적화 기법을 결합하는 데 있습니다.

1. 병목 지점을 찾는 3단계 진단법

10개 테이블 조인이 포함된 느린 쿼리를 만났을 때, 즉시 쿼리를 수정하는 것은 금물입니다. 다음 진단 절차로 근본 원인을 먼저 파악하세요.

1.1 EXPLAIN으로 실행 계획 분석

EXPLAIN 명령으로 각 테이블의 접근 방식을 확인합니다. 주목해야 할 항목:

  • type 컬럼: ALL(전체 테이블 스캔)이나 index(전체 인덱스 스캔)가 나오면 최적화 대상입니다. 이상적인 값은 ref, eq_ref, const입니다.
  • rows 컬럼: 스캔 예상 행 수로, 기대치보다 훨씬 크다면 인덱스 추가가 필요합니다.
  • Extra 컬럼: Using temporary(임시 테이블)나 Using filesort(파일 정렬)가 보이면 주요 병목입니다.

예시:

EXPLAIN SELECT o.id, o.amount, u.name, p.title
FROM orders o
JOIN users u ON o.user_id = u.id
JOIN products p ON o.product_id = p.id
WHERE o.status = 'PAID';

users 테이블의 type이 ALL이라면, user_id에 인덱스가 없어 전체 스캔이 발생하고 있다는 뜻입니다. 명확한 최적화 대상입니다.

1.2 실제 SQL 실행 시간 분포 확인

SET profiling = 1;
-- 느린 SQL 쿼리 실행
SELECT ...;
SHOW PROFILES;
SHOW PROFILE FOR QUERY 1;

각 단계(예: "sending data", "creating sort index")에서 소요된 시간을 확인하면, I/O 병목인지 CPU/정렬 병목인지 구분할 수 있습니다.

1.3 데이터베이스 파라미터 설정 검토

  • join_buffer_size: 값이 작으면 다중 스캔이 발생할 수 있습니다.
  • tmp_table_size / max_heap_table_size: 부족하면 임시 테이블이 디스크로 내려갑니다.
  • innodb_buffer_pool_size: 자주 접근하는 "핫" 데이터를 메모리에 담을 수 있어야 합니다.

2. 10개 테이블 조인이 느린 이유

MySQL(InnoDB)에서 다중 테이블 조인은 기본적으로 Nested Loop Join 알고리즘을 사용합니다. 드라이빙 테이블의 행을 하나씩 꺼내 연결된 테이블을 순차적으로 탐색하는 방식입니다.

대략적인 시간 복잡도: 드라이빙 테이블 행 수 × 연결 테이블당 인덱스 스캔 비용

드라이빙 테이블에 10만 행이 있고 각 연결 테이블 인덱스 스캔 비용이 1ms라면, 10개 테이블 기준 최악의 경우 10만 × (10 × 1ms) = 1,000초가 될 수 있습니다.

MySQL 8.0.18 이상에서는 특정 조건에서 Hash Join을 활용할 수 있어 성능이 개선되지만, 메모리와 해시 테이블 구성 오버헤드의 제약은 여전히 존재합니다.

3. SQL부터 아키텍처까지: 7가지 최적화 무기

아래 전략들은 비용이 낮고 빠른 SQL 수준 변경에서 시작해 더 높은 노력이 필요한 아키텍처 수준 해법으로 이어집니다.

무기 1: 인덱스 최적화 (가장 빠른 효과)

ON과 WHERE 절에 사용되는 모든 컬럼에 인덱스가 있는지 확인합니다. LEFT JOIN이라면 오른쪽 테이블의 조인 컬럼에 인덱스를 추가하세요. 복합 인덱스는 왼쪽 접두사 원칙을 따라야 합니다.

-- orders 테이블에 복합 인덱스 추가
ALTER TABLE orders ADD INDEX idx_create_user (create_time, user_id);

-- 연관 테이블에 인덱스 추가
ALTER TABLE users ADD INDEX idx_vip (vip_level);
ALTER TABLE categories ADD INDEX idx_status (status);

장점: 단순하고 직접적이며 비침습적입니다.
단점: 인덱스가 과도하면 쓰기 성능이 저하됩니다.
최적 대상: 선택도가 높은(중복 값이 적은) 컬럼

무기 2: JOIN 순서 조정

결과 집합이 가장 작은 테이블이 조인을 주도하게 하세요. 필요하다면 STRAIGHT_JOIN으로 순서를 강제합니다.

-- 작은 테이블(blacklist)이 조인을 주도하도록 강제
SELECT STRAIGHT_JOIN o.*
FROM blacklist b
JOIN orders o ON b.user_id = o.user_id;

무기 3: JOIN 분리 후 애플리케이션 레이어에서 조합

데이터 규모가 중간 수준인 목록 조회 쿼리라면, 여러 개의 단순 쿼리로 나눠 실행한 뒤 Java 등 애플리케이션 코드에서 결과를 합칩니다. 데이터베이스 부하를 줄이고 대형 카르테시안 곱을 피할 수 있습니다. IN 절을 활용한 배치 쿼리(500건 단위 분할)와 맵 자료구조로 효율적인 인메모리 조립이 가능합니다.

장점: DB 부하 감소, 쿼리별 독립 최적화 가능
최적 대상: 메인 테이블 크기가 중간 수준(10만 행 미만)이고 연관 테이블이 많은 경우

무기 4: 임시 테이블 또는 파생 테이블

자주 재사용되는 중간 결과를 미리 구체화합니다.

CREATE TEMPORARY TABLE tmp_user_stat (
    user_id BIGINT PRIMARY KEY,
    total DECIMAL(10,2),
    INDEX(user_id)
) ENGINE=InnoDB;

INSERT INTO tmp_user_stat (user_id, total)
SELECT user_id, SUM(amount) FROM orders GROUP BY user_id;

무기 5: 구체화 뷰(Materialized View) 또는 집계 테이블

고정된 BI 형태의 리포트라면, 스케줄 ETL 작업으로 결과를 사전 계산하여 전용 집계 테이블에 저장합니다. 쿼리는 집계 테이블을 직접 조회하여 밀리초 단위로 응답합니다.

무기 6: OLAP 엔진으로 전환

복잡한 분석 쿼리는 ClickHouse나 Apache Doris 같은 컬럼형 데이터베이스로 이관하세요. 와이드 테이블 및 스타 스키마 조인에 최적화되어 있습니다.

무기 7: 수직 파티셔닝 + 읽기/쓰기 분리

와이드 테이블을 기본 정보(고빈도) 테이블과 확장 정보(대용량 텍스트, 저빈도) 테이블로 분리합니다. 복잡한 읽기 쿼리는 레플리카 데이터베이스로 라우팅합니다.

4. 면접 답변 예시

"먼저 EXPLAIN을 실행해 전체 테이블 스캔, 임시 테이블 사용, 파일 정렬 여부를 확인합니다. 그 다음 적절한 인덱스를 추가하거나 STRAIGHT_JOIN으로 조인 순서를 조정합니다.

SQL 수준 튜닝으로 부족하다면, 여러 개의 단순 쿼리로 나눠 메모리에서 조합하는 애플리케이션 레이어 접근 또는 임시 테이블을 활용한 중간 결과 구체화를 검토합니다.

최적화는 단일 기법이 아니라 데이터 볼륨과 비즈니스 요건에 맞춰 인덱싱·쿼리 재작성·아키텍처 조정을 조합한 결과입니다. performance_schema나 프로파일링 도구로 병목이 I/O인지 CPU인지도 반드시 확인합니다."

마무리

복잡한 다중 테이블 조인 최적화에 만능 해법은 없습니다. 철저한 진단, 데이터 특성 파악, 단기 성과와 장기 유지보수성 사이의 균형이 핵심입니다. 이 다층적 기법들을 익혀두면 가장 까다로운 10개 테이블 쿼리도 효율적이고 확장 가능한 작업으로 바꿀 수 있습니다.
