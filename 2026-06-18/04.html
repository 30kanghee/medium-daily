Apache Spark WTF??? 😈 Vices & Virtues 😇

Ángel Álvarez Pascua · Towards Data Engineering · 23 min read · 원문 · freedium

요약

Apache Spark의 데이터 소스 API 진화를 DSv1의 한계부터 DSv2의 명시적 커넥터 계약, Spark 4.x의 새로운 기능까지 유머러스하게 설명한다.

이 글은 Apache Spark의 데이터 소스 API가 DSv1에서 DSv2로 진화한 과정을 종교적·악마적 은유를 곁들여 설명한다. DSv1은 단순했지만 레이크하우스 시대에 필요한 카탈로그 인식, 세밀한 쓰기 시맨틱, 명시적 능력 협상이 부족했다. DSv2는 컬럼 푸시다운, 집계 푸시다운, 카탈로그 통합 등 더 풍부한 인터페이스를 도입하여 Spark가 외부 시스템과 더 정확한 대화를 나눌 수 있게 한다. Spark 4.x에서는 Python 데이터 소스 API, 네이티브 XML 지원, S3 Magic Committer 기본 활성화 등 추가 개선이 이루어졌다. 그러나 각 기능은 장점만큼 부작용을 동반하므로 커넥터 구현자는 타임존·널·소수점 처리의 시맨틱 일치를 반드시 검증해야 한다.

DSv1은 어깨 위의 작은 악마가 속삭이는 것이다: "그냥 폴더 읽으면 되지."
DSv2는 더 나은 계약을 가진 천사다: 깔끔한 스캔 계획, 풍부한 카탈로그 통합, 열 가지치기, 그리고 커넥터가 실제로 지원할 때의 푸시다운.

왜냐하면 Spark에서 구원은 약속되지 않기 때문이다. 푸시다운될 뿐이다.

"No te vamos a enseñar nada que no sepas / Ni nada que no quieras aprender"
(Doble V의 "Vicios y Virtudes"에서 각색)

🤥 아주 그럴듯한 거짓말

데이터 엔지니어링에서 가장 예쁜 거짓말은 이것이다: spark.read.parquet("/path").

Spark가 안경을 고쳐 쓰고 파일을 공손히 열기 시작하는 것처럼 보인다. 하지만 현실은 그렇게 단순하지 않다.

Spark는 신뢰 문제가 있는 분산 엔진이다. DataFrame은 "메모리에 로드된 데이터"가 아니다. 느긋한 약속이다. 액션이 나타나 "자, 이제 돈을 써봅시다"라고 말하기 전까지는 아무 일도 일어나지 않는다.

그 게으름은 중요하다. Spark에게 열 가지치기, 필터 푸시다운을 수행하고 클라우드 청구서가 중세 고문 도구로 변하지 않도록 막을 시간을 준다.

V1 악마의 등장

그러고 나서 오래된 악마가 등장한다: Data Source V1. DSv1은 단순하고, 유용하고, 아주 2010년대스러웠다.

Spark가 소스에게 물었다: 행을 줄 수 있어?
소스가 대답했다: 물론이죠, 보스.

내부적으로 커넥터는 보통 BaseRelation을 노출하고, 참회 수준을 선택했다.

세 가지 고해 모드

  • TableScan은 모든 것을 읽고 나중에 고해했다.
  • PrunedScan은 적어도 불필요한 열을 건너뛰었다.
  • PrunedFilteredScan은 일부 필터도 푸시했지만, Spark는 프로덕션이 원래 신뢰 문제가 있기 때문에 나중에 다시 확인할 수 있었다.

V1이 땀을 흘리기 시작한 곳

이것은 "데이터 소스"가 주로 "경로에 있는 일부 파일"을 의미했을 때는 잘 작동했다.

하지만 현대 레이크하우스 테이블은 야망을 가진 폴더가 아니다. 카탈로그, 스냅샷, 파티션 규칙, 쓰기 시맨틱, 삭제, 업데이트, 그리고 점심 5분 전에 "빠른 덮어쓰기"를 말하는 사용자가 있다.

바로 그곳에서 V1이 땀을 흘리기 시작했다.

계약이 너무 모호했다. 능력 협상이 어색했다. 카탈로그 인식이 약했다. 쓰기가 원래 편안한 영역을 벗어나 확장되었다. 그리고 모든 진지한 커넥터는 너무 적은 API 뒤에 너무 많은 영리함을 숨겨야 했다.

🎬 2편은 항상 별로였나... 정말?

기술에서 위험한 문구가 있다: 우리는 V2를 만들고 있습니다.
방 안의 모든 엔지니어는 즉시 7년을 더 먹는다.

소프트웨어 속편은 나쁜 평판을 가지고 있다. Jaws 2. Speed 2. 회사의 두 번째 "통합 데이터 플랫폼" 시도. 모두 새로운 악당, 더 나쁜 설정, 그리고 세금 양식처럼 이름 붙여진 인터페이스를 도입한다.

그래서 Spark가 Data Source V2를 이야기에 가져왔을 때 의심은 건강했다. 하지만 이 속편은 필요했다.

세상이 변했다. 커넥터는 더 이상 "행을 줄 수 있어?"에만 답하지 않았다. 카탈로그, 스키마, 파티션, 푸시다운, 컬럼형 읽기, 배치 쓰기, 스트리밍, 테이블 기능에 대해 질문받고 있었다.

V1은 나쁘지 않았다. 격식을 갖추지 못했을 뿐이다. 레이크하우스 결혼식에 슬리퍼를 신고 나타났다.

V2의 세 가지 저주

DSv2는 Spark 2.3.0에서 실험적으로 처음 등장했으며 그 이후로 진화해 왔다. 세 가지 저주를 견뎌야 했다.

  • 저주 하나: 오래된 고통 고치기. V1은 너무 많은 것을 암묵적으로 만들었다: 무엇을 푸시다운할 수 있는지, 무엇을 가지치기할 수 있는지, 테이블이 무엇을 할 수 있는지, 쓰기가 어떻게 동작해야 하는지.
  • 저주 둘: 왕국을 부수지 않기. Spark는 V1을 삭제하고 지구 절반에게 "행운을 빌어요, 죄인들"이라고 말할 수 없었다. 너무 많은 프로덕션 작업, 커넥터, 레거시 설정이 그것에 의존했다.
  • 저주 셋: 레이크하우스 미래 예측하기. 테이블은 순수한 경로이기를 멈췄다. 카탈로그, 스냅샷, 트랜잭션, 파티션 스펙, 거버넌스 규칙, 그리고 후드티를 입은 화난 비즈니스 기대가 되었다.

Catalyst와의 취업 면접

DSv2는 Spark와 외부 시스템 간의 대화를 바꾼다. 하나의 모호한 커넥터 계약 대신 DSv2는 org.apache.spark.sql.connector 아래 더 작은 인터페이스를 사용한다.

커넥터는 지원하는 것을 명시적으로 말할 수 있다: 배치 읽기, 컬럼형 읽기, 필터 푸시다운, 집계 푸시다운, 제한 푸시다운, 카탈로그 작업, 테이블 기능.

이것이 중요한 이유는 Spark가 커넥터가 표현할 수 없는 것을 최적화할 수 없기 때문이다. 커넥터가 "그냥 모든 것 주세요, 나중에 필터링할게요"라고 말하면 Catalyst는 기적을 위한 공간이 제한적이다.

쓰기, 배치, 그리고 성인의 감독

DSv2는 Spark에 더 구조화된 쓰기 경로를 제공한다: 작성자 팩토리, 커밋 메시지, 드라이버 측 커밋, 중단 훅. 이것이 모든 커넥터를 마법처럼 트랜잭션 가능하게 만들지는 않지만, 진지한 테이블 형식에 진지한 동작을 구현할 더 깔끔한 프로토콜을 제공한다.

성능에도 같은 아이디어가 적용된다. DSv2는 커넥터가 ColumnarBatch를 통해 컬럼형 읽기에 옵트인할 수 있게 한다. 소스가 지원할 때 Spark는 벡터화된 실행 경로에 더 가깝게 유지되고 불필요한 객체 형태의 고통을 피할 수 있다.

카탈로그 지원을 추가하면 Spark는 "이 의심스러운 경로 읽기"에서 "기능이 있는 이 명명된 테이블에서 작업하기"로 졸업한다.

🚀 Spark 4.x: 새로운 기능과 새로운 악마들

Spark 4.x 시대에 이르면 DSv2는 더 이상 실험적 미리보기가 아니라 실행 엔진의 도전받지 않는 군주다. 하지만 최고 권력에는 최고의 유혹이 따르며, Spark 4.0은 신앙을 시험하는 전혀 새로운 마이그레이션 악마 군단을 풀어놓는다.

Python이 커넥터 대성당에 들어오다

Spark 4.0은 커스텀 PySpark 소스를 가능하게 했다.
Spark 4.1은 필터 푸시다운으로 더 날카롭게 만들었다: Python 커넥터는 행이 Spark로 넘어오기 전에 더 일찍 프레디케이트를 받아 더 적은 데이터를 읽을 수 있다.

😇 장점: 데이터 이동 감소, Python/JVM 경계 고통 감소, "왜 이 API 리더가 전체 인터넷을 가져왔지?" 순간 감소.

😈 단점: 푸시다운은 약속이다. Python 소스가 날짜, 널, 소수점, 타임존을 잘못 처리하면 더 빠르게 틀려진다 — 그래도 여전히 틀린 것이다.

네이티브 XML과 프레임워크 강화

Spark 4.0은 XML을 공식 내장 데이터 소스로 승격하여 구시대의 불안정한 외부 패키지 의존성을 제거한다. 동시에 광범위한 DSv2 프레임워크는 대대적인 엔진 수준 강화를 받는다:

  • 스토리지 파티션 조인: V2 소스의 네이티브 데이터 분포를 직접 활용하여 비용이 많이 드는 네트워크 셔플을 제거한다.
  • 집계 및 백분위수 푸시다운: 고급 소스 협업을 허용하여 백분위수 계산 같은 집약적 함수를 바이트가 네트워크에 닿기 전에 스토리지 레이어에서 직접 실행한다.
  • 카탈로그 프로시저: 명시적 저장 프로시저 카탈로그 API와 구조적 메타데이터 쿼리를 위한 네이티브 지원을 도입한다.

마이그레이션 고블린: 스키마와 기본값 변화

Spark 4.x는 더 나은 최적화를 가져오지만, 더 날카로운 마이그레이션 함정도 가져온다.

JDBC가 더 투명해졌다 — 그리고 더 엄격해졌다. Spark 4.0은 EXPLAIN에서 원격 데이터베이스로 전송되는 원시 SQL을 보여줄 수 있어 데이터베이스가 왜 우는지 디버깅하기 좋다. 하지만 JDBC 타입 매핑도 더 엄격해졌다. 느슨한 다운스트림 스키마는 갑자기 종교를 찾은 것처럼 실패하기 시작할 수 있다.

ANSI 모드가 이제 기본값이다. 잘못된 캐스트, 오버플로, 파싱 오류가 조용한 널이 될 가능성이 낮아졌다. 정확성에는 좋지만, 레거시 파이프라인에는 고통스럽다.

S3 커밋 동작이 Spark 4.1에서 변경되었다. Hadoop S3A Magic Committer가 S3 버킷의 기본값으로 활성화되었다. 이름 변경 기반 쓰기 비용을 줄일 수 있지만, 커밋 메커니즘은 가볍게 볼 수 없다. 감정적 깊이를 가진 고아 파일을 좋아하지 않는다면 테이블 형식, 재시도, 중단, 실패 경로를 테스트하라.

🛠️ 장점 지도: DSv2가 고치려는 것

이제 악마가 고해했다: DSv1은 유용하고, 실전에서 검증되었으며, 레이크하우스 시대에는 너무 모호했다.

DSv2가 모든 것을 고치지는 않는다. 저주받은 커넥터를 위한 스파 주말이 아니다. 주로 Spark와 외부 시스템 간의 대화를 고친다: 무엇을 푸시하고, 가지치기하고, 쓰고, 신뢰하고, 실제 테이블로 취급할 수 있는지.

분산 데이터 재앙의 대부분은 악의로 시작하지 않기 때문이다. 모호함으로 시작한다.

명시적 기능: 더 이상 추측 게임 없음

DSv1은 종종 별자리 특성을 통해 커넥터의 성격을 읽는 것 같았다.

DSv2는 더 직접적이다. 소스는 지원하는 것을 명시적으로 노출할 수 있다: 읽기, 쓰기, 카탈로그 작업, 필터 푸시다운, 집계 푸시다운, 제한 푸시다운, 컬럼형 읽기, 파티션 보고 등.

😇 장점: Catalyst와 더 나은 대화.

😈 단점: 기능은 약속이다. 커넥터가 "네, 그 필터를 푸시했어요"라고 말하지만 실제로 Spark와 다르게 널, 타임존, 소수점, 문자열을 평가한다면 — 쿼리를 최적화한 것이 아니다. 데이터에서 진실을 최적화한 것이다.

테이블 지향 쓰기: 경로에서 실제 테이블로

예전 Spark 쓰기는 종종 경로 모양의 혼돈이었다: 이 DataFrame을 저기에 저장하세요.

DSv2는 DataFrameWriterV2와 카탈로그 인식 API를 통해 Spark 쓰기를 추진한다. 맹목적으로 경로에 파일을 떨어뜨리는 대신, Spark는 명명된 테이블을 대상으로 하고 추가, 생성, 교체, 조건부 덮어쓰기, 동적 파티션 덮어쓰기 같은 더 명확한 작업을 사용할 수 있다.

😇 장점: 의도가 더 명시적이 된다. 하지만 명시적이 무적을 의미하지는 않는다. 커넥터는 여전히 분산 쓰기를 올바르게 구현해야 한다.

😈 단점: DSv2는 더 나은 쓰기 기계를 제공한다. 기계를 테스트할 필요성을 없애지는 않는다.

컬럼형 실행: 행 모양 고통 덜기

DSv2는 커넥터가 지원할 때 ColumnarBatch 데이터를 사용하여 컬럼형 읽기에 참여할 수 있게 한다. Spark를 벡터화된 실행 경로에 더 가깝게 유지하고 불필요한 객체 형태의 고통을 줄일 수 있다.

😇 장점: 불필요한 행 변환 감소. 더 나은 CPU 동작. 현대 컬럼형 형식과의 더 깔끔한 통합.

😈 단점: 컬럼형 경로는 더 용서하지 않는다. 메모리 수명 주기 버그, 오프힙 압력, 과도하게 큰 배치, 잘못 관리된 리더는 성능 엔지니어링을 귀신 든 탈출 방으로 바꿀 수 있다.

DSv2는 커넥터 수명 주기를 더 구조화한다.

             ⤷ InputPartition
                ⤷ PartitionReaderFactory
                   ⤷ PartitionReader

드라이버가 작업을 계획하고, 익스큐터가 작업을 수행한다.

그 분리가 중요하다. 드라이버는 계획을 수립하고, 푸시다운할 수 있는 것을 푸시다운하고, 가벼운 파티션 설명을 만들어야 한다. 익스큐터는 실제 리더를 만들고 외부 시스템에 접근해야 한다.

😇 장점: 계획과 실행 간의 더 깔끔한 분리.

😈 단점: 무거운 클라이언트, 열린 소켓, 직렬화할 수 없는 괴물을 그 경계의 잘못된 쪽에 두면 Spark는 분산 시스템이 감정을 가지고 있다는 것을 상기시켜 줄 것이다. 대부분 부정적인 감정.

🏢 Data Source V2 아키텍처: 천사의 관료주의

DSv2는 Spark의 천사 관료주의다: 많은 작은 인터페이스, 각각 할 일, 클립보드, 그리고 모호함에 대한 깊은 두려움을 가지고 있다.

요점은 우아함을 위한 우아함이 아니다. 요점은 생존이다.

👑 황금 규칙: 드라이버가 계획한다. 익스큐터가 읽는다. 이것을 섞으면 악마가 루트 접근권을 얻는다.

드라이버 측 체인

드라이버에서 Spark는 실제 분산 읽기가 시작되기 전에 계획을 협상한다.

TableProvider는 안내데스크다. 사용자 옵션과 형식 문자열을 Spark가 이해할 수 있는 소스로 매핑한다.

😈 단점: 안내데스크를 지게차로 만들지 마라. 여기서 무거운 목록 작성, 네트워크 호출, 비용이 많이 드는 추론은 작업이 시작되기도 전에 전체 작업을 느리게 할 수 있다.

Table은 논리적 자산이다: 테이블, 스트림, 디렉토리 기반 소스, 또는 외부 시스템. 스키마와 기능을 노출한다.

😈 단점: 기능은 증언이다. 여기서 거짓말하면 계획이 팬픽션이 된다.

ScanBuilder와 Scan은 협상 방이다. 여기서 Spark는 스캔이 확정되기 전에 필터를 푸시하고, 열을 가지치기하고, 제한을 적용하거나 다른 최적화를 요청할 수 있다.

😈 단점: 푸시다운은 소스 시맨틱이 Spark의 기대와 일치할 때만 안전하다.

Batch는 프로젝트 매니저다. planInputPartitions()를 통해 스캔을 병렬 작업으로 바꾸고 리더 팩토리를 제공한다.

😈 단점: 하나의 거대한 파티션은 클러스터를 하나의 슬픈 노트북으로 만든다. 수백만 개의 작은 파티션은 드라이버를 에스프레소를 마신 항공교통관제사로 만든다.

익스큐터 측 체인

계획 후 Spark는 가벼운 작업 설명을 익스큐터에 전송한다.

InputPartition은 작업 티켓이다. 데이터 자체가 아닌 어느 조각을 읽을지 설명해야 한다.

😈 단점: 열린 소켓, 라이브 클라이언트, 직렬화할 수 없는 짐승을 내부에 넣으면 Spark가 의자를 던질 것이다.

PartitionReaderFactory는 도구 상자다. Spark는 이것을 익스큐터에 직렬화하여 태스크가 로컬에서 리더를 만들 수 있게 한다.

😈 단점: 무거운 클라이언트를 너무 일찍 초기화하면 드라이버 측 상태를 네트워크를 통해 가방 속 너구리처럼 밀수할 수 있다.

PartitionReader는 작업자다. 태스크 내부에서 실행되고, next()로 진행하고, get()으로 데이터를 반환하고, close()로 정리한다.

😈 단점: 재시도가 발생한다. 리더는 실패, 정리, 부작용을 신중하게 처리해야 한다. 그렇지 않으면 커넥터가 뱃지를 단 중복 생성기가 된다.

행, 배치, 그리고 로컬 단축키

DSv2는 다양한 물리적 형태로 데이터를 읽을 수 있다.

행 기반 읽기는 InternalRow를 방출한다. 신뢰할 수 있고, 친숙하며, 때로는 완벽히 괜찮다.

컬럼형 읽기는 ColumnarBatch를 방출할 수 있어, 커넥터가 지원할 때 Spark가 벡터화된 실행에 더 가깝게 유지된다.

😈 단점: 컬럼형 코드는 빠르고, 날카롭고, 용서가 없다. 배치 수명 주기 버그와 메모리 압력은 "고성능"을 "왜 익스큐터가 사라졌지?"로 바꿀 수 있다.

로컬 스캔도 있는데, Spark가 작은 로컬 결과를 위해 분산 스케줄링을 피할 수 있다.

😈 단점: "로컬"을 "무료"와 혼동하지 마라. 예상치 못하게 커지면 드라이버가 희생양이 된다.

실용적인 냄새 테스트

df.explain("formatted")로 커넥터를 고해시켜라.

  • 큰 쿼리가 하나의 스캔 태스크를 만든다면, 파티션 계획이 무너진 것이다. 클러스터는 이제 장식용 가구다.
  • 좁은 쿼리가 여전히 전체 스키마를 읽는다면, 열 가지치기가 실패한 것이다. 넓은 스키마 세금을 내고 있다.
  • 필터링된 쿼리가 푸시된 프레디케이트를 보여주지 않는다면, Spark가 소스가 더 일찍 했어야 할 정리 작업을 하고 있다.
  • 재시도가 Spark 외부에서 중복을 만든다면, 리더나 작성자가 안전하게 반복할 수 없는 부작용을 가지고 있다. 그것은 재시도 문제가 아니다. 정확성 범죄 현장이다.

📉 푸시다운: 클러스터를 구하는 미덕

"푸시다운"은 데이터 엔지니어들이 중세 사제가 전장을 축복하는 것처럼 자신감 있게 말하는 단어 중 하나다. 모두가 고개를 끄덕인다.

그러고 나서 쿼리가 열두 행을 반환하기 위해 48TB를 스캔하고, 갑자기 팀 전체가 EXPLAIN FORMATTED에 깊이 영적이 된다.

아이디어는 단순하다: 데이터에 더 가까이서 작업하라.
위험한 부분은: 푸시다운은 Spark의 쿼리 의미의 일부를 다른 시스템에 위임한다.

😇 장점: 두 시스템이 동의하면, I/O, CPU, 메모리, 그리고 월별 예산 검토를 절약할 수 있다.

😈 단점: 동의하지 않으면, 엄청난 속도로 틀린 답을 얻는다.

최적화 대화

Spark는 커넥터에 무작위로 소원을 던지지 않는다. DSv2에서 Catalyst는 명시적 인터페이스를 통해 소스와 대화한다. 커넥터가 지원하는 것에 따라 Spark는 필터를 푸시하고, 열을 가지치기하고, 집계를 푸시하고, 제한을 적용하고, 통계를 보고하거나 런타임 필터를 사용할 수 있다.

소스는 무엇을 수락했는지, 무엇을 거부했는지, Spark가 나중에 여전히 평가해야 하는 것을 명확히 말해야 한다.

😇 좋은 커넥터: "이 프레디케이트를 처리했습니다. 저것은 안전하게 처리할 수 없었습니다. 스캔 후 이것을 다시 확인해 주세요."

😈 나쁜 커넥터: "믿으세요."

가지치기 vs. 푸시다운: 주문 혼동 멈추기

모든 최적화가 같은 주문이 아니다.

  • 열 가지치기는 너비를 줄인다. Spark는 쿼리에 필요한 열만 요청한다.
  • 파티션 가지치기는 메타데이터를 사용하여 전체 파티션이나 파일을 건너뛴다.
  • 프레디케이트 푸시다운은 필터를 소스에 더 가까이 보낸다.
  • 집계 푸시다운은 소스가 카운트, 최솟값, 최댓값, 부분 집계 등을 계산하게 한다.
  • 제한 및 Top-N 푸시다운은 소스에 일찍 멈추거나 가장 잘 맞는 행만 반환하도록 요청한다.

같은 가족. 다른 악마. 모든 것을 "푸시다운"이라고 부르는 것은 회의를 더 싸게 만들고 장애를 더 비싸게 만드는 방법이다.

열 가지치기: 너비 다이어트

열 가지치기는 지루하게 들린다. 그래서 아름답다.

500열 테이블에서 두 열이 필요하다면, Spark는 하나의 의자를 사용하기 위해 익스큐터에 전체 가구점을 끌어들여서는 안 된다.

DSv2에서 커넥터는 필요한 열 푸시다운을 구현하여 스캔이 필요한 스키마만 읽을 수 있다.

😇 장점: I/O 감소. 디코딩 감소. 네트워크 트래픽 감소. 익스큐터 메모리 드라마 감소.

😈 단점: 중첩 스키마는 자신감이 죽는 곳이다. 최상위 가지치기는 쉽다. 깊은 구조체, 대소문자 민감성, 필드 순서, 스키마 진화, 중첩 프로젝션이 진짜 시험이다.

프레디케이트 푸시다운: 빠르고, 날카롭고, 의심스럽다

프레디케이트 푸시다운은 Spark가 더 적은 데이터를 읽도록 필터를 소스로 보낸다.

JDBC의 경우 WHERE 절이 될 수 있다. 파일과 레이크하우스 테이블의 경우 메타데이터, 파일 통계, 파티션 값, 또는 데이터 스킵 구조를 사용할 수 있다.

😇 장점: Spark가 땀을 흘리기 전에 대규모 데이터 감소.

😈 단점: 소스는 Spark가 의미하는 것과 같은 것을 의미해야 한다. 널, 소수점, 타임존, 문자열 비교, 대소문자 민감성, NaN, 타임스탬프 경계. 소스가 정밀도나 널 논리를 다르게 해석한다면, 쿼리는 더 빨라지지 않았다. 창의적으로 틀렸다.

👍 좋은 소식: DSv2는 부분 푸시다운을 지원한다. 커넥터는 이해하는 프레디케이트를 수락하고 위험한 것은 스캔 후 Spark가 다시 확인하도록 남길 수 있다.

집계, 제한, Top-N 푸시다운

집계 푸시다운. 집계 푸시다운은 소스에 Spark가 원시 행을 읽기 전에 요약을 계산하도록 요청한다. 테이블 형식은 메타데이터에서 count(*)에 답할 수 있다. 파일 인덱스는 최솟값/최댓값 통계를 알 수 있다. 데이터베이스는 Spark가 네트워크를 통해 모든 것을 읽는 것보다 훨씬 효율적으로 집계를 계산할 수 있다.

😈 단점: 수학은 의견이 있다. 소수점, 부동소수점, 널, NaN, 그룹화 동작, 오버플로 처리, 부분 집계 시맨틱이 Spark의 기대와 일치해야 한다.

제한 푸시다운. 제한 푸시다운은 소스에 일찍 멈추도록 요청한다. Spark가 100개 행만 필요하다면, 커넥터는 1억 개 읽기를 피할 수 있다.

😈 단점: 순서 없는 LIMIT은 안정적인 행의 약속이 아니다. "일부 주세요"다. 미리보기에는 괜찮을 수 있다. 누군가가 그 행이 결정론적이거나 대표적이라고 생각할 때는 괜찮지 않다.

Top-N 푸시다운. Top-N 푸시다운은 정렬이 방에 들어온 후의 제한 푸시다운이다. "일부 10개 행"을 요청하는 대신 Spark는 "이 순서에 따라 최상위 10개 행"을 요청한다.

😈 단점: 순서 시맨틱이 정확히 일치해야 한다. 널 순서, 문자열 비교, 대소문자 민감성, 콜레이션, 타임스탬프 처리, 동점 처리가 모두 결과를 바꿀 수 있다.

런타임 필터, 통계, Python 소스

  • 런타임 필터링은 Spark가 실행 중에, 종종 조인에서, 필터를 추론하고 이미 계획된 V2 입력 파티션을 줄이는 데 사용할 수 있게 한다.
  • 통계 보고는 소스가 푸시다운 후 스캔 크기와 행 수를 추정할 수 있게 한다. 좋은 통계는 Spark가 더 나은 조인을 선택하도록 돕는다. 환상적인 통계는 Spark가 자신 있게 재앙을 선택하도록 돕는다.
  • Python 데이터 소스 필터 푸시다운이 Spark 4.1에서 도착한다. Python 우선 커스텀 소스는 이제 프레디케이트 푸시다운에 참여하고 데이터 이동을 줄일 수 있다.

프로덕션 죄악의 박물관

푸시다운 실패는 보통 네 가지 저주받은 복장 중 하나를 입고 온다.

  • 조용한 세금. 커넥터가 아무것도 푸시하지 않는다. 결과는 정확하지만 Spark는 우주를 읽고 청구서는 시를 쓰기 시작한다.
  • 연극 아이. 계획이 푸시다운이 발생했다고 주장하지만 커넥터는 여전히 모든 것을 읽는다. UI는 최적화되어 보인다. 스토리지 레이어는 울고 있다.
  • 스테이징 천사, 프로덕션 고블린. 푸시다운이 스테이징에서 작동한다. 프로덕션에서 하나의 드리프트된 설정이 기적을 전체 스캔으로 바꾼다.
  • 빠른 방해자. 커넥터가 진정으로 이해하지 못하는 것을 푸시한다: 타임존, 널, 소수점, 문자열, NaN, 반올림. 쿼리가 즉시 완료된다. 대시보드가 업데이트된다. 비즈니스가 축하한다. 데이터에는 귀신이 들었다.

✍️ V2로 쓰기: 클립보드를 든 악마

DSv2 쓰기 경로는 조정과 실행을 분리한다.

Table
 ⤷ WriteBuilder
    ⤷ Write
       ⤷ BatchWrite
          ⤷ DataWriterFactory
             ⤷ DataWriter
                ⤷ WriterCommitMessage

드라이버가 계획하고 조정한다. 익스큐터가 쓴다.

각 태스크는 DataWriter를 받고, 레코드를 쓰고, 로컬로 커밋하고, 작은 WriterCommitMessage를 드라이버로 돌려보낸다.

모든 태스크가 성공하면 드라이버가 전역 커밋을 호출한다. 뭔가 실패하면 Spark가 중단을 호출한다.

DSv2는 프로토콜을 제공한다. 불멸을 제공하지는 않는다.

세 가지 쓰기 고블린

1. 추가 함정. 추가는 무해하게 들린다. 그렇지 않다. 재시도, 투기적 실행, 불안정한 네트워크, 외부 부작용은 작성자가 멱등성이 없으면 데이터를 중복시킬 수 있다.

2. 덮어쓰기 갭. overwrite(condition)은 훌륭하게 명확하다. 이 조건과 일치하는 행만 교체하라. 하지만 테이블은 메타데이터 레이어에서 삭제-더하기를 원자적으로 만들어야 한다. 오래된 데이터가 사라지기 전에 새 데이터가 보이게 된다면, 독자는 데이터 갭에 빠질 수 있다.

3. 동적 파티션 전기톱. overwritePartitions()는 일일 재로드에 훌륭하다. 들어오는 행이 닿은 파티션만 교체한다. 하지만 들어오는 데이터가 저주받은 파티션 키를 포함한다면, Spark는 순종하여 잘못된 왕국을 만들거나 교체할 것이다. 동적 파티션 덮어쓰기는 유용하다. 캘린더 지원이 있는 전기톱이기도 하다.

🗂️ 카탈로그: 경로가 기관이 되는 곳

스토리지 경로는 바이트가 어디 있는지 알려준다. 귀엽지만 거버넌스에는 쓸모없다. 소유권, 스키마, 네임스페이스, 테이블 속성을 모른다.

플랫폼이 성장함에 따라 경로는 귀신 든 주소가 된다. 카탈로그는 "저기 어딘가의 일부 파일"을 명명된, 거버넌스된, 검색 가능한 자산으로 바꾼다.

경로 문자열에서 명명된 자산으로

대신 Spark는 구성된 카탈로그 플러그인에 이름을 해결하도록 요청한다.

spark.sql.catalog.prod = com.company.ProdCatalog

DSv2로의 두 가지 문

DSv2에는 두 개의 중요한 입문 문이 있다.

TableProvider는 선술집 악수다. 실제 카탈로그가 없는 V2 소스에 사용된다. 형식/옵션 기반 접근에 잘 작동한다: 이 옵션으로 이것을 읽으세요.

CatalogPlugin은 실제 카탈로그 통합을 위한 루트 인터페이스다. 거기서 Spark는 테이블 관리, 네임스페이스, 함수, 프로시저, 더 풍부한 메타데이터 동작을 레이어링할 수 있다.

😇 장점: 이름이 일급 시민이 된다.

😈 단점: 이름이 중요해지면 잘못 이름 짓는 것이 더 나은 문서가 있는 프로덕션 장애가 된다.

Spark 4: 더 많은 동사, 더 많은 서류

Spark 4.0은 저장 프로시저를 위한 카탈로그 API를 추가하고, V2 테이블 생성과 CTAS 지원을 개선하고, ALTER NAMESPACE ... UNSET PROPERTIES를 지원하고, V2 테이블에 SHOW COLUMNS 지원을 추가했다.

카탈로그는 이제 수동적인 주소록이 아니다. 운영 제어 표면이 되고 있다.

😈 단점: 모든 관리 동사는 장전된 도구다. CALL compact_table()은 유용하다. CALL regret_everything()은 보통 다른 이름으로 불리지만, 티켓 볼륨으로 알아볼 수 있다.

🐍 Python 깊이 파기: JVM 사제단 없는 데이터 소스

수년간 커스텀 Spark 소스는 JVM 장막 뒤에 살았다: Scala 예복, Java 향, 문 앞의 Maven. Python 사용자는 노트북을 들고 밖에서 기다리며 속삭였다: 나도 데이터가 있어요.

Spark 4.0은 Python 데이터 소스 API로 옆문을 열었다. 트랜잭션 레이크하우스 엔진을 점심 시간에 다시 작성하기 위한 것이 아니라, 내부 API 리더, 합성 생성기, 테스트 픽스처, 가벼운 싱크, "이 이상한 것을 그냥 읽어주세요" 커넥터를 만드는 Python 우선 팀을 위한 것이다.

JVM 순례 필요 없음.

Python 리더의 해부학

단순화된 리더 흐름은 다음과 같다:

DataSource
 ⤷ DataSourceReader
    ⤷ partitions()
       ⤷ read(partition)
          ⤷ Python iterator

DataSource는 진입점이다. 소스에 이름을 부여하고, 스키마를 선언하고, 리더나 작성자를 반환한다.

partitions()는 작업 티켓을 만든다. 경로, 오프셋, 페이지 번호, ID 같은 지루한 Python 딕셔너리나 단순 객체여야 한다.

😈 단점: 파티션 객체 안에 라이브 클라이언트, 소켓, 연결 풀, 열린 파일 핸들을 숨기지 마라. Spark는 이것들을 워커에 전송해야 하는데, 라이브 네트워크 연결을 피클링하는 것은 작업이 현대 무용을 발견하는 방법이다.

read(partition)은 워커 측에서 실행된다. 하나의 파티션 티켓을 받고 행을 Spark로 yield한다.

😈 단점: yield된 행은 선언된 스키마와 일치해야 한다. 필드 수, 타입, 널 허용 여부는 제안이 아니다. 스키마가 정수를 말하고 API가 "lol"을 보내면 Spark는 결국 알아낼 것이다.
