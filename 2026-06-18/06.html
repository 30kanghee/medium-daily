Claude Certified Architect: The Complete Guide to Passing the CCA Foundations Exam

Richard Hightower · Towards Artificial Intelligence · 원문 · freedium

요약

2026년 3월 출시된 Claude Certified Architect(CCA) Foundations 시험의 형식, 5개 역량 도메인, 6개 프로덕션 시나리오, 합격 전략을 총정리한 완벽 학습 가이드.

Anthropic이 2026년 3월 출시한 CCA Foundations 시험은 Claude로 프로덕션 시스템을 실제로 구축할 수 있는지를 검증하는 301 수준의 전문가 자격증이다. 시험은 60문항·120분·합격점 720점(1,000점 만점)으로 구성되며, 에이전트 아키텍처(27%), Claude Code(20%), 프롬프트 엔지니어링(20%), 도구 설계(18%), 컨텍스트 관리(15%) 5개 도메인을 다룬다. Accenture 3만 명, Cognizant 35만 명, Deloitte 47만 명 등 $1억 클로드 파트너 네트워크 투자와 맞물려 CCA 자격증의 시장 가치가 빠르게 높아지고 있다. 이 글은 도메인별 학습 전략, 반드시 피해야 할 안티패턴 카탈로그, 4주 학습 계획, 핵심 용어 사전을 포함한 종합 합격 가이드다.

2026년 3월 12일, Anthropic이 Claude Certified Architect(CCA) Foundations 시험을 공식 출시했다. AI 업계에 드디어 전문 자격증이 생겼다. Claude로 실제 프로덕션 시스템을 구축할 수 있는지를 검증하는 자격증이다. 영리한 프롬프트를 작성할 수 있는지가 아니다. 튜토리얼을 시청했는지도 아니다. 실제 프로덕션에서 동작하는 진짜 소프트웨어를 설계할 수 있는지를 묻는다.

CCA 시험을 준비 중이고 첫 시도에 합격하고 싶다면, 이 가이드가 그 길잡이다. 시험 형식, 5개 역량 도메인 전체, 6개 프로덕션 시나리오, 합격자와 불합격자를 가르는 멘탈 모델, 그리고 실제로 점수를 잃는 지점을 중심으로 구성된 주차별 학습 계획까지 모든 것을 다룬다.

Anthropic은 이 시험을 다음과 같이 정의한다. "Claude로 직접 개발한 경험이 최소 6개월 이상인 숙련된 전문가를 위한 301 수준의 시험."

문제는 시나리오 기반이며, 오답 보기도 매우 그럴듯하게 설계되어 있다. 시험은 무엇을 해야 하는지만큼이나 무엇을 하면 안 되는지도 함께 테스트한다.

지금 CCA가 중요한 이유

Anthropic은 이 자격증을 1억 달러 규모의 Claude 파트너 네트워크 투자와 함께 출시했다. Accenture는 3만 명의 전문가를 교육하고 Anthropic 전담 사업부를 신설했다. Cognizant는 35만 명의 직원에게 Claude 접근권을 부여했다. Deloitte는 47만 명의 직원을 플랫폼에 온보딩했다. Infosys는 전담 CoE(Center of Excellence)를 구축했다. 이것은 파일럿 프로그램이 아니다. 수십만 명 단위로 측정되는 인력 전환이다.

이 모든 조직은 전문가 수준으로 Claude 기반 시스템을 설계할 수 있는 사람이 필요하다. CCA 자격증은 현재 아무런 표준 신호가 없는 시장에서 채용 담당자에게 표준화된 신호를 제공한다. 시장이 아직 형성 중인 지금 CCA 배지를 가지고 있다는 것은, 2년 후 모두가 자격증을 가진 상황에서 나타나는 것과는 전혀 다른 의미를 가진다.

CCA는 Anthropic이 지금까지 출시한 첫 번째 자격증이다. 커뮤니티가 아직 작고 자격증이 신선한 지금 인증을 받는 것은, 곧 매우 길어질 줄의 맨 앞자리를 차지하는 일이다.

시험 형식: 무엇을 마주하게 되는가

  • 60문항, 120분
  • 합격 점수: 1,000점 만점에 720점
  • 응시 비용: $99
  • 감독관 시험 — 외부 도구, 문서, Claude 사용 불가

각 문제가 150~200단어 분량의 프로덕션 시나리오인 상황에서 문제당 2분은 빡빡하다. 900점대를 기록한 응시자들은 트랩 패턴을 즉각적으로 인식하는 것에서 속도가 나왔다고 말한다.

6개 프로덕션 시나리오 중 4개가 무작위로 출제된다. 6개 전부를 공부하라.

시간 관리 전략: 빠르게 1회독하면서 잠시 멈추게 만드는 문제에 플래그를 달아라. 확실히 아는 것부터 먼저 답하고, 그 다음 플래그된 문제로 돌아가라.

5개 역량 도메인

도메인 1: 에이전트 아키텍처 및 오케스트레이션 (27%)

이 도메인이 핵심이다. 시험의 4분의 1을 넘는다. 여기에 충분히 투자하지 않으면 다른 곳에서 만회할 수 없다.

알아야 할 내용

  • 멀티 에이전트 패턴: 코디네이터-서브에이전트 패턴과 허브-앤-스포크. 각각이 적절한 상황을 파악하라.
  • 핵심 규칙: 서브에이전트는 컨텍스트를 자동으로 상속받지 않는다. 코디네이터가 서브에이전트를 생성할 때, 서브에이전트는 빈 컨텍스트에서 시작한다. 필요한 모든 것을 명시적으로 전달해야 한다.
  • 턴 간 세션 상태 관리
  • 태스크 분해: 복잡한 요청을 개별 서브태스크로 분리하기
  • 에이전트 루프 설계: 입력 수신 → 추론 → 도구 호출 → 결과 평가 → 다음 단계 결정
  • 에스컬레이션 로직: 모델 주도 에스컬레이션이 아닌 결정론적 에스컬레이션 규칙

시험 팁: 도메인 1에서 가장 흔한 함정은 "슈퍼 에이전트" 안티패턴이다. 즉 4~5개의 도구를 가진 3~4개의 전문화된 에이전트 대신, 15개 이상의 도구를 가진 에이전트 하나로 구성하는 방식이다. 모든 것을 하나의 에이전트에 통합하는 선택지가 보이면, 그 답은 거의 항상 틀렸다.

학습 전략: 소규모 멀티 에이전트 시스템을 직접 구축하라. 코디네이터가 두 개의 서브에이전트를 생성하고, 각각에게 서로 다른 컨텍스트를 명시적으로 전달하고, 결과를 종합하게 만들어라.

도메인 2: Claude Code 설정 및 워크플로우 (20%)

알아야 할 내용

  • CLAUDE.md 계층 구조: 프로젝트 수준(.claude/CLAUDE.md, 버전 관리로 공유)과 사용자 수준(~/.claude/CLAUDE.md, 개인 설정). 프로젝트 수준은 "테크 리드" 파일이다. 사용자 수준은 개인 커스터마이징이다.
  • 커스텀 슬래시 커맨드 및 스킬: 마크다운 기반 스킬로 재사용 가능한 워크플로우
  • 플랜 모드 vs. 직접 실행: 복잡한 멀티 스텝 태스크에는 플랜 모드; 잘 정의된 저위험 태스크에는 직접 실행
  • CI/CD 통합: 비대화형 파이프라인을 위한 -p(--print) 플래그. CI/CD에서 -p를 빠뜨리는 것은 자주 출제되는 안티패턴이다.
  • --bare 플래그: 재현 가능한 동작을 보장하기 위한 스크립트 CI/CD 호출에 사용
  • --output-format json: 구조화된 JSON 출력을 위해

시험 팁: CI/CD 관련 문제는 거의 항상 -p 플래그를 잊은 응시자를 함정에 빠뜨린다. CI/CD 컨텍스트가 보이는 순간, -p가 있는지 확인하라. 제안된 솔루션에 -p가 없다면, 그 솔루션은 틀렸다.

도메인 3: 프롬프트 엔지니어링 및 구조화된 출력 (20%)

핵심 키워드는 "신뢰성"이다. Claude가 대부분의 경우에 JSON을 생성한다는 것은 프로덕션에 충분하지 않다.

알아야 할 내용

  • tool_choice를 활용한 JSON 스키마 강제: 입력 스키마와 함께 tool_choice를 사용하여 구조화된 출력을 강제하기. 시스템 프롬프트에서 JSON을 생성하도록 Claude에게 요청하는 것이 아니라, API의 도구 메커니즘을 사용하여 프로그래밍 방식으로 출력을 제한한다.
  • 구조화된 출력 API: Pydantic 모델을 사용한 client.messages.parse() (베타, anthropic-beta: structured-outputs-2025-11-13 헤더 필요)
  • 퓨샷 프롬프팅: 예시 입력과 출력 제공
  • 유효성 검사-재시도 루프: 스키마에 대해 출력을 검증하고, 실패하면 오류를 Claude에 다시 전송하여 수정
  • 핵심 안티패턴: JSON 준수를 위해 프롬프트에만 의존하기

시험 팁: JSON 준수 문제의 솔루션으로 "시스템 프롬프트에 지침을 추가하라"고 말하는 선택지는 거의 항상 틀렸다. 시험은 프롬프트만 사용하는 방식보다 tool_choice, 구조화된 출력 API, 유효성 검사-재시도 루프를 높이 평가한다.

도메인 4: 도구 설계 및 MCP 통합 (18%)

MCP(Model Context Protocol)는 2024년 11월 출시되었으며 OpenAI와 Google DeepMind도 채택했다. 커뮤니티 피드백에 따르면 이 도메인에서 대부분의 응시자들이 예상치 못한 점수를 잃는다.

알아야 할 내용

  • MCP 프리미티브:
    • Tools(도구): 모델이 호출할 수 있는 실행 가능한 함수. 액션에 사용한다.
    • Resources(리소스): 수집을 위한 데이터. 읽기 전용 컨텍스트에 사용한다.
    • Prompts(프롬프트): 미리 정의된 템플릿 및 워크플로우.
  • 설정 파일: .mcp.json은 프로젝트 수준(버전 관리); ~/.claude.json은 사용자 수준
  • 라우팅으로서의 도구 설명: 도구에 작성하는 설명이 Claude가 어느 도구를 호출할지 결정하는 주된 메커니즘이다. 자신의 코드베이스를 본 적 없는 개발자를 위한 문서를 작성하듯이 서술하라.
  • 4~5개 도구 규칙: Anthropic의 공식 지침은 에이전트당 4~5개의 집중된 도구다. 18개의 도구는 선택 신뢰성을 저하시킨다.

시험 팁: 시험 문제에서 무언가가 도구여야 하는지 리소스여야 하는지 물을 때 — Claude가 액션을 취하기 위해 호출해야 하는가, 아니면 컨텍스트로서 데이터가 필요한가? 액션은 도구다. 컨텍스트는 리소스다.

도메인 5: 컨텍스트 관리 및 신뢰성 (15%)

15%의 비중이 시사하는 것보다 더 많이 공부하라. 이 도메인은 다른 모든 도메인을 가로지른다.

알아야 할 내용

  • "중간 소실" 효과: Claude는 컨텍스트 윈도우의 시작과 끝에 있는 정보에 더 많은 주의를 기울인다. 중요한 정보는 시작 또는 끝에 배치하라.
  • 긴 세션에서의 컨텍스트 저하: 구조화된 요약 사용, 중요한 사실의 주기적 재명시
  • 에스컬레이션 설계: 모델의 자체 평가가 아닌 결정론적 규칙으로 에스컬레이션 유발
  • 토큰 이코노믹스:
    • 프롬프트 캐싱: 최대 90% 비용 절감, 실시간 지연. 반복되는 시스템 프롬프트, 정책 문서에 이상적.
    • 메시지 배치 API: 50% 비용 절감, 최대 24시간 지연(대부분 1시간 내 완료). 야간 감사, 대량 처리에 적합.
    • 실시간 API: 표준 가격, 실시간 지연. 사용자 대면 블로킹 워크플로우에 사용.

시험 팁: 사용자가 응답을 기다리고 있을 때 정답은 절대 배치 API가 아니다. 프롬프트 캐싱이다. 야간 배치 처리에는 배치 API가 거의 항상 올바른 비용 최적화 선택이다.

6개 프로덕션 시나리오

모든 시험은 이 6개 중 4개를 무작위로 추출한다. 6개 전부를 공부하라.

시나리오 1: 고객 지원 해결 에이전트. 모호도가 높은 요청을 처리하는 지원 에이전트 구축. 에스컬레이션 로직과 컴플라이언스 제약을 테스트한다. 가장 큰 함정: 에스컬레이션 여부를 결정하기 위해 Claude의 자체 보고 신뢰도 점수를 사용하는 것. LLM이 생성한 신뢰도 점수는 프로덕션 라우팅에 교정되어 있지 않다. 대신 프로그래밍 방식의 규칙을 사용하라: 티켓 카테고리, 금액, 계정 등급, 이슈 유형.

시나리오 2: Claude Code를 활용한 코드 생성. 대규모 코드베이스 탐색 및 코드 생성. 컨텍스트 저하와 플랜 모드 vs. 직접 실행을 테스트한다. 함정: 더 큰 컨텍스트 윈도우가 주의 분산 문제를 해결한다고 믿는 것. 그렇지 않다. "중간 소실" 효과는 컨텍스트 크기에 상관없이 적용된다.

시나리오 3: 멀티 에이전트 연구 시스템. 코디네이터가 전문화된 서브에이전트에게 연구 태스크를 분배한다. 함정: "슈퍼 에이전트" 안티패턴 — 서브에이전트별로 4~5개의 도구를 분배하는 대신 18개의 도구를 가진 에이전트 하나로 구성하는 것. 또한 서브에이전트가 코디네이터 컨텍스트를 자동으로 상속받지 않는다는 점도 테스트한다.

시나리오 4: 개발자 생산성 도구. 커스텀 슬래시 커맨드와 CLAUDE.md 설정을 활용한 Claude Code. 플랜 모드 vs. 직접 실행, CLAUDE.md 계층 구조, 재사용 가능한 스킬을 테스트한다.

시나리오 5: CI/CD용 Claude Code. 파이프라인에서의 자동 코드 리뷰, 테스트 생성, PR 피드백. 핵심 패턴: 비대화형 모드를 위한 -p 플래그 + 재현성을 위한 --bare + --output-format json. CI/CD에서 -p가 없는 솔루션은 틀렸다.

시나리오 6: 구조화된 데이터 추출. 비구조화된 문서에서 구조화된 데이터 추출. 함정: 프롬프트만으로 강제하기. 올바른 접근법: 입력 스키마와 함께 tool_choice를 사용하거나, 프로그래밍 방식의 유효성 검사 및 재시도 루프를 갖춘 구조화된 출력 API를 사용하기.

시험을 위한 핵심 멘탈 모델

멘탈 모델 1: 프로그래밍 방식의 강제가 프롬프트 기반 지침을 항상 이긴다

예외 없이. 비즈니스 규칙, JSON 준수, 라우팅 결정 등 모든 경우에서: 프롬프트가 아닌 코드로 강제하라. 프롬프트는 지침이다. 코드는 법이다.

멘탈 모델 2: 서브에이전트는 컨텍스트를 상속받지 않는다

코디네이터가 서브에이전트를 생성할 때, 서브에이전트는 빈 상태에서 시작한다. 필요한 모든 컨텍스트를 명시적으로 전달해야 한다. 이것은 직관에 반하기 때문에 가장 많이 테스트되는 개념 중 하나다.

멘탈 모델 3: 도구 설명이 라우팅을 결정한다

Claude는 도구의 이름이 아니라 설명을 기반으로 도구를 선택한다. 구체적이고 정밀하며 명확한 문서로 설명을 작성하라.

멘탈 모델 4: "중간 소실" 효과는 실재한다

중요한 정보는 컨텍스트의 시작 또는 끝에 배치하라. 중간에 있는 내용은 주의를 덜 받는다.

멘탈 모델 5: API를 지연 요구사항에 맞게 선택하라

백그라운드 작업에는 배치 API. 사용자 대면 워크플로우에는 실시간 API. 실시간 요구사항이 있는 반복 컨텍스트에는 프롬프트 캐싱.

안티패턴 카탈로그

도메인 1: 에이전트 아키텍처

  • 15개 이상의 도구를 가진 슈퍼 에이전트: 도구 선택 정확도를 저하시킨다. 에이전트당 4~5개의 도구를 사용하고 전문화된 서브에이전트에 위임하라.
  • 라우팅을 위한 자체 보고 신뢰도: LLM 신뢰도는 프로덕션에 교정되어 있지 않다. 결정론적 비즈니스 규칙을 사용하라.
  • 서브에이전트가 컨텍스트를 상속받는다고 가정하기: 그렇지 않다. 컨텍스트를 명시적으로 전달하라.

도메인 2: Claude Code

  • CI/CD에서 -p 없이 Claude Code 실행: 헤드리스 환경에서 멈춘다. -p와 --bare를 함께 사용하라.
  • 프로젝트 CLAUDE.md에 사용자 설정 넣기: 개인 설정이 모든 팀원에게 전파된다. 사용자 수준 설정은 ~/.claude/CLAUDE.md를 사용하라.
  • 복잡한 멀티 파일 변경에 직접 실행 사용: 대신 플랜 모드를 사용하라.

도메인 3: 프롬프트 엔지니어링

  • 프롬프트만으로 JSON 강제: 최선의 노력일 뿐, 엣지 케이스에서 실패한다. tool_choice 또는 구조화된 출력 API를 사용하라.
  • 유효성 검사-재시도 루프 없음: 유효성 검사 실패 시 오류 컨텍스트와 함께 재시도하는 시스템을 만들어라.

도메인 4: 도구 설계

  • 읽기 전용 데이터를 도구로 노출: 읽기 전용 데이터에는 MCP 리소스를 사용하고, 실행 가능한 액션에만 도구를 사용하라.
  • 모호한 도구 설명: 도구가 하는 일, 기대하는 입력, 사용 시점을 명확하게 서술하는 정밀하고 구체적인 설명을 작성하라.

도메인 5: 컨텍스트 관리

  • 라이브 고객 지원에 배치 API: 최대 24시간 지연. 사용자 대면 워크플로우에는 실시간 API를 사용하라.
  • 에스컬레이션 핸드오프 시 원시 대화 기록 전달: 원시 기록 대신 핵심 필드를 담은 구조화된 JSON 요약을 제공하라.
  • 컨텍스트 중간에 중요 정보 삽입: 중요한 정보는 시작 또는 끝에 배치하라.

4주 학습 계획

1주차: 기초 및 멘탈 모델

강좌: Claude 101 및 AI Fluency: Framework and Foundations
목표: 용어와 멘탈 모델을 내면화한다.
에이전트 루프, 컨텍스트 윈도우, 중단 이유, tool_choice를 학습한다. 1주차 말까지 다음을 사전 없이 정의할 수 있어야 한다: 에이전트 루프, 코디네이터-서브에이전트 패턴, 컨텍스트 포킹, 중단 이유(stop reason), tool_choice, 유효성 검사-재시도 루프, "중간 소실" 효과.

2주차: 핵심 스킬 (가장 집중해야 할 주차)

강좌: Building with the Claude API (8~10시간; 가장 중요한 단일 강좌)
목표: 가장 많이 테스트되는 패턴을 직접 경험한다.
유효성 검사-재시도 루프를 처음부터 구현하라. tool_choice와 JSON 스키마로 구조화된 출력을 실습하라. 2주차 말까지: 유효성 검사-재시도 루프 동작 코드, tool_choice로 강제된 구조화된 출력 호출, 3~4개 도구를 가진 간단한 에이전트를 직접 구현해야 한다.

3주차: 고급 주제 (도메인 4 지뢰 주차)

강좌: MCP Mastery, Claude Code in Action, Introduction to Agent Skills
목표: MCP 도구 설계 및 Claude Code 설정의 부족한 부분을 보완한다.
3개의 도구와 1개의 리소스를 포함한 MCP 서버를 Python으로 구축하라. 실제 프로젝트에서 CLAUDE.md를 설정하라. CI/CD 파이프라인 시뮬레이션에서 -p 플래그와 함께 Claude Code를 실행하라.

4주차: 실전 연습, 안티패턴, 시험 시뮬레이션

실제와 동일한 조건에서 공식 연습 시험을 응시하라. 실제 시험 일정을 잡기 전에 900점 이상을 목표로 하라.
캡스톤 프로젝트 구축: 각각 4~5개의 도구를 가진 두 개의 서브에이전트를 거느린 코디네이터 에이전트를 만들고, 서브에이전트 간에 명시적 컨텍스트를 전달하라.

도메인별 학습 시간 배분

  • 에이전트 아키텍처 및 오케스트레이션 (27%) — 8~10시간 — 최우선순위
  • Claude Code 설정 및 워크플로우 (20%) — 6~7시간
  • 프롬프트 엔지니어링 및 구조화된 출력 (20%) — 6~7시간 — 유효성 검사-재시도 루프 직접 구현
  • 도구 설계 및 MCP 통합 (18%) — 6~8시간 — 비중보다 어렵기 때문에 추가 시간 투자 필요
  • 컨텍스트 관리 및 신뢰성 (15%) — 4~5시간 — 토큰 이코노믹스 표 암기
  • 합계: 30~37시간

초기 응시자 피드백

  • 난이도는 실제다. 커뮤니티 공통 의견: "튜토리얼 보고 합격하는 자격증이 아니다."
  • 안티패턴 인식이 올바른 답을 아는 것만큼 중요하다.
  • MCP 도구 경계가 다른 어떤 주제보다 사람들을 더 많이 실패하게 만든다.
  • 특히 시나리오 체인에서는 시간 관리가 매우 중요하다.
  • 한 응시자는 985/1,000을 기록했다. 합격 점수는 720점 — 실수를 위한 여유가 있다. 완벽할 필요는 없다. 핵심 패턴을 확실히 알고 함정을 빠르게 인식하면 된다.

핵심 용어 사전

  • 에이전트 루프(Agentic Loop): 입력 수신, 추론, 도구 호출, 결과 평가, 다음 단계 결정의 순환
  • 컨텍스트 포킹(Context Forking): 컨텍스트를 서로 다른 서브에이전트를 위한 별도 브랜치로 분리하기
  • 코디네이터-서브에이전트 패턴(Coordinator-Subagent Pattern): 코디네이터 에이전트가 전문화된 서브에이전트에 위임하고 결과를 종합하는 패턴
  • 프로그래밍 방식 훅(Programmatic Hook): 도구 출력을 검증하거나 변환하는 코드 수준 강제 장치(예: PostToolUse)
  • 중단 이유(Stop Reason): 생성이 중단된 이유에 대한 API 신호(예: tool_use, end_turn, max_tokens)
  • tool_choice: 도구 선택을 제어하는 API 파라미터: auto(모델이 결정), any(반드시 도구 사용), 또는 특정 도구 지정
  • 유효성 검사-재시도 루프(Validation-Retry Loop): 스키마에 대해 출력을 검증하고, 실패 시 오류를 Claude에 다시 전송하여 수정하는 루프
  • MCP 리소스(MCP Resources): 수집을 위해 노출된 데이터 카탈로그/스키마 (읽기 전용 컨텍스트)
  • MCP 도구(MCP Tools): 모델이 액션을 취하기 위해 호출할 수 있는 실행 가능한 함수
  • MCP 프롬프트(MCP Prompts): 미리 정의된 지침 템플릿 또는 워크플로우
