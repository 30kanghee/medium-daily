Why I No Longer Use oh-my-zsh

GitHubDaily · 8 min read · 원문 · freedium

요약

수년간 oh-my-zsh에 쌓여온 레이어들이 개발 터미널 환경을 무겁고 불투명하게 만들어, sheldon·starship·atuin·zoxide·mise 조합으로 관심사를 분리해 전환한 실전 이야기다.

Ghostty 터미널에서 zsh 셸 시작 시간이 2.43초에 달하는 문제를 파고든 저자는 원인이 oh-my-zsh 플러그인들의 중복 로드와 레이어 혼재임을 밝혀냈다. 단순한 프레임워크 교체가 아닌 '관심사 분리'를 목표로, 플러그인 관리는 sheldon, 프롬프트는 starship, 히스토리 검색은 atuin, 디렉터리 이동은 zoxide, 런타임 관리는 mise로 각 책임을 나눴다. Ruby 2.7.8 재설치 실패 같은 현실적 장애물을 프래그매틱하게 해결하고, 버전 숫자가 아닌 which 명령으로 소유권 전환을 검증하는 방법을 강조했다. 마이그레이션 후 셸 시작 시간은 1.79초로 단축됐지만, 저자가 가장 소중히 여기는 것은 속도보다 '설명 가능성' — 문제 발생 시 어느 레이어를 봐야 하는지 명확히 아는 것이다.

서문

이번에 커맨드라인 환경을 손댄 이유는 더 멋진 도구 세트로 갈아타고 싶어서가 아니었다. Ghostty에서 zsh를 여는 것이 일상적인 업무 리듬을 방해할 만큼 느려졌기 때문이다.

그 느림은 벤치마크에서나 보이는 수치가 아니었다. 새 탭을 열 때, 새 창을 열 때, 셸을 전환할 때마다 몸으로 느껴지는 느림이었다. 터미널을 장기적인 주요 업무 인터페이스로 사용하는 사람에게 이런 지연은 계속해서 흐름을 끊는다.

처음에는 문제가 터미널 자체에 있다고 생각했다. 진지하게 트러블슈팅을 해본 뒤에야 Ghostty는 문제가 없고, 수년간 써온 oh-my-zsh 환경이 점점 무거워지고 있다는 것을 깨달았다. 더 정확히는, oh-my-zsh를 더 이상 쓸 수 없게 된 것이 아니라 내 설정이 모든 것을 이것 하나에 맡기기에는 더 이상 적합하지 않았던 것이다.

이 글은 세 가지를 명확히 하고자 한다: 왜 oh-my-zsh를 그만뒀는지, 최종적으로 어떤 커맨드라인 도구 조합을 선택했는지, 그리고 Ruby / Node / Zsh를 어떻게 매끄럽게 이전했는지.

결론 먼저

결론부터 원한다면, 이번 마이그레이션에서 내린 핵심 판단을 한 문장으로 요약하면:

oh-my-zsh는 환경을 빠르게 구성하기에는 훌륭하지만, 수년간 주 개발 머신에 계속 쌓아 올리기에는 적합하지 않다.

내가 선택한 조합은 다음과 같다:

  • sheldon
  • starship
  • atuin
  • zoxide
  • mise

마이그레이션 전 인터랙티브 zsh 시작 시간은 약 2.43초였고, 이후에는 약 1.79초다. 물론 이 향상은 의미 있지만, 이번 마이그레이션에서 가장 중요한 부분은 아니다. 더 중요한 것은 "누가 플러그인을 관리하고, 누가 프롬프트를 담당하고, 누가 Ruby를, 누가 Node를 관리하는지"를 드디어 분리했다는 점이다.

돌이켜보면, 이번 마이그레이션의 진짜 가치는 "새 도구 묶음으로 갈아탄 것"이 아니라, 점점 역사적 부채처럼 변해가던 커맨드라인 환경을 경계가 명확하고, 설명 가능하며, 롤백 가능한 시스템으로 재편한 것이다.

oh-my-zsh를 그만두기로 한 이유

먼저 분명히 하자면, 나는 oh-my-zsh를 비판하는 것이 아니다.

오히려 항상 훌륭한 출발점이라고 생각했다. 설치가 쉽고, 바로 동작하며, 플러그인이 풍부하고, 기본 alias도 편리하다. 터미널 환경을 처음으로 진지하게 만져보는 사람에게는 비용 대비 가치가 높은 선택이다.

하지만 문제는, oh-my-zsh는 "빠른 설정"에는 탁월하지만 "장기적인 최소 유지보수"는 잘 처리하지 못한다는 것이다.

내 이전 머신 환경은 꽤 전형적이었다:

  • 터미널: Ghostty
  • 셸: zsh
  • 플러그인 프레임워크: oh-my-zsh
  • Ruby: rbenv
  • Node: 일부는 nvm, 일부는 Homebrew로 직접 설치
  • 기타: 구문 강조, 자동 완성, 디렉터리 이동, 히스토리 검색, 각종 alias, 서드파티 스크립트 초기화

각각은 나름의 이유가 있었고, 많은 개발자 머신이 비슷하게 생겼다. 진짜 문제는 이것들이 한 번에 설계된 것이 아니라 수년에 걸쳐 하나씩 쌓여왔다는 것이다. 처음에는 git alias 몇 개를 추가하고, 그다음에는 구문 강조, 자동 완성, 버전 관리, PATH 주입, 셸 통합, 명령어 향상이 차례로 붙었다. 각각은 크지 않지만, 셸 시작 시에 모두 실행되며 결국 터미널 시작 체인을 엉킨 실타래로 만들었다.

이번 트러블슈팅에서 발견한 문제들은 매우 전형적이었다:

  • zsh-syntax-highlighting이 여러 번 로드되는 현상
  • rbenv 플러그인과 직접 작성한 rbenv init가 초기화를 중복 실행
  • brew shellenv가 모든 인터랙티브 셸에서 실행
  • thefuck이 초기화 중 실행되며 권한 오류 발생

공통점은 환경을 바로 망가뜨리지는 않지만 지속적인 숨은 비용을 만들어낸다는 것이다. 더 나쁜 건, 플러그인 관리, 프롬프트, 히스토리, 디렉터리 이동, 런타임 관리, 개인 alias가 모두 동일한 진입점에 뒤섞이면 기본적인 질문에 답하기가 어려워진다:

  • 누가 플러그인을 관리하는가?
  • 누가 프롬프트를 관리하는가?
  • 누가 Ruby를 관리하는가?
  • 누가 Node를 관리하는가?
  • 실제로 시작 속도를 늦추는 것은 무엇인가?

결국 oh-my-zsh를 포기한 것은 그것이 "나쁘기" 때문이 아니라, 내 커맨드라인 환경이 너무 복잡해져서 더 이상 하나의 올인원 프레임워크에 의존해서는 안 되는 상태가 됐기 때문이다.

이번에 진짜 원한 것

이런 글을 읽는 많은 사람들은 핵심이 "oh-my-zsh를 더 빠른 프레임워크로 교체하는 것"이라고 생각할 수 있다. 하지만 내가 이번에 진짜 목표로 한 것은 단일 지점 교체가 아니라 레이어 분리(decoupling)였다.

최종 선택한 도구 조합은 다음과 같다:

  • 플러그인 관리: sheldon
  • 프롬프트: starship
  • 히스토리 검색: atuin
  • 디렉터리 이동: zoxide
  • Ruby / Node 런타임 관리: mise

이 조합이 "궁극의 해법"은 아니다 — 다른 도구로 교체해도 된다. 하지만 이번에 내가 가장 중요하게 생각한 것은 도구의 이름이 아니라 경계였다.

이전에는 커맨드라인 환경이 모든 것이 .zshrc에 쌓여가는 거대한 진입 파일 같았다면, 이제는 책임이 명확한 여러 개의 작은 모듈처럼 만들고 싶었다. 플러그인이 어떻게 로드되는지, 프롬프트가 어떻게 렌더링되는지, 명령 히스토리를 어떻게 검색하는지, 디렉터리를 어떻게 이동하는지, 런타임은 누가 결정하는지 — 각각이 자신의 영역을 갖는 것. 나중에 문제가 생겨도 어떤 레이어를 보면 되는지 알 수 있고, 계속 늘어나는 .zshrc를 뒤지며 추측하는 일은 없어진다.

그래서 이번 마이그레이션의 핵심 이점을 한마디로 표현한다면, "oh-my-zsh에서 sheldon으로 갈아탄 것"이 아니라 "드디어 커맨드라인 환경을 도구 더미에서 구조로 정리한 것"이다.

마이그레이션의 진짜 어려움은 zsh가 아니었다

이번 마이그레이션에서 나는 "삭제 후 재설치" 방식이 아닌 매우 보수적인 원칙을 따랐다:

구조를 먼저 분리하고, 병렬로 인수하며, 마지막으로 점진적으로 전환한다.

이 원칙은 주 개발 머신에서 매우 중요하다. 장난감 환경에서 실험하는 게 아니라, 매일 일하는 머신의 인프라를 건드리는 것이기 때문이다.

내 첫 번째 단계는 설정을 바꾸는 것이 아니라 문제를 수치화하는 것이었다. 당시 인터랙티브 zsh 시작 시간은 약 2.43초였다. 무엇을 최적화하는지 정확히 알아야 하며, 그렇지 않으면 "더 빠른 것 같다"는 자기 착각에 빠지기 쉽기 때문에 이 단계는 중요하다.

다음으로 셸 시작 구조를 분리했다. 플러그인은 sheldon으로, 프롬프트는 starship으로, 히스토리는 atuin으로, 디렉터리 이동은 여전히 zoxide로, 내 alias/함수는 ~/.zshrc에 남겼다. 이 단계에서 가장 눈에 띄는 이점은 성능이 아니라 인지 부하 감소였다. 나중에 문제가 생겨도 어떤 레이어를 봐야 하는지 안다.

진짜 까다로운 부분은 런타임 관리, 즉 Ruby / Node를 mise에 넘기는 것이었다.

Node는 비교적 순조로웠다. 당시 머신에서 v24.10.0을 사용하고 있었는데, "명령이 성공적으로 실행됐는지"보다 이것들에 집중했다:

which node
mise which node
node -v

환경을 이전할 때 많은 사람들이 버전 번호만 확인하지만, 버전이 일치한다고 소유권이 진짜로 전환된 것은 아니다. 보이는 node -v는 바뀌지 않았을 수 있지만, 그 뒤에서 Homebrew인지 nvm인지 mise인지는 전혀 다를 수 있다. 주 개발 머신에서는 버전 번호보다 which를 더 믿는다.

Ruby가 이번 마이그레이션의 진짜 함정이었다. 목표 버전은 2.7.8이었고, 처음에는 mise로 처음부터 재설치하려 했지만 다운로드 중 다음과 같은 핵심 오류로 실패했다:

curl: (56) Recv failure: Connection reset by peer

이 문제는 실제 마이그레이션과 튜토리얼 글의 차이를 잘 보여준다. 튜토리얼은 보통 "명령 실행, 설치 성공, 다음 단계"이지만, 실제 머신 마이그레이션에서는 네트워크 불안정, 구 런타임 버전, 업스트림 배포 변경, 미러 문제 등 외부 요인을 만날 수 있다.

결국 "Ruby 2.7.8을 새로 다운로드해서 컴파일하는 것"에 고집하지 않고 더 실용적인 접근을 택했다: 현재 머신에 이미 안정적으로 동작하는 Ruby 2.7.8이 있으므로, mise가 이를 매끄럽게 인수하도록 했다. 주 개발 머신에서는 사용 가능성이 이론적 순수성보다 중요하다.

mise가 ruby와 node를 올바르게 인수할 수 있음을 확인한 후, ~/.zprofile을 수정해 명확히 만료된 PATH 잔재만 정리했다 — 예전 rbenv shim이나 nvm 버전 경로 같은 것들. 전체 PATH를 한 번에 다시 쓰는 것은 다른 툴체인을 같이 망가뜨릴 수 있어 피했다.

마지막으로, "기존 환경을 삭제"하는 것이 아니라 "먼저 롤백을 준비"했다. 이번에는 백업만 두는 것이 아니라 원클릭 복구 스크립트를 만들었다. 개발 환경 마이그레이션에서 가장 무서운 것은 전환 실패가 아니라 — 실패 후 되돌리려 할 때, "이전에 어떤 파일을 바꿨지?"를 기억에만 의존해야 하는 상황이다.

마이그레이션 후 또 한 번의 "정리"

이번 마이그레이션에서 매우 실질적인 후속 작업이 있었다: 첫 번째 전환을 마쳐도 곧바로 "완료"로 처리하지 않았다.

이유는 간단하다. which node와 which ruby가 mise를 가리키고 있었더라도, 나중에 인터랙티브 셸의 PATH를 다시 확인했을 때 예전 ~/.nvm/versions/..., ~/.rbenv/shims, ~/.rbenv/bin이 최종 환경에 여전히 남아 있었다. 가장 짜증스러운 점은 즉각적으로 문제를 일으키지 않아 마이그레이션이 성공했다고 생각할 수 있지만, 다음에 PATH 순서가 바뀔 때 은밀하게 예전 경로로 소유권이 다시 돌아갈 수 있다는 것이다.

그래서 새로운 마이그레이션 원칙을 추가했다: "기능이 실행되는지"만 확인하지 말고 "예전 경로가 진짜 빠져나갔는지"를 확인하라. 나에게는 node -v나 ruby -v 확인보다 이 명령들이 더 가치 있다:

which node
which ruby
mise which node
mise which ruby
print -l $path | rg '(.nvm/|.rbenv/)'

나중에 두 가지 범주에서 추가 정리를 했다. 하나는 중복 역사적 잔재로, 예를 들어 디렉터리 이동이 zoxide로 전환됐으므로 예전 autojump가 계속 은밀하게 로드되어선 안 된다. 다른 하나는 인터랙티브 셸에서만 존재해야 하는 기능들로 — bun, OpenClaw, fzf 자동 완성과 인터랙션 향상 — 이들이 스크립트 셸에 시작 비용을 부담시켜선 안 된다.

이 정리 과정에서 전형적인 함정을 만났다: 설정 파일이 "올바르게 보이는 것"이 실제로 실행된다는 뜻은 아니다. 예를 들어 나중에 Yazi 설정을 추가할 때 버전 호환성 문제가 있었다 — 파일 오프너 실패가 아니라 도구 자체가 시작되지 않았다. 그 순간 무언가를 확인했다: 주 개발 머신에서 환경 마이그레이션은 설정이 문서 예시와 일치하는지 눈으로만 확인해서는 안 되고, 실제 시작 검증이 필요하다.

yazi --version
tmux -V
lazygit --version
zsh -i -c 'echo ok'

마지막으로, "개인 정보"와 "기능 설정"을 분리했다. 이전에는 API 키, PATH 주입, 프록시 alias, 함수, 자동 완성 스크립트를 하나의 진입 파일에 모두 넣었다. 최근 들어 이것이 단기적으로 시간을 절약하지만 장기적으로 인지 부하를 증폭시킨다는 것을 점점 더 느낀다. 기능 설정은 기능과 함께 있어야 하고, 개인 자격 증명은 최소한 별도의 비공개 환경 파일에 있어야 한다. 그렇지 않으면 나중에 셸 초기화 문제를 디버깅하는 건지 환경 변수 문제를 디버깅하는 건지 구분하기 어렵다.

마이그레이션 후 가장 소중한 것은 속도가 아니라 설명 가능성이다

결과적으로 이번 마이그레이션은 가치 있었다. 인터랙티브 zsh 시작 시간이 약 2.43초에서 1.79초로 줄었고, 확실히 가벼운 느낌이다.

하지만 "0.6초 빨라진 것"으로 요약한다면 과소평가하는 것이다. 진짜 가치는 이전에는 불분명했던 많은 문제들을 이제 명확하게 설명할 수 있다는 것이다.

예를 들어:

  • 누가 플러그인을 관리하는가?
  • 누가 프롬프트를 관리하는가?
  • 누가 히스토리 검색을 관리하는가?
  • 누가 디렉터리 이동을 관리하는가?
  • 누가 Ruby / Node를 관리하는가?

이제 나중에 문제가 생겨도 역사적 설정들의 혼돈 속에서 경험에 의존해 추측하지 않아도 된다; 구조에서부터 트러블슈팅할 수 있다. 개발 머신을 장기적으로 유지하는 사람에게 이 설명 가능성은 조금 빨라지는 것보다 더 중요하다.

이번 마이그레이션에서 당부하고 싶은 것들

  1. 기존 환경을 바로 삭제하지 마라: 특히 ~/.rbenv, ~/.nvm, 구 런타임 디렉터리 같은 것은 전환 당일에 삭제하지 마라. 더 안전한 순서는 새 도구를 도입하고, 전환 성공을 확인하고, 며칠 정상적으로 사용한 후 구 디렉터리를 정리하는 것이다.
  2. 버전 번호만 보지 말고 반드시 which를 확인하라: ruby -v와 node -v는 "현재 버전이 변하지 않았다"고만 말할 뿐, 진짜 누가 담당하는지는 알려주지 않는다. 핵심은 which ruby, which node, mise which ruby, mise which node다.
  3. 전체 PATH를 한 번에 재작성하지 마라: PATH는 환경을 사고로 망가뜨리기 가장 쉬운 곳이다. 가능한 한 적게 건드리고, 명확히 만료된 세그먼트만 삭제하며, 다른 툴체인의 순서를 함부로 바꾸지 마라.
  4. 주 개발 머신 마이그레이션에는 반드시 롤백 스크립트가 있어야 한다: 백업만으로는 충분하지 않다. 원클릭 복구가 최선이다. 정말로 롤백이 필요할 때는 보통 망가진 상태이고, 수동 기억에 의존할수록 2차 함정에 빠지기 쉽다.
  5. 버전 번호와 표면적 기능만 보지 말고 which, 최종 PATH, 예전 경로가 진짜 빠져나갔는지 확인하라.
  6. 자동 완성, fzf, 프롬프트 등 인터랙티브 기능은 인터랙티브 셸에서만 로드하라; 스크립트 셸에도 시작 비용을 부담시키지 마라.
  7. 어떤 단일 도구도 맹목적으로 따르지 마라: 결국 sheldon, starship, atuin, zoxide, mise라는 이름이 핵심이 아니다. 명확한 경계, 독립적인 책임, 문제 발생 시 명확한 트러블슈팅 진입점이 핵심이다.

마지막으로

이번 마이그레이션을 마치고 가장 크게 느낀 것은 "드디어 더 멋진 커맨드라인 도구 세트로 갈아탔다"가 아니라:

커맨드라인 환경도 코드처럼 시간이 지나면서 부패할 수 있다.

많은 개발자들이 비즈니스 코드를 리팩토링하는 데 시간을 기꺼이 투자하지만, 매일 사용하는 터미널 환경을 진지하게 리팩토링하는 사람은 드물다. 그러나 터미널은 일상적인 개발에서 가장 자주 접하는 업무 인터페이스 중 하나다. 느린 시작, 엉킨 구조, 디버깅하기 어려운 문제들 — 이 비용들은 한꺼번에 폭발하지 않지만, 조금씩 집중력을 갉아먹는다.

지금 내가 궁극적인 해법을 찾았다고 생각하지는 않는다. 하지만 적어도 이번에는 점점 무거워지는 역사적 부담에서, 경계가 더 명확하고, 유지보수가 더 쉬우며, 설명 가능성이 더 좋은 도구 조합으로 정리했다. 나에게 그것만으로도 충분히 가치 있다.
