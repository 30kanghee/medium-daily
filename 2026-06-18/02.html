Apache Spark 4.1 is Here: The Next Chapter in Unified Analytics

Cheng Zhi Zhao · Data Engineering Space · 5 min read · 원문 · freedium

요약

Apache Spark 4.1은 진정한 실시간 스트리밍, 선언적 파이프라인, Python 네이티브 지원, SQL 스크립팅을 도입해 데이터 엔지니어의 일상을 실질적으로 개선한다.

2025년 12월에 출시된 Apache Spark 4.1은 단순한 속도 향상을 넘어 실제 데이터 파이프라인 작업의 불편함을 정면으로 해결합니다. 가장 큰 변화는 마이크로배치 방식을 탈피한 진정한 실시간 모드(RTM)로, 무상태 연산에서 한 자리 수 밀리초의 지연을 달성합니다. Spark 선언적 파이프라인(SDP)은 제어 흐름 대신 데이터 흐름만 정의하면 되는 방식으로 복잡한 오케스트레이션 코드를 줄입니다. Python 측면에서는 Arrow 네이티브 UDF와 개선된 에러 메시지로 PySpark 개발 경험이 크게 향상됩니다. SQL 스크립팅이 정식 출시(GA)되어 SQL 내에서 직접 변수 선언, 루프, 예외 처리가 가능해졌습니다.

오랫동안 주요 Spark 릴리즈는 순수한 속도 벤치마크에만 집중하는 것처럼 느껴졌습니다 — 페타바이트를 얼마나 빠르게 처리할 수 있는지. 그것도 중요하지만, Apache Spark 4.1은 다른 느낌입니다.

2025년 12월에 출시된 이 버전은 매일 데이터 파이프라인을 실제로 작성하는 사람들이 만든 것처럼 느껴집니다. 일상적인 워크플로우의 짜증스러운 부분들을 해결합니다: 지연 지체, 의존성 지옥, Java 기반 세계에서 Python을 사용하는 마찰.

업그레이드할 가치가 있는지 궁금하다면, 실제로 작업 방식을 바꿀 기능들의 요약이 여기 있습니다.

1. 드디어, "진짜" 실시간

Structured Streaming을 사용해 본 적이 있다면, 이미 알고 있는 사실이 있습니다: 그것은 기술적으로 실시간이 아니었습니다. "마이크로배치"였습니다. 항상 다음 배치가 트리거될 몇 초를 기다리고 있었습니다. 이것이 많은 사람들이 Spark의 스트리밍 데이터 처리 오버헤드 때문에 여전히 Flink를 선호하는 이유 중 하나입니다.

Spark 4.1은 실시간 모드(RTM, Real-Time Mode)를 도입했으며, 이것은 대대적인 변화입니다.

  • 변화: 쿼리를 이제 배치 단위가 아닌 연속적으로 실행할 수 있습니다.
  • 결과: 무상태 연산의 경우 지연이 한 자리 수 밀리초로 줄어듭니다.
  • 분위기: 서브초 응답이 필요하다고 해서 Flink나 Kafka Streams로 전환할 필요가 없어졌습니다. Spark가 이제 "빠른 차선"을 처리할 수 있습니다.

전통적인 마이크로배치에서 실시간 모드로의 전환은 주로 "Trigger" 전략의 변화입니다. 사실상 Spark에게 X초마다 데이터를 확인하는 것을 멈추고 대신 연결을 열린 상태로 유지해 이벤트가 도착하는 대로 처리하라고 지시하는 것입니다.

1단계: 표준 마이크로배치 (현재 상태)

아마도 처리 시간으로 설정된 트리거가 있을 것입니다. Spark가 간격을 기다리고, 작업을 시작하고, 처리하고, 태스크를 종료하기 때문에 지연이 발생합니다.

# 지연: ~10초 - 1분
stream = spark.readStream.format("kafka")...

query = stream.writeStream \
    .trigger(processingTime='10 seconds') \  # ❌ "대기" 사이클
    .start()

2단계: RTM으로의 마이그레이션 (Spark 4.1)

마이그레이션하려면 트리거를 continuous로 변경합니다. Spark는 이제 소스를 지속적으로 폴링하는 장기 실행 태스크를 시작해 태스크 시작 오버헤드를 제거합니다.

stream = spark.readStream.format("kafka")...

query = stream.writeStream \
    .trigger(continuous='1 second') \  # ✅ "항상 켜짐" 사이클
    .start()
  • 상태: 가장 낮은 지연을 위해 연산이 무상태인지 확인하세요(예: filter, map, 간단한 프로젝션). 유상태 연산(집계)은 정확한 엄격성 보장에 따라 여전히 마이크로배치 의미론이 필요할 수 있습니다.
  • 리소스: RTM은 코어를 영구적으로 점유합니다(배치 사이에 해제하지 않음). 이러한 장기 실행 태스크를 위해 클러스터에 전용 코어가 있는지 확인하세요.

2. 스파게티 코드 없는 파이프라인

우리 모두 그런 경험이 있습니다: "테이블 A를 실행한 뒤 테이블 B를 실행하라"는 것만 말하기 위해 복잡한 오케스트레이션 로직을 작성하는 것.

Spark 선언적 파이프라인(SDP)이 등장했습니다. 제어 흐름("어떻게")을 작성하는 대신, 데이터 흐름("무엇")만 정의합니다.

  • 어떻게 도움이 되는지: Spark에게 "이 데이터셋이 이렇게 보이길 원한다"고 말합니다. Spark가 의존성 그래프, 재시도 로직, 체크포인팅을 파악합니다.
  • 왜 좋아할지: 훨씬 더 "현대적 데이터 스택" 경험에 가깝습니다. 저장소를 막고 있던 보일러플레이트 코드 층을 제거합니다.

"함께 더 나은" 아키텍처: Airflow를 완전히 버리지 않습니다. 대신, Airflow가 SDP의 "트리거"가 됩니다.

  1. Airflow가 말합니다: "오전 8시입니다. 영업 파이프라인을 깨워주세요."
  2. SDP가 인계받습니다: "알겠습니다. raw_clicks 테이블에 새 데이터가 있습니다. daily_metrics를 업데이트해야 합니다. 백필, 체크포인트, 병합 로직을 처리하겠습니다."
  3. Airflow가 기다립니다: "데이터가 준비되면 알려주세요."

이것은 관심사의 분리를 만듭니다: Airflow는 일정을 관리하고, Spark는 데이터 무결성을 관리합니다.

3. Python이 드디어 네이티브처럼 느껴진다

수년간 PySpark 개발자들은 직렬화 오버헤드와 암호화된 Java 스택 트레이스와 씨름하며 이등 시민처럼 느꼈습니다. Spark 4.1은 "Python화" 릴리즈입니다.

  • Arrow 네이티브 UDF: 이제 데코레이터를 사용해 PyArrow를 직접 사용해 데이터를 처리하는 UDF를 작성할 수 있습니다. 느린 변환 과정을 건너뛰고 커스텀 Python 함수를 빠르게 실행합니다.
  • 더 나은 에러: 작게 들리지만, 엄청납니다. Python 코드가 실패하면, Spark 4.1은 PhD가 필요한 50줄 JVM 스택 트레이스가 아닌 Python 에러를 제공하려 합니다.
  • 디버깅: Python 워커 내부에서 정확히 무슨 일이 일어나는지 볼 수 있는 새로운 python_worker_logs() 함수가 있습니다.

다음은 이전 버전의 표준 UDF 작성 방식과 Spark 4.1의 새로운 Arrow 네이티브 방식의 구체적인 차이입니다.

이전 방식 (표준 Python UDF) — 이전에는 표준 Python 직렬화(pickle)를 사용했으며, JVM과 Python 사이에서 행 단위로 데이터를 변환하기 때문에 느렸습니다.

from pyspark.sql.functions import udf
from pyspark.sql.types import IntegerType

@udf(returnType=IntegerType())
def calculate_score(value):
    # 순수 Python 로직
    return (value * 2) + 10

df.select(calculate_score("score")).show()

Spark 4.1 방식 (Arrow 네이티브 UDF) — 이제 플래그를 추가하거나 전역적으로 활성화해 Arrow 직렬화를 강제할 수 있습니다. 이것은 데이터를 열 단위 배치로 처리해 로직을 pandas로 다시 작성하지 않고도 대폭적인 속도 향상을 제공합니다.

from pyspark.sql.functions import udf
from pyspark.sql.types import IntegerType

@udf(returnType=IntegerType(), useArrow=True)
def calculate_score(value):
    return (value * 2) + 10

df.select(calculate_score("score")).show()

4. SQL에서의 절차적 프로그래밍

SQL은 쿼리에는 훌륭하지만 로직에는 끔찍합니다. 루프를 실행하거나 변수를 설정하기 위해 보통 Python으로 SQL을 감싸야 했습니다.

이제 정식 출시(GA)된 SQL 스크립팅으로 SQL 스크립트 내에서 직접 절차적 로직을 처리할 수 있습니다:

  • 변수 DECLARE
  • FOR와 WHILE 루프 실행
  • CONTINUE HANDLER로 예외 처리
-- 1. 품질 메트릭을 담을 변수 정의
DECLARE total_rows INT DEFAULT 0;
DECLARE null_count INT DEFAULT 0;

-- 2. 스테이징 테이블에서 메트릭 계산
SET total_rows = (SELECT COUNT(*) FROM new_sales_data);
SET null_count = (SELECT COUNT(*) FROM new_sales_data WHERE customer_id IS NULL);

-- 3. 로직 블록
-- 데이터가 있고 "불량 데이터" 비율이 허용 가능한지 확인 (예: < 1%)
IF total_rows > 0 AND (null_count / total_rows) < 0.01 THEN
    -- 성공 경로: 데이터 병합
    MERGE INTO sales_history AS target
    USING new_sales_data AS source
    ON target.id = source.id
    WHEN MATCHED THEN UPDATE SET target.amount = source.amount
    WHEN NOT MATCHED THEN INSERT *;
    SELECT 'Success: Data merged successfully.' AS status;
ELSE
    -- 실패 경로: 메인 테이블 건드리지 않음
    -- 작업을 명시적으로 실패시키기 위해 에러를 발생시킬 수도 있음
    SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = 'Data Quality Check Failed: Too many nulls';

END IF;

마무리 생각

Spark 4.1은 단순히 더 빠른 것이 아닙니다; 더 스마트합니다. 역사적으로 데이터 엔지니어들이 작업을 완료하기 위해 다섯 가지 다른 도구를 함께 붙여야 했던 마찰을 제거합니다. 오케스트레이션, 실시간 스트리밍, 절차적 로직을 "사내"로 가져옴으로써 Spark는 여러분에게 필요한 유일한 엔진이 되기 위한 강력한 주장을 펼치고 있습니다.
