I Used Git Wrong for Years

Tushar Kanjariya · 6 min read · 원문 · freedium

요약

수년간 Git을 겨우 세 가지 명령어로만 쓰던 개발자가 시니어 개발자를 보며 깨달음을 얻고, 실전을 바꾼 14가지 Git 습관과 명령어를 소개한다.

저자는 오랫동안 git add . / git commit -m "update" / git push만 반복했지만, 시니어 개발자가 패닉 없이 삭제된 커밋을 복구하고 지저분한 히스토리를 정리하는 모습을 목격하며 Git을 제대로 배워야겠다고 결심한다. Git을 백업 도구가 아닌 프로젝트의 전체 타임라인으로 이해하는 것이 출발점이다. 핵심 팁으로는 커밋 전 git status와 git diff로 상태를 확인하고, git add -p로 변경 사항을 선택적으로 스테이징하며, git bisect로 버그를 도입한 커밋을 이진 탐색으로 찾고, git stash로 작업을 안전하게 임시 저장하고, git reflog로 잃어버린 줄 알았던 커밋을 복구하는 방법 등이 포함된다. 또한 인터랙티브 리베이스(git rebase -i)로 커밋 히스토리를 정리하고, git worktree로 브랜치 전환 없이 동시에 여러 작업을 진행하는 고급 기법도 다룬다. 저자는 좋은 Git 실력의 본질이 명령어 암기가 아니라 실수했을 때 침착하게 복구할 수 있다는 자신감이라고 결론 짓는다.

나는 Git이 그냥 세 가지 명령어라고 생각했다.

git add .
git commit -m "update"
git push

그것이 수년간 내 워크플로우였다.

어느 날 나는 시니어 개발자가 삭제된 커밋을 복구하고, 6개월치 지저분한 히스토리를 정리하고, 구글링도 없이, 패닉도 없이 나쁜 병합을 되돌리는 것을 지켜봤다.

그 사이 나는 여전히 구글에서 이것을 검색하고 있었다. "마지막 커밋을 취소하되 변경 사항은 유지하는 방법"

그 순간 깨달았다.

대부분의 개발자는 Git을 겨우 살아남을 수준만큼만 안다. 빠르게 움직이는 사람들은 Git이 실제로 어떻게 작동하는지 안다.

여기 내가 너무 늦게 배운 Git 요령과 습관들이 있다. 실제로 매일 일하는 방식을 바꾼 것들이다.

짧은 답변: 시니어 개발자들이 Git에서 더 빠른 것은 더 많은 명령어를 외워서가 아니다.

그들은 Git을 타임라인으로, 즉 프로젝트가 지금까지 존재했던 모든 상태의 기록으로 이해하기 때문에 빠르다. 그것이 클릭되면 모든 명령어가 직관적으로 이해되기 시작한다.

Git은 단순한 버전 관리가 아니다

오랫동안 나는 Git을 백업 시스템처럼 다뤘다.

변경 → 커밋 → 푸시.

하지만 Git은 프로젝트 전체 삶의 타임라인에 가깝다.

그것을 이해하고 나서 명령어들이 무섭게 느껴지지 않았다.

그리고 실수 수정이 훨씬 쉬워졌다.

팁 #1: git status가 가장 중요한 명령어다

당연하게 들린다. 나는 수년간 무시했다.

이제 모든 커밋 전, 브랜치 전환 전, 리베이스 전에 지속적으로 실행한다.

git status

대부분의 Git 실수는 현재 상태를 완전히 모를 때 일어난다. 이 단계를 건너뛰었다가 .env 파일, 디버그 로그, 미완성 코드, 무작위 스크린샷을 실수로 커밋한 적이 있다.

명령어 하나. 2초 소요. 몇 시간을 아낀다.

팁 #2: git diff가 부끄러운 커밋을 막아준다

이것이 내 커밋을 완전히 바꿨다.

커밋 전:

git diff

무엇이 바뀌었는지 정확히 보여준다. 프로덕션 푸시 직전에 API 키, 깨진 임포트, 남겨진 console.log를 잡은 적이 있다.

더 좋은 것:

git diff --staged

이미 추가한 파일만 보여준다. 이제 거의 모든 커밋 전에 이것을 사용한다.

팁 #3: git add . 를 맹목적으로 사용하지 말라

git add .는 모든 것을 스테이징한다.

때로는 괜찮다. 하지만 종종 작업하는 동안 손댄 다른 4가지가 아니라 특정 변경 사항 하나만 커밋하고 싶을 때가 있다.

대신 이것을 사용하라:

git add -p

이것은 변경 사항을 조각별로 추가하게 해준다. 각 청크를 검토하고 스테이징할지 건너뛸지 결정한다.

실제 예: 같은 파일에서 로그인 수정, UI 리디자인, 무작위 정리 작업을 동시에 하고 있었다. git add -p로 로그인 수정만 깔끔하게 커밋할 수 있었다.

더 깔끔한 커밋. 나중에 디버깅이 더 쉽다. 추가 30초가 가치 있다.

팁 #4: git bisect가 마법사가 된 기분을 준다

수년간 너무 고급스러워 보여서 이 명령어를 무시했다.

상황: 프로덕션에서 무언가 망가진 것을 안다. 3주 전에는 작동했다. 그 사이 80개의 커밋 중 어느 것이 문제를 도입했는지 모른다.

git bisect start
git bisect bad          # 현재 상태가 망가짐
git bisect good v2.3.1  # 이 버전은 괜찮았음

그러면 Git이 good과 bad 사이의 정확히 절반 지점의 커밋을 체크아웃한다.

테스트한다. Git에게 good인지 bad인지 알려준다.

다시 검색 공간이 절반으로 줄어든다. 약 7번 반복하면 80개의 커밋을 1개로 좁힐 수 있다.

한 달간 열려 있던 버그에 이것을 사용했다. 11분 만에 정확한 커밋을 찾았다. 커밋 메시지는 "minor cleanup"이었다. 마이너하지 않았다.

팁 #5: git stash는 구원자다

셀 수 없이 많은 번 나를 구해줬다.

상황: 기능 작업 중간에 있다. 긴급 버그가 발생했다. 지금 당장 브랜치를 전환해야 한다.

전에는 패닉했다.

이제는:

git stash

작업이 안전한 사물함으로 사라진다. 브랜치를 전환하고, 버그를 수정하고, 돌아온다:

git stash pop

떠난 그대로 모든 것이 돌아온다.

브랜치는 기본적으로 샌드박스 환경이다. git stash는 그 사이의 긴급 일시정지다.

cd -의 Git 브랜치 버전이다.

팁 #6: git log가 마침내 Git을 이해하게 만들었다

수년간 Git 히스토리는 내게 노이즈처럼 보였다.

그런 다음 이것을 사용하기 시작했다:

git log --oneline --graph --decorate

갑자기 프로젝트의 전체 타임라인이 브랜치, 병합, 커밋 흐름 모두 명확하게 펼쳐졌다.

즉시 별칭을 만들었다:

alias gl='git log --oneline --graph --decorate'

이제 gl은 내 일상 워크플로우에서 가장 많이 사용하는 명령어 중 하나다.

팁 #7: 브랜치는 저렴하다 — 더 많이 사용하라

main에서 직접 작업하는 것은 함정이다. 너무 오랫동안 그랬다.

이제 거의 모든 것에 브랜치를 만든다:

git switch -c fix-login-bug

이것이 내가 얼마나 대담하게 실험하는지를 바꿨다. 안정적인 코드에 절대 손대지 않고 아이디어를 테스트하고, 공격적으로 리팩터링하고, 위험한 수정을 시도할 수 있다.

브랜치는 무료다. main을 망가뜨리는 비용은 무료가 아니다.

팁 #8: 필요하기 전에 git restore를 배워라

이 명령어가 심장마비에서 나를 구해줬다.

실수로 잘못된 파일을 편집했다.

변경 사항을 수동으로 되돌리는 대신:

git restore app.js

파일이 마지막으로 커밋된 상태로 즉시 복원된다.

첫 달에 이것을 배웠으면 좋았을 것이다.

팁 #9: git worktree는 범죄적으로 과소평가되어 있다

이것은 "Hot take: Worktrees are underrated"라는 Reddit 스레드에서 나왔고, 전체를 읽으러 갔다. 그 핫테이크는 맞았다.

보통 기능 작업 중간에 작업을 전환할 때 나는:

  • 변경 사항을 stash
  • 브랜치 전환
  • 핫픽스 작업
  • 다시 전환
  • unstash
  • 충돌이 없기를 기도

했었다. git worktree는 다르다:

git worktree add ../hotfix-branch hotfix/login-timeout

이것은 hotfix/login-timeout을 별도의 디렉토리로 체크아웃한다.

현재 브랜치는 건드리지 않는다. 다른 디렉토리로 cd하고, 버그를 수정하고, 푸시하고, 다시 cd로 돌아온다.

stash 없음. 체크아웃 없음. "아, 내가 뭘 하던 중이었지" 하는 상황 없음.

솔직히 말하면: 아직 자동으로 이것을 사용하지는 않는다. 아직 조금 낯설다. 하지만 더 긴 중단(한 시간 이상)의 경우 stash-and-pray 워크플로우보다 훨씬 깔끔하다.

팁 #10: git reflog는 시간 여행처럼 느껴진다

이것이 시니어 개발자들을 마법처럼 보이게 만드는 명령어다.

한번은 리베이스 중에 커밋을 영구적으로 잃어버렸다고 생각했다.

대부분의 개발자들이 모르는 것: Git은 거의 모든 것을 기억한다.

git reflog

삭제했다고 생각한 커밋, 떨어뜨린 브랜치, reset으로 넘어간 상태까지 포함하여 모든 최근 Git 동작을 보여준다.

복구는 명령어 하나다:

git checkout <commit-id>

그 순간이 Git 실수에 대해 생각하는 방식을 완전히 바꿨다. 이제 실수하는 것이 훨씬 덜 무섭다.

git reflog는 안전망이다. 동작 후 약 90일 동안 존재한다. 무언가가 사라졌다고 가정하기 전에 사용하라.

팁 #11: 더 나은 커밋 메시지가 생각보다 훨씬 중요하다

예전의 나:

git commit -m "fix"

미래의 나는 그 사람이 싫었다.

이제:

git commit -m "Fix token refresh issue in auth middleware"

좋은 커밋 메시지는 Git을 위한 것이 아니다. 인간을 위한 것이다. 보통 6개월 후 목요일 밤 11시에 무언가가 왜 바뀌었는지 이해하려는 미래의 자신을 위해.

팁 #12: 인터랙티브 리베이스가 내 히스토리를 더 깔끔하게 만들었다

수년간 위험하게 들려서 git rebase를 피했다.

그런 다음 배웠다:

git rebase -i HEAD~5

이것은 커밋을 squash하고, 이름을 바꾸고, 순서를 변경할 수 있게 해준다. PR이 나가기 전 모두.

전:

fix
another fix
small update
final fix

후:

Improve authentication error handling

하나의 깔끔하고 의미 있는 커밋. 팀원들이 차이를 알아챌 것이다.

팁 #13: git blame은 놀랍도록 유용하다

무서운 이름에도 불구하고.

git blame app.js

누가 각 줄을 바꿨는지, 언제, 어느 커밋에서인지 보여준다.

프로덕션 버그를 도입한 정확한 커밋까지 추적하고, 이상한 코드 조각이 왜 존재하는지 이해하고, 몰랐던 관련 커밋들을 찾는 데 사용했다.

기본적으로 코드 고고학이다. 찾는 것을 발견하면 놀랍도록 만족스럽다.

팁 #14: 별칭이 마찰을 없앴다

긴 명령어 입력에 지쳤다.

그래서 별칭을 추가했다:

alias gs='git status'
alias ga='git add'
alias gc='git commit -m'
alias gp='git push'
alias gl='git log --oneline --graph --decorate'

각 별칭은 약 3초를 아낀다. 하루 50번 사용, 수개월 동안 — 쌓인다. 더 중요한 것은, 마찰이 낮아지면 실제로 실행해야 할 명령어들을 실행하게 된다는 것이다.

진짜 변화

수년간 나는 좋은 Git 실력이 명령어 암기를 의미한다고 생각했다.

이제 이것이 대부분이라고 생각한다:

일이 잘못될 때 침착함을 유지하는 것.

내가 아는 최고의 개발자들은 Git을 두려워하지 않는다.

왜냐하면 그들은 복구를 이해하기 때문이다.

그것이 진짜 차이점이다.

속도 타이핑이 아니다.

화려한 워크플로우도 아니다.

자신감이다.

그리고 솔직히 대부분의 Git 자신감은 배우는 것에서 온다:

  • 변경 사항을 검사하는 방법
  • 실수를 되돌리는 방법
  • 안전하게 복구하는 방법

그것들을 알면 Git이 더 이상 위험하게 느껴지지 않는다.

다음으로 시작하라:

  • git diff
  • git stash
  • git reflog

그 세 가지만으로 수많은 시간을 아낄 수 있다.

결론

대부분의 개발자는 Git을 백업 도구처럼 사용한다.

더 나은 개발자들은 타임머신처럼 사용한다.
