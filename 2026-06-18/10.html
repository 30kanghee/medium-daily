Stop Using ROW_NUMBER() for Deduplication in Spark — Use max_by() Instead

Dh Soni · 6 min read · 원문 · freedium

요약

Spark에서 중복 제거 시 ROW_NUMBER() 윈도우 함수 대신 max_by() 집계 함수를 사용하면 셔플·정렬 비용을 획기적으로 줄이고 성능을 크게 개선할 수 있다.

ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ...) 패턴은 Spark에서 전체 셔플과 정렬을 강제하여 5단계 실행 과정을 거쳐야 하며, 처리된 행의 대부분이 버려지는 비효율적인 구조다. 반면 Spark 3.0+에서 도입된 max_by(값, 정렬기준)은 집계 함수이므로 맵 사이드 부분 집계가 가능하여 셔플 전에 각 실행기가 로컬에서 데이터를 크게 줄일 수 있다. 이 함수는 교환법칙·결합법칙·점진적 계산 특성을 가져 분산 처리와 스트리밍 파이프라인에서 특히 강력하다. 다만 Top-N(N>1) 조회, 복합 정렬 기준 타이브레이킹, 또는 순위값 자체가 필요한 경우에는 여전히 윈도우 함수가 필요하다.

단순한 집계 함수 하나가 어떻게 비용이 많이 드는 셔플+정렬 윈도우 연산을 대체할 수 있는지, 그리고 왜 그 대수적 특성이 당신이 생각하는 것보다 훨씬 강력한지 알아봅니다.

Spark를 어느 정도 규모 이상에서 다뤄봤다면 이런 패턴을 수백 번은 작성해봤을 것입니다: "고객별로 가장 최신 레코드를 가져와라." 그리고 아마 매번 같은 무기를 집어들었을 겁니다 — ROW_NUMBER()에 윈도우 파티션을 걸고 rn = 1로 필터링하는 방법이죠.

이 방법은 맞습니다. 그러나 Spark에게 요청할 수 있는 가장 비싼 작업 중 하나이기도 합니다. 이 글에서는 max_by()가 거의 항상 더 나은 선택인 이유, 내부 동작 방식, 그리고 교환법칙(commutative), 결합법칙(associative), 누적 특성(cumulative)이 어떻게 이 함수를 분산 연산의 일급 시민으로 만드는지 보여드리겠습니다.

고전적인 패턴: 윈도우 + ROW_NUMBER

수백만 행이 있는 orders 테이블에서 고객별 가장 최근 주문을 원한다고 가정해봅시다. 교과서적인 접근법은 다음과 같습니다.

-- ROW_NUMBER() 패턴: 비용이 많이 드는 셔플 + 정렬
SELECT *
FROM (
  SELECT *,
    ROW_NUMBER() OVER (
      PARTITION BY customer_id
      ORDER BY order_date DESC
    ) AS rn
  FROM orders
)
WHERE rn = 1

이 방법은 정확합니다. 하지만 Spark가 실제로 이를 실행하기 위해 무엇을 해야 하는지 분석해봅시다.

ROW_NUMBER() 접근법은 전체 셔플 + 정렬을 강제하며, 5단계를 거쳐 모든 단일 행을 처리하지만 그룹당 하나만 남깁니다. 전체 작업의 99.6%가 낭비됩니다.

이것은 5단계이며, 그 중 셔플과 정렬 두 단계는 분산 컴퓨팅에서 가장 비용이 많이 드는 연산입니다. 모든 단일 행이 네트워크를 통해 이동하고, 정렬되고, 행 번호가 부여된 다음… 99.6%가 버려집니다.

데이터 스큐의 숨겨진 비용

한 고객에게 1,000만 개의 주문이 있고 대부분의 고객에게는 5개만 있다면, 윈도우 함수는 단일 파티션에서 1,000만 행을 모두 정렬해야 합니다. 이는 고전적인 데이터 스큐 문제를 일으킵니다 — 하나의 실행기가 모든 작업을 하는 동안 나머지는 유휴 상태입니다. Spark는 윈도우 함수를 부분적으로 집계할 수 없습니다.

윈도우 함수는 부분 집계가 불가능하므로 스큐된 파티션 전체를 단일 실행기에서 정렬해야 합니다. 이것이 중복 제거 작업에서 OOM(메모리 부족)의 1위 원인입니다.

더 나은 방법: max_by()

Spark 3.0+는 max_by(값, 정렬기준)을 도입했습니다 — 최대 정렬기준에 해당하는 값을 반환하는 집계 함수입니다. 같은 쿼리를 다시 작성하면:

-- max_by() 패턴: 부분 집계, 정렬 없음
SELECT
  customer_id,
  max_by(order_id,    order_date) AS order_id,
  max_by(product,     order_date) AS product,
  max_by(amount,      order_date) AS amount,
  max(order_date)                 AS order_date
FROM orders
GROUP BY customer_id

끝입니다. 서브쿼리도, 윈도우도, 필터도 없습니다.

집계 함수인 max_by()는 맵 사이드 부분 집계의 혜택을 받아 5억 행을 네트워크를 통해 전송하기 전에 약 600만 행으로 줄입니다. 셔플 I/O가 27배 감소합니다.

핵심 차이점: max_by()는 윈도우 함수가 아니라 집계 함수이므로, Spark는 맵 사이드에서 부분 집계를 적용할 수 있습니다. 즉, 각 실행기는 데이터가 네트워크를 통해 전송되기 전에 로컬 데이터를 고객당 한 행으로 줄입니다.

max_by()가 데이터를 처리하는 방법 — 단계별

ROW_NUMBER() 윈도우:

  • 모든 행의 전체 셔플
  • O(n log n) 정렬 필요
  • 부분 집계 불가
  • 스큐 시 OOM 발생
  • 5단계 실행
  • 결합 불가

max_by() 집계:

  • 부분 결과만 셔플
  • O(n) — 최대값만 추적
  • 맵 사이드 부분 집계
  • 스큐를 우아하게 처리
  • 3단계 실행
  • 교환법칙 + 결합법칙 성립

벤치마크: "더 빠르다"는 것이 실제로 무엇을 의미하는가

약 200만 명의 고유 고객이 있는 5억 행 이벤트 테이블에서 Databricks 클러스터를 실행한 대표적인 성능 수치입니다.

셔플 감소가 핵심 수치입니다. 48GB의 셔플에서 2GB 미만으로 이동할 때, 단순히 더 빠른 것이 아닙니다 — 다른 작업을 위한 클러스터 리소스를 확보하고, 클라우드 비용을 줄이며, 파이프라인에서 OOM 실패의 주요 원인을 제거합니다.

max_by()를 특별하게 만드는 대수적 특성

이제 흥미로운 부분입니다. max_by()가 부분 집계와 병렬화를 이렇게 효과적으로 수행할 수 있는 이유는 세 가지 수학적 특성 덕분입니다. 이것들은 단순한 학문적 개념이 아닙니다 — Spark의 Catalyst 옵티마이저가 실행 계획을 어떻게 재배열하고 최적화할 수 있는지를 직접 결정합니다.

1. 교환법칙 — f(A, B) = f(B, A)

레코드를 처리하는 순서가 결과에 영향을 미치지 않습니다. 행 A와 행 B를 먼저 비교하든, 행 B와 행 A를 먼저 비교하든 같은 승자를 얻습니다.

왜 중요한가: Spark는 실행기 간 조율 없이 임의의 순서로 파티션을 읽을 수 있습니다.

2. 결합법칙 — f(A∪B) = merge(f(A), f(B))

원하는 방식으로 계산을 그룹화하고 하위 그룹화할 수 있습니다. 이것이 부분 집계를 가능하게 하는 특성입니다 — Spark는 각 실행기에서 로컬 승자를 계산한 다음 실행기 간에 승자를 병합할 수 있습니다.

max_by()는 결합법칙이 성립하므로 Spark는 먼저 로컬 승자를 계산한 다음 8행 대신 3행만 셔플할 수 있습니다. 규모에서는 10억 행과 100만 행을 셔플하는 차이입니다.

형식적으로: max_by(A ∪ B, ord) = merge(max_by(A, ord), max_by(B, ord)). 데이터를 원하는 방식으로 분할하고 부분 결과를 독립적으로 계산한 다음 병합할 수 있습니다. 이것이 바로 Spark가 맵 사이드 결합 → 셔플 → 리듀스 파이프라인에서 사용하는 패턴입니다.

3. 누적 특성(점진적 계산) — f(old ∪ new) = merge(f(old), f(new))

스트리밍과 Delta Lake 증분 파이프라인에서 중요해지는 특성입니다: 처음부터 다시 계산하지 않고 max_by() 결과를 점진적으로 업데이트할 수 있습니다.

새 데이터가 도착하면 전체 테이블을 다시 스캔할 필요가 없습니다. 새 배치의 max_by() 결과를 기존 저장된 결과와 비교하고 승자를 유지하기만 하면 됩니다. 이것이 max_by()를 Structured Streaming과 Delta Live Tables에 자연스럽게 적합하게 만듭니다.

ROW_NUMBER()가 이를 할 수 없는 이유

윈도우 함수는 결합법칙이 성립하지 않습니다. 일부에 대해 ROW_NUMBER()를 계산하고 다른 부분과 병합할 수 없습니다 — 번호 매기기는 전체 정렬된 파티션이 있어야 합니다. 즉, 모든 증분 업데이트는 전체 윈도우를 완전히 재계산해야 합니다.

Databricks에서의 완전한 실용 예제

-- 구방식 — 윈도우 (비용이 많이 드는 셔플 + 정렬)
WITH ranked AS (
  SELECT *,
    ROW_NUMBER() OVER (
      PARTITION BY customer_id
      ORDER BY event_timestamp DESC
    ) AS rn
  FROM catalog.schema.customer_events
)
SELECT
  customer_id, event_type, event_timestamp,
  page_url, device, session_id, geo_region
FROM ranked
WHERE rn = 1

-- 신방식 — max_by (부분 집계, 정렬 없음)
SELECT
  customer_id,
  max_by(event_type,      event_timestamp) AS event_type,
  max(event_timestamp)                     AS event_timestamp,
  max_by(page_url,        event_timestamp) AS page_url,
  max_by(device,          event_timestamp) AS device,
  max_by(session_id,      event_timestamp) AS session_id,
  max_by(geo_region,      event_timestamp) AS geo_region
FROM catalog.schema.customer_events
GROUP BY customer_id

PySpark 동등 코드

from pyspark.sql import functions as F
df = spark.table("catalog.schema.customer_events")
result = (
    df.groupBy("customer_id")
    .agg(
        F.max_by("event_type",  "event_timestamp").alias("event_type"),
        F.max("event_timestamp").alias("event_timestamp"),
        F.max_by("page_url",    "event_timestamp").alias("page_url"),
        F.max_by("device",      "event_timestamp").alias("device"),
        F.max_by("session_id",  "event_timestamp").alias("session_id"),
        F.max_by("geo_region",  "event_timestamp").alias("geo_region"),
    )
)
result.display()

여전히 ROW_NUMBER()가 필요한 경우

공정하게 말씀드리면, max_by()가 작동하지 않는 경우가 있습니다.

그룹당 Top-N (N > 1): 고객별로 가장 최근 주문 3개가 필요하다면, max_by()는 단 하나의 최고 결과만 제공합니다. 윈도우 함수 또는 LATERAL VIEW explode와 slice(sort_array(collect_list(...)))를 조합해야 합니다.

보조 정렬이 있는 타이브레이킹: 정렬 기준 컬럼에 타이가 있고 결정론적 타이브레이커가 필요한 경우(예: ORDER BY event_timestamp DESC, event_id DESC), 복합 정렬 기준을 구성해야 합니다 — Spark는 필드 단위로 struct를 비교하므로 struct(event_timestamp, event_id)를 정렬 인자로 사용하는 방식으로 처리할 수 있습니다.

순위값 자체에 대한 접근: 다운스트림 로직에 실제 행 번호 값이 필요한 경우, 윈도우 함수가 유일한 옵션입니다.

타이브레이킹을 위한 struct 트릭

max_by()에서 정렬 인자로 struct를 전달하여 다중 컬럼 정렬을 처리할 수 있습니다:

max_by(value_col, struct(timestamp_col, id_col))

Spark는 struct를 사전식으로 비교하므로 정렬 연산 없이 ORDER BY timestamp DESC, id DESC 의미론을 제공합니다.

TL;DR — 의사결정 프레임워크

대부분의 중복 제거 시나리오(N=1)에서는 max_by()가 올바른 선택입니다. ROW_NUMBER()는 Top-N 및 순위 접근 사용 사례를 위해 남겨두세요.

마치며

ROW_NUMBER() WHERE rn = 1 패턴은 Databricks에서 제가 가장 자주 보는 성능 안티패턴 중 하나입니다. 틀린 것은 아닙니다 — 단지 그 작업에 비해 지나치게 과대하게 설계되었을 뿐입니다. 최대값을 찾기 위해 전체 정렬을 사용하는 것은, 꽃 한 송이를 따기 위해 불도저를 사용하는 것과 같습니다.

max_by()는 이 문제를 위해 만들어졌습니다. 그 대수적 특성 — 교환법칙, 결합법칙, 점진적 계산 가능성 — 은 단순한 이론적 장점이 아닙니다. Spark가 부분 집계, 최소 셔플, 정렬 없이 실행할 수 있는 이유입니다. 이것은 직접적으로 더 빠른 작업, 낮은 비용, 그리고 규모에서 무너지지 않는 파이프라인으로 이어집니다.

다음에 ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ...) = 1을 작성하려고 할 때, 멈추세요. 대신 max_by()를 사용하세요. 클러스터가 감사할 것입니다.
