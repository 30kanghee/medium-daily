How I Built a Production-Grade Metric Logging System for Apache Spark (PySpark) and Created a Web UI Dashboard

One Step Analytics (Anurag Sharma) · 3 min read · 원문 · freedium

요약

Apache Spark용 프로덕션 수준의 메트릭 로깅 시스템을 구축하고 React 기반 웹 UI 대시보드로 시각화한 실전 구현기.

표준 Spark UI는 실시간 디버깅에는 뛰어나지만 이력 분석, 커스텀 비즈니스 KPI 추적, 팀 전체 가시성 확보에는 한계가 있다. 저자는 PySpark Accumulator와 커스텀 Scala SparkListener를 활용해 시스템 메트릭과 비즈니스 메트릭을 모두 수집하고, Kafka를 통해 Delta Lake로 전송하는 파이프라인을 구축했다. 백엔드는 FastAPI로, 프론트엔드는 React + Recharts로 구현해 스테이지별 소요 시간, 셔플 크기, 비즈니스 KPI를 시각화했다. 이 시스템 도입 후 디버깅 시간이 60~70% 단축되었으며, 성능 저하와 데이터 스큐를 조기에 탐지할 수 있게 됐다. 핵심 교훈은 Accumulator로 시작해 SparkListener로 발전시키고, 시스템 메트릭과 비즈니스 메트릭을 결합하며, 히스토리 비교가 실시간보다 더 가치 있다는 점이다.

대규모 데이터 파이프라인에서 표준 Spark UI는 실시간 디버깅에는 탁월하지만, 이력 분석, 커스텀 비즈니스 메트릭, 트렌드 추적, 알림, 팀 전체 가시성 확보에는 부족합니다. 저는 PySpark 애플리케이션에 특화된 완전한 메트릭 로깅 및 모니터링 시스템을 구축하고, 그 위에 모던 웹 UI를 얹었습니다.

직면했던 문제

  • 실행 간 잡/스테이지 메트릭의 영속적인 이력이 없었음.
  • 커스텀 비즈니스 KPI 추적이 어려웠음 (예: 고객 세그먼트별 처리 레코드 수, 데이터 유형별 에러율).
  • 팀원들이 Spark UI, 로그, 모니터링 도구 사이를 전환하느라 시간을 낭비함.
  • 성능 회귀 탐지와 리소스 최적화가 필요했음.

전체 아키텍처

핵심 흐름:

  1. Spark 내부에서 메트릭을 수집합니다 (시스템 + 커스텀 + 비즈니스).
  2. 중앙 저장소로 전송합니다.
  3. Python 백엔드가 데이터를 제공합니다.
  4. React UI가 풍부한 시각화를 제공합니다.

1단계: PySpark에서 메트릭 수집

옵션 A: Accumulator를 이용한 커스텀 비즈니스 메트릭 (Python)

from pyspark.sql import SparkSession
from pyspark.accumulators import AccumulatorParam
import json
import time

class MetricAccumulator(AccumulatorParam):
    def zero(self, initialValue):
        return initialValue

    def addInPlace(self, v1, v2):
        # Merge two metric dicts
        for k, v in v2.items():
            if k in v1:
                if isinstance(v, (int, float)):
                    v1[k] += v
                else:
                    v1[k] = v  # overwrite for latest
            else:
                v1[k] = v
        return v1

spark = SparkSession.builder.appName("Metrics").getOrCreate()

# Register accumulator
metrics_acc = spark.sparkContext.accumulator({}, MetricAccumulator())

def record_metric(key: str, value):
    current = metrics_acc.value
    current[key] = current.get(key, 0) + value if isinstance(value, (int, float)) else value
    metrics_acc.add({key: value})

트랜스포메이션에서의 사용 예 (foreachPartition을 이용해 효율 높이기):

def process_partition(iterator):
    local_metrics = {"records_processed": 0, "errors": 0}
    for row in iterator:
        try:
            # process row
            local_metrics["records_processed"] += 1
        except Exception:
            local_metrics["errors"] += 1
    # Send to accumulator (thread-safe)
    metrics_acc.add(local_metrics)
    yield ...  # processed data

df.rdd.foreachPartition(process_partition)

옵션 B: 커스텀 SparkListener (Scala)

SparkListener는 Scala/Java로 구현됩니다. PySpark에서는 경량 Scala 리스너 JAR을 작성해 연결하는 것이 모범 사례입니다.

Scala 리스너 예시 (JAR로 컴파일):

// CustomMetricsListener.scala
package com.example.spark

import org.apache.spark.scheduler._
import org.apache.spark.sql.SparkSession
import java.time.Instant

class CustomMetricsListener extends SparkListener {
  override def onStageCompleted(stageCompleted: SparkListenerStageCompleted): Unit = {
    val info = stageCompleted.stageInfo
    val metrics = Map(
      "app_id" -> SparkSession.getActiveSession.map(_.sparkContext.applicationId),
      "stage_id" -> info.stageId,
      "num_tasks" -> info.numTasks,
      "stage_duration_ms" -> info.completionTime.getOrElse(0L),
      "input_bytes" -> info.taskMetrics.inputMetrics.bytesRead,
      "shuffle_write_bytes" -> info.taskMetrics.shuffleWriteMetrics.bytesWritten,
      "timestamp" -> Instant.now().toEpochMilli
    )
    // Push to Kafka, HTTP endpoint, or Delta Lake
    sendMetrics(metrics)
  }

  private def sendMetrics(metrics: Map[String, Any]): Unit = {
    // Example: HTTP post or Kafka producer
    println(s"Metrics: $metrics")  // Replace with real sink
  }
}

PySpark에서 리스너 연결하기:

spark = (SparkSession.builder
         .appName("MyApp")
         .config("spark.extraListeners", "com.example.spark.CustomMetricsListener")
         .config("spark.driver.extraClassPath", "/path/to/your-listener.jar")
         .config("spark.executor.extraClassPath", "/path/to/your-listener.jar")
         .getOrCreate())

2단계: 전송 계층

높은 처리량을 위해 Kafka를 사용했습니다:

# In the listener sink or a separate consumer
from kafka import KafkaProducer

producer = KafkaProducer(bootstrap_servers='kafka:9092')  # your kafka broker

def send_to_kafka(topic: str, metrics: dict):
    producer.send(topic, json.dumps(metrics).encode('utf-8'))

대안: 드라이버에서 Delta Lake로 직접 쓰기.

3단계: 수집 및 저장 서비스 (Python)

경량 FastAPI 서비스가 Kafka에서 소비하여 Delta Lake에 씁니다.

# ingestion_service.py
from fastapi import FastAPI
from delta import *
import pyspark
from datetime import datetime

# Spark session for ingestion
spark = (SparkSession.builder
         .appName("ingestion_service")
         .getOrCreate())

app = FastAPI()

@app.post("/metrics")
async def receive_metrics(metrics: dict):
    df = spark.createDataFrame([metrics])
    df.write.format("delta").mode("append").partitionBy("date").save("/delta/spark_metrics")
    return {"status": "ok"}

저장소 스키마 예시 (Delta Lake):

  • spark_apps: 앱 수준 요약
  • spark_stages: 상세 스테이지 메트릭
  • business_metrics: 차원(팀, 도메인 등)을 포함한 커스텀 KPI

프론트엔드 (React + TypeScript + Recharts/ECharts):

빠른 성과를 위해 Delta Lake / ClickHouse에 Grafana를 직접 연결하기도 했습니다.

주요 과제와 해결책

  • 높은 카디널리티 및 볼륨 — 수집 시점에 메트릭을 집계하고, 장기 테이블에는 시간 기반 파티셔닝과 다운샘플링 적용.
  • 리스너 오버헤드 — 전송을 비동기/배치 처리로 전환.
  • PySpark + Scala 브릿지 — 리스너는 최소화하고 무거운 로직은 Python에서 처리.
  • 스큐 탐지 — 태스크 소요 시간 분산이 높은 스테이지에 플래그를 추가하는 커스텀 로직 구현.
  • 스키마 진화 — Delta Lake의 schema-on-read 유연성 활용.

달성한 결과

  • 디버깅 시간 60~70% 단축.
  • 성능 문제와 데이터 스큐의 조기 탐지.
  • 더 나은 리소스 할당과 비용 절감.
  • 데이터 팀과 플랫폼 팀을 위한 단일 진실 공급원 확보.

얻은 교훈

  • Accumulator로 간단하게 시작하고 → 점차 완전한 SparkListener로 발전시키세요.
  • 시스템 메트릭(셔플, 스필, GC)과 비즈니스 메트릭을 결합하세요.
  • 영속성 + 이력 비교가 실시간만의 가치보다 더 큽니다.
  • 전체 요구사항의 80%는 Grafana로 충족하고, 세련된 경험과 워크플로우에는 커스텀 React UI를 사용하세요.
