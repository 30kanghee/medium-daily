Why Your Spark Job Is Slow: The Data Skew Problem Nobody Warns You About

Nitesh Thakur · 7 min read · 원문 · freedium

요약

Spark 작업이 99% 완료 상태에서 멈추는 진짜 이유는 데이터 쏠림(skew)이며, 클러스터 설정이 아닌 데이터 분포 문제이므로 솔팅·브로드캐스트 조인·AQE 등 데이터 구조적 해법이 필요하다.

클러스터가 정상처럼 보이고 메모리도 충분한데 Spark 작업이 99%에서 수십 분째 멈추는 현상은 데이터 쏠림(data skew) 때문입니다. 특정 파티션 하나가 나머지 전체의 데이터를 합친 것보다 훨씬 많은 데이터를 처리하는 구조적 문제입니다. Spark UI의 태스크 메트릭을 확인해 한 태스크가 다른 태스크보다 수십 배 더 오래 걸린다면 쏠림이 원인입니다. 집계 연산의 쏠림은 솔팅(salting)으로 핫 키를 여러 파티션에 분산시켜 해결하고, 조인 쏠림은 작은 테이블의 브로드캐스트 조인, 핫 키 격리 처리, 또는 AQE 활성화로 해결합니다. 핵심 교훈은 데이터 쏠림은 클러스터 자원 부족이 아닌 데이터 분포 문제이므로, 더 많은 익스큐터를 추가하는 것은 해결책이 되지 않는다는 것입니다.

엔지니어들이 인프라 문제로 착각하는 특정한 종류의 Spark 작업 고통이 있습니다. 클러스터는 정상으로 보입니다. 익스큐터 메모리가 부족하지 않습니다. 네트워크 처리량도 정상입니다. 그런데도 3분 걸려야 할 작업이 99% 완료 상태에서 40분째 멈춰 있고, 끝내기를 거부하는 단 하나의 태스크에 발목이 잡혀 있습니다.

아마도 이런 순간 Spark UI를 멍하니 바라본 적이 있을 것입니다. 199개의 태스크가 녹색으로 완료되고 단 하나의 태스크 — 항상 같은 것, 보통 3단계, 항상 마지막 것 — 가 다른 모든 것과는 완전히 다른 규모의 데이터를 처리하는 것처럼 보이는 속도로 움직입니다. 왜냐하면 실제로 그렇기 때문입니다.

이것이 데이터 쏠림(data skew)입니다. Spark 워크로드에서 가장 흔한 성능 저하 요인 중 하나이며, 엔지니어들이 먼저 확인하는 것들 — 익스큐터 크기 조정, 셔플 파티션 수, 메모리 구성 — 이 거의 원인이 아닙니다. 그 설정들도 중요하지만, 근본 원인이 하나의 파티션에 1,000만 개의 행이 있고 나머지 199개에 각각 50,000개씩 있는 경우에는 도움이 되지 않습니다.

쏠림이 왜 발생하는지, 어떻게 확인하는지, 어떤 원인에 어떤 해결책이 맞는지 이해하는 것이 3분에 끝나는 Spark 작업과 파이프라인을 한 시간 동안 지연시키는 작업을 구분합니다.

Spark가 쏠림을 자동으로 "수정"할 수 없는 이유

진단에 들어가기 전에, Spark가 이것을 투명하게 처리하지 못하는 이유를 이해하는 것이 가치 있습니다. 짧은 답변은 Spark가 키에 의해 데이터를 파티션화하며, 키가 균일하게 분포되지 않으면 파티션도 그렇지 않다는 것입니다. Spark는 데이터를 검사하지 않고는 주어진 키 분포가 예상되는 비즈니스 로직인지 쏠림 문제인지 알 수 없습니다.

셔플이 발생할 때 — groupBy, join, reduceByKey, 또는 같은 키를 가진 데이터를 함께 위치시켜야 하는 모든 연산 중에 — Spark는 각 행의 키를 해시해 어느 파티션에 속하는지 결정합니다. 같은 키를 가진 모든 행이 같은 파티션에 들어갑니다. 데이터셋의 40%가 키 "US"를 가지고 있고 그것이 파티션 7로 해시된다면, 파티션 7이 전체 데이터의 40%를 얻습니다. 하나의 익스큐터가 작업의 40%를 처리합니다. 나머지 모두가 기다립니다.

이것은 버그가 아닙니다. 해시 파티셔닝이 작동하는 방식의 직접적인 결과입니다. 해결책은 Spark의 파티셔너를 바꾸는 것이 아닙니다 — Spark에게 더 균등하게 파티션할 수 있는 데이터를 제공하거나, 셔플이 구조화되는 방식을 바꾸는 것입니다.

다른 것을 가정하기 전에 쏠림을 확인하는 방법

느린 Spark 작업에는 여러 가능한 원인이 있습니다 — 불충분한 병렬성, GC 압력, 디스크 스필, 네트워크 병목, 잘못된 브로드캐스트 조인 선택. 문제를 쏠림으로 처리하기 전에 실제로 쏠림인지 확인하세요. Spark UI가 이것을 간단하게 만들어줍니다.

느린 단계의 단계 상세를 엽니다. 태스크 메트릭 테이블을 확인하세요. 지속 시간으로 정렬하세요. 이런 분포를 보인다면:

Task 0:   1m 58s    — 48,231 행, 12MB
Task 1:   2m 03s    — 51,004 행, 13MB
...
Task 198: 2m 11s    — 49,887 행, 12MB
Task 199: 44m 37s   — 4,021,445 행, 1.1GB

그것이 쏠림입니다. 하나의 태스크가 다른 모든 태스크보다 80배 더 많은 데이터를 처리하고 있습니다. 익스큐터 크기를 늘린다고 이것이 해결되지 않습니다 — 필요 없는 199개 태스크에 더 많은 리소스를 주고 있는 동안, 실제로 필요한 하나의 태스크는 여전히 단일 익스큐터에 제약받습니다.

다른 신호: 태스크당 셔플 읽기 메트릭을 확인하세요. 하나의 태스크가 다른 것들보다 훨씬 높은 셔플 읽기 크기를 보인다면, 쏠림은 셔플 중에 발생했습니다. 태스크들이 비슷한 셔플 읽기 크기를 보이지만 완전히 다른 지속 시간을 보인다면, 쏠림은 처리 시간에 있습니다 — 잠재적으로 핫 키에 적용되는 UDF나 복잡한 변환.

코드에서 셔플이 발생하기 전에 직접 쏠림을 정량화할 수 있습니다:

# groupBy나 join 전에 키 분포 확인
df.groupBy("country_code") \
  .count() \
  .orderBy("count", ascending=False) \
  .show(20)

상위 키가 10번째 키보다 10배 더 많은 행을 가지고 있다면, 쏠림이 있습니다. 100배라면, 심각한 쏠림이 있습니다. 규모는 해결책이 얼마나 공격적이어야 하는지를 알려줍니다.

솔팅 해결책: 핫 키 분산

집계 연산 — groupBy, sum, count, avg — 에서 키 쏠림의 표준 해결책은 솔팅(salting)입니다. 아이디어는 간단합니다: 쏠린 키에 무작위 접미사를 추가해 여러 파티션에 인위적으로 분산시키고, 집계를 수행한 다음, 부분 결과를 병합합니다.

다음은 패턴입니다:

from pyspark.sql import functions as F

# 1단계: 솔트 버킷 수는? 쏠림 심각도에 맞게 설정
SALT_FACTOR = 10
# 2단계: 모든 행에 무작위 솔트 추가
df_salted = df.withColumn(
    "salted_key",
    F.concat(
        F.col("country_code"),
        F.lit("_"),
        (F.rand() * SALT_FACTOR).cast("int").cast("string")
    )
)
# 3단계: 솔팅된 키에 부분 집계 - 작업 분산
partial = df_salted.groupBy("salted_key").agg(
    F.sum("revenue").alias("partial_revenue"),
    F.count("*").alias("partial_count")
)
# 4단계: 솔트 제거 후 최종 집계
final = partial.withColumn(
    "country_code",
    F.split(F.col("salted_key"), "_")[0]
).groupBy("country_code").agg(
    F.sum("partial_revenue").alias("total_revenue"),
    F.sum("partial_count").alias("total_count")
)

하나의 파티션으로 가던 핫 키 "US"는 이제 US_0, US_1, US_2, ..., US_9 — 하나 대신 열 개의 파티션으로 분산됩니다. 각 파티션은 전체 데이터의 약 4%를 얻게 됩니다, 40% 대신. 45분 걸리던 단계가 이제 약 5분 걸립니다.

솔트 계수는 쏠림 심각도와 일치해야 합니다. 평균보다 10배 더 많은 데이터를 가진 키는 균등화하기 위해 최소 10의 솔트 계수가 필요합니다. 100배 더 많은 데이터를 가진 키는 50~100이 필요합니다. 너무 낮으면 핫 파티션이 여전히 다른 것들보다 크지만 — 개선됐지만 해결되지 않습니다. 너무 높으면 불필요한 추가 집계 작업을 하게 됩니다.

솔팅이 할 수 없는 것: 조인에는 도움이 되지 않습니다. 조인 키에 무작위로 솔팅할 수 없습니다 — 솔팅 후 조인을 위해 함께 위치해야 하는 행들이 더 이상 매칭되지 않습니다. 조인에는 다른 접근 방식이 필요합니다.

쏠린 조인 해결: 브로드캐스트, 격리, AQE

조인에서 쏠림이 진짜 고통스러워지는 이유는 선택지가 더 제한적이기 때문입니다.

브로드캐스트 조인 (작은 테이블용)

조인의 한쪽이 익스큐터 메모리에 맞을 만큼 작다면, 셔플을 완전히 건너뛰세요. Spark는 작은 테이블의 복사본을 모든 익스큐터에 보내고, 각 익스큐터가 큰 테이블의 파티션과 로컬에서 조인합니다. 셔플이 없다는 것은 쏠림 증폭이 없다는 것을 의미합니다.

from pyspark.sql.functions import broadcast

result = large_df.join(broadcast(small_df), on="customer_id")

Spark의 기본 브로드캐스트 임계값은 10MB (spark.sql.autoBroadcastJoinThreshold)입니다. 룩업 테이블이 더 크지만 여전히 관리 가능하다면 높이세요 — 100MB나 200MB는 현대 익스큐터 힙 크기에서 합리적입니다:

spark.sql.autoBroadcastJoinThreshold=209715200  # 200MB

브로드캐스트 조인은 크기 조건이 충족될 때 Spark에서 가장 효과적인 단일 조인 최적화입니다. 조인의 한쪽에 차원 테이블이나 룩업 테이블이 있다면, 다른 것을 탐색하기 전에 메모리에 맞는지 확인하고 브로드캐스트하세요.

핫 키 격리

조인의 양쪽이 모두 크고 브로드캐스트가 불가능할 때, 한 가지 접근법은 쏠린 키를 별도로 처리하는 것입니다:

# 큰 테이블을 쏠린 부분과 쏠리지 않은 부분으로 분리
skewed_df = large_df.filter(F.col("country_code") == "US")
normal_df = large_df.filter(F.col("country_code") != "US")

# 각 부분을 별도로 조인
skewed_result = skewed_df.join(other_df.filter(F.col("country_code") == "US"), on="country_code")
normal_result = normal_df.join(other_df.filter(F.col("country_code") != "US"), on="country_code")

# 결과 합치기
final = skewed_result.union(normal_result)

이것은 장황해 보이지만 효과적입니다. 쏠린 부분은 자체 조인을 얻고, 파티션 수와 설정을 독립적으로 조정할 수 있습니다. 쏠리지 않은 부분은 표준 설정을 사용합니다. 마지막의 union은 저렴합니다.

적응형 쿼리 실행 (AQE)

Spark 3.0에서 도입한 적응형 쿼리 실행(Adaptive Query Execution)은 런타임 중에 자동으로 쏠림을 감지하고 처리할 수 있습니다. 다음으로 활성화하세요:

spark.sql.adaptive.enabled=true
spark.sql.adaptive.skewJoin.enabled=true

AQE가 활성화되면, Spark는 spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes(기본 256MB)보다 큰 파티션을 감지하고 더 작은 서브태스크로 분할해, 각 서브태스크를 다른 쪽의 해당 파티션의 복제본과 조인합니다.

AQE는 마법이 아닙니다 — 여전히 런타임에 쏠림을 감지해야 하며, 감지 임계값은 데이터 특성에 맞게 조정해야 합니다. 그러나 Spark 3.x를 실행하는 팀에게는 AQE를 쏠림 조인 감지와 함께 활성화하는 것이 코드 변경 없이 일반적인 쏠림 패턴을 처리하는 저노력 첫 번째 단계입니다.

spark.sql.adaptive.enabled=true
spark.sql.adaptive.skewJoin.enabled=true
spark.sql.adaptive.skewJoin.skewedPartitionFactor=5
spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes=268435456

skewedPartitionFactor 설정은 파티션이 중앙값 파티션 크기의 5배보다 크면 쏠린 것으로 처리한다는 의미입니다. 쏠림이 덜 극단적이지만 여전히 지연자를 유발한다면 이것을 낮추세요.

모든 것을 더 나쁘게 만드는 설정

하나의 설정 변경이 쏠림을 극적으로 악화시키며 생각 없이 Spark 설정에 복사됩니다: spark.sql.shuffle.partitions를 너무 낮게 설정하는 것.

기본값은 200입니다. 작은 데이터셋에는 200이 괜찮거나 너무 높기도 합니다. 쏠린 키가 있는 대규모 데이터셋에는 200개의 파티션이 전체를 의미합니다 — 쏠린 키가 각 파티션의 데이터에서 더 큰 몫을 얻는다는 것을 의미합니다. 셔플 파티션을 늘리면 해시 함수에 키를 분산할 더 많은 버킷이 생겨, 각 파티션의 평균 크기와 핫 키가 줄 수 있는 상대적 피해가 줄어듭니다.

대규모 작업의 경우 2,000에서 4,000개의 셔플 파티션이 합리적인 범위입니다. AQE가 활성화되면 Spark는 런타임에 작은 파티션을 자동으로 합칩니다, 그래서 이 숫자를 높게 설정하는 비용이 줄어듭니다. 작은 AQE 파티션을 결합하는 설정은:

spark.sql.adaptive.coalescePartitions.enabled=true
spark.sql.adaptive.advisoryPartitionSizeInBytes=67108864  # 파티션당 64MB 목표

AQE 병합이 활성화되면 셔플 파티션을 4,000으로 설정하는 것은 거의 비용이 없습니다 — 작은 파티션이 병합됩니다 — 하지만 쏠림이 분산될 더 많은 여지를 줍니다.

확인할 또 다른 설정: spark.default.parallelism은 RDD 연산의 병렬성을 제어합니다(DataFrame 셔플 파티션과 구별됩니다). 클러스터가 완전히 활용되도록 총 익스큐터 코어 수의 최소 2~3배로 설정하세요.

쏠림은 데이터 문제이지 클러스터 문제가 아닙니다

Spark 작업이 느릴 때 직관적으로 클러스터 설정을 조정하려 합니다 — 더 많은 익스큐터, 더 큰 익스큐터 메모리, 더 높은 병렬성. 그 설정들은 불충분한 리소스로 인한 문제를 해결합니다. 불균등한 데이터 분포로 인한 문제는 해결하지 못합니다.

199개의 태스크가 2분 안에 끝나고 하나가 45분 걸릴 때, 클러스터는 충분한 이상의 리소스를 가지고 있습니다. 문제는 그 모든 리소스가 유휴 상태로, 데이터의 불공평한 몫을 얻은 단일 파티션을 기다리고 있다는 것입니다. 더 많은 익스큐터를 추가해도 그 파티션을 분할하지 않습니다. 이미 거기서 기다리는 것들 옆에 더 많은 유휴 익스큐터를 추가할 뿐입니다.

해결책은 항상 데이터 구조적입니다: 핫 키를 솔팅하고, 조인의 작은 쪽을 브로드캐스트하고, 쏠린 키를 별도 처리를 위해 격리하거나, AQE가 런타임에 처리하도록 합니다. 구체적인 기법은 쏠림이 집계인지 조인인지, 쏠린 키를 미리 식별할 수 있는지에 따라 달라집니다.

구성을 변경하기 전에 Spark UI에서 태스크 메트릭을 확인하세요. 지속 시간 분포가 균일하다면 리소스 문제입니다. 하나의 태스크가 이상값이라면 쏠림 문제입니다. 두 가지 다른 진단. 두 가지 완전히 다른 해결책.

45분짜리 태스크는 익스큐터가 약해서 고전하는 것이 아닙니다. 데이터에게 스스로 퍼지라고 아무도 말하지 않았기 때문에 고전하는 것입니다.
