I Crashed a $35,000 Databricks Cluster at 11 PM — And What It Taught Me

Pravash · Towards Data Engineering · 8 min read · 원문 · freedium

요약

새벽 11시 Databricks 클러스터 OOM 장애를 겪은 후 비용을 78% 절감하고 하루 50TB를 처리하는 파이프라인을 구축한 실전 경험담.

저자는 화요일 밤 11시, 500GB 데이터를 처리하려다 Databricks 클러스터가 OOM으로 충돌하여 20개 노드까지 자동 확장된 끝에 $35,000짜리 청구서를 받은 경험을 공유한다. 이 사고를 계기로 클러스터 크기 산정, 자동 확장 설정, Delta Lake 최적화, 비용 관리 전략을 철저히 학습했다. 구체적인 개선 조치로 클러스터 구성을 재설계하고, 파티셔닝 전략을 최적화하며, 파이프라인 아키텍처를 정비한 결과 비용을 78% 줄이고 하루 50TB 처리 파이프라인을 완성했다. 이 글은 단순한 장애 보고서가 아니라 Databricks를 프로덕션에서 운영할 때 반드시 알아야 할 실전 교훈의 모음이다.

그 밤, 모든 것이 무너졌다

화요일 밤 11시였다. 나는 긴 하루를 마치고 막 자리에 앉아 노트북 화면을 응시하고 있었다. Databricks 파이프라인이 500GB의 고객 이벤트 데이터를 처리하는 중이었고, 진행률 표시줄은 천천히 앞으로 나아가고 있었다. 별로 걱정하지 않았다. 이런 작업은 전에도 수도 없이 돌려봤으니까.

그런데 갑자기 Spark UI가 멈췄다. 드라이버 로그에 무시무시한 붉은 글씨가 넘쳐흘렀다.

java.lang.OutOfMemoryError: GC overhead limit exceeded
  at org.apache.spark.sql.execution.joins.SortMergeJoinExec...
ERROR SparkContext: Error initializing SparkContext.
java.lang.OutOfMemoryError: Java heap space

클러스터는 이미 자동 확장 한계인 20개 노드까지 치솟아 있었다. 그리고 그 상태에서도 결국 충돌했다. 다음 날 아침, 클라우드 비용 대시보드를 열었을 때 나는 눈을 의심했다. 단 하룻밤의 실패한 작업에 $35,000가 청구되어 있었다.

이 사고는 내가 지금껏 경험한 가장 비싼 학습이었다. 그리고 동시에 가장 값진 교훈이었다.

무엇이 잘못되었는가: 근본 원인 분석

1. 데이터 스큐(Data Skew) — 보이지 않는 함정

사후 분석을 시작했을 때 가장 먼저 눈에 띈 것은 태스크별 처리 시간의 극단적인 불균형이었다. Spark UI의 Stage 탭을 열어보니 대부분의 태스크가 2~3초 만에 끝나는 반면, 몇몇 태스크는 40분 넘게 실행되다가 OOM으로 죽고 있었다.

원인은 조인 키였다. 우리 데이터셋에는 특정 사용자 ID가 전체 레코드의 30% 이상을 차지하고 있었다. SortMergeJoin을 수행할 때 이 키를 기준으로 셔플이 일어나면서, 단 몇 개의 파티션이 수백만 건의 레코드를 떠안게 되었다. 메모리는 순식간에 포화상태가 되었다.

# 문제가 된 원래 코드
result = large_df.join(dim_df, on="user_id", how="left")

# 스큐 확인
large_df.groupBy("user_id").count().orderBy("count", ascending=False).show(10)

2. 잘못 설정된 자동 확장(Auto-scaling)

우리 클러스터는 최소 2노드, 최대 20노드로 설정되어 있었다. 표면적으로는 합리적으로 보인다. 하지만 문제는 Databricks의 자동 확장이 처리량 기반이 아니라 큐에 쌓인 태스크 수 기반으로 동작한다는 것이다.

스큐된 몇 개의 파티션이 계속 실패를 반복하자, Spark은 "더 많은 리소스가 필요하다"고 판단하고 노드를 계속 추가했다. 그러나 근본 원인인 데이터 스큐가 해결되지 않은 상태에서 노드를 늘리는 것은 아무런 도움이 되지 않았다. 오히려 실패하는 노드 수만 늘어났고, 비용만 기하급수적으로 쌓였다.

3. 브로드캐스트 조인 임계값 미설정

우리가 조인하던 차원 테이블(dim_df)은 실제로 약 50MB짜리 소규모 테이블이었다. Spark이 이것을 브로드캐스트 조인으로 처리했다면 셔플 자체가 발생하지 않았을 것이다. 그러나 기본 설정인 spark.sql.autoBroadcastJoinThreshold가 10MB로 설정되어 있어 브로드캐스트가 일어나지 않았다.

4. 파티션 크기 최적화 부재

데이터를 읽어들일 때 기본 파티션 수를 그대로 사용하고 있었다. 500GB 데이터를 처리할 때 Spark이 생성한 파티션은 200개였는데, 이는 파티션당 평균 2.5GB라는 말이다. 권장 파티션 크기인 128~256MB를 훨씬 초과했고, 각 이그제큐터가 GC 압박에 시달리는 것은 필연적이었다.

비용 78% 절감: 단계별 최적화 전략

1단계: 클러스터 구성 재설계

가장 먼저 한 일은 클러스터 유형을 재검토하는 것이었다. 기존에는 범용 컴퓨팅 인스턴스를 사용하고 있었지만, 우리 워크로드는 메모리 집약적인 대규모 조인과 집계 연산이 핵심이었다. 메모리 최적화 인스턴스(AWS 기준 r5 시리즈, Azure 기준 E 시리즈)로 교체하자 동일한 작업을 훨씬 적은 수의 노드로 처리할 수 있었다.

또한 스팟 인스턴스(Spot/Preemptible)를 워커 노드에 적극적으로 활용하기 시작했다. 드라이버는 안정성을 위해 온디맨드 인스턴스를 유지하되, 워커 노드의 80%를 스팟 인스턴스로 전환했다. 이것만으로 컴퓨팅 비용이 60~70% 감소했다.

{
  "node_type_id": "r5.4xlarge",
  "driver_node_type_id": "r5.2xlarge",
  "autoscale": {
    "min_workers": 2,
    "max_workers": 8
  },
  "aws_attributes": {
    "availability": "SPOT_WITH_FALLBACK",
    "spot_bid_price_percent": 100,
    "first_on_demand": 1
  }
}

2단계: 데이터 스큐 해결 — 솔트 키(Salt Key) 기법

데이터 스큐 문제를 해결하기 위해 솔팅(Salting) 기법을 도입했다. 핵심 아이디어는 간단하다. 한쪽에 쏠린 키를 인위적으로 여러 개의 하위 키로 분산시켜 조인 부하를 균등하게 만드는 것이다.

from pyspark.sql import functions as F

# 솔트 계수 (파티션 수에 맞게 조정)
SALT_FACTOR = 20

# 대용량 테이블에 솔트 키 추가
large_df_salted = large_df.withColumn(
    "salt", (F.rand() * SALT_FACTOR).cast("int")
).withColumn(
    "user_id_salted", F.concat_ws("_", F.col("user_id"), F.col("salt"))
)

# 차원 테이블을 솔트 계수만큼 복제
from pyspark.sql.functions import array, explode, lit

dim_df_replicated = dim_df.withColumn(
    "salt_array", array([lit(i) for i in range(SALT_FACTOR)])
).withColumn(
    "salt", explode("salt_array")
).withColumn(
    "user_id_salted", F.concat_ws("_", F.col("user_id"), F.col("salt"))
).drop("salt_array", "salt")

# 솔트된 키로 조인
result = large_df_salted.join(dim_df_replicated, on="user_id_salted", how="left")

이 방식으로 기존에 40분이 걸리던 스큐 파티션이 2~3분으로 줄었다. 전체 파이프라인 실행 시간이 4시간에서 45분으로 단축되었다.

3단계: 브로드캐스트 조인 적극 활용

소규모 참조 테이블에는 반드시 브로드캐스트 조인을 강제 적용하도록 코드를 수정했다.

# 브로드캐스트 임계값을 늘리거나 hint를 명시적으로 사용
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "100m")

# 또는 조인 힌트로 명시적 제어
from pyspark.sql.functions import broadcast

result = large_df.join(
    broadcast(dim_df),  # 명시적 브로드캐스트
    on="user_id",
    how="left"
)

4단계: 파티션 수 최적화

파티션당 128~256MB를 목표로 repartition을 명시적으로 설정했다. 500GB 데이터라면 최소 2,000개 이상의 파티션이 필요하다.

# 데이터 크기 기반 동적 파티션 계산
def optimal_partitions(df, target_partition_size_mb=200):
    # 샘플링으로 데이터 크기 추정
    sample_count = df.count()
    # 대략적인 메모리 사용량 기반 계산
    estimated_size_mb = sample_count * 0.001  # 레코드당 평균 1KB 가정
    return max(200, int(estimated_size_mb / target_partition_size_mb))

# Shuffle 파티션 수 동적 설정
spark.conf.set("spark.sql.shuffle.partitions", "2000")

# 또는 AQE(Adaptive Query Execution) 활용
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")
AQE(Adaptive Query Execution)는 Spark 3.0부터 도입된 기능으로, 런타임에 파티션 수를 자동으로 최적화한다. 가능하다면 항상 활성화해두라.

Delta Lake 최적화: 숨겨진 성능 보물

Z-Ordering으로 데이터 스킵 극대화

우리 파이프라인은 대부분의 쿼리에서 event_date와 user_id를 필터 조건으로 사용했다. Delta Lake의 Z-Ordering을 이 두 컬럼에 적용하자 데이터 스캔량이 극적으로 줄었다.

-- Delta Lake Z-Ordering 적용
OPTIMIZE events_table
ZORDER BY (event_date, user_id);

-- 데이터 스킵 효과 확인
DESCRIBE DETAIL events_table;

Z-Ordering 적용 전후를 비교하면, 일별 집계 쿼리 기준으로 스캔하는 파일 수가 평균 450개에서 23개로 줄었다. I/O 비용과 시간이 동시에 절감된 것이다.

소형 파일 문제 해결 — OPTIMIZE와 VACUUM

스트리밍 파이프라인이나 자주 업서트(upsert)하는 환경에서는 소형 파일이 급격히 누적된다. 수십만 개의 작은 파일을 읽는 것은 메타데이터 오버헤드만으로도 성능을 심각하게 저하시킨다.

-- 정기적인 최적화 (일배치 후 실행 권장)
OPTIMIZE events_table;

-- 오래된 파일 정리 (7일 보관 기본값)
VACUUM events_table RETAIN 168 HOURS;

-- Python으로 자동화
def optimize_delta_table(spark, table_path, zorder_cols=None):
    if zorder_cols:
        cols_str = ", ".join(zorder_cols)
        spark.sql(f"OPTIMIZE delta.`{table_path}` ZORDER BY ({cols_str})")
    else:
        spark.sql(f"OPTIMIZE delta.`{table_path}`")
    spark.sql(f"VACUUM delta.`{table_path}` RETAIN 168 HOURS")

파티셔닝 전략: 카디널리티와 쿼리 패턴의 균형

Delta Lake 테이블의 파티셔닝은 쿼리 패턴에 맞게 설계해야 한다. 카디널리티가 너무 높은 컬럼(예: user_id)으로 파티셔닝하면 수백만 개의 파티션 디렉토리가 생겨 오히려 성능을 해친다.

우리의 경우 event_date(일 단위)로 파티셔닝하고, 그 안에서 Z-Ordering으로 세부 필터링을 처리하는 전략이 최적이었다.

-- 테이블 생성 시 파티션 설계
CREATE TABLE events_table (
  event_id STRING,
  user_id STRING,
  event_type STRING,
  event_date DATE,
  created_at TIMESTAMP,
  payload MAP<STRING, STRING>
)
USING DELTA
PARTITIONED BY (event_date)
LOCATION 's3://my-bucket/delta/events/';

하루 50TB 파이프라인 아키텍처

전체 아키텍처 개요

비용 위기를 극복한 후 우리는 파이프라인 아키텍처 전체를 재설계했다. 목표는 명확했다. 일일 50TB 데이터를 안정적으로, 저비용으로 처리하는 것이다.

최종 아키텍처는 크게 세 개의 레이어로 구성된다.

  • 수집 레이어: Kafka → Databricks Auto Loader (스트리밍 수집)
  • 처리 레이어: Delta Live Tables (DLT) 기반의 선언적 파이프라인
  • 서빙 레이어: Delta Lake Gold 테이블 → BI 도구 / Feature Store

Auto Loader로 효율적인 데이터 수집

기존에는 S3/ADLS의 특정 경로를 주기적으로 폴링하는 방식으로 데이터를 수집했다. 이 방식은 신규 파일 감지 비용이 높고 중복 처리 위험이 있었다. Databricks Auto Loader로 전환하자 파일 감지 오버헤드가 사라지고 exactly-once 처리가 보장되었다.

from pyspark.sql import functions as F

# Auto Loader를 사용한 스트리밍 수집
df_stream = (
    spark.readStream
    .format("cloudFiles")
    .option("cloudFiles.format", "json")
    .option("cloudFiles.schemaLocation", "/mnt/schema/events")
    .option("cloudFiles.inferColumnTypes", "true")
    .option("cloudFiles.maxFilesPerTrigger", 10000)
    .load("s3://raw-data-bucket/events/")
)

# 처리 후 Delta Lake에 쓰기
query = (
    df_stream
    .withColumn("event_date", F.to_date("created_at"))
    .writeStream
    .format("delta")
    .partitionBy("event_date")
    .option("checkpointLocation", "/mnt/checkpoints/events")
    .trigger(availableNow=True)  # 배치 모드로 실행
    .start("/mnt/delta/events/")
)

Delta Live Tables로 파이프라인 선언적 관리

DLT(Delta Live Tables)는 데이터 품질 검증과 의존성 관리를 선언적으로 처리할 수 있게 해준다. 이전에는 파이프라인 단계 간 의존성을 수동으로 관리했고, 한 단계가 실패하면 하위 단계가 오염된 데이터를 처리하는 문제가 빈번했다.

import dlt
from pyspark.sql import functions as F

@dlt.table(
    name="bronze_events",
    comment="원시 이벤트 데이터 (Auto Loader 수집)",
    table_properties={"quality": "bronze"}
)
def bronze_events():
    return (
        spark.readStream
        .format("cloudFiles")
        .option("cloudFiles.format", "json")
        .load("s3://raw-data-bucket/events/")
    )

@dlt.table(
    name="silver_events",
    comment="정제된 이벤트 데이터",
    table_properties={"quality": "silver"}
)
@dlt.expect_or_drop("valid_user_id", "user_id IS NOT NULL")
@dlt.expect_or_drop("valid_event_type", "event_type IN ('click', 'view', 'purchase')")
def silver_events():
    return (
        dlt.read_stream("bronze_events")
        .withColumn("event_date", F.to_date("created_at"))
        .withColumn("user_id", F.trim(F.col("user_id")))
        .dropDuplicates(["event_id"])
    )

@dlt.table(
    name="gold_daily_summary",
    comment="일별 사용자 행동 요약",
    table_properties={"quality": "gold"}
)
def gold_daily_summary():
    return (
        dlt.read("silver_events")
        .groupBy("event_date", "user_id", "event_type")
        .agg(
            F.count("*").alias("event_count"),
            F.countDistinct("session_id").alias("session_count")
        )
    )

비용 모니터링 자동화

가장 중요한 교훈 중 하나는 비용 모니터링을 코드 레벨에서 자동화해야 한다는 것이다. 장애가 발생한 날 밤, 만약 비용 임계값 알림이 있었다면 $35,000짜리 청구서는 없었을 것이다.

import requests

def send_cost_alert(dbu_used, threshold=100):
    """DBU 사용량이 임계값 초과 시 Slack 알림 발송"""
    if dbu_used > threshold:
        message = f"경고: DBU 사용량 {dbu_used}개 초과 (임계값: {threshold})"
        requests.post(

클러스터 설계 원칙

이 사고를 통해 정립한 클러스터 설계 원칙을 공유한다.

  • 워크로드 유형에 맞는 인스턴스 선택: 메모리 집약적 작업은 메모리 최적화 인스턴스, CPU 집약적 작업은 컴퓨팅 최적화 인스턴스를 사용한다.
  • 자동 확장 상한선은 보수적으로: 최대 노드 수를 필요 이상으로 높게 설정하면 비용 폭탄의 원인이 된다. 실제 워크로드를 프로파일링한 후 적정 상한선을 설정하라.
  • 클러스터 자동 종료 시간 설정: 유휴 클러스터가 밤새 실행되는 것을 막기 위해 자동 종료 시간(Auto Termination)을 반드시 설정한다. 개발 클러스터는 30분, 프로덕션 클러스터는 작업 완료 후 즉시 종료를 권장한다.
  • Spot 인스턴스 + 폴백 전략: 워커 노드의 70~80%를 Spot 인스턴스로 운영하되, 최소 1~2개는 온디맨드로 유지해 스팟 중단 시에도 파이프라인이 완전히 멈추지 않도록 한다.

코드 레벨 최적화 체크리스트

매번 파이프라인을 작성할 때 아래 체크리스트를 반드시 확인한다.

  • 조인 전 데이터 스큐 여부 확인 (groupBy + count + orderBy)
  • 소규모 테이블에 브로드캐스트 조인 적용
  • AQE(Adaptive Query Execution) 활성화
  • 셔플 파티션 수 명시적 설정 (데이터 크기 / 200MB)
  • Delta Lake 테이블에 정기 OPTIMIZE 및 VACUUM 예약
  • Z-Ordering 컬럼을 쿼리 패턴 기반으로 선정
  • 데이터 품질 검증을 파이프라인 내부에 내재화 (DLT expectations)

운영(Ops) 관점의 교훈

기술적인 최적화만큼이나 중요한 것이 운영 체계다.

  • 비용 알림 자동화: Databricks의 예산 알림(Budget Alert) 또는 클라우드 네이티브 비용 알림을 설정해 임계값 초과 시 즉시 팀에 알린다.
  • 스테이징 환경에서 비용 추정: 프로덕션 배포 전 소규모 샘플로 DBU 사용량을 추정하고, 전체 데이터 규모로 외삽해 예상 비용을 계산한다.
  • Unity Catalog로 리소스 거버넌스: Databricks Unity Catalog를 활용하면 팀별, 프로젝트별 컴퓨팅 사용량을 추적하고 할당량을 관리할 수 있다.
  • Photon 엔진 활용: Databricks Photon은 SQL 워크로드와 DataFrame API에 대해 C++ 기반 벡터화 처리를 제공한다. 비용 대비 성능이 크게 향상된다.

그래서, 결과는?

이 모든 최적화를 적용한 결과는 숫자로 명확하게 나타났다.

  • 월간 Databricks 비용: 기존 대비 78% 감소
  • 일일 처리 데이터량: 기존 500GB에서 50TB로 확장
  • 평균 파이프라인 실행 시간: 4시간에서 45분으로 단축
  • 데이터 품질 인시던트: DLT 도입 후 0건

Databricks는 강력하다. 하지만 그 강력함에는 그만한 비용이 따른다. 클러스터를 제대로 이해하지 못한 채 자동 확장에만 의존하는 것은 신용카드를 아이에게 맡기는 것과 다르지 않다. 데이터 엔지니어라면 클러스터 크기 산정, 스큐 탐지, Delta Lake 최적화는 기본 역량으로 갖춰야 한다.
