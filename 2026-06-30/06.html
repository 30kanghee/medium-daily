Automating the Entire Data Engineering Lifecycle With AI: An AI-First Approach to TDLC, SDLC, and Beyond

Nayan Paul · 17분 읽기 · 원문 · freedium

Claude Code의 슬래시 명령어와 자율 서브에이전트로 PySpark + Delta Lake 파이프라인 전체를 자동화하는 AI 우선 접근법

요약

데이터 엔지니어링 프로젝트를 처음부터 구축하는 데는 폴더 구조 생성, 보일러플레이트 코드 작성, 스키마 추론, 점진적 워터마크 처리 등 반복적이고 기계적인 작업들이 수일을 잡아먹는다. 이 글은 Claude Code의 슬래시 명령어와 자율 서브에이전트로 구성된 'SDLC Data Pipeline' 패키지를 소개한다. 이 패키지는 /project-init부터 /checkin까지 7단계의 파이프라인을 통해 PySpark + Delta Lake 프로젝트를 처음 디렉토리 생성부터 git push까지 완전히 자동화한다. AI가 기관의 모범 사례를 코드로 인코딩하고, 인간은 비즈니스 판단이 필요한 세 개의 승인 관문에만 개입하는 방식으로, AI 우선 데이터 엔지니어링이 실제로 어떤 모습인지를 보여준다.

슬래시 명령어와 자율 서브에이전트 패키지가 어떻게 프로덕션급 PySpark + Delta Lake 파이프라인을 스캐폴딩하고, 구축하고, 큐레이션하고, 버전 관리하고, 배포할 수 있는지, 그리고 모든 핵심 결정 지점에 사람이 개입하는 구조.

우리가 해결하려 한 문제

데이터 엔지니어링 프로젝트를 처음부터 구축해본 사람이라면 이 의식이 익숙할 것이다. 폴더 구조를 만들고, SparkSession 팩토리 보일러플레이트를 작성하고, 수집 스크립트를 스텁으로 채우고, 스키마 추론·점진적 워터마크·파티션 전략과 씨름하고, SCD Type 2 머지 로직을 손으로 코딩하고, README를 쓰고, .env를 .gitignore에 넣는 걸 혹시 빠뜨리지 않았나 확인한다.

이 일련의 작업은 대부분 기계적이다. 창의적이지도 않고, 차별화 요소도 아니다. 그런데 매 프로젝트마다 며칠이 사라진다. 더 심각한 문제는 일관성이 없다는 것이다. 같은 팀의 두 엔지니어도 점진적 워터마크 추적을 서로 다르게 구현한다. 한 명은 불량 행을 격리하고, 다른 한 명은 조용히 드롭한다. 한 명은 브론즈 레이어를 ingest_date로 파티션하고, 다른 한 명은 비즈니스 컬럼으로 파티션해 다운스트림 큐레이션 단계를 망가뜨린다.

우리가 답하려 한 질문은 단순했다. mkdir부터 git push까지 전체 파이프라인을 당신의 선택을 알고, 맥락을 유지하며, 일관된 아키텍처를 따르는 프로덕션급 코드를 생성하는 대화형 AI가 이끌 수 있다면?

그 답이 SDLC Data Pipeline이다. Claude Code의 슬래시 명령어와 자율 서브에이전트 집합으로 PySpark + Delta Lake 프로젝트의 기술 전달 라이프사이클(TDLC), 소프트웨어 개발 라이프사이클(SDLC), 그리고 DevOps 체크인 프로세스를 자동화한다.

우리가 만든 것: 7단계 AI 파이프라인

이 패키지는 각각 Claude Code에서 입력하는 슬래시 명령어로 구성된 7단계로 프로젝트를 이끈다.

/project-init → /ingest → /curate → /cdc → /data-product → /test-harness → /checkin

각 명령어는 질문을 던지는 상태 기반 대화형 프론트엔드다. 각 명령어 뒤에는 스키마를 읽고, 코드를 생성하고, 파일을 작성하는 하나 이상의 자율 서브에이전트가 있다. 이들 사이를 잇는 것은 내구성 있는 상태 파일 .sdlc/state.json으로, 모든 것을 앞으로 전달하여 같은 질문을 두 번 묻는 일이 없다.

각 단계가 무엇을 하는지, 왜 존재하는지, AI 우선 설계 관점에서 무엇이 흥미로운지 살펴보자.

1단계: /project-init — 기초 다지기

새 데이터 프로젝트는 구조에서 시작한다. /project-init 명령어는 다섯 가지를 묻는다. 프로젝트 이름, 설명, 기술 스택 확인(기본값 pyspark-delta), 기본 출력 경로, 팀 이름. 딱 이것뿐이다. 이 다섯 가지 답변으로 project-init-agent 서브에이전트에게 넘기고, 에이전트는 전체 프로젝트 트리를 스캐폴딩한다.

my-pipeline/
├── README.md
├── .env.example
├── .gitignore
├── requirements.txt
├── conf/
│   └── project.yaml
├── src/
│   ├── __init__.py
│   └── common/
│       ├── spark.py       ← 모든 단계가 공유하는 SparkSession 팩토리
│       └── config.py
├── tests/
│   └── __init__.py
└── .sdlc/
    └── state.json         ← 전체 파이프라인의 내구성 있는 컨텍스트 객체

모든 다운스트림 단계는 src/common/spark.py를 임포트한다. Delta Lake 확장이 활성화된 SparkSession을 빌드하는 재사용 가능한 헬퍼다. 단일 공유 팩토리를 두면 프로젝트에서 생성되는 모든 스크립트가 일관성을 갖는다. 동일한 카탈로그 설정, 동일한 Delta 확장, 환경 변수를 통한 동일한 확장 패턴.

state.json 파일은 여기서 프로젝트 메타데이터와 다섯 개 파이프라인 단계가 모두 pending으로 초기화된다. 이 파일이 파이프라인의 기억이다. 단계가 완료되면 아티팩트, 선택 사항, 결과물을 이 파일에 기록한다. 이후 단계들은 사용자에게 반복해서 묻지 않고 이 파일에서 맥락을 복원한다.

에이전트가 협상 없이 강제하는 설계 규칙이 하나 있다. 기존 .sdlc/state.json을 절대 덮어쓰지 않는다. 이미 상태가 있는 디렉토리에서 /project-init을 실행하면 에이전트가 멈추고 프로젝트가 이미 초기화되었다고 보고한다. 반만 구축된 파이프라인의 상태를 초기화 재실행으로 날려버리는 흔한 실수를 막는 장치다.

AI 우선 관점에서의 핵심 통찰은 프로젝트 초기화가 창의적 작업이 아니라는 것이다. 이것은 계약이다. 모든 단계가 동의하는 일련의 관행. 그 계약을 사람의 기억이 아닌 서브에이전트의 지침으로 인코딩함으로써, 모든 프로젝트는 동일한 기반에서 시작한다.

2단계: /ingest — 소스에서 브론즈 Delta Lake로

수집 단계에서 진정한 다양성이 시작된다. 소스는 ADLS/S3/로컬 파일, JDBC 데이터베이스, Kafka 토픽, REST API, 또는 Snowflake 테이블일 수 있다. 로드는 배치 또는 스트리밍, 전체 또는 점진적일 수 있다. 점진적 전략은 소스마다 다르다. JDBC에는 타임스탬프 워터마크, 파일 소스에는 Auto Loader, Kafka에는 오프셋 체크포인트, REST API에는 커서 매개변수.

/ingest 명령어는 선택에 따라 분기하는 질문 시리즈를 던진다. "배치 JDBC 점진적"이라고 하면 워터마크 컬럼, 타입(타임스탬프 vs 숫자), 초기 워터마크 값을 묻는다. "스트리밍 Kafka"라고 하면 startingOffsets와 체크포인트 경로를 확인한다. "REST API 점진적"이라고 하면 커서 매개변수 이름과 마지막 커서를 어디에 유지할지 묻는다.

명령어가 명시적으로 묻지 않는 것은 어떤 비즈니스 파티션 키를 사용할지다. 답은 항상 ingest_date이며, 이것이 패키지가 인코딩하는 가장 중요한 아키텍처 결정 중 하나다.

브론즈는 항상 ingest_date(로드 날짜)로 파티션된다. 모든 실행은 자체 ingest_date=YYYY-MM-DD/ 폴더에 저장된다. 파티션이 로드 경계다.

이 말은 "새 로드를 처리하라"가 항상 "새 ingest_date 파티션을 읽어라"와 동일하다는 뜻이다. 워터마크 메타데이터도 없고, "마지막 실행 날짜가 언제였나?" 같은 질문도 없다. 큐레이션 단계는 어떤 로드를 처리할지 파티션으로 자연스럽게 범위를 잡을 수 있다.

3단계: /curate — 브론즈에서 실버로, 데이터 품질 포함

큐레이션은 이 패키지에서 가장 정교한 단계다. 스키마가 추론되고, 데이터 품질 규칙이 권장되고, 나쁜 행이 격리되거나 드롭되거나 경고를 발생시키고, SCD Type 2 머지가 히스토리를 보존한다. 이 모든 것이 코드를 생성하기 전 사람의 승인을 받는다.

이 단계가 두 개의 에이전트로 분리된 이유가 여기 있다. curate-advisor는 스키마를 분석하고 데이터 품질 규칙을 추천하는 분석 에이전트다. 규칙이 생성되면 대화에 표시되고, 사용자가 편집, 추가, 삭제할 수 있다. 사용자가 승인하면 비로소 curate-agent가 코드를 작성한다.

이 구분은 핵심적이다. 데이터 품질 규칙에서 어드바이저와 제너레이터를 분리하면, AI가 실제 데이터 의미를 모르고도 추천을 제공할 수 있으면서 사람이 의미 있는 방식으로 검토하고 수정할 수 있다.

이 단계의 또 다른 아키텍처 결정은 DQ 심각도를 warn과 error로 분리하는 것이다. warn 규칙 위반은 격리 파티션에 기록되지만 처리는 계속된다. error 규칙 위반은 실행을 중단시킨다. 이 분리는 에이전트 지침에 인코딩되어 있으며, 모든 생성된 큐레이션 스크립트가 이 패턴을 따른다.

4단계: /cdc — 실버에서 골드로, SCD Type 2 포함

CDC 단계는 변경 데이터 캡처를 처리한다. 실버의 새 ingest_date 파티션을 읽고, 활성 레코드와 현재 레코드를 구분하는 SCD Type 2 스테이징 테이블에 머지한다.

구현은 두 페이즈 접근법을 따른다. 먼저 변경된 레코드의 현재 버전을 만료시키고(is_current = false, end_date 설정), 그 다음 신규 레코드를 is_current = true로 삽입한다. 이 두 페이즈 패턴은 단순화된 단일 스텝 머지가 조용히 실패하거나 잘못된 결과를 내는 엣지 케이스를 처리한다.

5단계: /data-product — 비즈니스 계약 정의

데이터 프로덕트 단계는 골드 레이어가 BI 도구에 노출되는 방식을 정의한다. 큐레이션 단계처럼 두 에이전트로 분리된다. data-product-advisor는 골드 데이터를 분석하고 뷰와 KPI를 추천한다. 사용자가 BI 계약을 승인하면 data-product-agent가 뷰 DDL과 KPI 로직을 생성한다.

이 단계의 핵심 통찰은 데이터 계약이 비즈니스 지식을 필요로 한다는 것이다. AI는 컬럼 이름과 데이터 타입을 기반으로 합리적인 KPI를 제안할 수 있다. 하지만 어떤 KPI가 실제 비즈니스 정의와 일치하는지는 사람만 알 수 있다. 어드바이저-승인-제너레이터 패턴은 이 구분을 강제한다.

6단계: /test-harness — 생성된 코드와 일치하는 테스트

테스트 하네스 단계가 특별한 이유는 전체 파이프라인 상태를 알고 있다는 것이다. state.json에서 실제 생성된 코드의 경로와 구조를 읽고, 그 코드에 맞는 테스트를 생성한다. 추상적인 템플릿이 아니다. 실제로 생성된 수집 스크립트의 로직을 테스트하고, 실제 큐레이션 규칙을 검증하고, 실제 SCD Type 2 머지가 올바르게 동작하는지 확인하는 테스트다.

이것이 이 패키지의 AI 우선 접근법이 가진 주요 이점 중 하나다. 테스트를 생성하는 에이전트는 테스트할 코드를 생성한 에이전트와 같은 구조적 지식을 공유한다.

7단계: /checkin — 보안 가드레일이 있는 git push

체크인 단계는 파이프라인의 마지막 관문이다. checkin-agent는 코드를 커밋하거나 푸시하기 전에 사전 점검 목록을 실행한다.

  • 비밀 스캔: .env 파일이 스테이징되어 있는가? 하드코딩된 자격증명이 보이는가?
  • 테스트 통과: 테스트 하네스가 실행되었고 통과했는가?
  • README 최신화: 생성된 아키텍처를 문서화했는가?

사전 점검이 통과하면 에이전트는 스테이징된 파일 목록을 표시하고 사용자의 명시적 승인을 요청한다. 승인이 있을 때만 커밋하고 푸시한다.

상태 파일: 파이프라인의 기억

전체 패키지에서 .sdlc/state.json이 하는 역할을 이해하는 것이 중요하다. 이것은 단순한 설정 파일이 아니다. 파이프라인을 통한 전체 여정의 기록이다.

상태 파일 덕분에 각 단계는 독립적으로 실행 가능하다. DQ 규칙을 변경한 후 큐레이션 스크립트를 재생성해야 한다면, /curate를 직접 실행하면 된다. state.json을 읽고 이전 단계에서 필요한 모든 것을 복원하며, 큐레이션 경로와 샘플 스키마만 묻는다. 수집을 다시 실행하거나 처음부터 모든 질문에 다시 답할 필요가 없다.

세 개의 승인 관문: 인간이 루프 안에 있는 곳

패키지에는 사람이 명시적으로 승인하기 전까지 코드가 생성되지 않는 세 단계가 있다.

  1. /curate: curate-advisor가 추천한 큐레이션 규칙을 검토·편집하고, 나쁜 행 처리 방식을 선택한 후 승인해야 curate-agent가 코드를 생성한다.
  2. /data-product: data-product-advisor가 추천한 BI 계약, 뷰, KPI를 검토·편집하고 승인해야 data-product-agent가 코드를 생성한다.
  3. /checkin: 사전 안전 점검, 스테이징된 파일 검토, 명시적 푸시 확인 후에야 checkin-agent가 실행된다.

이 관문들은 관료적 마찰이 아니다. AI가 아는 것(무엇이 가능한가)과 인간이 아는 것(무엇이 적절한가)이 만나는 지점이다. 큐레이션 규칙셋은 데이터 의미와 일치할 때만 가치가 있다. KPI 정의는 비즈니스 정의와 일치할 때만 유용하다. 어드바이저→관문→제너레이터 패턴이 이것을 강제한다. AI가 분석하고(항상), 인간이 승인하고(항상), AI가 구현한다(항상). 어떤 단계도 건너뛰지 않는다.

레이어드 파티션 모델: 관행으로 인코딩된 아키텍처

패키지가 제공하는 가장 가치 있는 것 중 하나는 세 개의 Delta Lake 레이어 전반에 걸쳐 일관되고 방어 가능한 파티션 전략을 강제한다는 것이다.

이 아키텍처는 임의적이지 않다. 브론즈를 ingest_date로 파티션하면 모든 로드가 격리된다. 다른 로드에 영향을 주지 않고 어떤 로드든 재처리할 수 있고, 어떤 날짜에 무엇이 도착했는지 감사할 수 있으며, 큐레이션 단계는 워터마크 없이 각 실행의 자연스러운 범위를 갖는다. 실버는 큐레이션이 한 번에 하나의 로드를 처리하므로 같은 파티션을 유지한다. 여기에 비즈니스 파티션을 추가하면 재파티션 작업과 분리된 쓰기가 발생한다. 골드는 BI 도구가 실제로 쿼리하는 유일한 레이어이므로, 이곳에서 비즈니스 파티션이 효과를 발휘한다.

에이전트 지침에 이 모델을 인코딩함으로써("브론즈는 어떤 비즈니스 키로도 파티션하지 않는다. 커스텀/비즈니스 파티션은 CDC 레이어에서만") — 이 패키지로 생성되는 모든 프로젝트가 동일한 모델을 따른다. 규모에서의 일관성은 AI 우선 접근법이 제공하는 가장 깊은 가치 중 하나다.

AI 우선 데이터 엔지니어링이 의미하는 것

이 패키지가 엔드-투-엔드로 하는 일을 돌아보면, 단일 단계를 넘어서는 더 큰 패턴이 드러난다. AI 우선 SDLC/TDLC가 실제로 어떤 모습인지가 그것이다.

AI는 기관의 지식을 인코딩한다. JDBC 점진적 로드의 워터마크 패턴, 두 페이즈 SCD Type 2 머지, 데이터 품질의 warn/error 심각도 분리 — 이것들은 경험 있는 데이터 엔지니어는 알고 주니어 엔지니어는 힘들게 배우는 것들이다. 패키지는 이것들을 에이전트 지침에서 협상 불가능한 행동으로 인코딩한다. 파이프라인을 실행하는 사람이 누구든 그 지식의 혜택을 받는다.

AI는 설정 드리프트를 제거한다. 이 패키지 없이는 두 엔지니어가 두 파이프라인을 구축할 때 파티션 키, 큐레이션 순서, DQ 규칙 명명 관행, 테스트 구조에 대해 서로 다른 선택을 한다. 이 패키지와 함께하면 선택이 아키텍처에 내장된다. 사람들이 가끔 따르는 스타일 가이드 때문이 아니라 생성기가 그 선택을 하기 때문에 프로젝트가 일관성을 갖는다.

인간은 오직 인간만 결정할 수 있는 것에 집중한다. 승인 관문은 비즈니스 지식이 필요한 결정 지점에 정확히 배치된다. 어떤 큐레이션 규칙이 데이터 의미에 맞는지, 어떤 KPI가 비즈니스에 중요한지, 데이터 계약이 소비자에게 무엇을 약속하는지. AI가 나머지를 처리한다.

라이프사이클은 연속적이다. TDLC는 파이프라인을 구축하는 것만이 아니다. 테스트하고, 문서화하고, 안전하게 배포하는 것이다. 파이프라인은 data_product.py에서 끝나지 않는다. /test-harness(실제 생성된 코드와 일치하는 테스트를 생성)와 /checkin(보안 가드레일과 함께 커밋하고 푸시)까지 이어진다. "데이터 파이프라인이 필요하다"에서 "코드가 main에 있고 PR이 열려 있다"까지의 전체 흐름이 하나의 자동화된 워크플로다.

아키텍처는 투명하고 확장 가능하다. 이 패키지는 블랙박스가 아니다. ARCHITECTURE.md는 하이브리드 모델, 단계별 계약, 상태 스키마, 새 단계를 추가하는 패턴을 문서화한다. 새 단계를 추가하는 것은 세 단계다. 명령어(대화형 프론트엔드)를 만들고, 에이전트(상태 비저장 제너레이터)를 만들고, 상태 읽기 → 공백 질문 → 위임 → 상태 업데이트 → 자동 호출 패턴을 따른다.

시작하기

패키지는 두 개의 하위 디렉토리를 가진 .claude/에 있다.

패키지 디렉토리의 Claude Code 세션에서 새 파이프라인을 시작하려면

/project-init my-sales-pipeline

파이프라인은 각 단계에서 총 일곱 가지 질문을 하며, 큐레이션, 데이터 프로덕트 설계, 체크인에서 사람의 승인 관문을 거친다. 나머지는 자동화된다. 마지막에는 완전하고, 문서화되고, 테스트되고, 버전 관리된 PySpark + Delta Lake 파이프라인이 생긴다. mkdir부터 git push까지.

결론

SDLC에 대한 기존 접근법은 AI를 조수로 취급한다. 보일러플레이트를 더 빨리 작성하는 수단, 더 나은 자동완성, 코드도 생성할 수 있는 러버덕. 여기서 설명한 접근법은 다르다. AI가 프로세스다. 단계를 이끌고, 맥락을 유지하고, 아키텍처를 강제하고, 코드를 생성한다. 인간은 루프 밖에 있지 않다. 인간은 정확히 맞는 결정 지점에 있다. AI가 나머지를 처리한다.

이것이 AI 우선의 실제 의미다. 당신이 사용하는 도구로서의 AI가 아니라, 당신이 위에 구축하는 인프라로서의 AI. 가치는 생성되는 어떤 단일 파일에 있지 않다. 일관성, 완전성, 규모에서 인코딩된 기관의 지식, 그리고 엔지니어들이 실제로 인간의 판단이 필요한 결정에 집중할 수 있는 자유에 있다.

파이프라인은 완성됐다. 코드는 일관적이다. 테스트는 코드와 일치한다. README는 전체를 문서화한다. 그리고 이 어떤 것도 워터마크 헬퍼, DQ 심각도 분리, SCD Type 2 스테이징 업데이트 머지, 또는 사전 비밀 스캔을 수동으로 작성할 필요가 없었다.

그것이 핵심이다.
