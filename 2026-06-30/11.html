Nobody Tells You This About Becoming a Lead Data Engineer

Think Data · · 3분 · 원문 · freedium

수석 데이터 엔지니어가 되려면 코딩 실력보다 시스템 사고의 전환이 필요하다.

요약

대부분의 사람들은 수석 데이터 엔지니어로의 성장을 경험 축적과 더 나은 코딩 실력의 자연스러운 결과로 생각하지만, 실제 전환의 핵심은 기술이 아닌 사고방식의 변화에 있다. 개별 쿼리와 파이프라인을 최적화하는 것에서 시작해, 오케스트레이션·의존성 그래프 수준의 파이프라인 사고를 거쳐, 데이터 소유권·스키마 변경·데이터 컨트랙트를 다루는 시스템 사고로 나아간다. 리드 수준에서는 다른 엔지니어들이 안전하고 효율적으로 파이프라인을 구축할 수 있는 플랫폼 자체를 설계하는 역할을 맡는다. 가장 높은 단계에서는 기술적 문제를 넘어 조직 전체의 데이터 아키텍처를 설계하는 조직적 사고가 요구된다.

대부분의 사람들은 수석 데이터 엔지니어가 되는 것을 자연스러운 승진으로 여긴다. 경험이 쌓이고, 코딩 실력이 늘고, 더 큰 프로젝트를 맡으면 되는 것이라고. 하지만 현실에서 데이터 엔지니어에서 리드로의 전환은 더 나은 파이프라인을 작성하는 것과는 관계가 없다. 핵심은 시스템 관점으로 사고하는 법을 익히는 데 있다.

커리어 초반에 데이터 엔지니어는 개별 컴포넌트에 집중한다.

  • 변환 하나
  • 파이프라인 하나
  • DAG 하나
  • 웨어하우스 테이블 하나

리드 엔지니어는 전혀 다른 것을 생각해야 한다. 바로 데이터 플랫폼 전체의 동작 방식이다. 그리고 이 전환은 거의 모든 설계 결정을 바꿔버린다.

커리어 사다리 vs 사고 사다리

레벨 1 — 쿼리 사고

커리어 초반의 목표는 단순하다. 데이터를 올바르게 변환하는 것이다.

주요 관심사는 다음과 같다.

  • SQL
  • 조인
  • 집계
  • 변환
  • 성능 튜닝

이 단계의 핵심 질문은 이것이다.

"이 쿼리가 올바른 데이터셋을 반환하는가?"

대부분의 엔지니어는 처음 몇 년을 개별 쿼리를 최적화하는 데 보낸다.

레벨 2 — 파이프라인 사고

곧 도전 과제가 바뀐다. 문제는 더 이상 쿼리 자체가 아니다. 수많은 변환들이 어떻게 함께 작동하느냐가 문제가 된다.

이제 엔지니어들은 다음을 고민하기 시작한다.

  • 오케스트레이션
  • 의존성 그래프
  • 재시도 처리
  • 백필(backfill)
  • 스케줄링
  • 멱등성(idempotent) 파이프라인

이 단계의 질문은 이렇게 바뀐다.

"이 파이프라인들이 어떻게 함께 안정적으로 실행될 수 있는가?"

오케스트레이션 프레임워크 같은 도구들이 중요해지는 시점이다. 하지만 파이프라인만으로는 여전히 전체 시스템을 이해하기에 부족하다.

레벨 3 — 시스템 사고

데이터 플랫폼이 성장하면 파이프라인은 더 이상 주된 문제가 되지 않는다. 진짜 도전 과제들이 드러나기 시작한다.

  • 스키마 변경
  • 업스트림 의존성
  • 데이터 소유권
  • 데이터셋 탐색 가능성
  • 파이프라인 난립

이 단계에서 엔지니어들은 파이프라인이 아닌 데이터 시스템 전체를 사고해야 한다. 질문도 달라진다.

  • 이 데이터셋의 소유자는 누구인가?
  • 이 데이터는 어떤 보장을 제공하는가?
  • 스키마가 변경되면 어떤 일이 생기는가?
  • 다운스트림 팀은 이 데이터에 어떻게 안전하게 의존할 수 있는가?

데이터 컨트랙트, 버전 관리된 스키마, 데이터셋 소유권 같은 개념들이 핵심이 되는 지점이다.

레벨 4 — 플랫폼 사고

리드 레벨로의 전환이 시작되는 지점이다. 초점이 다시 한 번 이동한다. 파이프라인을 직접 구축하는 대신, 다른 엔지니어들이 의존하는 플랫폼 역량을 설계하기 시작한다.

구체적인 예를 들면 다음과 같다.

  • 표준화된 변환 프레임워크
  • 재사용 가능한 데이터 모델
  • 자동화된 테스트
  • 옵저버빌리티(observability) 시스템
  • 비용 모니터링
  • 개발자 도구

이 단계의 핵심 질문은 이것이다.

"다른 엔지니어들이 어떻게 하면 안전하고 효율적으로 데이터 파이프라인을 구축할 수 있을까?"

달리 말하면, 이제 파이프라인을 만드는 것이 아니라 파이프라인이 만들어지는 환경을 구축하는 것이다.

레벨 5 — 조직적 사고

마지막 전환은 순수하게 기술적인 문제가 아니다. 팀과 도메인을 가로질러 데이터가 어떻게 흐르는지의 문제다.

대규모 조직은 다음과 같은 어려움을 겪는다.

  • 중복된 데이터셋
  • 일관성 없는 지표
  • 불명확한 소유권
  • 충돌하는 데이터 정의

이 단계에서의 도전 과제는 조직 수준의 데이터 아키텍처다. 다음과 같은 아이디어들이 중요해진다.

  • 도메인별 데이터 소유권
  • 데이터 프로덕트
  • 플랫폼 거버넌스
  • 팀 간 컨트랙트

이 단계의 질문은 더 이상 기술적이지 않다. 아키텍처적 질문이 된다.

"조직은 데이터 시스템을 어떻게 구조화해야 하는가?"

모든 리드 데이터 엔지니어가 배우는 3가지 아키텍처 교훈

플랫폼 책임을 맡게 되면서 반복적으로 등장하는 교훈들이 있다.

1. 파이프라인은 선형으로 늘어나지만, 복잡성은 그보다 훨씬 빠르게 늘어난다

파이프라인을 추가하는 것은 쉽다. 수백 개를 유지 관리하는 것은 전혀 다른 이야기다. 구조가 없으면 데이터 플랫폼은 서서히 이렇게 변해간다.

  • 문서화되지 않은 테이블들
  • 중복된 변환 로직
  • 깨지기 쉬운 의존성들

해결책은 플랫폼 수준의 제약이다. 엄격한 데이터셋 소유권, 일관된 모델링 레이어, 문서화된 데이터 컨트랙트가 그것이다. 코드보다 구조가 더 중요해지는 순간이다.

2. 데이터 컨트랙트가 파이프라인 로직보다 더 중요하다

실제 프로덕션 장애의 대부분은 파이프라인이 깨져서 생기는 것이 아니다. 예상치 못한 업스트림 변경 때문에 생긴다. 예를 들면 이런 것들이다.

  • 새로운 컬럼 추가
  • 데이터 타입 변경
  • 필드 누락
  • 의미론적 변경

명확한 컨트랙트 없이는 모든 다운스트림 파이프라인이 불안정해진다. 현대 데이터 플랫폼은 데이터셋을 API처럼 안정적인 인터페이스로 취급하는 방향으로 나아가고 있다.

3. 옵저버빌리티가 오케스트레이션보다 더 중요하다

오케스트레이션 시스템은 한 가지 질문에 답한다. 파이프라인이 실행됐는가?

하지만 이것에는 답하지 못한다.

  • 데이터가 올바른가?
  • 볼륨이 정상인가?
  • 분포가 바뀌었는가?
  • 데이터 신선도가 저하되었는가?

파이프라인이 성공적으로 실행되었다고 해서 데이터가 정상이라는 의미는 아니다. 그래서 현대 데이터 플랫폼은 데이터셋 자체의 건강 상태를 모니터링하는 데이터 옵저버빌리티 시스템에 점점 더 의존하게 된다.

데이터 엔지니어에서 리드로의 진짜 전환

가장 중요한 전환은 기술적인 것이 아니다. 개념적인 것이다. 데이터 엔지니어는 컴포넌트에 집중하고, 리드 엔지니어는 성장해도 살아남는 시스템에 집중한다.

질문이 이렇게 바뀐다.

"이 파이프라인을 어떻게 만들까?"가 아니라 "회사가 성장함에 따라 이 플랫폼 전체가 어떻게 동작해야 하는가?"로.

그렇게 생각하기 시작하면, 역할은 자연스럽게 파이프라인 빌더에서 시스템 설계자로 진화한다.

마치며

데이터 엔지니어링 커리어는 변환을 작성하는 것에서 시작한다. 하지만 시간이 지나면서 문제는 점점 커진다.

  • 파이프라인 신뢰성
  • 플랫폼 확장성
  • 조직의 데이터 아키텍처

리드 역할로 성장하는 엔지니어들은 대개 미묘한 전환을 이뤄낸 사람들이다. 파이프라인을 개별적으로 바라보는 것을 멈추고, 수년에 걸쳐 데이터를 안정적으로 이동시키는 시스템을 생각하기 시작한 사람들이다.
