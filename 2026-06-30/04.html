There Are 5 Types of CDC Techniques — But Most Engineers Use the Wrong One

Think Data · 5분 읽기 · 원문 · freedium

CDC의 5가지 기법은 선택지가 아니라 이전 방식의 실패에서 진화한 것이다.

요약

CDC(Change Data Capture)는 처음에는 간단해 보이지만 운영 환경에서 곧 가장 어려운 데이터 엔지니어링 문제 중 하나로 드러난다. 타임스탬프 기반, 로그 기반, 트리거 기반, 스냅샷 비교, MERGE 기반의 다섯 가지 기법은 각각 이전 방식의 한계를 극복하기 위해 탄생했다. 대부분의 엔지니어는 한 가지 기법만 선택하려 하지만, 실제 성숙한 시스템은 다섯 가지를 상황에 맞게 조합해 사용한다. 각 기법이 어떤 문제를 해결하고 어떤 새로운 문제를 만들어내는지 이해하는 것이 핵심이다.

"프로덕션 데이터베이스를 몇 분 간격으로 웨어하우스에 동기화할 수 있을까요?"

처음에는 별것 아닌 것처럼 들린다.

그런데 6개월이 지나면 이런 일이 벌어진다.

  • 대시보드 수치가 재무팀 숫자와 맞지 않는다.
  • 삭제된 주문이 분석 시스템에 여전히 남아 있다.
  • Kafka 지연이 세 시간을 넘어섰다.
  • 늦게 들어온 이벤트가 최신 데이터를 덮어쓴다.
  • 누군가 "전체 재적재가 필요할 수도 있다"고 말한다.

그 순간 CDC는 데이터 엔지니어링에서 가장 어려운 문제 중 하나가 된다.

CDC의 5가지 주요 기법은 애초에 "선택지"로 설계된 게 아니었다. 이전 방식이 규모 앞에서 무너졌기 때문에 하나씩 진화한 것이다.

Shopify와 유사한 시스템을 예시로 삼아 이 진화 과정을 살펴보자.

예시: Shopify 주문 시스템

다음과 같은 핵심 테이블을 갖춘 Shopify 유사 회사를 상상해보자.

orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    status TEXT,
    total_amount DECIMAL,
    updated_at TIMESTAMP,
    is_deleted BOOLEAN DEFAULT FALSE
)

이 테이블 하나가 다음을 모두 담당한다.

  • 매출 대시보드
  • 고객 알림
  • 사기 탐지 시스템
  • 주문 처리 파이프라인
  • 웨어하우스 분석

회사가 성장하면서 CDC 아키텍처는 다섯 단계를 거쳐 진화한다.

기법 1 — 타임스탬프 기반 CDC

거의 모든 회사가 여기서 시작한다.

왜 만들어졌나

초기 Shopify는 이런 환경이다.

  • Postgres 데이터베이스 하나
  • 분석 엔지니어 한 명
  • 스트리밍 인프라 없음
  • Kafka 없음
  • Debezium 없음

가장 간단한 해결책은 이것이다.

SELECT *
FROM orders
WHERE updated_at > :last_watermark;

이걸 15분마다 실행하면 끝이다.

엔지니어들이 처음에 좋아하는 이유

  • 구현이 단순하다.
  • 비용이 거의 들지 않는다.
  • 디버깅이 쉽다.
  • 어떤 데이터베이스에서도 동작한다.

WAL(Write Ahead Log) 파싱도 없고, Kafka 오프셋도 없으며, 분산 시스템 고민도 필요 없다.

운영 현실이 찾아오면

다음 시나리오를 생각해보자.

-- 주문 생성
INSERT INTO orders VALUES (
  42, 1001, 'PENDING', 250.00, '10:00', false
);

-- 이후 결제 실패
UPDATE orders
SET status = 'FAILED',
    updated_at = '10:20'
WHERE order_id = 42;

CDC 작업이 10:10에 실행됐다면, 웨어하우스는 영구적으로 다음과 같이 인식한다.

  • 주문 상태는 여전히 PENDING
  • 매출 수치는 유효한 것으로 처리

재무 데이터가 틀려진다.

이 기법이 만드는 문제들

삭제가 사라진다

DELETE FROM orders WHERE order_id = 42;

CDC는 이 삭제를 포착하지 못한다.

시계 불일치로 데이터 정합성이 깨진다

한 서비스는 UTC로 타임스탬프를 기록하고, 다른 서비스는 로컬 시간을 쓴다. 그러면 일부 행이 누락되거나 중복 처리된다.

늦게 들어온 업데이트가 분석 데이터를 조용히 오염시킨다

가장 위험한 점은 파이프라인이 실패하지 않는다는 것이다. 데이터가 그냥 틀려진다. 이런 버그가 가장 무섭다.

이 기법이 아직 살아남은 이유

다음과 같은 용도에서는 충분히 "쓸 만"하기 때문이다.

  • 대시보드
  • MVP 시스템
  • 중요도가 낮은 분석

하지만 결국 규모가 커지면 무너진다.

기법 2 — 로그 기반 CDC

타임스탬프 기반 CDC가 정확성을 보장할 수 없다는 것이 드러나면서 대부분의 엔지니어가 결국 이 방식으로 마이그레이션한다.

근본적인 발상의 전환

기존 질문: "어떤 행이 변경됐나?"

새로운 질문: "어떤 트랜잭션이 커밋됐나?"

이 관점의 전환이 모든 것을 바꾼다.

Shopify식 아키텍처

Postgres WAL
    ↓
Debezium
    ↓
Kafka
    ↓
Warehouse + 다운스트림 시스템

주요 도구는 다음과 같다.

  • Debezium: 데이터베이스의 모든 행 단위 변경을 스트리밍하는 오픈소스 로그 기반 CDC 플랫폼
  • Apache Kafka: 대규모 실시간 데이터를 스트리밍, 저장, 처리할 수 있는 분산 내결함성 커밋 로그 시스템

CDC 이벤트의 실제 모습

{
  "op": "u",
  "before": {
    "status": "PENDING"
  },
  "after": {
    "status": "SHIPPED"
  },
  "updated_at": "10:20"
}

데이터베이스 트랜잭션 로그 자체가 단일 진실 공급원(source of truth)이 된다.

웨어하우스 컨슈머 예시

from kafka import KafkaConsumer
import json
consumer = KafkaConsumer(
    "orders.cdc",
    bootstrap_servers="localhost:9092",
    value_deserializer=lambda v: json.loads(v.decode())
)
for msg in consumer:
    event = msg.value
    print(event)

왜 이 기법이 만들어졌나

타임스탬프 폴링으로는 감당할 수 없는 다음 요구사항들이 생겼기 때문이다.

  • 실시간 업데이트
  • 삭제 처리
  • 신뢰할 수 있는 이벤트 순서
  • 확장 가능한 이벤트 스트리밍

하지만 새로운 문제가 생긴다

중복 이벤트

Kafka는 at-least-once(최소 한 번) 전달을 보장한다. 따라서 이런 일이 생긴다.

SHIPPED
SHIPPED (again)

멱등성(idempotency) 처리가 없으면 다운스트림 지표가 중복 집계되고 집계 값이 틀어진다.

이벤트 순서 역전

네트워크 지연으로 다음과 같은 상황이 발생할 수 있다.

SHIPPED가 먼저 도착
PENDING이 나중에 도착

그러면 웨어하우스의 상태가 거꾸로 돌아간다. 해결책은 다음과 같다.

WHERE incoming.updated_at > existing.updated_at

초기 스냅샷의 복잡성

CDC 스트림은 앞으로 일어날 변경만 캡처한다. 따라서 엔지니어는 다음을 직접 처리해야 한다.

  1. 현재 테이블 스냅샷 생성
  2. WAL 오프셋 확보
  3. 정확한 위치부터 스트리밍 시작

이 과정을 잘못 처리하면 데이터 중복이나 유실이 발생한다.

로그 기반 CDC가 업계 표준이 된 이유

다음 세 가지를 동시에 만족하기 때문이다.

  • 정확성
  • 낮은 지연
  • 확장성

기법 3 — 트리거 기반 CDC

현실은 더 복잡해진다. Shopify가 레거시 ERP 시스템을 운영하는 회사를 인수했는데, 트랜잭션 로그에 접근할 수 없는 상황이다. 그렇다면?

왜 이 기법이 만들어졌나

다음과 같은 상황이 생기기 때문이다.

  • 벤더 데이터베이스가 잠겨 있다.
  • WAL/binlog 접근이 금지되어 있다.
  • SQL 접근만 가능하다.

그래서 엔지니어들은 CDC를 데이터베이스 내부로 밀어넣었다.

트리거 기반 설계

CREATE TABLE orders_audit (
    order_id INT,
    operation TEXT,
    old_status TEXT,
    new_status TEXT,
    changed_at TIMESTAMP
);

트리거 로직

CREATE OR REPLACE FUNCTION audit_orders()
RETURNS TRIGGER AS $$
BEGIN
    INSERT INTO orders_audit VALUES (
        NEW.order_id,
        TG_OP,
        OLD.status,
        NEW.status,
        now()
    );
RETURN NEW;
END;
$$ LANGUAGE plpgsql;

해결하는 문제

  • 삭제를 포착할 수 있다.
  • 업데이트를 추적한다.
  • 로그 접근 없이도 동작한다.

망가뜨리는 것

쓰기 증폭(Write Amplification): 모든 업데이트가 두 번의 쓰기로 늘어난다.

orders에 1회 쓰기
+ audit 테이블에 1회 쓰기

규모가 커지면 데이터베이스 성능이 급격히 저하된다.

숨겨진 지연: 트리거가 조용히 쓰기 처리량을 낮춘다.

디버깅 악몽: 프로덕션의 트리거 체인은 추론하기가 극도로 어렵다.

이 기법이 아직 살아남은 이유

인프라를 통제할 수 없고 SQL만 다룰 수 있는 상황이 실제로 존재하기 때문이다.

기법 4 — 스냅샷 비교(Snapshot + Diff) CDC

Shopify가 데이터 레이크와 컴플라이언스 시스템으로 확장하면서 또 다른 문제가 생긴다. 일부 시스템은 전체 내보내기만 제공한다. 로그도 없고, 트리거도 없고, 타임스탬프도 없다. 거대한 스냅샷만 있을 뿐이다.

왜 이 기법이 만들어졌나

배치 시스템에는 다음이 필요했기 때문이다.

  • 과거 데이터 재구성
  • 컴플라이언스 아카이브
  • 외부 벤더 데이터 수집

일별 스냅샷 예시

SELECT * FROM orders;

저장 위치:

s3://orders/dt=2026-05-25/
s3://orders/dt=2026-05-26/

차이 비교 로직

import pandas as pd
old = pd.read_parquet("orders_2026_05_25.parquet")
new = pd.read_parquet("orders_2026_05_26.parquet")
merged = old.merge(
    new,
    on="order_id",
    how="outer",
    suffixes=("_old", "_new")
)

해결하는 문제

  • 어떤 형태의 데이터 덤프에도 적용 가능하다.
  • 완전한 이력 비교가 가능하다.
  • 복구가 용이하다.

망가뜨리는 것

막대한 컴퓨팅 비용: 전체 데이터셋을 반복적으로 스캔한다.

유령 변경(Fake Diffs): 서로 다른 시점에 찍힌 스냅샷이 실제로는 없는 변경처럼 보이게 만든다.

높은 지연: 데이터 최신성이 아무리 좋아도 몇 시간, 보통은 하루 단위다.

기업들이 여전히 사용하는 이유

배치 데이터 레이크는 실시간 처리 속도보다 다음을 우선시하기 때문이다.

  • 완전성
  • 이력 복구
  • 스키마 유연성

기법 5 — MERGE 기반 CDC

마지막으로 모든 데이터가 웨어하우스에 도달한다. Snowflake, BigQuery, Databricks. 이제 핵심 과제는 이것이다.

"이 모든 변경 사항을 안전하게 어떻게 조정할 것인가?"

왜 이 기법이 만들어졌나

웨어하우스에는 다음이 필요하기 때문이다.

  • 결정론적(deterministic) 최종 상태
  • 멱등성 있는 업데이트
  • 확장 가능한 조정(reconciliation)

스테이징 테이블

staging_orders (
    order_id INT,
    status TEXT,
    updated_at TIMESTAMP
)

최종 MERGE

MERGE INTO orders t
USING staging_orders s
ON t.order_id = s.order_id
WHEN MATCHED
AND s.updated_at > t.updated_at THEN
  UPDATE SET
    status = s.status,
    updated_at = s.updated_at
WHEN NOT MATCHED THEN
  INSERT (
    order_id,
    status,
    updated_at
  )
  VALUES (
    s.order_id,
    s.status,
    s.updated_at
  );

해결하는 문제

  • 중복 이벤트 처리
  • 재실행 안전성
  • 최종 상태 일관성

여전히 생기는 문제

늦은 이벤트가 최신 데이터를 덮어쓴다: 타임스탬프 검증이 없으면 오래된 업데이트가 현재 상태를 오염시킨다.

삭제를 자주 잊는다: 결과적으로 좀비 레코드가 영구히 남는다.

대용량 MERGE는 비싸다: 대형 웨어하우스 테이블에서는 파티션 잠금, 느린 쿼리, 높은 컴퓨팅 비용이 발생한다.

대부분의 엔지니어가 놓치는 것

많은 팀이 CDC를 "기법 하나를 고르면 된다"고 생각한다. 하지만 실제 시스템이 진화하는 방식은 그렇지 않다.

현대 기업들은 대개 다섯 가지 기법을 모두 조합해 사용한다. CDC 기법은 도시의 교통 시스템과 같다.

도시는 이런 선택을 하지 않는다.

  • 도로만 쓴다.
  • 또는 기차만 쓴다.
  • 또는 공항만 쓴다.

서로 다른 문제를 해결하고 서로 다른 트레이드오프를 가지기 때문에 모든 수단을 함께 쓴다. CDC도 마찬가지다.

+---------------------------------------------+
| 기법            | 존재 이유                  |
| ------------- | --------------------------- |
| 타임스탬프 CDC  | 단순성                      |
| 로그 기반 CDC   | 정확성 + 확장성              |
| 트리거 CDC      | 레거시/벤더 시스템           |
| 스냅샷 비교     | 배치 복구 + 컴플라이언스     |
| MERGE CDC      | 웨어하우스 조정              |
+---------------------------------------------+
                ┌────────────────────┐
                │   Postgres Orders  │
                └─────────┬──────────┘
                          │
         ┌────────────────┼────────────────┐
         │                │                │
         ▼                ▼                ▼

 타임스탬프 CDC      로그 기반 CDC     트리거 CDC
  (배치 작업)          (Kafka)         (벤더 ERP)

         │                │                │
         └────────────────┴────────────────┘
                          │
                          ▼

                  스냅샷 복구
                  (일별 내보내기)

                          │
                          ▼

                  웨어하우스 MERGE
                 (최종 조정)

시니어 엔지니어들이 결국 깨닫는 것

CDC의 본질은 결국 이 질문으로 귀결된다.

"우리는 어디에서 불일관성을 허용할 것인가?"

모든 CDC 아키텍처는 결국 다음과 싸우게 된다.

  • 늦게 도착하는 이벤트
  • 중복
  • 삭제 처리
  • 순서 역전
  • 재실행 버그
  • 규모에 따른 병목
