Stop Using != null — This Is How Senior Developers Actually Think

Gain Java Knowledge · Real World Java · 4분 읽기 · 원문 · freedium

요약

시니어 개발자는 null 검사를 줄이는 것이 아니라, null이 필요 없는 시스템을 설계한다.

많은 개발자들이 습관적으로 != null을 코드 곳곳에 뿌리지만, 시니어 개발자들은 전혀 다른 방식으로 생각한다. 그들은 null을 시스템 가장자리로 격리하고, 생성자와 불변 설계로 객체의 유효 상태를 보장하며, Optional과 Null Object 패턴으로 null 자체를 소거한다. 이 글은 주니어와 시니어를 가르는 사고 전환을 Java 코드 예시와 함께 구체적으로 보여준다.

재능이 부족해서가 아니다. 노력이 부족해서도 아니다. 문법을 몰라서도 절대 아니다.

개발자의 성장이 조용히 멈추는 순간은 훨씬 이른 시점에 찾아온다 — 바로 다음과 같은 코드를 작성하면서 안전하다는 착각에 빠질 때다.

if (x != null) {
    // 계속 진행
}

처음에는 책임감 있는 코드처럼 느껴진다. 그러다 손에 익어 근육 기억이 된다. 결국에는 눈에 보이지도 않게 된다.

세월이 흐른다. 기능을 출시하고, 버그를 고치고, 주니어를 가르치기까지 한다. 그런데 어쩐지 시니어 자리는 늘 손에 잡히지 않는다. 아무도 그 이유를 말해주지 않는다.

시니어 개발자들이 입 밖에 내지 않는 불편한 진실이 있다.

핵심은 null이 아니다. null을 마주쳤을 때 어떻게 생각하느냐가 핵심이다.

시니어 개발자들이 버그를 덜 만드는 이유는 조건을 더 많이 검사해서가 아니다. 그들은 끊임없이 검사해야 하는 코드를 처음부터 설계하지 않는다.

이 시각의 전환을 한 번 보고 나면 다시는 예전 눈으로 볼 수 없다.

!= null의 진짜 문제

솔직히 말하자. != null은 작동한다. 하지만 코드베이스 전체가 이렇게 생겼다면 어떨까.

if (order != null && order.getCustomer() != null &&
    order.getCustomer().getAddress() != null) {
    // 무언가를 한다
}

이건 안전한 코드가 아니다. 두려움에 의한 코드다.

시니어 개발자의 눈에 이 코드는 이렇게 읽힌다.

"이 코드는 자신의 설계를 신뢰하지 않는다."

그것이 진짜 문제다.

주니어의 사고방식 vs 시니어의 사고방식

주니어의 사고:

"무엇이든 null이 될 수 있으니 어디서나 방어해야 한다."

시니어의 사고:

"애초에 왜 이게 null이 될 수 있도록 허용했나?"

시니어는 조건문에서 출발하지 않는다. 계약, 경계, 그리고 보장에서 출발한다.

1. 시니어는 null을 시스템 가장자리로 밀어낸다

잘 설계된 시스템에서 null은 오직 가장자리(edge)에서만 허용된다.

  • 컨트롤러 입력값
  • 외부 API 응답
  • 데이터베이스 조회 결과
  • 역직렬화(deserialization) 처리

핵심 비즈니스 로직 내부에서는? null은 시나리오가 아니라 버그다.

경계 유효성 검사 예시 (Spring Boot)

@PostMapping("/users")
public ResponseEntity<Void> createUser(@RequestBody UserRequest request) {
    Objects.requireNonNull(request.getEmail(), "Email must not be null");
    userService.createUser(request);
    return ResponseEntity.ok().build();
}

경계에서 검증을 마치면, 서비스 레이어는 데이터가 올바르다는 전제 위에서 동작한다. 메서드마다 != null을 흩뿌릴 필요도, 불안을 코드에 심어놓을 필요도 없다.

2. 시니어는 생성자로 유효 상태를 강제한다

이렇게 하는 대신:

User user = new User();
user.setEmail(email);

시니어 개발자는 이렇게 한다.

public class User {

    private final String email;

    public User(String email) {
        this.email = Objects.requireNonNull(email, "email is required");
    }

    public String getEmail() {
        return email;
    }
}

이제 다음이 보장된다.

  • User 객체는 유효하지 않은 상태로 존재할 수 없다.
  • 버그는 발생 즉시 실패한다(fail-fast).
  • 하위 코드가 단순하고 안전해진다.

이것은 방어적 프로그래밍이 아니라 의도적인 설계다.

3. Optional은 설계 결정이지, 세련된 null 검사가 아니다

많은 개발자들이 Optional을 이렇게 잘못 쓴다.

Optional<User> user = Optional.ofNullable(findUser());
if (user.isPresent()) {
    user.get().getName();
}

이건 != null에 정장을 입힌 것에 불과하다.

시니어가 실제로 Optional을 쓰는 방식

public Optional<User> findUserById(Long id) {
    return repository.findById(id);
}

호출 지점에서는 이렇게 쓴다.

User user = userService.findUserById(id)
        .orElseThrow(() -> new UserNotFoundException(id));

이렇게 하면 선택이 강제된다. 부재를 처리하거나, 아니면 명시적으로 실패하거나. 시스템 전체를 조용히 떠도는 null은 없다.

4. 시니어는 null을 동작(behavior)으로 대체한다

시니어 개발자들이 가장 저평가되어 있다고 여기는 패턴이 있다. 바로 Null Object 패턴이다.

이렇게 하는 대신:

Discount discount = order.getDiscount();
if (discount != null) {
    price -= discount.apply(price);
}

시니어는 동작 자체를 모델링한다.

public interface Discount {
    BigDecimal apply(BigDecimal price);
}

public class NoDiscount implements Discount {
    public BigDecimal apply(BigDecimal price) {
        return price;
    }
}

그러면 코드가 이렇게 된다.

price = order.getDiscount().apply(price);

조건도 없다. 분기도 없다. 오직 의도만 남는다.

5. 시니어가 != null을 볼 때 실제로 생각하는 것

아프게 들릴 수 있지만 사실이다. 시니어 개발자가 과도한 null 검사를 보면 이렇게 생각한다.

  • 도메인 모델링이 취약하다.
  • 유효성 검증의 책임이 불분명하다.
  • 계약이 명확하지 않다.
  • 설계가 아니라 두려움이 코드를 이끌고 있다.
"왜 이 시스템은 이토록 많은 유효하지 않은 상태를 허용하고 있나?"

6. 시니어 면접에서 이것이 중요한 이유

시니어 면접은 NullPointerException을 피하는 방법에 관한 것이 아니다. 다음과 같은 질문을 다룬다.

  • 왜 이 값이 null이 될 수 있는가?
  • 부재(absence)는 허용해야 하는가, 금지해야 하는가?
  • 유효성 검증의 소유권은 누구에게 있는가?
  • 이것은 도메인 개념인가, 오류인가?

면접관은 얼마나 빠르게 코드를 짜는지가 아니라 어떻게 생각하는지를 보고 있다.

!= null이 실제로 괜찮은 경우

  • 시스템 경계(boundary)에서
  • 외부 API와 상호작용할 때
  • 레거시 시스템을 다룰 때
  • 성능이 극도로 중요한 경로에서

차이점은 단 하나다. 그들은 왜 그것이 거기 있는지 정확히 안다. 습관이 아니라, 두려움도 아니라 — 의도다.

마지막으로

주니어 개발자는 null에서 살아남는 코드를 짠다. 시니어 개발자는 null이 숨을 곳이 없는 시스템을 설계한다.

다음번에 손가락이 자동으로 이것을 타이핑하려 할 때:

if (x != null)

잠깐 멈추고 스스로에게 물어보라.

"애초에 이것이 null이 될 수 있어야 하는가?"

그 질문 하나가 시니어 수준의 사고가 시작되는 지점이다.
