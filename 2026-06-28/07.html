DuckDB: The Quiet Superpower in Data Engineering

Nexumo · 6분 읽기 · 원문 · freedium

요약

DuckDB는 서버도 클러스터도 클라우드 요금도 없이 노트북 위에서 Parquet 파일을 즉시 분석할 수 있는 '파일 속의 데이터 웨어하우스'다.

현대 데이터 엔지니어링은 클러스터 워밍업과 백필 대기라는 고질적인 병목에 시달린다. DuckDB는 프로세스 내부에서 직접 실행되는 컬럼형 OLAP 엔진으로 이 병목을 근본적으로 없앤다. Parquet 파일을 테이블처럼 즉시 쿼리하고, CI 파이프라인에서 경량 데이터 테스트를 실행하고, 로컬에서 로직을 검증한 뒤 안심하고 프로덕션에 배포하는 워크플로우가 가능해진다. 새 플랫폼을 구축하자는 얘기가 아니라, 기존 작업 흐름의 빈틈을 메우는 실용적인 유틸리티다.

레이크하우스 시대가 만들어낸 새로운 수요

솔직히 인정하자. 현대 데이터 엔지니어링에는 묘한 문제가 있다. 컴퓨트는 그 어느 때보다 풍부하고, 스토리지도 마찬가지인데, 여전히 절반의 시간을 기다리며 보낸다. 클러스터가 워밍업되기를, 잡이 끝나기를, "딱 한 번만 더" 백필이 완료되기를.

DuckDB는 그 느낌을 뒤집는다. 워크플로우 속으로 사라질 만큼 가볍고, 동시에 노트북을 진지한 분석 머신처럼 느끼게 할 만큼 강력하다.

지금 우리는 이런 세상에 살고 있다.

  • 오브젝트 스토리지와 Parquet이 어디에나 있다.
  • ELT 파이프라인은 빠른 유효성 검증을 요구한다.
  • 팀은 로컬에서 프로토타입을 만들고 나중에 전 세계에 배포한다.
  • 분석이 대화형으로 이루어지길 기대하는 문화가 자리 잡았다.

여기서 함정이 있다. 모든 질문이 분산 시스템을 필요로 하지는 않는다는 것이다. 데이터셋을 훑어보거나, Parquet 폴더 두 개를 조인하거나, 조정 쿼리를 실행하거나, 증분 모델을 검증하거나, 깔끔한 결과물을 내보내는 작업이 필요할 때—바로 그것이 DuckDB가 빛나는 영역이다.

DuckDB가 다르게 느껴지는 이유

DuckDB는 흔히 "분석을 위한 SQLite"라고 불린다. 좋은 비유지만 중요한 세부 사항을 놓친다. DuckDB는 스캔, 조인, 집계 같은 OLAP 방식의 쿼리를 위해 설계된, 컬럼형 벡터화 실행 엔진이다.

인 프로세스 실행이 모든 것을 바꾼다

DuckDB는 "배포"하는 것이 아니라 사용하는 것이다. Python 노트북, CLI 세션, 데이터 앱, CI 잡, 심지어 임베디드 서비스 등 어떤 프로세스 안에서도 직접 실행된다. 데몬도, 클러스터 핸드셰이크도, "컴퓨트 프로비저닝 티켓 제출"도 필요 없다. 이것만으로도 데이터 작업의 리듬이 "예약하고 기다리기"에서 "질문하고 바로 답 얻기"로 바뀐다.

컬럼형 + 벡터화 실행 = 꼭 필요한 순간에 빠름

왜 빠를까? DuckDB는 분석의 형태에 맞게 최적화됐기 때문이다.

  • 행(row)이 아닌 열(column) 단위로 읽는다.
  • CPU 친화적인 배치 단위로 데이터를 처리한다.
  • 필터와 프로젝션을 파일 스캔 단계에서 미리 처리한다(pushdown).
  • 필요한 곳에서 병렬 처리를 수행한다.

쉽게 말하면, Parquet를 번거로운 import 과정 없이 일급 시민으로 다루는 쿼리 엔진이다.

DuckDB가 비밀 병기가 되는 다섯 가지 상황

웨어하우스를 대체하자는 이야기가 아니다. 웨어하우스 주변의 모든 것을 업그레이드하자는 이야기다.

1) 개발용 '파일 속 웨어하우스'

데이터 엔지니어들은 원격 시스템에서 로직을 디버깅하다가 며칠을 날린다. 문제의 원인은 대개 조인 키 불일치나 타임스탬프 엣지 케이스처럼 사소한 것들이다. DuckDB를 쓰면 대표 데이터 샘플(혹은 규모가 적당한 전체 데이터셋)을 로컬로 가져와 변환 로직을 실행하고, 소프트웨어 엔지니어처럼 빠르게 반복할 수 있다. 마찰이 줄고, 피드백이 빨라지고, "감으로 커밋하는" 배포가 사라진다.

2) 파이프라인 유효성 검증 및 단위 테스트

가장 저평가된 활용 사례가 바로 테스트다. 웨어하우스 실행이 끝난 뒤에야 로직을 검증하는 대신, 더 이른 시점에 검증할 수 있다. 스키마 기대값, 유일성 제약, 조정 합계, 증분 로직을 경량으로, 결정론적으로 CI에서 확인할 수 있다.

3) Parquet 직접 분석(로드 단계 없이)

데이터가 이미 Parquet 형태라면 질문 하나를 던지기 위해 굳이 "로드"할 필요가 없다. 임시 조사, 백필 감사, "어제 지표가 왜 12% 떨어졌지?" 같은 즉각적인 점검, 다운스트림 잡을 위한 파생 Parquet 생성 등에 DuckDB는 탁월하다.

4) 제품 내장 분석

가끔 분석 기능이 시스템 안에 직접 들어가야 하는 경우가 있다. 고객 대면 대시보드, 로컬 분석 도구, 오프라인 워크플로우, 내부 어드민 패널 등이 그 예다. DuckDB는 앱을 분산 플랫폼으로 키우지 않고도 응답성을 유지시켜 주는 내장 엔진 역할을 할 수 있다.

5) 로컬과 클라우드를 잇는 하이브리드 워크플로우

"로컬에서 시작하고 필요할 때 클라우드로 확장"하는 생태계가 성장하고 있다. 노트북에서 시작해 나중에 협업이나 더 큰 컴퓨트가 필요해질 때 하이브리드 접근법이 점점 일반화되고 있다.

아키텍처 흐름: 현대 스택에서 DuckDB의 자리

실제로 사용할 수 있는 두 가지 패턴을 소개한다.

패턴 A: 로컬 우선 ELT 개발 → 클라우드 실행

(레이크의 Parquet/CSV/JSON)
                   |
                   v
        +----------------------+
        |  DuckDB (로컬 개발)   |
        |  - 프로파일링 & 디버깅 |
        |  - 빠른 조인          |
        |  - 유효성 테스트      |
        +----------------------+
                   |
            Parquet 내보내기
                   |
                   v
        +----------------------+
        |  웨어하우스 / Spark   |
        |  - 전체 규모 실행     |
        |  - 스케줄된 잡        |
        +----------------------+

패턴 B: 데이터 모델 CI 검증 게이트

Pull Request
   |
   v
dbt 모델 실행 (샘플 데이터)  --->  DuckDB 검사 (SQL 테스트)
   |                                   |
   v                                   v
빠른 실패 (저렴)                  통과 → 머지 → 전체 실행

두 패턴의 공통 주제는 하나다. DuckDB는 빠른 피드백 레이어가 된다.

실제로 재사용하게 될 코드 세 가지

1) Parquet 폴더를 테이블처럼 쿼리하기 (Python)

import duckdb

con = duckdb.connect()

# import 단계 없이 Parquet 직접 쿼리
df = con.execute("""
  SELECT
    date_trunc('day', event_time) AS day,
    count(*) AS events
  FROM read_parquet('data/events/*.parquet')
  WHERE event_type = 'checkout'
  GROUP BY 1
  ORDER BY 1
""").df()

print(df.head())

왜 중요한가: "파일 속에 있는 데이터"를 몇 초 만에 "질문을 던질 수 있는 데이터"로 바꿔준다.

2) 확장 기능으로 S3의 Parquet 읽기 (SQL)

INSTALL httpfs;
LOAD httpfs;

-- 환경 변수 또는 선호하는 방법으로 자격증명을 설정한다.
SELECT *
FROM read_parquet('s3://my-bucket/warehouse/events/2026-01/*.parquet')
LIMIT 100;

이 시점부터 DuckDB는 이동 가능한 레이크하우스 쿼리 엔진처럼 느껴진다. 디버깅과 감사 작업에 특히 유용하다.

3) 깔끔한 Parquet 결과물 생성

import duckdb

con = duckdb.connect("pipeline.duckdb")

con.execute("""
  CREATE OR REPLACE TABLE daily_revenue AS
  SELECT
    date_trunc('day', paid_at) AS day,
    sum(amount) AS revenue
  FROM read_parquet('data/payments/*.parquet')
  WHERE status = 'paid'
  GROUP BY 1
""")

con.execute("""
  COPY daily_revenue
  TO 'out/daily_revenue.parquet'
  (FORMAT PARQUET);
""")

유용한 이유: 무거운 인프라를 전혀 올리지 않고도 다운스트림 도구가 선호하는 고품질 컬럼형 결과물을 만들 수 있다.

DuckDB가 아닌 것(그리고 그것이 괜찮은 이유)

DuckDB는 다음을 목표로 하지 않는다.

  • 동시 접속이 많은 OLTP 데이터베이스
  • 수백 명의 BI 사용자를 위한 멀티테넌트 웨어하우스
  • 엔터프라이즈 데이터 플랫폼의 대체재

트랜잭션 백엔드처럼 사용하면 실망한다. 반면 이동 가능한 분석 엔진, 즉 "배포할 수 있는 쿼리 도구"로 대하면 기대 이상의 결과를 얻는다.

현실적인 사례: "깨진 지표" 문제

팀이 새 증분 모델을 배포했다. 대시보드 수치가 9% 떨어진다. 모두가 패닉 상태다.

기존 방식: 웨어하우스 쿼리를 실행하고, 로그를 살펴보고, 잡을 재실행하고, 기다린다.

DuckDB 방식:

  1. 관련 Parquet 파티션 이틀치를 로컬로 가져온다.
  2. 모델 로직을 로컬에서 실행한다.
  3. 핵심 차원별로 카운트를 비교(diff)한다.
  4. 원인을 찾는다. 대개 조인 조건, 늦게 도착한 데이터, 또는 타임존 불일치다.
  5. 자신 있게 패치하고 재실행한다.

화려하지는 않다. 그냥 빠르다. 디버깅의 속도는 곧 신뢰성의 한 형태다.

오후 한 나절에 DuckDB 도입하기

  1. 하나의 고통 지점에서 시작한다. 로컬 디버깅, 유효성 검증, 데이터 조정, Parquet 분석 중 하나를 택한다.
  2. 소규모 '데이터 툴박스 레포'를 만든다. 반복되는 점검 작업을 위한 DuckDB 스크립트를 모아둔다.
  3. CI 잡을 추가한다. 샘플 파티션에서 DuckDB 기반 테스트를 실행한다.
  4. 내보내기를 표준화한다. 다운스트림 단계를 위해 DuckDB에서 Parquet 결과물을 생산한다.
  5. 유틸리티 엔진으로 다룬다. 새로운 플랫폼을 "소유"하려 하지 말고.

마이그레이션 계획은 필요 없다. 첫 번째 성공 경험만 있으면 된다.

결론: 최고의 도구는 기다림을 없앤다

DuckDB가 과대광고 없이 주목받는 이유는, 전면 재작성을 요구하지 않기 때문이다. 데이터 엔지니어링이 느리고 불편해지는 틈새—로컬 반복, 빠른 검증, Parquet 우선 워크플로우, 경량 분석—에 조용히 스며든다.

데이터 엔지니어라면 일주일만 써볼 가치가 있다. 내려놓게 되거나, 아니면 어느새 모든 곳에 쓰게 될 것이다.
