10 Data Models Every Data Engineer Must Know (Before They Break Production)

Rohan Dutt · 11분 읽기 · 원문 · freedium

프로덕션을 망가뜨리기 전에 반드시 알아야 할 확장성·정확성·안정성을 결정하는 10가지 핵심 데이터 모델 패턴.

잘못된 데이터 모델 선택은 프로덕션 장애로 이어진다. 이 글은 스타 스키마부터 통합 서빙 레이어까지 10가지 데이터 모델을 실제 SQL 코드와 함께 상세히 다룬다. 각 모델의 핵심 문제점, 실제 재난 사례, 그리고 현대적인 대안을 제시해 어떤 상황에서 어떤 모델을 써야 하는지 명확한 기준을 제공한다. 스타·스노우플레이크 스키마의 성능 한계, 데이터 볼트의 분석 취약점, 컬럼형 저장소의 위력, 그리고 Iceberg 기반 통합 서빙 레이어까지 망라한 실전 가이드다.

프로덕션 시스템의 장애를 막는 필수 데이터 모델을 익혀보자. 올바른 모델을 선택하면 확장성, 일관성, 신뢰할 수 있는 분석이 보장된다.

10. 스타 스키마: 규모에서 실패하는 레거시 워크호스

왜 중요한가

스타 스키마는 직관적이고 분석가에게 친화적이지만, 대규모에서는 성능 병목이 된다. 특히 방대한 팩트 테이블, 높은 카디널리티 차원, 근실시간 워크로드에서 문제가 심각하다.

핵심 문제

  • 팩트 테이블이 커질수록(수십억 행) 쿼리 성능이 급격히 저하된다.
  • 원시 팩트에 대한 복잡한 조인 + GROUP BY = 느린 스캔, 타임아웃.
-- 원시 팩트 테이블에 대한 고전적인 스타 스키마 쿼리
SELECT
    d.region,
    d.store_type,
    SUM(f.sales_amount) AS total_sales
FROM fact_sales f
JOIN dim_store d  ON f.store_id = d.store_id
JOIN dim_date dt  ON f.date_id = dt.date_id
WHERE dt.date BETWEEN '2025-11-28' AND '2025-11-29'
GROUP BY d.region, d.store_type;

실제 재난: 한 유통 대기업의 블랙프라이데이 대시보드가 시간당 거래가 1천만 건을 넘어서자 타임아웃이 발생했다. 스타 스키마가 감당하지 못한 것이다.

해결책: 집계 인식 모델링

CREATE TABLE sales_daily_summary AS
SELECT
    date_id,
    store_id,
    SUM(sales_amount) AS total_sales,
    COUNT(*) AS txn_count
FROM fact_sales
GROUP BY date_id, store_id;

-- 최적화된 쿼리
SELECT
    d.region,
    d.store_type,
    SUM(s.total_sales) AS total_sales
FROM sales_daily_summary s
JOIN dim_store d  ON s.store_id = d.store_id
JOIN dim_date dt  ON s.date_id = dt.date_id
WHERE dt.date BETWEEN '2025-11-28' AND '2025-11-29'
GROUP BY d.region, d.store_type;

결과: 10~100배 빠른 쿼리, 피크 부하에서도 예측 가능한 성능.

결론: 스타 스키마는 정적이고 소용량 분석에는 적합하다. 고처리량 OLAP에서는 사전 집계, 와이드 테이블, 현대적인 레이크하우스 모델이 선택이 아니라 생존 도구다.

9. 스노우플레이크 스키마: 과설계되고 느리다

왜 중요한가

스노우플레이크 스키마는 쿼리 성능이 아닌 저장소를 최적화한다. 현대 분석(클라우드 OLAP, 대시보드, 임시 쿼리)에서 병목은 디스크가 아니라 연산이다. 과도한 정규화는 조인 깊이를 폭발시키고 지연 시간을 죽인다.

-- 스노우플레이크 스타일 쿼리 (고통스러운 버전)
SELECT
    r.region_name,
    c.category_name,
    SUM(f.sales_amount) AS total_sales
FROM fact_sales f
JOIN dim_product p        ON f.product_id = p.product_id
JOIN dim_category c       ON p.category_id = c.category_id
JOIN dim_subcategory sc   ON c.subcategory_id = sc.subcategory_id
JOIN dim_store s          ON f.store_id = s.store_id
JOIN dim_region r         ON s.region_id = r.region_id
GROUP BY r.region_name, c.category_name;

현대적 대안: 차원 비정규화

-- 평탄화된 차원 테이블
CREATE TABLE dim_product_flat AS
SELECT
    p.product_id,
    p.product_name,
    c.category_name,
    sc.subcategory_name
FROM dim_product p
JOIN dim_category c     ON p.category_id = c.category_id
JOIN dim_subcategory sc ON c.subcategory_id = sc.subcategory_id;

결론: 스노우플레이크 스키마는 어제의 저장소 문제를 해결하면서 오늘의 성능 악몽을 만든다. 대규모 분석에서는 차원을 평탄화하고, 조인을 구체화하며, 컬럼형 엔진이 무거운 작업을 처리하게 하라.

8. 데이터 볼트: 감사 가능성이 필요할 때의 엔터프라이즈 괴물

왜 중요한가

데이터 볼트는 감사 가능성, 계보, 완전한 이력화에 탁월하다. 규제 산업(금융, 의료)에서 중요하지만, 다층 아키텍처는 저지연 분석에 근본적으로 부적합하다.

  • 허브 → 비즈니스 키
  • 링크 → 관계
  • 새틀라이트 → 설명적 + 이력 속성

모든 쿼리가 시간 변형 테이블에 걸친 조인 폭발이 된다.

SELECT
    h_cust.customer_id,
    s_cust.customer_name,
    SUM(s_tx.amount) AS total_spend
FROM hub_customer h_cust
JOIN link_customer_tx l_ct  ON h_cust.hub_customer_key = l_ct.hub_customer_key
JOIN hub_transaction h_tx   ON l_ct.hub_transaction_key = h_tx.hub_transaction_key
JOIN sat_customer s_cust    ON h_cust.hub_customer_key = s_cust.hub_customer_key AND s_cust.is_current = true
JOIN sat_transaction s_tx   ON h_tx.hub_transaction_key = s_tx.hub_transaction_key AND s_tx.is_current = true
GROUP BY h_cust.customer_id, s_cust.customer_name;

결론: 데이터 볼트는 통합 및 규정 준수 레이어이지 분석 레이어가 아니다. 규제 기관이 이력을 요구할 때 사용하라. 속도를 위해서는 스트리밍 CDC + 차원 또는 와이드 마트를 위에 레이어링하라.

7. 시계열 혼돈을 위한 와이드 컬럼 저장소 (Cassandra, Bigtable)

왜 중요한가

와이드 컬럼 데이터베이스는 쓰기가 멈추지 않는 고속 수집(IoT, 메트릭, 로그)에서 지배적이다. 하지만 쿼리 유연성을 희생한다. 조인 없음, 제한된 필터링, 엄격한 접근 패턴.

-- 올바른 테이블 설계 (접근 패턴 기반)
CREATE TABLE device_metrics (
    device_id TEXT,
    metric_date DATE,
    event_ts TIMESTAMP,
    metric_type TEXT,
    metric_value DOUBLE,
    PRIMARY KEY (
        (device_id, metric_date),  -- 파티션 키
        event_ts                   -- 클러스터링 키
    )
) WITH CLUSTERING ORDER BY (event_ts DESC);

-- 잘 확장되는 쿼리
SELECT event_ts, metric_value
FROM device_metrics
WHERE device_id = 'dev_42'
  AND metric_date = '2025-12-26'
  AND event_ts >= '2025-12-26 10:00:00'
  AND event_ts <  '2025-12-26 11:00:00';

결론: 와이드 컬럼 저장소는 쓰기 최적화된 시계열 엔진이지 분석 데이터베이스가 아니다. 미리 쿼리에 맞게 설계하거나, OLAP 엔진과 페어링하라.

6. 숨겨진 관계를 위한 그래프 모델 (Neo4j, TigerGraph)

왜 중요한가

인사이트가 관계에 있을 때(사기 고리, 소셜 영향력, 네트워크 홉), 관계형 조인은 재귀 깊이에서 무너진다. 그래프 데이터베이스는 관계를 일급 시민으로 취급해 다중 홉 순회를 빠르고 자연스럽게 만든다.

-- SQL에서 친구의 친구 (2홉) - 안티패턴
SELECT DISTINCT f2.user_id
FROM friendships f1
JOIN friendships f2 ON f1.friend_id = f2.user_id
WHERE f1.user_id = 123 AND f2.user_id <> 123;

-- 그래프 모델 (승리)
MATCH (u:User {id: 123})-[:FRIEND*2..4]->(conn)
RETURN DISTINCT conn.id
LIMIT 100;

-- 사기 탐지
MATCH (a:Account)-[:TRANSFER]->(b:Account)-[:TRANSFER]->(c:Account)
WHERE a.country <> c.country
RETURN a, b, c;

결론: "누가 누구와 연결되어 있는가"로 시작하는 질문에는 SQL이 잘못된 도구다. 관계 중심 인사이트에는 그래프 데이터베이스가 조인보다 수십 배 뛰어나다.

5. 스트리밍 이벤트 소싱 (Kafka + CDC)

왜 중요한가

배치 ETL은 실시간 시스템과 근본적으로 호환되지 않는다. CDC는 데이터베이스 변이를 불변 이벤트로 전환해 거의 제로에 가까운 지연 파이프라인, 재생 가능한 상태, 마이크로서비스 간 시스템 전체 일관성을 가능하게 한다.

-- 소스 테이블
CREATE TABLE orders (
    order_id BIGINT PRIMARY KEY,
    user_id  BIGINT,
    status   TEXT,
    amount   DECIMAL(10,2),
    updated_at TIMESTAMP
);

-- CDC 이벤트 (Debezium 스타일)
{
  "op": "u",
  "before": { "status": "CREATED" },
  "after":  { "status": "PAID" },
  "ts_ms": 1735209123123,
  "source": { "table": "orders" }
}

-- 멱등성 싱크 (핵심)
INSERT INTO orders_state (order_id, status, amount)
VALUES (:order_id, :status, :amount)
ON CONFLICT (order_id)
DO UPDATE SET
  status = EXCLUDED.status,
  amount = EXCLUDED.amount;

결론: 이벤트 소싱은 최적화가 아니라 아키텍처 전환이다. 실시간 일관성, 감사 가능성, 복구가 필요하다면 Kafka + CDC가 근간이다. 단, 정확성을 먼저 엔지니어링해야 한다.

4. 저렴하고 빠른 분석을 위한 컬럼형 저장소 (Parquet, Delta Lake)

왜 중요한가

행 기반 데이터베이스는 포인트 조회에 최적화되어 있지 분석 워크로드에는 그렇지 않다. 분석 워크로드는 수십억 행에 걸쳐 몇 개의 컬럼만 읽는다. 컬럼형 저장소가 바로 이를 위해 만들어졌다.

CREATE TABLE sales_parquet (
    order_id BIGINT,
    region   STRING,
    amount   DECIMAL(10,2),
    order_ts TIMESTAMP
)
USING PARQUET
PARTITIONED BY (region, order_date);

-- 최적화된 쿼리
SELECT
    region,
    SUM(amount) AS total_sales
FROM sales_parquet
WHERE order_date = '2025-12-25'
  AND region = 'US'
GROUP BY region;

-- Delta Lake MERGE (실제 운영 안전성)
MERGE INTO sales_parquet t
USING updates u ON t.order_id = u.order_id
WHEN MATCHED THEN UPDATE SET amount = u.amount
WHEN NOT MATCHED THEN INSERT *;

결론: 컬럼형 저장소는 원시 데이터를 분석 슈퍼파워로 전환한다. 행 저장소에서 빅 데이터를 스캔한다면 돈과 시간을 낭비하는 것이다.

3. 멀티 모델 하이브리드 (SQL + NoSQL 충돌 시)

왜 중요한가

실제 데이터는 하나의 형태가 아닌 경우가 많다. 현대 앱은 관계형 팩트, 반구조화된 JSON, 관계를 혼합한다. 멀티 모델 데이터베이스를 사용하면 어색한 ETL이나 데이터 중복 없이 한 곳에서 모든 것을 쿼리할 수 있다.

CREATE TABLE listings (
    listing_id BIGINT PRIMARY KEY,
    city TEXT,
    price NUMERIC,
    attributes JSONB
);

-- 구조화 + JSON 쿼리
SELECT
    city,
    COUNT(*) AS luxury_listings
FROM listings
WHERE price > 300
  AND attributes @> '{"amenities": ["pool", "wifi"]}'
  AND (attributes->>'pet_friendly')::boolean = true
GROUP BY city;

-- 성능을 위한 인덱싱
CREATE INDEX idx_listings_attrs
ON listings
USING GIN (attributes);

결론: 멀티 모델 하이브리드는 실용적인 타협안이다. 데이터가 테이블, 문서, 관계에 걸쳐 있을 때 뷰와 인덱스 뒤에 복잡성을 숨기는 한 민첩성을 제공한다.

2. 리버스 ETL (운영 분석) — 데이터를 앱에 돌려놓기

왜 중요한가

전통적인 분석은 대시보드에서 멈춘다. 리버스 ETL은 웨어하우스에서 모델링된 데이터를 운영 도구에 다시 밀어넣어 루프를 닫는다. 그래서 인사이트가 보고서가 아니라 행동을 이끈다.

-- customer_360.sql
SELECT
    customer_id,
    MAX(last_login_ts)              AS last_active,
    SUM(lifetime_spend)             AS ltv,
    COUNT(DISTINCT order_id)        AS order_count,
    CASE
        WHEN SUM(lifetime_spend) > 5000 THEN 'VIP'
        WHEN SUM(lifetime_spend) > 1000 THEN 'High'
        ELSE 'Standard'
    END AS customer_tier
FROM orders
GROUP BY customer_id;

-- 리버스 ETL 싱크 흐름 (개념적)
warehouse.customer_360
   → Hightouch / Census
       → Salesforce (Account Tier)
       → HubSpot (Lifecycle Stage)
       → Zendesk (Support Priority)

-- 증분 업데이트
SELECT *
FROM customer_360
WHERE updated_at > :last_sync_ts;

결론: 리버스 ETL은 웨어하우스를 보고 무덤에서 운영 엔진으로 변환한다. 인사이트가 고객 대면 도구에 도달하지 않는다면 가치를 창출하지 못하는 것이다.

1. 통합 서빙 레이어 — 프로덕션 데이터의 미래

왜 중요한가

현대 데이터 스택은 OLTP, OLAP, 검색, 스트리밍 시스템에 걸쳐 데이터를 분산시켜 동기화 지연과 중복 로직을 만든다. 통합 서빙 레이어는 하나의 논리적 데이터 레이어(Iceberg/Hudi/Delta)를 다양한 접근 모드로 사용한다: SQL 분석, 근실시간 읽기, ML, 심지어 그래프/검색 워크로드까지.

-- Iceberg 통합 테이블
CREATE TABLE user_events (
    user_id BIGINT,
    event_type STRING,
    event_ts TIMESTAMP,
    attributes MAP<STRING, STRING>
)
USING ICEBERG
PARTITIONED BY (days(event_ts));

-- OLAP 쿼리 (Trino / Spark SQL)
SELECT
    event_type,
    COUNT(*) AS cnt
FROM user_events
WHERE event_ts >= current_date - INTERVAL '7' DAY
GROUP BY event_type;

-- 증분 / 근실시간 읽기 (타임 트래블)
SELECT *
FROM user_events
FOR SYSTEM_TIME AS OF TIMESTAMP '2025-12-26 10:00:00';

결론: 통합 서빙 레이어는 최종 아키텍처다: 하나의 데이터셋, 다양한 워크로드, 사일로 없음. 다음 10년을 위해 설계한다면 오픈 테이블 형식은 선택이 아니다.

데이터 엔지니어로서 이 10가지 모델을 이해하면 단순히 데이터 파이프라인을 구축하는 것을 넘어, 올바른 도구를 올바른 문제에 적용하는 진정한 아키텍처 판단력을 갖추게 된다.
