Value Objects in .NET 10: Stop Passing Strings Around Your Domain

Michael Maurice · 13 min read · 원문 · freedium

.NET에서 원시 타입(특히 string) 과도 의존을 Value Object 패턴으로 해결하여 타입 안전하고 자기 검증하는 도메인 모델을 만드는 방법을 실전 코드로 안내한다.

원시 타입 집착(Primitive Obsession)은 개발자가 이메일, 금액, 주소 같은 복잡한 도메인 개념을 string, int, decimal로 표현할 때 발생하는 코드 악취다. 이로 인해 파라미터 순서 오입력, 유효성 검사 로직 중복, 컴파일 타임 보호 부재 같은 문제가 생긴다. Value Object는 불변성, 값 동등성, 자기 검증의 세 가지 특성으로 이 문제를 해결한다. 이 글은 추상 기반 클래스부터 이메일, 이름, 금액, 주소 같은 구체적인 Value Object 구현까지, 그리고 Entity Framework Core와의 통합 방법까지 .NET 코드 예제로 상세히 설명한다. 도메인 모델이 자기 문서화되고 런타임 오류가 컴파일 타임 오류로 전환되는 효과를 실제로 느낄 수 있다.

원시 타입 집착(Primitive Obsession)은 소프트웨어 개발에서 가장 만연한 코드 악취 중 하나다. 개발자가 복잡한 도메인 개념을 표현하기 위해 내장 원시 타입(문자열, 정수, 소수)에 과도하게 의존할 때 발생하며, 불명확한 코드, 분산된 유효성 검사 로직, 그리고 버그 위험 증가로 이어진다. Value Object는 이 문제에 대한 우아한 해결책으로, 타입 안전성, 캡슐화, 표현력을 제공한다.

원시 타입 집착 이해하기

원시 타입 집착이란?

원시 타입 집착은 복잡한 도메인 개념을 표현하기 위해 원시 데이터 타입을 과도하게 사용하는 것이다. 의미 있는 추상화를 만드는 대신 개발자들이 비즈니스 엔티티를 모델링하는 데 string, int, decimal같은 기본 타입에 의존한다.

원시 타입 집착의 흔한 예시:

  • 이메일 주소, 전화번호, 우편번호에 string 사용
  • 통화 정보 없이 금액을 decimal로 표현
  • 상태 코드에 열거형 대신 int 상수 사용
  • 주소를 별도의 string 속성들로 모델링
  • 시간대나 비즈니스 맥락 없이 DateTime 사용

원시 타입 집착이 유발하는 문제

원시 타입 집착이 해로운 이유:

  • 표현력 부족: string은 이메일일 수도 있고, 이름일 수도 있고, 주소일 수도 있다 — 아무런 의미론적 의미를 제공하지 않는다
  • 오류 위험 증가: 잘못된 파라미터를 전달하거나 유사한 원시 타입을 혼동하기 쉽다
  • 분산된 유효성 검사: 코드베이스 전체에 유효성 검사 로직이 중복된다
  • 약한 타입 안전성: 잘못된 연산에 대한 컴파일 타임 보호가 없다
  • 어려운 리팩토링: 비즈니스 규칙 변경이 코드베이스 전체에 걸친 수정을 요구한다
// 나쁜 예: 원시 타입 집착
public class User
{
    public string FirstName { get; set; } = string.Empty;  // 규칙이 뭔가요?
    public string LastName { get; set; } = string.Empty;   // 비어있을 수 있나요?
    public string Email { get; set; } = string.Empty;      // 검증됐나요?
    public decimal Salary { get; set; }                    // 어떤 통화?
    public int Status { get; set; }                        // 숫자들의 의미는?
}

// 위험한 사용법 - 컴파일되지만 잘못됨
var user = new User
{
    FirstName = "user@example.com",  // 이메일이 이름 필드에!
    Email = "John",                   // 이름이 이메일 필드에!
    Salary = -50000,                  // 음수 급여?
    Status = 99                       // 유효하지 않은 상태 코드
};

Value Object 소개

Value Object란?

Value Object는 고유한 정체성 없이 도메인의 서술적 측면을 표현하는 불변 객체다. 속성만으로 정의되며 그것을 사용하는 엔티티와 독립적인 생명주기가 없다.

Value Object의 핵심 특성:

  • 불변성: 생성 후 변경 불가
  • 값 동등성: 모든 속성이 같으면 두 객체는 동등
  • 자기 검증: 항상 유효한 상태
  • 정체성 없음: 고유 식별자가 아닌 값으로 정의됨
  • 교체 가능: 수정보다는 전체 교체

.NET에서 Value Object 구현하기

기본 Value Object 구현

모든 Value Object의 기반을 만든다:

// Domain/Common/ValueObject.cs
namespace Domain.Common;

public abstract class ValueObject : IEquatable<ValueObject>
{
    protected abstract IEnumerable<object?> GetEqualityComponents();

    public override bool Equals(object? obj)
    {
        return obj is ValueObject other && Equals(other);
    }

    public bool Equals(ValueObject? other)
    {
        return other is not null &&
               GetEqualityComponents().SequenceEqual(other.GetEqualityComponents());
    }

    public override int GetHashCode()
    {
        return GetEqualityComponents()
            .Where(x => x != null)
            .Aggregate(1, (current, obj) => current * 23 + obj!.GetHashCode());
    }

    public static bool operator ==(ValueObject? left, ValueObject? right)
        => Equals(left, right);

    public static bool operator !=(ValueObject? left, ValueObject? right)
        => !Equals(left, right);
}

EmailAddress Value Object

// Domain/ValueObjects/EmailAddress.cs
namespace Domain.ValueObjects;

public sealed class EmailAddress : ValueObject
{
    public string Value { get; }

    private EmailAddress(string value) => Value = value;

    public static Result<EmailAddress> Create(string? email)
    {
        if (string.IsNullOrWhiteSpace(email))
            return Result.Failure<EmailAddress>("이메일은 비어 있을 수 없습니다.");

        email = email.Trim().ToLowerInvariant();

        if (email.Length > 254)
            return Result.Failure<EmailAddress>("이메일이 너무 깁니다.");

        if (!Regex.IsMatch(email, @"^[^@\s]+@[^@\s]+\.[^@\s]+$"))
            return Result.Failure<EmailAddress>($"'{email}'은 유효하지 않은 이메일 형식입니다.");

        return Result.Success(new EmailAddress(email));
    }

    protected override IEnumerable<object?> GetEqualityComponents()
    {
        yield return Value;
    }

    public override string ToString() => Value;
    public static implicit operator string(EmailAddress email) => email.Value;
}

Money Value Object

// Domain/ValueObjects/Money.cs
namespace Domain.ValueObjects;

public sealed class Money : ValueObject
{
    public decimal Amount { get; }
    public string Currency { get; }

    private static readonly HashSet<string> ValidCurrencies =
        new() { "USD", "EUR", "GBP", "KRW", "JPY" };

    private Money(decimal amount, string currency)
    {
        Amount = amount;
        Currency = currency;
    }

    public static Result<Money> Create(decimal amount, string? currency)
    {
        if (string.IsNullOrWhiteSpace(currency))
            return Result.Failure<Money>("통화는 비어 있을 수 없습니다.");

        currency = currency.Trim().ToUpperInvariant();

        if (!ValidCurrencies.Contains(currency))
            return Result.Failure<Money>($"'{currency}'은 지원되지 않는 통화입니다.");

        if (amount < 0)
            return Result.Failure<Money>("금액은 음수일 수 없습니다.");

        if (decimal.Round(amount, 2) != amount)
            return Result.Failure<Money>("금액은 소수점 2자리까지만 가능합니다.");

        return Result.Success(new Money(amount, currency));
    }

    public Money Add(Money other)
    {
        if (Currency != other.Currency)
            throw new InvalidOperationException($"다른 통화는 더할 수 없습니다: {Currency} vs {other.Currency}");
        return new Money(Amount + other.Amount, Currency);
    }

    protected override IEnumerable<object?> GetEqualityComponents()
    {
        yield return Amount;
        yield return Currency;
    }

    public override string ToString() => $"{Amount:F2} {Currency}";
}

Address Value Object

// Domain/ValueObjects/Address.cs
namespace Domain.ValueObjects;

public sealed class Address : ValueObject
{
    public string Street { get; }
    public string City { get; }
    public string PostalCode { get; }
    public string Country { get; }

    private Address(string street, string city, string postalCode, string country)
    {
        Street = street;
        City = city;
        PostalCode = postalCode;
        Country = country;
    }

    public static Result<Address> Create(
        string? street, string? city,
        string? postalCode, string? country)
    {
        var errors = new List<string>();

        if (string.IsNullOrWhiteSpace(street)) errors.Add("거리는 필수입니다.");
        if (string.IsNullOrWhiteSpace(city)) errors.Add("도시는 필수입니다.");
        if (string.IsNullOrWhiteSpace(postalCode)) errors.Add("우편번호는 필수입니다.");
        if (string.IsNullOrWhiteSpace(country)) errors.Add("국가는 필수입니다.");

        if (errors.Any())
            return Result.Failure<Address>(string.Join(", ", errors));

        return Result.Success(new Address(
            street!.Trim(), city!.Trim(),
            postalCode!.Trim(), country!.Trim()));
    }

    protected override IEnumerable<object?> GetEqualityComponents()
    {
        yield return Street;
        yield return City;
        yield return PostalCode;
        yield return Country;
    }

    public override string ToString() =>
        $"{Street}, {City}, {PostalCode}, {Country}";
}

Entity Framework Core와의 통합

EF Core에서 Value Object는 소유(Owned) 타입으로 매핑한다:

// Infrastructure/Persistence/Configurations/EmployeeConfiguration.cs
public class EmployeeConfiguration : IEntityTypeConfiguration<Employee>
{
    public void Configure(EntityTypeBuilder<Employee> builder)
    {
        builder.HasKey(e => e.Id);

        // EmailAddress Value Object 매핑
        builder.OwnsOne(e => e.Email, email =>
        {
            email.Property(e => e.Value)
                .HasColumnName("Email")
                .HasMaxLength(254)
                .IsRequired();
            email.HasIndex(e => e.Value).IsUnique();
        });

        // Money Value Object 매핑
        builder.OwnsOne(e => e.Salary, salary =>
        {
            salary.Property(s => s.Amount)
                .HasColumnName("SalaryAmount")
                .HasPrecision(18, 2)
                .IsRequired();
            salary.Property(s => s.Currency)
                .HasColumnName("SalaryCurrency")
                .HasMaxLength(3)
                .IsRequired();
        });

        // Address Value Object 매핑
        builder.OwnsOne(e => e.HomeAddress, addr =>
        {
            addr.Property(a => a.Street).HasColumnName("Street");
            addr.Property(a => a.City).HasColumnName("City");
            addr.Property(a => a.PostalCode).HasColumnName("PostalCode");
            addr.Property(a => a.Country).HasColumnName("Country");
        });
    }
}

Value Object 구현 모범 사례

1. 항상 private 생성자 사용

팩토리 메서드를 통해서만 생성하여 유효하지 않은 객체가 만들어지는 것을 방지한다.

2. 모든 검증을 Value Object 내부에

검증 로직이 도메인 전체에 분산되는 것을 방지한다. 유효하지 않은 Value Object가 존재할 수 없어야 한다.

3. 불변성 강제

모든 속성을 { get; private set; } 또는 { get; init; }으로 선언한다. 내부 컬렉션의 경우 IReadOnlyList 사용을 고려한다.

4. 의미 있는 연산 제공

도메인과 관련된 메서드를 Value Object에 추가한다. 예: Money.Add(), EmailAddress.GetDomain().

5. 암묵적 변환 고려

기존 코드와의 통합을 위해 implicit operator를 활용하면 점진적 마이그레이션이 쉬워진다.

마치며

Value Object는 단순한 패턴이 아니다. 그것은 코드가 하는 말을 도메인 언어로 바꾸는 방법이다. string을 EmailAddress로, decimal을 Money로 바꾸는 순간 세 가지 일이 동시에 일어난다:

  • 런타임 오류가 컴파일 타임 오류가 된다 — 이메일 필드에 이름을 넣으면 컴파일러가 잡는다
  • 코드가 자기 문서화된다 — Employee(EmailAddress, PersonName, Money)는 Employee(string, string, string, decimal, string)보다 훨씬 명확하다
  • 비즈니스 규칙이 한 곳에 모인다 — 이메일 검증 로직이 여러 서비스에 흩어지지 않고 EmailAddress.Create() 안에 있다

처음에는 보일러플레이트처럼 느껴질 수 있다. 하지만 새 팀원이 잘못된 파라미터 순서로 버그를 만들거나, 검증 없는 빈 이메일이 데이터베이스에 들어가거나, 통화가 다른 금액끼리 더해지는 사고가 일어난 후에는 — 보일러플레이트가 보험처럼 느껴진다.

도메인이 자신의 언어로 말하게 하라. string을 넘기는 것을 멈추고, 의미를 넘겨라.
