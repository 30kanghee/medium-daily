Google L5 level Interview: If You Join 10 Tables in an SQL Query, How Would You Optimize Performance?

Umesh Kumar Yadav · Javarevisited · 4분 읽기 · 원문 · freedium

10개 테이블을 조인하는 느린 SQL 쿼리를 Google L5 면접 수준으로 체계적으로 진단하고 최적화하는 7가지 전략.

프로덕션 환경에서 10개 테이블을 조인하며 30초 이상 걸리는 SQL 쿼리를 어떻게 최적화할 것인지는 Google L5 면접에서 실제로 출제된 질문이다. 단순히 SQL을 잘 작성하는 것이 아니라 체계적인 진단 방법론과 다층적 최적화 기술을 조합하는 능력이 핵심이다. 진단 단계에서는 EXPLAIN 실행 계획 분석, 프로파일링으로 병목 단계 파악, DB 파라미터 확인을 먼저 해야 한다. 최적화는 인덱스 추가 같은 간단한 방법부터 애플리케이션 레이어 분리, 임시 테이블 물리화, OLAP 엔진 전환까지 7단계로 단계적으로 접근해야 한다.

"프로덕션 환경에서 10개 테이블을 조인하고 실행에 30초 이상 걸리는 SQL 쿼리가 있다면, 성능을 어떻게 단계적으로 진단하고 최적화하겠습니까?"

진정한 어려움은 단순히 SQL을 작성하는 것이 아니라, 체계적인 문제 해결 방법론을 적용하고 다층적인 최적화 기술을 결합하는 데 있습니다.

이 글은 해당 주제를 상세히 다루며, 면접과 실제 현업 모두에서 활용할 수 있는 구조화된 접근 방식을 제공합니다.

1. 병목 지점을 파악하는 세 가지 단계

10개 테이블을 조인하는 느린 SQL 쿼리를 만났을 때, 쿼리를 바로 수정하려 들지 마세요. 대신 다음 진단 단계를 따라 근본 원인을 파악하세요.

1.1 EXPLAIN으로 실행 계획 분석

EXPLAIN 명령을 실행하여 각 테이블의 액세스 타입을 확인합니다. 다음 항목에 주의를 기울이세요:

  • type 컬럼: ALL(전체 테이블 스캔) 또는 index(전체 인덱스 스캔) 같은 값은 최적화 기회를 나타냅니다. 이상적인 값은 ref, eq_ref, 또는 const입니다.
  • rows 컬럼: 스캔할 행 수를 추정합니다. 예상보다 훨씬 높은 수치는 추가 인덱스가 필요함을 시사합니다.
  • Extra 컬럼: Using temporary(임시 테이블) 또는 Using filesort(파일 정렬)의 존재는 주요 성능 병목을 나타냅니다.

예시:

EXPLAIN SELECT o.id, o.amount, u.name, p.title
FROM orders o
JOIN users u ON o.user_id = u.id
JOIN products p ON o.product_id = p.id
WHERE o.status = 'PAID';

users 테이블의 type이 ALL을 나타낸다면, user_id에 인덱스가 없어서 전체 테이블 스캔이 발생하는 것입니다 — 명확한 최적화 대상입니다.

1.2 실제 SQL 실행 시간 분포 확인

SET profiling = 1;
-- 느린 SQL 쿼리 실행
SELECT ...;
SHOW PROFILES;
SHOW PROFILE FOR QUERY 1;

이를 통해 각 단계(예: "sending data", "creating sort index")에서 소요된 시간을 파악하여 I/O 병목인지 CPU/정렬 병목인지 구분할 수 있습니다.

1.3 데이터베이스 파라미터 설정 검토

  • join_buffer_size: 값이 작으면 다중 스캔이 발생할 수 있습니다.
  • tmp_table_size / max_heap_table_size: 값이 부족하면 임시 테이블이 디스크로 넘어갑니다.
  • innodb_buffer_pool_size: 자주 접근하는 "핫" 데이터를 수용할 수 있는지 확인합니다.

2. 10개 테이블 조인이 느린 이유

MySQL(InnoDB)에서 다중 테이블 조인은 기본적으로 Nested Loop Join 알고리즘을 사용합니다. 드라이빙 테이블에서 행을 선택하고 이후 테이블을 순환하여 일치 항목을 찾습니다.

대략적인 시간 복잡도: 드라이빙 테이블의 행 수 × 각 연관 테이블당 인덱스 스캔 비용.

드라이빙 테이블에 100,000개의 행이 있고 각 연관 테이블 인덱스 스캔에 1ms가 걸린다면, 10개 테이블의 경우 최악의 경우 100,000 × (10 × 1ms) = 1,000초에 달할 수 있습니다.

MySQL 8.0.18 이후부터는 특정 조건 하에 Hash Join을 사용할 수 있어 인덱스가 있을 때 성능이 향상되지만, 메모리와 해시 테이블 구성 오버헤드의 제약이 여전히 존재합니다.

3. SQL부터 아키텍처까지: 7가지 최적화 기법

다음 전략들은 비용이 낮고 빠른 SQL 레벨 변경부터 더 많은 노력이 필요한 아키텍처 솔루션까지 순차적으로 제시됩니다.

무기 1: 인덱스 최적화 (가장 빠른 효과)

ON과 WHERE 절에 사용된 모든 컬럼이 인덱스를 가지는지 확인합니다. LEFT JOIN의 경우, 오른쪽 테이블의 조인 컬럼에 인덱스를 추가합니다. 복합 인덱스는 가장 왼쪽 접두사 원칙을 따라야 합니다.

최적화 예시:

-- orders에 복합 인덱스 추가
ALTER TABLE orders ADD INDEX idx_create_user (create_time, user_id);

-- 연관 테이블에 인덱스 추가
ALTER TABLE users ADD INDEX idx_vip (vip_level);
ALTER TABLE categories ADD INDEX idx_status (status);

장점: 간단하고 직접적이며 비침습적입니다.

단점: 과도한 인덱스는 쓰기 성능을 저하시킬 수 있습니다.

적합한 경우: 선택도가 높은 컬럼(중복값이 적은 컬럼).

무기 2: JOIN 순서 조정

가장 작은 결과 집합이 조인을 드라이빙하도록 합니다. 필요한 경우 STRAIGHT_JOIN을 사용하여 순서를 강제합니다.

예시:

-- 작은 테이블(블랙리스트)이 조인을 드라이브하도록 강제
SELECT STRAIGHT_JOIN o.*
FROM blacklist b
JOIN orders o ON b.user_id = o.user_id;

무기 3: JOIN 분리 후 애플리케이션 레이어에서 조합

적당한 데이터 볼륨의 목록 표시 쿼리의 경우, 데이터를 별도의 쿼리로 가져와 애플리케이션 코드(예: Java)에서 결과를 병합합니다.

이 접근 방식은 데이터베이스 부하를 줄이고 대규모 카테시안 곱을 방지합니다. IN 절을 사용한 배치 쿼리(제한 초과를 방지하기 위해 500개씩 분할)를 사용하고 효율적인 인메모리 조합을 위해 맵을 활용합니다.

장점: 데이터베이스 부하를 낮추고 독립적인 쿼리 최적화를 가능하게 합니다.

적합한 경우: 여러 연관 테이블이 있는 적당한 메인 테이블 크기(<100,000행).

무기 4: 임시 테이블 또는 파생 테이블

자주 재사용되는 중간 결과를 물리화합니다.

예시:

CREATE TEMPORARY TABLE tmp_user_stat (
    user_id BIGINT PRIMARY KEY,
    total DECIMAL(10,2),
    INDEX(user_id)
) ENGINE=InnoDB;

INSERT INTO tmp_user_stat (user_id, total)
SELECT user_id, SUM(amount) FROM orders GROUP BY user_id;

무기 5: 구체화된 뷰 또는 집계 테이블

고정된 BI 스타일 보고서의 경우, 예약된 ETL 작업을 통해 결과를 미리 계산하고 전용 집계 테이블에 저장합니다. 쿼리는 그런 다음 밀리초 수준의 응답을 위해 집계 테이블을 조회합니다.

무기 6: OLAP 엔진으로 전환

복잡한 분석 쿼리를 와이드 테이블 및 스타 스키마 조인에 뛰어난 ClickHouse나 Apache Doris 같은 컬럼형 데이터베이스로 마이그레이션합니다.

무기 7: 수직 파티셔닝 + 읽기/쓰기 분리

와이드 테이블을 기본(고빈도)과 확장(대용량 텍스트, 저빈도) 테이블로 분리합니다. 복잡한 읽기 쿼리를 레플리카 데이터베이스로 라우팅합니다.

4. 최적화 방법 비교

5. 면접 답변 예시

"면접에서 다음과 같이 답변할 수 있습니다:

먼저 EXPLAIN을 실행하여 전체 테이블 스캔, 임시 테이블, 파일 정렬을 파악할 것입니다. 그다음 적절한 인덱스를 추가하거나 STRAIGHT_JOIN을 사용하여 조인 순서를 조정할 것입니다.

SQL 레벨 튜닝이 여전히 불충분하다면, 메모리에서 조합되는 여러 타겟 쿼리 같은 애플리케이션 레이어 변경을 고려하거나 임시 테이블을 사용하여 중간 결과를 물리화할 것입니다.

최적화는 단일 기술로 달성되는 것이 아니라 데이터 볼륨과 비즈니스 요구사항에 맞춘 인덱싱, 쿼리 재작성, 아키텍처 조정의 조합을 통해 달성됩니다.

또한 performance_schema 또는 프로파일링 도구를 활용하여 병목이 I/O 바운드인지 CPU 바운드인지 확인할 것입니다."

결론

복잡한 다중 테이블 조인을 최적화하는 데는 보편적인 해결책이 없습니다. 성공은 철저한 진단, 데이터 특성, 그리고 단기 이익과 장기 유지보수성 간의 균형에 달려 있습니다.

이러한 계층화된 기술을 마스터함으로써 개발자와 아키텍트는 가장 까다로운 10개 테이블 쿼리도 효율적이고 확장 가능한 운영으로 변환할 수 있습니다.
