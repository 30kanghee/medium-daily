6개월을 낭비했다 — Claude Code를 잘못 쓰고 있었다. 모든 것을 바꿔준 14가지 명령어

Mouez Yazidi · Towards AI · 9 min read · 원문 · freedium

Claude Code를 6개월간 명령어 하나만 알고 써온 개발자가, 그 누구도 제대로 알려주지 않았던 14개의 숨겨진 명령어를 발견하고 생산성을 완전히 바꾼 경험담이다.

저자는 긴 디버깅 세션에서 동료가 우연히 보여준 /compact 명령어 하나로 Claude Code의 숨겨진 생태계가 있음을 깨달았다. 이 글은 14개 명령어를 4개 범주로 나눠 소개한다: 한 번 설정으로 영구 적용되는 셋업 명령어(/init, /memory, /pr_comments), 매일 사용하는 핵심 명령어(/btw, /compact, !, /cost), 고급 사용자 기능(/fast, /review, /model), 그리고 위기 상황용 명령어(/clear, /doctor, /terminal-setup, /help)다. 저자는 이 명령어들을 통해 200시간 이상의 시간을 절약했다고 밝히며, 7일 마스터리 챌린지를 제안한다.

고백

화요일 새벽 2시였습니다. 30분이면 끝날 디버깅 세션이 세 시간째 이어지고 있었습니다. Claude Code 채팅 창에는 147개의 메시지가 뒤엉켜 있었고, 컨텍스트 창은 한계를 넘어 응답은 점점 느려지고 있었습니다. 탭 간에 코드를 복사 붙여넣기하면서 생각의 흐름을 잃고, 도구와 싸우는 느낌이 들었습니다.

그때 시니어 개발자 친구가 아무렇지도 않게 말했습니다. "/compact 써봤어?"

저는 그를 멍하니 바라봤습니다. "뭐라고?"

그는 Claude 세션에 세 글자를 입력했습니다. 전체 대화가 깔끔한 요약으로 압축되었습니다. 컨텍스트가 확보되었습니다. 우리는 아무 막힘 없이 계속 작업했습니다.

그 순간, 부끄러운 사실을 깨달았습니다. 저는 6개월 동안 명령어 하나만 알고 Claude Code를 써왔습니다. 그냥 코드를 입력하고 결과를 기대했을 뿐입니다. 그러는 동안, 생산성을 극대화해주는 14개의 명령어가 눈에 보이는 곳에 숨어 있었습니다.

이후 몇 주 동안 이 명령어들을 발견하면서 200시간 이상을 절약했습니다. 이것이 첫날부터 누군가 알려줬으면 했던 가이드입니다.

파트 1: 셋업 명령어 — 한 번 설정, 영구 적용

아무도 이야기하지 않는 문제

모든 새 프로젝트는 같은 방식으로 시작됩니다: 빈 에디터, 빈 머릿속.

첫 한 시간은 컨텍스트 설정에만 씁니다. 기술 스택, 코딩 스타일, 프로젝트 구조를 설명하는 반복적이고 소모적인 작업입니다. 첫 코드 한 줄도 쓰기 전에 의욕이 꺾입니다.

저도 새 프로젝트 시작이 싫었습니다. 그러다 모든 것을 바꿔놓은 것을 발견했습니다. 바로 /init입니다.

명령어 하나의 마법

/init: CLAUDE.md 자동 생성

프로젝트 문서를 직접 작성하는 대신, /init을 입력하고 자리를 뜨면 됩니다. Claude가 프로젝트 전체 구조를 읽고 코드베이스를 분석해 다음 항목들이 포함된 포괄적인 CLAUDE.md 파일을 작성합니다:

  • 프로젝트 요약
  • 아키텍처 결정 사항
  • 코딩 컨벤션
  • 주요 의존성
  • 설정 지침

이전: 프로젝트당 45분의 수동 문서 작업
이후: 30초, 명령어 하나, 완료

하지만 여기서 더 좋아집니다. /init을 몇 번 쓰다 보면 여전히 반복하고 있다는 것을 깨닫게 됩니다. 모든 프로젝트마다 Claude에게 같은 것들을 말하고 있습니다:

  • 함수형 프로그래밍 패턴을 선호한다.
  • 항상 에러 처리를 추가한다.
  • TypeScript strict mode를 사용한다.

그때 /memory를 만나게 됩니다.

이 명령어는 글로벌 메모리 파일을 열어줍니다. 모든 프로젝트, 모든 세션에 영구적으로 적용되는 설정입니다. 선호하는 톤, 코딩 표준, 불필요한 습관들을 한 번 설정하면 Claude가 기억합니다.

실제 메모리 파일 예시:

- Always suggest tests first
- Prefer composition over inheritance
- Explain complex logic with comments
- Never use any without explicit permission

이것이 절약해준 시간은 이루 헤아릴 수 없습니다. 하지만 정신적 에너지는? 정말 귀중합니다.

코드 리뷰를 위한 비밀 무기

저를 괴롭히던 시나리오가 있었습니다: 풀 리퀘스트 작업을 시작하고, GitHub 댓글을 20분 동안 읽고, 구현을 논의하러 Claude로 전환하고, 어떤 댓글을 처리했는지 잊어버리고, 다시 GitHub으로 돌아가는 악순환입니다.

/pr_comments는 코드 한 줄 쓰기 전에 GitHub 풀 리퀘스트 댓글을 전부 Claude 세션으로 불러옵니다. 완전한 컨텍스트, 탭 전환 없음, 리뷰어가 지적한 내용 완전 파악.

이전에는 PR당 30~40분을 컨텍스트 관리에만 썼습니다. 지금은? 댓글을 한 번 로드하고 작업이 끝날 때까지 플로우 상태를 유지합니다.

파트 2: 매일 쓰는 명령어 — 치팅처럼 느껴지는 것들

모든 것을 탈선시키는 질문

복잡한 기능을 개발하는 중입니다. Claude가 솔루션 설계를 도와주고 있습니다. 그런데 갑자기:

잠깐, JWT와 세션 쿠키의 차이가 뭐였지?

어떻게 하시겠습니까?

  • A) 구글링하러 흐름을 깨기
  • B) Claude에게 물어보고 대화의 맥락을 잃기
  • C) 메모해두고 나중에 잊어버리기

저는 몇 달 동안 C를 선택했습니다. /btw를 알기 전까지는요.

이 명령어는 순수한 천재입니다. /btw 리프레시 토큰 처리하는 가장 좋은 방법이 뭐야?를 입력하면 Claude가 백그라운드에서 메인 작업을 계속하면서 곁다리 질문에 답해줍니다. 컨텍스트 전환 없음. 진행 상황 손실 없음. 뇌의 병렬 처리.

제가 잘못 쓰고 있었다는 것을 깨달은 순간: /btw를 3주 동안 쓰다가 진정한 힘을 이해했습니다. 이것은 단순히 질문을 위한 것이 아닙니다. 인지 과부하 없이 여러 생각의 흐름을 유지하는 것입니다.

세션이 숨을 헐떡일 때

다들 아는 상황입니다. Claude 세션이 메시지 200개 깊이에 빠져 있습니다. 응답이 느려지고 있습니다. 컨텍스트 한계 경고가 뜨고 있습니다. 패닉이 옵니다:

처음부터 다시 시작해야 하나? 이 모든 진행 상황을 잃게 되는 건가?

멈추세요. 숨 쉬세요. /compact를 입력하세요.

이 명령어는 정말 중요한 것만 남기고 불필요한 것들을 제거해 대화를 정리합니다. Claude가 중요한 모든 것을 요약하고, 핵심 결정과 코드를 보존하며, 컨텍스트 창을 확보합니다. 진행 상황은 유지됩니다. Claude는 새로운 컨텍스트를 얻습니다. 아무도 울지 않습니다.

계산해보면: 처음부터 다시 시작 = 45분 손실. /compact = 10초.

지난 3개월 동안 이 명령어를 127번 사용했습니다. 절약된 시간을 계산해보세요.

쉘 명령어 혁명

이전 워크플로우는 이랬습니다:

  1. Claude에게 디버깅 도움 요청
  2. Claude가 제안: "수정된 파일을 확인하려면 git status를 실행하세요"
  3. 터미널로 전환
  4. 명령어 입력
  5. 출력 복사
  6. Claude로 다시 전환
  7. 출력 붙여넣기
  8. 대화 계속

일곱 단계. 매번. 항상.

그리고 ! 명령어를 발견했습니다.

Claude에서 직접 ! git status를 입력하면 출력이 바로 대화 안에 들어옵니다. 탭 전환 없음. 복사 붙여넣기 없음. 컨텍스트 손실 없음.

가장 좋아하는 사용 사례:

! npm install
! npm run test
! git add .

채팅을 떠나지 않고 전체 워크플로우를 실행할 수 있습니다. 치팅 같은 느낌이 드는 건 실제로 그렇기 때문입니다.

모르는 것의 불안

토큰 한계는 물리기 전까지는 보이지 않습니다. 세션을 이어가다 보면 Claude가 대화의 앞부분을 잊기 시작합니다. 중요한 컨텍스트를 잃게 됩니다. 다시 시작해야 합니다.

/cost는 정확히 어디에 서 있는지 보여줍니다:

  • 총 비용: $0.08
  • 사용된 토큰: 42k
  • 남은 컨텍스트: 60%

세션의 연료 게이지 같습니다. 길 가에서 멈추기 전에 언제 /compact를 써야 할지 알게 됩니다.

파트 3: 레벨업 준비가 되었을 때의 파워 무브

속도 vs. 품질: 모델 전환 게임

대부분의 사용자가 모르는 비밀이 있습니다. 세션당 하나의 모델에 고정될 필요가 없습니다.

/model을 만나보세요. 대화 중에 Claude의 다른 두뇌로 전환할 수 있는 명령어입니다:

  • Opus: 깊은 추론이 필요한 어려운 문제에.
  • Sonnet: 균형 잡힌 속도와 품질을 위해.
  • Haiku: 즉각적인 응답이 필요한 빠른 작업에.

실제 시나리오: 복잡한 데이터베이스 마이그레이션을 설계하고 있습니다. 깊은 생각을 위해 Opus로 시작합니다. 계획이 세워지면 빠른 코드 생성을 위해 Haiku로 전환합니다. 그런 다음 코드 리뷰를 위해 Sonnet으로 돌아갑니다. 모두 한 세션에서. 재시작 없음. 컨텍스트 손실 없음.

시간 절약? 복잡한 프로젝트에서 3~4배 빠름.

시가 아닌 속도가 필요할 때

아이디어를 테스트하거나, 프로토타이핑하거나, 솔루션을 탐색할 때처럼 빠르게 반복하고 싶을 때가 있습니다. 가장 신중하고 깊이 있는 응답이 필요한 것이 아닙니다. 빠른 것이 필요합니다.

/fast는 같은 모델을 사용하되 깊이보다는 속도에 최적화된 더 빠른 응답을 토글합니다. 다음과 같은 경우에 완벽합니다:

  • 브레인스토밍 세션.
  • 빠른 리팩토링.
  • 반복적 디버깅.
  • "일단 되게 만들어" 순간들.

초기 개발 중에는 켜두고, 신중한 검토가 필요할 때 끕니다. 두 세계의 최고.

실제로 버그를 잡는 코드 리뷰

저는 코드 리뷰를 체크박스처럼 다루곤 했습니다. "응, Claude가 봤어." 하지만 실제 리뷰였을까요, 아니면 그냥 도장 찍기였을까요?

/review가 그것을 바꿨습니다. 이것은 가벼운 피드백이 아닙니다. 다음 항목들을 체계적으로 검토하는 코드 리뷰 패스입니다:

  • 버그와 엣지 케이스.
  • 논리 오류.
  • 코드 명확성.
  • 성능 문제.
  • 보안 우려 사항.

/review의 실제 출력:

Line 42: potential null reference — handle missing key
Line 87: inefficient loop — consider using map instead
Line 103: security risk — sanitize user input

새벽 2시에 풀 리퀘스트를 읽어주는 시니어 엔지니어가 있는 것 같습니다. 판단 없이.

파트 4: 위기 상황용 명령어 — 일이 잘못될 때

나선형 추락

이 느낌을 아실 겁니다. Claude가 원을 그리고 있습니다. 같은 조언을 반복합니다. 듣지를 않습니다. 대화가 탈선했고 출구를 찾을 수 없습니다.

본능: 탭을 닫는다. 처음부터 시작한다. 진행 상황을 모두 잃는다.

하지 마세요.

/clear는 대화를 지우되 셋업을 유지합니다. 도구, 권한, 프로젝트 컨텍스트 모두 보존됩니다. 소프트 리셋이지, 핵 옵션이 아닙니다.

차이점: 처음부터 다시 시작 = 20분 재설정. /clear = 몇 초 만에 새 출발.

아무것도 작동하지 않을 때

뭔가 고장난 것 같습니다. 명령어가 작동하지 않습니다. 응답이 이상합니다. 설치, API 키, 권한 문제인지, 아니면 그냥 우주선의 영향인지 모르겠습니다.

모든 것을 재설치하기 전에 /doctor를 실행하세요.

다음을 확인합니다:

  • ✓ API 키 유효
  • ✓ Node 18+
  • ✓ 권한 OK
  • ✓ 설정 파일 존재
  • ✓ 터미널 통합

10초 만에 해결할 수 있는 IT 지원 티켓 같습니다.

말하지 않는 터미널

Claude가 터미널 출력을 볼 수 없습니다. 2010년처럼 복사 붙여넣기를 하고 있습니다. 더 나은 방법이 있어야 합니다.

있습니다. /terminal-setup은 Claude가 실제로 터미널에서 일어나는 일을 볼 수 있도록 쉘 통합을 설정합니다. 명령어 하나로, 갑자기 미래에 살게 됩니다.

길을 잃었다면? 도움을 구하세요

무엇이 가능한지 모르겠나요? 어떤 명령어가 무엇을 하는지 기억나지 않나요?

/help는 간단한 설명과 함께 모든 명령어를 나열합니다. 길을 잃었을 때의 출발점, 막혔을 때의 참고 자료, 이 글의 다른 모든 것을 잊었을 때의 안전망입니다.

프로 팁: /help를 즐겨찾기에 저장해두었습니다. 팀의 새 멤버들에게 이 명령어들을 가르칠 때, 깊이 들어가기 전에 먼저 /help를 실행해 전체 영역을 보게 합니다.

완전한 명령어 참조 목록

여기에 치트 시트가 있습니다. 스크린샷을 찍어두세요.

한 번 셋업:

  • /init: 프로젝트 문서 자동 생성.
  • /memory: 글로벌 선호도 영구 설정.
  • /pr_comments: GitHub PR 댓글을 컨텍스트로 로드.

매일 사용:

  • /btw: 흐름을 깨지 않고 곁다리 질문하기.
  • /compact: 대화 압축, 계속 진행.
  • ! 명령어: 채팅을 떠나지 않고 쉘 실행.
  • /cost: 토큰 사용량 확인.

파워 무브:

  • /fast: 더 빠른 응답 토글.
  • /review: 체계적인 코드 리뷰.
  • /model: 세션 중 모델 전환.

위기 상황:

7일 마스터리 챌린지

이 글을 북마크하고 잊어버리지 마세요. 실행 계획이 있습니다:

  • 1일차: 현재 프로젝트에서 /init 실행. 마법을 느끼세요.
  • 2일차: 상위 5가지 코딩 선호도로 /memory 설정.
  • 3일차: 모든 곁다리 질문에 /btw 사용. 플로우 상태에 주목하세요.
  • 4일차: 컨텍스트 한계에 도달했나요? 패닉 대신 /compact.
  • 5일차: 쉘 명령어에 ! 시도. 다시는 탭을 전환하지 마세요.
  • 6일차: 다음 PR에 /review 실행. 출시 전 버그를 잡으세요.
  • 7일차: /model 전환을 실험해보세요. 자신만의 리듬을 찾으세요.

8일차 이후: 이제 당신은 명령어 하나만 아는 사람이 아닙니다. 모두가 조언을 구하는 파워 유저입니다.
