이 5가지 데이터 엔지니어링 용어를 이해한다면, 당신은 업계 상위 10%다

B V Sarath Chandra · Towards Data Engineering · 6 min read · 원문 · freedium

멱등성, 네트워크 셔플, 1:N 팬아웃, 메달리온 아키텍처, 시맨틱 레이어—이 5가지 핵심 개념을 진정으로 이해하면 데이터 엔지니어링 업계에서 상위 10%에 속할 수 있다.

대부분의 데이터 엔지니어들은 Airflow, Snowflake, dbt 같은 도구 이름은 줄줄 외우지만, 파이프라인이 실패했을 때 왜 실패했는지 설명하지 못한다. 이 글은 단순한 정의 암기가 아닌 실제 이해가 필요한 5가지 아키텍처 개념을 설명한다. 멱등성은 파이프라인이 몇 번 재실행되어도 결과가 동일하게 유지되도록 보장하고, 네트워크 셔플은 분산 컴퓨팅에서 가장 비싼 연산이며, 1:N 팬아웃은 클라우드 예산을 폭발시키는 조용한 킬러다. 메달리온 아키텍처는 데이터 레이크하우스에 질서를 부여하고, 시맨틱 레이어는 AI가 데이터를 올바르게 해석할 수 있게 만드는 핵심 인프라다.

솔직하게 말하겠다.

오늘날 데이터 엔지니어링에 대해 이야기하는 대부분의 사람들은 SaaS 도구 목록을 읽는 것처럼 들린다 — Airflow, Snowflake, dbt, Kafka — 아니면 PySpark 작업이 클러스터를 다운시킨 이유를 실제로 물어보면 완전히 모른다는 게 드러난다.

어느 쪽이 될 필요가 없다.

5가지 용어, 5가지 핵심 아키텍처 개념이 있다. 이것들을 단순히 정의만 외우는 것이 아니라 진정으로 이해한다면, 거의 모든 방 안에서 다른 사람들보다 훨씬 앞서게 될 것이다. 주니어 개발자든, 망가진 파이프라인에 지친 데이터 과학자든, 클라우드 비용이 왜 이렇게 높은지 이해하려는 리더든 상관없이.

시작하겠다.

1. 멱등성 (Idempotency)

가장 먼저 뇌리에 새겨야 할 것은 데이터 파이프라인은 실패한다는 것이다. 끊임없이 실패한다. API가 다운되고, 클러스터가 메모리 부족에 걸리거나, 소스 데이터가 늦게 도착한다.

그렇다면 파이프라인이 중간에 실패해서 재실행해야 할 때 무슨 일이 벌어질까?

이것이 바로 멱등성이 등장하는 지점이다.

멱등적 파이프라인이란 한 번이든, 두 번이든, 쉰 번이든 연속으로 실행할 수 있으며, 데이터베이스의 최종 상태가 단 한 번만 실행한 것과 정확히 동일하게 유지되는 파이프라인이다.

파이프라인이 단순히 INSERT 명령을 사용한다면, 두 번 실행하면 데이터가 모두 중복된다. 대시보드가 이제 수익을 두 배로 보여주고, CEO가 패닉 상태에 빠진다. 파이프라인이 멱등적이라면 MERGE (업서트) 같은 명령을 사용하거나 특정 파티션을 안전하게 삭제하고 교체한다. 데이터베이스를 보면서 이렇게 말하는 것이다: "이 행이 이미 존재하면 업데이트하라. 존재하지 않으면 삽입하라."

이것이 왜 중요한가?

주니어 엔지니어들은 모든 것이 잘 될 것이라고 가정하고 파이프라인을 만든다. 시니어 엔지니어들은 모든 것이 잘못될 것을 알고 멱등적 파이프라인을 만든다. 멱등성을 이해하면 Airflow DAG의 "재시도" 버튼을 눌러서 프로덕션 데이터베이스를 손상시켰을지 걱정하며 새벽 3시에 식은땀을 흘리며 깨어날 필요가 없어진다.

2. 네트워크 셔플 (The Network Shuffle)

상업 주방을 상상해 보라. 10명의 요리사(Executor)가 각자의 자리에서 채소를 썰고 있다. 엄청나게 빠르다.

그런데 갑자기 요리사들에게 썬 양파 전부를 특정 테이블 하나에, 당근 전부를 다른 테이블에, 셀러리 전부를 세 번째 테이블에 모아놓으라고 지시한다. 혼돈이 시작된다. 요리사들이 주방을 가로질러 뛰어다니고, 서로 부딪히고, 채소 그릇을 들고 다니고, 줄을 서서 기다린다. 아무도 실제로 채소를 썰지 않는다; 그냥 음식을 이리저리 옮기고 있을 뿐이다.

이것이 바로 셔플이다.

분산 컴퓨팅(Spark나 Databricks 같은)에서 데이터는 여러 워커 노드에 분산되어 있다. .filter() 같은 명령을 실행하면 노드들이 데이터가 있는 자리에서 바로 처리한다. 번개처럼 빠르다. 하지만 .groupBy()나 .join() 같은 명령을 실행하면 엔진이 매칭되는 키들을 함께 묶기 위해 방대한 양의 데이터를 네트워크를 통해 물리적으로 이동시켜야 한다.

이것이 왜 중요한가?

셔플은 데이터 엔지니어링에서 가장 비싸고 시간이 많이 걸리는 연산이다. 파이프라인이 4분이 아닌 4시간이 걸리는 이유다.

셔플을 이해하면 느린 코드를 고치기 위해 클러스터에 더 많은 메모리를 추가하는 방식으로 무작정 접근하지 않게 된다. 대신 로직을 바꾼다. 조인 전에 데이터를 필터링한다. 더 작은 테이블은 브로드캐스트한다. 하드웨어가 아닌 데이터의 물리적 구조를 최적화한다.

3. 1:N 팬아웃 트랩 (The 1:N Fan-Out Trap)

이것은 개인적으로 가장 좋아하는 설명 주제다. 클라우드 컴퓨팅 예산을 조용히 잡아먹는 킬러이기 때문이다.

SQL로 두 테이블을 조인할 때 키를 기반으로 행을 매칭한다.

테이블 A가 1,000개의 행을 갖고 있고 테이블 B에 LEFT JOIN을 한다면, 결과도 여전히 1,000개의 행일 것이라고 기대할 것이다, 맞지 않나?

테이블 B에 중복이 없다면 그럴 것이다.

users 테이블을 purchases 테이블에 조인하는데, 한 사용자가 5건의 구매를 했다면, 그 단일 사용자 행이 출력에서 5번 복제된다. 이것이 1대다(1:N) 관계다.

이제 이 작업을 실수로 1억 개의 행이 있는 테이블에서, 수십억 번의 클릭이 있는 로그 테이블과 조인한다고 상상해 보라.

이것이 왜 중요한가?

카테시안 폭발을 일으킨 것이다. 1억 개짜리 테이블이 순식간에 500억 개짜리 괴물로 "팬아웃"된다. Spark 클러스터가 RAM을 소진하고 디스크로 스필되다가 결국 다운되어, 수백 달러의 클라우드 컴퓨팅 비용을 불태운다.

팬아웃 트랩을 이해하면 연결하는 테이블의 세분성(고유성)을 엄격하게 확인하지 않고는 절대 JOIN을 작성하지 않게 된다.

4. 메달리온 아키텍처 (Medallion Architecture)

이것은 모두가 사용하는 용어이지만, 제대로 실행하는 사람은 거의 없다.

앱에서 나온 원시 JSON 파일을 비즈니스가 읽을 대시보드에 직접 던져놓으면 힘든 일이 생긴다. 데이터가 지저분하고, 중첩되어 있으며, 오류로 가득 차 있다.

메달리온 아키텍처는 데이터 레이크하우스에 질서를 가져오는 방법이다. 데이터를 세 가지 품질 수준으로 정리하는 간단한 개념이다:

  • 브론즈: 도착한 그대로의 원시, 비필터링 데이터. 소스 시스템이 망가지더라도 항상 원본 브론즈 데이터를 재생할 수 있다.
  • 실버: 정제되고, 필터링되고, 타입이 지정된 데이터. 중복 제거 및 표준화 완료.
  • 골드: 비즈니스 수준의 집계. 고도로 정제된 스타 스키마로, 데이터가 사전 조인되어 임원들이 쿼리할 준비가 된 상태.

이것이 왜 중요한가?

비즈니스 이해관계자가 지표가 잘못되었다고 불평하면 메달리온 아키텍처가 디버깅 방법을 정확히 알려준다. 500줄짜리 SQL 쿼리를 풀어헤칠 필요가 없다. 골드 테이블을 확인한다. 거기서 로직이 잘못되었으면 고친다. 골드 테이블이 괜찮으면 실버 테이블에서 누락된 데이터를 확인한다. 혼돈스러운 데이터 늪을 관리 가능하고 감사된 공급망으로 바꾸어 놓는다.

5. 시맨틱 레이어 (The Semantic Layer)

이것은 다섯 가지 중 가장 오해받는 개념이며, 특히 인공지능이 등장한 지금은 더욱 그렇다.

지금 모든 CEO들이 원하는 것은 "Text-to-SQL" 챗봇이다: "지난달 수익이 얼마였지?" 라고 타이핑하면 즉각적인 답변이 나오는 것.

LLM을 원시 데이터베이스에 직접 연결하면 환각이 발생한다. LLM은 "수익"이 gross_sales 컬럼인지 net_arr 컬럼인지 알지 못한다. 환불된 거래를 필터링해야 한다는 것도 알지 못한다.

이것이 바로 시맨틱 레이어가 등장하는 지점이다.

시맨틱 레이어(dbt 같은 도구 사용)는 비즈니스 지표를 코드로 정의하는 곳이다. 이렇게 엄격한 정의를 작성한다: "수익 = gross_sales — refunds, 상태 = 'active'인 경우."

이것이 왜 중요한가?

시맨틱 레이어가 있으면 AI에게 SQL 조인을 추측하게 하지 않아도 된다. 그냥 AI가 시맨틱 레이어를 쿼리하게 하면 된다. AI는 "수익" 지표만 요청하면 되고, 시맨틱 레이어가 그것을 완벽하고 인간이 승인한 SQL 코드로 변환한다.

실제로 기업에서 작동하는 AI 에이전트를 만들고 싶다면, 더 나은 프롬프트 엔지니어링이 필요한 것이 아니다. 시맨틱 레이어가 필요하다.

상위 10%에 오신 것을 환영한다

파이썬 스크립트 작성법만 아는 사람과 분산 시스템이 실제로 어떻게 작동하는지 이해하는 사람 사이의 격차는 어마어마하다.

ProductHunt에서 출시되는 모든 새로운 도구를 외울 필요는 없다. 하지만 멱등성을 이해하면 파이프라인이 밤새 살아남는다. 셔플과 팬아웃 트랩을 이해하면 회사의 클라우드 컴퓨팅 비용 수천 달러를 절약한다. 메달리온 아키텍처를 이해하면 데이터가 실제로 관리된다. 그리고 시맨틱 레이어를 이해하면 내일의 AI 인프라를 구축할 준비가 된 것이다.

이것이 전부다. 다섯 가지 개념. 진짜 엔지니어링.

상위 10%에 오신 것을 환영한다.
