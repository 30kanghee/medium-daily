Data Engineer Interview Question: How to Design a Scalable Data Lake on S3 (Step-by-Step Guide)

Sriw World of Coding · Towards Data Engineering · 7 min read · 원문 · freedium

요약

Amazon S3 기반 데이터 레이크를 제대로 설계하려면 레이어 분리, 파티셔닝 전략, 파일 포맷 선택, 메타데이터 카탈로그, 보안·거버넌스, 비용 제어까지 단계별로 체계적으로 접근해야 한다.

데이터가 폭발적으로 늘어나는 환경에서 계획 없이 S3에 쌓다 보면 금세 "데이터 늪"이 된다. 이 글은 데이터 엔지니어 면접에서 자주 나오는 "하루 10TB 규모의 S3 데이터 레이크를 어떻게 설계하겠는가"라는 질문에 대한 단계별 답변을 제공한다. 핵심은 Raw·Processed·Curated라는 세 레이어를 별도 버킷으로 분리하고, 날짜·앱·국가 등 자주 필터링하는 차원으로 파티션을 나누며, 분석 레이어에는 Parquet 포맷을 쓰는 것이다. 여기에 AWS Glue 데이터 카탈로그로 테이블을 등록하고, IAM·Lake Formation으로 최소 권한 접근 제어를 구현하고, 수명주기 정책으로 비용을 통제하면 기가바이트에서 페타바이트까지 확장되는 안정적인 데이터 레이크가 완성된다.

문제의 시작: 데이터 늪을 피하는 법

데이터가 매달 폭발적으로 늘어나는 회사를 상상해보자. 로그, 클릭스트림, 사용자 이벤트, 트랜잭션 등 온갖 데이터가 쏟아진다. 계획 없이 쌓다 보면 믿을 수 있는 데이터 레이크가 아니라 혼돈스러운 "데이터 늪"이 된다.

S3 데이터 레이크는 대부분 단순하게 시작된다.

  • 버킷 하나.
  • 평탄한 폴더 구조.
  • JSON/CSV를 마구잡이로 쌓는다.

그러고 몇 달 지나면 모두가 묻기 시작한다.

  • "지난주 데이터는 어디 있죠?"
  • "왜 Spark 작업이 이렇게 느리죠?"
  • "여기 보안은 어떻게 관리하죠?"

이런 상황을 겪어봤다면 혼자가 아니다. S3에 확장 가능한 데이터 레이크를 설계하는 것은 S3 자체보다 구조, 거버넌스, 패턴에 관한 문제다. 면접에서 채용 담당자는 "S3가 무엇인가요?"라고 묻지 않는다. 이렇게 묻는다.

"하루 10TB의 데이터를 처리하는 회사를 위해 확장 가능한 S3 데이터 레이크를 어떻게 설계하겠습니까?"

바로 이것을 지금 함께 풀어보겠다.

핵심 구성 요소

1. 올바른 스토리지 기반 선택

Amazon S3가 거의 모든 클라우드 데이터 레이크의 뼈대다. 핵심 특성은 세 가지다.

  • 사실상 무한한 확장성 (GB에서 PB까지).
  • 높은 내구성 (99.999999999% — 11나인).
  • 모든 데이터 타입 지원: JSON, CSV, Parquet, Avro, 이미지, 로그 등.

S3는 무엇이든 저장할 수 있는 거대한 창고다. 나중에 어떻게 구성하고 최적화할지 결정하면 된다.

2. 데이터를 레이어(데이터 존)로 구성

흔히 쓰는 패턴은 데이터를 레이어로 나누는 것이다(흔히 "존(zone)"이라고도 한다).

Raw / 랜딩 존

  • 원천 그대로 수집한다.
  • 불변, 추가 전용이다.
  • 감사, 재처리, 디버깅에 활용한다.

Cleaned / 처리 존

  • 구조화되고 표준화된다.
  • 불량 레코드를 걸러낸다.
  • 스키마가 강제된다.

Curated / 분석 존

  • 쿼리에 최적화된다 (Parquet/ORC).
  • BI와 ML을 위한 스타 스키마 형태의 레이아웃.

레이어별로 S3 버킷을 분리하는 것이 보안, 거버넌스, 수명주기 관리를 위한 모범 사례다.

3. 올바른 폴더 구조와 파티셔닝 패턴

잘못된 폴더 구조는 쿼리 성능을 망친다.

좋은 패턴(사용자 이벤트 예시):

s3://my-data-lake/raw/events/date=2026-04-17/app=mobile/
s3://my-data-lake/curated/events/date=2026-04-17/country=IN/

핵심 포인트:

  • 자주 필터링하는 고 카디널리티 차원으로 파티션을 나눈다: date, app, country, region 등.
  • 일별 파티션 수를 적절히 유지한다. 파티션이 너무 많으면 메타데이터 부하가 커진다.
  • user_id 같은 매우 높은 카디널리티의 ID로 직접 파티셔닝하는 것은 피해야 한다. 그룹화하거나 버킷팅을 활용하자.

이 패턴이 있어야 Athena, Spark, Presto, Redshift Spectrum 같은 도구들이 관련 파티션만 스캔할 수 있다.

4. 올바른 파일 포맷 선택

데이터를 어떻게 저장하느냐는 쿼리 속도, 스토리지 비용, 압축에 큰 영향을 미친다.

분석/큐레이션 존에서는 Parquet이 거의 모든 현대 S3 데이터 레이크의 기본값이다.

5. 메타데이터 레이어(데이터 카탈로그) 추가

파일을 저장하는 것은 절반에 불과하다. 다음을 알고 있는 데이터 카탈로그가 필요하다.

  • 어떤 테이블이 있는지
  • 각 테이블의 스키마
  • 데이터가 어디에 있는지

주요 도구:

  • AWS Glue 데이터 카탈로그 (서버리스, Athena, EMR, Redshift Spectrum과 통합).
  • Apache Hive Metastore (EMR, 오픈소스).

S3 위치를 크롤링해 파티션이 있는 테이블로 등록한다. 이후 BI 도구와 SQL 엔진이 일반 테이블처럼 쿼리할 수 있다.

6. 보안과 거버넌스

확장 가능한 데이터 레이크도 보안이 없으면 무용지물이다. 핵심 패턴은 다음과 같다.

  • 수집, ETL, 쿼리 등 각 프로세스에 최소 권한 IAM 역할을 부여한다.
  • 버킷 수준 및 객체 수준 암호화를 적용한다 (SSE-S3, KMS, 클라이언트 사이드).

접근 제어는 다음으로 구현한다.

  • 버킷 정책
  • IAM 정책
  • AWS Lake Formation (행·열 수준 보안용)

별도 AWS 계정을 활용할 수도 있다.

  • 데이터 레이크 코어
  • 수집 / ETL
  • 분석 / BI

이렇게 하면 폭발 반경을 줄이고 거버넌스가 쉬워진다.

실전 예시: 이커머스 앱 데이터 레이크 단계별 설계

사용자 이벤트를 매일 로그로 남기는 이커머스 앱을 위한 확장 가능한 S3 데이터 레이크를 설계해보자.

1단계: 데이터 레이어 정의

S3에 세 개의 버킷을 생성한다 (운영 전용 리전, 예: ap-south-1).

  • my-company-data-lake-raw
  • my-company-data-lake-processed
  • my-company-data-lake-curated

2단계: 파티셔닝 전략 정의

주요 이벤트 타입이 user_click이고 필드는 다음과 같다고 가정한다.

  • event_id, user_id, product_id, page, country, timestamp

파티션을 이렇게 고른다.

  • 최상위: date=YYYY-MM-DD
  • 선택: app=web/mobile, country=XX

Raw 데이터 경로 예시:

s3://my-company-data-lake-raw/events/user_click/date=2026-04-17/app=web/

큐레이션된 Parquet 레이아웃:

s3://my-company-data-lake-curated/events/user_click/date=2026-04-17/country=IN/

3단계: Raw 데이터 수집

이벤트당 Raw JSON 파일 샘플:

{
  "event_id": "e123",
  "user_id": "u456",
  "product_id": "p789",
  "page": "product_detail",
  "country": "IN",
  "timestamp": "2026-04-17T15:00:00Z"
}

수집 경로:

  • Kafka → Kinesis → Lambda → S3
  • Kafka → EMR / Spark → S3
  • 직접 API → S3 PUT

핵심:

  • 추가 전용 쓰기.
  • 버킷에 버전 관리를 활성화한다.

4단계: Cleaned 레이어로 변환

Spark (EMR / Glue)로 raw에서 읽어, 정제·검증 후 processed에 쓴다.

Scala/PySpark 예시 (의사코드):

val df = spark.read
  .option("header", "true")
  .json("s3a://my-company-data-lake-raw/events/user_click/date=2026-04-17/*")

val cleaned = df
  .na.drop()                    // null user_id, product_id 제거
  .filter($"country".isNotNull) // 잘못된 country 필터링

cleaned.write
  .partitionBy("app", "country")
  .mode("overwrite")
  .format("parquet")
  .save("s3a://my-company-data-lake-processed/events/user_click/date=2026-04-17/")

이렇게 하면 processed에 분석을 위한 파티션된 Parquet 파일이 생성된다.

5단계: 분석 준비 테이블 생성

처리된 데이터를 Glue 데이터 카탈로그에 등록한다.

  • 데이터베이스 ecom_events를 생성한다.
  • user_click 테이블을 다음 경로를 가리키도록 생성한다: s3://my-company-data-lake-processed/events/user_click/
  • 파티션 프로젝션 또는 파티션 크롤러로 새 date 파티션이 자동으로 발견되게 한다.

이제 Athena에서 쿼리할 수 있다.

SELECT country, COUNT(*) AS clicks
FROM ecom_events.user_click
WHERE date = '2026-04-17'
GROUP BY country;

6단계: 수명주기와 비용 제어 추가

  • Raw 레이어: 365일 보관 후 S3 Glacier Deep Archive로 이전한다.
  • Processed / Curated: 90일 보관 후 아카이브한다.
  • 감사를 위해 S3 버전 관리와 버킷 로깅을 켠다.

S3 수명주기 정책으로 스토리지 클래스 간 객체 전환을 자동화한다.

흔한 실수와 오해

실수 1: 모든 것을 하나의 버킷에

실수: s3://my-data-lake 하나에 raw, processed, analytics를 전부 넣는다.

왜 나쁜가: 관심사 분리가 없다. 레이어별로 다른 정책, 암호화, 수명주기 규칙을 적용하기 어렵다.

해결: 최소한 세 개의 버킷을 사용한다 (raw, processed, curated).

실수 2: 파티셔닝 전략 없음

실수: 날짜별 평탄한 폴더, 파티션 없음.

s3://my-data-lake/raw/events/2026-04-17.json

왜 나쁜가: 모든 쿼리가 하루치 데이터 전체를 스캔한다 (파티션 제거 불가). 데이터가 늘수록 성능이 떨어진다.

해결: date, app, country 등으로 파티셔닝하고 폴더 깊이를 적절히 유지한다.

실수 3: 분석에 CSV 사용

실수: Raw 존에서 CSV는 괜찮지만, 분석 존에서도 CSV를 쓰는 것.

왜 나쁜가: 모든 쿼리에서 파일 전체를 스캔한다. 압축률이 낮고 컬럼 제거도 안 된다.

해결: Processed/Curated 레이어에서 Parquet (또는 Avro)으로 변환한다.

실수 4: 보안과 거버넌스 무시

실수: 모두에게 s3:* 권한, 암호화 없음, 데이터 카탈로그 없음.

왜 나쁜가: 데이터 손실이나 유출 위험. 소유권이나 검색 가능성이 없다.

해결: IAM 역할 + 버킷 정책 + Lake Formation을 사용한다. SSE-S3 또는 KMS 암호화를 적용한다.

실전 팁과 모범 사례

  1. 두 레이어 패턴으로 시작하라
    • Raw + Curated (단순하고 관리하기 쉽다).
    • 복잡한 변환이 필요해질 때 "processed" 레이어를 추가한다.
  2. 일관된 명명 규칙을 사용하라
    • bucket_name = <company>_<purpose>_<layer>
    • s3://mycompany_datalake_raw 등.
  3. 수집과 카탈로그 등록을 자동화하라
    • EventBridge + Lambda 또는 Glue Jobs + Crawler로 새 데이터를 자동 등록한다.
  4. 파티션을 현명하게 나눠라
    • 파티션이 너무 많으면 메타데이터 지옥이 된다.
    • 너무 적으면 전체 스캔이 된다.
    • 쿼리 파티션당 파일 100~10,000개를 목표로 한다.
  5. 비용을 모니터링하고 최적화하라
    • S3 Analytics + Cost Explorer로 버킷별 스토리지를 추적한다.
    • 접근 패턴이 자주 바뀐다면 S3 Intelligent-Tiering을 켠다.
  6. 스키마 진화를 고려해 설계하라
    • Raw에서는 스키마-온-리드를 쓰되, Curated에서는 스키마-온-라이트를 쓴다.
    • 장기적 확장성을 위해 Parquet + 스키마 진화를 활용한다.

정리

  • S3 위에 확장 가능한 데이터 레이크는 하나의 거대한 버킷이 아니라 레이어(raw, processed, curated)로 구성된다.
  • 올바른 파티셔닝 (date, app, country)과 분석 준비 데이터를 위한 Parquet을 사용한다.
  • 도구들이 S3 위에서 구조화된 테이블을 쿼리할 수 있도록 데이터 카탈로그(Glue, Hive)를 추가한다.
  • 보안(IAM, 버킷 정책, 암호화)과 거버넌스(계정 분리, 수명주기 정책)를 강제한다.
  • 단순하게 시작하고, 자동화하고, 데이터 볼륨과 사용 사례가 성장함에 따라 반복 개선한다.
