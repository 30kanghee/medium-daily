Enterprise Data Architecture: From Data Lakes to Data Products

Shraddha Zope-Ladhe · 퍼블리케이션 없음 · 8 min read · 원문 · freedium

요약

데이터 레이크에 데이터를 쌓는 것과 그 데이터로 실제 가치를 창출하는 것은 전혀 다른 역량이며, 기업 데이터 아키텍처(EDA)는 이 간극을 메우기 위한 조직적 규율이다.

지난 10년간 기업들은 페타바이트 규모의 데이터 레이크를 구축하는 데 막대한 투자를 쏟아부었지만, 정작 "우리 데이터가 경쟁사보다 빠른 의사결정을 가능하게 하는가?"라는 질문에는 여전히 답하지 못하고 있다. 이 글은 데이터 웨어하우스에서 데이터 레이크로, 다시 데이터 레이크하우스와 데이터 메시를 거쳐 데이터 프로덕트 시대로 이어지는 아키텍처 진화를 추적한다. 저자는 금융 서비스 분야의 20년 이상 경험을 바탕으로, 데이터를 조직의 부채가 아닌 전략 자산으로 전환하는 데 필요한 아키텍처 원칙과 거버넌스 체계를 구체적으로 설명한다. 데이터 프로덕트 사고방식의 핵심은 데이터를 단순히 저장하는 것을 넘어, 명확한 소유권·품질 기준·SLA를 갖춘 제품으로 취급하는 것이다.

서론: 데이터 볼륨과 데이터 가치 사이의 간극

지난 10년간 함께 일해 온 조직들은 하나같이 같은 역설에 빠져 있었다. 데이터는 넘쳐나는데 인사이트는 메말라 있다는 것이다.

금융 서비스 분야에서 이 문제는 특히 심각하다. 중간 규모의 은행 하나만 해도 하루에 수십억 건의 이벤트를 생성한다. 카드 거래, UPI 결제, 대출 신청, 사기 신호, 시장 피드, 고객 인터랙션, Open Banking API 호출까지. 이 모든 것을 수집할 인프라는 이미 갖춰져 있다. 하지만 그것을 실제로 유용하게 만드는 규율은 대부분의 조직에서 아직 자리를 잡지 못했다.

지난 10년은 데이터 레이크를 쌓는 데 바쳤다. 페타바이트 단위의 스토리지, 정교한 수집 파이프라인, 컨퍼런스실 벽을 가득 채운 인상적인 아키텍처 다이어그램. 그럼에도 핵심 질문, 즉 "우리 데이터가 경쟁사보다 더 빠른 의사결정을 가능하게 하는가?"는 대부분의 조직에서 여전히 답을 찾지 못하고 있다.

데이터를 저장하는 역량과 데이터로 가치를 창출하는 역량은 완전히 다른 이야기다. 이 둘을 혼동하는 데는 엄청난 비용이 따른다.

기업 데이터 아키텍처(EDA)는 바로 이 문제를 해결하기 위해 존재한다. 단순한 기술적 연습이 아니라 조직 차원의 규율로서 말이다.

아키텍처의 진화: 데이터 웨어하우스에서 데이터 프로덕트까지

1세대: 데이터 웨어하우스의 시대

1990년대부터 2000년대까지 기업들은 데이터 웨어하우스를 중심으로 움직였다. 정형 데이터를 미리 정의된 스키마에 따라 적재하고, SQL로 분석하는 방식이다. 빠른 쿼리 성능과 일관된 데이터 품질이 강점이었지만, 스키마 변경에 유연하지 못하고 비정형 데이터를 처리하기 어렵다는 한계가 뚜렷했다. 대규모 확장에도 비용이 급격히 올라갔다.

2세대: 데이터 레이크의 부상

2010년대에 들어서면서 Hadoop과 클라우드 오브젝트 스토리지가 등장하며 데이터 레이크 패러다임이 자리를 잡았다. "일단 모든 것을 저장하고, 나중에 필요할 때 처리하자"는 접근법이다. 정형·반정형·비정형 데이터를 원시 형태로 저렴하게 저장할 수 있다는 점이 매력적이었다.

하지만 현실은 녹록지 않았다. 데이터 레이크는 빠르게 '데이터 늪(data swamp)'으로 변질됐다. 누가 어떤 데이터를 소유하는지 불분명하고, 데이터 품질을 보장할 방법이 없었다. 분석가들은 원하는 데이터를 찾는 데 분석하는 것보다 더 많은 시간을 써야 했다.

3세대: 레이크하우스와 데이터 메시

데이터 레이크의 한계에 대응하는 두 가지 접근이 등장했다. 하나는 기술적 해법인 레이크하우스(Lakehouse)이고, 다른 하나는 조직적 해법인 데이터 메시(Data Mesh)다.

레이크하우스는 Delta Lake, Apache Iceberg, Apache Hudi 같은 오픈 테이블 포맷을 통해 데이터 레이크에 ACID 트랜잭션, 스키마 진화, 타임 트래블 기능을 더한다. 저렴한 스토리지의 확장성과 웨어하우스의 안정성을 동시에 추구하는 접근이다.

데이터 메시는 중앙화된 데이터 팀이 모든 데이터를 관리하는 대신, 각 도메인 팀이 자신의 데이터를 직접 소유하고 관리하자는 철학이다. Zhamak Dehghani가 제안한 이 개념은 데이터를 도메인 중심으로 분산 소유하되, 연합 거버넌스로 일관성을 유지하는 구조다.

4세대: 데이터 프로덕트 시대

현재 선도적인 조직들이 향하고 있는 방향은 데이터를 '프로덕트'로 취급하는 것이다. 데이터 프로덕트란 명확한 소유자, 문서화된 인터페이스, 측정 가능한 품질 기준, 그리고 서비스 수준 협약(SLA)을 갖춘 데이터 자산이다.

소프트웨어 제품이 사용자 요구를 충족해야 하듯, 데이터 프로덕트는 데이터 소비자의 요구를 충족해야 한다. "우리가 이 데이터를 갖고 있는가?"가 아니라 "이 데이터가 소비하기 쉽고, 신뢰할 수 있으며, 제때 전달되는가?"를 묻는 것이 데이터 프로덕트 사고방식의 핵심이다.

기업 데이터 아키텍처의 핵심 원칙

1. 데이터 소유권의 명확화

모든 데이터 자산에는 명확한 소유자가 있어야 한다. 여기서 소유권은 기술적 관리만을 의미하지 않는다. 데이터의 정확성, 최신성, 접근 가능성에 대한 책임을 지는 것을 의미한다. 소유권이 불명확한 데이터는 반드시 품질이 저하된다.

2. 데이터 품질을 파이프라인에 내재화

데이터 품질은 사후 검증이 아니라 수집 시점부터 파이프라인에 내장되어야 한다. 완전성(completeness), 정확성(accuracy), 일관성(consistency), 시의성(timeliness)의 네 가지 차원을 지속적으로 모니터링하고, 품질 지표를 데이터 카탈로그에 노출해야 한다.

3. 메타데이터가 데이터만큼 중요하다

데이터를 찾고 이해하고 신뢰하는 것은 메타데이터 없이는 불가능하다. 비즈니스 용어 정의(business glossary), 데이터 계보(lineage), 기술 메타데이터를 통합 관리하는 데이터 카탈로그는 현대 EDA의 필수 구성 요소다.

4. 거버넌스와 민첩성의 균형

엄격한 거버넌스는 민첩성을 죽인다. 반대로 거버넌스 없는 민첩성은 혼란을 낳는다. 연합 거버넌스(federated governance)는 이 긴장을 해소하는 접근이다. 중앙에서 표준과 정책을 정의하되, 실행은 도메인 팀에 위임한다. 중앙 집중식 통제 없이도 일관성을 유지하는 것이 목표다.

5. 자가 서비스 역량

데이터 팀이 모든 분석 요청의 병목이 되어서는 안 된다. 비즈니스 사용자가 중앙 팀의 도움 없이도 안전하게 데이터를 탐색하고 분석할 수 있는 자가 서비스 인프라가 필요하다. 이는 민주적 데이터 접근과 적절한 접근 제어의 균형을 요구한다.

금융 서비스 관점: 실전에서의 EDA

금융 서비스 분야에서 EDA는 특별한 맥락을 갖는다. 규제 보고, 사기 탐지, 고객 360도 뷰, 실시간 리스크 평가가 모두 데이터 아키텍처에 의존하기 때문이다.

실시간 결제 시스템을 예로 들어보자. 카드 거래 이벤트가 생성되는 순간부터 사기 탐지 모델이 판단을 내리기까지의 지연 시간은 수백 밀리초 이내여야 한다. 이를 위해서는 스트리밍 파이프라인, 피처 스토어, 저지연 서빙 레이어가 유기적으로 연결되어야 한다. 동시에 규제 기관에 제출해야 하는 T+1 보고 데이터는 높은 정확성과 완전한 감사 추적을 요구한다.

이처럼 실시간성과 정확성, 접근성과 보안이라는 상충하는 요구를 동시에 충족하는 것이 금융 서비스 EDA의 핵심 도전이다.

데이터를 전략 자산으로 만들기 위한 실천 사항

데이터를 조직의 부채에서 전략 자산으로 전환하는 여정은 하루아침에 이루어지지 않는다. 저자가 제안하는 실천 사항은 다음과 같다.

첫째, 현재 상태를 솔직하게 진단한다. 데이터 인벤토리를 만들고, 어떤 데이터가 어디에 있으며 누가 소유하는지 파악하는 것이 출발점이다. 불편한 진실을 직면하는 것이 변화의 시작이다.

둘째, 비즈니스 가치 기준으로 우선순위를 정한다. 모든 데이터를 동시에 개선할 수는 없다. 가장 높은 비즈니스 가치를 창출하는 데이터 도메인부터 시작해 성공 사례를 만들어야 한다.

셋째, 기술보다 문화를 먼저 바꾼다. 최신 기술 스택을 도입하더라도 데이터 품질에 대한 책임 문화 없이는 결국 같은 문제가 반복된다. 데이터 프로덕트 오너십을 성과 지표에 반영해야 한다.

넷째, 작게 시작해 점진적으로 확장한다. 전사적 데이터 메시를 한 번에 구축하려는 시도는 대부분 실패한다. 파일럿 도메인에서 시작해 학습하고, 그 경험을 바탕으로 조직 전체로 확산하는 것이 현실적이다.

결론

데이터 레이크에서 데이터 프로덕트로의 전환은 기술적 업그레이드가 아니라 조직적 성숙이다. 스토리지 비용은 계속 떨어지고 있다. 진짜 경쟁 우위는 데이터를 얼마나 많이 저장하느냐가 아니라, 얼마나 빠르고 신뢰할 수 있게 가치로 전환하느냐에서 온다.

EDA는 이 전환을 가능하게 하는 조직 차원의 규율이다. 데이터를 제품으로 취급하고, 소유권을 명확히 하며, 품질을 파이프라인에 내재화하는 것이 그 시작점이다. 데이터가 넘쳐나는 시대에 진짜 부족한 것은 저장 공간이 아니라 데이터로 무언가를 실제로 해내는 역량이다.
