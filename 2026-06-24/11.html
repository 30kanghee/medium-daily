12 Companies. Billions of Users. Countless Engineering Lessons.

The Latency Gambler (@kanishks772) · 5 min read · 원문 · freedium

요약

YouTube, AWS, Google Docs, Kafka 등 12개 대형 시스템의 실제 아키텍처를 분석해 수십억 사용자 규모로 확장하는 공통 패턴을 정리한 실용 케이스 스터디 모음이다.

이 글은 인터넷을 움직이는 12개 시스템의 설계 결정을 통해 확장성 패턴을 실용적으로 정리한다. 규모와 문제 유형이 각기 다른 사례들을 살펴보면, 결국 공통된 원칙이 반복된다. 캐싱을 적극 활용하고, 관심사를 분리해 독립적으로 확장 가능한 컴포넌트를 구성하며, 복제·재시도·점진적 성능 저하로 장애에 대응한다. 가장 중요한 교훈은 확장은 혁명이 아니라 반복이라는 것, 그리고 성숙한 기술도 올바른 추상화를 갖추면 인터넷 규모까지 올라갈 수 있다는 사실이다.

규모 있는 시스템이 어떻게 작동하는지 이해하는 것은 2025년에 선택이 아니다. 이 12개의 케이스 스터디는 인터넷을 떠받치는 근본적인 패턴을 담고 있다. 각각 수십억 명의 사용자를 처리하는 것부터 대륙을 넘는 실시간 협업을 가능하게 하는 것까지, 서로 다른 종류의 문제를 풀어낸다.

YouTube: MySQL을 한계 너머로 밀어붙이다

YouTube는 25억 명에 가까운 사용자를 MySQL로 처리한다. MySQL은 애초에 이 규모를 위해 설계된 데이터베이스가 아니다. 비결은 Vitess다. MySQL을 분산 시스템으로 변환하는 추상화 계층이다. Vitess는 VTTablet 사이드카를 도입해 개별 MySQL 인스턴스를 관리하고, 비싼 쿼리를 재작성하며, 자주 접근하는 데이터를 캐싱한다. VTGate라는 프록시 계층이 일관 해싱을 사용해 쿼리를 올바른 샤드로 라우팅한다. 데이터베이스 페더레이션과 샤딩이 투명하게 처리되기 때문에, YouTube는 SQL의 운영 편의성을 유지하면서 수평 확장이 가능하다.

이 아키텍처가 증명하는 것은, 성숙한 기술도 올바른 추상화를 갖추면 인터넷 규모까지 올라갈 수 있다는 점이다. YouTube의 접근 방식은 최신 기술보다 운영 단순성을 우선시한다. 실용적인 시스템 설계가 무엇인지 보여주는 사례다.

AWS: 0명에서 1천만 사용자까지

Amazon이 제시하는 1천만 사용자 확장 레퍼런스 아키텍처는 점진적으로 성장하는 시스템을 구축하는 템플릿이다. 시작은 단순하다. 애플리케이션과 데이터베이스를 한 서버에서 돌린다. 그 다음 데이터베이스를 전용 장비로 분리한다. 읽기 복제본을 추가해 쿼리 부하를 분산한다. ElastiCache로 캐싱을 도입하고, CloudFront로 콘텐츠 전달을 처리한다. 로드 밸런서 뒤에 오토 스케일링 그룹을 구성한다. 쓰기 경합이 문제가 될 때는 결국 페더레이션과 샤딩이 필요해진다.

이 과정이 가르쳐주는 것은 확장은 혁명이 아니라 반복이라는 점이다. 각 병목 구간마다 특정 해법이 있다. 읽기가 많은 작업에는 캐싱, 가용성에는 복제, 쓰기 확장성에는 샤딩이다. 500만~1천만 사용자 구간에서는 기능별로 데이터를 분리하는 페더레이션과, 개별 데이터셋을 여러 호스트에 분산하는 샤딩이 함께 작동한다.

Bluesky: 탈중앙화 소셜, 중앙화처럼 느끼게

Bluesky는 탈중앙화 시스템을 중앙화된 것처럼 느끼게 만드는 방법을 보여준다. AT 프로토콜은 제공자가 바뀌어도 유지되는 탈중앙화 식별자(DID)를 사용한다. 개인 데이터 서버(PDS)는 머클 트리로 구조화된 사용자 저장소를 호스팅한다. 릴레이가 여러 PDS의 데이터를 집계해 단일 스트림으로 제공하고, AppView가 그 스트림을 소비해 사용자에게 보여준다.

혁신의 핵심은 계정 이식성이다. 사용자 신원이 서버가 아니라 DID에 암호학적으로 묶여 있기 때문에, 팔로워나 콘텐츠 손실 없이 제공자를 이동할 수 있다. 신원, 저장소, 표현을 분리해 각각 독립적으로 교체 가능하게 만든 아키텍처다.

Google Docs: 실시간 동시 편집의 어려운 문제

Google Docs는 분산 시스템에서 가장 어려운 문제 중 하나를 풀어냈다. 여러 사용자가 같은 문서를 동시에 편집하면서도 일관성을 유지하는 것이다. 해법은 운영 변환(Operational Transformation, OT)이다. 동시 편집을 순서에 관계없이 교환 가능하게 변환하는 알고리즘이다. 두 사용자가 같은 위치를 편집하면 OT가 작업 파라미터를 결정론적으로 조정해 충돌을 해결한다.

아키텍처는 상태를 메모리에 유지하는 협업 서버들로 구성되며, 이 서버들이 변환을 적용하고 WebSocket으로 변경 사항을 브로드캐스트한다. 메시지 큐가 WebSocket 서비스와 문서 서비스를 분리해 가용성과 신뢰성을 높인다. 모든 클라이언트가 같은 작업을 같은 순서로 적용하면 동일한 문서 상태에 수렴하는 최종 일관성 모델이다.

Apache Kafka: 분산 이벤트 스트리밍

Kafka는 데이터를 브로커에 분산된 추가 전용(append-only) 로그로 취급한다. 프로듀서가 클러스터 전체에 파티셔닝된 토픽에 쓴다. 컨슈머는 오프셋으로 추적하며 자신의 속도로 읽는다. 이 모델은 리플레이, 병렬 처리, 프로듀서와 컨슈머의 시간적 분리를 가능하게 한다. ZooKeeper가 클러스터 멤버십을 조율하고 브로커가 데이터 지속성과 복제를 담당하는 방식으로 관심사를 분리한다.

복제를 통한 내구성과 파티션 내 순서 보장이 시스템의 핵심 특성이다. 여러 컨슈머 그룹이 조율 없이 같은 스트림을 독립적으로 처리할 수 있어, 다양한 다운스트림 처리가 동시에 가능하다.

URL 단축기: 단순함 속의 시스템 설계 원칙

URL 단축기는 핵심 시스템 설계 원칙들을 작은 규모로 보여준다. 자동 증가 ID의 base62 인코딩이나 분산 ID 생성기로 고유한 단축 코드를 만든다. 읽기에 최적화된 키-값 저장소에 매핑을 저장한다. 대부분의 접근이 인기 있는 URL에 집중되므로 적극적으로 캐싱한다. 영구 리다이렉트는 HTTP 301, 임시 리다이렉트는 HTTP 302로 처리한다.

확장성은 캐싱과 지리적 분산에서 나온다. CDN이 전 세계적으로 캐싱된 리다이렉트를 최소 지연으로 제공한다. 데이터베이스는 단축 코드를 기준으로 샤딩해 고른 분산을 보장한다. 분석은 핵심 리다이렉트 경로와 분리되어 비동기로 처리된다.

Uber: 실시간 위치 서비스

Uber는 전 세계 도시에서 승객과 가까운 드라이버를 실시간으로 매칭한다. 아키텍처의 핵심은 S2 지오메트리나 지오해시를 사용해 세상을 셀로 분할하는 공간 인덱싱이다. 드라이버가 지속적인 연결을 통해 계속 위치 업데이트를 보내고, 매칭 서비스가 공간 인덱스를 조회해 픽업 위치 근처의 드라이버를 찾는다. 도착 예상 시간과 드라이버 평점 같은 요소도 함께 고려된다.

시스템은 지리적 샤딩으로 수백만 건의 동시 GPS 업데이트를 처리한다. 각 도시나 지역이 특정 샤드에 매핑되어 대부분의 작업이 로컬에서 처리된다. 상태 머신이 요청부터 완료까지 운행 과정을 추적하고, WebSocket이 승객과 드라이버 모두에게 양방향 실시간 업데이트를 가능하게 한다.

Slack: 엔터프라이즈 메시징의 규모

Slack은 채널로 이루어진 워크스페이스로 커뮤니케이션을 조직한다. 메시지는 채널 내에서 순서가 있는 시퀀스로, 스레드 대화와 읽음 확인 같은 기능을 가능하게 한다. 실시간 전달은 WebSocket으로 하되, 연결 실패 시 롱 폴링으로 폴백한다.

메시지 전달 보장은 승인 전에 메시지를 지속하는 메시지 큐에서 나온다. 검색 기능은 Elasticsearch로 메시지를 인덱싱해 대화 이력에서 전문 검색을 가능하게 한다. 현재 상태 표시기는 하트비트 메시지로 사용자 온라인 상태를 추적한다. 시스템은 워크스페이스 단위로 샤딩해 대부분의 작업이 단일 샤드 내에서 처리되게 한다.

ChatGPT: 대규모 언어 모델의 프로덕션 서빙

ChatGPT는 수십억 개의 파라미터를 가진 모델에 대한 추론 요청을 어떻게 서빙하는지 보여준다. 아키텍처는 모델 서빙과 대화 관리를 분리한다. 로드 밸런서가 GPU 클러스터의 모델 복제본들에 요청을 분산시킨다. 각 요청은 토큰을 순차적으로 생성하며 결과를 사용자에게 스트리밍으로 돌려보낸다.

비용 관리와 공정한 접근을 위해 속도 제한과 할당량 관리를 처리한다. 대화 이력은 빠른 키-값 저장소에 유지되며 모델 컨텍스트 창에 맞게 압축된다. 자주 요청되는 응답은 이전 완성 결과를 캐싱해 처리한다. 모니터링은 추론 지연, GPU 사용률, 모델 품질 지표를 추적한다.

Meta: 10억 사용자의 소셜 그래프

Facebook의 아키텍처는 초당 1,150만 건의 서버리스 함수 호출을 처리한다. 소셜 그래프는 사용자, 페이지, 콘텐츠 간의 관계를 분산 그래프 데이터베이스에 저장한다. 뉴스피드 생성은 사용자 참여를 예측하는 머신러닝 모델로 콘텐츠를 랭킹한다. 시스템은 여러 계층에서 공격적으로 캐싱한다. 정적 자산에는 CDN, 데이터베이스 쿼리에는 memcached, 활성 사용자에게는 사전 계산된 피드다.

쓰기 작업은 비동기 처리를 가능하게 하는 메시징 시스템을 거친다. 사진 업로드, 상태 업데이트, 댓글이 큐에 들어가고 워커 플릿이 처리한다. 이 분리 덕분에 프론트엔드는 백그라운드 작업이 계산 비용이 큰 작업을 처리하는 동안에도 반응성을 유지할 수 있다.

Reddit: 커뮤니티 중심 콘텐츠

Reddit의 아키텍처는 각자의 규칙과 모더레이터를 가진 독립적인 커뮤니티인 서브레딧을 중심으로 돌아간다. 콘텐츠 랭킹은 최근 투표에 더 가중치를 두는 투표 알고리즘을 사용한다. 시스템은 핫 랭킹, 논란, 새것, 상위 보기를 분리해 처리한다. 각각 다른 계산 방식이 필요하다.

스레딩은 수천 개의 댓글로 이루어진 중첩 토론을 가능하게 한다. 아키텍처는 랭킹을 주기적으로 사전 계산해 결과를 캐싱, 빠른 페이지 로딩을 지원한다. 속도 제한은 스팸과 남용을 방지하면서 정상적인 고빈도 사용은 허용한다. 시스템은 서브레딧 단위로 샤딩하며, 인기 있는 커뮤니티는 여러 데이터베이스 노드에 분산된다.

Redis: 서비스로서의 인메모리 자료구조

Redis는 문자열, 해시, 리스트, 집합, 정렬된 집합, 스트림 등 자료구조를 서비스로 제공한다. 모든 데이터를 메모리에 두어 마이크로초 지연을 달성하면서 디스크에 선택적으로 지속해 내구성을 확보한다. 마스터-복제본 복제는 읽기 확장과 장애 조치를 가능하게 한다. 클러스터 모드는 해시 슬롯을 사용해 데이터를 노드에 샤딩한다.

캐싱부터 세션 저장, 실시간 분석까지 다양한 용도로 쓰인다. pub/sub 시스템이 메시지 브로드캐스팅을 가능하게 하고, 트랜잭션이 여러 키에 걸친 원자적 작업을 제공한다. Lua 스크립팅으로 사용자 정의 로직을 서버에서 원자적으로 실행할 수 있다. 단순한 데이터 모델 덕분에 예측 가능한 성능을 유지하면서 다양한 애플리케이션에 활용된다.

12개 시스템을 관통하는 교훈

이 12개 시스템은 공통된 패턴을 공유한다. 대부분이 캐싱을 적극적으로 활용한다. 모두 관심사를 독립적으로 확장 가능한 컴포넌트로 분리한다. 각각 복제, 재시도, 점진적 성능 저하를 통해 장애에 대응한다. 최선의 시스템은 일반적인 경우에 최적화하면서 예외 상황도 올바르게 처리한다.

단순한 것에서 복잡한 것으로 가는 과정은 시사하는 바가 크다. 작동하는 가장 단순한 해법부터 시작하라. 병목이 어디서 발생하는지 측정하라. 측정된 문제에 구체적인 최적화를 적용하라. AWS의 확장 가이드에서 드러나는 이 반복적 접근 방식은 어디에나 적용된다.

이 시스템들을 이해하는 것은 면접 준비 그 이상이다. 제약 조건 속에서 아키텍처 결정을 내리는 직관을 키워준다. 당신이 다음에 구축할 시스템은 이 설계들로부터 패턴을 빌려올 것이고, 자신의 특정 컨텍스트와 요구사항에 맞게 변형될 것이다.
