DS-STAR: How Google built a Data Science agent that actually works

Jose Parreño (@joparga3) · Data Science Collective · 22 min read · 원문 · freedium

요약

Google의 DS-STAR는 모델을 바꾸지 않고 시스템 설계만으로 Gemini 2.5 Pro의 데이터 사이언스 정확도를 12%에서 45%로 끌어올린 7모듈 에이전트 파이프라인이다.

저자 Jose Parreño는 Claude Code 같은 범용 코딩 에이전트가 분포 비교, p-값, 신뢰 구간 같은 심층 통계 분석에서는 역부족이라는 한계를 직접 경험하고 DS-STAR 논문을 찾아냈다. DS-STAR의 핵심 주장은 강력한 모델이 아니라 그 주변의 시스템 설계가 성능을 결정한다는 것으로, Gemini 2.5 Pro 단독으로 12.70%였던 DABStep 하드 태스크 정확도를 7모듈 파이프라인 하나로 45.24%까지 끌어올렸다. 7개 에이전트(분석기, 플래너, 코더, 디버거, 검증기, 라우터, 최종화기)가 순차적으로 연결된 피드백 제어 루프 구조이며, DS-STAR+는 이를 확장해 개방형 질문에 대한 종합 리포트를 생성한다. 데이터 파일 자동 분석, 단계별 순차 계획, LLM 기반 검증이라는 세 가지 혁신이 성능 향상의 핵심이다.

왜 범용 AI 코딩 도구는 데이터 사이언스에서 부족한가

저자는 약 2년간 AI 코딩 도구를 써왔고, Claude Code를 지난 1년간 꾸준히 사용해왔다. 일반적인 엔지니어링 작업에서는 정말 인상적인 결과를 낸다. 하지만 실제 데이터 사이언스 작업으로 밀어붙이면 계속 벽에 부딪혔다. 평균 몇 개를 계산하는 정도로는 쓸모가 있었지만, 분포 비교, p-값, 신뢰 구간, 데이터 품질 점검처럼 데이터 사이언티스트라면 당연히 해야 할 심층 분석은 빠져 있었다.

제너럴리스트에게 스페셜리스트 일을 시키는 느낌. 유용하긴 한데 최상의 결과는 아니었다. 그래서 직접 찾아나섰고, 드디어 기다리던 논문을 발견했다.

DS-STAR란 무엇인가

DS-STAR(Data Science Agent for Solving Diverse Tasks)는 Google이 개발한 데이터 사이언스 에이전트다. 논문의 핵심 주장은 이렇다. 성능을 높이는 것은 새로운 프런티어 모델이 아니라 그 주변의 프레임워크와 시스템 설계다.

수치로 확인해보면:

  • Gemini 2.5 Pro 단독: DABStep 하드 태스크에서 12.70%
  • DS-STAR + Gemini 2.5 Pro: 45.24%
  • 차이: 32퍼센트포인트 상승
  • 모델은 바뀌지 않았다. 그 주변의 시스템이 달라졌을 뿐이다.

이 논리는 저자에게 익숙하게 다가온다. Claude Code가 다른 AI 코딩 도구를 앞서는 이유도 항상 최고의 프런티어 모델을 쓰기 때문이 아니라 시스템이 어떻게 연결되어 있느냐 덕분이다. DS-STAR는 같은 논리를 데이터 사이언스에 적용한다.

논문은 어디서 찾을 수 있나

논문 제목은 "DS-STAR: Data Science Agent for Solving Diverse Tasks across Heterogeneous Formats and Open-Ended Queries"이며 arXiv(2509.21825)에서 확인할 수 있다. 2025년 9월 제출, 2026년 2월 4차 개정판이 나왔다. KAIST의 Jaehyun Nam과 Google Cloud의 Jinsung Yoon, Jiefeng Chen, Raj Sinha, Tomas Pfister가 공동 저자다. Google이 공식 블로그에도 요약본을 게재했다.

논문이 풀고자 하는 문제는 야심 차다. 다양한 데이터 형식과 개방형 질문에 걸쳐 어떤 데이터 사이언스 질문에도 답할 수 있는 시스템을 만드는 것.

DS-STAR가 해결하는 문제: 왜 기존 에이전트는 실패하는가

실제 데이터 사이언스 작업이 어려운 이유는 데이터 자체에 있다. 하나의 깔끔한 CSV 파일과 명확히 정의된 질문으로 이루어지는 경우는 거의 없다. 여러 정형 데이터셋에 걸친 다단계 추론, 규칙에 녹아 있는 도메인 지식, 그리고 단 한 번의 코드 생성으로 답할 수 없는 질문들이 뒤엉켜 있다.

DABStep 벤치마크가 이 현실을 잘 포착한다. 450개 태스크 중 72개는 쉬운 문제(단일 데이터셋, 최소한의 맥락 지식)이고, 378개는 어려운 문제(여러 데이터셋, 도메인 지식 필요, 단일 패스 해결 불가)다. KramaBench는 여기서 더 나아가 104개 도전 과제에 1,700개 파일과 6개 도메인에 걸친 24개 데이터 소스를 담았다. 에이전트가 어떤 파일이 관련 있는지 스스로 발견해야 한다.

두 벤치마크 모두 "코드 블록 하나 쓰고 답 제출"하는 접근 방식을 깨뜨리도록 설계됐다. DS-STAR는 바로 그 접근 방식을 대체한다.

DS-STAR의 구조: 두 겹으로 쌓인 시스템

DS-STAR는 사실 두 가지 시스템이 하나 위에 다른 하나를 얹은 구조다.

  • DS-STAR: 내부 엔진. 명확하게 정의된 질문을 처리한다.
  • DS-STAR+: 외부 확장. 개방형 질문을 세부 하위 질문으로 분해하고, 각각을 DS-STAR에 넘긴 뒤, 작성 에이전트가 답변을 종합해 리포트를 만든다.

구조적으로 DS-STAR+가 DS-STAR를 내부 엔진으로 품는 형태다.

DS-STAR 깊이 파보기: 7개 모듈

모듈 1. 분석기(ANALYSER)

DS-STAR가 계획을 세우기 전에 가장 먼저 하는 일은 받은 데이터 파일 전체를 프로파일링하는 것이다. 각 파일마다 Python 스크립트를 생성해 실행하고, 출력 결과를 구조화된 설명으로 저장한다. 일종의 인덱서이자 소형 RAG라고 볼 수 있다.

흥미로운 점은 파일 설명이 LLM이 자유롭게 분석하는 방식이 아니라 Python으로 결정론적으로 생성된다는 것이다. 아래는 분석기 모듈의 프롬프트다.

You are an expert data analysist.
Generate a Python code that loads and describes the content of {filename}.

# Requirement
- The file can both unstructured or structured data.
- If there are too many structured data, print out just few examples.
- Print out essential informations. For example, print out all the column names.
- The Python code should print out the content of {filename}.
- The code should be a single-file Python program that is self-contained and can be
executed as-is.
- Your response should only contain a single code block.
- Important: You should not include dummy contents since we will debug if error occurs.
- Do not use try: and except: to prevent error. I will debug it later.

모듈 2. 플래너(PLANNER)

데이터 설명과 사용자 질문을 받아 실행 계획의 첫 번째 단계를 생성한다. 핵심은 전체 계획을 한 번에 만들지 않고, 딱 한 단계씩만 만든다는 것이다.

플래너에는 두 가지 모드가 있다. planner_init(첫 단계, 이전 계획 맥락 없음)과 planner_next(이후 단계, 현재 계획과 이전 실행 결과를 받아 다음 단계 제안). 이전 시도 결과를 다음 계획에 반영함으로써 시스템이 깊은 맥락을 갖는 기억처럼 작동한다.

모듈 3. 코더(CODER)

현재 계획 단계를 받아 실행 가능한 Python 코드를 생성한다. 플래너처럼 두 가지 모드(coder_init, coder_next)가 있으며, 이전 결과를 인식하면서 다음 코드를 작성한다. 코드는 단일 자립형(self-contained) 스크립트로 생성된다.

모듈 4. 디버거(DEBUGGER)

코더의 출력이 실제 데이터 파일에 실행됐을 때 실패하면(문법 오류, 없는 컬럼, 타입 불일치 등) 디버거가 작동한다. 실패한 코드와 오류 메시지를 받아 수정된 스크립트를 생성한다.

논문 Figure 1에서 두드러지지 않지만 알고리즘에는 명확히 존재하는 컴포넌트다. 코더와 검증기 사이에 위치해 실행 실패를 계획 평가 전에 처리한다. 디버거는 두 가지 프롬프트를 사용한다. 하나는 오류를 요약하고(트레이스백이 길어질 수 있으니), 다른 하나는 실제로 버그를 수정한다.

모듈 5. 검증기(VERIFIER)

코드가 성공적으로 실행된 후 검증기가 현재 계획이 원래 질문에 답하기 충분한지 평가한다. 판정 v는 이진값이다. sufficient 또는 insufficient.

누적 계획, 사용자 질문, 실행 코드, 실행 결과를 종합해 판단한다. 검증기는 이 시스템의 "센서" 역할을 한다.

모듈 6. 라우터(ROUTER)

검증기가 insufficient를 반환했을 때만 작동한다. 현재 계획, 질문, 실행 결과를 보고 이진 결정을 내린다.

  • 옵션 1. 새 단계 추가(Add Step): 계획의 끝에 새 단계를 덧붙인다.
  • 옵션 2. 기존 단계 교체(Replace Step K): 특정 단계 K를 수정한다.

옵션 2가 특히 영리하다. 계획 전체를 다시 쓰는 대신 오류가 있는 단일 단계만 정확히 수정할 수 있기 때문이다. 라우터는 이 시스템의 "컨트롤러" 역할을 한다.

모듈 7. 최종화기(FINALISER)

검증기가 sufficient를 반환하면 최종화기가 모든 실행 출력을 종합해 최종 답변을 만든다. 누적된 실행 이력에서 관련 결과를 추출하고 원래 질문에 대한 응답 형식으로 정리한다.

중요한 설계 원칙: 순차 파이프라인

파이프라인은 병렬이 아닌 순차로 실행된다. 각 에이전트의 출력이 다음 에이전트의 입력이 된다. 어느 하나를 바꾸면 그 이후 에이전트가 보는 것도 달라진다.

저자는 로보틱스 엔지니어링 배경에서 이 알고리즘이 피드백 제어 루프처럼 읽힌다고 말한다.

  • 검증기 = 센서
  • 라우터 = 컨트롤러
  • 플래너와 코더 = 액추에이터
  • 센서가 "완료" 신호를 보내면 루프가 종료된다.

DS-STAR+ 깊이 파보기

DS-STAR+는 개방형 질문을 다루는 확장판이다. 흐름은 다음과 같다.

  1. 분해기(Decomposer): 개방형 쿼리를 집중된 하위 질문들로 분해한다.
  2. DS-STAR 내부 엔진: 각 하위 질문을 처리한다.
  3. 작성 에이전트(Writer): 모든 답변을 종합해 구조화된 리포트를 작성한다.
  4. 평가자(Evaluator): 추가 분석 라운드 생성 여부를 결정한다.

성능 평가 결과

DS-STAR는 DABStep, KramaBench, DA-Code 세 가지 벤치마크에서 기존 최고 성능 방법들을 모두 앞질렀다.

  • DABStep: 최고 대안 대비 41.0% → 45.2%
  • KramaBench: 39.8% → 44.7%
  • DA-Code: 37.0% → 38.5%

특히 여러 데이터 파일이 필요한 하드 태스크에서 격차가 더 벌어졌다. DS-STAR는 2025년 9월 18일 기준 DABStep 공개 리더보드 1위를 차지했다.

구성 요소별 기여도 분석(Ablation Study)

데이터 파일 분석기가 없으면(Variant 1) DABStep 하드 태스크 정확도가 26.98%로 급락한다. 데이터 맥락이 계획과 구현의 토대임이 확인된다.

라우터가 없으면(Variant 2) 시스템은 새 단계를 순차적으로 추가하기만 한다. 실수를 수정하는 것이 단순히 단계를 추가하는 것보다 훨씬 효과적임을 보여준다.

LLM 범용성: GPT-5로 테스트했을 때도 좋은 결과가 나왔다. 쉬운 태스크는 GPT-5 버전이, 어려운 태스크는 Gemini 2.5 Pro 버전이 더 강했다.

반복 횟수 분석

어려운 태스크는 자연히 더 많은 반복을 필요로 한다. DABStep 벤치마크에서 하드 태스크는 평균 5.6라운드, 이지 태스크는 평균 3.0라운드가 필요했다. 이지 태스크의 50% 이상은 단 1라운드만에 해결됐다.

이 논문이 의미하는 바

DS-STAR에서 진짜 중요한 메시지는 모델 자체가 아니다. 모델 주변의 시스템 아키텍처가 성능을 결정한다는 것이다. 7개의 전문화된 에이전트, 결정론적 데이터 프로파일링, 단계별 계획과 검증, 그리고 정확한 오류 수정 능력이 결합됐을 때 범용 모델도 데이터 사이언티스트 수준의 분석을 수행할 수 있다.

실제 사용 사례가 단순한 내부 데이터베이스와 명확한 질문으로 이루어져 있더라도 이 아키텍처 선택은 여전히 유효하다. 이질적인 혼란을 처리할 수 있는 시스템은 깔끔한 데이터도 당연히 처리한다. 반대는 성립하지 않는다.
