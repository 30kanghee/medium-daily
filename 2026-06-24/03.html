Top 25 Generative AI System Design Interview Questions and Answers

Dr. Sanjay Kumar · 5 min read · 원문 · freedium

요약

LLM 기반 시스템을 프로덕션 환경에 설계하고 배포하는 데 필요한 핵심 개념과 실전 질문들을 아키텍처·비용·안전·확장 관점에서 총정리한 인터뷰 가이드다.

이 글은 대규모 언어 모델(LLM)을 실제 서비스에 적용하는 시스템 설계 인터뷰에 자주 등장하는 핵심 주제들을 다룬다. RAG(검색 증강 생성), 추론 최적화, 비기능 요구사항(NFR), 멀티 에이전트 설계, 비용 절감, 모니터링, 윤리적 리스크 등을 실제 사례와 함께 설명한다. 각 문항은 "무엇을 설계할 것인가"뿐 아니라 "왜 그 선택인가"라는 판단 근거까지 명확히 제시하는 형식으로 구성됐다. 고객 지원 챗봇, PDF 문서 대화 시스템, 대규모 트래픽 확장 시나리오 등 구체적인 케이스 스터디도 포함돼 있다.

Q1. LLM 시스템 설계란 무엇이며 왜 중요한가?

LLM 시스템 설계는 대규모 언어 모델을 프로덕션 환경에 배포하기 위한 엔드투엔드 아키텍처를 말한다. 구체적으로는 인프라(하드웨어, 클라우드 서비스, GPU/TPU 최적화), 추론 파이프라인(지연 시간 단축, 캐싱, 배치 처리), 통합(API, RAG, 안전 필터), 확장성(고트래픽 처리, 비용-성능 트레이드오프) 전반을 다룬다.

제대로 된 시스템 설계 없이는 높은 지연 시간, 환각 현상, 보안 위험, 지속 불가능한 비용 문제에 직면한다. 잘 설계된 시스템은 안정성, 확장성, 사용자 만족을 모두 확보할 수 있다.

Q2. RAG(검색 증강 생성)를 설명하고 LLM 시스템에서의 역할을 말하라

RAG는 외부 데이터(예: 벡터 데이터베이스)를 동적으로 가져와 LLM 출력을 사실에 기반하게 만드는 방식이다.

처리 순서는 다음과 같다. 첫째, 사용자 질의를 임베딩(예: SentenceTransformers)한다. 둘째, FAISS나 Pinecone 같은 도구로 가장 관련성 높은 문서 k개를 검색한다. 셋째, 이 컨텍스트를 프롬프트에 추가한 뒤 LLM에 전송한다.

RAG를 쓰는 이유는 세 가지다. 환각을 줄이고, 모델을 재학습하지 않고도 지식을 업데이트할 수 있으며, 고객 지원 같은 도메인 특화 질의의 정확도를 높인다.

Q3. 낮은 지연 시간과 높은 처리량을 위한 LLM 추론 최적화 방법은?

  • 양자화(Quantization): 모델 정밀도를 낮춰 추론을 가속한다(예: FP32 → INT8).
  • 캐싱: 자주 묻는 질의("환불 정책이 뭔가요?")의 응답을 저장해둔다.
  • 배치 처리: 여러 요청을 병렬로 처리한다.
  • 모델 증류(Distillation): 더 작게 파인튜닝된 모델(예: DistilBERT)을 사용한다.
  • 하드웨어: Triton이나 vLLM으로 효율적인 GPU/TPU 서빙 환경을 구성한다.

실제 사례로, 챗봇에 캐싱과 양자화를 함께 적용하면 지연 시간이 2초에서 200ms로 줄어들 수 있다.

Q4. LLM 시스템의 핵심 비기능 요구사항(NFR)을 설명하라

  • 지연 시간: 실시간 앱(예: 채팅)은 1초 미만이어야 한다.
  • 비용: 단순한 작업에는 저렴한 모델을 써서 비용을 최적화한다.
  • 확장성: 트래픽 급증에 대응하는 오토스케일링(Kubernetes).
  • 보안: 데이터 암호화, GDPR·HIPAA 등 규정 준수.
  • 신뢰성: 폴백 모델, 재시도 로직, 서킷 브레이커.
  • 편향/안전: 유해 콘텐츠 필터(예: Perspective API)와 공정성 점검.

Q5. 고객 지원 챗봇 설계 케이스 스터디

요구사항: 동시 접속 1만 명 이상 처리, 1초 미만 응답, 주문 추적 데이터베이스 연동.

솔루션:

  • 프론트엔드: 스트리밍 응답을 지원하는 React 기반 채팅 UI.
  • 백엔드: Pinecone 벡터 DB에서 주문 정보를 가져오는 RAG 파이프라인, LangChain 오케스트레이션, FastAPI 게이트웨이(속도 제한 포함).
  • 최적화: 양자화된 Llama-2 모델, 자주 묻는 질의를 위한 Redis 캐시, 지연 시간/SLO 모니터링을 위한 Prometheus.

결과: 비용 40% 절감, 의도 파악 정확도 95% 달성.

Q6. 환각이나 편향 같은 LLM의 한계는 어떻게 처리하는가?

  • 환각: RAG와 사실 검증 도구(예: VerifAI)를 함께 사용한다.
  • 편향: 정제된 데이터로 파인튜닝하고, 후처리 필터를 적용한다.
  • 안전: 규칙 기반 차단(욕설 등)과 적대적 테스트.
  • 투명성: 감사 목적으로 프롬프트와 응답을 기록한다.

Q7. 클라우드 vs. 엣지 배포 비교

클라우드 배포는 대규모 모델과 높은 트래픽에 적합하며 유연한 확장이 가능하지만 네트워크 지연과 비용이 단점이다. 엣지 배포는 오프라인 환경이나 개인정보 보호가 중요한 사용 사례에서 빛을 발한다. 저지연이 필수인 의료 기기나 산업 현장처럼 인터넷 연결이 불안정한 환경에서는 엣지가 유리하다. 실시간 고객 서비스나 다국어 번역처럼 대규모 처리가 필요한 경우에는 클라우드가 적합하다.

Q8. LLM 모니터링과 평가에 쓰는 도구는?

  • 지연 시간/처리량: Prometheus + Grafana.
  • 환각 감지: Luna 또는 수동 검토.
  • 유해성: Detoxify 또는 Perspective API.
  • 사용자 피드백: A/B 테스트 + 좋아요/싫어요 버튼.

Q9. LLM 시스템에서 비용과 성능 균형을 어떻게 맞추는가?

  • 계층형 모델: 복잡한 질의에는 GPT-4를, 단순한 작업에는 Mistral 같은 소형 모델을 사용한다.
  • 콜드 스타트: 소형 모델은 항상 워밍업 상태를 유지하고, 대형 모델은 필요할 때만 기동한다.
  • 스팟 인스턴스: 중요도가 낮은 워크로드에는 저렴한 클라우드 VM을 활용한다.

Q10. 프롬프트 일관성을 위한 접근법은?

  • 템플릿: "당신은 이커머스에 특화된 친절한 어시스턴트입니다" 같은 시스템 프롬프트를 사용한다.
  • 퓨샷 예시: 프롬프트에 2~3개의 예시를 포함한다.
  • 함수 호출: OpenAI의 tools API를 활용해 동적 데이터(예: "사용자 주문 상태 조회")를 가져온다.

Q11. LLM API 요청의 속도 제한(Rate Limiting) 시스템 설계

  • 토큰 버킷 알고리즘: 사용자/IP별 요청을 추적한다(예: 분당 100건).
  • API 게이트웨이: Kong, NGINX, AWS API Gateway로 제한을 적용한다.
  • 폴백: 한도 초과 시 캐시된 응답이나 가벼운 모델(예: Mistral)을 제공한다.
  • 우선순위 큐: 프리미엄 사용자에게 더 높은 처리량을 배정한다.

Q12. 양자화가 LLM 효율성을 어떻게 개선하는가? 트레이드오프는?

양자화는 모델 가중치를 32비트 부동소수점에서 8비트 정수로 줄이는 기법이다. 추론이 빨라지고 메모리가 줄어드는(예: 모델 크기 4배 감소) 이점이 있지만, 정확도가 소폭 떨어진다(벤치마크 기준 약 1~2%). 주요 도구로는 ONNX Runtime과 TensorRT가 있다.

Q13. 새 LLM 버전의 무중단 롤아웃 방법은?

  • 블루-그린 배포: 두 개의 동일한 환경 사이에서 트래픽을 전환한다.
  • 섀도우 테스트: 두 모델에 동시에 쿼리를 보내 출력을 비교한다.
  • 카나리 릴리즈: 트래픽을 1%에서 100%로 점진적으로 전환한다.
  • 롤백 계획: 지연 시간, 오류율 등 지표를 모니터링하며 문제 시 이전 버전으로 되돌린다.

Q14. 서로 다른 모델이 각기 다른 작업을 처리하는 멀티 에이전트 LLM 시스템 설계

  • 라우터 모델: 쿼리를 분류한다(예: "고객 지원" → GPT-4, "번역" → NLLB).
  • 오케스트레이터: LangChain이나 Celery로 워크플로를 관리한다.
  • 데이터 전달: 에이전트 간 JSON 페이로드로 컨텍스트를 넘긴다.
  • 폴백: 오류 시 범용 모델(예: Claude)로 기본 응답한다.

Q15. 트래픽이 많은 LLM 앱에서 GPU 비용을 줄이는 방법은?

  • 모델 증류: 대형 모델을 모방하는 소형 모델(예: TinyLlama)을 훈련한다.
  • 희소 MoE(Mixture of Experts): 쿼리별로 관련 모델 레이어만 활성화한다.
  • 스팟 인스턴스: 배치 작업에 AWS/GCP 스팟 VM을 활용한다.
  • 캐싱: 자주 묻는 응답(예: "반품 정책이 뭔가요?")을 캐시한다.

Q16. LLM 기반 고객 지원 봇의 기능 우선순위는 어떻게 정하는가?

영향 대비 노력 매트릭스를 활용한다. 영향이 크고 노력이 적은 것(예: RAG 통합으로 환각 감소)을 먼저 한다. 영향이 크지만 노력도 많이 드는 것(예: 전화 지원을 위한 음성-텍스트 변환)은 이후 단계로 미룬다. 반복적인 상위 20% 질의(예: 주문 추적)에서 시작해서 CSAT, 해결 시간, 전환율을 지표로 삼는다.

Q17. LLM 프로젝트의 ROI를 어떻게 측정하는가?

  • 비용 절감: 인건비와 LLM 운영 비용을 비교한다.
  • 매출 영향: 업셀 성공률(예: 챗봇의 제품 추천 전환율).
  • 효율성: 시간당 처리 질의 수, 상담사 지원 소요 시간.
  • 정성적 지표: 사용자 피드백(NPS 설문).

Q18. LLM과 관련해 제품 관리자가 고려해야 할 윤리적 리스크는?

  • 편향: 학습 데이터가 다양한 인구 집단을 대표하는지 확인한다.
  • 개인정보: 개인 식별 정보(PII) 로깅을 피하고 익명화를 적용한다.
  • 투명성: AI 사용 여부를 공개한다("이 응답은 자동화된 것입니다").
  • 악용 방지: 스팸/허위정보 차단을 위한 속도 제한.

Q19. LLM이 잘못된 법률 조언을 제공하는 상황은 어떻게 처리하는가?

두 가지 안전 장치를 적용한다. 전처리 단계에서는 법률 키워드("소송", "계약")를 감지해 사람 상담원에게 라우팅한다. 후처리 단계에서는 "변호사와 상담하세요"라는 면책 문구를 자동으로 추가한다. 모든 법률 관련 질의는 감사 로그에 플래그를 달아 검토한다. 필요하다면 법률 학습 데이터를 제거하거나 Harvey AI 같은 전문화된 모델을 사용하는 파인튜닝도 검토한다.

Q20. 'PDF와 대화하기' 시스템 설계 (RAG 집중 사례)

처리 흐름:

  1. PDF 업로드: PyPDF나 Unstructured.io로 텍스트를 추출한다.
  2. 청크 분할 및 임베딩: 단락 단위로 쪼개고 OpenAI 임베딩으로 벡터화한다.
  3. 쿼리 처리: 사용자가 "이 문서를 요약해줘"라고 물으면 RAG가 관련 청크를 검색하고 LLM이 요약한다.

도구: LangChain(오케스트레이션), Pinecone(벡터 DB), Next.js(UI).

최적화: 지연 시간을 줄이기 위해 업로드 시점에 미리 임베딩을 생성해둔다. 자주 묻는 질의(예: "핵심 내용이 뭔가요?")는 캐시한다.

Q21. 사용자 100명에서 100만 명으로 LLM 앱 확장하기

인프라:

  • 무상태 API: Kubernetes로 수평 확장한다.
  • CDN: 정적 자산(JS, 모델 가중치 등)을 캐시한다.

데이터베이스:

  • 읽기 복제본: 벡터 DB(Weaviate/Chroma) 읽기 부하를 분산한다.
  • 샤딩: 사용자 데이터를 지역별로 파티셔닝한다.

비용 통제:

  • 오토스케일링: 트래픽이 주기적이라면 야간에 스케일다운한다.
  • 스팟 인스턴스: 문서 재임베딩 같은 배치 작업에 활용한다.
