Customizing CI/CD with Custom Pipelines in dbt: A Practical Guide

Abhishek Kumar Gupta (@abhishekkrgupta0) · Tech with Abhishek · 10 min read · 원문 · freedium

요약

dbt와 Snowflake 환경에서 GitHub Actions를 활용해 CI/CD 파이프라인을 처음부터 구축하는 방법을 단계별로 안내하는 실용 가이드다.

애널리틱스 엔지니어링이 성숙해가면서 데이터 파이프라인에도 CI/CD는 선택이 아닌 필수가 됐다. 이 글은 dbt와 Snowflake 조합에 GitHub Actions를 연결해 PR마다 자동 검증이 돌아가는 파이프라인을 실제 코드와 함께 보여준다. CI 단계에서는 격리된 스키마를 생성해 테스트를 실행하고, CD 단계에서는 리뷰와 CI 통과 후에만 프로덕션에 배포한다. 격리 스키마의 자동 정리, Slack 알림 연동, 롤백 전략, GenAI 기반 SQL 린팅 통합 같은 실무 팁도 함께 다룬다.

데이터 팀에 CI/CD가 왜 필요한가

애널리틱스 엔지니어링이 무르익으면서 CI/CD는 더 이상 소프트웨어 개발만의 이야기가 아니게 됐다. 데이터 파이프라인에도 없어서는 안 될 요소다. dbt와 Snowflake 환경에서 CI/CD를 갖추면 모든 모델, 테스트, 변환이 버전 관리되고, 검증되고, 자신감 있게 배포된다.

GitHub Actions는 클라우드 네이티브 자동화, 확장성, 투명성을 dbt + Snowflake 워크플로우에 가져다 준다. 오류를 일찍 잡아내고, 변경 사항을 더 빠르게 출시하며, 팀 전체가 역량을 발휘할 수 있게 해 준다.

이 가이드에서는 GitHub Actions로 dbt와 Snowflake를 위한 견고한 CI/CD 파이프라인을 실제 코드 샘플과 실무 팁과 함께 구축해 본다.

CI/CD 아키텍처 전체 그림

기본 워크플로우는 다음과 같다.

개발자 PR → GitHub Actions(린트, dbt 빌드, 테스트, 문서 생성) → 리뷰/승인 → Snowflake 프로덕션 배포

지속적 통합(CI): 모든 pull request가 자동 검사를 트리거한다. 코드 린팅, dbt 컴파일과 빌드, 모델 테스트, 문서 생성이 자동으로 실행된다.

지속적 배포(CD): 승인된 변경 사항은 프로덕션 Snowflake 환경에 자동으로 배포된다.

단계별 구축: 파이프라인 처음부터 만들기

1단계. GitHub 저장소 설정

dbt 프로젝트를 GitHub 저장소에 올린다. 브랜치 전략을 명확하게 잡아두는 것이 중요하다. main, develop, feature/* 같은 패턴이 일반적이다.

2단계. CI/CD용 dbt 프로필 구성

CI/CD 전용 Snowflake 사용자와 역할을 최소 권한 원칙으로 만든다. 자격 증명은 GitHub Secrets에 보관한다. SNOWFLAKE_USER, SNOWFLAKE_PASSWORD, SNOWFLAKE_ACCOUNT 같은 이름으로 등록해 두면 워크플로우 파일에서 참조할 수 있다.

3단계. GitHub Actions 워크플로우 작성

.github/workflows/dbt-ci-cd.yml 파일을 생성한다.

name: dbt CI/CD

on:
  pull_request:
    branches: [main, develop]
  push:
    branches: [main]

jobs:
  dbt-checks:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: Install dbt and dependencies
        run: |
          pip install dbt-snowflake

      - name: Configure dbt profile
        run: |
          mkdir -p ~/.dbt
          echo "
          snowflake:
            target: ci
            outputs:
              ci:
                type: snowflake
                account: ${{ secrets.SNOWFLAKE_ACCOUNT }}
                user: ${{ secrets.SNOWFLAKE_USER }}
                password: ${{ secrets.SNOWFLAKE_PASSWORD }}
                role: ${{ secrets.SNOWFLAKE_ROLE }}
                database: ${{ secrets.SNOWFLAKE_DATABASE }}
                warehouse: ${{ secrets.SNOWFLAKE_WAREHOUSE }}
                schema: CI_${{ github.run_id }}
                threads: 4
                client_session_keep_alive: False
          " > ~/.dbt/profiles.yml

      - name: dbt deps
        run: dbt deps

      - name: dbt compile
        run: dbt compile

      - name: dbt build (run + test)
        run: dbt build --target ci

      - name: dbt docs generate
        run: dbt docs generate --target ci

      - name: Cleanup CI schema
        if: always()
        run: |
          dbt run-operation drop_schema \
            --args '{"schema_name": "CI_${{ github.run_id }}"}' \
            --target ci

워크플로우의 핵심을 하나씩 살펴보자.

트리거 조건: main과 develop 브랜치에 대한 pull request, 그리고 main에 대한 직접 push 모두에서 실행된다.

격리 스키마: schema: CI_${{ github.run_id }}가 핵심이다. 각 CI 실행마다 고유한 스키마를 생성해서 테스트들이 서로 충돌하지 않는다.

항상 정리: if: always()로 설정된 정리 스텝은 빌드가 성공하든 실패하든 CI 스키마를 반드시 삭제한다. 저장 비용을 줄이는 중요한 설정이다.

4단계. 실무 모범 사례

격리 스키마 사용: CI 실행마다 CI_<run_id> 같은 고유 스키마를 써서 충돌을 방지하고 테스트 데이터를 격리한다.

자동 정리: 테스트 후에는 항상 CI 스키마를 삭제한다. 저장 비용 절감에 직접적인 효과가 있다.

빠른 실패(Fail Fast): dbt 모델이나 테스트에서 오류가 발생하면 워크플로우가 즉시 실패하도록 설정한다. 문제를 일찍 발견할수록 수정 비용이 낮다.

문서를 아티팩트로: 생성된 dbt 문서를 GitHub Pages나 워크플로우 아티팩트로 게시한다. 리뷰어가 변경 사항의 영향을 시각적으로 확인할 수 있다.

Slack/Teams 알림: 빌드 성공과 실패 알림을 채팅 도구와 연동한다. 팀원들이 파이프라인 상태를 실시간으로 파악할 수 있다.

5단계. 프로덕션 배포(CD)

프로덕션에는 PR 리뷰와 CI 검사를 모두 통과한 후에만 배포한다. main 브랜치 머지에 트리거되는 별도 잡이나 워크플로우를 사용한다.

  • 프로덕션 자격 증명으로 dbt build 또는 dbt run을 실행한다.
  • 배포 전후로 dbt source freshness 검사를 추가하면 데이터 신선도까지 확인할 수 있다.

모니터링, 관찰 가능성, 롤백

파이프라인을 구축하고 나면 운영 관점에서 세 가지를 챙겨야 한다.

dbt Cloud 아티팩트 활용: run_results.json과 manifest.json은 감사 추적과 디버깅에 유용하다. 특정 실행에서 어떤 모델이 어떤 상태로 끝났는지 확인할 수 있다.

Snowflake 쿼리 이력 모니터링: 실패한 쿼리나 비정상적으로 오래 걸리는 쿼리를 정기적으로 살펴본다. 데이터 품질 문제가 쿼리 성능으로 먼저 드러나는 경우가 많다.

자동 롤백: 배포가 실패하면 GitHub Actions로 이전 커밋으로 되돌아가도록 설정할 수 있다. 롤백 절차를 미리 자동화해 두면 장애 상황에서 빠르게 대응할 수 있다.

배포 태깅: GitHub에서 릴리즈마다 태그를 달아 추적 가능성을 확보한다. 어떤 버전이 언제 프로덕션에 나갔는지 명확하게 알 수 있어야 한다.

실제 사례: 글로벌 리테일 팀의 구성

한 글로벌 유통 기업이 이 설정을 다음과 같이 운영하고 있다.

  • 새 모델 개발은 모두 feature 브랜치에서 시작한다.
  • 모든 PR이 격리된 Snowflake 스키마에서 dbt 빌드와 테스트를 자동으로 실행한다.
  • 빌드 실패 시 Slack으로 즉시 알림이 간다.
  • 동료 리뷰와 CI 녹색 통과 후에만 프로덕션 배포가 진행된다.
  • dbt 문서가 비즈니스 사용자를 위해 자동으로 게시된다.

이 구성 덕분에 팀은 매주 수십 건의 모델 변경을 안정적으로 배포한다. 프로덕션 장애는 크게 줄었고, 신규 팀원도 빠르게 온보딩된다.

고급 팁과 주의사항

한 발 더 나아가기

GenAI 파이프라인 통합: LLM 기반 SQL 린터나 코드 리뷰 봇을 워크플로우에 연결해 보자. SQL 스타일 가이드 준수 여부를 자동으로 검사하거나, 잠재적 성능 문제를 사전에 탐지할 수 있다.

다중 환경 배포: GitHub Actions의 matrix 잡을 활용하면 dev, staging, prod 같은 여러 Snowflake 환경에서 동시에 테스트할 수 있다. 환경별 동작 차이를 조기에 발견하는 데 효과적이다.

데이터 계약 강제: CI의 일부로 스키마와 품질 계약(data contract)을 검사하는 단계를 추가한다. 모델 인터페이스가 다운스트림 소비자와 계속 호환되는지 보장할 수 있다.

관찰 가능성 도구 연동: Monte Carlo나 Datafold 같은 도구를 연동하면 데이터 계보(lineage)와 이상 탐지 기능을 CI/CD 파이프라인에 더할 수 있다. 데이터 품질 문제를 프로덕션 도달 전에 잡는 것이 목표다.

자주 빠지는 함정

CI 스키마 정리 누락: if: always() 없이 정리 스텝을 두면 실패한 빌드 후에 CI 스키마가 쌓인다. 저장 비용이 눈에 띄지 않게 늘어나는 원인이 된다.

하드코딩된 자격 증명: 모든 Snowflake 자격 증명은 반드시 GitHub Secrets를 통해 주입해야 한다. 워크플로우 파일에 직접 쓰는 것은 보안 사고로 이어진다.

테스트 없는 배포: dbt build는 dbt run과 dbt test를 함께 실행한다. dbt run만 쓰면 모델 테스트가 빠진다. CI 단계에서는 반드시 dbt build를 사용한다.

프로덕션 배포 전 리뷰 생략: 자동화가 갖춰져 있어도 프로덕션 배포는 인간의 리뷰를 거쳐야 한다. CI가 통과됐다고 해서 코드가 비즈니스 요구사항을 충족한다는 의미는 아니다.

마무리: 더 빠르게, 더 안전하게, 더 스마트하게

GitHub Actions, dbt, Snowflake로 구동되는 CI/CD를 갖추면 데이터 팀이 세계 수준의 소프트웨어 엔지니어링과 같은 속도와 엄밀함으로 분석을 전달할 수 있다.

모든 것을 자동화하고, 문제를 일찍 잡고, 팀이 진짜 중요한 것에 집중할 수 있도록 하라. 인사이트를 발굴하고 비즈니스 가치를 만드는 것이다. 파이프라인 관리에 지치는 것이 아니라.

dbt CI/CD를 어떻게 자동화하고 있는지, 어떤 도구를 쓰고 어떤 시행착오를 겪었는지 댓글로 나눠보자.
